The bug is in the translate method's loop when a translator consumes characters. After translate_codepoint returns consumed > 0, the loop incorrectly uses pos as the starting index for char_count calculations, but pos hasn't been advanced yet. This causes code_point_at(code_units, pos) to be called repeatedly on the same position, and when consumed=2 for a surrogate pair at the end of short strings, it tries to access index 2 (out of bounds). Fixed by advancing pos first by the consumed codepoints using proper UTF-16 char counting.
