The bug occurs because the translate method incorrectly advances the position `pos` after `translate_codepoint` returns a non-zero `consumed` value. It calls `code_point_at(code_units, pos)` (which uses the original `pos`) inside a loop `range(consumed)`, but `pos` hasn't been updated yet, and for surrogate pairs this can cause invalid index access. The fix updates `pos` incrementally by consuming the correct number of code units for each codepoint consumed by the translator.
