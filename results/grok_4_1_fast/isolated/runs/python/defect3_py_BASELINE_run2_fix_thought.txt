The bug occurs because after `translate_codepoint` consumes multiple code units (e.g., 2 for a surrogate pair), the loop incorrectly iterates over each consumed unit and calls `code_point_at(code_units, pos)` multiple times from the same starting `pos`, causing out-of-bounds access. The fix is to advance `pos` by the total number of code units consumed in one step, matching Java's behavior where translators report code units consumed.
