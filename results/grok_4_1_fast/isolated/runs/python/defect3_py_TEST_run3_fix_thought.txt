The bug is in the translate method's surrogate pair advancement logic. When translate_codepoint returns 2 (indicating a surrogate pair was consumed), the loop incorrectly advances pos by summing char_count for each of the 2 codepoints starting from the current pos (which is still the high surrogate). This causes it to process the high surrogate again. The fix is to advance pos by the total number of code units consumed (consumed * average char units, but since consumed represents codepoints, we calculate total code units directly).
