The bug diagnosis incorrectly identifies the issue. `to_utf16_code_units` correctly populates `code_units` from the input string. The real bug is in `translate`: when a translator returns `consumed > 0`, the loop incorrectly uses `char_count(code_point_at(code_units, pos))` which calls `code_point_at` with the wrong index `pos` (already advanced). It should simply advance by `consumed` code units since `translate_codepoint` already consumed them and wrote output.
