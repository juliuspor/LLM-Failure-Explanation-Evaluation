The bug occurs because the implementation finds the earliest match across all search strings but doesn't consider overlapping matches or the longest match rule. When 'b' matches at position 1 but 'a' matches earlier at position 0, it applies 'a'->'c' first, then incorrectly finds the next 'a' at position 2 instead of 'b' at position 1 after advancing. Fixed by using a position-based approach that checks all search strings starting from current position and selects the longest match at each step, properly handling null replacements by skipping them and advancing past matched positions.
