The bug is in the `translate` method's position advancement logic. When `consumed == 0`, it correctly advances by `len(c)` (UTF-16 code unit count), but when `consumed > 0`, it incorrectly advances by summing `char_count(code_point_at(code_units, pos))` for each consumed codepoint, which calls `code_point_at` with an already-consumed position, causing IndexError on surrogate pairs. Fix by advancing `pos += consumed` since `translate_codepoint` returns the number of code units consumed.
