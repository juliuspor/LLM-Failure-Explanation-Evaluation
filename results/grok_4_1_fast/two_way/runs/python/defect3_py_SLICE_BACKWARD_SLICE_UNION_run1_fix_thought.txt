The diagnosis misidentifies the bug. `to_utf16_code_units` correctly populates the list (used in `translate` method, not `code_point_at`). The real bug is in `translate` method: when translators return `consumed > 0`, the loop `for pt in range(consumed): pos += char_count(code_point_at(code_units, pos))` uses the wrong `pos` (already advanced incorrectly) and redundant codepoint lookups. Fix by advancing `pos` directly by `consumed` code units.
