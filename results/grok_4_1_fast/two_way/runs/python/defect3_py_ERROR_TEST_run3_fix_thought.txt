The bug is in the translate method's loop. When translate_codepoint returns consumed > 0, the code incorrectly advances pos by summing char_count(code_point_at(code_units, pos)) for each consumed item, but pos isn't updated between calls, causing it to repeatedly read the same position and potentially over-advance. Fixed by advancing pos by the total UTF-16 code units consumed (consumed * average char size, but simplified to correct surrogate handling). For surrogate pairs, when consumed=2, advance by exactly 2 code units.
