The comment states translators handle surrogate pairs by consuming multiple code units (e.g., `consumed=2` for a pair). However, inside the loop, `code_point_at(code_units, pos)` is called repeatedly at the **same** `pos` value each iteration because `pos` isn't updated until after the call.

But since `pos` doesn't advance between iterations, every call to `code_point_at(code_units, pos)` returns the **same** large codepoint.

Then `char_count(large_codepoint)` returns 2 each time. For `consumed=2`, `pos` gets incremented by `2 + 2 = 4`, skipping ahead incorrectly and corrupting the position tracking for subsequent UTF-16 code units.
