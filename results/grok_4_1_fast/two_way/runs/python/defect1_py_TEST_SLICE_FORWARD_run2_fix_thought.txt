The bug is in `for_offset_hours_minutes`: range checks for hours (-23 to +23) and minutes (0 to 59) are performed, but minutes can be negative due to input validation allowing it, and boundary cases like hours=24 don't trigger ValueError because the final offset check uses _MAX_MILLIS which allows up to 23:59:59.999. Fix by strengthening input validation: ensure minutes >= 0 and total absolute hours (hours + minutes/60) stays within -23 to +23 bounds.
