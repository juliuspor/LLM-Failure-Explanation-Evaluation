The bug is in the `translate` method's index advancement logic after `translate_codepoint` consumes code units. When a translator like `CsvEscaper` returns `consumed=2` for a surrogate pair, the loop incorrectly uses `char_count(code_point_at(code_units, pos))` which calls `code_point_at` at the current `pos` (start of pair), not the advanced position. This causes double advancement. Fixed by simply advancing `pos` by `consumed` directly, as `translate_codepoint` returns the exact number of code units consumed.
