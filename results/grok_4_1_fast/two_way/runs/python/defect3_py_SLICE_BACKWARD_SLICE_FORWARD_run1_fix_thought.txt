The bug diagnosis is incorrect - code_units is properly populated by to_utf16_code_units() before code_point_at() is called. The real bug is in the translate() method's main loop: after translate_codepoint() returns a non-zero consumed count, the loop incorrectly advances pos using char_count(code_point_at(code_units, pos)) in a for loop, which re-reads the already-consumed codepoint and causes infinite loop or incorrect advancement. Fix by advancing pos by consumed code units directly.
