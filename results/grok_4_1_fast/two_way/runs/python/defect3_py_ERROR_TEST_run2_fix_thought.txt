The bug is in the translate method's index advancement logic after a translator consumes codepoints. When consumed > 0, it incorrectly advances pos by summing char_count(code_point_at(pos)) for each consumed codepoint, but pos hasn't been updated yet, causing code_point_at to be called at the wrong position and overshooting. Fix by advancing pos by the consumed count directly, as translate_codepoint returns the number of code units consumed.
