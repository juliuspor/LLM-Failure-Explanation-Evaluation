The bug diagnosis references a non-existent `from_utf16_code_unit` function and incorrect line numbers. The actual issue is in `CharSequenceTranslator.translate`: after `translate_codepoint` returns `consumed > 0`, the loop `for pt in range(consumed): pos += char_count(code_point_at(code_units, pos))` incorrectly calls `code_point_at(code_units, pos)` where `pos` points to already-consumed surrogate pairs, causing infinite loops or incorrect advancement. Fix by advancing `pos` directly by `consumed` code units.
