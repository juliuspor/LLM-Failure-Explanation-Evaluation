The bug occurs because `random_letters_numbers` passes `start=0, end=0` to `_random_internal`, which sets `gap=0` and causes `_next_int_java(rnd, 0)` to raise ValueError('bound must be positive'). The fix sets proper ASCII bounds (32-127) in `random` when `start==end==0` to match Java behavior and test expectations, preventing non-positive `gap`.
