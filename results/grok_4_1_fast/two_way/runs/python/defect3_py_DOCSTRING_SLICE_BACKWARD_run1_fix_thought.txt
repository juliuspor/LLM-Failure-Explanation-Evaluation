The bug diagnosis is incorrect - code_units is properly populated by to_utf16_code_units(input_seq) before the loop in translate(). The real issue is in the surrogate pair handling loop: after translate_codepoint() consumes characters, pos is advanced using char_count(code_point_at(code_units, pos)), but code_point_at() uses the current pos (already advanced by consumed chars), causing double-skipping and infinite loops or incorrect positioning. Fix by calculating the correct advancement using the returned consumed count directly with proper UTF-16 unit counting.
