The bug occurs because `RandomStringUtils.random` passes `start=0` to `_random_internal`, which derives `gap = end - start`. When `letters` or `numbers` are True and `chars` is None, `end` is set to `ord('z') + 1` but `start` remains 0, making `gap` too large. The fix sets appropriate `start` values matching Java's behavior: `ord('a')` for letters, `ord('0')` for numbers.
