The code fails because of inconsistent handling of negative `minutes_offset` values between two code paths. On line 377, if `minutes_offset < 0`, it raises `ValueError` immediately. However, in the `SLICE_UNION` path (lines 383-385), when `minutes_offset` is negative, the code executes `minutes_offset = hours_in_minutes - minutes_offset`, making it positive (e.g., -30 becomes 30 if hours_in_minutes=60). Then on line 386, `FieldUtils.safe_multiply(minutes_offset, DateTimeConstants.MILLIS_PER_MINUTE)` detects the resulting large offset value and throws an exception, reaching line 388's `raise ValueError("Offset is too large")`. This creates a bug: negative offsets that could be valid (like -30 minutes) are rejected with the wrong error message instead of being normalized correctly, such as by always using `hours_in_minutes + minutes_offset` for negative values to get 30 minutes.
