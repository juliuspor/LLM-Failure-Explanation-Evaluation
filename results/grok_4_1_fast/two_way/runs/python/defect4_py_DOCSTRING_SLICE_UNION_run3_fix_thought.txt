The bug is in the logic for updating min/max indices. When an index is already set (>=0), the code compares the new period's time with the current min/max period's time but always overwrites the index with the new `index` regardless of the comparison result. This causes indices to always track the most recently added item. Fixed by only updating the index when the comparison actually shows the new period is smaller/larger. Also fixed a bug where `_max_middle_index` incorrectly used `_min_middle_index`.
