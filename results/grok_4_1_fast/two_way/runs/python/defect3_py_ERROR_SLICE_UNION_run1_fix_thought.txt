The bug diagnosis is incorrect - the IndexError occurs because `code_point_at` is called with an index >= len(code_units), which is properly caught by the bounds check. The real issue is in the `translate` method's main loop: when `translate_codepoint` returns a non-zero `consumed` value, the loop incorrectly calls `code_point_at(code_units, pos)` (where `pos` is the unchanged starting position) inside `char_count` calls, causing repeated IndexErrors for surrogate pairs. The fix modifies the loop to advance `pos` by `consumed` code units directly, matching Java's CharSequenceTranslator behavior where `consumed` represents UTF-16 code units consumed.
