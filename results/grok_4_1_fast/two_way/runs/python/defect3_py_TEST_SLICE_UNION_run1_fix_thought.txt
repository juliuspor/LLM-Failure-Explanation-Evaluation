The bug is in the translate method's surrogate pair advancement logic. When a translator consumes codepoints but returns consumed=1 (not 0), the loop incorrectly calls code_point_at(pos) where pos points to a low surrogate or single code unit, then advances by char_count of that, mismatching the consumed count. Fix by advancing pos by consumed code units directly when consumed > 0, since translators must return the exact number of code units they consumed.
