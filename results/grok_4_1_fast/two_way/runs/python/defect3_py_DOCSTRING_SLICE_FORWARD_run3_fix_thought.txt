The bug is in the translate method's surrogate pair handling after successful translation. When translate_codepoint consumes characters (including surrogate pairs), the loop incorrectly advances pos using char_count(code_point_at(code_units, pos)) which always uses the current pos (first code unit of pair). This fails to properly skip the low surrogate. Fixed by advancing pos by the raw number of code units consumed (consumed * char unit size), matching Java's char[] advancement logic.
