The bug is in the translate method's main loop. After translate_codepoint returns a non-zero consumed count, the loop incorrectly advances pos by summing char_count for each of 'consumed' codepoints starting from the current pos (which is wrong). Instead, it should simply advance pos by the returned consumed value, as translate_codepoint is responsible for consuming that many code units. The existing bounds check in code_point_at prevents the IndexError, but the advancement logic is flawed. Fix by changing the advancement to pos += consumed.
