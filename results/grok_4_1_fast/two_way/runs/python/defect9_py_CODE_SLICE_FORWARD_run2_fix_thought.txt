The bug occurs because after the first replacement, `text_index` and `replace_index` are reset to -1, then the next match search may fail to find any match (leaving both as -1), but the while loop condition checks only `text_index != -1` using the stale value before the failed search update. This causes the loop body to execute with `replace_index = -1`, leading to IndexError. Fix by checking both `text_index != -1 and replace_index != -1` in the while condition.
