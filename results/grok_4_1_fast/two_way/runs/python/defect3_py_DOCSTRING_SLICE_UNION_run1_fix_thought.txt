The bug diagnosis misidentifies the problematic function. `to_utf16_code_units` is correct and doesn't reference undefined `index`. The actual issue is in `CharSequenceTranslator.translate`: after `consumed > 0`, the loop `for pt in range(consumed): pos += char_count(code_point_at(code_units, pos))` uses the wrong `pos` (already advanced) and calls `code_point_at(code_units, pos)` which may be invalid since `pos` has changed. Fix by calculating total code units to skip before advancing `pos`.
