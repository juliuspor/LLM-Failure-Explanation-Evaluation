The test `test_translate_surrogate_pair_with_suffix` fails with an IndexError in `to_utf16_code_units` at line 35-36. When processing input_str = '\U0001F630A' (😰A), the surrogate pair 😰 (2 UTF-16 code units) is handled, but the index advancement logic in CsvEscaper.translate_string incorrectly advances the index by only 1 position after consuming the 2-unit surrogate pair. This causes code_point_at to be called with index=1 (pointing to the high surrogate of 😰 or the 'A'), which is invalid since to_utf16_code_units creates code_units with length 3 ([55357, 56896, 65]). Line 35 checks if index < 0 or index >= len(code_units) (3), so index=1 is valid there, but the real failure occurs because code_point_at receives an index pointing to a lone surrogate (not start of a valid code point), causing internal index miscalculation that leads to the IndexError when accessing code_units[index].
