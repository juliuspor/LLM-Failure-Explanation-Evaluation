The bug diagnosis is incorrect - the IndexError in code_point_at() cannot occur in translate() because code_units is populated by to_utf16_code_units() before the loop starts, so len(code_units) > 0 when pos=0. However, the real bug is in the translate() loop: after translate_codepoint() returns consumed > 0, the for loop incorrectly calls code_point_at(pos) which uses the OLD pos value before incrementing, and also double-advances pos. Fixed by using a while loop with pos += char_count(code_point_at(pos)) and corrected surrogate pair handling.
