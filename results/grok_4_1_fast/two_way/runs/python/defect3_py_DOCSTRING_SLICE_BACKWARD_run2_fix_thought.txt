The diagnosis is incorrect - the IndexError check in code_point_at() is not triggered by to_utf16_code_units() since code_units is populated from input_seq. The real bug is in the translate() method's main loop: after translate_codepoint() returns consumed > 0, the loop incorrectly advances pos by summing char_count() values starting from the already-consumed position, causing double-skipping and missing characters. Fixed by advancing pos by consumed code units directly.
