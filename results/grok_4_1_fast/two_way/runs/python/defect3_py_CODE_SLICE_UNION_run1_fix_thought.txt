The bug is in the post-consumption advancement loop. When `consumed > 0`, the loop calls `code_point_at(code_units, pos)` repeatedly with the same `pos` value since `pos` isn't updated until after the call. For surrogate pairs, this causes incorrect advancement (adds 4 instead of 2). Fix by advancing `pos` by `consumed` code units directly, as `translate_codepoint` already consumed that many code units.
