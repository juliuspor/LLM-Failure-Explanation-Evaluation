The bug occurs because the for loop updates `pos` multiple times using the same initial `pos` value for `code_point_at`, causing out-of-bounds access on subsequent iterations. Fix by advancing `pos` incrementally: after `translate_codepoint` consumes `consumed` codepoints, use a while loop to skip exactly `consumed` codepoints by repeatedly advancing by `char_count` of the codepoint at the current `pos`.
