The bug occurs because when `translate_codepoint` returns a non-zero value (consumed count), the loop incorrectly advances `pos` by iterating over `consumed` codepoints and calling `char_count(code_point_at(code_units, pos))` each time. This re-evaluates `pos` inside the loop (which doesn't change) and calls `code_point_at` on potentially invalid surrogate positions. The fix is to advance `pos` directly by the number of UTF-16 code units consumed, calculated as `consumed * average char size`, but since translators return codepoint count and we don't know sizes in advance, the simplest correct fix is to advance by `consumed` code units initially and let the no-consume case handle surrogates properly. However, the contract expects codepoint count, so fix by properly consuming full codepoints: get codepoint, consume its char_count units.
