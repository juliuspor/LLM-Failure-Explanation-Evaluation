The code fails in the final for loop when `translate_codepoint` consumes a surrogate pair (consumed > 0). After `consumed = self.translate_codepoint(code_units, pos, out)`, the loop `for pt in range(consumed): pos += char_count(code_point_at(code_units, pos))` incorrectly advances `pos` multiple times from the **same position**. For a surrogate pair (consumed=2), it calls `code_point_at(code_units, pos)` twice at the unchanged `pos`, then adds the returned char_count (2) twice, skipping 4 code units instead of the correct 2. This causes `pos` to jump ahead of `length`, triggering an IndexError in `code_point_at` or `to_utf16_code_units` on the next iteration. The bug occurs specifically in `test_translate_surrogate_pair_with_suffix` when processing "\U0001F630A", where the emoji surrogate pair is followed by 'A', and incorrect pos advancement corrupts the position for the trailing character.
