The bug is in the translate method's index advancement logic after translate_codepoint consumes codepoints. When CsvEscaper consumes 2 code units for a surrogate pair, the loop incorrectly uses char_count(code_point_at(pos)) which only advances by 1 initially, causing infinite loop or IndexError. Fixed by advancing pos by the exact number of code units consumed (consumed * code unit size), matching Java's char[] indexing behavior.
