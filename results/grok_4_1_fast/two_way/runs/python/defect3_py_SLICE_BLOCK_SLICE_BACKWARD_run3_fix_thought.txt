The bug diagnosis is incorrect - the index check in `code_point_at` (line 35) occurs during translation after `code_units` is fully populated by `to_utf16_code_units`, not when it's empty. The real bug is in `translate` method: after `translate_codepoint` consumes characters, `pos` is incorrectly advanced using `char_count(code_point_at(code_units, pos))` in a loop, but `pos` is already at the next unconsumed position, causing double-skipping and missing characters. Fix by simply advancing `pos` by the number of code units consumed.
