The bug diagnosis is incorrect - `to_utf16_code_units` correctly populates `code_units` from input string `s` in `translate`. The real bug is in `translate`: after `translate_codepoint` returns `consumed > 0`, the loop `for pt in range(consumed): pos += char_count(code_point_at(code_units, pos))` incorrectly calls `code_point_at(code_units, pos)` where `pos` is still the original index, causing redundant surrogate pair decoding and double advancement. Fix by advancing `pos` directly by the number of code units consumed.
