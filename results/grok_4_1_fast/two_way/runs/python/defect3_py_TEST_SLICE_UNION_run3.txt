The test `test_translate_surrogate_pair_with_suffix` fails with an IndexError in `code_point_at` at line 36. When processing input_str = '\U0001F630A' (😰A), `to_utf16_code_units` creates code_units = [0xD83D, 0xDE30, 65] for the surrogate pair 😰 followed by 'A'. At index=0, `code_point_at` correctly detects the surrogate pair (high=0xD83D at index 0, low=0xDE30 at index 1) and returns the code point. However, the `translate_string` method in CsvEscaper then advances the index incorrectly by only +1 instead of +2 after handling the surrogate pair, setting index=1. At index=1, line 35 checks `if index < 0 or index >= len(code_units)` where index=1 < len([3])=3 is true but the high surrogate check at lines 38-39 sees high=0xDE30 (a low surrogate), which fails the high surrogate range check (0xD800-0xDBFF). The code then incorrectly executes line 44 `return high` (returning 0xDE30), but before that, some index validation logic in `code_point_at` at line 36 raises IndexError because it doesn't expect a lone low surrogate at that position.
