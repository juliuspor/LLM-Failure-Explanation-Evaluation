The bug occurs because `pos` is incorrectly advanced in the loop after `translate_codepoint` returns `consumed > 0`. The loop calls `code_point_at(code_units, pos)` with the already-advanced `pos`, causing IndexError when `pos` becomes invalid. The fix removes this redundant advancement since `translate_codepoint` already handled the consumed codepoints by writing to `out`. Simply set `pos += consumed` after the call.
