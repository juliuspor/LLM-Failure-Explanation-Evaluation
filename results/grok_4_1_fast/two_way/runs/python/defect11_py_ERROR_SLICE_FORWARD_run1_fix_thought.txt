The bug occurs because when `start=0` and `chars=None`, `gap = end - start` becomes `end`, and `_next_int_java(rnd, end)` is called with `bound=end`. In `random_ascii`, `end=127` which is positive so it works, but in `random_default` and similar cases where `end=127` and `start=0`, if the surrogate logic or rejection loops don't fill all positions, it may attempt `idx = 0 + _next_int_java(rnd, 127)`. The diagnosis indicates this fails with 'bound must be positive', suggesting the test expects different behavior or error handling when `start=0`. The fix adjusts the internal logic to handle `start=0` properly by ensuring valid bounds in `_next_int_java` calls.
