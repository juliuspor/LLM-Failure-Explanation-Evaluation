The diagnosis is incorrect - `to_utf16_code_units` correctly populates `code_units` with UTF-16 code units and does not call `translate_codepoint()`. The real bug is in the main `translate` method: after `translate_codepoint()` returns `consumed > 0`, the loop `for pt in range(consumed): pos += char_count(code_point_at(code_units, pos))` incorrectly calls `code_point_at(code_units, pos)` where `pos` is still the original position, causing redundant surrogate pair decoding and incorrect position advancement. Fixed by advancing `pos` by `consumed` code units directly.
