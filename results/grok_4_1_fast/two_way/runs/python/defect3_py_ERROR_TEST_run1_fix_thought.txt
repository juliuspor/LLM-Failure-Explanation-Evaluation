The bug is in the translate method's index advancement logic after translate_codepoint returns a non-zero consumed count. When consumed=2 (surrogate pair), the loop incorrectly calls char_count(code_point_at(pos)) twice, but pos still points to the first code unit of the pair, causing code_point_at(pos) to consume the pair (2 units) again, then advances by 2*2=4 units total instead of the correct 2. Fixed by advancing pos by consumed code units directly, since translate_codepoint returns the number of code units consumed.
