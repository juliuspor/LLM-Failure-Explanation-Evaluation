The bug diagnosis claims IndexError at line 36 in code_point_at, but the diagnosis misidentifies the root cause. The actual issue is in the translate method's main loop: when a translator returns consumed > 0, the loop incorrectly advances pos using char_count(code_point_at(code_units, pos)) in a for loop over range(consumed), but pos is not updated between iterations, causing code_point_at to be called with the same pos repeatedly and eventually exceeding bounds. Fixed by advancing pos by the consumed codepoint count directly using the translator's return value.
