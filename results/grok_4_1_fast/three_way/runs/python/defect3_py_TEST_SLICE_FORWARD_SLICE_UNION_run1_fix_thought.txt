The bug is in the translate method's index advancement logic. When translate_codepoint returns a non-zero consumed count, the loop incorrectly calls code_point_at(pos) repeatedly and advances by char_count for each, but pos hasn't been updated yet and consumed doesn't directly map to code unit count. Instead, simply advance pos by the returned consumed value, which represents the number of codepoints consumed (1 or 2 code units). This fixes the surrogate pair handling and prevents index overshoot.
