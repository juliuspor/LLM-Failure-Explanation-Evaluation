The diagnosis is incorrect - `to_utf16_code_units` correctly populates `code_units` from the input string before `code_point_at` is called, so empty list issue doesn't occur there. The real bug is in `translate` method: after `consumed == 0`, `pos += len(c)` uses code unit count instead of codepoint count, and the final position advancement loop incorrectly calls `char_count(code_point_at(code_units, pos))` where `pos` is already advanced past consumed codepoints. Fixed by using `pos += char_count(codepoint)` consistently.
