The code fails in the while loop's fallback branch when `translate_codepoint` returns 0. After `out.extend(c)` where `c = to_chars(codepoint)`, it does `pos += len(c)`. Here `len(c)` is the number of UTF-16 code units for the codepoint (1 or 2), but `code_units` is already a list of UTF-16 code units. This advances `pos` by 1 or 2 code units, which is correct. However, later when `consumed > 0`, the loop `for pt in range(consumed): pos += char_count(code_point_at(code_units, pos))` calls `code_point_at(code_units, pos)` repeatedly. The first call succeeds and returns the codepoint, but `char_count` returns 1 or 2, so `pos` advances by that amount. The next iteration calls `code_point_at(code_units, new_pos)`, but `new_pos` now points to the next code unit after the current codepoint. If `code_point_at` expects `pos` to be at the start of a codepoint and checks `if index < 0 or index >= len(code_units)` (as shown in SLICE_BACKWARD lines 35-36), it may raise IndexError if the logic inside `code_point_at` improperly handles positions mid-codepoint or has off-by-one errors. This happens because after the first `pos` update, subsequent calls to `code_point_at(code_units, pos)` use a `pos` that no longer aligns with codepoint boundaries expected by the function.
