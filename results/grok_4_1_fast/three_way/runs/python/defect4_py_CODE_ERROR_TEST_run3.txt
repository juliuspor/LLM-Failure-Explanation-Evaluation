The code fails because of a bug in the `if self._max_middle_index >= 0:` block. It incorrectly uses `self.get_data_item(self._min_middle_index)` instead of `self.get_data_item(self._max_middle_index)` to fetch the period for comparison. This means it always compares the new `middle` value against the middle time of the current minimum middle index item, not the maximum middle index item.

In the test case:
- Index 0: SimpleTimePeriod(0,100), middle=50 → sets _max_middle_index=0
- Index 1: SimpleTimePeriod(100,200), middle=150 → compares against item 0 (middle=50), 150>50 so correctly sets _max_middle_index=1
- Index 2: SimpleTimePeriod(0,50), middle=25 → sets _min_middle_index=2 (not relevant here)
- Index 3: SimpleTimePeriod(50,100), middle=75 → fetches item at _min_middle_index=2 (middle=25), compares 75>25, so incorrectly sets _max_middle_index=3

Expected max middle=150 at index 1, but got index 3 because it compared against the wrong period.
