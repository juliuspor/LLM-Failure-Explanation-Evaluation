The bug occurs because the parser doesn't handle the language__variant format (empty country) like 'fr__POSIX'. After validating the first '_' at position 2, it immediately checks characters at positions 3-4 as uppercase country codes, but in this case position 3 is another '_'. Fix by checking for a second '_' at position 5 after the first '_', and if found, treat positions 3-4 as empty country and parse variant from position 6. Also update length check to allow length >=7 for this case.
