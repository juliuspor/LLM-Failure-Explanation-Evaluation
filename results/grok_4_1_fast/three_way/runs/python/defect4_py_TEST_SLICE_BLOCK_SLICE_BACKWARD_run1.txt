The code fails in the `add` method of `TimePeriodValues` when updating `_min_start_index`. In the `_update_bounds` logic (around lines 343-352), after comparing `start < min_start` at line 349, the code **unconditionally** sets `self._min_start_index = index` at line 352 in the `if self._min_start_index >= 0:` block. This overwrites the index even when `start >= min_start`, causing the minimum start index to always be the **last added** period instead of the true minimum. This breaks `test_min_start_index` where it expects index 1 (start=50) but gets index 2 (start=200). The same bug affects min_end_index and max_middle_index calculations.
