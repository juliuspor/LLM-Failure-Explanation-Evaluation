The bug is in the always-set logic for all min/max indices. After each comparison block (if self._xxx_index >= 0), there's an unconditional 'else' clause that sets the index to the current index even when the new period doesn't actually update the min/max. This causes the indices to always point to the last added item instead of the true min/max. Fixed by removing all the unconditional 'else' clauses - indices are only set when actually updating the min/max or on first item (index >=0 check handles initialization).
