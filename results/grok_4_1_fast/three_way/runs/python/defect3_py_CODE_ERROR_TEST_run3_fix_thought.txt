The bug occurs because after translate_codepoint returns consumed=2 (handling a surrogate pair), the for loop incorrectly iterates range(2) and calls code_point_at(code_units, pos) multiple times from the same starting position, causing out-of-bounds access. The fix removes the unnecessary for loop since translate_codepoint already handled the advancement. Simply set pos += consumed, treating consumed as the number of UTF-16 code units advanced.
