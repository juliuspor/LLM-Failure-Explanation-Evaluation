The code fails in the _update_bounds method when updating _min_start_index. After lines 343-348 compare the new period's start time with the current minimum start time from self._min_start_index, line 352 unconditionally sets self._min_start_index = index regardless of whether the new start time is actually smaller. This overwrites the correct minimum index every time a new period is added, so get_min_start_index() always returns the index of the last added period instead of the true minimum start index.

This breaks test_min_start_index (expects index 1 for start=50, gets last index), test_generic_time_period_support (expects min_start_index=2), and similar min-index tests. The max-index updates likely have the same bug with unconditional assignment after comparison.
