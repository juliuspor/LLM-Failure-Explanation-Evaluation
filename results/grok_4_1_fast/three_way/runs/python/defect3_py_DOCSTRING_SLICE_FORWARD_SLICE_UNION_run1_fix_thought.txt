The bug diagnosis is incorrect - `to_utf16_code_units` correctly populates `code_units` with UTF-16 code units from the input string `s`. The real bug is in the `translate` method's main loop: after `translate_codepoint` returns `consumed > 0`, the loop incorrectly advances `pos` by calling `char_count(code_point_at(code_units, pos))` *consumed* times starting from the already-consumed position, causing double-skipping and missing characters. Fix by advancing `pos` directly by `consumed` code units.
