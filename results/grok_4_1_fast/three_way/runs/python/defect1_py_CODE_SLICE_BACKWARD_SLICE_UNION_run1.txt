The code fails due to incorrect logic when computing the total offset minutes on lines 383 and 385. When hours_offset is negative (hours_in_minutes < 0), line 383 sets minutes_offset = hours_in_minutes - minutes_offset. Since hours_in_minutes is negative and minutes_offset is positive (0-59, validated on line 377), this subtracts a positive value from a negative number, making minutes_offset more negative than hours_in_minutes alone. For example, hours_offset=-1 (hours_in_minutes=-60) and minutes_offset=30 results in minutes_offset = -60 - 30 = -90. Later on line 386, FieldUtils.safe_multiply(-90, DateTimeConstants.MILLIS_PER_MINUTE) triggers ArithmeticError due to the large negative magnitude, which is caught and raises ValueError('Offset is too large') on line 388. This happens for valid timezone offsets like -01:30, which should succeed but incorrectly computes -90 minutes total instead of the correct -90 minutes (-60 -30). The bug is in the sign handling: it should add the absolute contributions (hours_in_minutes + minutes_offset) regardless of sign, as both represent offset direction.
