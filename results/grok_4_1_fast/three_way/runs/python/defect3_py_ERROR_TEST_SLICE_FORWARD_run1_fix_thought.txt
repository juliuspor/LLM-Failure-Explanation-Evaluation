The bug is in the `translate` method's advancement logic after `translate_codepoint` consumes characters. When it consumes codepoints (including surrogate pairs), the loop incorrectly uses `char_count(code_point_at(code_units, pos))` where `pos` is already advanced past the consumed surrogate pair's starting position, causing `code_point_at` to be called on invalid low surrogate indices. This leads to incorrect position advancement and eventual IndexError. Fix by tracking the exact code unit count consumed instead of recalculating via codepoints.
