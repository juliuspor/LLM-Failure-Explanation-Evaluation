The bug is in the `translate` method, not `to_utf16_code_units`. When `translate_codepoint` returns 0, the code incorrectly uses `len(c)` (always 1 or 2) instead of `char_count(codepoint)` to advance `pos`. This causes `pos` to skip too few code units for surrogate pairs, leading to `code_point_at` being called with an invalid index (like 2 on an empty list in some recursive scenarios). Fixed by using `char_count(codepoint)` consistently for position advancement.
