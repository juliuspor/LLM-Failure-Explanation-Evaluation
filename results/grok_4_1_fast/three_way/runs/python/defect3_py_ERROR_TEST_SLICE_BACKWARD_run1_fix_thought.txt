The bug is in the `translate` method's index advancement logic. After `translate_codepoint` consumes characters (including surrogate pairs), the loop incorrectly uses `pos` in `char_count(code_point_at(code_units, pos))` before advancing `pos`, causing repeated lookups at the same position and eventual index overflow. Fixed by advancing `pos` by `consumed` code units directly, matching Java's char[] indexing semantics.
