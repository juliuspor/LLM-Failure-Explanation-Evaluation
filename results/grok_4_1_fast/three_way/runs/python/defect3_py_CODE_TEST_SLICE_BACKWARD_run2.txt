The code fails with an IndexError in the final for loop of the translate method when processing a surrogate pair followed by another character (like "\U0001F630A"). Here's what happens:

1. translate_codepoint() processes the surrogate pair at position pos=0 and returns consumed=2 (two UTF-16 code units).
2. The loop "for pt in range(consumed):" runs twice (pt=0, pt=1).
3. Inside the loop, code_point_at(code_units, pos) is called with the ORIGINAL pos=0 both times.
4. On the first iteration (pt=0), code_point_at succeeds and char_count returns 2, so pos becomes pos=2.
5. On the second iteration (pt=1), code_point_at(code_units, 0) is called AGAIN with pos=0 (the original value, not the updated pos=2).
6. But now code_point_at sees index=0 refers to the high surrogate of a surrogate pair, which requires looking ahead to index=1. Since the contract expects full codepoints, it likely fails validation and raises IndexError (as shown in SLICE_BACKWARD at line 36).

Root cause: The for loop uses range(consumed) with the original pos value instead of advancing pos incrementally by 1 code unit each iteration, causing code_point_at to be repeatedly called on the same invalid surrogate position.
