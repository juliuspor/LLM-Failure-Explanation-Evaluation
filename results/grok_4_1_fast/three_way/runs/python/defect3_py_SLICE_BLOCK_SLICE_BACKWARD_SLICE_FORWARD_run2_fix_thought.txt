The bug diagnosis is incorrect - `to_utf16_code_units` correctly populates the code units list from the input string, so `code_units` is not empty in `translate`. The real bug is in the `translate` method's main loop: after `translate_codepoint` consumes characters, the position advancement logic is incorrect. It calls `code_point_at(code_units, pos)` (which uses the current `pos`) inside the loop `for pt in range(consumed)`, causing it to repeatedly consume the same surrogate pair or codepoint. Fixed by first computing the codepoints to skip, then advancing `pos` by their total UTF-16 code unit lengths.
