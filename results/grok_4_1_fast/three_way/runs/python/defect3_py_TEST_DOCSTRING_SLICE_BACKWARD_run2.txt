The test `test_translate_surrogate_pair_with_suffix` fails with an IndexError from `code_point_at` at line 35-36. When processing input_str = "\U0001F630A" (😰A), the surrogate pair 😰 (U+D83D U+DE30) is correctly detected. However, after consuming the 2 UTF-16 code units for the surrogate pair, the index advances to position 2 (pointing to 'A'). The `code_point_at(index=2)` call then checks `if index < 0 or index >= len(code_units)` where code_units has length 2 (only the surrogate pair units so far), causing `2 >= 2` to be True and raising IndexError("String index out of range: 2"). This happens because `code_point_at` incorrectly uses the partially-built `code_units` list length instead of the input string length to validate the index, leading to premature out-of-bounds detection when processing characters after surrogate pairs.
