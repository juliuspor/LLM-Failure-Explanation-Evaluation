The test `test_translate_surrogate_pair_with_suffix` fails with an IndexError in the `translate` method of `CharSequenceTranslator`. The bug occurs in the surrogate pair handling logic around lines 38-44 in `code_point_at`. When processing the input "\U0001F630A" (emoji 😰 followed by 'A'), the code correctly identifies the surrogate pair at `index` (high surrogate at `code_units[index]`, low at `code_units[index+1]`). After consuming the pair in `translate_codepoint()`, it advances the index by 2 positions. However, the next iteration starts checking for another surrogate pair from this new position, where `code_units[index]` is now the ASCII 'A' (value 65). The condition `0xD800 <= high <= 0xDBFF` falsely passes because 65 falls within the unsigned 16-bit range interpretation, then `index + 1 < len(code_units)` is true (pointing to the end), but `code_units[index + 1]` raises IndexError since it accesses beyond the string length. This happens because the surrogate check doesn't properly filter for high surrogates only and lacks bounds checking before accessing `low`.
