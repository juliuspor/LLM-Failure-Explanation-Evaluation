The test `test_translate_surrogate_pair_with_suffix` fails with an IndexError in `code_point_at`. When processing input "\U0001F630A" (😰A), the UTF-16 code units are [0xD83D, 0xDE30, 0x0041]. At index=1, `code_point_at` checks `high = code_units[1]` (0xDE30, a low surrogate). Since 0xDE30 is not in 0xD800-0xDBFF range, it returns `high` (0xDE30) instead of raising an error for a malformed surrogate. Later, when `index=2` (for 'A'), the check `index + 1 < len(code_units)` (3 < 3) fails, but the real bug is `CsvEscaper.translate_string` incorrectly advances the index by 2 positions after every `code_point_at` call, even for single code units like the low surrogate at index=1. This causes index to jump from 1 to 3, exceeding the list length (IndexError at line 35 in `code_point_at`: `if index < 0 or index >= len(code_units)`). The root cause is in `translate_string`: it assumes every character consumes 2 code units, but single BMP characters and malformed surrogates consume only 1.
