The bug occurs because null replacements are skipped during search but not during append. When a null replacement is selected (replace_index points to it), we must skip appending and advance the start position past the matched search string to avoid incorrect replacements later. Add a check after finding the earliest match: if replacement_list[replace_index] is None, skip appending, update start = text_index + len(search_list[replace_index]), and continue searching from the new start.
