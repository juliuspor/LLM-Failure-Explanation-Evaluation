The bug is in the `translate` method's main loop. When `translate_codepoint` returns a non-zero `consumed` value (e.g., 2 for a surrogate pair), the code incorrectly enters a for loop that calls `code_point_at(code_units, pos)` with the unchanged `pos`, then advances `pos` using `char_count` in a way that doesn't match the contract. Per the contract comment, translators handle surrogate pairs themselves, so `pos` should simply advance by `consumed` code units directly after a successful translation.
