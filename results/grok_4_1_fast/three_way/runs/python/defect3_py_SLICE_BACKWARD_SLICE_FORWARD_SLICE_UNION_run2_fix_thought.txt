The bug diagnosis is incorrect - `code_units` in `code_point_at` is populated by `to_utf16_code_units` in `translate`, not empty. The real bug is in `translate`: after `translate_codepoint` consumes codepoints, the loop incorrectly uses `char_count(code_point_at(code_units, pos))` where `pos` is already advanced past consumed codepoints, causing double-skipping or incorrect advancement. Fix by advancing `pos` directly by `consumed` code units, as `translate_codepoint` returns code units consumed.
