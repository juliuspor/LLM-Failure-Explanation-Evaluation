The code in TimePeriodValues._update_bounds fails because it uses the wrong variable for comparison when updating max_middle_index. In the test case, periods are added with middle points: index 0 (mid=50), index 1 (mid=150), index 2 (mid=25), index 3 (mid=75). The maximum middle is 150 at index 1.

When adding the period at index 3 (mid=75), the code incorrectly compares the new middle timestamp against max_end_time (which is larger due to previous end times) instead of max_middle_time. This causes the condition `if new_middle > max_end_time:` (or similar wrong variable) to incorrectly evaluate to True, setting max_middle_index = 3 instead of preserving the correct value of 1.

The bug occurs in the max middle update logic within _update_bounds, where max_end_time is used instead of max_middle_time for the comparison.
