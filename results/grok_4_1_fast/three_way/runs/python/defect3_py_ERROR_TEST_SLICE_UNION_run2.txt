The code fails in the `to_utf16_code_units` function during the `test_translate_surrogate_pair_with_suffix` test with input "\U0001F630A" (😰A). The `code_point_at` function calls `to_utf16_code_units(s)` which converts the string to UTF-16 code units: [55357, 56896, 65] (high surrogate, low surrogate, 'A'). When `index=2` is passed to access the 'A' code unit at line 35 (`if index < 0 or index >= len(code_units):`), it succeeds. However, lines 38-39 then attempt `high = code_units[2]` and check `index + 1 < len(code_units)` (3 < 3, false). The bug occurs because the code incorrectly assumes all code units could be surrogate highs and tries to access `code_units[index + 1]` without bounds checking before the condition, but the actual IndexError at index 2 indicates the check at line 35 is the failure point. The root cause is mishandling the transition from a surrogate pair (indices 0-1) to a single code unit (index 2), causing the index validation to reject a valid position.
