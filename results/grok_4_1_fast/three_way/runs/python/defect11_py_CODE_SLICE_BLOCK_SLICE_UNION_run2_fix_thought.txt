The bug occurs because when chars=None, letters=False, numbers=False, start/end=0/0, the code sets start=0, end=127 creating a valid range (gap=127), but the diagnosis misidentifies the issue. The real problem is in surrogate handling: when generating a high surrogate pair, it uses _next_int_java(rnd, 128) where bound=128 is valid, but for low surrogate it uses the same which should also be valid. However, the infinite loop occurs when all generated characters are rejected by the letters/numbers filter or skipped due to surrogate/private surrogate logic without consuming positions. The fix ensures we always generate printable ASCII when no specific constraints are given by setting start=32, end=127 explicitly in that case, matching random_ascii behavior and Java default.
