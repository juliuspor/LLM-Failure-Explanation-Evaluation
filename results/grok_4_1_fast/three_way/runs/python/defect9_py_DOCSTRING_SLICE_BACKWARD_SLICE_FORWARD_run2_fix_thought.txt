The bug occurs because the code checks `replacement_list[i] is None` before finding matches but doesn't re-check after finding a match. When `text.find()` succeeds for a search string whose replacement is None, `replace_index` gets set to that index, then `parts.append(replacement_list[replace_index])` tries to access None, and later `len(search_list[replace_index])` would fail. Fix by using `StringUtils.default_string(replacement_list[replace_index])` to convert None replacements to empty strings, matching the None-safe behavior of other methods.
