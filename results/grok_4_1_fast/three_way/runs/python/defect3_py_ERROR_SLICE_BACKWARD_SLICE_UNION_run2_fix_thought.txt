The bug diagnosis incorrectly identifies the issue. `to_utf16_code_units` correctly populates code_units from the input string. The real bug is in the main `translate` method's loop: after `consumed > 0`, it increments `pos` using `char_count(code_point_at(code_units, pos))` where `pos` is already advanced by the translator, causing `code_point_at` to receive an out-of-bounds index. Fixed by using the `consumed` count directly to advance `pos` by the correct number of code units.
