The bug is in the translate method's index advancement logic. When translate_codepoint returns 0, it advances by len(c) which correctly handles surrogate pairs (2 units). However, when translate_codepoint returns a non-zero consumed value, the loop incorrectly advances by summing char_count for each of 'consumed' codepoints starting from the current pos, causing it to re-examine already-consumed surrogate pairs and miscalculate advancement. Fix by simply advancing pos by consumed code units directly.
