The test `test_translate_surrogate_pair_with_suffix` fails with an IndexError on line 36 in `code_point_at`: `raise IndexError(f"String index out of range: {index}")`. This happens because `CsvEscaper.translate_string` calls `code_point_at(index, code_units)` where `code_units` comes from `to_utf16_code_units(input_str)`. For input_str = "\U0001F630A" (😰A), `to_utf16_code_units` creates a list of 3 UTF-16 code units: [55357, 56360, 65] (high surrogate, low surrogate, 'A'). When processing the surrogate pair at index 0-1, the translator advances `index` to 2 to process the 'A'. But `code_point_at(2, code_units)` checks `if 2 >= len(code_units)` (3 is false), yet fails because the surrogate pair logic in `translate` incorrectly calculates the next index, passing an out-of-bounds value (>=3) to `code_point_at`. The root cause is in `CharSequenceTranslator.translate`: after handling a surrogate pair (2 code units), `index += 2` skips to index=2 correctly, but a subsequent bounds check or loop condition passes index=3 (or higher) to `code_point_at(3, code_units)`, triggering `index >= len(code_units)` (3>=3 is true). This bug only manifests with surrogate pairs *followed by another character*.
