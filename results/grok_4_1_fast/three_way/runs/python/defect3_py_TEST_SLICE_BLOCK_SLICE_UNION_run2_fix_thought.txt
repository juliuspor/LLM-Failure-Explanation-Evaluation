The bug is in the translate method's index advancement logic. When translate_codepoint returns 2 (consuming a surrogate pair), the loop incorrectly uses `for pt in range(consumed)` and calls `code_point_at(code_units, pos)` with the unchanged pos, then advances by char_count of that position multiple times. This causes incorrect advancement. The fix is to calculate the total code units to skip once using code_point_at(pos), then advance pos by that amount, regardless of consumed count.
