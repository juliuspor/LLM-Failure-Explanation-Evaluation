The bug diagnosis is incorrect - `to_utf16_code_units()` correctly populates the `code_units` list from the input string, so `code_point_at()` receives a non-empty list and the IndexError doesn't occur there. The real bug is in `translate()`: after `translate_codepoint()` returns `consumed > 0`, the loop `for pt in range(consumed): pos += char_count(code_point_at(code_units, pos))` calls `code_point_at(code_units, pos)` with the SAME `pos` each iteration (since `pos` isn't updated until after `char_count()`), causing it to repeatedly consume the same codepoint(s). Fixed by calculating the correct advancement once using `char_count(code_point_at(code_units, pos)) * consumed` or properly advancing `pos` in the loop.
