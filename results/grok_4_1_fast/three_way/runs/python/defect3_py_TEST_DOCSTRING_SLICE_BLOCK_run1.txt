The input string '\U0001F630A' contains a surrogate pair 😰 (UTF-16 code units at indices 0-1) followed by 'A' at index 2. When `CharSequenceTranslator.translate` processes this string, it calls `code_point_at(code_units, index)` to get the next code point. After consuming the surrogate pair at index 0, the index advances to 2. The `CsvEscaper.translate_codepoint()` then returns without consuming further characters, so the next `code_point_at(code_units, 2)` is called. However, the surrogate pair detection logic in `code_point_at` at lines 35-36 incorrectly treats the high surrogate from the previous pair as a standalone invalid surrogate when index=2 is reached, or the index advancement in `translate` skips past the string length, causing `index >= len(code_units)` to trigger the IndexError. This happens because the translator's index handling doesn't properly account for the 2-unit width of surrogate pairs when followed by single-unit characters like 'A'.
