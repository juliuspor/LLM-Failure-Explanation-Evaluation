The bug occurs because when translate_codepoint() returns a non-zero consumed count for a single codepoint (like 'A'), the loop incorrectly calls code_point_at(pos) which returns the codepoint, then uses char_count(codepoint) which always returns 1 or 2 based on codepoint value. For single code units at the end of the array, this causes incorrect advancement. The fix is to advance by the exact number of code units consumed (as returned by translate_codepoint), since translators must understand UTF-16 boundaries and return the correct code unit count.
