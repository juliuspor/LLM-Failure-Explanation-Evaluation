The bug diagnosis incorrectly identifies the issue in `to_utf16_code_units` - that function correctly populates `code_units` by iterating over each character in `s` and appending UTF-16 code units. The real bug is in `CharSequenceTranslator.translate`: after `translate_codepoint` returns `consumed > 0`, the loop `for pt in range(consumed): pos += char_count(code_point_at(code_units, pos))` incorrectly calls `code_point_at(code_units, pos)` where `pos` is still the original index, causing re-consumption of the same codepoint(s). Fix by advancing `pos` by the correct number of code units consumed: `pos += consumed`.
