The code fails in the `_update_bounds` method of `TimePeriodValues` when updating `self._min_start_index`. At lines 349-352, after comparing `if start < min_start:` and setting `self._min_start_index = index` on line 350 (correctly), there's an unconditional `self._min_start_index = index` on line 352. This always overwrites the index with the current `index` regardless of whether the new period actually has a smaller start time. 

In `test_min_start_index`, after adding periods with starts 100 (idx0), 50 (idx1), 200 (idx2):
- First period sets `_min_start_index=0`
- Second period (start=50 < 100) correctly sets it to 1 on line 350
- Third period (start=200 > 50) skips line 350 but line 352 sets it to 2

Thus `get_min_start_index()` returns 2 instead of the expected 1. The same bug affects min_end_index, max_start_index, max_end_index, and max_middle_index updates.
