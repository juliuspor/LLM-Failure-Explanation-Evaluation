The bug occurs because `_next_int_java` is called with `gap = 0` (when `start == end == 0` and no `chars` provided), causing `bound <= 0` and raising ValueError('bound must be positive'). The fix updates the default character range logic in `random` to set `start=32, end=127` (matching `random_ascii`) when `start==end==0` and no `chars`, preventing zero gap.
