The test `test_translate_surrogate_pair_with_suffix` fails with an IndexError on line 36 in `code_point_at`. The input "\U0001F630A" (😰A) converts to UTF-16 code units [55357, 56360, 65] via `to_utf16_code_units`. When `code_point_at` processes index 0, it detects the surrogate pair (high=55357 at index 0, low=56360 at index 1) and correctly returns the code point. However, `CsvEscaper.translate_string` (or its underlying `CharSequenceTranslator.translate`) then advances the index by only 1 position instead of 2, setting next index=1. At index=1, `code_point_at(1)` checks `high = code_units[1] = 56360` (0xDC40), which is a low surrogate (0xDC00-0xDFFF). Since 56360 is not a high surrogate (0xD800-0xDBFF), the trailing surrogate check fails, but index+1=2 equals len(code_units)=3, so no low surrogate is consumed. The code then returns high=56360, but before that, line 35 checks `if index < 0 or index >= len(code_units)` which passes. The real failure occurs because processing a lone low surrogate (56360) at index 1 triggers an invalid UTF-16 state in `translate`, causing the IndexError during subsequent index advancement past the string end.
