The code fails in TimePeriodValues._update_bounds because it incorrectly updates max_middle_index when adding a new period. In test_max_middle_index_simple, after adding periods with middle points 50 (index 0), 150 (index 1), 25 (index 2), and 75 (index 3), it returns 3 instead of 1.

The bug occurs around lines similar to 352 in the slice: when updating _max_middle_index, the code has logic like:

if self._max_middle_index >= 0:
    # get current max middle value
    if middle > current_max_middle:
        self._max_middle_index = index
# BUG: This unconditional line always executes
self._max_middle_index = index

This mirrors the _min_start_index pattern at line 352. The final unconditional assignment sets _max_middle_index to the latest index (3) regardless of middle point values, overwriting the correct index (1 for middle=150). The same bug likely affects _max_end_index and _max_middle_index updates.
