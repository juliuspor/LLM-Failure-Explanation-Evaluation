The code fails in the while loop's final pos update due to an infinite loop or incorrect position advancement. After `self.translate_codepoint(code_units, pos, out)` returns `consumed > 0`, the code enters this block:

```python
for pt in range(consumed):
    pos += char_count(code_point_at(code_units, pos))
```

The issue is that `code_point_at(code_units, pos)` is called repeatedly with the *same* `pos` value each iteration because `pos` hasn't been updated yet inside the loop body when the function is called. From the slice union, `code_point_at` reads `code_units[index]` (where `index=pos`) and checks if it's a high surrogate (0xD800-0xDBFF). If the codepoint at `pos` is a high surrogate, `char_count` returns 2 (since surrogates form pairs). Thus, each iteration computes `char_count=2` for the same position and adds 2 to `pos` *after* the call.

However, `translate_codepoint` already consumed `consumed` code units (likely surrogate pairs or multi-unit sequences) and advanced `out` accordingly. Recomputing `char_count` from the *original* `pos` double-counts the advancement, causing `pos` to overshoot. Worse, if `consumed > 1` (e.g., multiple surrogate pairs), the loop calls `code_point_at(code_units, pos)` with an unchanged `pos`, always returning the same high surrogate codepoint, so it adds `char_count=2` exactly `consumed` times.

This mismatches the contract: `translate_codepoint` consumed `consumed` *code units* (UTF-16 chars), but the loop advances by `consumed * char_count(codepoint at original pos)` code units, leading to incorrect skipping and potential infinite loops if advancement undershoots.
