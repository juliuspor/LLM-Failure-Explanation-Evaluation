The code fails in TimePeriodValues._update_bounds method when updating _min_start_index. At lines 343-350, after calculating 'start' for the new period, the code checks if self._min_start_index >= 0. If true, it gets the current min_start_period = self.get_data_item(self._min_start_index).get_period() and calculates min_start from it. Then at line 349, it compares 'if start < min_start:' to update self._min_start_index = index only if smaller. However, at line 352, it ALWAYS sets self._min_start_index = index regardless of the comparison result. This means the _min_start_index is incorrectly overwritten with the current index even when the new period's start time is NOT the minimum, causing tests like test_min_start_index to fail because it picks the wrong index (last added instead of actual minimum start). The same bug likely affects other index calculations (_max_start_index, _min_end_index, etc.) with identical always-overwrite logic.
