The bug is in the translate method's index advancement logic after translate_codepoint returns a non-zero consumed count. The current code incorrectly advances pos by summing char_count for each of the consumed codepoints, which fails for surrogate pairs. When CsvEscaper returns 2 for a surrogate pair (consuming 2 code units), it then calls char_count(code_point_at(pos)) twice, but pos still points to the first code unit of the pair, causing incorrect advancement. The fix is to simply advance by the number of code units consumed: pos += consumed.
