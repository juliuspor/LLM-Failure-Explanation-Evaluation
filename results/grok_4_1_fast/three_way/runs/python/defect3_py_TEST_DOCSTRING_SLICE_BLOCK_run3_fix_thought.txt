The bug occurs because after `translate_codepoint()` returns a non-zero `consumed` value, the loop incorrectly advances `pos` by calling `char_count(code_point_at(code_units, pos))` *consumed* times, causing double advancement for surrogate pairs. The fix is to advance `pos` by `consumed` code units directly, since `translate_codepoint()` already specifies exactly how many UTF-16 code units were consumed.
