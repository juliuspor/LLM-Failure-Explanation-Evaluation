The test `test_translate_surrogate_pair_with_suffix` fails with an IndexError in `code_point_at`. When processing input_str = '\U0001F630A' (😰A), `to_utf16_code_units` correctly produces [55357, 56384, 65] where indices 0-1 are the surrogate pair for 😰 and index 2 is 'A'. In `code_point_at`, at index=0, line 38 gets high=55357 (0xD8D8, a high surrogate). Lines 39-41 detect the surrogate pair with low=56384 (0xDC00) at index+1. Line 43 returns the code point 128560. However, the translator's index advancement logic (not shown but implied by the bug description) only advances by 1 position instead of 2 after consuming the surrogate pair, setting next index=1. The next `code_point_at(index=1)` then hits line 35: index=1 is valid (1 < 3), but line 38 tries `high = code_units[1]` where code_units[1]=56384 (0xDC00, a low surrogate). Line 39's check `0xD800 <= 56384 <= 0xDBFF` fails because 0xDC00 > 0xDBFF, so it falls through. But the real failure occurs because the translator later calls `code_point_at` with an out-of-bounds index (likely index=2 when len(code_units)=3, or mishandles the end), triggering line 36's IndexError: 'String index out of range: 2'.
