The bug occurs because `pos` is advanced inside the `for pt in range(consumed)` loop using `code_point_at(code_units, pos)`, but `pos` is not updated within the loop body, causing repeated calls with the same `pos` value. For `consumed=2` and a surrogate pair (2 code units), it calls `code_point_at` twice at the same position without advancing `pos`, then incorrectly advances `pos` by 1+1=2 units from the original position. When `pos` reaches or exceeds `length`, the next iteration's `code_point_at(code_units, pos)` in the continue block raises IndexError. Fix by pre-computing codepoints and their char counts before advancing `pos`.
