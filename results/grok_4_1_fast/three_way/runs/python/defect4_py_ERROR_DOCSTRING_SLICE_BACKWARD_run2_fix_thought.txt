The bug is in the max_middle_index block where `self._min_middle_index` is incorrectly used instead of `self._max_middle_index` to fetch the comparison period, causing wrong max middle tracking. Additionally, all blocks unconditionally set the index to the new `index` when `self.*_index < 0` (first item), but fail to update when `>= 0` because the comparison values are fetched before the new period's values are fully computed. Fixed by computing all current min/max values first, then comparing the new period's values against them to update indices correctly.
