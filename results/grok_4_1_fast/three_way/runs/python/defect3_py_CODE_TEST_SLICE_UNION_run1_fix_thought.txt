The bug is in the translate method's pos advancement logic after translate_codepoint consumes code units. When consumed > 0, the loop incorrectly calls code_point_at(pos) repeatedly while pos is still the start position, then advances pos by char_count values that don't account for the fact that translate_codepoint already processed those code units. This causes incorrect pos advancement (double-skipping surrogate pairs). Fix: advance pos by the exact number of code units consumed (consumed value directly represents UTF-16 units consumed).
