The bug diagnosis incorrectly identifies the issue in to_utf16_code_units() - that function works correctly. The real problem is in the translate() method's loop: after translate_codepoint() returns consumed > 0, the loop incorrectly advances pos by summing char_count() for each of 'consumed' codepoints starting from the already-advanced position, causing pos to overshoot and trigger IndexError in code_point_at(). Fixed by simply advancing pos by consumed code units.
