The code fails in the _update_bounds method when updating self._min_start_index. After line 349's 'if start < min_start:' check, line 352 unconditionally executes 'self._min_start_index = index' regardless of whether the new period actually has a smaller start time. This overwrites the correct minimum index with the current index every time a new period is added, causing get_min_start_index() to always return the index of the last added period instead of the true minimum start time index.

For example, in test_min_start_index:
- Add SimpleTimePeriod(100,200) -> _min_start_index = 0 (correct)
- Add SimpleTimePeriod(50,150) -> start=50 < min_start=100, so line 350 sets index=1, then line 352 also sets index=1 (still correct)
- Add SimpleTimePeriod(200,300) -> start=200 > min_start=50, skips line 350, but line 352 sets _min_start_index=2 (incorrect)

The test expects index 1 but gets 2. The same bug affects _max_start_index, _min_end_index, and _max_end_index updates.
