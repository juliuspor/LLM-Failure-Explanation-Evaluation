The bug occurs because when `consumed > 0`, the loop advances `pos` by summing `char_count` for each of the `consumed` codepoints starting at the current `pos`, but `translate_codepoint` may have already consumed surrogate pairs or multi-codepoint sequences. This double-advances `pos`, causing overshoot. Fix by advancing `pos` directly by the total UTF-16 code units consumed (matching Java's char consumption semantics).
