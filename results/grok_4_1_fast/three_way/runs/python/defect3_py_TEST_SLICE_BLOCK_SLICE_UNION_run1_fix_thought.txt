The bug occurs because when translate_codepoint returns 0 for a surrogate pair, the main loop advances pos by len(c) which is 2, but when it does return a non-zero value (like 2 for surrogate pair in CsvEscaper), the subsequent loop incorrectly advances pos by summing char_count(code_point_at(pos)) for each 'consumed' codepoint, where code_point_at(pos) on a low surrogate returns the invalid low surrogate value and char_count(56360) returns 2, over-advancing past the string end. Fix by advancing pos by 'consumed' code units directly after translate_codepoint succeeds.
