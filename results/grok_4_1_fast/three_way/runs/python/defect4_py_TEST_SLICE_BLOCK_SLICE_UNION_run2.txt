The code fails in `TimePeriodValues._update_bounds` (lines 343-352) due to a logic error in updating `_min_start_index`. When a new period is added:

1. If `_min_start_index >= 0` (line 343), it fetches the current min start value (lines 344-348).
2. It compares the new `start` with `min_start` (line 349).
3. **If `start < min_start` (line 349), it correctly sets `self._min_start_index = index` (line 350).**
4. **But then, regardless of the comparison, line 352 always executes `self._min_start_index = index`, overwriting the correct index with the new one.**

This causes `get_min_start_index()` (line 440) to always return the index of the *last* added period, not the one with the actual minimum start time.

**Test impact:** `test_min_start_index` fails because it expects index 1 (start=50) but gets index 2 (start=200). All other min/max index tests fail similarly.
