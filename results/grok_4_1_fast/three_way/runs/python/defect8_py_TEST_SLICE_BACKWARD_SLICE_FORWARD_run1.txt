The test `test_locale_with_empty_country_variant` fails because `LocaleUtils.to_locale("fr__POSIX")` raises a `ValueError` instead of returning `Locale("fr", "", "POSIX")`. The bug occurs at lines 227-230 where the code checks `ch3 = locale_str[3]` ('_') and `ch4 = locale_str[4]` ('P'). Line 229 validates that both must be uppercase letters A-Z, but `ch3` is '_', which fails the condition `ch3 < 'A' or ch3 > 'Z'`, raising `ValueError(f"Invalid locale format: {locale_str}")`. This check assumes position 3 is always the start of a 2-letter country code, but in the `language__variant` format, position 3 is an empty country delimiter '_', causing the invalid locale format error just like the original Java bug.
