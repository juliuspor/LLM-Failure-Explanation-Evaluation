The bug is in the translate method's advancement logic after translate_codepoint returns 0. When no translation occurs, it calls code_point_at(pos) which correctly handles surrogate pairs by looking ahead, but then advances pos by len(c) where c=to_chars(codepoint) which is correct (1 or 2). However, the diagnosis reveals the real issue is in CsvEscaper: when it encounters a high surrogate, it returns 2 (consuming the pair), but the main translate loop then incorrectly advances by calling char_count(code_point_at(pos)) *twice* for consumed=2, causing pos to overshoot by 1 extra unit, landing on the low surrogate and corrupting state. Fix by changing the advancement to simply pos += consumed, since translators already return the correct UTF-16 code unit count consumed.
