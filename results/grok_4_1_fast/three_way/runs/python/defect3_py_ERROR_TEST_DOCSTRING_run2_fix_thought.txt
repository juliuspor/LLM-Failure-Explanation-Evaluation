The bug is in the `translate` method's index advancement logic after `translate_codepoint` consumes codepoints. When it returns `consumed > 0`, the loop incorrectly advances `pos` by calling `char_count(code_point_at(code_units, pos))` *consumed* times, but `pos` hasn't been updated yet, causing `code_point_at` to be called at the same index repeatedly and potentially accessing invalid indices. The fix advances `pos` directly by `consumed` (number of code units consumed), matching how surrogate pairs are handled.
