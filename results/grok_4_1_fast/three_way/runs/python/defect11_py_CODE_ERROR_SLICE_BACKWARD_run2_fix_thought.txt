The bug occurs because `random_default` calls `random_letters_numbers` which passes `start=0, end=0` to `_random_internal`. Inside `_random_internal`, when `chars=None` and `letters=False, numbers=False`, it sets `end=127, start=0` AFTER computing `gap = end - start` (which is 0), causing `_next_int_java(rnd, 0)` to raise ValueError('bound must be positive'). The fix reorders the default range derivation BEFORE computing `gap`.
