The test `test_translate_surrogate_pair_with_suffix` fails with an IndexError on line 36 in `code_point_at`: `raise IndexError(f"String index out of range: {index}")`. This happens because `CsvEscaper.translate_string` calls `code_point_at(index, code_units)` where `code_units` comes from `to_utf16_code_units(input_str)`.

For `input_str = "\U0001F630A"` (😰A), `to_utf16_code_units` creates `code_units` with 3 elements: [55357, 56880, 65] (high surrogate, low surrogate for 😰, then 'A').

When processing the surrogate pair at `index=0`, the code correctly identifies it spans `code_units[0:2]`. After handling the pair, `index` advances to 2 (pointing to 'A'). Then `code_point_at(2, code_units)` is called, but line 35 checks `if index < 0 or index >= len(code_units)`, and since `len(code_units)=3`, `index=2` is valid (2 < 3). However, inside `code_point_at`, it likely does `code_units[index+1]` assuming a surrogate pair exists at position 2, but position 2 is the single code unit 'A' (65), so `index+1=3` exceeds the list length, triggering the bounds check failure.

Root cause: `code_point_at` at index 2 (ASCII char) incorrectly assumes a surrogate pair and accesses `code_units[3]`, which doesn't exist.
