The test `test_translate_surrogate_pair_with_suffix` fails with an IndexError in `code_point_at`. When processing input "\U0001F630A" (😰A), the UTF-16 code units are [0xD83D, 0xDE30, 0x0041]. At index=0 in `code_point_at`, it detects the high surrogate (0xD83D) at `code_units[0]`, checks `index + 1 < len(code_units)` (1 < 3, true), reads low surrogate `code_units[1]` (0xDE30), and correctly returns the code point. However, `CsvEscaper.translate_string` then advances the index by only 1 position instead of 2 after consuming the surrogate pair, so next iteration starts at index=1. At index=1, `code_point_at` reads `high = code_units[1]` (0xDE30), which is in 0xDC00-0xDFFF range, but the check `0xD800 <= high <= 0xDBFF` fails since 0xDE30 > 0xDBFF. It then checks `index + 1 < len(code_units)` (2 < 3, true) anyway, reads `low = code_units[2]` (0x0041), but since the high surrogate check failed, it returns `high` (0xDE30). The real bug is later in `translate_string`: after processing this, index advances by 1 to index=2, then `code_point_at(2)` succeeds, but when index reaches 3, `code_point_at` checks `index < len(code_units)` (3 < 3, false) and raises IndexError on line 36.
