The code fails with an IndexError in the test_translate_surrogate_pair_with_suffix test when processing input "\U0001F630A" (emoji 😰 followed by 'A'). Here's what happens:

1. Emoji 😰 converts to UTF-16 code units: [0xD83D, 0xDE30, 0x0041] (high surrogate, low surrogate, then 'A')
2. At pos=0, self.translate_codepoint(code_units, 0, out) consumes 2 code units (surrogate pair) and returns consumed=2
3. The loop then executes: for pt in range(2): (pt=0, pt=1)
4. First iteration (pt=0): code_point_at(code_units, pos=0) returns 0x1F630, char_count(0x1F630)=2, pos += 2 → pos=2
5. Second iteration (pt=1): code_point_at(code_units, pos=2) → index=2 is valid, gets 0x0041 ('A'), char_count(0x0041)=1, pos += 1 → pos=3
6. Loop exits normally (pos=3 == length=3)

However, the IndexError occurs inside translate_codepoint(code_units, pos, out) during a subsequent call or in code_point_at when pos=2. The bug is the unnecessary for pt in range(consumed) loop after translate_codepoint succeeds - it double-advances pos over the same surrogate pair region that translate_codepoint already handled, causing misalignment in later calls to code_point_at or translate_codepoint.
