The bug occurs because `no_more_matches_for_repl_index[i]` is set to True when no match is found from the *current* start position, but it should only be set when no match exists from the *original* text position (i.e., globally no more matches after previous replacements). This causes valid later matches to be skipped. Fix by only setting the flag when `temp_index == -1` *and* the search string had at least one match before (track via `matched_once[i]` flag), ensuring we don't prematurely skip search strings that still have matches ahead.
