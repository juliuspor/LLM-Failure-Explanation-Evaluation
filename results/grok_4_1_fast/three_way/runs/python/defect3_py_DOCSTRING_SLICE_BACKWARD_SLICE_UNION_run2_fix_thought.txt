The bug diagnosis incorrectly identifies the issue. `to_utf16_code_units` correctly populates `code_units` by iterating over each character in `s` and appending UTF-16 code units. The real bug is in the `translate` method's main loop: after `translate_codepoint` returns `consumed > 0`, the code incorrectly advances `pos` by calling `char_count(code_point_at(code_units, pos))` *consumed* times starting from the already-consumed position, causing double-skipping and missing characters. Fix by advancing `pos` directly by `consumed` code units.
