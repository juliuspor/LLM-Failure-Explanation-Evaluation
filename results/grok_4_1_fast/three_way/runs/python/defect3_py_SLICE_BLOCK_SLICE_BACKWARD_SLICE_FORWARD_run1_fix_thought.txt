The bug diagnosis is incorrect - `to_utf16_code_units` correctly populates the list from the input string `s`. The real issue is in the main `translate` method's loop: when a translator returns `consumed > 0`, the loop incorrectly advances `pos` using `char_count(code_point_at(code_units, pos))` which calls `code_point_at` with the current `pos` (already consumed), causing IndexError on empty or advanced positions. Fixed by advancing `pos` directly by `consumed` code units.
