The bug is in the index advancement logic after `translate_codepoint` consumes codepoints. When a surrogate pair is consumed (consumed=2), the loop incorrectly calls `code_point_at(code_units, pos)` where `pos` is still the original index (0), then advances by `char_count` of that codepoint twice, but `pos` isn't updated between calls, causing incorrect advancement. Fix by first advancing `pos` by the consumed code units, then using the new `pos` for any remaining advancement.
