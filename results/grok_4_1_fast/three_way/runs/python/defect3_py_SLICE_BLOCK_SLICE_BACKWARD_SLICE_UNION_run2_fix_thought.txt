The diagnosis is incorrect - code_units IS populated via to_utf16_code_units() before code_point_at is called. The real bug is in the translate() method's main loop: after translate_codepoint() consumes characters, the pos advancement incorrectly uses char_count(code_point_at(code_units, pos)) in a loop, but pos hasn't been updated yet, causing it to repeatedly consume the same codepoint. Fix by advancing pos by the number of code units consumed (consumed * average char size isn't correct; instead calculate total code units spanned by consumed codepoints).
