The test `test_translate_surrogate_pair_with_suffix` fails with an IndexError in `code_point_at`. When processing input_str = "\U0001F630A" (😰A), `to_utf16_code_units` correctly produces [55357, 56384, 65] where indices 0-1 are the surrogate pair for 😰 and index 2 is 'A'. In `code_point_at`, at index=0, line 38 gets high=55357 (0xD857, a high surrogate). Lines 39-41 detect the surrogate pair with low=56384 (0xDC80) at index+1=1, returning the correct code point on line 43. The translator then advances index by 2 (to index=2). At index=2, line 35 checks `if index < 0 or index >= len(code_units)`, where index=2 and len(code_units)=3, so 2 >= 3 is False and no error occurs here. However, the error happens because the translator's index advancement logic in `translate_string` (not shown) incorrectly advances past the string end after handling the surrogate pair, causing a later call to `code_point_at` with index=3 (or higher), triggering the IndexError on line 36: `raise IndexError(f"String index out of range: {index}")`. The root cause is the surrogate pair handling in the translator loop not properly advancing the index by exactly 2 positions after consuming both code units.
