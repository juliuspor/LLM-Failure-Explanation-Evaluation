The bug occurs because after a translator consumes a surrogate pair (2 code units), the main translate loop incorrectly advances `pos` by calling `char_count(code_point_at(code_units, pos))` repeatedly for `consumed` times, using the wrong starting position each iteration. This causes index misalignment. The fix simplifies advancement: after consumption, advance `pos` directly by the number of code units consumed (1 or 2), matching Java's char[] indexing.
