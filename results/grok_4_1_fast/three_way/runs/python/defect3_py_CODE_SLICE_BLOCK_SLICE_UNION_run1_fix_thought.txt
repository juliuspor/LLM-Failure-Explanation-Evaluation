The bug occurs because after `translate_codepoint` consumes `consumed` codepoints, the loop calls `code_point_at(code_units, pos)` repeatedly with the current `pos`, but `pos` is already at or beyond the end after advancing. The fix is to calculate the advancement upfront by summing `char_count` for each consumed codepoint starting from the original position, avoiding repeated `code_point_at` calls at invalid positions.
