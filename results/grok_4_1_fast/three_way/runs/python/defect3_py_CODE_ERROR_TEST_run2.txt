The code fails in test_translate_surrogate_pair_with_suffix() when processing input "\U0001F630A" (😰A). Here's what happens:

1. code_units = to_utf16_code_units("😰A") creates [0xD83D, 0xDE30, 0x0041] (3 UTF-16 code units)
2. At pos=0, self.translate_codepoint() processes the surrogate pair 😰 and returns consumed=2
3. The loop then executes 'for pt in range(consumed):' (pt=0, pt=1)
4. Inside the loop, code_point_at(code_units, pos) is called with pos=0 both times
5. code_point_at(..., 0) internally accesses code_units[2] to check the second surrogate
6. But after first iteration, pos should advance, yet it doesn't before second call
7. IndexError occurs because code_point_at tries to access index 2 when only indices 0-2 exist

Root cause: The for loop calls code_point_at(code_units, pos) TWICE with unchanged pos=0, but pos only advances AFTER both iterations. The comment assumes translators advance pos by code units, but the code advances by codepoints, causing double-processing of the same surrogate pair location.
