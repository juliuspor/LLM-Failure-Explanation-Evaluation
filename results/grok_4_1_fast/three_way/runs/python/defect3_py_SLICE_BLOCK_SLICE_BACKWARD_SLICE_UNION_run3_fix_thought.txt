The diagnosis is incorrect - `to_utf16_code_units` correctly populates `code_units` from the input string. The real bug is in the main `translate` method's loop: after `translate_codepoint` returns `consumed > 0`, it incorrectly advances `pos` by summing `char_count` values starting from the already-consumed position, causing double-skipping and missing characters.
