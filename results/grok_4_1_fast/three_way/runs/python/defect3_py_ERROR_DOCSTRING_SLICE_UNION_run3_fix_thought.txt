The bug diagnosis misidentifies the error location (it's in code_point_at, not to_utf16_code_units). The real issue is in translate(): when translate_codepoint returns 0, we call code_point_at(pos) which can raise IndexError if pos == length (valid end condition). Additionally, after consumed > 0, the loop incorrectly calls code_point_at(pos) which is now invalid since pos hasn't been advanced yet, and increments pos char-by-char which is wrong since translate_codepoint already consumed those codepoints. Fix: handle end-of-input before calling code_point_at, and advance pos by consumed code units directly.
