The code fails in the final for loop when `translate_codepoint` consumes a surrogate pair (returns consumed=2). After `consumed = self.translate_codepoint(code_units, pos, out)`, the loop `for pt in range(consumed): pos += char_count(code_point_at(code_units, pos))` incorrectly advances `pos`.

For input "\U0001F630A" (😰A), `code_units` has length 3: [high surrogate at 0, low surrogate at 1, 'A' at 2]. When `pos=0`, `translate_codepoint` consumes the surrogate pair and returns 2. The loop runs twice:

1st iteration: `code_point_at(code_units, 0)` succeeds (surrogate pair), `char_count=2`, `pos = 0 + 2 = 2`
2nd iteration: `code_point_at(code_units, 2)` → index 2 is valid, but inside `code_point_at`, it checks `index + 1 < len(code_units)` (3 < 3 is false), returns code unit 65 ('A'), `char_count=1`, `pos = 2 + 1 = 3`

Now `while pos < length` (3 < 3 is false), loop exits. But the 'A' at position 2 was never passed to `translate_codepoint` - it was skipped entirely, causing incorrect translation.

The error "IndexError: String index out of range: 2" occurs later in other tests or code when `code_point_at` is called with pos=2 on a shorter string, but the root bug is this incorrect position advancement that skips characters after surrogate pairs.
