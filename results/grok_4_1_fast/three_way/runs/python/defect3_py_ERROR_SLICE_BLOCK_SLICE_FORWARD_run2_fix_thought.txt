The bug occurs because `translate_codepoint` implementations (like CsvEscaper) access `code_units[index + 1]` without checking if `index + 1 < len(code_units)`, bypassing the bounds check in `code_point_at`. The fix adds an explicit bounds check in the main `translate` loop before calling `translate_codepoint`, and ensures proper advancement of `pos` when no translation occurs by using `char_count(codepoint)`.
