The bug is in the `translate` method's surrogate pair advancement logic. When `translate_codepoint` consumes 2 code units (a surrogate pair), the loop incorrectly calls `code_point_at(code_units, pos)` where `pos` is still the original index, then advances `pos` by `char_count` of that codepoint (2) *per* consumed codepoint in a nested loop. This double-advances past the string length. Fix: replace the nested loop with `pos += consumed * 2` for surrogate pairs or use `pos += sum(char_count(code_point_at(code_units, pos + i)) for i in range(consumed))`, but simplest is `pos += consumed` since translators return code unit counts, not codepoints.
