The code fails in the `TimePeriodValues._update_bounds` method when updating `_min_start_index`. After line 343, if the current period is a `SimpleTimePeriod`, `start` is correctly set using `period.get_start_millis()`. Then at line 344-348, it retrieves `min_start` from the period at `self._min_start_index`. However, line 352 unconditionally sets `self._min_start_index = index` regardless of whether `start < min_start` (checked at line 349). This overwrites the correct minimum index with the current `index` every time `add()` is called, so `get_min_start_index()` always returns the index of the *last* added period instead of the true minimum start time index.

This breaks `test_min_start_index`: after adding periods with starts 100 (idx0), 50 (idx1), 200 (idx2), it returns 2 instead of 1.
