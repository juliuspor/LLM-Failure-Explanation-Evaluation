The bug occurs because `RandomStringUtils.random(3, 5, 5, False, False)` passes `end=5` to `_random_internal`, resulting in `gap = end - start = 0`. The `_next_int_java` function raises `ValueError("bound must be positive")` without mentioning `start` or `end`. The fix adds validation in `random` to catch `start == end` early and raise a `ValueError` with a message referencing both parameters, matching the Java behavior expected by the test.
