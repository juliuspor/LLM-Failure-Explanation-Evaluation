The code fails in the SLICE_FORWARD and SLICE_UNION functions at line 34 when `end` is negative. The validation at line 32 checks `if start < 0 or end < 0 ...` and raises IndexError if `end < 0`. However, the docstring specifies that `upper` can be `-1` to mean 'no limit'. When `upper=-1` is passed as `end`, the check `end < 0` triggers IndexError before slicing, even though Python slicing supports negative `end` values (like `s[start:]` when `end=-1`). This prevents the intended 'no limit' behavior.
