The code fails with an infinite loop in the `translate` method's `while pos < length` loop when `self.translate_codepoint` returns `consumed == 0` for a surrogate pair (UTF-16 high/low pair spanning `code_units[pos]` and `code_units[pos+1]`). Here's what happens:

1. `code_point_at(code_units, pos)` calls `code_units[pos]` (line 38 in SLICE_FORWARD), detects a high surrogate (0xD800-0xDBFF at line 39), sees the following low surrogate (line 40-41), and returns a supplementary codepoint (>0xFFFF at line 43).

2. `to_chars(codepoint)` converts this single supplementary codepoint to 2 UTF-16 code units (a high/low surrogate pair).

3. `len(c)` is 2, so `pos += len(c)` advances `pos` by 2, correctly skipping the original surrogate pair.

4. The loop's final `for pt in range(consumed)` iterates 0 times (since `consumed == 0`), so `pos` receives **no additional advancement** from this for-loop.

**Root cause**: When `consumed == 0`, the code only advances `pos` once (via `pos += len(c)`), but the comment and structure expect the for-loop to handle **additional** advancement for each 'consumed' codepoint. Since the loop doesn't run, `pos` doesn't advance enough for surrogate pairs that `translate_codepoint` skips, causing `pos` to stall and loop forever on the same surrogate pair.
