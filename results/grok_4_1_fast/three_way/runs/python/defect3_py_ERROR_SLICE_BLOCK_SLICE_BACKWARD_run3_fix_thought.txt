The bug diagnosis is incorrect - `code_point_at` is called in the `translate` method after `code_units` is fully populated by `to_utf16_code_units`. The real bug is in the `translate` loop: after `translate_codepoint` consumes characters, `pos` is advanced using `code_point_at(code_units, pos)` where `pos` is the new position, but the loop incorrectly uses the old `pos` value for `char_count(code_point_at(code_units, pos))`. Fix by tracking the starting position and advancing `pos` by `consumed * char_count(codepoint_at_start)` or using a while loop to consume exactly `consumed` codepoints.
