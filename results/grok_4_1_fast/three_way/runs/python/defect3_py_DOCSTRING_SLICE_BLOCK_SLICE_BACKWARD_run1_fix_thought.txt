The bug diagnosis is incorrect - the validation in code_point_at() is not the issue as code_units is properly populated before translate_codepoint calls. The real bug is in the translate() method's main loop: after translate_codepoint consumes characters, the pos advancement incorrectly uses char_count(code_point_at(code_units, pos)) in a loop, but pos hasn't been updated yet, causing it to repeatedly consume the same codepoint. Fix by advancing pos by the number of code units consumed (consumed * average char size isn't reliable), instead calculate total code units consumed properly.
