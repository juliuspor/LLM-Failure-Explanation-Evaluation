The bug is in the main `translate` method, not `to_utf16_code_units`. The issue occurs when `translate_codepoint` consumes a surrogate pair (returns 2), but the position advancement loop incorrectly calls `code_point_at(code_units, pos)` where `pos` is still the original index, causing IndexError. Fixed by advancing `pos` by the number of code units consumed (`consumed`), since `translate_codepoint` returns the count of code units (not codepoints) it consumed.
