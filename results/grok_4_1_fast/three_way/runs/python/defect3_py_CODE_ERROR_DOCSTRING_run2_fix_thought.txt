The bug occurs because the for loop calls `code_point_at(code_units, pos)` repeatedly without advancing `pos` inside the loop body first. When `consumed > 1`, it indexes the same position multiple times. After the first iteration, `pos` jumps ahead (especially for surrogate pairs where `char_count` returns 2), causing subsequent `code_point_at` calls to access out-of-bounds indices. Fix by advancing `pos` incrementally inside the loop: get codepoint at current `pos`, advance `pos` by its `char_count`, and repeat `consumed` times.
