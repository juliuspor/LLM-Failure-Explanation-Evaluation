The bug is in the translate method's position advancement logic after translator consumption. When a translator consumes characters (including surrogate pairs), the loop incorrectly uses `char_count(code_point_at(code_units, pos))` which re-calls `code_point_at(pos)` repeatedly with the same pos, causing it to treat the low surrogate as a standalone codepoint (returning raw value since no following high surrogate), advancing by only 1 instead of 2, leading to pos overflow. Fix by advancing directly by the consumed count since translators already guarantee correct UTF-16 advancement.
