The code fails in TimePeriodValues._update_bounds because it incorrectly updates max_middle_index when adding a new period. In test_max_middle_index_simple, after adding periods with middle points 50 (idx0), 150 (idx1), 25 (idx2), and 75 (idx3), it returns 3 instead of 1.

The bug occurs around lines 334-352 in the else branch (for non-SimpleTimePeriod). When a new period is added:
1. It calculates the new middle time correctly
2. If _max_middle_index >= 0 (true after first period), it gets the CURRENT max middle period using self.get_data_item(self._max_middle_index)
3. It compares new_middle > current_max_middle
4. If true, it correctly sets self._max_middle_index = index
5. **But then it ALWAYS executes self._max_middle_index = index regardless** (similar to the min_start_index bug at lines 349-352)

This means EVERY new period overwrites max_middle_index with its own index, so the final value is always the last added index (3), not the index of the actual maximum middle time (1). The min_start_index logic has the same flaw at lines 349-352.
