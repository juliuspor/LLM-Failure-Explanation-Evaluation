The bug occurs because when `_min_start_index` (and similar indices) is -1, the comparison block is skipped and the index is always set to the new index without comparing values. This fails to maintain the true minimum/maximum. The fix ensures that when an index is -1 (first item), it is set to the current index; when >=0, it compares the new period's value with the current extremum and updates if necessary. Fixed a bug in max_middle where wrong index was used.
