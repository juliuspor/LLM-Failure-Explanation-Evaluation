The bug occurs because after `translate_codepoint` consumes code units and modifies `out`, the loop `for pt in range(consumed): pos += char_count(code_point_at(code_units, pos))` calls `code_point_at(code_units, pos)` with the *updated* `pos` value, which can be near the end of `code_units`. When `pos` points to a high surrogate near the end, `code_point_at` tries to access `code_units[pos+1]` (out of bounds), causing IndexError. The fix calculates the advancement once using the *original* `pos` before the loop, ensuring `code_point_at` is only called with safe indices.
