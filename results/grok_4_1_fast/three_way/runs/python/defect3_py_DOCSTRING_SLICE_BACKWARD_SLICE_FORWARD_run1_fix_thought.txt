The bug diagnosis incorrectly identifies `to_utf16_code_units` as empty - it actually populates correctly. The real bug is in `translate` method: after `translate_codepoint` returns `consumed > 0`, the loop `for pt in range(consumed): pos += char_count(code_point_at(code_units, pos))` uses the wrong `pos` (already advanced incorrectly) and calls `code_point_at` with potentially invalid indices. Fix by calculating total code units to advance using `char_count` on each consumed codepoint before advancing `pos`.
