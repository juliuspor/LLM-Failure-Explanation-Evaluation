The test `test_max_middle_index_simple` fails because `get_max_middle_index()` returns 3 instead of the expected 1. The periods added have middle points: index 0 (mid=50), index 1 (mid=150), index 2 (mid=25), index 3 (mid=75). Index 1 has the maximum middle value (150).

In `TimePeriodValues._update_bounds()`, the code incorrectly uses `self._max_middle_index` to compare against the new period's middle value when updating the maximum. It should use the middle value of the *current max holder* (`self._periods[self._max_middle_index].get_middle_ms()`) for comparison.

This bug causes the code to think index 3 (mid=75) has a larger middle than index 1 (mid=150), so it wrongly sets `self._max_middle_index = 3`. The same logic error likely affects min_start_index and other bounds calculations, but the test specifically catches the max_middle_index failure.
