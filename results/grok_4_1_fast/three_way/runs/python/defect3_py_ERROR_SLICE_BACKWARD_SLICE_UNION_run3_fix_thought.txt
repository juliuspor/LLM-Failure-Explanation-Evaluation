The bug occurs because `translate_codepoint` implementations (like CsvEscaper) access `code_units[index+1]` without checking if `index+1 < len(code_units)`, causing `code_point_at` to be called later with an out-of-bounds index (e.g., index=2 on empty list). Fix by adding bounds check before surrogate pair handling in `translate` loop and ensuring proper advancement.
