The bug is in the offset calculation logic for negative hours and positive minutes. When hours_offset is negative, the code incorrectly sets minutes_offset = hours_in_minutes - minutes_offset (e.g., -120 - 15 = -135), then multiplies by 60,000 to get -8,100,000ms (-2:15), but the validation passes since it only checks minutes_offset bounds before recalculation. The fix normalizes to total minutes correctly: always compute total_minutes = hours_offset * 60 + minutes_offset, then validate the final total_minutes range (-1439 to 1439) before creating the offset.
