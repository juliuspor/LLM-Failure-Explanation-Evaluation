The code fails with an IndexError in the final for loop when processing surrogate pairs followed by additional characters (like in test_translate_surrogate_pair_with_suffix with input '\U0001F630A'). Here's what happens:

1. translate_codepoint() processes a surrogate pair (2 UTF-16 code units, e.g., 😰) and returns consumed = 2
2. The loop enters the 'if consumed == 0' branch (skipped) and reaches the for loop: for pt in range(consumed): (runs twice)
3. Inside the loop, pos is still the original position (high surrogate start). code_point_at(code_units, pos) is called
4. code_point_at() at a high surrogate position internally advances index by 1 to check the next code unit (low surrogate), causing index == pos + 1
5. For the second iteration (pt=1), pos hasn't advanced yet, so code_point_at() again starts at the high surrogate, computes index = pos + 1 (now pointing to the 'A' character or beyond end), and raises IndexError at line 36 when index >= len(code_units)

Root cause: pos is not advanced before the for loop, so code_point_at(code_units, pos) repeatedly processes the same surrogate pair position, causing internal index overstep on second iteration.
