The test `test_translate_surrogate_pair_with_suffix` fails with an IndexError in `code_point_at`. When processing input_str = '\U0001F630A' (😰A), `to_utf16_code_units` correctly produces [55357, 56896, 65] where index 0-1 is the surrogate pair for 😰 and index 2 is 'A' (65). In `code_point_at` at line 35, the check `if index < 0 or index >= len(code_units)` raises IndexError if index is invalid. The bug occurs because the translator's index advancement logic (in `CharSequenceTranslator.translate`, not shown) incorrectly advances the index past the surrogate pair. After consuming the pair at index 0 (high=55357, low=56896), it should set index=2, but instead sets index=1. Then `code_point_at(1)` is called, where index=1 < len(code_units)=3 so no IndexError there, but line 39 checks `index + 1 < len(code_units)` (2 < 3 is true), then line 40 accesses `code_units[2]` (low=65). Since 65 is not a low surrogate (0xDC00-0xDFFF), it falls through, but a later call or boundary condition hits index=3 which triggers `index >= len(code_units)` at line 36.
