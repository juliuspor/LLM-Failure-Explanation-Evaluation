2. For "\U0001F630A", the UTF-16 encoding produces exactly 3 code units: [55357, 56896, 65] (high surrogate, low surrogate for 😰, then 'A').
3. Later code (likely in `CsvEscaper.translate_string` or `code_point_at`) attempts to access `code_units[2]`.
4. At the moment of access, `index == 2` but `len(code_units) == 3`, so line 35's check `if index < 0 or index >= len(code_units)` incorrectly raises `IndexError` because `2 >= 3` is False, but the code still treats it as out of range or the index gets incremented incorrectly afterward.

Root cause: The index advancement logic after processing the surrogate pair (😰, which consumes indices 0 and 1) fails to correctly set `index = 2` for the next character 'A', causing an incorrect bounds check or access attempt.
