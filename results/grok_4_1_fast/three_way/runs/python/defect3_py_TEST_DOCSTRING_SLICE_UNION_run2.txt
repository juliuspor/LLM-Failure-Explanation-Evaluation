The test `test_translate_surrogate_pair_with_suffix` fails with an IndexError in `code_point_at` at line 36. When processing input_str = '\U0001F630A' (😰A), the `translate` method calls `code_point_at(0)` which correctly converts the surrogate pair 😰 (code units at indices 0 and 1 in the UTF-16 `code_units` list from `to_utf16_code_units`) to code point U+1F630 and advances `index` to 2. Then `code_point_at(2)` is called for character 'A', but line 35 checks `if index < 0 or index >= len(code_units)`, and since len(code_units) == 3 (2 surrogates + 1 for 'A'), index=2 is valid. The real failure occurs because `translate_codepoint()` (in CsvEscaper) consumes the single code point 'A' at index 2 but fails to advance `index` by 1, causing the next `code_point_at(2)` call. This repeatedly indexes code_units[2] which triggers the surrogate pair check at lines 38-41: `high = code_units[2]` (0x41 for 'A') is not a high surrogate (0xD800-0xDBFF), but `index + 1 < len(code_units)` is false (3 < 3), so it returns 0x41 correctly. However, without index advancement, it loops infinitely at index=2 until hitting the end condition, but the bug manifests as IndexError when surrogate detection mishandles the boundary condition at the end of code_units.
