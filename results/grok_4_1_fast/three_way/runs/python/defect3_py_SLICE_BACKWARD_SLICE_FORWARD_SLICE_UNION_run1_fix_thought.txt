The diagnosis is incorrect - `to_utf16_code_units` correctly populates `code_units` with UTF-16 code units from the input string. The real bug is in the `translate` method's main loop: after `translate_codepoint` returns `consumed > 0`, the position advancement loop incorrectly calls `code_point_at(code_units, pos)` (where `pos` is the original index) instead of advancing `pos` by `consumed` code units directly. This causes infinite loops or incorrect positioning for surrogate pairs. Fixed by simply advancing `pos += consumed`.
