The bug is in the translate method's fallback path. When no translator consumes a codepoint (consumed == 0), it calls code_point_at(pos) to get the codepoint, then appends to_chars(codepoint) and advances pos by len(c) (number of code units). However, when a translator like CsvEscaper returns 0 for surrogate pairs, this causes overshooting because code_point_at is called at pos=0 (surrogate pair), consumes 2 code units, but subsequent calls at pos=2 fail bounds check. Fix: always advance by char_count(code_point_at(pos)) in fallback to match translator contract.
