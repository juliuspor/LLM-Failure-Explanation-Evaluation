The bug diagnosis misidentifies the problem location. The actual issue is in `translate` method: after `consumed == 0`, it calls `code_point_at(code_units, pos)` which validates `pos < len(code_units)`, but then incorrectly advances `pos += len(c)` using UTF-16 code unit count instead of codepoint count. This can make `pos` exceed `length`, causing next `code_point_at` to fail validation. Fix by advancing `pos` by the correct codepoint length using `char_count(codepoint)`.
