The bug is in the translate method's pos advancement logic. When a translator consumes codepoints (consumed > 0), the loop incorrectly calls code_point_at(pos) which can raise IndexError if pos is at the last single code unit. Additionally, it advances pos one char_count at a time for 'consumed' times, but should advance by the total UTF-16 code units consumed. Fix: when consumed > 0, advance pos by consumed * average char size conservatively, but primarily avoid code_point_at(pos) call at boundary by checking bounds first.
