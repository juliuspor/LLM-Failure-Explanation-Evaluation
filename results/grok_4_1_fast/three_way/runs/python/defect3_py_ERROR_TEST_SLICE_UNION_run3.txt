The code fails in the `to_utf16_code_units` function during the `test_translate_surrogate_pair_with_suffix` test with input "\U0001F630A" (emoji 😰 followed by 'A'). The `code_point_at` function calls `to_utf16_code_units(s)` which converts the string to UTF-16 code units: [55357, 56384, 65] (high surrogate, low surrogate, 'A'). When `index=2` is passed to access the 'A' code unit at line 38 (`high = code_units[index]`), the prior check at line 35 (`if index < 0 or index >= len(code_units)`) incorrectly raises IndexError because `len(code_units)` is 3, making `index=2` valid. This reveals a bug in the index advancement logic within `CsvEscaper.translate_string`: after processing the surrogate pair at indices 0-1, it advances the index by only 1 instead of 2, causing `index=2` (out of bounds for the remaining logic expecting position after the pair). The surrogate pair consumes 2 code units but the loop doesn't skip both correctly.
