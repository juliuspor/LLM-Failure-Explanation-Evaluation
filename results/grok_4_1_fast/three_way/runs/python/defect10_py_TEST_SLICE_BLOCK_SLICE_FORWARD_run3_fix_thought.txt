The bug occurs because `char_at(numeric, 0)` is called on 'bobL'[:-1] = 'bob', where numeric[0] = 'b' passes the is_digits check incorrectly. However, char_at is called with index 0 which is safe. The real issue is the logic allows invalid non-numeric strings to proceed to create_long/create_big_integer without proper validation. Added early validation using is_number() to reject invalid inputs like 'bobL' before any slicing or parsing, ensuring ValueError is raised consistently.
