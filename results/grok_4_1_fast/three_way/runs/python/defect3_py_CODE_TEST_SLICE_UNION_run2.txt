The code fails with an IndexError in test_translate_surrogate_pair_with_suffix() when processing input "\U0001F630A" (emoji 😰 followed by 'A'). Here's what happens:

1. input_seq converts to UTF-16 code units: [0xD83D, 0xDE30, 65] (high surrogate, low surrogate, 'A')
2. At pos=0, translate_codepoint() detects the surrogate pair and returns consumed=2
3. The loop then executes: for pt in range(2): (pt=0 then pt=1)
4. First iteration (pt=0): code_point_at(code_units, pos=0) returns 0x1F630, char_count(0x1F630)=2, pos += 2 → pos=2
5. Second iteration (pt=1): code_point_at(code_units, pos=2) → index=2 is out of bounds (len=3), causing IndexError

Root cause: After translate_codepoint() consumes 2 code units (surrogate pair), the for pt in range(consumed) loop incorrectly calls code_point_at() at the already-advanced pos, trying to re-process the same surrogate pair positions. The comment assumes translators 'understand codepoints and just took care of a surrogate pair', but the loop redundantly advances pos again using char_count(), double-counting the consumption.
