The code fails with an IndexError in `code_point_at(code_units, pos)` when processing surrogate pairs followed by another character, like "\U0001F630A" (😰A). Here's what happens:

1. `translate_codepoint(code_units, pos, out)` consumes 2 UTF-16 code units for the surrogate pair 😰 and returns `consumed = 2`.
2. The loop then executes `for pt in range(consumed):` (runs twice).
3. First iteration: `pos` is still the original position (0). `code_point_at(code_units, pos)` works fine, returning the emoji codepoint. `char_count()` returns 1, so `pos += 1` (now pos=1).
4. Second iteration: `pos=1` points to the low surrogate of the pair. `code_point_at(code_units, 1)` sees a low surrogate (0xDC00-0xDFFF), treats it as a standalone BMP char, and `char_count()` returns 1, so `pos += 1` (now pos=2).
5. Next `while pos < length` iteration: `pos=2` points to 'A'. `translate_codepoint()` returns `consumed=0` (no translation needed).
6. Falls to fallback: `code_point_at(code_units, 2)` is called, but `length=3`, so index 2 is valid.

**The real crash**: On the *next* call or malformed input, `pos` can reach/exceed `length`. The bug comment "they just took care of a surrogate pair" is wrong - after `consumed=2`, the loop manually advances `pos` by *codepoint counts* (1+1=2), but `translate_codepoint` already expects to skip 2 UTF-16 units. This double-advancement corrupts `pos`, eventually causing `code_point_at` IndexError at line 36 when `pos >= len(code_units)`.
