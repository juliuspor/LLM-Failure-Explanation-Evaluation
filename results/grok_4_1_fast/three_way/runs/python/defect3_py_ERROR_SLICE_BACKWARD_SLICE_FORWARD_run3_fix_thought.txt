The bug diagnosis is incorrect - `to_utf16_code_units` correctly populates `code_units` and `code_point_at` has proper bounds checking. The real bug is in the main `translate` loop: after `translator.translate_codepoint` consumes characters, the loop incorrectly calls `code_point_at(code_units, pos)` where `pos` is the same unchanged position, causing IndexError when it exceeds bounds. Fixed by advancing `pos` by `consumed` code units directly, matching the contract that translators return code units consumed.
