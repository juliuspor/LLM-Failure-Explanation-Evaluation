The bug is in the `translate` method of `CharSequenceTranslator`. When `consumed` is non-zero (i.e., the translator handled some codepoints), the loop advances `pos` incorrectly. The original code uses `for pt in range(consumed): pos += char_count(code_point_at(code_units, pos))`. However, `code_point_at` expects an index into `code_units`, but `pos` is being updated inside the loop, causing the second call to `code_point_at` to potentially start in the middle of a surrogate pair if the first consumed codepoint was a surrogate pair (2 code units). The fix is to compute the total code units consumed by iterating over the codepoints but using a temporary offset that doesn't modify `pos` until after the loop. Alternatively, we can simply advance `pos` by the number of code units that were actually consumed by the translator? But the translator's `translate_codepoint` returns the number of codepoints consumed, not code units. So we need to convert consumed codepoints to code units by examining the input code units. The safest way is to iterate over the consumed codepoints, each time getting the codepoint at the current position (which is pos + offset) and adding its char count. This ensures we skip the correct number of code units. Also, note that when `consumed == 0`, we already advance by `char_count(codepoint)`, which is correct. So we only need to fix the else branch.
