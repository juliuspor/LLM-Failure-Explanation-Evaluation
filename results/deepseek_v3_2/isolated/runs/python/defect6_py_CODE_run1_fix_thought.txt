The bug is an infinite loop when mantissa becomes 0. This occurs for x = 0 because abs(0) >= 100 is false, so the loop is not entered. However, if x is a non‑zero integer that is a multiple of 10, mantissa eventually becomes ±1, not 0. But there is a subtle case: when x is a float that equals an integer and has absolute value >= 100, and its integer representation has trailing zeros, the loop works correctly. The real issue is that the loop condition does not guard against mantissa becoming 0 after integer division. For example, if mantissa is 10 and we divide by 10, we get 1, not 0. However, if mantissa were 0 (which can't happen with the current logic), the loop would run forever. To make the code robust, we add a check mantissa != 0. Additionally, the check_val calculation uses floor division, which for negative numbers is correct but may be confusing; we keep it as is because it works. The fix is to change the while condition to while mantissa != 0 and mantissa % 10 == 0.
