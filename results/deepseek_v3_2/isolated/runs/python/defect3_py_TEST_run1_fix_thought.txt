The bug is in the `translate` method's handling of the `consumed` return value from `translate_codepoint`. The original code incorrectly assumed that `consumed` is the number of code units (chars) to advance, but the contract says it's the number of codepoints consumed. For surrogate pairs, each codepoint may be one or two code units. The fix is to correctly advance `pos` by the total code units for the consumed codepoints. When `consumed == 0`, we already advance by `char_count(codepoint)`. When `consumed > 0`, we need to iterate through the consumed codepoints, advancing by the char count for each. This ensures proper handling of surrogate pairs and avoids index errors.
