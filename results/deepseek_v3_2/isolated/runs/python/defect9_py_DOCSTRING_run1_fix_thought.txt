The bug was that the algorithm did not correctly handle overlapping search strings because after a replacement, it continued scanning from the end of the original search string in the original text, but the replacement might have changed the length. However, the algorithm builds the result incrementally and only looks at the original text for matches, so the replacement length does not affect the scanning position in the original text. The real issue is that the algorithm picks the earliest match in the original text, but after replacing, it should continue scanning from the end of the matched substring in the original text, not from the start of the replacement. This is correct because overlapping matches that start before the end of the matched substring would have been found earlier (since we always pick the earliest start). The original implementation had a flaw: it used a boolean array to mark search strings that no longer have matches, but it didn't reset the search after each replacement. The fix is to simplify the algorithm: filter out invalid pairs, then repeatedly find the earliest match among all search strings, append the part before the match and the replacement, and move the current position past the matched substring in the original text. This ensures overlapping patterns are handled correctly because we always pick the earliest match, and after replacement, we move past the entire matched substring, so any overlap that starts inside it is ignored (as it should be, because we already replaced that part).
