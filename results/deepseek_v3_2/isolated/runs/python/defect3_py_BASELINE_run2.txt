The code fails because the `while` loop incorrectly advances `pos` after a surrogate pair is handled by `translate_codepoint`. In the loop, when `consumed > 0` (meaning `translate_codepoint` processed some code units), the `for` loop increments `pos` by calling `char_count(code_point_at(code_units, pos))` for each consumed unit. However, `code_point_at` is called with the current `pos`, which may be at a surrogate pair (two UTF-16 code units). The `code_point_at` function returns a single Unicode code point for the pair, but `char_count` returns 2 for a surrogate pair. The `for` loop then adds 2 to `pos` for each consumed unit, causing `pos` to overshoot the length of `code_units`. Specifically, for a surrogate pair (e.g., emoji 😰) followed by another character (e.g., 'A'), `consumed` might be 1 (if `translate_codepoint` handled the pair), but the `for` loop runs once and adds 2 to `pos`, skipping past the pair. Then, in the next iteration, `pos` is 2, but `code_units` has length 3 (two units for the surrogate pair and one for 'A'). The `while` condition `pos < length` is true (2 < 3), but `code_point_at(code_units, pos)` is called with `pos=2`, which is within bounds. However, if `translate_codepoint` returns 0 for the next character, the `if consumed == 0` block calls `code_point_at` again, which is fine. The error occurs later when the `for` loop runs again and `pos` becomes 3, and on the next `while` iteration, `code_point_at` is called with `pos=3`, which is out of range (since `len(code_units)` is 3), triggering the IndexError from line 35-36 in `code_point_at`. The root cause is that the `for` loop overcounts the advance for surrogate pairs: it should advance `pos` by the number of UTF-16 code units consumed, not by the `char_count` of the code point at the current position. The fix is to change the `for` loop to advance `pos` by 1 for each consumed code unit, or better, to track the consumed code units directly.
