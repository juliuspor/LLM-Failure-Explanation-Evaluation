The bug is in the loop that advances `pos` after a translator consumes characters. The original code used a for-loop that incorrectly added char counts for each consumed codepoint, which could overshoot. The fix is to simply add `consumed` (which is the number of code units consumed) to `pos`. This matches the contract that `translate_codepoint` returns the number of code units consumed, not codepoints. Also, when no translation occurs (consumed == 0), we already advance by the char count of the current codepoint, which is correct. The fixed code removes the problematic for-loop and replaces it with `pos += consumed`.
