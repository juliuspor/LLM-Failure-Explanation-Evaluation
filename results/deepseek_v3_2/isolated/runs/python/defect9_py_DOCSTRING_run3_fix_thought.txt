The bug is that the original function uses the original text for all find operations, ignoring the changes made by replacements. This causes missed matches because the text length and content change after each replacement. The fix is to process the text incrementally: after each replacement, we continue searching in the remaining part of the original text (the part after the matched substring). This ensures that all matches are found in the original order, and replacements do not affect the search for later matches (since we only search in the unprocessed suffix). This approach also naturally handles overlapping search strings because we only remove the matched substring from the text, so overlapping matches that start before the end of the matched substring are not possible (they would have been found earlier). The algorithm is similar to the Java Apache Commons Lang implementation, which also processes left-to-right without backtracking. The key change is to update the text being searched after each match.
