The bug is that the algorithm uses the original text to find matches after replacements, which is incorrect because replacements change the string. The fix is to process replacements sequentially on a mutable string (or using a string builder pattern) and find matches in the updated string. We filter out invalid search/replacement pairs, then iteratively find the earliest match among all search strings from the current position in the working string, perform the replacement, and continue. This ensures that each match is found in the current state of the string, and we avoid scanning already replaced parts. The new implementation avoids the original bug and matches the expected behavior of Apache Commons Lang's replaceEach.
