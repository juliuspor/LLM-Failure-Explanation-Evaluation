The bug is in the `translate` method's loop when handling consumed codepoints. The original code incorrectly assumed that each codepoint corresponds to exactly one code unit, but supplementary characters (codepoints > 0xFFFF) are represented by two code units (a surrogate pair). The fix is to properly advance the position by the number of code units for each consumed codepoint, using `char_count`. Also, in the zero-consumption branch, we already use `char_count`, but we should ensure consistency. The fix updates the loop that processes consumed codepoints to use `char_count` for each codepoint.
