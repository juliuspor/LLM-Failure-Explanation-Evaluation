The bug is in the `_random_internal` method, not the `random` method. The diagnosis indicates that when `letters=True` and `numbers=True`, the character set may be empty if the range [start, end) contains no letters or digits. This leads to `gap = end - start` being used as bound in `_next_int_java`, but the actual allowed characters are filtered later. However, the issue is that the algorithm may loop indefinitely because no characters are accepted. The Java implementation handles this by throwing an exception if no valid characters are in the range. The fix is to add a check in `_random_internal` to ensure that at least one character in the range satisfies the letters/numbers constraints. If not, raise a ValueError. Also, ensure that when chars is provided and empty, we already raise an error. The fix will be applied to `_random_internal`.
