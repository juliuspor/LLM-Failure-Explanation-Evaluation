The bug occurs because `translate` incorrectly increments `pos` when `consumed > 0`. The loop `for pt in range(consumed):` calls `char_count(code_point_at(code_units, pos))` which reads the codepoint at the current `pos`, but `pos` is not updated within the loop, leading to repeated reads of the same codepoint and an infinite loop. When `consumed` is 2 (e.g., for a surrogate pair), the loop runs twice, each time consuming the same surrogate pair, causing `pos` to jump by 4 code units instead of 2, skipping past the end of the list. The fix is to compute the total code units consumed directly from `consumed` (which is in codepoints) by iterating through the codepoints and summing their `char_count`.
