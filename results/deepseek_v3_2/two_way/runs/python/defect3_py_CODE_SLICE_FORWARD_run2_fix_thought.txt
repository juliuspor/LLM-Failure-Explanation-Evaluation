The bug is that when `consumed == 0`, the code incorrectly advances `pos` by the number of output code units (`len(c)`) instead of the number of input code units consumed to form the codepoint. The function `code_point_at` may read one or two input code units (if a surrogate pair), but doesn't return that count. The fix is to compute `input_consumed` by checking if the current code unit is a high surrogate followed by a low surrogate. If so, advance by 2; otherwise by 1. Then use `input_consumed` to update `pos`. This ensures the loop progresses correctly through the input array.
