The bug is that the `to_locale` function fails on locale strings with double underscores (e.g., 'fr__POSIX') because it expects exactly one underscore between language and country, and one between country and variant. The fix is to split the locale string by underscore and handle empty segments appropriately, similar to Java's Locale constructor. We'll parse the parts: language (required, lowercase), country (optional, uppercase), variant (optional, as-is). This accommodates double underscores and is more robust.
