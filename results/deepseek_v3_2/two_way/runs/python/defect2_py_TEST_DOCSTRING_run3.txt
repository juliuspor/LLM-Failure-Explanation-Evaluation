However, the original Java implementation likely threw an exception for out-of-bounds values because it did not clamp. The Python implementation may have replicated this behavior, causing test failures for test_lower_bound_clamping and test_upper_bound_clamping. Specifically, the method likely computes an intermediate value (e.g., via linear interpolation) without first clamping the input, leading to a color component outside 0..255, which triggers a ValueError as per the docstring. The issue occurs in the get_paint method where value is used directly in the formula without clamping. For example, with default bounds (0.0, 1.0), a value of -0.5 would produce a negative component, causing ValueError. The fix is to clamp value to [lower, upper] before interpolation.
