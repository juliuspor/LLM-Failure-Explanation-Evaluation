The bug is that the original `to_locale` function did not handle locale strings with empty country codes (e.g., "fr__POSIX"). The validation logic assumed that a variant is always preceded by a single underscore after a two-letter country code. However, when the country is empty, the format is language + "__" + variant (double underscore). The fix restructures the parsing to allow for empty country by checking if the character after the first underscore is another underscore. If so, country is empty and variant starts after the second underscore. Otherwise, parse two uppercase letters as country, then optionally an underscore and variant. This matches Java's Locale.toString() behavior.
