The bug is in the `translate` method's loop where it advances `pos` after a translator consumes codepoints. The current code uses `char_count(code_point_at(code_units, pos))` for each consumed codepoint, but `char_count` returns 1 for BMP characters and 2 for supplementary characters (surrogate pairs). However, `code_point_at` already handles surrogate pairs and returns the full codepoint, so `char_count` correctly returns 2 for the emoji. The issue is that the loop iterates `pt` over `range(consumed)`, where `consumed` is the number of codepoints consumed, not code units. For the emoji, `consumed` should be 1 (one codepoint), but the emoji occupies two code units. The loop then calls `char_count(code_point_at(code_units, pos))` with `pos` still at 0, which returns 2, advancing `pos` by 2. That's correct. But the test failure suggests a different problem: the `CsvEscaper.translate_codepoint` returns 2 for a surrogate pair (lines 200-205), meaning it consumed two code units. However, the `translate` method treats `consumed` as codepoints, not code units. When `consumed` is 2, the loop iterates twice, each time calling `char_count` on the same position (0), which returns 2, advancing `pos` by 4 total, skipping past the 'A'. The fix is to change the advancement logic to treat `consumed` as the number of code units consumed, not codepoints. In the `translate` method, after a translator returns `consumed`, we should advance `pos` by `consumed` directly, because `consumed` is already the number of code units consumed (as per the contract of `translate_codepoint`). The existing loop incorrectly assumes `consumed` is codepoints and tries to convert each codepoint to code units via `char_count`. We'll replace the loop with a simple `pos += consumed`.
