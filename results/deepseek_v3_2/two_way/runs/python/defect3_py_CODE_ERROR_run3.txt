The error occurs in the while loop when the code calls `char_count(code_point_at(code_units, pos))`. The `char_count` function likely returns 1 for a BMP character and 2 for a surrogate pair. However, the loop variable `pt` iterates from 0 to `consumed-1`, and inside the loop, `pos` is incremented by `char_count(code_point_at(code_units, pos))` for each iteration. This means `pos` changes within the loop, but `code_point_at(code_units, pos)` is called with the updated `pos` each time, which can skip over characters incorrectly. Specifically, if `consumed` is 2 (meaning two code units were processed), the loop runs twice, incrementing `pos` by the character count of the character at the current `pos` each time. But after the first increment, `pos` may point to a new character, and calling `code_point_at` again may read a surrogate pair incorrectly, leading to an index out of range because it tries to access beyond `code_units` length. The issue is that `code_point_at` should use the original position plus an offset based on `pt`, not the updated `pos`. For example, if `pos` starts at 0 and `consumed` is 2, the first iteration increments `pos` by 2 (for a surrogate pair), making `pos` 2. The second iteration then calls `code_point_at(code_units, 2)`, which may be out of range if `length` is 2. The fix is to compute the increment based on the original position: store the starting position before the loop, and for each `pt`, add the character count of the character at `start_pos + offset`.
