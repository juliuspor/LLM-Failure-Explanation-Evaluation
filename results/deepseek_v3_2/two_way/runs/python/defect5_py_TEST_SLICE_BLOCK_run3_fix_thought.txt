The bug occurs because when both array and element are None, inferred_type becomes object, and the code raises a TypeError if expected_type is not object. However, adding None to None should succeed regardless of expected_type, as the expected_type is just a hint for Java-style type checking and should not block the operation when the inferred type is object. The fix is to only raise the TypeError when inferred_type is object AND the array is not None (i.e., we are actually trying to cast an existing object list to a more specific type). If array is None, we are creating a new list, so no cast is needed.
