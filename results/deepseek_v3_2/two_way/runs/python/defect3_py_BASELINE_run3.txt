The IndexError occurs in the while loop of translate() when processing surrogate pairs. The bug is in the line `pos += char_count(code_point_at(code_units, pos))` inside the for loop. After `translate_codepoint()` consumes a surrogate pair (consumed > 0), the for loop iterates `consumed` times. Each iteration calls `code_point_at(code_units, pos)`, which reads the UTF-16 code unit at index `pos`. However, `pos` is not updated between iterations, so the same index is read multiple times. For a surrogate pair, `char_count()` returns 2, so `pos` increases by 2 each iteration. If `consumed` is 1 (meaning one Unicode codepoint was handled), the for loop runs once, and `pos` increases by 2 correctly. But if `consumed` is greater than 1, the second iteration will read `code_point_at(code_units, pos)` where `pos` is now 2 positions ahead, potentially exceeding the string length. In the failing test, input "\U0001F630A" has a surrogate pair (2 code units) followed by 'A'. `translate_codepoint()` likely returns consumed=1 for the emoji. The for loop runs once, `pos` increases by 2, moving past the surrogate pair. The while loop continues, `pos` (2) equals `length` (3), so the loop exits correctly. However, if `translate_codepoint()` returned consumed=2 or more, the second iteration would cause an out-of-range error at `code_point_at()` because `pos` would be 4, exceeding `length` (3). The fix is to advance `pos` by the total code units consumed, not per codepoint in the loop. Replace the for loop with `pos += consumed_char_count` where `consumed_char_count` sums the UTF-16 units for all consumed codepoints.
