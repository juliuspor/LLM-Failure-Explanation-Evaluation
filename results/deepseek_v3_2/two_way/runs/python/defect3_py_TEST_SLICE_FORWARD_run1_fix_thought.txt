The bug is in the `translate` method's loop when handling consumed codepoints. The original code incorrectly assumed that each consumed codepoint corresponds to exactly one UTF-16 code unit, but a codepoint > 0xFFFF (represented by a surrogate pair) occupies two code units. The fix is to advance `pos` by the number of code units for each consumed codepoint, using `char_count`. Also, in the `consumed == 0` branch, we already used `char_count` correctly, but we need to ensure we use `char_count(codepoint)` instead of `len(c)` (which is the same, but consistent). The loop for consumed codepoints now correctly advances `pos` by the code unit count for each codepoint.
