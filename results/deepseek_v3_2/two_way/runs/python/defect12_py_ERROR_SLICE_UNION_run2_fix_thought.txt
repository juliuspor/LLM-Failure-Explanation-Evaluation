The bug is in `substring_java` which incorrectly raises an IndexError when `end > len(s)`. Python slicing handles out-of-range indices gracefully, so we should remove the bounds checks for `start` and `end` being greater than the string length. However, we must still check for negative indices and `start > end` because those are invalid in both Java and Python. The fix is to adjust `substring_java` to only raise for negative indices or `start > end`, and let Python slicing handle the rest.
