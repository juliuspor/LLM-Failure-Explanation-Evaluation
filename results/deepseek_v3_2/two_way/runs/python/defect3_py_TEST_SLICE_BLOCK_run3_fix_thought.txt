The bug is in the `translate` method's loop logic. When a surrogate pair is processed by `translate_codepoint` (consumed=2), the loop incorrectly advances `pos` by summing `char_count` for each consumed codepoint, but `pos` is already at the start of the surrogate pair. The inner loop `for pt in range(consumed):` calls `char_count` on the same codepoint multiple times, leading to an over-advance. Instead, we should advance `pos` by the number of UTF-16 code units consumed, not by codepoint counts. Since `translate_codepoint` returns the number of codepoints consumed, we need to convert that to code units. We can do this by iterating through the consumed codepoints and summing their `char_count`. But note: `translate_codepoint` already wrote the appropriate code units to `out`. So we just need to advance `pos` by the total code units corresponding to the consumed codepoints. We'll compute that by iterating from `pos` and counting code units until we've covered `consumed` codepoints.
