The bug is in the `translate` method's loop logic. When `translate_codepoint` returns 0, we correctly handle the current codepoint by appending its UTF-16 code units and advancing `pos` by the number of code units (1 or 2). However, when `translate_codepoint` returns a non-zero `consumed` (number of codepoints consumed), the loop incorrectly advances `pos` by iterating over each consumed codepoint and adding the `char_count` for each. This double-counts code units because `char_count` returns the number of UTF-16 code units per codepoint, but `pos` is already an index into the `code_units` list (which is in code units, not codepoints). Instead, we should advance `pos` by the total number of code units corresponding to the consumed codepoints. We can compute this by summing `char_count(code_point_at(code_units, pos + offset))` for each consumed codepoint, but a simpler fix is to directly advance `pos` by the number of code units that were actually consumed from the `code_units` list. Since `translate_codepoint` is expected to understand codepoints and may have consumed a surrogate pair, we can't assume each codepoint corresponds to one code unit. Instead, we can compute the advance by iterating over the consumed codepoints and accumulating their `char_count`. However, note that the `code_units` list is in UTF-16 code units, and the `index` parameter to `translate_codepoint` is an index into that list. If `translate_codepoint` consumes a surrogate pair, it should return 1 (one codepoint) but it actually consumed two code units from the input. The current loop incorrectly adds `char_count` for each consumed codepoint, but `pos` is already at the start of the surrogate pair. Adding `char_count` once for the single codepoint would advance by 2, which is correct. But the loop does a `for pt in range(consumed)`, and for each iteration it adds `char_count(code_point_at(code_units, pos))`. Since `pos` updates inside the loop, the second iteration would look at the wrong position. We need to compute the total code units consumed before updating `pos`. Let's refactor: when `consumed > 0`, compute `total_code_units = 0` and for each consumed codepoint, get the codepoint at `pos + total_code_units` and add its `char_count`. Then increment `pos` by `total_code_units`. This ensures we skip the correct number of code units in the input list.
