The error occurs in the while loop when `consumed` is greater than 0. The loop uses `char_count(code_point_at(code_units, pos))` to advance `pos`. The function `char_count` likely returns 1 for BMP characters and 2 for surrogate pairs (UTF-16 code units). However, the `for` loop iterates `pt in range(consumed)`, where `consumed` is the number of code units processed by `translate_codepoint`. Inside the loop, `pos` is incremented by `char_count(code_point_at(code_units, pos))` for each iteration. This can cause `pos` to exceed `length` because `char_count` may return 2 for a surrogate pair, but `consumed` might already account for the pair. For example, if `consumed` is 2 (a surrogate pair), the loop runs twice. The first iteration increments `pos` by 2 (if the pair is a high surrogate followed by a low surrogate), making `pos` jump past the low surrogate. The second iteration then calls `code_point_at(code_units, pos)` with an index that is out of bounds, causing the IndexError. The issue is that the loop double-counts the code units. The fix is to increment `pos` by `consumed` directly after the loop, or adjust the loop logic to not double-count.
