The bug is that when `consumed > 0`, the loop incorrectly uses `char_count(code_point_at(...))` to advance `pos`. The `translate_codepoint` method returns the number of code units consumed, not codepoints. The original code looped over `range(consumed)` and for each iteration added `char_count(...)`, which could cause index out of range. The fix is to simply add `consumed` to `pos` directly, because `consumed` already represents the number of code units processed. This matches the contract that `translate_codepoint` returns the count of code units consumed.
