The test fails because the `code_point_at` function incorrectly advances the index when handling a surrogate pair. The function returns a full Unicode code point for a surrogate pair, but the calling code in `translate_string` increments the index by 1 after each call. For a surrogate pair (two UTF-16 code units), the index should advance by 2, not 1. This causes an `IndexError` when the next iteration tries to access `code_units[index]` with an index that is now out of bounds, because the index only moved past the high surrogate, leaving the low surrogate unprocessed. The bug occurs in the loop that calls `code_point_at` for each position, but does not adjust the index for surrogate pairs.
