The bug is in the `translate` method's loop where `pos` can exceed `length` due to incorrect handling of surrogate pairs. Specifically, when a surrogate pair is consumed, the loop increments `pos` by the number of code units (2) but the `char_count` function returns 2 for a supplementary codepoint, which is correct. However, the `for pt in range(consumed):` loop uses `char_count(code_point_at(code_units, pos))` which may call `code_point_at` with an out-of-range index if `pos` is at the end of the list. The fix is to ensure `pos` is incremented correctly without calling `code_point_at` when it would be out of bounds. We can replace the loop with a direct increment of `pos` by `consumed` because `consumed` is already the number of codepoints, not code units. But the contract of `translate_codepoint` returns the number of codepoints consumed, not code units. So we need to convert codepoints to code units for incrementing. We'll compute the number of code units consumed by iterating through the codepoints and summing their `char_count`. This avoids the out-of-bounds access.
