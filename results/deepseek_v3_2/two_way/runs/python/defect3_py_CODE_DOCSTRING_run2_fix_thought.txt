The bug is in the `translate` method's loop where it advances `pos` after a translator consumes codepoints. The current `for` loop incorrectly uses `pos` as both the loop variable and the index for `code_point_at`, and it updates `pos` inside the loop while also using it to compute the next codepoint. This can lead to skipping or misreading code units, and potentially an infinite loop if `consumed` is zero or if the advancement logic fails. The fix is to advance `pos` by the total number of code units consumed, not by iterating over codepoints. Since `translate_codepoint` returns the number of codepoints consumed, we need to convert that to code units by scanning the input from the current position. We'll implement a helper to count code units for a given number of codepoints.
