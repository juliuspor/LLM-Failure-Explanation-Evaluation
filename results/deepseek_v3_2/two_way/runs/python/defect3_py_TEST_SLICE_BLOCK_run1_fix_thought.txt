The bug is in the `translate` method's index advancement logic. When `translate_codepoint` returns a non-zero `consumed` (number of codepoints consumed), the loop advances `pos` incorrectly. It uses `char_count(code_point_at(code_units, pos))` inside a loop, but `pos` is not updated between iterations, leading to double counting. Moreover, the original code incorrectly assumed that `consumed` corresponds to code units, but it's codepoints. The fix is to advance `pos` by the total code units for the consumed codepoints. In the `consumed == 0` branch, we already correctly advance by `char_count(codepoint)`. In the non-zero branch, we need to iterate over each consumed codepoint, get its codepoint at the current `pos`, and increment `pos` by its `char_count`. This ensures proper handling of surrogate pairs.
