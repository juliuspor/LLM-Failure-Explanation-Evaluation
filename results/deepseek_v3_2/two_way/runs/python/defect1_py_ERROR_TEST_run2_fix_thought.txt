The bug is that the method incorrectly rejects negative minutes when hours are zero. According to the rule, minutes must have the same sign as hours, but when hours are zero, minutes can be either positive or negative because zero is neither positive nor negative. The fix is to adjust the validation: allow minutes in range -59 to 59, and then enforce sign matching only when hours are non-zero. For hours > 0, minutes must be >= 0; for hours < 0, minutes must be <= 0. For hours == 0, minutes can be any value in [-59, 59]. Also, the calculation of total minutes must handle sign correctly. Currently, the code does: if hours_in_minutes < 0: minutes_offset = hours_in_minutes - minutes_offset else: minutes_offset = hours_in_minutes + minutes_offset. This is wrong because minutes_offset already has its own sign. The correct formula is total_minutes = hours_offset * 60 + minutes_offset. So we can simply compute total_minutes = hours_offset * 60 + minutes_offset. Then multiply by MILLIS_PER_MINUTE. This also simplifies the sign handling. We'll update the calculation accordingly.
