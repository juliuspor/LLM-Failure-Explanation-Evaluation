The bug is in the `translate` method's loop logic. When `translate_codepoint` returns a non-zero `consumed` (e.g., 2 for a surrogate pair), the loop incorrectly advances `pos` by calling `char_count` for each consumed codepoint, but `consumed` is already the number of *codepoints* consumed, not code units. The loop should advance `pos` by the number of UTF-16 code units corresponding to those codepoints. The fix is to compute the code units consumed directly from the `consumed` codepoints by iterating through them and summing their `char_count`. This ensures `pos` moves correctly through the code unit list.
