The bug had two main issues: 1) The condition `x == int(x)` could fail for large floats due to precision loss, leading to incorrect representation. 2) The while loop for computing exponent had a logic error: `exp` was not incremented inside the loop, causing `check_val` to be computed incorrectly and potentially infinite loop. Also, the loop could reduce mantissa to zero incorrectly. The fix: replace the integer check with a safe check for representable integers (within 2^53) using `math.trunc`. Ensure the while loop increments `exp` correctly and only proceeds when the reduced mantissa times 10^(exp+1) equals the original value. Added early return after integer path to avoid fallthrough.
