The bug is in the `translate` method's loop when handling consumed codepoints. After `translate_codepoint` returns a non-zero `consumed` (number of codepoints consumed), the loop incorrectly advances `pos` by summing `char_count` for each codepoint, but it recalculates the codepoint at the current `pos` each iteration, which can lead to misalignment with surrogate pairs. Instead, we should advance `pos` by the number of UTF-16 code units corresponding to the consumed codepoints. We can compute this by iterating through the consumed codepoints and adding the `char_count` for each, but we must ensure we read the codepoints sequentially from the `code_units` list. The fix is to track a temporary position while processing consumed codepoints, advancing by the `char_count` of each codepoint at that temporary position.
