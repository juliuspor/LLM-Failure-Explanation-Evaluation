The bug is in the `translate` method's loop when handling consumed codepoints. After `translate_codepoint` returns a non-zero `consumed`, the loop incorrectly advances `pos` by summing `char_count` for each consumed codepoint, but `code_point_at` is called with the same `pos` each iteration, leading to an infinite loop and eventually an out-of-bounds index. Instead, we should advance `pos` by the total number of code units consumed, not by counting codepoints. We can compute the total code units consumed by iterating through the consumed codepoints and summing their `char_count`. However, a simpler fix is to let `translate_codepoint` return the number of *code units* consumed, not codepoints, because the input is a list of code units. But the existing contract expects codepoints. To keep the contract, we can compute the total code units consumed by iterating through the consumed codepoints, but we must update `pos` correctly. The fix: after a non-zero `consumed`, compute the total code units consumed by iterating through the first `consumed` codepoints starting at `pos`, summing their `char_count`, and then add that total to `pos`.
