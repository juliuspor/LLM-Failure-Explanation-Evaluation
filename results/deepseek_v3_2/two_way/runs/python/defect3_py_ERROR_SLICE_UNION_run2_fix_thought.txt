The bug was in the loop logic: after a translator consumed codepoints, the code incorrectly used `char_count(code_point_at(...))` to advance `pos`. This could skip too many code units because `consumed` already represents the number of codepoints consumed, not code units. The fix changes the advancement to `pos += consumed`, ensuring proper indexing. Also, in the zero-consumption branch, `pos` should advance by `char_count(codepoint)` (which matches the number of code units added) instead of `len(c)` (which is the same but less clear).
