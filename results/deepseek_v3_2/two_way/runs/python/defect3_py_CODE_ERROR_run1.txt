The function `char_count` likely returns 1 for BMP characters and 2 for surrogate pairs. The issue is that `pos` is updated inside the `for` loop, but `code_point_at(code_units, pos)` is called with the updated `pos` each iteration, which may cause an index out of range. Specifically, if `consumed` is 2 (a surrogate pair), the first iteration updates `pos` by 2 (since `char_count` returns 2 for a surrogate pair), making `pos` equal to or exceed `length`. Then the second iteration calls `code_point_at(code_units, pos)` with an out-of-range index. The fix is to compute the advance per code unit outside the loop or adjust the loop logic.
