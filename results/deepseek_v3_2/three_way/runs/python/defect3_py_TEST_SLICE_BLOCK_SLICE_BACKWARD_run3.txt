The test fails because the `code_point_at` function incorrectly accesses the `code_units` list. The function is called with a string index (position in the original string), but it uses that index to look up a value in the `code_units` list, which is a list of UTF-16 code units. For characters that are surrogate pairs (like emoji), the string index and the code unit index differ. For example, the emoji \U0001F630 is one character in the string (index 0) but corresponds to two code units in the list (indices 0 and 1). When `code_point_at` is called with index=0, it checks if 0 >= len(code_units) (which is 2), which is false, so it returns the code unit at index 0 (the high surrogate). However, the calling code in `translate_string` then increments the string index by 1, but the loop also increments the position in the code units list by the number of code units consumed (2). This mismatch causes an out-of-sync state. Later, when processing the next character 'A' at string index 1, `code_point_at` is called with index=1, but the code units list only has indices 0 and 1 (the two surrogates). Since index=1 is valid, it returns the low surrogate (incorrectly). The loop then tries to advance the code unit position by 1 (since it thinks it consumed one code point), but the position was already at 2 after the emoji, leading to an IndexError when accessing code_units[2] (line 35). The root cause is that `code_point_at` uses the string index as an index into the code units list, which is wrong. It should use the code unit position that is maintained separately in the translation loop.
