The code fails because the `code_point_at` function uses `len(code_units)` to check bounds, but `code_units` is a local variable inside `to_utf16_code_units`. The `code_point_at` function is called with a string `s` and an index `i`, but it does not convert `s` to code units. Instead, it incorrectly uses `code_units` from the outer function's scope, which is empty or from a previous call. This causes an IndexError when `index` is out of range of `code_units`. Specifically, in `test_translate_surrogate_pair_with_suffix`, when processing the emoji (surrogate pair), `code_point_at` is called with the string and index, but `code_units` is not the UTF-16 representation of that string. The bug is in `code_point_at`: it should convert the input string to UTF-16 code units each time, not rely on an outer variable.
