The bug is in the loop that advances `pos` after a non-zero consumption. The original code incorrectly used `char_count(code_point_at(code_units, pos))` inside a loop over `consumed` (codepoints), but it always evaluated at the same `pos` each iteration, leading to wrong advancement. Moreover, the index update after zero consumption also had a subtle issue: it used `len(c)` which is the same as `char_count(codepoint)`, but the comment said 'codepoints consumed' which is misleading. Actually, the fix is to correctly compute the total number of code units consumed for the given number of codepoints. We need to iterate over the consumed codepoints, each time getting the codepoint at the current position (which advances by the code units of previous codepoints) and add its char count. This ensures proper handling of surrogate pairs. Also, the zero-consumption branch already correctly advances by `char_count(codepoint)`. So we only need to fix the non-zero branch.
