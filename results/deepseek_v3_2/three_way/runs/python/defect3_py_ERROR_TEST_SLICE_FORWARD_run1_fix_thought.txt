The bug occurs because the `translate` method incorrectly advances the index after a surrogate pair is processed. When `translate_codepoint` returns 2 (for a surrogate pair), the loop at lines 120-121 increments `pos` by `char_count(code_point_at(code_units, pos))` for each consumed codepoint. However, `char_count` returns 2 for a supplementary codepoint, but `pos` is already at the start of the surrogate pair. The inner loop adds 2 for each codepoint, causing an overrun. The fix is to directly advance `pos` by the number of code units consumed, not by codepoint counts. Since `translate_codepoint` returns the number of codepoints consumed, we need to convert that to code units. We can do this by summing `char_count(code_point_at(code_units, pos + offset))` for each consumed codepoint, where offset tracks the code units already counted. Alternatively, we can compute the total code units consumed by iterating through the codepoints. The simplest fix is to compute the total code units consumed using a helper that counts code units for each codepoint sequentially, then add that to `pos`.
