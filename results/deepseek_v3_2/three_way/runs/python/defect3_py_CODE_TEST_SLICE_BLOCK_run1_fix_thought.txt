The bug is in the `for` loop that advances `pos` when `consumed > 0`. It incorrectly uses `char_count(code_point_at(...))` which returns the number of UTF-16 code units for the codepoint at the current `pos`. This double-counts surrogate pairs because `pos` is updated inside the loop, but `code_point_at` is called again with the new `pos` (which may now point to the second half of a surrogate pair). Instead, we should compute the total number of UTF-16 code units consumed by the translator and advance `pos` by that amount directly. We can do this by tracking the initial `pos` and counting how many code units were appended to `out` by `translate_codepoint` (or by other means). However, the contract of `translate_codepoint` returns the number of *codepoints* consumed, not code units. So we need to convert consumed codepoints to code units by iterating through the input code units from the current position. The fix: when `consumed > 0`, we iterate through the next `consumed` codepoints in the input, summing their `char_count` to get total code units consumed, then add that to `pos`.
