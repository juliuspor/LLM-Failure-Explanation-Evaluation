The bug is that the validation incorrectly rejects negative minutes when hours are zero. The original validation allowed minutes only between 0 and 59, but the test case expects (0, -15) to be valid, producing offset -15 minutes. The fix is to expand the allowed range for minutes_offset to -59..59, and compute total minutes as hours_offset*60 + minutes_offset. This allows negative minutes when hours are zero, and also handles mixed signs correctly (e.g., -1, 30 => -60+30 = -30 minutes). The validation for hours remains -23..23. The total offset will be computed and passed to for_offset_millis, which will handle the range check. Also, we need to adjust the sign handling in the calculation: previously there was a conditional that adjusted minutes_offset based on sign of hours_in_minutes, but that was incorrect. We should simply add hours_in_minutes and minutes_offset. The conditional was causing errors for negative minutes. Remove that conditional and compute total_minutes directly.
