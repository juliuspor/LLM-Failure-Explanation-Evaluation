The bug is in the `translate` method's loop when handling consumed codepoints. After `translate_codepoint` returns a non-zero `consumed` (number of codepoints consumed), the loop incorrectly advances `pos` by iterating over each consumed codepoint and adding the UTF-16 length of the codepoint at the current `pos`. However, `pos` is updated inside the loop, causing the `code_point_at` call to use an updated `pos` for each iteration, which can lead to misalignment with surrogate pairs. Instead, we should compute the total code units consumed directly from the original `code_units` list, using the `consumed` codepoints count. The fix is to compute the total code units to advance by summing the UTF-16 lengths of each consumed codepoint, without modifying `pos` until after the loop.
