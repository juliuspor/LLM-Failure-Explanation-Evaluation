The original `to_locale` method had a rigid length-based check that failed for locale strings like "fr__POSIX" because it expected exactly one underscore between language and country. The fix replaces the manual character-by-character parsing with a split-based approach that handles multiple underscores and empty parts correctly. It now: 1) splits the string by underscores; 2) validates the language code (2 or 3 lowercase letters); 3) determines if the second part is a country code (2 uppercase letters) or a variant; 4) handles empty country parts (double underscore) as per POSIX format. This makes the method more robust and compliant with common locale string conventions.
