The bug is in the `translate` method's handling of the `consumed` return value from `translate_codepoint`. The original code incorrectly advanced `pos` by iterating over `range(consumed)` and calling `char_count(code_point_at(code_units, pos))` each time, but `pos` is updated inside the loop, leading to potential out-of-bounds access if `consumed` is greater than the number of remaining code units. Moreover, the logic for advancing `pos` when `consumed == 0` used `len(c)` which is correct but inconsistent with the other branch. The fix ensures that `pos` is advanced by the number of code units for each consumed codepoint, using `char_count` on the codepoint at the current position before moving forward. This avoids index errors and correctly handles surrogate pairs.
