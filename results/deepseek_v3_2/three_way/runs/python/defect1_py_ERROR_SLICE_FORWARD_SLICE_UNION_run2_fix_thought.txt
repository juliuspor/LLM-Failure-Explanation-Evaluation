The bug is that the validation of minutes_offset is too strict. It should allow negative values (down to -59) because minutes_offset can be negative when representing a negative offset component (e.g., -05:30). The fix changes the validation range from 0..59 to -59..59. This matches the later logic that adds or subtracts minutes_offset based on the sign of hours_offset. The total minutes offset will be computed correctly, and any out-of-range total offset will be caught by for_offset_millis.
