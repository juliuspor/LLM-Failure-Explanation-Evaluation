The original to_locale method had a bug: it assumed that a locale string with length > 5 must have a country code at positions 3 and 4 (0-indexed). This fails for strings like "fr__POSIX" where the country is empty (double underscore) and variant follows. The fix is to rewrite the method using a more robust parsing approach: split by underscore and handle the cases: 1 part (language only), 2 parts (language+country or language+variant), 3 parts (language+country+variant). This matches Java's Locale semantics and handles the double underscore case correctly.
