The fix is to restructure the logic to handle the case where the third character (index 3) is an underscore, indicating an empty country code, and the variant starts after that. We also need to validate the language part and ensure proper lengths. The updated method first validates the language code (two lowercase letters). If length is 2, return language only. If longer, check for underscore at position 2. Then, if the next character (index 3) is also an underscore, treat as empty country with variant (if any). Otherwise, check for a two-letter uppercase country code. If present, handle with or without variant. This matches Java's Locale behavior for strings like "fr__POSIX".
