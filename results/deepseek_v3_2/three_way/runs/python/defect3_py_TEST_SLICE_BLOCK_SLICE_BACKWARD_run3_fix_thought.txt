The bug is that the loop incorrectly advances the position `pos` after a translator consumes codepoints. The original code used `for pt in range(consumed): pos += char_count(code_point_at(code_units, pos))`. However, `pos` is an index into the `code_units` list (UTF-16 code units), not a string index. The `translate_codepoint` method returns the number of codepoints consumed, not code units. To advance `pos` correctly, we need to convert the consumed codepoints into the corresponding number of code units. The fix is to iterate over the consumed codepoints, get each codepoint at the current `pos` (which is in code units), and increment `pos` by the number of code units for that codepoint (1 or 2). This ensures `pos` stays aligned with the code units list. Also, the `code_point_at` function is already designed to work with code unit indices, so it's correct as is. The fix modifies the loop after `consumed != 0` to properly advance `pos`.
