The error occurs in the while loop of the translate method. The variable `consumed` is the number of code units processed by `translate_codepoint`. In the `for` loop at the end, `pt` iterates `consumed` times, and each iteration calls `char_count(code_point_at(code_units, pos))`. The `code_point_at` function is called with the current `pos`, but `pos` is not updated inside the loop. This means `code_point_at` is repeatedly called with the same `pos` value, which may be invalid after the first call if `consumed > 1`. Specifically, if `consumed` is 2 (meaning a surrogate pair was processed), the first iteration reads the high surrogate at `pos`, and the second iteration tries to read at `pos` again, but `code_point_at` expects a valid index. Since the high surrogate is at index `pos` and the low surrogate is at `pos + 1`, after the first iteration the high surrogate has been 'consumed', but `pos` hasn't changed, so the second iteration may try to access `code_units[pos + 1]` (via the `if` check in `code_point_at` at line 39) if the high surrogate is valid, but if `consumed` is 2, the entire surrogate pair is already processed, and the second iteration should not read the low surrogate separately. The loop incorrectly assumes each consumed unit corresponds to a full codepoint, but `char_count` may return 2 for a surrogate pair, causing `pos` to be incremented incorrectly. The root cause is that the `for` loop updates `pos` multiple times based on `char_count`, but `pos` should only be incremented by `consumed` (the number of code units processed). The fix is to replace the `for` loop with `pos += consumed`.
