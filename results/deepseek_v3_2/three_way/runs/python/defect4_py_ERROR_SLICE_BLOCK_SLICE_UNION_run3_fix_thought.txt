The bug is that the `_update_bounds` method incorrectly updates the min/max indices when the new period does not beat the current extreme. The `else` clause (setting the index) should only run when the current index is -1 (uninitialized). For each metric (start, middle, end), we must keep the old index if the new period is not more extreme. The fix is to restructure each block: compute the current extreme value, compare with the new value, and update the index only if the new value is more extreme OR the index is -1. This ensures indices are only updated when a new extreme is found.
