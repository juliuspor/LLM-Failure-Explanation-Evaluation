The bug occurs because `translate` incorrectly increments `pos` by `len(c)` (which is the number of code units) instead of the number of codepoints consumed. When a surrogate pair is processed, `len(c)` is 2, but the number of codepoints consumed is 1. This mismatch causes `pos` to advance too far, leading to an out-of-bounds index in `code_point_at`. The fix is to use `char_count(codepoint)` to advance `pos` by the correct number of code units per codepoint, both when no translation occurs and within the loop for consumed codepoints.
