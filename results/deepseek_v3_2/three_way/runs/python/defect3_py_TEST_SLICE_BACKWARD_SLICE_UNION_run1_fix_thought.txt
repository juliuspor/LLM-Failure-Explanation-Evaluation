The bug was that `translate` used `pos` as an index into `code_units` (UTF-16 code units) but later called `code_point_at` with `pos` as the index, which expects a code unit index. However, the loop advancement logic incorrectly used `len(c)` (which is code units) instead of `char_count(codepoint)` (which correctly accounts for surrogate pairs). This mismatch could cause misalignment when handling surrogate pairs. The fix changes `pos += len(c)` to `pos += char_count(codepoint)` to ensure `pos` always advances by the number of code units corresponding to the consumed codepoint, maintaining consistency with the code unit list.
