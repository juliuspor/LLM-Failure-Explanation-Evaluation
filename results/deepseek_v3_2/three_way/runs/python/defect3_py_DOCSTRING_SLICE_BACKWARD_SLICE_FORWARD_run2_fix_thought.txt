The bug occurs because when a translator returns a consumed count greater than 0, the loop that advances the position may call code_point_at with an index that is out of bounds. This can happen if the translator incorrectly returns the number of code units instead of codepoints (as in CsvEscaper for surrogate pairs). To make the translate method robust, we add a bounds check inside the loop that advances pos, breaking if pos reaches or exceeds the length of code_units. This prevents the IndexError from code_point_at.
