The bug is in the while loop's advancement when consumed > 0. The for loop uses char_count(code_point_at(...)) which re-evaluates at the updated pos each iteration, leading to incorrect advancement. Instead, we should compute the total advance once based on the consumed codepoints. Since translate_codepoint returns the number of codepoints consumed, we can advance pos by iterating over those codepoints and adding their charCounts, but we must compute from the original position. The fix is to compute advance by iterating from 0 to consumed-1, using the original code_units list and a temporary index that we update correctly.
