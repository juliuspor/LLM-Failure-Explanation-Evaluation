The bug was in the for-loop that advanced `pos` incorrectly when `consumed` > 0. The loop used `code_point_at(code_units, pos)` inside each iteration, but `pos` was updated within the same iteration, causing out-of-bounds access. The fix is to compute the codepoint at the current `pos` before updating it, then increment `pos` by the correct number of code units for that codepoint. This ensures `pos` moves correctly through the code units list, matching the number of consumed code points.
