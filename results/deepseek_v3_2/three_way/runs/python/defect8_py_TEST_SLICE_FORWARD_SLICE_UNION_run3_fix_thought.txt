The bug occurs because the original method incorrectly assumes a country code is always present when length > 2 and the third character is '_'. It fails for the double underscore format (e.g., 'fr__POSIX') where the country is empty. The fix restructures the logic: after validating the language and the first underscore, check if the next character is another underscore (indicating empty country). If so, parse language and variant directly. Otherwise, validate the country code as uppercase letters and proceed with the original parsing for language, country, and optional variant. This handles both standard formats (e.g., 'en_GB') and the double underscore format correctly.
