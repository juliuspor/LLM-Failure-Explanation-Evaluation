The bug is in the while loop's advancement of `pos` after a non-zero `consumed`. The original code incorrectly used a for-loop to increment `pos` by `char_count(code_point_at(...))` for each consumed codepoint, but `consumed` is already the number of code units processed (as returned by `translate_codepoint`). This mismatch could cause `pos` to exceed the length of `code_units`. The fix is to simply advance `pos` by `consumed` directly, because `translate_codepoint` returns the number of code units it consumed (as per its contract, it handles surrogate pairs correctly). Also, in the zero-consumed case, we already advance by `char_count(codepoint)`, which is correct. The fixed code removes the erroneous for-loop and replaces it with `pos += consumed`.
