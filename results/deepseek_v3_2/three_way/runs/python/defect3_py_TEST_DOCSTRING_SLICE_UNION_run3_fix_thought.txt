The bug was in the loop that advances the position after a translator consumes codepoints. The original code incorrectly used `pos += len(c)` for the zero-consumption case and a separate loop for the non-zero case, but both should advance based on the actual codepoint count. The fix ensures that in both cases, `pos` is incremented by the number of UTF-16 code units (via `char_count`) for each consumed codepoint, correctly handling surrogate pairs.
