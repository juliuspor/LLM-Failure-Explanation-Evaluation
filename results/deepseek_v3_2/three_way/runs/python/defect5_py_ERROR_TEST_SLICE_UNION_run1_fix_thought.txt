The bug occurs when both array and element are None, and expected_type is provided. The original code sets inferred_type to object in this case, which later triggers a TypeError when expected_type is not object. The fix is to prioritize expected_type when both inputs are None: if expected_type is provided, use it as inferred_type; otherwise fall back to object. This ensures the type check passes when expected_type is provided (e.g., str) and the inferred_type matches, preventing the spurious ClassCastException.
