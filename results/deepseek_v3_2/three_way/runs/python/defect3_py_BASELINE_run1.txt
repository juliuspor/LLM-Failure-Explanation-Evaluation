The IndexError occurs in the while loop of translate() when processing a surrogate pair. The bug is in the line `pos += char_count(code_point_at(code_units, pos))` inside the for loop. After `translate_codepoint` returns `consumed > 0`, the for loop iterates `consumed` times. In each iteration, it calls `code_point_at(code_units, pos)` to get a codepoint, then `char_count` to determine how many UTF-16 code units that codepoint occupies (1 for BMP, 2 for surrogate pair). However, `code_point_at` itself may read *two* code units (index `pos` and `pos+1`) if `pos` points to a high surrogate. The function `char_count` then returns 2. So `pos` increases by 2. But the for loop then increments `pt` and repeats, calling `code_point_at(code_units, pos)` again, where `pos` is now already advanced past the high surrogate. If `consumed` is 2 (meaning the translator handled two code units), the second iteration will call `code_point_at` with `pos` pointing to the low surrogate (index `pos+1` of the original pair). `code_point_at` will see a low surrogate without a preceding high surrogate (since `pos` is now at the low surrogate), treat it as a standalone code unit, and return that low surrogate value. Then `char_count` returns 1, advancing `pos` by 1. This double counting leads to `pos` advancing too far, eventually exceeding `length` and causing an IndexError in `code_point_at` when checking bounds. The fix is to advance `pos` by `consumed` directly after the translator succeeds, not by re-evaluating each codepoint.
