The bug is that the original to_locale method assumes a strict format with fixed positions for language, country, and variant, and does not handle the case where the country is empty but a variant is present (i.e., language__variant). The fix is to change the parsing logic to split the string by underscores and then examine the parts. This allows us to detect when the second part is not a valid country code (two uppercase letters) and treat it as part of the variant instead, with an empty country. This matches the behavior of Java's Locale.forLanguageTag and similar utilities. The new implementation also validates the language code and country code (if present) for correct casing and length.
