The bug is in the `translate` method's loop where it processes consumed codepoints. When `translate_codepoint` returns a non-zero `consumed` (number of codepoints consumed), the loop incorrectly advances `pos` by iterating over each codepoint and adding `char_count` for each. This double-counts because `char_count` returns the number of UTF-16 code units per codepoint, but `pos` is an index into `code_units` (which are UTF-16 code units). Instead, we should advance `pos` by the total number of code units corresponding to the consumed codepoints. We can compute this by summing `char_count` for each consumed codepoint, but we must be careful to get the codepoint at the correct position each time. However, a simpler fix is to directly advance `pos` by the number of code units that were actually consumed by `translate_codepoint`? Wait: `consumed` is the number of codepoints consumed, not code units. The contract is that `translate_cepoint` returns the number of codepoints consumed. The caller must then advance the index by the number of code units corresponding to those codepoints. The original loop does that but incorrectly because it uses `pos` inside the loop while updating `pos`. Let's rewrite the loop to compute the total code units to advance before updating `pos`. We'll compute `advance = 0`, then for each consumed codepoint, get the codepoint at `pos + advance` and add its `char_count` to `advance`. Then add `advance` to `pos`. This ensures we skip the correct number of code units.
