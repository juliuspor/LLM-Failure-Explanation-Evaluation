The bug is in the loop that advances the position after a translator consumes codepoints. The original code used `code_point_at(code_units, pos)` inside the for‑loop but did not update `pos` correctly for surrogate pairs: it called `char_count` on the same codepoint repeatedly, causing the index to jump by an incorrect number of code units. The fix ensures that each iteration of the for‑loop reads the codepoint at the current position and then advances by the correct number of UTF‑16 code units (1 for BMP, 2 for a surrogate pair). This guarantees that `pos` always points to the start of a valid codepoint and never lands on a low surrogate, preventing IndexError and incorrect processing.
