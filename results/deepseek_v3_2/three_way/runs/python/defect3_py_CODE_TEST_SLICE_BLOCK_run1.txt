The code fails because the loop incorrectly advances `pos` when `consumed > 0`. In the `while` loop, after `consumed` is returned from `translate_codepoint`, the code enters a `for` loop that iterates `pt in range(consumed)`. Inside this loop, `pos` is incremented by `char_count(code_point_at(code_units, pos))`. However, `code_point_at` is called with the current `pos`, which may be at the start of a surrogate pair. The `char_count` function likely returns 2 for a surrogate pair (since it consists of two UTF-16 code units). This means `pos` is incremented by 2 for each iteration of the `for` loop. If `consumed` is 1 (meaning one codepoint was processed), the loop runs once and increments `pos` by 2, skipping over the second half of the surrogate pair. This leads to an `IndexError` when `code_point_at` is called later with an index out of bounds (as seen in the test failure). The issue is that the `for` loop is meant to advance `pos` by the number of codepoints consumed, but it incorrectly uses `char_count` on the same `pos` each iteration, causing double counting for surrogate pairs. The correct behavior should advance `pos` by the number of UTF-16 code units consumed, not by the char count of the codepoint at the current position repeatedly.
