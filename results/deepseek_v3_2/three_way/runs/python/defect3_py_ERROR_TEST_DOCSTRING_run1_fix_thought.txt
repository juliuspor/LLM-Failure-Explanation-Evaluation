The bug is in the `translate` method's loop where it advances the index after a surrogate pair. When `translate_codepoint` returns 0, the method uses `code_point_at` to get the codepoint and `to_chars` to write it, but then advances `pos` by `len(c)`, which is correct (2 for surrogate pair). However, when `translate_codepoint` returns a non-zero `consumed` (which is the number of codepoints consumed, not code units), the loop advances `pos` incorrectly. It uses `char_count(code_point_at(code_units, pos))` for each consumed codepoint, but `pos` is updated inside the loop, causing an out-of-range error because `code_point_at` may be called with an index pointing to the second half of a surrogate pair. The fix is to compute the total code units consumed by summing `char_count` for each codepoint, using a temporary index.
