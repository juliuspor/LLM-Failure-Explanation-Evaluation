The bug occurs when both `array` and `element` are None, and `expected_type` is provided (e.g., str). The code sets `inferred_type` to `object`, then raises a TypeError because `inferred_type == object` and `expected_type != object`. This is incorrect because there's no actual type conflict when the list is being created from scratch with a None element. The fix is to only raise the TypeError when `array` is not None, indicating an existing list that might have a type mismatch. If `array` is None, we are creating a new list, so no cast conflict exists. Therefore, we add a condition to check if `array` is not None before raising the error.
