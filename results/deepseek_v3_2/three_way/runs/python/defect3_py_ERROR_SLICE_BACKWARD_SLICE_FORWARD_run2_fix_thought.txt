The bug occurs because the while loop in `translate` uses `pos` to index into `code_units`, but after consuming a surrogate pair via `translate_codepoint`, the loop increments `pos` by the number of codepoints consumed, not by the number of code units. However, the `for pt in range(consumed):` loop incorrectly uses `char_count(code_point_at(...))` to advance `pos`. This can cause `pos` to exceed the length of `code_units` because `char_count` returns 1 or 2, but `code_point_at` may read ahead and cause an index error if `pos` is near the end. The fix is to compute the total code units consumed directly from the `consumed` codepoints, by iterating and summing their `char_count`. Also, ensure `pos` never exceeds `length`.
