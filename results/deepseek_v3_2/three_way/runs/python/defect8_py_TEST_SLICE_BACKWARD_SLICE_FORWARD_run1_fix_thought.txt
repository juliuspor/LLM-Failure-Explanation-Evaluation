The bug is that to_locale incorrectly assumes that a locale string with length > 5 must have a two-letter uppercase country code at positions 3 and 4. However, the format language__variant (double underscore) is valid and indicates an empty country. The fix is to check for a double underscore pattern after the language. If the character at index 2 is '_' and index 3 is also '_', then the country is empty and the variant starts at index 4. We also need to adjust validation for the case where there is a country code (single underscore after language).
