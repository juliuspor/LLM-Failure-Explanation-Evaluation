The bug is in the loop where consumed != 0. The original code attempted to iterate over consumed codepoints and for each, call char_count on the codepoint at the current pos, but pos is not updated within the loop, leading to repeated calls to code_point_at at the same position, which could cause an index out of range if the codepoint is a surrogate pair (2 code units) and the list length is insufficient. Moreover, the loop incorrectly assumes that each iteration consumes exactly one codepoint, but the translator may have consumed multiple codepoints at once. The fix is to compute the total code units consumed by scanning forward from pos, counting codepoints until we have accounted for all consumed codepoints, then update pos accordingly.
