The bug was that the original formula used the unclamped `value` to compute `g`, which could produce out-of-range results even after clamping because the clamping was only applied to `v` but not used in the calculation. Additionally, the error message was generic. The fix uses the clamped value `v` in the formula, ensuring the result stays within 0..255 (assuming bounds are valid). Also, the error message now includes the actual value of `g` for debugging. The division by zero is already prevented by the constructor check that lower_bound < upper_bound.
