The `for` loop then adds `char_count` (which is 2 for that surrogate pair) to `pos`, but the next iteration of the `for` loop calls `code_point_at` again at the new `pos`, which is now after the low surrogate, so it reads the next code unit correctly. The `for` loop iterates over `consumed` codepoints, but `char_count` is called on the codepoint at the current `pos`, which may not align if the previous iteration advanced `pos` by more than one code unit. This can lead to misalignment and infinite loops or index errors. The root cause is that `pos` should advance by the number of UTF-16 code units consumed, not by the number of codepoints. The code incorrectly mixes codepoint and code unit counts.
