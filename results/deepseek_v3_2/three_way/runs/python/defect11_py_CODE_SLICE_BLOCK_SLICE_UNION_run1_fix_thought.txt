The bug occurs when `start >= end`, causing `gap = end - start` to be zero or negative, which later leads to `_next_int_java(rnd, gap)` raising a ValueError because `bound` must be positive. The fix is to validate that `start < end` when a character range is used (i.e., when `chars` is None and `start` and `end` are not both zero). This mirrors Java's behavior, which throws an IllegalArgumentException if `start >= end` in such cases. The validation should be placed in `_random_internal` after the logic that sets default `start` and `end` values.
