The bug occurs when `translate_codepoint` returns a consumed count that causes `pos` to advance incorrectly, potentially exceeding the length of `code_units`. Specifically, the loop `for pt in range(consumed):` increments `pos` by `char_count(code_point_at(code_units, pos))` each iteration, but if `consumed` is greater than the remaining codepoints, `code_point_at` may be called with an out-of-range index. The fix is to ensure `pos` never exceeds `length` by moving the `char_count` calculation outside the loop and using a while loop that checks bounds.
