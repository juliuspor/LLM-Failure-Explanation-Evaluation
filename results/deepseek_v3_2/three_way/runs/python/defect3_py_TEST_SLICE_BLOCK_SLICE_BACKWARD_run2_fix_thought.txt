The bug is in the index advancement logic inside the `translate` method. After a translator consumes a surrogate pair (2 code units), the loop incorrectly advances `pos` by the number of code points consumed, but each code point may be represented by 1 or 2 code units. The current code uses `char_count(code_point_at(...))` for each consumed code point, which is redundant and can cause an out-of-bounds access when `pos` is already at the end. Instead, we should advance `pos` by the number of code units corresponding to the consumed code points. We can compute this by iterating over the consumed code points and summing their `char_count`. However, a simpler fix is to directly use the `consumed` value returned by `translate_codepoint` as the number of code points, and then advance `pos` by the total code units for those code points. We'll compute the total code units by iterating from the current position and using `char_count` for each consumed code point until we've accounted for all consumed code points.
