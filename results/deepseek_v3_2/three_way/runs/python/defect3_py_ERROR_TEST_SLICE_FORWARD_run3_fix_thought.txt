The bug was in the `translate` method's index increment logic. After `translate_codepoint` returns a non-zero `consumed` (number of code units consumed), the method incorrectly added the number of code points (`char_count`) for each consumed code point, causing an index overshoot. The fix changes the increment to directly add `consumed` (which is already in code units) to `pos`. Also, in the `consumed == 0` branch, `pos` should increment by `char_count(codepoint)` instead of `len(c)` for consistency, though both are equal. This ensures correct iteration over UTF-16 code units.
