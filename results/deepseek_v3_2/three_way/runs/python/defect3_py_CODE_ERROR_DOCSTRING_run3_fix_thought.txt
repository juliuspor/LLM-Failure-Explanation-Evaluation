The bug was in the while loop's update logic for `pos`. When `consumed == 0`, the code incorrectly used `len(c)` from `to_chars(codepoint)`, which returns the number of UTF-16 code units for that codepoint, but that may not match the number of code units at `pos` if the input already is in UTF-16 code units. Instead, we should directly copy the code units from `code_units` and advance `pos` by 1 or 2 based on whether we have a surrogate pair. When `consumed > 0`, the inner for-loop must advance `pos` by the char count of each consumed codepoint, but the previous code recalculated `code_point_at` each iteration with the updated `pos`, which could skip over surrogate pairs incorrectly. The fix ensures that `pos` is incremented by the correct number of code units in both branches, using the existing `char_count` and `code_point_at` functions to handle surrogate pairs properly. The `consumed == 0` branch now checks for a valid surrogate pair directly in `code_units` and advances accordingly.
