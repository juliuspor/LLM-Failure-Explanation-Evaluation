The bug occurs because the `random` function does not validate the `start` and `end` parameters before passing them to `_random_internal`. In `_random_internal`, when `chars` is `None`, `gap = end - start` is computed, and `_next_int_java(rnd, gap)` is called. If `start >= end`, `gap` becomes zero or negative, causing `_next_int_java` to raise a ValueError with 'bound must be positive'. The fix adds a validation check at the beginning of `random` to ensure `start < end`, raising a ValueError with a clear message that includes 'start' and 'end' values, matching the expected error format from the diagnosis.
