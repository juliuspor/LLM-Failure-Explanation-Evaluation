The bug occurs because the `translate` method incorrectly advances `pos` when a translator consumes codepoints. The loop uses `char_count(code_point_at(code_units, pos))` to increment `pos`, but `code_point_at` can raise an IndexError if `pos` is at the last code unit of a surrogate pair (e.g., index 2 in a 3-unit list where index 2 is a low surrogate). The fix is to ensure `pos` is incremented by the number of code units consumed, not by the number of codepoints. We should track consumed code units directly, not via codepoint counts. We'll change the loop to use the `consumed` value (which is the number of codepoints consumed) and compute the corresponding code units by iterating through the codepoints and summing their `char_count`. This matches the contract that `translate_codepoint` returns codepoints consumed, not code units.
