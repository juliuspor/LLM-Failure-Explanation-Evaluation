The bug is that the contract of translate_codepoint is ambiguous: it returns number of codepoints consumed, but the implementation in CsvEscaper returns number of code units (2 for a surrogate pair). This causes misalignment in the advance loop. The fix is to clarify that translate_codepoint should return the number of UTF-16 code units consumed, matching Java's behavior. Then in translate, we advance pos by consumed directly. Also, for the non-translated case, we copy code units directly from the list using surrogate pair detection, avoiding codepoint conversion.
