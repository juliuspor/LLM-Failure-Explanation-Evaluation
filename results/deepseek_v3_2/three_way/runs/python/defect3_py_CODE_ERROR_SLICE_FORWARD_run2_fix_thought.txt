The bug was in the loop that advanced the position after a successful translation. The original code incorrectly used a for-loop to increment pos by char counts for each consumed code point, which could lead to index out of range. The fix is to simply add 'consumed' (which is the number of code units processed) directly to pos, as the contract of translate_codepoint is to return the number of code units consumed. This matches the behavior when consumed == 0, where pos is incremented by char_count(codepoint) (which is the number of code units for that codepoint).
