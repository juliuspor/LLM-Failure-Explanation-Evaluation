The code fails because the `to_utf16_code_units` function returns a list of UTF-16 code units (integers), but the `code_point_at` function incorrectly uses `len(code_units)` to check bounds. When a surrogate pair (like an emoji) is present, the string length in characters (code points) is less than the number of UTF-16 code units. For example, the string "\U0001F630A" has 2 characters (emoji + 'A') but 3 UTF-16 code units (high surrogate, low surrogate, 'A'). The `code_point_at` function expects an index in terms of code units, but the test passes a character index (from the string iteration). This mismatch causes an IndexError when trying to access index 2 in a list of 3 code units, because the index is derived from string length (2) which is out of range for the code units list (length 3). The bug is in `code_point_at` line 35-36: it should compare index to the length of the string's code points, not the code units list.
