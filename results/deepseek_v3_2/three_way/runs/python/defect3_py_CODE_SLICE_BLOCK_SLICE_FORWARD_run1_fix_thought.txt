The bug is that `pos` increments incorrectly when `consumed > 0`. The loop iterates over codepoints, but `pos` should advance by the number of UTF-16 code units consumed, not by the number of codepoints. The fix is to compute the total code units consumed by the translator and advance `pos` by that amount. When `consumed == 0`, we already advance by the code units of the current codepoint (via `len(c)`). When `consumed > 0`, we should advance by the code units corresponding to the `consumed` codepoints, which we can compute by iterating through the code units from the current position, counting codepoints until we've consumed the required number.
