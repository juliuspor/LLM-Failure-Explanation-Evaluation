The bug is in the loop that advances `pos` after a translator consumes characters. The loop incorrectly uses `char_count(code_point_at(...))` for each consumed code point, which can cause over-advancement when multiple code points are consumed. Instead, we should directly advance `pos` by the number of code units consumed, which can be calculated by summing the `char_count` of each consumed code point, but we must ensure we don't re-evaluate `code_point_at` on an already advanced position. The fix is to compute the total code units consumed by iterating over the consumed code points using a temporary index.
