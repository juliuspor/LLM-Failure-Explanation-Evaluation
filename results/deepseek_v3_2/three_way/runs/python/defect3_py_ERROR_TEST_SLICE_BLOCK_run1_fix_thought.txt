The bug is in the `translate` method's loop where it processes consumed codepoints. When `translate_codepoint` returns a non-zero `consumed` (number of codepoints consumed), the loop incorrectly advances `pos` by summing `char_count` for each codepoint, but `char_count` is called on the codepoint at the current `pos`, which may be a surrogate pair. This leads to double-counting because `char_count` returns 2 for a surrogate pair, but the loop iterates `consumed` times (which is 1 for a single codepoint like an emoji). So for an emoji (one codepoint, two code units), `consumed=1`, then the loop runs once, adds `char_count(emoji)=2` to `pos`, moving `pos` by 2 correctly. However, the loop variable `pt` is unused, and the `char_count` call is repeated each iteration, but the codepoint at `pos` changes after the first addition? Actually, `code_point_at(code_units, pos)` is called each iteration, but `pos` hasn't been updated yet. So for `consumed=1`, it's fine. But if `consumed>1`, the logic is flawed. The real issue is that `consumed` is the number of codepoints, not code units. The loop should advance `pos` by the total code units for those codepoints. The fix: compute total code units to advance directly, not by iterating. Also, ensure we don't call `code_point_at` with an index that might be a low surrogate (if previous codepoint was a surrogate pair). We'll compute advance by iterating through codepoints until we've consumed `consumed` codepoints, summing `char_count` for each.
