The bug is in the loop that advances `pos` when `consumed > 0`. The original code incorrectly uses `pos` inside the for‑loop while also updating it, but the real issue is that `consumed` is the number of *code points* consumed, and the loop must advance `pos` by the total number of UTF‑16 code units corresponding to those `consumed` code points. The fix is to compute the total advance correctly: iterate `consumed` times, each time adding the `char_count` of the code point at the current `pos`. This ensures proper handling of surrogate pairs (2 code units) and BMP characters (1 code unit).
