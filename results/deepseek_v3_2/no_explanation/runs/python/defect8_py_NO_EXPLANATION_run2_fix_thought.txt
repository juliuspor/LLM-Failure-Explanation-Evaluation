The original `to_locale` function had several bugs:
1. It only accepted locale strings of length 2, 5, or >=7, which is too restrictive. Locale strings can have language codes of 2 or 3 characters (ISO 639), and country codes can be 2 letters or 3 digits (UN M.49). Variants can be of any length.
2. The validation logic incorrectly checked character ranges using direct character comparisons, which would fail for 3-letter language codes or numeric country codes.
3. It didn't handle empty country parts correctly (e.g., "en__variant" where country is empty but variant exists).
4. It didn't validate that language codes are lowercase and country codes are uppercase (for alphabetic codes).

I rewrote the function to:
- Split the locale string by underscores into parts.
- Validate the number of parts (1-3).
- Validate language code: 2-3 letters, all lowercase.
- Validate country code: if present, either 2 letters (converted to uppercase) or 3 digits (for UN M.49).
- Allow empty country string (when there's a double underscore).
- Validate variant: if present, must be non-empty.
- Return a Locale object with appropriate casing.
This matches the Java implementation's behavior more accurately and handles more cases.
