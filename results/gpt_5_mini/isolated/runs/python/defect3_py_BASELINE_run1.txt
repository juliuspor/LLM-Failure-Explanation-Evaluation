The translate method advances `pos` incorrectly after a translator consumes a code point. When consumed != 0, the loop uses: for pt in range(consumed): pos += char_count(code_point_at(code_units, pos)) This calls code_point_at(code_units, pos) repeatedly while incrementing pos inside the loop, but code_point_at expects a valid index into code_units. On the first iteration it may read a surrogate pair and advance pos by 2 (char_count returns 2), so the next loop iteration calls code_point_at with the new pos which may be equal to length (out of range) or point past the buffer, causing IndexError: String index out of range. The root cause is using pos as both the moving index inside the for-loop and as the argument to code_point_at, so pos can move beyond bounds between iterations. The correct approach is to compute the code point(s) or their char counts without mutating pos each time, for example by reading code_point_at at the current starting index and advancing pos by the total number of UTF-16 code units consumed (sum of char_count for each code point consumed) or by iterating consumed times while calculating counts from a separate index variable. Specifically: at line with 'for pt in range(consumed): pos += char_count(code_point_at(code_units, pos))' the code should save the starting index (e.g., start = pos) and either compute increments from that or update a temporary index. This fixes the IndexError when a surrogate pair is followed by another character because you avoid calling code_point_at with an out-of-range index.
