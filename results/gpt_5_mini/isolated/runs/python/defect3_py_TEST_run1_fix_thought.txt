The bug is that translate increments pos by len(c) (number of code units of the translated output) when no translation occurs, but len(c) is length of produced UTF-16 units for the codepoint, which is correct. However the real bug is in the loop after consumed !=0: it increments pos by char_count(code_point_at(code_units, pos)) inside a for-loop that iterates 'consumed' times but calls code_point_at with same pos each iteration leading to repeated advancement by same amount; also when translator consumed >1 codepoints, we must advance by the sum of char_counts of each consumed codepoint. Fix by iterating consumed times, each time computing cp = code_point_at(code_units, pos) then pos += char_count(cp). Also ensure when consumed==0 path uses code_point_at then pos increments by char_count. Replace previous logic with correct advancement and bounds handling.
