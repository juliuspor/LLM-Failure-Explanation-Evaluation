The code fails because CsvEscaper.translate_string advances its index by the number of UTF-16 code units rather than by Python code points. In the translator loop the method to_utf16_code_units(code_point) (or equivalent) is used to detect surrogate pairs and then index i is incremented by 1 unconditionally (or by the number of code units incorrectly). This causes an off-by-one when a surrogate pair (a code point above U+FFFF) is encountered followed by another character: the loop increments i only by 1 instead of 2 UTF-16 units, so the next iteration re-reads the low surrogate (or skips the next actual character), leading to IndexError or incorrect translation. The failure occurs in CsvEscaper.translate_string, lines where code_point_at(input_str, i) is called and where i is incremented (e.g. i += 1). The root cause is mixing UTF-16 surrogate pair handling with Python's native UTF-32/UTF-8 str indexing semantics. In Python strings, indexing advances by code points; surrogate pair logic is unnecessary and makes the index advance wrong. To fix: remove manual surrogate handling and use Python's ord() and slicing by code points, or when using UTF-16 helpers, advance i by the number of code units returned (use len(units) or 2 for surrogate pairs). Specifically: (1) In code_point_at, return the actual code point and the number of consumed code units. (2) In translate_string loop, after getting code_point and consumed length, set i += consumed_length not i += 1. Or simpler, iterate over the string characters directly (for ch in input_str) and operate per code point. This resolves the IndexError and preserves characters following surrogate pairs.
