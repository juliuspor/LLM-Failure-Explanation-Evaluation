Code fails because pos advancement in the for-loop uses char_count(code_point_at(code_units, pos)) while pos is already being incremented each loop, and code_point_at expects index within code_units; after consuming a surrogate pair, char_count returns 2 for the high surrogate (or for the codepoint), causing pos to jump incorrectly and eventually become equal to length+1. Specifically: in translate(), when consumed > 0 the code runs "for pt in range(consumed): pos += char_count(code_point_at(code_units, pos))". On the first iteration pos points to the high surrogate (valid), char_count(...) returns 2, so pos increases by 2, skipping the low surrogate; on the next iteration the loop body executes again (because consumed is 1 or more), calls code_point_at with pos now equal to a value >= len(code_units) and then code_point_at raises IndexError("String index out of range: {index}"). The bug is here: the loop should advance pos by the number of UTF-16 code units consumed (1 for BMP or 2 for surrogate pair) exactly once per consumed codepoint, but calling code_point_at again after advancing off the end causes the out-of-range access. Also the inner use of char_count(code_point_at(...)) is redundant: code_point_at already reads both code units when forming a supplementary code point; instead the code should check if the current code unit is a high surrogate and if so add 2, otherwise add 1, or use the consumed value directly. In short: incorrect index advancement in the surrogate-pair handling loop at the final for-loop (pos += char_count(code_point_at(code_units, pos))) triggers code_point_at with an out-of-range index. Recommended fix: advance pos by 1 or 2 based on the code unit at the current pos without calling code_point_at after advancing, or replace the loop with pos += 1 if code unit is BMP else pos += 2, or use the number of code units consumed returned by translate_codepoint if it reports code units instead of codepoints.
