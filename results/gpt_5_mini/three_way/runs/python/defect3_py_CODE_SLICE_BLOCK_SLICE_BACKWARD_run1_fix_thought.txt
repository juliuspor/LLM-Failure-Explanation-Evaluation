The bug is caused by mixing code points and UTF-16 code unit counts. translate_codepoint implementations (like CsvEscaper) return number of code points consumed, but translate() treats consumed as count of code points and then advances pos by summing char_count(code_point_at(...)) which can call code_point_at beyond bounds. Simpler fix: treat consumed as number of code points; advance pos by iterating consumed times but ensure we check bounds before calling code_point_at. For each iteration, get codepoint at current pos, then add char_count(codepoint) to pos; but first check pos<length before calling code_point_at. Also ensure that when translate_codepoint returns >0, we do not attempt to call code_point_at if pos is at or beyond length; so add guard. Implement loop that for _ in range(consumed): if pos<length: cp = code_point_at(...); pos += char_count(cp) else: break
