The code advances the position incorrectly after a translator consumes a codepoint. In translate(), when consumed == 0 it handles a scalar by reading codepoint = code_point_at(code_units, pos), converting to chars c = to_chars(codepoint) and then does pos += len(c) — that correctly advances by 1 or 2 UTF-16 code units. But in the consumed > 0 case it does: for pt in range(consumed): pos += char_count(code_point_at(code_units, pos)). This is wrong because char_count(code_point_at(code_units, pos)) returns 1 for codepoints in BMP and 2 for supplementary codepoints; however code_point_at(code_units, pos) expects pos to be a valid index into code_units. On the first loop iteration it reads the code point at pos, then advances pos by 1 or 2. On the next iteration, if pos advanced by 2 (surrogate pair), code_point_at will be called with the new pos which now points to the low surrogate (out of range or invalid), causing code_point_at to raise IndexError or read a low surrogate as if it were a high surrogate. Specifically, for a consumed translator that claimed to consume a supplementary codepoint (consumed == 1) the loop still calls char_count using code_point_at at the same pos repeatedly causing incorrect advancement. In the failing test the input is a surrogate pair followed by 'A' (code units: high, low, 'A'). translate_codepoint returns consumed > 0 (translator handled the surrogate pair). The for-loop then iterates and on the second iteration (or when pos points to the low surrogate) code_point_at(code_units, pos) triggers the IndexError at slice block line 35-36 because pos indexes the low surrogate or goes past length. Root cause: the code assumes consumed counts UTF-16 code units or that calling code_point_at repeatedly at the moving pos is safe; instead it should advance pos by a fixed number of code units consumed (e.g., pos += consumed) or compute the char count from the original codepoint(s) before mutating pos. In short: pos is advanced by reading code_point_at at positions that are invalid after prior increments, causing IndexError when handling surrogate pairs.
