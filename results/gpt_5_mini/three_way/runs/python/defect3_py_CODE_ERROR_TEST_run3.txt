The code advances pos incorrectly in the for-loop that consumes 'consumed' codepoints. Specifically, inside:

    for pt in range(consumed):
        pos += char_count(code_point_at(code_units, pos))

it calls code_point_at(code_units, pos) after pos has been advanced by previous iterations. code_point_at expects pos to point at a valid code unit index; when 'pos' starts at the last code unit of a surrogate pair, char_count(code_point_at(...)) may be 2, and the loop then advances pos beyond length, causing the next call to code_point_at to use an out-of-range index and raise IndexError: String index out of range: 2. The root cause is using pos as both the iterator and the argument to code_point_at while incrementing it inside the loop. The intended behavior is to advance pos by the UTF-16 char count of each original codepoint consumed, not to recompute code_point_at at the already-advanced pos. This happens when translate_codepoint returned 1 (consumed a single codepoint represented by two code units) and the loop then steps through 'consumed' iterations but reads code_point_at with an invalid index.

Where it occurs: in translate(), at the for pt in range(consumed) loop after a non-zero 'consumed' return from translate_codepoint. Variables involved: pos, length, consumed, code_units. Line: pos += char_count(code_point_at(code_units, pos)).

Why: code_point_at(code_units, pos) is being called with pos values that may be equal to length or outside valid range because pos is updated inside the loop; the loop should instead compute the char count from the original codepoint(s) or advance by 1 or 2 safely without re-reading code_point_at at an advanced index.

Fix steps:
1) Replace the for-loop with a single advance using a stable value: if consumed == 1: pos += char_count(code_point_at(code_units, pos)) before any increment, or better, compute cp = code_point_at(code_units, pos) and then do for pt in range(consumed): pos += char_count(cp) if all consumed codepoints share the same size (they do not). Correct approach: increment pos by the number of UTF-16 code units consumed by each codepoint returned by translate_codepoint; track a temporary index tmp = pos and in each iteration call code_point_at(code_units, tmp) then tmp += char_count(...); after loop set pos = tmp. This ensures code_point_at is never called with an out-of-range pos.
2) Alternatively, have translate_codepoint return the number of UTF-16 code units consumed instead of codepoints.

These changes prevent calling code_point_at with an invalid index and fix the IndexError.
