The code fails because to_utf16_code_units builds a list of code units and then code_point_at (or the translator) uses an index against that list but compares the index against len(code_units) while index is in code points space. Specifically, in to_utf16_code_units (around lines shown) code_units is a list of UTF-16 code units; later, code_point_at checks “if index < 0 or index >= len(code_units): raise IndexError(...)”. When the translator iterates over the original Python str by code point positions but then uses that same index to access the UTF-16 code unit list, surrogate pairs (which become two UTF-16 units) cause the lengths to differ. Advancing the index by 1 past a surrogate pair ends up out-of-range for code_units and triggers IndexError. The root cause is mixing two different index domains: Python code point index vs UTF-16 code unit index. The failure occurs when translating a supplementary character (like \U0001F630) followed by another character (as in test_translate_surrogate_pair_with_suffix) because the translator advances by one code point but the UTF-16 list expects two units, so an access at that index is invalid. To fix: keep indices consistent—either iterate and index the original string (code points) and convert code points to UTF-16 units when appending, or operate directly on the UTF-16 code unit list and advance by the correct unit count for surrogates. Concretely: (1) In translate, use the string length and indexing for positions passed to translate_codepoint; (2) In code_point_at, accept a code point index or compute a code unit index by summing surrogate pairs; (3) When appending to out, append both units for a surrogate pair and advance the loop index by 1 for code points, not by 2. Ensure checks that compare index to len(code_units) use the same domain as index.
