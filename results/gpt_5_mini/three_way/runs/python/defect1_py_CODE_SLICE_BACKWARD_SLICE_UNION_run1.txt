The code fails because it unintentionally reassigns the input parameter minutes_offset to hold the total minutes (hours_in_minutes ± minutes_offset) and then passes that to FieldUtils.safe_multiply. When hours_offset is negative, the code does: minutes_offset = hours_in_minutes - minutes_offset (line ~383). For example, hours_offset = -1 and minutes_offset = 30 yields hours_in_minutes = -60 then minutes_offset = -60 - 30 = -90, producing -90 minutes (correct total -1:30) but this changes the semantics of the original minutes_offset variable. The real bug causing failures is that when hours_offset is negative, the subtraction should add the minutes (i.e. total = hours_in_minutes - minutes_offset is correct only if minutes_offset represents positive minutes to subtract), but more importantly the parameter validation allows hours_offset == 0 and minutes_offset nonzero earlier; if hours_offset is 0 the code returns UTC only when both are zero (line ~374). The actual runtime failure occurs because FieldUtils.safe_multiply can raise an ArithmeticError if minutes_offset is outside a safe range, and because minutes_offset was reused and can overflow or produce an incorrect sign leading to unexpected large absolute value. Specifically, reusing minutes_offset and computing minutes_offset = hours_in_minutes - minutes_offset (line 383) or + (line 385) leads to wrong total minutes when hours_in_minutes is negative and minutes_offset is positive, causing the multiply to overflow and raise an ArithmeticError, which is caught and translated to ValueError("Offset is too large") at line ~388. The fix is to compute a new variable total_minutes = hours_in_minutes + (1 if hours_in_minutes >=0 else -1) * minutes_offset or better: total_minutes = hours_in_minutes + (minutes_offset if hours_offset >=0 else -minutes_offset), then multiply total_minutes by MILLIS_PER_MINUTE and pass that to for_offset_millis. Also do not mutate the input minutes_offset.
