The max_middle_index variable ends up larger than expected because the update logic treats 'middle' period bounds incorrectly when new items are added. Specifically, in the method that updates cached min/max indices (docstring shows function around lines ~440), the code compares and updates _max_middle_index using the new period's index whenever the new period's middle timestamp is >= current middle, but it fails to restrict updates to periods within the current visible bounds or to correctly compute the middle timestamp for the incoming period. As a result, when adding an item with a middle timestamp that is equal or slightly greater than the current max, the code sets _max_middle_index = index (line where _max_middle_index is assigned). Because there is no check to ensure the updated index falls inside the active slice or to recompute middle consistently, later assertions expecting max_middle_index==1 fail and return 3. The root cause is an incorrect comparison/update path in the update_cached_min_max_indices routine: it uses the wrong timestamp field (start/end instead of middle) or incorrect relational operator, and it does not guard against updating when expanding bounds should affect min/max differently.
