The code fails because inferred_type is set incorrectly: when array is not None it uses type(array) (which returns 'list' class), not the element type; but more importantly, for the failing test both array and element are None, so inferred_type becomes object. Later the TypeError check treats expected_type=str and inferred_type==object so it raises a TypeError. The root cause: logic assumes using 'object' when both inputs are None is acceptable, but the caller expects a str list; raising the TypeError is correct per the current code but the test expects ArrayUtils.add(None, None, expected_type=str) to return [None] without error. The error occurs at the if expected_type is not None block where inferred_type == object triggers the raise; the check should instead allow creating a list whose component type is expected_type (str) when provided, or use expected_type to build the new list. Also new_list creation uses ArrayUtils._copy_list_grow1(array, inferred_type) so inferred_type should be expected_type when provided (especially when array is None). In summary: lines setting inferred_type and the conditional that raises TypeError (the expected_type handling) are wrong: when expected_type is provided and array and element are None, inferred_type should be expected_type and no TypeError should be raised.
