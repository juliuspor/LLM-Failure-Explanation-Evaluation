The bug arises from mixing character indices with UTF-16 code unit indices. translate() should advance `pos` by the number of UTF-16 code units consumed, not by len(c) which returns number of code units but the loop after consumed should increment correctly. The problematic part is the for-loop that advances pos using char_count(code_point_at(code_units, pos)) which uses pos already advanced; simpler fix: when consumed>0, increment pos by the total UTF-16 code units corresponding to the consumed code points. To do that, compute units = 0; for i in range(consumed): cp = code_point_at(code_units, pos + units); units += char_count(cp); then pos += units. Also ensure when consumed==0 we advance by char_count of current codepoint as before. Replace the existing loop with this calculation to avoid mis-indexing.
