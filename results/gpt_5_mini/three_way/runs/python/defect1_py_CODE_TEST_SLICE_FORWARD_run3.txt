The failure is caused by reusing and overwriting the input variable minutes_offset with a computed total minutes value, and using the wrong arithmetic when hours_offset is negative. Specifically, at lines 383–385 the code does: hours_in_minutes = hours_offset * 60; if hours_in_minutes < 0: minutes_offset = hours_in_minutes - minutes_offset else: minutes_offset = hours_in_minutes + minutes_offset. This mutates minutes_offset (losing the original sign information) and for negative hours subtracts minutes_offset instead of subtracting the absolute minutes. For example, calling for_offset_hours_minutes(-2, -15): hours_in_minutes becomes -120, then minutes_offset becomes -120 - (-15) = -105 (i.e. -1h45m) instead of the expected -135 minutes (i.e. -2h15m). Similarly for (-2, 15): minutes_offset becomes -120 - 15 = -135 which is correct, but the inconsistent handling means some combinations produce wrong totals. The bug is in lines 383–385: the code should compute a separate total_minutes = hours_in_minutes + (minutes_offset if hours_offset>=0 else -abs(minutes_offset)) or otherwise combine hours and minutes without overwriting minutes_offset or flipping signs. Also mutating the input minutes_offset makes later debugging confusing (variable name no longer holds minutes). The error therefore occurs during the minutes calculation and leads to incorrect offsets returned by for_offset_hours_minutes.
