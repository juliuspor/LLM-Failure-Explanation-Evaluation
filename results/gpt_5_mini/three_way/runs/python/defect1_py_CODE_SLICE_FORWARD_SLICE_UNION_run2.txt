The code fails because it reuses the parameter name minutes_offset to store the total offset in minutes, which overwrites the original minutes sign and leads to wrong arithmetic when hours_offset is negative. Specifically, at line 383/385 the code assigns minutes_offset = hours_in_minutes - minutes_offset for negative hours or + minutes for positive hours, but minutes_offset was the input minutes (0..59) and should not be repurposed for the total minutes variable. This causes incorrect values passed to FieldUtils.safe_multiply and can produce an ArithmeticError or wrong offset when hours_offset is negative (e.g., hours_offset=-1, minutes_offset=30 becomes -60 - 30 = -90 instead of -30). The correct approach is to compute a new variable total_minutes = hours_in_minutes +/- minutes_offset and keep the input minutes_offset unchanged. Also ensure using subtraction for negative hours: total_minutes = hours_in_minutes - minutes_offset and for non-negative hours: total_minutes = hours_in_minutes + minutes_offset, then multiply that by MILLIS_PER_MINUTE. The bug occurs in for_offset_hours_minutes around lines 383-386 where minutes_offset is overwritten and passed to safe_multiply.
