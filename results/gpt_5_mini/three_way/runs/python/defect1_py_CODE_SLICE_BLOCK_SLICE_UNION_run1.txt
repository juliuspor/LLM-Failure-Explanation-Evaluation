The code fails because minutes_offset is mutated and reused for both validation and later arithmetic, causing wrong sign handling and overflow. Specifically: after validating minutes_offset (lines 377-378), the code computes hours_in_minutes = hours_offset * 60 and then overwrites minutes_offset with either hours_in_minutes - minutes_offset (line 383) or hours_in_minutes + minutes_offset (line 385). This mutates the validated minutes value and mixes units (minutes_offset becomes total minutes), then passes that to FieldUtils.safe_multiply with MILLIS_PER_MINUTE (line 386). If hours_offset is negative, the subtraction on line 383 computes a value with incorrect sign (should add the negative hours to the minutes), causing the final offset to be wrong or to overflow. Also, validating minutes_offset only for 0..59 at lines 377-378 is correct for a separate minutes parameter but after mutation the variable name no longer represents the same quantity, making error messages and exceptions misleading. The root cause is reusing the minutes_offset variable for total minutes; use a new variable (e.g., total_minutes) and compute total_minutes = hours_in_minutes + sign_adjusted_minutes, then multiply. This preserves the original minutes_offset for validation and avoids sign/overflow bugs.
