The bug arises because when translate_codepoint consumes 0 and we write the codepoint ourselves, we advanced pos by len(c) (number of UTF-16 code units), which is correct. However when translate_codepoint returns non-zero (consumed codepoints), the code increments pos by iterating consumed times and calling char_count(code_point_at(code_units, pos)), but code_point_at can raise IndexError if pos is at the last index and previous translator consumed a surrogate pair. Fix by using a safe approach: when consumed>0, advance pos by summing char_count of code_point_at only while pos < length, but avoid calling code_point_at beyond bounds. Specifically, for each iteration, determine codepoint = code_point_at(code_units, pos) but handle IndexError by treating remaining as single code unit and increment pos by 1. Alternatively, simpler: when a translator returns consumed, advance pos by the number of UTF-16 code units that represent the consumed codepoints. We can compute that by iterating through code_units starting at pos and for cnt in range(consumed): get current code_unit and if it's a high surrogate and next exists and is low surrogate, advance by 2 and skip next; else advance by 1. This avoids calling code_point_at beyond end. Implement loop that increments pos_by_units accordingly and add to pos.
