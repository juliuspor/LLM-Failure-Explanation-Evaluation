The code fails because the loop advancing pos after a translator consumes a codepoint uses char_count(code_point_at(code_units, pos)), which calls code_point_at with pos possibly equal to length (out of range) when consumed >0 and pos already at end or when previous increments moved pos beyond bounds. Specifically, after consumed >0 the for-loop does: for pt in range(consumed): pos += char_count(code_point_at(code_units, pos)). If pos is at the start of a surrogate pair, char_count(...) returns 2, but code_point_at reads code_units[pos] and code_units[pos+1]; when pos becomes equal to length (or requests code_point_at on an index that equals length) code_point_at raises IndexError at slice block line 35-36 (index >= len), causing the test failure. Root cause: using code_point_at in the increment loop with the same pos that is being advanced consumes beyond bounds instead of simply adding 1 for each consumed code unit. Also the logic misunderstands 'consumed' meaning: consumed is number of codepoints consumed, but loop advances by code units. The fix is to advance pos by the number of code units consumed (i.e., increment by 1 for BMP codepoint or 2 for surrogate pair) without calling code_point_at at the current pos after it may be out of range; specifically use char_count on the codepoint that was consumed (e.g., compute code_point_at before loop, or better: advance pos by 1 for each code unit consumed, or add 2 when a surrogate pair was consumed).
