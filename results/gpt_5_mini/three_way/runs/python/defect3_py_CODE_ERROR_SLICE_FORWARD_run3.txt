The code fails because the loop that advances pos after a translator consumes a codepoint is incrementing pos incorrectly, causing pos to advance past the end of the code_units list and then be used as an index. Specifically: in translate(), when consumed > 0 the code runs: for pt in range(consumed): pos += char_count(code_point_at(code_units, pos)). Here code_point_at(code_units, pos) (see SLICE_FORWARD) reads code_units[pos] and may try to access code_units[pos+1] only if pos+1 < len(code_units). But the loop calls code_point_at repeatedly while increasing pos by char_count(...) which can be 1 or 2. If pos is near the end (for example pos == len(code_units)-1) and consumed >= 1 the first call to code_point_at is okay, returns the trail or single unit; char_count may return 2 if code_point_at sees a high surrogate and there is a following low, but if pos is at the last index the check index + 1 < len(code_units) fails and code_point_at returns the high unit; however the loop can later increment pos by 2 (if char_count returns 2 for a surrogate) and make pos == len(code_units)+1 which is out of range. More directly, the logic assumes translators report 'consumed' in codepoints and that advancing pos by char_count(code_point_at(...)) per codepoint is safe, but char_count reads code_point_at at the current pos each iteration; when pos points at a high surrogate near the end, char_count may compute 2 (expecting a pair) while there is no low surrogate, or the loop may call code_point_at after pos has been increased past the last valid index causing IndexError: "String index out of range: 2".
