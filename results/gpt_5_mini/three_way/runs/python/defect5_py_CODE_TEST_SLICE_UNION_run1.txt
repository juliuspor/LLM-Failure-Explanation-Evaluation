The code incorrectly uses type(array) to infer the element type when array is non-None. At line 332-333 it sets inferred_type = type(array). For a Python list, type(array) returns <class 'list'>, not the element type (e.g. <class 'str'>). That makes inferred_type equal to list rather than the contained element type, so the later logic and the call to ArrayUtils._copy_list_grow1 receive the wrong inferred_type. Separately, when both array and element are None the code sets inferred_type = object (line 337), and then enforces a TypeError only if inferred_type == object and expected_type != object (lines 343-344). In the failing test test_add_null_to_null_with_expected_type, ArrayUtils.add(None, None, expected_type=str) sets inferred_type = object, so it does not raise at inference time, but ArrayUtils._copy_list_grow1 is called with inferred_type == object which produces a list of generic objects (e.g. [None]) that the caller expects to be a string-list. The root cause: using type(array) instead of inferring the element type from the array contents (e.g. using type(array[0]) or using expected_type when provided) causes wrong type inference and incorrect downstream behavior. Specific lines: 332-333 (inferred_type = type(array)) and 343-344 (TypeError raised only for object->expected_type mismatch) are where the logic breaks. Fix steps: 1) If array is not None and has elements, set inferred_type = type(array[0]) (or inspect all elements and determine common type). 2) If array is empty but expected_type provided, use expected_type as inferred_type. 3) If both array and element are None and expected_type provided, use expected_type instead of object. 4) Adjust the condition that raises TypeError to check when the created list type (object) cannot satisfy expected_type, and raise immediately before copying/growing the list to avoid creating wrong-typed lists.
