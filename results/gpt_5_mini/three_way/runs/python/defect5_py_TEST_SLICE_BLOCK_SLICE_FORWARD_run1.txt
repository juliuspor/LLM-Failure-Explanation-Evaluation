The code raises a TypeError in ArrayUtils.add when inferred_type becomes 'object' while expected_type is a concrete type (e.g., str). At lines 343–347 the function checks: if inferred_type == object and expected_type != object: raise TypeError(...). This happens because adding None or starting from a None array leaves the element type unknown, so inferred_type defaults to object. When expected_type is supplied (expected_type=str) the code refuses to treat the object-typed list as a String list and raises a TypeError simulating Java's ClassCastException. Root cause: the method conflates Python's dynamic None/typing with Java-style array casting, enforcing an unnecessary check that blocks valid cases (like adding None into a list of expected_type). It occurs specifically in the add path handling null/None arrays and None elements where inferred_type is set to object. Fix: remove or relax the check at lines 343–347 so that when inferred_type is object but expected_type is provided, the function constructs a list of the expected_type (or allows mixed types) instead of raising. Alternatively, determine inferred_type from expected_type when array is None or when all elements are None: set inferred_type = expected_type if expected_type is not None; and only raise if there is an actual incompatible concrete type conflict (e.g., existing non-None elements incompatible with expected_type).
