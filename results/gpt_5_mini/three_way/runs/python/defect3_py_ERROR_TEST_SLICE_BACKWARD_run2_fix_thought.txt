The bug is an index advancement issue in translate(): when translate_codepoint returns 0 we advanced pos by len(c) where c are UTF-16 units of current codepoint — that's correct. When consumed>0 we had a loop that incremented pos by char_count(code_point_at(code_units, pos)) repeatedly, but calling code_point_at after advancing pos leads to out-of-range because we call code_point_at with pos equal to length. Fix by computing advancement based on codepoints consumed using code_point_at at the starting pos only and summing char_count for each consumed codepoint while ensuring bounds checks. Simpler: when consumed>0, advance pos by the number of UTF-16 code units consumed (i.e., sum char_count(codepoint_at(...)) but avoid calling code_point_at with pos==length). We'll compute a local idx and for i in range(consumed): get cp = code_point_at(code_units, idx) then idx += char_count(cp). Then set pos = idx. Also ensure code_point_at is only called while idx < length. If translator claims to consume beyond available, break. This prevents index==len access.
