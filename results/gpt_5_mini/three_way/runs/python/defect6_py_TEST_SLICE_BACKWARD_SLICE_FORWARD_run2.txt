The code fails because when x is an integer-valued float (x == int(x)) the branch at lines ~185-203 converts x into an int and assigns value = int(x). For large or special cases the code later decides whether to emit scientific notation or plain integer. However this logic runs before checking for negative zero. Negative zero (-0.0) compares equal to 0 when using x == int(x) so it enters the integer branch. At line 186 value becomes 0 losing the sign; later the code emits str(value) or str(x) but the path that adds str(value) (line ~201) prints "0" not "-0.0". Also the final fallback self.add(str(x)) (line ~203) is not reached for the negative-zero case because the integer branch already handled it. The root cause is relying on x == int(x) which treats -0.0 as 0 and discards the sign bit; there is no early check using math.copysign or inspecting the sign bit (e.g., struct.pack or math.atan2) or a dedicated is_negative_zero(x) before converting to int. This occurs specifically between lines 185 and 201 where value/mantissa logic runs and ultimately calls self.add(str(value)) for zero, producing "0" instead of "-0.0".
