The bug arises because the translate loop advances pos by len(c) where c comes from to_chars(codepoint) (1 or 2), but when translate_codepoint consumed==0, that's fine. The error occurs in the branch when consumed != 0: we increment pos in a loop using char_count(code_point_at(code_units, pos)) but code_point_at raises IndexError if pos == length. To avoid calling code_point_at when pos may be at end, we should advance pos by the UTF-16 code units corresponding to the consumed codepoints without calling code_point_at at an index that may be out of range. Since consumed is number of codepoints consumed (not code units), we should iterate consumed times and for each, fetch the codepoint at current pos safely: if pos >= length: break else use code_point_at; but better: when consumed>0, we should increment pos by the number of UTF-16 code units consumed, which can be computed by checking if the starting code unit is a high surrogate and next is low surrogate -> that's 2, else 1; and repeat consumed times. Implement loop that checks pos < length before calling code_unit at pos. This prevents IndexError.
