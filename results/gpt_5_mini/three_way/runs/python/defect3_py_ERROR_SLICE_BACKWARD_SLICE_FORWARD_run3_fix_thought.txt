The bug stems from translate using UTF-16 code unit indices while translate_codepoint implementations (like CsvEscaper) assume index points to a code unit and may return counts in code points. The error arises when translate_codepoint returns a count >0 that is meant as codepoints but the loop increments pos by char_count(code_point_at(code_units, pos)) multiple times leading to out-of-range. Fix by interpreting consumed as number of codepoints (not code units) and advance pos by summing char_count for each consumed codepoint, but ensure we check bounds before calling code_point_at. Also handle case when consumed >0 but not enough remaining code units: advance by remaining units and break. Add safer checks.
