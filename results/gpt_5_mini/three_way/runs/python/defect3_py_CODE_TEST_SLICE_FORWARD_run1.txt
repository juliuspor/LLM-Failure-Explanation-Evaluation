The code fails because the loop that advances pos after a translator consumes a codepoint uses char_count(code_point_at(code_units, pos)) inside a for-loop iterating consumed times, but pos is not advanced before the first iteration, so code_point_at is called repeatedly on the same code unit index and char_count returns 2 for a surrogate pair, causing pos to be increased by 2 twice and jump past bounds (IndexError). Specifically: in translate (lines shown), after consumed = self.translate_codepoint(...), the for pt in range(consumed): pos += char_count(code_point_at(code_units, pos)) is wrong. When consumed>0 and the consumed codepoint was a surrogate pair (two UTF-16 code units), translate_codepoint already handled both units but pos still points to the high surrogate; the loop calls code_point_at(code_units, pos) multiple times without advancing between iterations, causing an incorrect double-count. The bug manifests in translate when processing a surrogate pair followed by another character (test_translate_surrogate_pair_with_suffix): pos increments overshoot the array length and subsequent code_point_at or indexing raises IndexError. The root cause is using char_count(code_point_at(...)) inside a loop that does not advance pos per-iteration correctly.
