The code fails because minutes_offset is mutated incorrectly when hours_offset is negative, causing minutes_offset to become greater than 59 which later leads to an out-of-range offset. Specifically: at line 377 the code checks minutes_offset range (0..59) and raises if invalid. Later (lines 383–386) the code attempts to adjust minutes_offset based on hours_offset sign using "minutes_offset = hours_in_minutes - minutes_offset" for negative hours and "minutes_offset = hours_in_minutes + minutes_offset" for positive hours. When hours_offset is negative, subtracting minutes_offset from hours_in_minutes (which is hours_offset * 60, a negative number) yields a value outside the expected 0..59 minute range and then is multiplied by MILLIS_PER_MINUTE. This causes either an incorrect total offset or triggers the "Offset is too large" check (line 388). The root cause is treating minutes_offset as an absolute minute value to add/subtract from a signed hours_in_minutes, rather than converting the combined hours and minutes into a single signed minute total first. As a result, inputs like hours_offset = -1 and minutes_offset = 30 produce the wrong signed minute total (-60 - 30 becomes -30 or similar depending on branch), making the offset calculation and range checks fail. Fix: compute total_minutes = hours_offset * 60 + (hours_offset < 0 ? -minutes_offset : minutes_offset) or normalize sign first; then use total_minutes directly to compute offset in milliseconds and remove the separate adjustments that invert minutes_offset.
