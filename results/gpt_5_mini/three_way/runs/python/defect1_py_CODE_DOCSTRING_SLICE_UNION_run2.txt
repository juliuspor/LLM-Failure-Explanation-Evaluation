The code incorrectly reassigns the minutes_offset variable and then multiplies it as if it were the total minutes offset; this causes wrong sign and value when hours_offset is negative, and also obscures the original minute component. Specifically: at line 383 (hours_in_minutes = hours_offset * 60) the code computes minutes_offset = hours_in_minutes - minutes_offset for negative hours_in_minutes and minutes_offset = hours_in_minutes + minutes_offset for non-negative hours_in_minutes. That changes the meaning of minutes_offset from "minutes component within the hour" to "total minutes offset", but the naming and later range checks no longer apply. The real bug manifests when hours_offset is negative: for hours_offset = -1 and minutes_offset = 30 the calculation yields -60 - 30 = -90 minutes, which is correct as total minutes but the code previously validated minutes_offset as being 0..59 and later reuses the same variable name, which can be confusing. The bigger problem is that when hours_offset is 0 the function returns early, but other edge cases such as hours_offset = -23 and minutes_offset > 0 produce a total beyond allowed range and may cause FieldUtils.safe_multiply to raise ArithmeticError; the catch converts it to a generic "Offset is too large" ValueError without indicating which input caused it. Also mutating the input variable minutes_offset prevents clearer logic and makes it easy to miss that minutes_offset was validated under a different meaning. Suggested fix: compute a separate total_minutes variable (e.g. total_minutes = hours_offset * 60 + ( -minutes_offset if hours_offset < 0 else minutes_offset )), keep input validation for minutes_offset, and then call offset = FieldUtils.safe_multiply(total_minutes, DateTimeConstants.MILLIS_PER_MINUTE). This keeps variable names clear and confines the risky arithmetic to a single well-named variable, and will avoid sign mistakes and confusion at lines 383-386.
