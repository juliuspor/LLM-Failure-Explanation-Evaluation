The code fails because the loop that advances pos after a translator consumes code points advances by char_count(code_point_at(code_units, pos)) while still at the original pos, but the for loop increments pos multiple times and uses code_point_at with pos beyond the end. Specifically, after consumed > 0, the code does: for pt in range(consumed): pos += char_count(code_point_at(code_units, pos)). If pos points to a high surrogate near the end, char_count(code_point_at(...)) may attempt to read the low surrogate at index pos+1 via code_point_at, which checks index bounds and raises IndexError when pos==len(code_units)-1 and it tries to access pos+1. The bug occurs when a surrogate pair is at position pos and is the last pair followed by another character: the loop miscomputes increments because code_point_at is called repeatedly while pos advances inside the loop, causing a call with index equal to len(code_units) (out of range). In short: the post-consume advancement uses code_point_at with a pos that can be incremented past the valid range, leading to IndexError: String index out of range: 2.
