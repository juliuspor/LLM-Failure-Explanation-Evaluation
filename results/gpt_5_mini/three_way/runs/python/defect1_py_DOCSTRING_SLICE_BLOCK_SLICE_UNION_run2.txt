The code fails because minutes_offset is treated incorrectly for negative hour offsets leading to an incorrect sign flip and an out-of-range total offset when converted to milliseconds. Specifically: at lines 377-378 the code validates minutes_offset is between 0 and 59. Later (slice union lines 383 and 385) the code attempts to adjust minutes_offset when hours_offset is negative by computing minutes_offset = hours_in_minutes - minutes_offset (line 383) or minutes_offset = hours_in_minutes + minutes_offset (line 385). Here hours_in_minutes is hours_offset * 60 and already carries the sign. For example, if hours_offset is -2 and minutes_offset is 30, hours_in_minutes = -120. The code does minutes_offset = -120 - 30 = -150 (line 383), which makes minutes_offset negative and larger in magnitude than expected. Then offset = FieldUtils.safe_multiply(minutes_offset, DateTimeConstants.MILLIS_PER_MINUTE) (line 386) produces a negative millisecond offset of -150 minutes instead of the intended -90 minutes (-1 hour 30 minutes). This mismatch can trigger the "Offset is too large" check (line 388) or produce an incorrect timezone. The root cause is treating minutes_offset as an absolute positive component while also combining it directly with a signed hours_in_minutes; the logic should apply the minutes sign consistently with hours (e.g., compute total_minutes = hours_in_minutes + (hours_offset < 0 ? -minutes_offset : minutes_offset)) and then multiply to milliseconds, and validate the final total_minutes range before creating the DateTimeZone.
