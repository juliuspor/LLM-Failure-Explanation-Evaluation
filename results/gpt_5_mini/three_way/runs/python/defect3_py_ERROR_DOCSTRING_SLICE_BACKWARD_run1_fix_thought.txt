The bug was an IndexError caused by advancing pos incorrectly after a translator consumes codepoints. The loop used char_count on code_point_at(code_units, pos) inside a for-loop which calls code_point_at with pos advanced, causing out-of-range access when consumed >1. Fix by computing the codepoint at pos once per iteration and advancing pos by char_count of that codepoint. Also ensure if consumed > 0 we advance pos correctly by summing char_count for each codepoint consumed, but avoid calling code_point_at with out-of-range index by checking pos < length before each access. Simpler: loop consumed times: get cp = code_point_at(code_units, pos) then pos += char_count(cp). This prevents using stale indices. Also handle malformed surrogate pairs by advancing by 1 if code_point_at would raise (but code_point_at already raises), so ensure bounds checked. Implemented change accordingly.
