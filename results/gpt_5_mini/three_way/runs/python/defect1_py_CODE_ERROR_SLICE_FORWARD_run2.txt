The code fails because minutes_offset is validated once near the top (line checking if minutes_offset < 0 or minutes_offset > 59), but later the code reassigns minutes_offset to a combined total of hours_in_minutes ± minutes_offset (lines 383 and 385). When hours_offset is negative and minutes_offset was positive, the expression hours_in_minutes - minutes_offset produces a negative value whose magnitude can exceed 59 (for example hours_offset=-1 and minutes_offset=15 gives hours_in_minutes = -60 and minutes_offset = -75), which violates the earlier intended bounds and leads to inconsistency. In the failing test the ValueError "Minutes out of range: -15" indicates minutes_offset was -15 at the initial validation, meaning the input minutes_offset was negative and got caught by the early check. The root cause is that the signing and combination logic is wrong: the method both validates minutes_offset as a 0..59 absolute value but also expects minutes_offset to be signed before combining with hours. Specifically, on lines 383 and 385 the code overwrites minutes_offset with a total minutes value instead of using a separate variable (e.g., total_minutes). Also the sign handling when hours_in_minutes < 0 should subtract the absolute minutes (hours_in_minutes - minutes_offset) but because minutes_offset is already allowed only 0..59 the calculation becomes incorrect for negative hour offsets. Fix by keeping minutes_offset as an unsigned offset (0..59) and computing total_minutes = hours_in_minutes - minutes_offset when hours_offset < 0, else hours_in_minutes + minutes_offset; or accept signed minutes_offset and remove the initial check. Then multiply total_minutes by MILLIS_PER_MINUTE and pass that to for_offset_millis. This ensures minutes stay within 0..59 before combination and prevents creating negative minute values that trigger the reported error.
