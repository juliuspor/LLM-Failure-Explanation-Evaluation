The code fails because the loop that advances pos after a translator consumes a codepoint uses char_count(code_point_at(code_units, pos)), which reads code_point_at at the current pos repeatedly while pos is being updated, causing an out-of-range access: when consumed > 0 it should advance pos by 1 for each consumed UTF-16 code unit rather than recomputing char counts at the advancing positions. Specifically, in the for loop starting at "for pt in range(consumed):" the body calls code_point_at(code_units, pos) then char_count(...). On the first iteration this may return 2 for a surrogate pair and then pos is incremented by 2, skipping past the intended units; on subsequent iterations code_point_at is called with a pos beyond length, raising IndexError: String index out of range: 2. In short, the loop both interprets consumed as a number of codepoints but then advances using variable-width char counts derived from the same advancing index, leading to over-advancing and an out-of-bounds index. The bug occurs in translate(), lines where the "for pt in range(consumed):" loop increments pos using char_count(code_point_at(code_units, pos)). The correct approach is to advance pos by the number of UTF-16 code units consumed (i.e., increment pos by 1 for each unit consumed, or directly add consumed if consumed counts units) rather than calling code_point_at on a position that may already be past the end.
