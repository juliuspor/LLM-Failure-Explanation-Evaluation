The bug is caused by advancing pos incorrectly after a translator consumes codepoints. translate_codepoint returns number of codepoints consumed (in codepoints), but the loop advanced pos by calling code_point_at at the changing position, which can go out of bounds. Fix by computing the number of UTF-16 code units to advance for each consumed codepoint without moving pos until done: iterate a temp index from pos and sum char_count(code_point_at(code_units,temp_index)) then set pos += total_units. This avoids calling code_point_at with out-of-range pos. Also handle consumed==0 case unchanged.
