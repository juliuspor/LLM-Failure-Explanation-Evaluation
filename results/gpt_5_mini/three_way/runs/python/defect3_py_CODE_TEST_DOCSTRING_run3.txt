The code fails because the loop that advances pos after a translator consumes a code point advances by char_count(code_point_at(code_units, pos)) while also iterating 'pt' over range(consumed). When consumed > 0 for a surrogate pair (a single code point represented by two UTF-16 code units), the inner loop computes char_count(code_point_at(code_units, pos)) but does so while pos still points to the start of the surrogate pair for the first iteration, causing the first increment to be 2 (correct) but on the next iteration pos now points after the pair; calling code_point_at(code_units, pos) can then read the next code unit(s). If the consumed value represents code points (not UTF-16 code units), the loop's logic is wrong: it multiplies increments, advancing pos too far or reading beyond the array when a surrogate pair is present followed by another character. Specifically, in the failing test case (translate, lines in translate method), consumed is 1 for the translator that handled the surrogate pair, but the for-loop runs once and calls char_count(code_point_at(code_units, pos)) with pos at the lead surrogate; char_count returns 2, so pos += 2 moves past the following 'A' character's code unit if subsequent iterations or surrounding logic expect different behavior. The root cause: confusion between "consumed" (number of code points) and UTF-16 code units, and using code_point_at(code_units, pos) inside the loop while mutating pos. This leads to index advancement errors and IndexError when a surrogate pair is followed by another character (as in test_translate_surrogate_pair_with_suffix). Suggested fixes: (1) Treat consumed as a number of code points and advance pos by summing char_count for each consumed code point by reading code points without changing pos (e.g., compute advance = 0; for i in range(consumed): cp = code_point_at(code_units, pos + advance); advance += char_count(cp); pos += advance), or (2) change translate_codepoint to return number of UTF-16 code units consumed instead of code points and then simply do pos += consumed, or (3) after a nonzero consumed, advance pos by the number of UTF-16 code units consumed computed once (not recomputed with moving pos).
