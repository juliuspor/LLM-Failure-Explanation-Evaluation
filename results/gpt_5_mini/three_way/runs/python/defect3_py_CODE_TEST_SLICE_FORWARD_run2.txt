The code fails because the loop that advances pos after a translator consumes a codepoint uses char_count(code_point_at(code_units, pos)) inside a for-loop iterating consumed times, but pos is not incremented before calling code_point_at, so on the first iteration code_point_at reads the same high surrogate again (not the combined codepoint) and char_count returns 1, causing pos to advance incorrectly by 1 per iteration and possibly re-reading surrogates — ultimately leading to an IndexError when a consumed surrogate pair at the end is advanced incorrectly. Specifically: in translate(), after consumed = self.translate_codepoint(...), the for-loop (lines shown) does "for pt in range(consumed): pos += char_count(code_point_at(code_units, pos))". If consumed==1 for a surrogate pair, code_point_at at that pos will return the full codepoint only if it sees the high and low surrogate; but because pos hasn't been advanced before the loop, and char_count expects a codepoint integer, calling code_point_at repeatedly without incrementing pos properly causes wrong advances. The root cause is conflating 'consumed' (number of codepoints translated) with the number of UTF-16 code units to advance; the code should advance by char_count of the codepoint read once per consumed iteration, but must first extract the codepoint at the current position and then advance pos by char_count(codepoint), not call code_point_at with the same pos repeatedly. This occurs around lines where consumed is handled and at the surrogate handling in code_point_at (slice_forward lines 38-44).
