The code fails because when parsing a locale string of length 5 or more (in LocaleUtils.to_locale), lines 227–230 index characters at positions 3 and 4 without first handling the special case of an empty country (double underscore). For input like "fr__POSIX" the character at index 2 is '_' and index 3 is '_' (not an uppercase letter). The code then executes: ch3 = locale_str[3]; ch4 = locale_str[4]; and immediately checks if ch3 and ch4 are between 'A' and 'Z' (line 229). Since ch3 == '_' it fails the uppercase-letter check and raises ValueError("Invalid locale format: fr__POSIX") at line 230. In short: the parser assumes that when there is a variant the country will be two uppercase letters, but it does not accept an empty country ("__"). This causes the method to reject valid language__variant strings. The failure happens in to_locale (starting at line 188), specifically around lines 227–230 where ch3/ch4 are read and validated; the root cause is missing handling for the empty country case (double underscore) prior to the uppercase checks.
