The code fails because the loop that advances pos when a translator consumes codepoints updates pos incorrectly and can use an out-of-range index. Specifically, in the for-loop at the end of translate (lines after the comment about surrogate pairs) pos is incremented by char_count(code_point_at(code_units, pos)) for each consumed codepoint. char_count expects a Unicode code point value, but code_point_at(code_units, pos) reads from code_units using the current pos. On the first iteration of that for-loop pos points to the start of the consumed codepoint sequence; char_count(code_point_at(code_units, pos)) returns 2 if the code unit at pos is a high surrogate followed by a low surrogate, otherwise 1. However the code then increments pos by that value and repeats the loop, calling code_point_at(code_units, pos) again. If the previously increment moved pos past the end of code_units (for example when the last codepoint was a surrogate pair and pos becomes equal to length), the next code_point_at call hits the bounds check at lines 35-36 and raises IndexError("String index out of range: {index}"). In other words, the loop reads beyond the end when consumed > 1 and the final increment advances pos to length, then the loop attempts another access. The root cause is using code_point_at on the advancing pos inside the for-loop and relying on pos to remain valid for every iteration. A clearer and safe approach is to compute the number of UTF-16 code units consumed per codepoint by inspecting the code units directly or to advance pos deterministically by 1 or 2 per iteration without re-calling code_point_at at the new pos. For example: capture initial_pos = pos; for _ in range(consumed): if is_high_surrogate(code_units[pos]) and pos+1<length and is_low_surrogate(code_units[pos+1]): pos += 2 else: pos += 1 This prevents calling code_point_at with pos==length and avoids the IndexError.
