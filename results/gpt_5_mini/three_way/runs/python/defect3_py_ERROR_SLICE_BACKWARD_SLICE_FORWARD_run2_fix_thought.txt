The bug is that translate may advance pos incorrectly after a translator consumes codepoints, causing pos to become equal/greater than length before next loop; but the primary issue in diagnosis is callers passing invalid index. Safer approach: after calculating 'consumed', if consumed > 0, advance pos by the number of UTF-16 code units corresponding to the consumed codepoints without calling code_point_at at positions that may be out of range. We'll compute codepoint at current pos for each of 'consumed' iterations but check bounds before calling. If out of range, break. This avoids IndexError. Also ensure that when consumed==0 we handle single code unit safely. Implemented defensive checks for bounds when calling code_point_at and char_count.
