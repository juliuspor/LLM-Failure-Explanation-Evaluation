The code infers types incorrectly when adding elements to arrays. At lines 332–335 it sets inferred_type = type(array) when array is not None and inferred_type = type(element) when element is not None; but type(array) returns the array/list class (e.g. list), not the element component type, so inferred_type becomes list instead of the element type. At line 337 inferred_type is set to object in the fallback case, and then lines 343–344 raise TypeError if inferred_type == object and expected_type != object. This causes a TypeError when both array and element are None but expected_type is provided (e.g. expected_type=str). The test calls ArrayUtils.add(None, None, expected_type=str) expecting [None], but the code treats inferred_type as object and raises TypeError because expected_type is not object. The root causes: using type(array) instead of inspecting array contents or using an expected_type parameter; and treating the absence of type info (object) as an error for a provided expected_type. Specific fixes: 1) When array is not None, determine the element type from array contents (e.g. use type(array[0]) if non-empty) or accept expected_type if supplied. 2) When element is not None use type(element) as currently done. 3) If both are None, default to expected_type if given, otherwise allow element type to be object and do not raise TypeError. 4) Change the check at lines 343–344 to only raise if inferred_type is object and expected_type is not None and incompatible, or better, set inferred_type = expected_type when expected_type is provided. This will let ArrayUtils.add(None, None, expected_type=str) return [None] without error.
