Fix ensures translate iterates over UTF-16 code units list and advances position in terms of code units, not Python codepoints. The original bug advanced pos by len(chars) where chars came from to_chars(codepoint) producing 2 for supplementary codepoints, but pos increment inside the loop over consumed used char_count(code_point_at(code_units, pos)) which could double-count. Simplify: when consumed==0 advance by char_count(code_point_at(...)) after writing; when consumed>0 simply advance pos by the total number of UTF-16 code units corresponding to the consumed code points. Compute advancement by summing char_count for each consumed codepoint starting at current pos before translation. Implement by computing advance = 0; for _ in range(consumed): advance += char_count(code_point_at(code_units, pos + advance)); pos += advance. This prevents overshoot.
