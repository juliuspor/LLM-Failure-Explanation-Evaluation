The code fails because in the for-loop that advances pos after a translator consumes one or more code points, pos is advanced by char_count(code_point_at(code_units, pos)) while the contract of 'consumed' counts code points, not UTF-16 code units. Specifically, when consumed > 0 and the current code point is a surrogate pair (two UTF-16 code units), the loop body calls code_point_at(code_units, pos) to get a code point and then char_count(...) to determine how many UTF-16 code units that code point occupies; it then increments pos by that amount for each iteration. However, char_count(code_point_at(...)) is computed using the code point at the current pos before pos is advanced inside the loop. As pos increases inside the loop, code_point_at may be called with stale or invalid positions, causing incorrect indexing. The real bug is that the inner for-loop iterates 'consumed' times but uses the dynamic pos inside the iteration to compute the advance amount; since pos is mutated each iteration, the code may skip or re-read code units leading to an IndexError when pos goes past length or to incorrect advancement. In the failing test (surrogate pair followed by 'A'), a translator consumed the surrogate pair (consumed==1) but the loop does: for pt in range(1): pos += char_count(code_point_at(code_units, pos)). If char_count returns 2 (for the surrogate pair), pos becomes 2. Later, outer while loop continues and attempts to process the next code unit at pos==2, but because length is the number of code units (3 for emoji+ 'A'), this should be fine. The actual failure arises when consumed corresponds to code points and the code uses char_count(code_point_at(code_units, pos)) inside the loop where code_point_at expects that pos indexes into code_units and may read a surrogate pair starting at the high surrogate; if pos is already at the low surrogate due to previous increments, code_point_at will raise because it sees an unpaired surrogate or goes out of bounds. Concretely, advancing pos by char_count(code_point_at(...)) while computing code_point_at at the same pos each iteration is fragile. The root cause: mixing indexes of UTF-16 code units (pos) and code points (consumed) and advancing pos incorrectly. The failure occurs in the for-loop that starts with 'for pt in range(consumed):' (near the end of the method). To fix: advance pos by the corresponding number of UTF-16 code units per consumed code point without re-reading code_point_at at a possibly invalid pos — for example, compute the char counts from the initially consumed code points or use the code_units length logic: if consumed==1 and code_units[pos] is a high surrogate and next exists, pos += 2 else pos +=1; or better, have translate_codepoint return the number of UTF-16 code units consumed instead of code points.
