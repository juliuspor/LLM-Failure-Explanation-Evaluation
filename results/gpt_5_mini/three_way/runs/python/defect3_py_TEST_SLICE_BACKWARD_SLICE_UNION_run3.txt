The function to_utf16_code_units (used by code_point_at) builds a list of UTF-16 code units and then code_point_at uses an index into that list. The failure occurs when code_point_at checks the high surrogate and then computes the code point by reading the low surrogate at index+1 without ensuring the original index is compared to the length of the original Python string rather than the length of the UTF-16 code_units list. Specifically, at lines 35-36 the function raises IndexError if index >= len(code_units), but elsewhere (lines 38-41) code_point_at assumes index is an index into code_units and tests index+1 < len(code_units) only after reading high = code_units[index]. The real bug is that the translator advances the loop index by 1 for a surrogate pair (treating the pair as two code units) but later code uses Python string indices (which are by Unicode code points) leading to a mismatch: when a surrogate pair is at the end and the translator tries to access code_units[index+1], index is equal to len(code_units)-1 and index+1 is out of range, causing IndexError. In short: index arithmetic mixes UTF-16 code unit indices and Python string indices, and the check at lines 35-36 plus the subsequent use of index+1 at line 40 fails for surrogate pairs at string end.
