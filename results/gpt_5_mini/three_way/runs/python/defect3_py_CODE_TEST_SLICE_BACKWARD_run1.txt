The code fails because the loop that advances pos after a translator consumes a codepoint uses char_count(code_point_at(code_units, pos)) while also iterating pt in range(consumed) and incrementing pos inside that loop. On the first iteration, char_count is called with pos still pointing to the start of the consumed codepoint; but char_count returns 2 for a surrogate pair, so pos is incremented by 2. On the next loop iteration the code calls code_point_at(code_units, pos) again, but pos already advanced past the consumed pair, so code_point_at may index out of range or misread. Specifically, when consumed==1 for a surrogate pair represented as two UTF-16 code units, the for-loop runs once but adds char_count(code_point_at(code_units, pos)) where pos is not moved before calling char_count; if char_count returns 2 this is fine, but when consumed>1 or in some cases the for-loop increments pos by char_count of a codepoint that itself could be 2, causing pos to jump too far and then the outer while continues and the next access to code_point_at(code_units, pos) can raise IndexError (see to_utf16_code_units and its bounds check at lines 35-36). The root cause is using pos inside the loop both to compute char_count and to advance, rather than consuming the known number of code units directly. In short: the code confuses 'consumed' (number of codepoints) with UTF-16 code units; it advances pos by char_count(code_point_at(...)) repeatedly, which double-counts surrogate pairs and can move pos past the buffer, causing IndexError. The problematic lines are the for pt in range(consumed): loop and the body pos += char_count(code_point_at(code_units, pos)).
