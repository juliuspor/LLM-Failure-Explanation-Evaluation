The code fails because to_utf16_code_units/code_point_at treats Python string indices as UTF-16 code unit indices but Python strings are sequences of Unicode code points. In the slice shown, code_point_at reads 'high = code_units[index]' and then if it's a high surrogate checks 'index + 1 < len(code_units)' and reads the low surrogate. However, elsewhere the translator advances `index` by 1 for every Python character processed. When a surrogate pair from UTF-16 is represented as a single Python code point (for characters above U+FFFF), the implementation still expects two entries and advances incorrectly, causing index to go out of range at the boundary (see IndexError raised at lines 35-36). Concretely: in translate_string the loop increments `index` without accounting for when code_point_at returns a value produced by combining a high+low surrogate (lines 38-44). If index reaches the last position and code_point_at attempts to access index+1, the bounds check at lines 35-36 may be bypassed because `len(code_units)` was measured in code points, not UTF-16 code units; then reading code_units[index+1] is out of range, raising IndexError. Root cause: mixing UTF-16 code unit indexing with Python's UTF-32/Unicode code point indexing.
