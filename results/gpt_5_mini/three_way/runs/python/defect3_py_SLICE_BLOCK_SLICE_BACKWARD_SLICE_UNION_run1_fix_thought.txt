The bug is an off-by-one when advancing pos when no translator consumed codepoints. The translate method incorrectly advances pos by len(c) (number of UTF-16 code units for the codepoint) which is correct; but when a translator consumes codepoints, the loop increments pos by char_count(code_point_at(code_units, pos)) inside a for-loop using consumed causing code_point_at to be called with pos already advanced beyond length. Simplify: when consumed>0, advance pos by sum of UTF-16 code units for each consumed codepoint starting from original pos. Compute for i in range(consumed): cp = code_point_at(code_units, pos); pos += char_count(cp). This ensures code_point_at is always called with valid pos. Also handle translator returning more codepoints than remain by breaking. Implement corrected translate method accordingly.
