Replace the buggy loop that advances pos after a translator consumed codepoints. The translators return the number of codepoints consumed (not variable-width units), but we should advance pos by summing the UTF-16 code unit lengths (char_count) of each consumed codepoint without repeatedly calling code_point_at on possibly out-of-range indices. The fix: iterate a local counter i from 0 to consumed-1, compute cp = code_point_at(code_units, pos) once per iteration, then advance pos by char_count(cp). This avoids reading beyond the end.
