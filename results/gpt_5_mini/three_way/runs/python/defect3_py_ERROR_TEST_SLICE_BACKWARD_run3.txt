The IndexError arises in to_utf16_code_units when code_units length is checked against an index computed incorrectly. In the tests the input contains a Unicode code point that maps to a UTF-16 surrogate pair (e.g. '\U0001F630'). to_utf16_code_units builds a list of 16-bit code units; later code_point_at (called by CsvEscaper.translate_string) passes an index that refers to the original Python string code point index rather than the position in the code_units list or advances the index by 1 after processing a surrogate pair instead of by 2. Specifically, at line 35-36 the function raises IndexError: f"String index out of range: {index}" because 'index' equals 2 while len(code_units) is 2 (valid indices are 0 and 1), so index >= len(code_units). This happens when the caller advanced past a surrogate pair only by 1, then called to_utf16_code_units(index) expecting a next code unit to exist at position 2 even though the code_units list length is 2. In short: a mismatch between how indices are counted (UTF-16 code units vs Python string code points) causes the caller to request an out-of-range index. The root cause location: the index advancement logic in CsvEscaper.translate_string or code_point_at doesn't increment by 2 after encountering a high surrogate; it increments by 1, leading to index==len(code_units) and the IndexError at lines 35-36 of to_utf16_code_units.
