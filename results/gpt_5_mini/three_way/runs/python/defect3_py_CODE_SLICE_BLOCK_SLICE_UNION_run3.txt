The code fails due to an IndexError in code_point_at when called with an index equal to len(code_units) or otherwise out of range. In translate, inside the loop when consumed == 0, you call code_point_at(code_units, pos) which may be safe, but when consumed > 0, the for loop does: for pt in range(consumed): pos += char_count(code_point_at(code_units, pos)). If a translator returned consumed >0 for a surrogate pair it expects to advance pos by consumed code units, but the loop increments pos by char_count(code_point_at(code_units, pos)) which itself can call code_point_at with pos equal to the last index +1 after previous increments, causing index >= len(code_units) and triggering the IndexError at the check "if index < 0 or index >= len(code_units): raise IndexError(...)" (shown at lines 35-36). Concretely, when pos is at the final code unit and a translator signaled it consumed one codepoint that spans two code units (surrogate pair), the inner loop calls code_point_at with pos pointing to the second code unit (or beyond), causing the out-of-range check to fail. The root cause is mixing two different notions: consumed is a count of codepoints the translator consumed, but the loop advances pos using char_count(code_point_at(...)) which reads code units and can read past the end. Fixes: (1) Change the loop to advance pos by the number of code units consumed directly (i.e., pos += consumed), or (2) ensure translators return the number of UTF-16 code units consumed instead of codepoints and document contract, or (3) guard calls to code_point_at by checking pos < length before calling, and if translator consumed indicates multiple codepoints, fetch codepoints safely. Preferred fix: have translate_codepoint return the number of code units consumed; then replace the for loop with pos += consumed. This avoids calling code_point_at at positions that may be out of range.
