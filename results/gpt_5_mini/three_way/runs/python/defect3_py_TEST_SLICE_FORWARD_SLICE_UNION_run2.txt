The code fails because to_utf16_code_units builds a list of UTF-16 code units (variable code_units) and code_point_at uses index into that list. In code_point_at (lines 35-44) when it detects a high surrogate at code_units[index] it checks index + 1 < len(code_units) and reads low = code_units[index + 1]. But later callers advance the index by 1 for a surrogate pair instead of 2; as a result, when a surrogate pair is at position N and the next character follows, code_point_at is later called with index equal to the low-surrogate position (N+1). On that call, line 35 does not raise, line 38 sets high = code_units[index] which is the low surrogate (0xDC00–0xDFFF), the condition on line 39 (0xD800 <= high <= 0xDBFF) fails, so it returns the low surrogate value (line 44) instead of a full code point, producing incorrect values. If callers then try to treat that returned value as a full code point and advance past it by two code units, they can compute an out-of-range next index and raise IndexError. In short: the function assumes callers never call it on the low surrogate index, but the translator advances index incorrectly (advancing only by 1 when it should skip both code units), causing code_point_at to see a low surrogate as a high surrogate or return wrong values and leading to IndexError. Specific variables/methods: to_utf16_code_units (which creates code_units), code_point_at (lines 35,38-44) misinterprets surrogate halves when index handling is wrong; the bug manifests when translating a surrogate pair followed by another character (the test input "\U0001F630A") where index advancement logic fails and triggers IndexError.
