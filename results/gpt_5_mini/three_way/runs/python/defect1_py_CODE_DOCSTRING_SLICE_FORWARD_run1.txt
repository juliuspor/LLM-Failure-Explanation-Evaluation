The code fails because it mutates the input parameter minutes_offset when computing total minutes, then passes that possibly-negative or incorrectly computed value into FieldUtils.safe_multiply. Specifically, on lines 383–386 the code assigns minutes_offset = hours_in_minutes - minutes_offset if hours_in_minutes < 0, otherwise minutes_offset = hours_in_minutes + minutes_offset. That reuses the parameter name minutes_offset for the total minutes, losing the original minutes component. If hours_in_minutes is negative, subtracting minutes_offset increases the negative magnitude incorrectly (it should subtract the absolute minutes component in the direction of the sign), and if hours_in_minutes is zero the initial check returned UTC, so not applicable. This mutation leads to wrong total minute calculation or sign errors causing FieldUtils.safe_multiply to raise ArithmeticError or produce an overflow, which is caught and turned into "Offset is too large" at line 388. The correct approach is to compute a new variable (e.g., total_minutes = hours_in_minutes + ( -minutes_offset if hours_in_minutes < 0 else minutes_offset)) and then multiply that by MILLIS_PER_MINUTE, avoiding reusing minutes_offset.
