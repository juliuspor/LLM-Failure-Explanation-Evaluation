The code fails because the loop that advances pos after a nonzero consumed value uses char_count(code_point_at(code_units, pos)) while still at the original pos; but pos is advanced inside the for loop by that expression repeatedly which can read past the end or miscount when consumed>1 because code_point_at may raise IndexError at line 36 when pos equals length. Specifically: in translate when consumed>0, the for pt in range(consumed): body calls code_point_at(code_units, pos). If pos has already reached the end (pos==length) this triggers the IndexError from code_point_at at line 36: "String index out of range: {index}". This happens when translate_codepoint indicated it consumed a surrogate pair (consumed==1 or 2) but the loop increments pos incorrectly using char_count of the codepoint at current pos, and may call code_point_at with pos==length. Root cause: wrong advancement logic after a successful consumption — it should increment pos by the number of UTF-16 code units consumed (typically 1 for BMP char or 2 for surrogate pair), not by char_count(code_point_at(code_units, pos)) evaluated repeatedly. Fix: replace the for-loop with pos += consumed (or with pos += 1 or 2 depending on whether the original codepoint at the start was a surrogate pair). Concretely: after consumed = self.translate_codepoint(...), use pos += consumed or compute cp = code_point_at(code_units, pos); pos += char_count(cp) once. Avoid calling code_point_at when pos==length.
