The code fails because of an incorrect index advancement in the while-loop when translate_codepoint returns a positive consumed value. Specifically, inside the for-loop (lines near the end) pos is incremented by char_count(code_point_at(code_units, pos)) for each iteration, but code_point_at reads using the same pos which changes inside the loop, causing a wrong value or IndexError when advancing past a surrogate pair. When consumed > 0 the translator contract means it consumed one code point (which may be a surrogate pair represented by two UTF-16 code units). The loop should advance pos by the number of UTF-16 code units that make up each consumed code point (i.e., char_count of the code point at the original start position or simply add 1 for BMP or 2 for supplementary). Currently, because pos is updated inside the loop and code_point_at may read at a position that is already beyond the high surrogate without adjusting for pair length, the function can index out of range (see code_point_at bounds check: it raises IndexError if index >= len(code_units)). Concretely, for a surrogate pair at pos: first iteration computes char_count for the high surrogate and advances pos by 2, but the loop then tries to run the second iteration (pt from 1 to consumed-1) and calls code_point_at at the new pos which is beyond length, triggering IndexError. The root cause: using a loop over consumed and recomputing char_count from the moving pos rather than advancing by the fixed number of code units consumed at the start. Fix: compute the number of code units for the original code point (e.g., ch_count = char_count(code_point_at(code_units, pos)) or use consumed to mean code units), then add ch_count once (or add ch_count*consumed) to pos. Alternatively, replace the for-loop with pos += ch_count_per_codepoint * consumed, or iterate using an original_index variable for code_point_at calls.
