The code fails because to_utf16_code_units or code_point_at uses an index that goes out of range when reading surrogate pairs. Specifically, in translate at the loop end (lines shown), the for loop increments pos by char_count(code_point_at(code_units, pos)) for each pt in range(consumed). If consumed > 0, translate_codepoint indicated it consumed a codepoint starting at pos that may be a surrogate pair (2 code units), but the for loop calls code_point_at(code_units, pos) repeatedly while advancing pos inside the loop; on the first iteration it may be valid, but on the second iteration pos has advanced past length leading to code_point_at raising IndexError at line 36: "String index out of range: {index}". The root cause is mixing two different notions: translate_codepoint returns number of codepoints consumed, but the loop advances by char_count(code_point_at(...)) which may advance by 1 or 2; iterating consumed times can overshoot. Instead the code should advance pos by the number of code units consumed (e.g., 1 for BMP or 2 for a surrogate pair) determined once from the initial codepoint, not recompute at each iteration. Also check inputs: out is annotated as List[int] but code checks if out is None after type Optional; that's okay. Also guard against consumed > remaining code units.
