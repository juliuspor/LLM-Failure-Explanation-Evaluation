The code fails because the translate() method checks `if out is None:` but its signature types `out: List[int]` and callers may still pass None; however the real bug is in the loop that advances pos when a translator consumes code points. Inside the `for pt in range(consumed):` loop the code calls `pos += char_count(code_point_at(code_units, pos))`. That uses code_point_at(index) which validates index against the length of code_units and raises IndexError if pos equals length. If `consumed` is greater than the remaining code points or if the loop attempts to read past the end (for example when `consumed` counts code points, but pos is advanced by UTF-16 char_count which can be 2 for a surrogate pair), the inner call to `code_point_at` can be given an out-of-range index and throw IndexError. Concretely: suppose pos points to a high surrogate and consumed==1; the loop runs once and calls code_point_at(code_units, pos) which may try to read the low surrogate (index+1) and succeed, returning a codepoint, then char_count returns 2 and pos increments by 2. But if consumed>1 (translator consumed multiple codepoints) the next iteration calls code_point_at with the new pos which may now equal length, causing the IndexError at the check on line 36 ("String index out of range: {index}"). The root cause is mixing two different unit/position domains: `consumed` counts codepoints but the loop advances `pos` by char_count(code_point_at(...)) leading to attempts to read past the end. To fix: advance pos by the number of UTF-16 code units consumed directly (e.g., have translate_codepoint return the number of code units consumed, not codepoints), or compute the codepoint and its char_count once per iteration and guard against reading beyond length. Also validate that `consumed` does not exceed the number of remaining codepoints before looping. This prevents IndexError at line 36 and fixes off-by-units bug.
