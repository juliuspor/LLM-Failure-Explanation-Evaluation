The translate loop increments pos incorrectly in the consumed case: inside the for-loop it calls char_count(code_point_at(code_units, pos)) but code_point_at expects index into code_units and uses code_units[index]; when pos points at a high surrogate, code_point_at returns the combined codepoint and char_count will return 2, but the loop increments pos by 2 each iteration, causing pos to skip forward too far because the loop itself runs 'consumed' times. Specifically, if consumed==1 the loop body increments pos by 2 causing pos to advance by 2 (correct) but for surrogate pairs consumed may be 1 while char_count returns 2 and loop runs once so OK; the real bug is when consumed>1 the loop recomputes code_point_at(code_units, pos) after pos was advanced, leading to reading past end or miscount. Also the char_count should be applied to the codepoint value, not to code_point_at(code_units, pos) which can return a codepoint combining two code units; calling char_count on that value may return 2 but the loop then advances pos twice per iteration causing over-advancement. Additionally, when pos is near end, code_point_at index check (line 35) raises IndexError if pos equals len(code_units) because caller sometimes advances pos incorrectly so code_point_at is called with out-of-range index (see slice_forward lines 35/36). In short: the loop that advances pos for the 'consumed' case advances by variable amounts per iteration while iterating 'consumed' times, which double-counts surrogate length and can call code_point_at with an out-of-range index, causing IndexError at line 36.
