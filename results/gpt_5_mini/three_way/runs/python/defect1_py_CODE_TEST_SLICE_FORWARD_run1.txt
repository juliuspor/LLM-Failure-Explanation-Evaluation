The code miscalculates minutes when hours_offset is negative because it reassigns minutes_offset incorrectly. On lines 383-385 it computes hours_in_minutes = hours_offset * 60, then if hours_in_minutes < 0 it does minutes_offset = hours_in_minutes - minutes_offset; else minutes_offset = hours_in_minutes + minutes_offset. This mutates the input minutes_offset variable and uses subtraction when hours are negative, which yields the wrong total minutes for cases like (-2, -15) and (-2, 15). For (-2, -15): hours_in_minutes = -120, then minutes_offset becomes -120 - (-15) = -105 minutes (which corresponds to -1h45m) instead of the expected -135 minutes (-2h -15m). For (-2, 15): minutes_offset becomes -120 - 15 = -135 (this one is correct), but because the same variable is reused, sign logic is inconsistent. The bug is that the code treats minutes_offset as both an input sign and a quantity to add/subtract from hours_in_minutes; it should compute a new total_minutes = hours_offset * 60 + minutes_offset (taking minutes_offset's sign into account) or, if following the intended rule that a negative hour always makes the whole offset negative, use total_minutes = hours_in_minutes - abs(minutes_offset) for negative hours. Additionally, mutating minutes_offset loses the original value and makes error checks and error messages harder to trace. The incorrect operation happens at lines 383-385 and causes tests like test_negative_hours_negative_minutes and test_boundaries to produce wrong offsets and IDs.
