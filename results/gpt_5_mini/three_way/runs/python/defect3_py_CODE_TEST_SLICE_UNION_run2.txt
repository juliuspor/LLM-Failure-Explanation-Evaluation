The code fails because the loop that advances pos after a translator consumes a codepoint advances by char_count(code_point_at(code_units, pos)) for each pt in range(consumed) while pos hasn't been moved yet, causing code_point_at to be called with the same pos repeatedly and then pos incremented by 1 or 2 depending on current code unit, which incorrectly advances when a surrogate pair is involved. Specifically: after consumed>0, the for-loop uses pos inside code_point_at before incrementing pos; on first iteration it correctly reads the surrogate pair and returns a codepoint requiring char_count=2, then pos is increased by 2. But on subsequent iterations (pt>0) code_point_at is called with the new pos which may be past the surrogate pair or equal to end, potentially raising IndexError (see to_utf16_code_units index check at lines 35–36) or causing wrong advancement. The root bug is misunderstanding of what 'consumed' measures (number of code points consumed by translator) vs how to advance through UTF-16 code units. The loop should compute the char_count once per code point and advance pos by that value per iteration, but it must call code_point_at with the current pos before incrementing; the implementation ends up calling code_point_at repeatedly at positions that move, but if 'consumed' refers to code points consumed starting at original pos, advancing based on code_point_at at shifting pos leads to skipping or out-of-range access when surrogate pairs are at the end or followed by another char. This occurs around lines in translate: pos management after consumed!=0 and triggers IndexError in to_utf16_code_units when code_point_at is called with index >= len(code_units).
