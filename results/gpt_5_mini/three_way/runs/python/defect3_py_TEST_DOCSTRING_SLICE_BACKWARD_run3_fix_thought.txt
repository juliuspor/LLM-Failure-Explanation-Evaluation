The bug is that after translate_codepoint returns 'consumed' which counts codepoints consumed not code units, the loop advances pos by summing char_count(code_point_at(code_units, pos)) per consumed iteration, but it recalculates code_point_at at the same pos repeatedly causing issues when a surrogate pair was consumed: we should compute once per iteration and advance pos by char_count of that codepoint, moving pos forward accordingly. Also handle consumed>0 case by iterating consumed times consuming codepoints safely. The main fix: inside for pt in range(consumed): compute cp = code_point_at(code_units, pos) then pos += char_count(cp). This prevents re-reading same pos. Also ensure when consumed==2 from translator (like CsvEscaper returning 2 for surrogate pair) we properly skip both units. Previously code did that but bug was that code_point_at used pos which might exceed length after increment — but computing code_point_at before increment solves it.
