The code fails because the loop that advances pos after a translator consumes codepoints increments pos incorrectly and can advance beyond the available code_units. Specifically, in the while loop after consumed != 0 the for-loop does: for pt in range(consumed): pos += char_count(code_point_at(code_units, pos)). That calls code_point_at(code_units, pos) while pos may already be equal to length (or near the end), so code_point_at raises IndexError("String index out of range: {index}") when index == len(code_units) or otherwise invalid. The root cause is conflating counts: consumed is a number of codepoints handled by translate_codepoint, but char_count(code_point_at(...)) returns 1 or 2 (UTF-16 code units) for the codepoint at the current pos. By iterating 'consumed' times and each time advancing by char_count of the current codepoint, you can overshoot because you're computing char_count at a position that may already be past the last unit. For example, if pos is at length-1 and consumed==1, the loop calls code_point_at(code_units, pos) which is fine, returns a high surrogate maybe, char_count returns 1, pos increments to length; next iteration (if any) would call code_point_at with index==length -> IndexError. In the failing run the index reported is 2, meaning code_point_at was called with index 2 while code_units length was <=2. The incorrect access happens in the for-loop at the line that calls code_point_at(code_units, pos). To fix: advance pos by summing char_count for each codepoint without calling code_point_at past the end; better: compute the codepoint at current pos once per iteration and use its char_count, or have translate_codepoint return the number of UTF-16 code units consumed (not codepoints), or simply increment pos by the total number of code units consumed (e.g., track units_consumed returned). Concretely: either change translate_codepoint to return units_consumed (1 or 2) and then do pos += units_consumed; or in the for-loop, compute cp = code_point_at(code_units, pos) then pos += char_count(cp) and repeat safely while ensuring pos < length before calling code_point_at. Also add bounds checks before calling code_point_at. These changes prevent calling code_point_at with index == len(code_units) which causes the IndexError.
