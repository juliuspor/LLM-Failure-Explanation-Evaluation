The code fails because in the while loop's for pt in range(consumed): body it calls char_count(code_point_at(code_units, pos)) and then advances pos by that value — but code_point_at already raises IndexError when pos >= length (see slice_union lines 35-36), and more importantly pos is mutated inside the loop while using code_point_at with the updated pos each iteration. If consumed represents number of code points consumed by translate_codepoint starting at the original pos, the loop should advance pos by the number of UTF-16 code units for each code point starting from the original positions; instead, calling code_point_at(code_units, pos) after incrementing pos can attempt to access past the end or mis-count and skip characters. Specifically: (1) If consumed == 1 and the code point at pos is a surrogate pair (two code units), char_count(code_point_at(...)) returns 2, so pos increments by 2 — correct. But the for-loop uses range(consumed) and calls code_point_at with the current pos each iteration; when consumed > 1 and there are surrogate pairs, the repeated calls may read beyond available indices and trigger IndexError at line 36. Also, the loop incorrectly mixes code points and UTF-16 code units: translate_codepoint consumed refers to code points, but pos and code_unit indexing operate on UTF-16 units. The net effect is an out-of-range access at code_point_at(code_units, pos) (slice_union lines 35-36) when the loop tries to advance past end or uses wrong starting pos. The fix: advance pos using char_count on the code point values returned by translate_codepoint or compute total units to skip without re-reading code_point_at at the updated pos; e.g., capture original_pos and for i in range(consumed): cp = code_point_at(code_units, original_pos); original_pos += char_count(cp); pos = original_pos after loop. Alternatively, add bounds checks before calling code_point_at or change translate_codepoint to return units consumed instead of code points.
