The failure arises from treating -0.0 and 0 as different outputs when comparing numeric string representations. In the code, when x is a float equal to an int (x == int(x)) and not infinite/NaN, it converts x to int: value = int(x) (line 186). For x == -0.0, int(x) yields 0, so value becomes 0 and sign information is lost. The block that tries to canonicalize large integers (lines 189–196) only runs when abs(x) >= 100, so it doesn't handle small-magnitude signed zeros. Later, when exp <= 2 the code does self.add(str(value)) (line 201), which emits "0". For other branch when not an exact integer (line 202–203) it emits str(x) which for -0.0 results in "-0.0". The test expected -0.0 but got 0 because the exact-integer branch converted -0.0 to int 0 and emitted "0", losing the negative zero sign. The bug is at lines 185–187 and the choice in line 201 to emit str(value) for integer-equal floats: it discards the negative zero sign and treats -0.0 as 0. Fixes: 1) special-case negative zero before int conversion: if x == 0.0 and math.copysign(1.0, x) < 0: self.add(str(x)) or explicitly emit "-0.0"; 2) or avoid using int(x) for floats that are zero-valued and preserve sign via math.copysign or format(x, ...) so -0.0 remains "-0.0"; 3) add a check before line 185: if x == 0.0 and math.copysign(1.0, x) < 0: self.add(str(x)); return. These changes preserve the negative zero while keeping existing integer normalization for other values.
