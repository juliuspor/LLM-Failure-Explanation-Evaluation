The bug is that in the _max_middle_index block the code fetches the period using self._min_middle_index instead of self._max_middle_index. Specifically, at the start of the block beginning "if self._max_middle_index >= 0:" it does: max_middle_period = self.get_data_item(self._min_middle_index).get_period() (line near the block) — which reads the wrong existing index. As a result the computed max_middle value comes from the existing minimum-middle entry rather than the existing maximum-middle entry, so comparisons (if middle > max_middle) compare against the wrong baseline and the stored _max_middle_index can be set incorrectly (e.g. expected 1 but got 3). The problem occurs where max_middle_period is retrieved; use self._max_middle_index there. Additionally, this leads to incorrect max selection when multiple items are added. Fix: change that line to use self._max_middle_index, i.e. max_middle_period = self.get_data_item(self._max_middle_index).get_period(). Optionally run tests after the change.
