The translate method advances pos incorrectly when ‘consumed’ > 0. Inside the for loop it calls char_count(code_point_at(code_units, pos)) and adds that to pos for each pt in range(consumed). But code_point_at(code_units, pos) reads at the current pos; after the first iteration it advances pos by the full char count (1 or 2), so the next iteration calls code_point_at at the new pos, effectively skipping code points and reading beyond the end. For a surrogate pair followed by another character (e.g. a pair at indices 0-1 and a char at index 2), suppose consumed == 1 (translator handled the pair) but the loop runs once and pos += char_count(code_point_at(code_units, pos)) where pos initially 0, char_count returns 2, so pos becomes 2. That’s correct for one pt, but when consumed > 1 the loop will run additional times and call code_point_at with pos already advanced, causing an index out of range when pos == length or when pos points to the low surrogate. The root cause: the loop is incrementing pos based on code_point_at inside the loop while also iterating consumed times — but consumed already indicates how many code points were consumed and should advance pos by the char_count of each original codepoint, not re-reading at the updated pos. Concretely, at the end of the translate when consumed > 0 the code should advance pos by the total number of UTF-16 code units consumed (sum of char_counts for the original code points), e.g. by iterating over a snapshot index or by using a single advance like pos += consumed if codepoints are single-unit, or by using the returned consumed in codepoint terms and computing unit advance correctly. The IndexError: String index out of range: 2 occurs because the loop calls code_point_at with pos==2 which is equal to length (out of range) when the surrogate pair occupied indices 0 and 1 and there was one following character at index 2; the second loop iteration tries to read past the end. Include concrete lines: the problematic lines are the for pt in range(consumed): pos += char_count(code_point_at(code_units, pos)) — this re-reads and advances pos incorrectly and causes the crash.
