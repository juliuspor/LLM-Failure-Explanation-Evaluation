The code raises ValueError("Minutes out of range: -15") because minutes_offset is mutated before the validation that checks it. Specifically, lines: "minutes_offset = hours_in_minutes - minutes_offset" (for negative hours) or "minutes_offset = hours_in_minutes + minutes_offset" change minutes_offset into a total-minute value. If hours_offset is negative and minutes_offset is positive (e.g. hours_offset=-1, minutes_offset=15), hours_in_minutes becomes -60 and the code sets minutes_offset = -60 - 15 = -75. After that mutation, the subsequent range check "if minutes_offset < 0 or minutes_offset > 59: raise ValueError(...)" (earlier in the function) has already executed once at the top and did not fail, but because the code uses the same variable name for both the input minute component and the computed total minutes, callers see a negative value when later validating or when computing offset, causing the observed ValueError. Root cause: reuse of the minutes_offset variable for two different meanings (input minute component vs total minutes) and performing arithmetic that produces a negative total before the intended bounds check or before returning. The offending lines are the mutation of minutes_offset in the try block (hours_in_minutes < 0 branch and else branch). Fix: use a new variable like total_minutes = hours_in_minutes - minutes_offset (or +) and leave minutes_offset unchanged; then compute offset = safe_multiply(total_minutes, MILLIS_PER_MINUTE) and pass that to for_offset_millis. Also ensure the initial validation of minutes_offset (0..59) stays in place.
