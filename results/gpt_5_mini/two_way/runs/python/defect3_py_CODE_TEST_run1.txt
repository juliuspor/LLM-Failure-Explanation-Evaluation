The code fails because the inner for-loop advances pos incorrectly when consumed > 0. After translate_codepoint reports it consumed N codepoints (surrogate pairs), the loop does: for pt in range(consumed): pos += char_count(code_point_at(code_units, pos)). This calls code_point_at(code_units, pos) repeatedly while pos is not advanced until after char_count, so if code_point_at expects a UTF-16 code unit index and returns a codepoint that might be a surrogate pair, char_count returns 2 for that codepoint; then pos increments by 2, skipping past the next code unit or advancing too far. Specifically, for a surrogate pair at index pos, code_point_at returns the pair's codepoint, char_count returns 2, and the loop increments pos by 2 once for pt=0, which is correct; but on the next iteration pt=1 it calls code_point_at(code_units, pos) again where pos is now after the pair and points to the next character; char_count then may return 1, and pos increments again, so overall pos advanced by 3 although consumed was 2. This miscounts because the inner loop should advance pos by 1 code unit per iteration or should call code_point_at on the original positions, not re-read at the advanced index. The root cause is mixing two different units: consumed counts codepoints, but pos and code_units are UTF-16 code unit indexes. The loop uses char_count(code_point_at(...)) which returns 1 or 2, causing pos to advance by the code unit length of codepoints, but repeating that per consumed causes over-advancement when consumed > 1. The failure occurs in the translate method, in the block starting 'for pt in range(consumed):' where pos is incremented incorrectly; this leads to skipping characters or IndexError on later accesses (seen in tests when a surrogate pair is followed by another character).
