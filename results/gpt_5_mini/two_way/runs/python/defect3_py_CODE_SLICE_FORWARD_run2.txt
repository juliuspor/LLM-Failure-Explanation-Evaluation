The code fails because translate() checks 'if out is None' but type hints show out: List[int]; however real bug is in the surrogate handling loop: after consumed>0 it runs 'for pt in range(consumed): pos += char_count(code_point_at(code_units, pos))'. Here consumed is number of code points consumed, but char_count(code_point_at(code_units,pos)) returns 1 or 2 (UTF-16 code units). However code_point_at reads a codepoint and when pos is at a high surrogate paired with low, code_point_at returns combined codepoint and char_count will return 2, so the loop increments pos by 2 repeatedly 'consumed' times, causing pos to advance too far (double-counting) when consumed>1. Worse, when consumed==1 but char_count returns 2, the loop still works; primary failure occurs when consumed is number of code units rather than codepoints: the contract comment says translators understand codepoints and just took care of a surrogate pair, so consumed is number of code units consumed, but for-loop treats it as number of codepoints. Specifically, at lines shown in slice_forward (lines ~38-44), code_point_at returns high surrogate value if no low surrogate, but logic returns combined codepoint when both; char_count likely returns 2 for supplementary codepoints. If consumed represents code units, then for pt in range(consumed) will iterate too many times, and pos will be advanced by char_count(...) each time causing incorrect pos and potential IndexError or skipping characters. Also initial check 'if out is None' then raising "Writer must not be null" is misleading. Fix: change loop to 'pos += consumed' (advance by number of code units consumed) or have translators return codepoint count and then advance by char_count once per consumed codepoint; clarify contract. Also ensure code_point_at handles indexing and returns int; ensure to_utf16_code_units returns ints in 0..0xFFFF.
