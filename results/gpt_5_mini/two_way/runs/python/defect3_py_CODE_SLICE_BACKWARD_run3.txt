The code fails because the loop that advances pos after consumed>0 calls char_count(code_point_at(code_units, pos)) repeatedly while also incrementing pos inside the loop. This uses code_point_at with pos values that may become out of range because pos is incremented incorrectly: char_count expects a codepoint but code_point_at expects a code unit index; calling code_point_at(code_units, pos) after pos was advanced can raise IndexError from to_utf16_code_units bounds check (see to_utf16_code_units index error at line 35-36). Specifically, in the for pt in range(consumed) loop, pos is incremented by char_count(code_point_at(code_units, pos)) which may be 2 for surrogate pairs, causing pos to jump beyond length or skip positions, and on next iteration code_point_at(code_units, pos) may access an index equal to length, triggering the IndexError. The root cause is mixing units: consumed counts codepoints consumed by translate_codepoint, but the loop increments pos in code units using char_count(code_point_at(...)) which can double-count or run off the end when consumed refers to codepoints consumed starting at the original pos. The error occurs in translate() when a translator handles a surrogate pair (consumed>0) and the subsequent loop advances pos incorrectly, leading to out-of-range index in code_point_at (see to_utf16_code_units bounds check).
