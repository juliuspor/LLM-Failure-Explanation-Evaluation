The code advances pos incorrectly in the for-loop after a translator returns a positive consumed value. It calls char_count(code_point_at(code_units, pos)) inside the loop while simultaneously incrementing pos each iteration; but code_point_at reads from code_units at the current pos which may be at the high surrogate of a surrogate pair. For example, consumed==1 for a surrogate pair means the translator consumed one code point that occupies two code units; the loop runs for pt in range(consumed) (so once), then computes char_count(code_point_at(code_units, pos)) which returns 2 for the surrogate pair, and then pos += 2; but the for-loop also increments pos for each pt? Actually the for-loop body only does pos += char_count(...). There is no extra increment; the real bug is that code_point_at(code_units, pos) expects pos to point to the start of the code point but after the first increment it may point after it when consumed>1 and multiple iterations occur, causing IndexError. More concretely: when a translator returns consumed>0 and the translator handled a surrogate pair (consumed==1), the loop increments pos by char_count of the code point at pos on each iteration, but code_point_at consumes the surrogate pair and advances pos by 2, so on next iteration pos moves past length boundary or double-counts. The actual bug is that pos is advanced using char_count(code_point_at(code_units, pos)) inside the loop, but code_point_at reads using pos and should not be called repeatedly during the loop—pos should be advanced by the number of UTF-16 code units consumed, which is consumed==1 maps to 2 code units for surrogate pair; instead the code mixes counts of code points and code units leading to overrun (IndexError) when encountering surrogate pairs followed by characters.
