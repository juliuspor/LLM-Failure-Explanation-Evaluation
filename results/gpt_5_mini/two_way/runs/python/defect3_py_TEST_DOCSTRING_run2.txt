The code fails because the translation loop advances the index by the number of UTF-16 code units written instead of the number of Unicode code points consumed by translate_codepoint(). Specifically, in translate (method implementing DOCSTRING), after calling translate_codepoint(input_seq, index, out) the code sets index += written, where 'written' is count of UTF-16 code units appended to 'out' (or uses out length difference). For a surrogate pair (a single code point represented as two UTF-16 code units), written == 2 while translate_codepoint consumed only 1 code point. Advancing index by 2 makes the loop skip the character following the surrogate pair or go past the end, causing IndexError when it later accesses input_seq[index] (or code_point_at). The bug occurs in the loop that updates 'index' (around where translate_codepoint is invoked). The root cause: mixing UTF-16 code unit counts with code point counts; index must be incremented by the number of code points consumed (usually 1 unless translate_codepoint consumes more), not by UTF-16 units/length difference. The failing variables/methods: index, out, translate_codepoint(), to_utf16_code_units()/code_point_at() and CsvEscaper.translate_string()/translate. The precise fix: compute consumed_codepoints (1 or more) from translate_codepoint return value or by comparing code point cursor before/after, and advance index by that amount. Alternatively, have translate_codepoint return the number of code points consumed instead of counting appended UTF-16 units.
