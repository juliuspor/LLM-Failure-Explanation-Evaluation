Implement correct conversion in to_utf16_code_units already present; but the bug report says to fix CharSequenceTranslator.translate. The translate function incorrectly advances pos when translators consume codepoints: it uses for pt in range(consumed): pos += char_count(code_point_at(code_units, pos)) which calls code_point_at with same pos repeatedly and may fail if code_units empty earlier. Root cause was to_utf16_code_units, but that function is implemented. Safer fix: when consumed>0, advance pos by sum of char_count for each consumed codepoint starting from current pos, but we can instead iterate consumed times computing cp=code_point_at(code_units, pos) then pos+=char_count(cp). That avoids repeated same-pos evaluation. Also ensure pos increment uses local variable. Update translate accordingly.
