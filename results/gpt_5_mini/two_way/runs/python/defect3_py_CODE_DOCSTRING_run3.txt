The code fails because it sometimes advances pos incorrectly when 'consumed' > 0. Specifically, in the for loop starting at the end, it iterates 'for pt in range(consumed): pos += char_count(code_point_at(code_units, pos))'. Here code_point_at expects an index into code_units (UTF-16 code units), but char_count(code_point_at(...)) returns 1 or 2 (number of UTF-16 code units for that code point). The loop uses the same pos both to read the code point and to advance by the number of code units for that code point; this double-uses pos while incrementing it inside the loop, which causes code_point_at to be called with updated pos values and can skip or misread code points. More critically, if consumed represents the number of code points consumed by translate_codepoint, the code should advance pos by the total number of UTF-16 code units corresponding to those consumed code points, not repeatedly re-read at the moving pos. The current approach also can call code_point_at beyond the end of code_units when surrogate pairs are involved, causing IndexError or invalid surrogate handling. In short: the loop incorrectly advances pos by repeatedly calling code_point_at at the changing position instead of summing the char counts of the originally consumed code points; this leads to skipping code points, wrong offsets, and potential out-of-range access.
