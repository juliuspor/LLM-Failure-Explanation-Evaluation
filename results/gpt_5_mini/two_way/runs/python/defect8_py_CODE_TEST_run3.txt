The code rejects locales that use an empty country with a double underscore (language__variant), e.g. "fr__POSIX" or "en__UNIX". Root cause: the conditional that validates length is wrong: `if length != 2 and length != 5 and length < 7:`. For an input like "fr__POSIX" length==8, the condition evaluates as: (8 != 2) True, (8 != 5) True, (8 < 7) False, combined with AND yields False so it does not raise there — that part is okay. The actual failure occurs later when validating the country letters at positions 3 and 4. After checking language letters, the code assumes position 2 is '_' and then reads ch3 = locale_str[3] and ch4 = locale_str[4] and enforces they are uppercase letters (`if ch3 < 'A' or ch3 > 'Z' or ch4 < 'A' or ch4 > 'Z': raise`). For a double underscore form like "fr__POSIX", locale_str[3] is '_' not an uppercase letter, so the function raises ValueError at that check. In other words, the code does not account for the special case where the country is empty (positions 3-4 are both '_' or position 2 and 5 are underscores), so it incorrectly requires uppercase letters at ch3/ch4. Location: in to_locale after `if locale_str[2] != '_'` check, the lines reading ch3/ch4 and the uppercase validation cause the failure. Fix summary: detect and allow empty country when locale_str[3] == '_' (i.e., language__variant). Concretely: after confirming locale_str[2] == '_', check if locale_str[3] == '_' -> then treat country as "" and set variant = locale_str[4+?] (adjust indices), or follow Java's Locale parsing: if length >= 7 and locale_str[3] == '_', then return Locale(lang, "", locale_str[4+...] ). Otherwise validate uppercase ch3/ch4 and continue as now.
