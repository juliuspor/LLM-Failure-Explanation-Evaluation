The provided code raises IndexError because pos is advanced incorrectly after translate_codepoint consumes a surrogate pair. In translate(), when consumed != 0 the loop: for pt in range(consumed): pos += char_count(code_point_at(code_units, pos)) calls code_point_at(code_units, pos) while pos still points at the high surrogate. char_count(code_point_at(...)) returns 2 for a surrogate pair, so on the first iteration pos increases by 2, skipping past both code units; but the for loop repeats for each pt, causing pos to be advanced multiple times (often more than the number of code units consumed). Specifically, if consumed == 1 for one codepoint (a surrogate pair), the body still uses char_count which yields 2, so pos increments by 2 once — that's OK. But when consumed >1 or when consumed counts code points, the loop increments by char_count(code_point_at(code_units, pos)) repeatedly while pos changes inside the loop; after the first increment pos may point beyond the available code_units length and the next call to code_point_at(code_units, pos) raises IndexError (see to_utf16_code_units index check at lines 35-36). The root cause: mixing two units of measurement — consumed (number of Unicode code points) vs pos (index into UTF-16 code units) — and calling code_point_at(code_units, pos) inside the loop while advancing pos with char_count leads to overshooting the buffer. The problematic lines are the for pt in range(consumed): pos += char_count(code_point_at(code_units, pos)) loop near the end of translate(); it should advance pos by the total number of UTF-16 code units consumed (i.e., sum of char_count for each code point starting from the original pos) or by using the return value from translate_codepoint to indicate units consumed in code units rather than code points. In short: wrong unit (code points vs. code units) and repeated calls using an index that becomes out of range cause the IndexError.
