The code fails because get_paint performs integer conversion before clamping and raises ValueError when input is out of bounds rather than clamping. Specifically: In GrayPaintScale.get_paint(value) the sequence is: compute scale = (value - self.lower) / (self.upper - self.lower) (likely line ~XX), then compute gray = int(scale * 255.0). For inputs below lower (e.g. -0.5 with default lower=0.0) scale becomes negative and int() produces a negative gray (e.g. -127), then later there is a check that raises ValueError if gray < 0 or gray > 255 (from the docstring "Raises: ValueError...") — so the method raises instead of clamping. Likewise for values > upper (1.5) scale >1 producing gray >255 and triggering the same ValueError. The root cause is wrong order: clamping must occur on scale (or input value) before conversion to gray. Also test_custom_bounds shows same failure when using custom lower=10.0 upper=20.0 and input 5.0 (scale <0). The specific variables/methods involved: get_paint, value, self.lower, self.upper, scale, gray. The failure occurs at the lines that compute gray and then validate it; because validation raises instead of clamping, the tests expecting (0,0,0) or (255,255,255) fail. Fix: clamp value to bounds with value = max(self.lower, min(self.upper, value)) before computing scale, or clamp scale = max(0.0, min(1.0, scale)) before int conversion. Then compute gray = int(scale * 255.0) and return (gray, gray, gray).
