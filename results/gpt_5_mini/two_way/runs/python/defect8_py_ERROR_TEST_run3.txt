The tests pass a locale string "fr__POSIX" but the implementation of LocaleUtils.to_locale incorrectly rejects it. Specifically, the parser expects either: a 2-letter language only (positions 0-1 lowercase), or language + underscore + 2-letter country (positions 3-4 uppercase), or language_country_variant with a single underscore separating country and variant. It does not handle the case when the country is empty and two underscores appear consecutively (language__variant). The failing check occurs when the code validates the character at index 3: it assumes a country letter must be present and uppercase, so when it finds '_' at position 3 the code raises ValueError("Invalid locale format") or similar. In short: LocaleUtils.to_locale wrongly enforces a non-empty country when parsing the language__variant form, causing ValueError for inputs like "fr__POSIX". The root cause is the parsing/validation logic around indexes 2-4 — it should allow an underscore at index 3 to indicate an empty country and accept the variant starting at index 4.
