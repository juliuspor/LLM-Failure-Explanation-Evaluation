The code fails because index advancement in the translator does not correctly account for UTF-16 surrogate pairs, causing code_point_at to be called with an out-of-range index (see slice check at lines 35-36). Specifically: In CsvEscaper.translate_string (the translator's main loop) the loop variable 'index' is incremented by 1 unconditionally after processing a code point. For characters outside the BMP (e.g. U+1F630), Python string indexing yields a single code point but the translator uses helpers to operate on UTF-16 code units via to_utf16_code_units and code_point_at. When code_point_at detects a high surrogate at position i it reads the low surrogate at i+1; if the translator then increments index by 1 instead of 2, the next iteration calls code_point_at with index == i+1, which is the low surrogate. code_point_at expects to be called only at code point boundaries and when given the low surrogate index it tries to look up a pair and may perform index checks like the one at lines 35-36, finding index >= len(code_units) or attempting to read beyond the end, raising IndexError: "String index out of range: {index}". Where it occurs: the failing check is in code_point_at around lines 35-36 (the slice boundary check). Why it happens: incorrect advancement of 'index' after handling a surrogate pair (off-by-one) leaves the loop pointing at the low surrogate or past the end. Reproduction: the test test_translate_surrogate_pair_with_suffix uses input '\U0001F630A' so the translator must consume the surrogate pair as a single code point and then proceed to the following 'A'; because it only advances by 1 it later calls code_point_at with index equal to the low surrogate's position or len(code_units), triggering the IndexError. Fix steps: 1) In CsvEscaper.translate_string, when code_point_at returns a supplementary code point (i.e. it used a surrogate pair), advance index by 2 code units instead of 1; if using UTF-16 helpers, detect whether the code point was from a pair (e.g. by checking returned width or comparing original code unit to high surrogate range) and increment accordingly. 2) Alternatively operate on Python code points (regular Python str) rather than manual UTF-16 code units so indexing matches Python's representation and no manual surrogate handling is needed. 3) Add unit tests (like the provided suite) to cover emoji followed by characters and surrogate at end.
