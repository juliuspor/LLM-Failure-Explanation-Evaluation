The code fails because the loop that advances pos after a translator consumes codepoints uses char_count(code_point_at(code_units, pos)) while pos already points inside the consumed surrogate pair; calling code_point_at with that pos can raise IndexError when pos is at the last code unit and the function attempts to access the next unit. Specifically, at the end of translate(), inside the for pt in range(consumed) loop, pos is incremented by char_count(code_point_at(code_units, pos)). When consumed > 0 and pos is at a high surrogate followed by a low surrogate, char_count returns 2, but code_point_at may access code_units[pos+1]. If pos is near length (e.g., pos == length-1) this leads to an IndexError "String index out of range" thrown from to_utf16_code_units/code_point_at (see lines 35-36 in the slice). Root cause: using code_point_at and char_count with a pos that can be advanced past the buffer without proper bounds checks. Fixes: (1) Instead of calling code_point_at inside that loop, advance pos by 1 for each consumed code unit: pos += 1 inside the for-loop, since consumed is already the number of UTF-16 code units handled; or (2) change consumed to mean number of codepoints and adjust logic accordingly; or (3) add explicit bounds checks before calling code_point_at to avoid indexing past the end. Also ensure translate_codepoint returns number of UTF-16 code units consumed (1 or 2) so the loop should do pos += consumed rather than iterating and calling char_count.
