The failure is caused by reassigning the input parameter minutes_offset when computing total minutes. In for_offset_hours_minutes, lines: hours_in_minutes = hours_offset * 60; if hours_in_minutes < 0: minutes_offset = hours_in_minutes - minutes_offset else: minutes_offset = hours_in_minutes + minutes_offset. This mutates minutes_offset and mixes sign semantics incorrectly: for negative hours with negative minutes (e.g. -2, -15) the code computes minutes_offset = -120 - (-15) = -105, which yields -105 minutes (i.e. -1h45m) instead of the expected -135 minutes (-2h15m). Similarly, for (-2,15) it does -120 - 15 = -135 which is correct, but for (0, -15) the earlier check hours_offset == 0 bypasses sign handling; more generally, overwriting minutes_offset causes confusion and wrong results. Also the conditional uses hours_in_minutes < 0 rather than hours_offset < 0 which is equivalent but unclear. The root bug: subtracting minutes_offset when hours are negative should use addition of the absolute minutes (total = hours_in_minutes - minutes) is wrong when minutes_offset is already negative; it should compute total_minutes = hours_offset * 60 + minutes_offset (simple sum) and then validate sign combinations if required. The incorrect arithmetic occurs at the lines that reassign minutes_offset and leads tests like test_negative_hours_negative_minutes failing (computed offset -105 minutes instead of -135).
