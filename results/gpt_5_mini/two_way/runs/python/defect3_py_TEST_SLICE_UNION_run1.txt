Specifically, at lines 35-36 the function checks 'if index < 0 or index >= len(code_units):' using len(code_units) (UTF-16 length). For strings containing a surrogate pair (emoji) the UTF-16 length is larger than the Python str length, so callers that advance the index by 1 when encountering a surrogate pair end up passing an index that points between the two UTF-16 code units. At lines 39-43 the function detects a high surrogate and reads the low surrogate at index+1 to form a code point; but higher-level code increments its index by 1 (treating the surrogate pair as one Python character) and later passes that incremented index again to code_point_at, which then may land on the low surrogate code unit alone. The root cause is treating Python string indexes as if they were UTF-16 code unit indexes; the fix is to either operate consistently on Python code point indexes (use ord() and slicing) or to ensure index advances and bounds checks are done against the UTF-16 code_units list when calling to_utf16_code_units()/code_point_at().
