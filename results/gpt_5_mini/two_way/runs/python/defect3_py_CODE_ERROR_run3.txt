The code fails because the loop that advances pos after a translator consumes codepoints uses char_count(code_point_at(code_units, pos)) inside the for-loop which recomputes and advances pos incorrectly and can read past the end. Specifically, when consumed > 0 the for pt in range(consumed): body calls code_point_at(code_units, pos) but pos may already be at or beyond length, causing code_point_at to index out of range. The root cause is mixing two different units: consumed counts codepoints consumed by translate_codepoint, but pos is an index into code_units (UTF-16 code units). The loop should advance pos by the number of code units corresponding to each consumed codepoint (1 for BMP, 2 for surrogate pairs) without calling code_point_at with a pos that might be at length. Concretely, on the last iteration when a translator reports consumed=1 for a codepoint represented by two code units, the inner loop may call code_point_at when pos == length-1, and code_point_at tries to access code_units[pos+1] and throws IndexError: String index out of range: 2. Fixes: (1) change the advancement loop to compute the code unit count by inspecting the current code unit(s) directly, e.g. if is_high_surrogate(code_units[pos]) and pos+1<length and is_low_surrogate(code_units[pos+1]) then advance by 2 else by 1; (2) or have translate_codepoint return the number of code units consumed instead of codepoints; (3) or ensure code_point_at is only called when pos < length and handles end-of-input safely. These fixes ensure pos is not used to read beyond code_units length.
