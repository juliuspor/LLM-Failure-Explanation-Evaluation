The code increments the index by 1 after handling a surrogate pair, not by 2 code units. In translate_string (and the helper code_point_at/to_utf16_code_units), when the current character is a high surrogate and forms a valid surrogate pair with the next low surrogate, the routine reads the pair as a single code point but then advances the loop index only by one. That leaves the loop to next iterate with i pointing at the low surrogate. When the code then tries to access the next character or form a pair again, it may read past the end of the string (for example when the surrogate pair is followed by one ASCII char). This produces IndexError: String index out of range: 2. The failure occurs in the loop that walks the UTF-16 code units (translate_string) when computing the next index after consuming a surrogate pair; variable i (or index) should be incremented by 2 to skip both code units of the surrogate pair. Specifically, check the lines where code_point_at is used and where the loop does index += 1 — change to index += (1 if code_point < 0x10000 else 2) or otherwise advance by the number of UTF-16 code units consumed. Also ensure boundary checks before accessing s[i+1] to form a surrogate pair to avoid reading past the string end.
