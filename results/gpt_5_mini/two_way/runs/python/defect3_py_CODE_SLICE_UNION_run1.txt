The code fails because the translate method checks 'if out is None' but its signature types out: List[int] and later uses out.extend — not None; however real bug is in surrogate handling loop: inside for pt in range(consumed): pos += char_count(code_point_at(code_units, pos)) it calls code_point_at with the current pos but char_count expects a codepoint, and code_point_at returns either a full codepoint (combining surrogate pair) or a single code unit. Then char_count(code_point_at(...)) returns 2 for non-BMP and 1 otherwise. But when code_point_at returns a combined codepoint for a surrogate pair, pos should be advanced by number of code units (1 or 2). The bug is that char_count is being passed a codepoint when it expects a code unit value to compute UTF-16 char count; more concretely char_count probably returns 2 only for values >= 0x10000, but code_point_at already consumes two units, so advancing by char_count(code_point_at(...)) inside a loop over consumed causes pos to be advanced too much or incorrectly and can index out of range. Specifically if consumed==1 and code_point_at at pos is a high surrogate without pair, code_point_at returns high surrogate value (0xD800..0xDBFF), char_count(high) may return 1 leading to pos increment by 1, ok. But when consumed>0 from translate_codepoint indicating it consumed a surrogate pair, the loop runs twice (pt range(consumed)) and each iteration calls code_point_at at same starting pos; after first iteration pos increases by char_count(code_point_at(...)) which returns 2, pos moves past the pair; second iteration calls code_point_at at new pos, which is the next code unit, leading to skipping or IndexError. Root cause: misuse of consumed and char_count; should simply pos += 1 for each unit consumed, or pos += consumed at end. The error occurs in translate method in the for-loop updating pos (around lines shown) leading to skipping or IndexError when handling surrogate pairs.
