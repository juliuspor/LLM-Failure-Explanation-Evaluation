The code fails because when adding elements to an array the implementation creates or returns a plain Object[] (heterogeneous list) instead of an array typed to the expected element type. The test calling ArrayUtils.add(None, None, expected_type=str) expects a String[] (in Python terms a list of str) but receives an Object[] (list of generic objects). At runtime the test harness (JVM bridge) tries to cast the returned Object[] to String[] and throws ClassCastException: [Ljava.lang.Object; cannot be cast to [Ljava.lang.String;. Root cause: ArrayUtils.add constructs the new array with component type Object (or uses a generic/new Object[]), losing the expected_type information or the component type of the original array. Specific places to check: the code path where the input array is None and element is None (both null), and the branch that creates a new array of length 1 — it should use expected_type (or the array's component type) when creating the new array, not Object.class. Also check the code that copies elements into the new array (it may use Arrays.copyOf with Object[].class). Steps to fix: 1) In ArrayUtils.add, when creating the result array, determine component_type = expected_type if provided, else (if input array not None) input_array.getClass().getComponentType(), else (if element not None) element.getClass(), otherwise default to Object.class only if no expected_type and element is also None. 2) Use java.lang.reflect.Array.newInstance(component_type, new_length) to create an array with the correct component type. 3) Copy existing elements into the new array (System.arraycopy) and set the new element. 4) Ensure method signatures and generic handling preserve runtime component type instead of returning Object[]. This will prevent ClassCastException in callers expecting String[] when adding nulls.
