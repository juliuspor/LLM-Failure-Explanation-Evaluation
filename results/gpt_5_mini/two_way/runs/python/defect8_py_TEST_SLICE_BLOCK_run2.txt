The failure occurs in LocaleUtils.to_locale when parsing a 5-character locale like "fr__POSIX" because the code assumes position 3 and 4 are letters of an uppercase country code. Specifically, at lines 229-230 the code does: if ch3 < 'A' or ch3 > 'Z' or ch4 < 'A' or ch4 > 'Z': raise ValueError(...). For an input like "fr__POSIX" the string positions are: index 0='f',1='r',2='_',3='_',4='P'... The code reads ch3 = locale_str[3] and ch4 = locale_str[4] and then enforces both ch3 and ch4 be uppercase letters. ch3 is '_' (the empty country separator), so the check fails and a ValueError is raised with the misleading message "Invalid locale format". Root cause: the validation at lines 229-230 doesn't allow the special case of an empty country (double underscore) followed by a variant. Fix steps: 1) In to_locale, detect the pattern language + '__' + variant by checking for locale_str[2]=='_' and locale_str[3]=='_' (or checking for two consecutive underscores) before enforcing uppercase country-letter checks. 2) When double underscore is present, set country = "" and variant = locale_str[4:] (ensure variant length and characters allowed). 3) Only apply the ch3/ch4 uppercase-letter validation when parsing a two-letter country (i.e., single underscore and enough length). Also update unit tests or add explicit branch for empty country. This change resolves the failure for inputs like "fr__POSIX" without affecting normal language_country parsing.
