The bug arises because translate_codepoint returns counts in codepoints, but caller advances pos by UTF-16 code units via char_count(code_point_at(...)) inside a loop for consumed; however translate_codepoint in CsvEscaper returns 2 when it consumed a surrogate pair but pos is advanced twice by char_count of code points causing overshoot. Simplest fix: when consumed>0, advance pos by counts of UTF-16 code units consumed, not by iterating consumed times using code_point_at. So change post-consumed handling to sum char_count for each codepoint consumed starting from current pos but using a loop that increments by char_count of each codepoint, avoiding calling code_point_at with out-of-range index. Also handle case where consumed can be negative or too large; ensure we break if pos>=length.
