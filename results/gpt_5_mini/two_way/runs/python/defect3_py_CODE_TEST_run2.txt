The code advances pos incorrectly inside the final for-loop: it calls char_count(code_point_at(code_units, pos)) repeatedly while incrementing pos, but code_point_at expects a position in the UTF-16 code_units array. After the translator consumed a codepoint (possibly a surrogate pair), the loop intends to skip the same number of UTF-16 code units as 'consumed' indicated. Instead, inside the loop it recalculates char_count(code_point_at(code_units, pos)) using the updated pos each iteration, which can double-count surrogate pairs or read past the end. Specifically, when a surrogate pair is present, char_count(code_point_at(code_units, pos)) returns 2 for the first iteration, so pos is advanced by 2 immediately, and the outer loop's for-range will then attempt another iteration if consumed>1, causing pos to be advanced again or to point beyond length, producing IndexError or skipping characters. The bug occurs in the lines:

    for pt in range(consumed):
        pos += char_count(code_point_at(code_units, pos))

Why it happens: 'consumed' already indicates how many UTF-16 code units to skip (1 for BMP, 2 for a surrogate pair). But the inner loop advances using char_count of the codepoint at the current pos, which for surrogate pairs returns 2 and causes the inner loop to over-advance. Where: inside translate(), after a non-zero 'consumed' return from translate_codepoint().

Fix steps:
1) Replace the inner loop with a single advance of pos by 'consumed': pos += consumed
2) Or, if consumed represents codepoints rather than code units, compute a single unit count before the loop: units_to_advance = sum(char_count(code_point_at(code_units, p)) for p in starting positions) and then add that once. But the simplest correct fix is: pos += consumed

This ensures the index advances exactly the number of UTF-16 code units the translator consumed and avoids reading past the end or skipping characters.
