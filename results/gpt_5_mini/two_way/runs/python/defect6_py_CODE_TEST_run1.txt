The code fails because it treats -0.0 as equal to int(-0.0) and then converts it to an int, losing the negative sign. Specifically, in add_number: the branch if x == int(x) and not math.isinf(x) and not math.isnan(x): is true for x == -0.0 because -0.0 == 0.0 and int(-0.0) == 0, so value = int(x) sets value to 0 (positive zero). Later the code uses value (and possibly mantissa) and ends up calling self.add(str(value)) which yields "0". That produces "0" instead of "-0.0" (test_negative_zero expects "-0.0"). The loss occurs at the line value = int(x) and later str(value). The root cause: using equality x == int(x) to decide integer representation collapses negative zero to zero; negative sign must be preserved by checking for negative zero with math.copysign or inspecting bits (e.g., math.copysign(1.0, x) < 0 or struct.unpack) before converting to int. Also when printing integers, the code should special-case negative zero (CodeConsumer.is_negative_zero) to print "-0.0" instead of "0".
