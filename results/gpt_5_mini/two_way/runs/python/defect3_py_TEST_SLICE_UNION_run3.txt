The code fails because to_utf16_code_units builds a list of UTF-16 code units, but code_point_at (lines 35-44 in slice) treats the list indices as code unit indices; when encountering a surrogate pair it returns a combined code point but callers advance index by 1 instead of 2. Specifically, at line 39 the function checks high surrogate and looks at index+1 for low surrogate, and at line 43 returns the combined code point, but elsewhere (CsvEscaper.translate_string) the loop increments its index by 1 unconditionally. As a result, when a surrogate pair is seen followed by another character (e.g. input "\U0001F630A"), the loop advances only one code unit past the pair, causing the next call to code_point_at to use an index that points at the low surrogate (a code unit between 0xDC00 and 0xDFFF). code_point_at then attempts to interpret that low surrogate as a high surrogate (line 39), doesn’t find a valid following low surrogate, and returns the low surrogate value. The translator then may try to advance again and eventually index arithmetic exceeds len(code_units), triggering the IndexError at line 36. Root cause: mismatch between how many code units code_point_at consumes (it returns a combined code point but does not communicate that it consumed two code units) and how the caller advances its index. Fix: make code_point_at return both the code point and the number of code units consumed (1 or 2), or have translate_string detect high surrogates and increment index by 2 when appropriate.
