The code fails because it reassigns the input parameter minutes_offset to hold a computed total-minute value, then later treats it as minutes when multiplying to milliseconds. Specifically, at lines 383 and 385 the code does: minutes_offset = hours_in_minutes - minutes_offset or + minutes_offset, which turns minutes_offset from a 0..59 value into a total minutes count (possibly negative) but still named minutes_offset. This is confusing but not the core bug. The real bug is sign handling when hours_in_minutes is negative: when hours_in_minutes < 0, the code does minutes_offset = hours_in_minutes - minutes_offset, which makes the total minutes more negative than intended. For example, for hours_offset = -1 and minutes_offset = 30, hours_in_minutes = -60, then minutes_offset becomes -60 - 30 = -90 (i.e. -1:30 becomes -1:30 as -90 minutes, but standard expectation is -1:30 should be -90 minutes; however elsewhere the check minutes_offset < 0 or >59 forbids negative minutes input). The actual failing scenario is when minutes_offset is negative or when hours_offset is -23 and minutes_offset non-zero: multiplying minutes_offset (now a large magnitude) by MILLIS_PER_MINUTE can overflow safe_multiply and raise ArithmeticError. The code also mistakenly allows hours_offset of 23 and minutes_offset up to 59 which yields total 1439 minutes OK; but the error arises because minutes_offset parameter is validated as 0..59 but then reused causing confusion. The root cause: variable shadowing and incorrect sign handling converting hours and minutes to total minutes leads to wrong arithmetic and potential overflow; safe_multiply raises ArithmeticError which is caught and rethrown as ValueError("Offset is too large"). The failure occurs at the lines where minutes_offset is reassigned (383,385) and when FieldUtils.safe_multiply is called (386).
