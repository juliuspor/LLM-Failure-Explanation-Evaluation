What happened (where it fails)
- The exception is raised in GrayPaintScale.get_paint. Specifically at the range check that raises ValueError:
  if g < 0 or g > 255:
      raise ValueError(f"Color parameter outside of expected range: Red Green Blue")
- That line runs when you call gps.get_paint(-0.5) in the test.

Why it fails (root cause)
- You clamp the input into v (lines 55–57): 
    v = max(value, self._lower_bound)
    v = min(v, self._upper_bound)
  but then you compute g using the original value variable, not the clamped v (line 59):
    g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- For value = -0.5 and the default bounds 0.0–1.0, the unclamped expression gives (−0.5 − 0.0)/(1.0 − 0.0) * 255 = −127.5 → int(...) = −127, so g < 0 and the ValueError is raised.
- In short: you intended to clamp the input to the bounds but then forgot to use the clamped variable v when computing the gray level.

Exactly where to change it (concrete fix)
- In GrayPaintScale.get_paint, replace the line that computes g so it uses v (the clamped value) instead of value. For example (line numbers approximate to your code):
    # current (faulty) line (around line 59):
    g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
    # change to:
    g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

- Optionally keep the same range check after that. With this change, gps.get_paint(-0.5) will clamp to 0.0, compute g = 0, and return (0, 0, 0) as expected.

Why this fix works
- Using v ensures values below lower_bound map to the lower-bound gray level (0) and values above upper_bound map to the upper-bound gray level (255), matching the intent of the clamping code and preventing negative or >255 RGB components.
