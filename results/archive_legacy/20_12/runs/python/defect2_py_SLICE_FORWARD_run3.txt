What happened (where it failed)
- The exception is raised inside GrayPaintScale.get_paint at the line that checks g’s range. In this code that is the block starting with
  if g < 0 or g > 255:
      raise ValueError(...)
- The test calls gps.get_paint(-0.5). The code clamps the input into local variable v, but then computes g using the original parameter value (value) instead of the clamped v. Because value is -0.5, the computed g becomes negative, triggering the ValueError.

Why it happened (root cause)
- In get_paint you intended to clamp the input to the scale bounds:
    v = max(value, self._lower_bound)
    v = min(v, self._upper_bound)
  but you never use v in the grayscale calculation. Instead you use value:
    g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
  So when value is out of bounds (like -0.5), the un-clamped expression yields a negative g and the range check fails.

Exact location
- Method: GrayPaintScale.get_paint
- Lines:
  - clamping: v = max(value, self._lower_bound); v = min(v, self._upper_bound)
  - buggy calculation: g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
  - range check immediately after that raises ValueError

How to fix it (step‑by‑step)
1. Use the clamped variable v in the calculation instead of value:
   Replace the line
     g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
   with
     g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

2. (Optional) For clarity, rename v to clamped or similar:
   clamped = max(min(value, self._upper_bound), self._lower_bound)
   g = int((clamped - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

3. Run tests (the provided test should then pass):
   gps = GrayPaintScale()
   color = gps.get_paint(-0.5)
   assert color == (0, 0, 0)

Why that fix works
- Using the clamped value ensures the normalized fraction is always in [0,1], so g is always between 0 and 255 and the ValueError will not be raised for inputs outside the bounds.

No other hidden issues
- The ValueError message string is generic but not the cause. The root cause is simply using value instead of the clamped variable v.
