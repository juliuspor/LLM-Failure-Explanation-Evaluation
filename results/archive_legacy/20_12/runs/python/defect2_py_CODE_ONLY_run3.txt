What happened (where it failed)
- The exception is raised inside GrayPaintScale.get_paint. Specifically the ValueError is raised at the check that validates the computed grayscale integer g.
- In your test the call gps.get_paint(-0.5) triggers the failure.

Why it failed (root cause)
- You intended to clamp the input value to the bounds (you compute v) but you never use v when computing g.
- Instead you compute g from the original value argument:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- For the test value -0.5 (with default lower_bound=0.0), this yields a negative ratio, so g < 0. The range check then raises the ValueError:
  "Color parameter outside of expected range: Red Green Blue"
- So the code clamps to v but then ignores v; the ignored clamp is the direct cause.

Where in the code (line references relative to snippet)
- In get_paint():
  - v = max(value, self._lower_bound)  (clamping starts here)
  - v = min(v, self._upper_bound)     (clamping ends here)
  - g = int((value - self._lower_bound) / ... )  <-- this line uses value instead of v and is the bug
  - if g < 0 or g > 255: raise ValueError  <-- exception triggered here for -0.5

How to fix it (concrete steps)
1. Use the clamped value v when computing g. Replace the g line with:
     g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
2. (Optional) To be extra safe against rounding issues, you can also clamp g into [0, 255] before the range check:
     g = max(0, min(255, g))
   then the explicit ValueError check becomes unnecessary (or you can keep it).
3. Run your test:
     gps = GrayPaintScale(); color = gps.get_paint(-0.5); assert color == (0, 0, 0)

That change ensures the calculation uses the clamped value and prevents negative or >255 g for inputs outside the bounds.
