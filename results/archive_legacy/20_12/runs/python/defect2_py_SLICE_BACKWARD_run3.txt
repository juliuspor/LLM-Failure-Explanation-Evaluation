What happened (where it fails)
- The exception is raised inside GrayPaintScale.get_paint at line 76 (the if g < 0 or g > 255 check).
- The test calls gps.get_paint(-0.5). get_paint clamps the input into v on lines 68–71, but then computes g using the original un-clamped variable value on line 73:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
  Because value is -0.5 and lower_bound defaults to 0.0, that formula produces a negative g, so the range check on line 76 triggers and raises ValueError.

Why this happens (root cause)
- You intended to clamp the incoming value into the bounds (you compute v) but you never use v when calculating g. Instead you use the un-clamped parameter value. So values below lower_bound or above upper_bound yield g outside 0–255 and cause the exception even though you already computed the clamped v.

Exact lines involved
- Clamping (does nothing because you don’t use it): lines 68–71
  v = max(value, self._lower_bound)
  v = min(v, self._upper_bound)
- Computation using the wrong variable: line 73
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- Range check and error: lines 76–77
  if g < 0 or g > 255:
      raise ValueError(...)

How to fix it (concrete steps)
1. Use the clamped variable v when computing g. Change line 73 to:
   g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
2. (Optional) For clarity, rename v to clamped or value_clamped:
   clamped = max(value, self._lower_bound)
   clamped = min(clamped, self._upper_bound)
   g = int((clamped - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
3. You may also replace the two-step clamp with:
   clamped = min(max(value, self._lower_bound), self._upper_bound)

After that change the test gps.get_paint(-0.5) will compute g using the clamped value 0.0, producing g == 0 and returning (0, 0, 0) instead of raising the ValueError.
