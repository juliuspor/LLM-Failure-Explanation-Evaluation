What happened (where it failed)
- Failure occurs in GrayPaintScale.get_paint (lines shown). Specifically the ValueError is raised at line 76 because the computed g is outside 0–255.

Why it happened (root cause)
- You clamp the input value into variable v (lines 68–69), but you never use v when computing g. Instead you compute g from the original value (line 73):
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- When the caller passes a value below lower_bound (e.g. value = -0.5 in the test), the formula produces a negative g (or >255 for values above upper_bound). Because g is negative, the range check at line 76 triggers and raises the ValueError even though you attempted to clamp the value.

Exactly where to change (step-by-step fix)
1. Use the clamped variable v in the computation of g instead of value. Replace line 73 with:
   g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
2. (Optional) You may also want to ensure rounding/limits are robust by bounding g to [0, 255] after computing:
   g = max(0, min(255, int(...)))
   and then you can remove or keep the explicit range check.

Why this fix works
- Using v ensures values below lower_bound map to 0 and values above upper_bound map to 255, so g cannot be outside 0–255 and the ValueError will not be raised for out-of-range inputs that were meant to be clamped.

Where it mattered in your test
- The test gps.get_paint(-0.5) produced g < 0 because the clamp was ignored; using v fixes that and returns (0, 0, 0) as expected.
