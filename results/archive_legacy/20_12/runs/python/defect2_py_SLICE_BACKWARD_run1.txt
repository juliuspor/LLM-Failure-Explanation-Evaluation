What happened (where it failed)
- The ValueError is raised in GrayPaintScale.get_paint, at the range check around line 76: 
  if g < 0 or g > 255: raise ValueError(...)
- The test calls gps.get_paint(-0.5). The code clamps the input into v but then does not use v when computing g: the computation on line 73 uses the original value variable instead of the clamped v:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
  (see lines 66–73 where v is set but never used)

Why that causes the error
- For value = -0.5, the code sets v = max(value, lower_bound) so v becomes 0.0 (lower_bound).
- But because g is computed from the un-clamped value (-0.5), the normalized fraction becomes negative and g is less than 0.
- The subsequent check (line 76) detects g < 0 and raises the ValueError you see.

Exactly where to change (how to fix)
1. In GrayPaintScale.get_paint, replace the use of value when computing g with the clamped v. Change this line:
   g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
   to:
   g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
2. (Optional) If you want consistent rounding behavior, consider using round(...) before int(...) or ensure the conversion choice is intentional.
3. Run the test again: gps = GrayPaintScale(); color = gps.get_paint(-0.5); assert color == (0, 0, 0) — it will pass after the change.

Summary
- Root cause: clamped value v is computed but not used; g is calculated from the original (possibly out-of-range) value, triggering the range check and ValueError.
- Fix: compute g using v (the clamped value) instead of value.
