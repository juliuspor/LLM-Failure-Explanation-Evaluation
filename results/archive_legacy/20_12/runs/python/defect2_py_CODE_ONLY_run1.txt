What happened (where it fails)
- The exception is raised inside GrayPaintScale.get_paint(), at the if g < 0 or g > 255 check. The test calls gps.get_paint(-0.5) and triggers the ValueError: "Color parameter outside of expected range: Red Green Blue".

Why it fails (root cause)
- The method clamps the input value into the allowed range and stores that clamped value in v:
  - v = max(value, self._lower_bound)
  - v = min(v, self._upper_bound)
  But when computing the gray channel it uses the original variable value instead of the clamped v:
  - g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
  Because value is -0.5 (below lower_bound 0.0), (value - lower_bound) is negative and the computed g is negative. The subsequent range check sees g < 0 and raises the ValueError.

Exactly where in the code
- In GrayPaintScale.get_paint(), line computing g uses value rather than v (the clamped value). The range-check immediately after that then raises the exception.

How to fix it (concrete steps)
1. Use the clamped variable v when computing the gray component. Replace the line
   g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
   with
   g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
2. (Optional) For safety, you can explicitly clamp g into [0, 255] after the computation, e.g.:
   g = max(0, min(255, g))
   but if v is correctly clamped and the arithmetic is correct this should be unnecessary.
3. Run the test again: gps = GrayPaintScale(); color = gps.get_paint(-0.5); assert color == (0, 0, 0)

Why this fix works
- Using v ensures the calculation uses a value guaranteed to be within [lower_bound, upper_bound], producing g in the 0–255 range and preventing the ValueError for inputs outside the bounds.
