What happened (where)
- The exception is raised in GrayPaintScale.get_paint, in this line:
  if g < 0 or g > 255:
      raise ValueError(f"Color parameter outside of expected range: Red Green Blue")
- That occurs after the code clamps the input to v but then computes g using the original value variable (value) instead of the clamped v. So when you call gps.get_paint(-0.5) the clamp sets v = 0.0, but g is computed from value = -0.5, producing a negative g and so the ValueError is raised.

Why it happened (root cause)
- The intent is to clamp the incoming value to the [lower_bound, upper_bound] range and then compute the gray level from the clamped value. The code correctly computes v as the clamped value, but then mistakenly uses value in the formula:
    g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
  That leaves negative inputs uncorrected and allows g to fall outside 0–255, which triggers the ValueError.

Exactly where to change (line/statement)
- In GrayPaintScale.get_paint, replace the use of value with the clamped variable v when computing g:
    g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

Suggested fix steps
1. Open the file and locate the get_paint method in class GrayPaintScale.
2. Find the three lines:
     v = max(value, self._lower_bound)
     v = min(v, self._upper_bound)
     g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
3. Change the last line to use v instead of value:
     g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
4. Run the failing test:
     gps = GrayPaintScale(); color = gps.get_paint(-0.5); assert color == (0, 0, 0)

Why this fix works
- Using v ensures the computation uses the clamped value so g will always be between 0 and 255 for inputs within or outside the bounds, preventing the ValueError and producing the expected grayscale tuple.

Optional small improvement
- Consider renaming v to clamped or clamped_value for clarity.
