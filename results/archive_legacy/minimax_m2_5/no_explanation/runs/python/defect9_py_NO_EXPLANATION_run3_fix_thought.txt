Looking at the `replace_each` function, I identified a critical bug: after performing a replacement, the loop doesn't properly update the `no_more_matches_for_repl_index` for the replaced index. When a replacement string contains the search string (e.g., searching for 'a' and replacing with 'ab'), it can cause infinite loops or incorrect behavior. Also, the algorithm doesn't check if the replacement contains any of the search strings, which could lead to re-replacements. Additionally, when searching for the next match after a replacement, it should mark the current replace_index as having no more matches in the current pass to prevent re-processing the same location.
