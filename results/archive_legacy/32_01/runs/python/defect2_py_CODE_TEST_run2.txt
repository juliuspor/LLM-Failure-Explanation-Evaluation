What happened (where it fails)
- In get_paint(), you clamp the input to v on lines:
  v = max(value, self._lower_bound)
  v = min(v, self._upper_bound)
  but the code that computes g still uses value instead of the clamped v:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- If value is below self._lower_bound (for example value == -0.5 in your test) the clamping produces v == self._lower_bound, but the calculation uses the original value (-0.5). That yields a negative g and triggers the ValueError check:
  if g < 0 or g > 255: raise ValueError(...)
- So the failure occurs because the calculation uses the un-clamped variable (value) instead of v, causing an out-of-range g.

Why it happens (root cause)
- Root cause: a simple variable mix-up. The clamp result is stored in v but never used in the subsequent normalized calculation; the code still uses value. The logic therefore does not apply the intended clamping.

Exact places in your snippet
- Clamping: the two lines assigning v.
- Computation: the line starting with g = int(...)
- Validation: the if g < 0 or g > 255: ... line that raises ValueError.

Concrete fixes (step-by-step)
1. Use the clamped value v in the normalized calculation:
   replace
     g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
   with
     g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

2. Guard against division by zero if lower and upper bounds could be equal:
   if self._upper_bound == self._lower_bound:
       g = 0  # or 255, or decide a sensible default
   else:
       compute g as above

3. (Optional) Use a clearer clamp and rounding, and a better error message:
   - Round before converting to int if desired: g = int(round(...))
   - Error message should include the bad value and the allowed range.

Example corrected method
    def get_paint(self, value: float) -> Tuple[int, int, int]:
        # Clamp value to bounds
        v = max(value, self._lower_bound)
        v = min(v, self._upper_bound)

        # Prevent division by zero
        if self._upper_bound == self._lower_bound:
            g = 0
        else:
            g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

        if g < 0 or g > 255:
            raise ValueError(f"Color parameter out of range: {g} not in [0,255]")

        return (g, g, g)

Validation
- With that change, gps.get_paint(-0.5) will use the clamped v and return (0, 0, 0) as expected.
