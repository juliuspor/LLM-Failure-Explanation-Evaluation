What happened (root cause)
- The exception comes from ClassUtils.to_class when it tries to call .upper() on an input element that is actually None. The traceback points at the line inside ClassUtils.to_class that calls value.upper() (or a similar expression like element.upper()). Because None has no .upper() method, Python raises AttributeError: 'NoneType' object has no attribute 'upper'.

Where it occurred
- Inside the method ClassUtils.to_class, on the line that contains value.upper() (or element.upper()). That line assumes every item is a string and calls .upper() without first checking the item’s type or value.

Why that assumption is wrong for your test
- Your test calls ClassUtils.to_class(['Test', None, 99.0]).
  - 'Test' is a str — calling .upper() is valid.
  - None is not a string — calling .upper() raises the AttributeError you saw.
  - 99.0 is a float — depending on implementation, it may be handled or converted differently; but the None already fails first.
- The method should treat None and non-string values explicitly instead of unconditionally calling string methods.

How to fix it (concrete steps)
1. In ClassUtils.to_class, locate the line that does value.upper() (or element.upper()). Example pseudo-line:
     mapped_name = value.upper()
2. Guard that call with a type check and a None check. Replace the unconditional call with branching that handles:
   - strings: normalize (call .upper())
   - None: map to type(None) (or whatever the method is supposed to return)
   - other objects: handle by type(value) or raise a clear error

Example change (one straightforward approach)
- Before:
    for value in values:
        mapped_name = value.upper()           # <-- fails for None
        cls = _lookup_class_by_name(mapped_name)
        result.append(cls)
- After:
    for value in values:

            result.append(type(None))
            continue

            mapped_name = value.upper()
            cls = _lookup_class_by_name(mapped_name)
            result.append(cls)
            continue
        # non-string, non-None: return its type (or convert as required)
        result.append(type(value))

3. If the method is supposed to accept strings naming types and also actual objects, make that behavior explicit in the docstring and tests. For your test, the above change yields [str, type(None), float] for ['Test', None, 99.0].

Additional recommendations
- Add unit tests that include None explicitly: assert ClassUtils.to_class([None]) == [type(None)].
- If you want to accept other representations (e.g., 'None' string vs actual None), decide and implement a clear rule: e.g., treat 'None' (string) as type(None) or as_lookup fails—document it.

Summary
- The immediate failure is because ClassUtils.to_class calls .upper() on every input item, and one of the items is None. Fix by checking for None and non-string types before calling .upper(), and update tests to cover None explicitly.
