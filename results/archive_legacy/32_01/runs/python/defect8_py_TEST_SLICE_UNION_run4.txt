What happened
- The test calls LocaleUtils.to_locale('fr__POSIX').
- The method rejects that input and raises ValueError("Invalid locale format: fr__POSIX").

Where it happens
- The rejection happens in to_locale at lines 242–245 (the lines that read ch3 = locale_str[3]; ch4 = locale_str[4]; then check whether ch3 and ch4 are 'A'..'Z' and raise ValueError if not).

Why it fails (root cause)
- The code assumes the two-character country code is always present and composed of two uppercase letters at positions 3 and 4.
- For the input 'fr__POSIX' the string layout is:
  - 0:'f' 1:'r' 2:'_' 3:'_' 4:'P' ...
- ch3 is '_' (not an uppercase letter), so the test ch3 < 'A' or ch3 > 'Z' is true and the method raises ValueError at line 245.
- In other words, the parser does not handle the "empty country" form (language + "__" + variant) where country is intentionally empty and two underscores separate language and variant. The code treats the second underscore as an invalid country letter.

What to change (concrete steps)
1. Detect the empty-country pattern (double underscore) before validating country letters. For example, after verifying the two-letter language:
   - If locale_str[2] == '_' and locale_str[3] == '_':
     - Return Locale(locale_str[0:2], '', locale_str[4:])  # variant starts at index 4
2. Otherwise continue the current path that expects two uppercase letters for country:
   - Validate ch3 and ch4 are uppercase letters and parse country from locale_str[3:5]
   - If there is a variant, ensure the separating '_' is present and read variant from the correct index (current code expects variant from index 6; keep that logic after parsing a present country).
3. Update unit tests to include both normal country-present forms (e.g., 'fr_FR') and the empty-country form ('fr__POSIX').
