What happens (where it fails)
- The failure is in get_paint at line 73: g is computed as
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- Immediately after (lines 76–77) the code raises ValueError if g < 0 or g > 255:
  raise ValueError(f"Color parameter outside of expected range: Red Green Blue")

Why it fails (root cause)
- That computation assumes two things that are not guaranteed:
  1) self._upper_bound != self._lower_bound. If those bounds are equal the division (upper - lower) is zero and Python raises ZeroDivisionError during the assignment to g.
  2) value lies between self._lower_bound and self._upper_bound. If value is outside that interval the normalized result can be <0 or >1, so g will be <0 or >255 and the code then raises the ValueError at lines 76–77.
- The ValueError message itself is unhelpful: it always says "Red Green Blue" and does not say which component failed, what the computed value was, or what the bounds/inputs were. That makes debugging harder.

Concrete examples that trigger each failure
- Bounds equal: self._lower_bound == self._upper_bound -> division by zero at line 73 (ZeroDivisionError).
- Value out of range: value < lower or value > upper -> g becomes negative or >255 -> code raises ValueError at line 76 with a non-informative message.

How to fix (clear steps)
1. Validate bounds before using them (in get_paint or earlier). If lower == upper, raise a clear error:
   - If you expect a single-valued range, raise ValueError("lower_bound must be different from upper_bound; got X and Y").
2. Clamp or validate the incoming value explicitly:
   - Option A (preferred when out-of-range is an input error): check if value < lower or value > upper and raise ValueError explaining the value and the bounds.
   - Option B (preferred when you want graceful behavior): clamp the normalized value into [0.0, 1.0] before mapping to 0–255.
3. Compute the normalized channel value only after those checks:
   norm = (value - lower) / (upper - lower)  # safe because upper != lower
   g = int(round(max(0.0, min(1.0, norm)) * 255.0))
4. Improve error messages to include the computed number and the input/ bounds, e.g.:
   raise ValueError(f"Computed green component {g} outside 0..255 for value={value} lower={lower} upper={upper}")

Minimal example replacement for lines 73–77
- Validate bounds:
  if self._upper_bound == self._lower_bound:
      raise ValueError(f"upper_bound must not equal lower_bound (both = {self._upper_bound})")
- Optionally validate or clamp value, then compute:
  if not (self._lower_bound <= value <= self._upper_bound):
      raise ValueError(f"value {value} not in bounds [{self._lower_bound}, {self._upper_bound}]")
  norm = (value - self._lower_bound) / (self._upper_bound - self._lower_bound)
  g = int(round(norm * 255.0))  # now guaranteed 0..255

Summary
- The immediate causes are either a ZeroDivisionError when upper == lower (line 73) or a ValueError when g is out of 0..255 (lines 76–77) because the code never validates bounds or the incoming value and uses an opaque error message. Fix by validating bounds, validating or clamping value, and improving the error message to show the offending numbers.
