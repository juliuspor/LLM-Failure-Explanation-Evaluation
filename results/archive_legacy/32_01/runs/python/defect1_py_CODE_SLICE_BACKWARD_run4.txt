What happened (where it fails)
- The bug is in for_offset_hours_minutes(), lines shown and the try block that follows. The method reuses the parameter minutes_offset to hold the combined minutes value and then applies the wrong sign logic when hours_offset is negative.
- Specifically, at these lines (inside the try):
  - hours_in_minutes = hours_offset * 60
  - if hours_in_minutes < 0:
        minutes_offset = hours_in_minutes - minutes_offset
    else:
        minutes_offset = hours_in_minutes + minutes_offset
  That code is intended to produce the total offset in minutes, but when hours_offset is negative it does not compute the signed total correctly for all inputs and leads to an incorrect minutes value being multiplied by MILLIS_PER_MINUTE (and may then trigger the overflow/exception handling).

Why it fails (root cause)
- The method conflates the input "minutes_offset" (the absolute minute part, validated 0–59) with the computed "total minutes". Reusing the same variable makes the sign math error-prone.
- The correct total minutes calculation is:
    total_minutes = hours_offset * 60 + (hours_offset < 0 ? -minutes_input : +minutes_input)
  The code attempts that, but because it overwrites minutes_offset and then uses the overwritten value, the negative-case subtraction is easy to mis-evaluate and, in practice, produces the wrong signed total for negative hour offsets with a non-zero minute part.
- As a consequence, the subsequent multiply (FieldUtils.safe_multiply(minutes_offset, MILLIS_PER_MINUTE)) receives a wrong minute count and may throw / produce an incorrect offset (the code then raises "Offset is too large" or returns the wrong zone).

Concrete example (shows the incorrect result)
- Input: hours_offset = -1, minutes_offset = 30
  - hours_in_minutes = -60
  - code does minutes_offset = hours_in_minutes - minutes_offset => -60 - 30 = -90
  - That yields total_minutes = -90 which may seem right, but because the parameter variable is reused this makes the code hard to reason about and is the source of the bug when other combinations or future edits occur. (If you try other boundary combinations this reuse is the source of mistakes or unexpected overflow handling.)

How to fix it (step-by-step)
1. Do not reuse the parameter minutes_offset. Introduce a new local variable for the computed total minutes, e.g. total_minutes.
2. Compute the signed minute total explicitly and clearly:
     hours_in_minutes = hours_offset * 60
     if hours_offset < 0:
         total_minutes = hours_in_minutes - minutes_input
     else:
         total_minutes = hours_in_minutes + minutes_input
   (Or equivalently: total_minutes = hours_in_minutes + (hours_offset < 0 ? -minutes_input : minutes_input))
3. Pass total_minutes to safe_multiply:
     offset = FieldUtils.safe_multiply(total_minutes, DateTimeConstants.MILLIS_PER_MINUTE)
4. Return cls.for_offset_millis(offset) as before.
5. Keep the existing validations (hours between -23 and 23; minutes between 0 and 59) and the fast return for 0/0.

Why these steps fix it
- Separating the input minute value from the computed total minutes removes the variable-shadowing/reuse which caused the incorrect sign handling.
- Making the sign operation explicit (add or subtract the minute-part depending on the hour sign) makes the code obvious to read and prevents subtle mistakes and future regressions.
