What happened (where): The error is raised in LocaleUtils.to_locale when parsing the string "fr__POSIX". The method splits the input on underscores and then runs strict validation on the language, country and variant tokens. The code path that handles three underscore-separated parts (the branch after splitting into parts and likely the block guarded by len(parts) == 3) rejects this input and raises ValueError("Invalid locale format: fr__POSIX").

Why it failed (root cause): "fr__POSIX" is the form language + empty country + variant (there are two underscores in a row). Java Locale allows that form (a Locale with language="fr", country="" and variant="POSIX"), but the to_locale implementation enforces that the country token, when present in a 3-part string, must be exactly two uppercase letters. In other words, the code treats the presence of a third token as meaning the middle (country) token must be length==2 and uppercase, and refuses an empty country token. So the validator incorrectly rejects the valid "language__variant" form (empty country + variant) and raises the ValueError.

- splitting: parts = locale_str.split('_')
- the branch handling len(parts) == 3 (the validation that checks country length and case)

What to change (concrete steps to fix):
1. In LocaleUtils.to_locale, find the branch that validates three-part locales (len(parts) == 3).
2. Replace the strict country check that requires country length == 2 with a check that allows either:
   - an empty country token (""), or
   - a 2-character uppercase country token.
3. Continue to validate language is 2 lowercase letters (or empty if your rules permit) and variant is non-empty and contains no underscores.

Example of a concrete conditional replacement (pseudo/pythonic code for the three-part branch):

    if len(parts) == 3:
