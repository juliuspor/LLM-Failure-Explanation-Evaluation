What happened (root cause)
- In get_paint the code correctly computes a clamped value and stores it in v (lines 2–4 of your snippet). But the grayscale calculation on line 6 still uses the original variable value instead of the clamped v. That means when value is outside the allowed range the computed g can be <0 or >255, which then triggers the ValueError check on lines 9–11.
- In short: the clamp is computed but never used in the calculation — you clamp v but compute g from value.

Why that fails
- If value < self._lower_bound or value > self._upper_bound, the calculation on line 6 still uses the original value, so the ratio can be negative or >1, producing g outside 0–255. The subsequent check (lines 9–10) then raises the ValueError you are seeing.

Other problems to address (also occur in the same method)
- Division by zero: if self._upper_bound == self._lower_bound, line 6 divides by zero.
- Error message is unhelpful: it doesn't include the bad value or which component failed.
- Even after fixing to use v, you should ensure numeric rounding/clamping and decide whether values exactly on bounds produce 0 or 255 as intended.

Concrete fixes (step-by-step)
1. Use the clamped variable v when computing g.
2. Guard against zero range (upper == lower) before dividing.
3. Optionally clamp the computed g to [0,255] and produce a clear error message if you still want to raise.

Fixed example (replace method body with this)
    def get_paint(self, value: float) -> Tuple[int, int, int]:
        # Clamp value to bounds
        v = max(value, self._lower_bound)
        v = min(v, self._upper_bound)

        # Avoid division by zero
        span = self._upper_bound - self._lower_bound
        if span == 0:
            # treat as middle/maximum/whatever behavior you want; here we return 0
            return (0, 0, 0)

        # Use the clamped value v (not the original value)
        g = int((v - self._lower_bound) / span * 255.0)

        # Clamp final g to [0,255] to be safe
        g = max(0, min(255, g))

        return (g, g, g)

If you prefer to keep raising an exception instead of clamping, change the span zero handling to raise ValueError with a clear message and include g (or v) in the message.

Summary
- The immediate bug: you clamp into v but compute g from value (line 6). Change that to use v, and add a zero-span guard and clearer error handling.
