  - v = max(value, self._lower_bound)
  - v = min(v, self._upper_bound)
- But on the next line you compute g using the original value, not the clamped v:
  - g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- If value is outside the bounds, that computation can produce a negative g or g > 255, which triggers the ValueError check immediately afterward.
- So the failure occurs at the g calculation and the subsequent range check (the lines shown above inside get_paint()) because clamping was done to v but never used.

Why it fails (root cause)
- A simple variable mix-up: you intended to use the clamped value v when mapping to 0–255, but you used value. The code clamps but then ignores the clamped result.
- Secondary issue to consider: if self._upper_bound == self._lower_bound the division (upper_bound - lower_bound) is zero and will raise ZeroDivisionError. That isn’t shown failing now but is a latent bug.
