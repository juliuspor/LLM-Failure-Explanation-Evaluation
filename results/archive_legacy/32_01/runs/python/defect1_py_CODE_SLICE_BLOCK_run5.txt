What happened (where it failed)
- The ValueError is raised on these lines (in for_offset_hours_minutes):
  - line 400–401:
    if minutes_offset < 0 or minutes_offset > 59:
        raise ValueError(f"Minutes out of range: {minutes_offset}")
- The method throws "Minutes out of range" when minutes_offset is negative.

Why it happened (root cause)
- The method validates minutes_offset as always non‑negative (0..59) but the rest of the method logic expects callers to be able to express a negative offset by giving a negative hours_offset and a positive minutes_offset OR by giving both negative values.
- If a caller passes hours_offset negative and minutes_offset negative (for example hours_offset = -1 and minutes_offset = -30), the validation rejects minutes_offset (< 0) and you get the ValueError at line 400.
- In short: the input contract is inconsistent. The code later combines hours and minutes with sign logic (lines that compute hours_in_minutes and then add/subtract minutes), but the precondition forbids negative minutes, so callers who naturally supply a negative minutes value for a negative offset are rejected before the combination logic runs.

What to change (concrete fixes, steps)
Pick one of these clear options depending on intended API:

Option A — Require minutes be non‑negative and document it (minimal change)
1. Keep the validation at lines 400–401 (minutes must be 0..59).
2. Update the API docs and callers so minutes is always provided as a non‑negative magnitude; the sign must come from hours_offset.
   - Example caller convention: (-1, 30) for -1:30 (hours negative, minutes positive).
3. No code change beyond documentation, or add an assertion error message that explains "minutes must be 0..59; sign comes from hours".

Option B — Accept signed minutes and normalize (recommended if callers already pass signed minutes)
1. Change the validation so signed minutes are allowed while ensuring magnitude <= 59:
   - Replace
       if minutes_offset < 0 or minutes_offset > 59:
           raise ValueError(...)
     with something like
       if abs(minutes_offset) > 59:
           raise ValueError(f"Minutes out of range: {minutes_offset}")
2. Compute the combined minute total explicitly using a single variable, using the sign of hours_offset if you want a consistent rule, or using the sign of minutes_offset if hours is zero. For example:
     hours_in_minutes = hours_offset * 60
     total_minutes = hours_in_minutes + (minutes_offset if hours_offset >= 0 else -abs(minutes_offset))
   or simpler and clearer:
     sign = -1 if (hours_offset < 0 or minutes_offset < 0) else 1
     total_minutes = sign * (abs(hours_offset) * 60 + abs(minutes_offset))
3. Then multiply total_minutes by MILLIS_PER_MINUTE and pass to for_offset_millis.
4. Keep the try/except for overflow as you have.

Why Option B is safer
- It fixes the real mismatch: callers that pass a negative minutes value (common when expressing negative offsets) will no longer be rejected by the validation on lines 400–401, and the combination logic will produce the intended negative total.

Summary
- The code fails because it rejects negative minutes at lines 400–401 even though the later combination logic either expects or should accept signed minutes. Choose to either enforce a non‑negative minutes convention and document it (Option A), or allow signed minutes and normalize them when combining hours and minutes (Option B, recommended).
