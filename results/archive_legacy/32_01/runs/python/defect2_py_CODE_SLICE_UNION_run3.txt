What happened (where it fails)
- In get_paint (starts at line 56) you clamp the input into variable v:
  - lines ~57–58: v = max(value, self._lower_bound); v = min(v, self._upper_bound)
- But on line 73 you compute g using the original value, not the clamped v:
  - g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- Because you used value instead of v, g can be <0 or >255 when value is outside the bounds, triggering the ValueError at lines 76–77.
- There is a second failure mode you should consider: if self._upper_bound == self._lower_bound the division on line 73 will raise ZeroDivisionError.

Why it happens (root cause)
- The code clamps into v but never uses v for the normalized calculation. That makes the clamping a no-op as far as g is concerned.
- If value < lower_bound or > upper_bound, the expression ((value - lower) / (upper - lower)) produces a negative number or >1, so after multiplication and int conversion g falls outside 0..255 and the guard raises.
- If upper == lower the denominator is zero and the expression fails with ZeroDivisionError.

Exact places to change and clear fixes (step-by-step)
1. Use the clamped variable v in the calculation (change line 73):
   - Replace
     g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
   - With
     g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

2. Handle equal bounds to avoid division by zero (add before computing g):
   - If self._upper_bound == self._lower_bound, decide a behavior:
     - raise a clear error: raise ValueError("upper_bound must be greater than lower_bound")
     - or return a fixed color (e.g., (0,0,0) or (255,255,255)) depending on your intent.
   - Example guard:
     if self._upper_bound == self._lower_bound:
         raise ValueError("get_paint: upper_bound equals lower_bound; cannot normalize value")

3. Improve the error message so it shows the offending value (optional but helpful). For example:
   if g < 0 or g > 255:
       raise ValueError(f"Color parameter outside expected range: {g} (computed from value={value})")

Summary
- Root cause: using value instead of the clamped v at line 73.
- Secondary issue: possible division by zero when upper_bound == lower_bound.
- Fix by computing g from v and adding a guard for equal bounds.
