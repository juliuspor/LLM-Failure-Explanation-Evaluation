What happened (where it fails)
- In get_paint, you clamp the input into variable v on lines 3–4:
  - Line 3: v = max(value, self._lower_bound)
  - Line 4: v = min(v, self._upper_bound)
- But on line 6 you compute g using the original value, not the clamped v:
  - Line 6: g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- If value is outside [self._lower_bound, self._upper_bound] this produces g < 0 or g > 255, so the check on lines 9–10 raises ValueError (line 10).
- Also note a secondary failure mode: if self._upper_bound == self._lower_bound the denominator is zero and the division on line 6 will raise ZeroDivisionError.

Why it fails (root cause)
- You intended to clamp the input before mapping it to 0–255, but you never used the clamped variable v when computing g. The code clamps v but then ignores it. That lets out-of-range values produce invalid g or trigger division by zero when the bounds are equal.

1. Use the clamped value v in the mapping on line 6:
   - Replace line 6 with:

2. Guard against identical bounds to avoid division by zero. For example, add before the calculation:
   - if self._upper_bound == self._lower_bound:
         g = 0  # or 255, or some defined constant/behavior
         # then skip the division
3. (Optional) After computing g, keep the existing range check; or better, clamp g to [0, 255] explicitly:
   - g = max(0, min(255, g))
