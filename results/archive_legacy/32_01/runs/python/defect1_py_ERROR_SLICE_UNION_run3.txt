What happened (where it failed)
- The ValueError was raised at line 400: the code runs the check
  if minutes_offset < 0 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")
  and so it threw "Minutes out of range: -15".
- That means minutes_offset at that point is -15, so the code believes the minute component is out of the allowed 0..59 range.

Why it happened (root cause)
- The code is checking the minute component before the sign/total offset logic is applied. Lines 406–409 suggest the code later adjusts minutes_offset for negative hour offsets:
    406:     minutes_offset = hours_in_minutes - minutes_offset
    408:     minutes_offset = hours_in_minutes + minutes_offset
  But those adjustments happen after the range check at line 400. So a negative minute value (for example when parsing "-8:15" the parser produced minutes = -15 or minutes_offset = -15) fails the early check even though the later arithmetic would have produced a valid total offset.
- In short: the method validates the minute value in isolation (rejecting negative minutes) before converting the sign and combining hours and minutes into a signed total. The negative minute value is legitimate as part of a signed offset, but the check catches it prematurely.

Where in code to change (specifics)
- The failing validation is at line 400. The subsequent adjustments are at lines 406–409 and the final conversion to milliseconds is at line 409 and cls.for_offset_millis(offset) at 412. You must move or change the validation so the sign is handled before you enforce 0..59 on minutes.
