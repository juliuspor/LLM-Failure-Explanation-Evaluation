What happened (where it fails)
- At line 56 you clamp the incoming value into variable v:
  - v = max(value, self._lower_bound)
  - v = min(v, self._upper_bound)
- But at line 73 you compute g using the original value variable, not the clamped v:
  - g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- If value is outside the [self._lower_bound, self._upper_bound] range, that calculation can produce g < 0 or g > 255 and then the check at line 76 raises ValueError.
- Also, if self._upper_bound == self._lower_bound you get a division-by-zero at line 73.

Why it happens (root cause)
- You intended to use the clamped number but forgot to use v when computing g. The computation therefore sometimes operates on an out-of-range value and triggers the range check (or a ZeroDivisionError when bounds are equal).

How to fix it (concrete steps)
1. Use the clamped v in the computation (change line 73):
   - Replace
     g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
     with
     g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

2. Guard against equal bounds (add a zero-division check before the division):
   - If self._upper_bound == self._lower_bound, decide a behavior (e.g., return 0 or 255 or a mid-gray). Example:
     if self._upper_bound == self._lower_bound:
         return (128, 128, 128)  # or choose a sensible default

3. (Optional) Improve the error message at line 76 so it shows the actual out-of-range value and bounds:
   - raise ValueError(f"Color component out of range: g={g} expected 0..255 (bounds {self._lower_bound}..{self._upper_bound})")

Minimal corrected example combining (1) and (2):
    def get_paint(self, value: float) -> Tuple[int, int, int]:
        # Clamp value to bounds
        v = max(value, self._lower_bound)
        v = min(v, self._upper_bound)

        if self._upper_bound == self._lower_bound:
            return (128, 128, 128)

        g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

        if g < 0 or g > 255:
            raise ValueError(f"Color component out of range: g={g}")

        return (g, g, g)

That addresses the bug (use of value instead of v) and the division-by-zero edge case.
