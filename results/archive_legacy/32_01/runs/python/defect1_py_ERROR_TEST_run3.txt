What happened
- The failure comes from passing a negative minutes value into DateTimeZone.for_offset_hours_minutes(-2, -15).
- The DateTimeZone.for_offset_hours_minutes method validates the minutes argument and throws ValueError/IllegalArgumentException when minutes is outside the allowed range (0–59). The exception message "Minutes out of range: -15" is produced by that validation.

Why it fails (root cause)
- for_offset_hours_minutes expects minutes to be a non‑negative number 0..59. The sign of the offset is determined by the sign of the hours parameter (and/or the total computed offset). Passing -15 for minutes violates the minutes range check, so the method rejects it before building the zone.

Where it occurs in the code
- The exception is thrown inside DateTimeZone.for_offset_hours_minutes (the method that builds a fixed-offset zone). Inside that method the code checks the minutes argument and raises "Minutes out of range: X" when minutes < 0 or minutes > 59. That check is the exact place the error originates.

How to fix it (concrete steps)
Option A — pass a non‑negative minutes argument
1. Call DateTimeZone.for_offset_hours_minutes(-2, 15) instead of (-2, -15).
   - Reason: the negative sign on the hours makes the whole offset negative; minutes must be positive (0–59).
2. Test:
   - zone = DateTimeZone.for_offset_hours_minutes(-2, 15)
   - assert zone.id == '-02:15'

Option B — build the offset from total milliseconds (if you prefer computing a signed value)
1. Compute total offset in milliseconds and use the for_offset_millis API (or equivalent) that accepts a signed millisecond offset:
   - total_ms = -((2 * 60 + 15) * 60 * 1000)
   - zone = DateTimeZone.for_offset_millis(total_ms)
2. This bypasses the two-parameter minutes validation and uses a signed total instead.

If you really want to accept negative minutes in that two-parameter API
- Change the validation in DateTimeZone.for_offset_hours_minutes (not recommended unless you understand consequences).
  1. In DateTimeZone.for_offset_hours_minutes, remove/adjust the check that rejects minutes < 0 and instead combine sign from hours and minutes into a single signed offset, then validate magnitude.
  2. Add unit tests for combinations like (-2, 15), (-2, -15), (2, -15) to define exact behavior.

Summary
- The method rejects negative minutes. Use a positive minutes value with a negative hours (for_offset_hours_minutes(-2, 15)), or supply a signed total milliseconds to for_offset_millis. This is the immediate and minimal change to make the test pass.
