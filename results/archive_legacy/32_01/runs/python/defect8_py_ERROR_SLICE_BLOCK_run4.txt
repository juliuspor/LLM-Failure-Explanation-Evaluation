What happened (where it fails)
- The ValueError is raised at lines 244–245 of your parser code:
  - Line 244 checks whether ch3 and ch4 are uppercase letters.
  - Line 245 raises ValueError(f"Invalid locale format: {locale_str}") when that check fails.
- The input locale_str that triggered the error is "fr__POSIX".
- For "fr__POSIX" the third character (ch3) is '_' (an underscore), not an uppercase letter, so the check ch3 < 'A' or ch3 > 'Z' is true and the ValueError is thrown.

Why it happened (root cause)
- Your code assumes the locale string always contains a two-letter language followed immediately by a two-letter country/region code (both letters A–Z) at the positions tested (ch3 and ch4).
- "fr__POSIX" is a valid-looking locale that has:
  - language = "fr"
  - an empty/omitted country/territory between the double underscores
  - a variant = "POSIX"
- Because the code only checks for A–Z at ch3 and ch4, it does not allow the underscore that indicates a missing country or the underscore that separates an empty country and a variant. That causes the exact ValueError you see.

Where to change it (concrete suggestions tied to lines)
- At lines 244–245, replace the single strict check with logic that accepts either:
  1) a two-letter country code (both ch3 and ch4 are A–Z), or
  2) an underscore at ch3 that marks an omitted country (and then handle the variant that follows).
- Concretely, instead of:
  if ch3 < 'A' or ch3 > 'Z' or ch4 < 'A' or ch4 > 'Z':
      raise ValueError(f"Invalid locale format: {locale_str}")
  use one of the fixes below.
