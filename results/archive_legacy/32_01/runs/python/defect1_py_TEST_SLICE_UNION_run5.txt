What happened (where it failed)
- The call DateTimeZone.for_offset_hours_minutes(-2, -15) throws a ValueError at the validation on line 400:
  400 |         if minutes_offset < 0 or minutes_offset > 59:
  That check sees minutes_offset == -15 and immediately raises "Minutes out of range", so execution never reaches the later code that combines hours and minutes (lines 406–409) and builds the offset.

Why it failed (root cause)
- The code treats the minutes argument as if it must be in the 0..59 range unconditionally. It does not accept a negative minutes value even when the hours argument is negative. The caller passed minutes as -15 (intending -2 hours and -15 minutes), but the guard on line 400 rejects negative minutes before any normalization or sign handling occurs.
- In short: the validation at line 400 is too strict and runs before sign-normalization, so valid inputs like (-2, -15) are treated as invalid.

- Lines 406–409: the later logic that attempts to compute minutes_offset when hours or minutes signs need adjustment (this code never runs because the earlier check already failed).
- Line 412 and 412 return: subsequent behavior depends on a correctly computed minutes_offset, but the pre-check prevents that.

3. Compute total_minutes = sign * (abs(hours) * 60 + minutes_abs).

Why these fixes work
- They allow minutes to be provided as negative when the caller intends a negative total offset (e.g., -2 hours and -15 minutes).
- They preserve the sensible constraint that minutes magnitude must be within 0..59 while allowing the sign to be expressed on either hours or minutes.

Which to choose

Summary
