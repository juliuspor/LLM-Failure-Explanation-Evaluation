What happened (what failed)
- The test called LocaleUtils.to_locale('fr__POSIX') expecting a Locale with language "fr", empty country, and variant "POSIX".
- Instead the method returned a Locale with country "POSIX" and empty variant (or otherwise mis-assigned the parts), so the assertion failed.

Where it happened (exact place)
- The bug is inside LocaleUtils.to_locale (the method that parses the underscore-separated locale string). The failure occurs in the parsing logic when handling consecutive underscores in the input "fr__POSIX".

Why it happened (root cause)
- The parsing code splits the input on "_" using a split operation that discards empty tokens. For example, many split implementations default to removing empty trailing or intermediate tokens (e.g. in Java String.split(String regex) with no limit argument).
- When you call split("_") on "fr__POSIX" you get ["fr", "POSIX"] — the empty country token between the two underscores is dropped. The parser then treats "POSIX" as the country part instead of the variant.
- As a result the parts are shifted left and the Locale constructor is called with the wrong values, so language/country/variant become (language="fr", country="POSIX", variant="") instead of the expected (language="fr", country="", variant="POSIX").

Exactly what to change (step-by-step fix)
1. In LocaleUtils.to_locale, change the split so empty tokens are preserved. If this is Java, replace:
   - String[] parts = input.split("_");
   with:
   - String[] parts = input.split("_", -1);
   Explanation: split(regex, -1) preserves empty tokens (so "fr__POSIX" becomes ["fr", "", "POSIX"]).

2. Ensure the subsequent assignment logic uses parts by index and allows empty strings:
   - if parts.length == 1: language = parts[0]; country = ""; variant = "";
   - if parts.length == 2: language = parts[0]; country = parts[1]; variant = "";
   - if parts.length >= 3: language = parts[0]; country = parts[1]; variant = join(parts[2..end], "_");
   This correctly handles "fr__POSIX" (parts[1] == "" and parts[2] == "POSIX").

3. Add or update unit tests:
   - Keep the failing test: to_locale("fr__POSIX") -> Locale("fr", "", "POSIX").
   - Add tests for other edge cases: "en", "en_US", "en_US_WIN", "zh__pinyin", and strings with more than one underscore in variant.

Why this fix works
- Preserving empty tokens guarantees the parser sees the empty country field when two underscores are adjacent, so the variant doesn't get shifted into the country slot.
- The assignment rules above match how Locale constructors expect their arguments (language, country, variant).
