What happened (symptom)
- Code calling LocaleUtils.to_locale(...) crashes with an IndexError rather than raising the intended ValueError for an invalid locale string.

Where it happened
- In LocaleUtils.to_locale, in the validation block that inspects the country letters. The failure occurs at the lines shown in the slice:
  - Line 244: if ch3 < 'A' or ch3 > 'Z' or ch4 < 'A' or ch4 > 'Z':
  - Line 245:     raise ValueError(f"Invalid locale format: {locale_str}")

Why it happened (root cause)
- The code reads ch3 and ch4 from locale_str (e.g. ch3 = locale_str[3], ch4 = locale_str[4]) before confirming that those indexes actually exist. If locale_str is too short (for example "en_" or "en" or any 3–4 character string), accessing locale_str[4] raises IndexError. That exception occurs before the condition can evaluate and before the code raises the intended ValueError describing the invalid format.
- In short: validation compares ch3/ch4 against 'A'..'Z' without first checking length/indices, so short inputs cause IndexError instead of the documented ValueError.

Exact place to fix
- LocaleUtils.to_locale, in the block that validates the country component (the code around lines 244–245). Ensure you check string length (or existence of positions 3 and 4) before indexing and comparing ch3 and ch4.

Concrete fixes (pick one)
1) Preferred: explicit length check before extracting characters
   - Before you read ch3/ch4, add:
     if len(locale_str) < 5:
         raise ValueError(f"Invalid locale format: {locale_str}")
   - Then you can safely do ch3 = locale_str[3]; ch4 = locale_str[4] and keep the existing comparisons.

2) Defensive indexing with try/except
   - Wrap the indexing in try/except IndexError and convert to ValueError:
     try:
         ch3 = locale_str[3]; ch4 = locale_str[4]
     except IndexError:
         raise ValueError(f"Invalid locale format: {locale_str}")
   - Then perform the same comparisons.

3) Verify positions explicitly without indexing
   - Use a pattern/length-based check up front to accept only lengths you want:
     - If expecting "ll_CC" (5 chars) or "ll" (2 chars) or "ll_CC_variant" (>=7), validate lengths and separator positions before indexing.

Why these fixes work
- They prevent out-of-range indexing by ensuring the characters exist first; that converts an accidental IndexError into the intended ValueError with a clear message about the invalid input, matching the method contract in the docstring.

Example concrete change (minimal)
- Insert near top of the country-validation block:
  if len(locale_str) < 5:
      raise ValueError(f"Invalid locale format: {locale_str}")

After this change, short/invalid inputs will raise ValueError as documented instead of IndexError.
