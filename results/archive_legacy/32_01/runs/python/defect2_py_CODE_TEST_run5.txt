What happened (root cause)
- In get_paint the code clamps the incoming value to the variable v on lines 2–3, but then computes g from the original parameter value on line 5.
- Because the clamped value v is not used, a value below the lower bound (your test: value == -0.5) produces a negative g. The negative g triggers the ValueError check on line 7, so the call fails.

- The clamp is done on lines 3–4 but the calculation that uses the clamped value should be on line 6; instead line 6 uses value (the original un-clamped input).

Why that causes the test failure
- Your test calls gps.get_paint(-0.5). If self._lower_bound is e.g. 0.0, the clamped v becomes 0.0 (lines 3–4). But because g is computed from value (-0.5) on line 6, g becomes negative. The explicit check on line 9 then raises ValueError, so the function does not return (0,0,0) as expected.

Concrete fixes (step-by-step)
1. Use the clamped variable v when computing g. Change line 6 to:

2. Add a guard against a zero-length range (division by zero) if lower and upper bounds can be equal. For example, before computing g:
   if self._upper_bound == self._lower_bound:
       g = 0  # or 255, or whatever is appropriate for your semantics
   else:
       compute g using v (as in step 1)
3. Keep the range check (lines 9–10) if you want to validate the result; with the above changes a clamped input like -0.5 will produce g == 0 and avoid the exception.
