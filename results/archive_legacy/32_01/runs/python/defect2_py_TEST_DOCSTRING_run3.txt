What happened
- The test calls gps.get_paint(-0.5) expecting the method to return the clamped black color (0, 0, 0).
- Instead get_paint raised ValueError. The method is treating the input -0.5 as an out-of-range input and raising, rather than clamping the value to the valid 0–1 interval and returning (0,0,0).

Why it happened (root cause)
- In GrayPaintScale.get_paint the code converts the input value into an 8-bit gray level and then enforces a strict range check. Concretely this happens where the method computes the RGB component (e.g. rgb = int(value * 255)) and then raises ValueError if rgb is outside 0–255.
- Because value == -0.5 produces rgb < 0, the range check trips and the method raises ValueError. The docstring says the method "Returns a paint (RGB tuple) for the specified value" and declares a ValueError "If value produces RGB outside 0-255 range", but the test expects the implementation to clamp negative inputs to 0. The implementation enforces the range by exception instead of clamping, so the behavior disagrees with the test's expectation.

Where it occurred
- In GrayPaintScale.get_paint, at the lines that (a) compute the integer gray level from value (rgb = int(value * 255) or similar) and (b) immediately raise ValueError when rgb < 0 or rgb > 255. Those lines are the direct cause of the failure when value = -0.5.

How to fix it (clear steps)
Pick one of two precise, small fixes depending on the intended API:

Option A — Clamp input values to [0,1] (recommended if you want negative values to map to black and >1 to white)
1. At the start of GrayPaintScale.get_paint, insert a clamp:
   value = max(0.0, min(1.0, float(value)))
2. Then compute the rgb channel:
   level = int(value * 255)
3. Return (level, level, level)
4. Remove (or relax) the later strict ValueError check that raises for out-of-range rgb.

Option B — Keep strict validation but make the docstring/test consistent
1. If the class should reject out-of-range inputs, update the test to expect a ValueError for -0.5.
2. Or update the docstring to say "Raises ValueError for values outside [0,1]" rather than phrasing it in terms of RGB after conversion.

Concrete code snippet for Option A (replace the problematic lines)
- Before (problematic pattern):
  rgb = int(value * 255)
  if rgb < 0 or rgb > 255:
      raise ValueError("RGB out of range")
  return (rgb, rgb, rgb)

- After (clamping):
  value = max(0.0, min(1.0, float(value)))   # clamp value to [0,1]
  level = int(value * 255)
  return (level, level, level)

Tests to add/update
- Keep your existing test:
  gps = GrayPaintScale(); color = gps.get_paint(-0.5); assert color == (0, 0, 0)
- Add symmetry tests:
  assert GrayPaintScale().get_paint(1.5) == (255, 255, 255)  # if clamping is chosen
- Or, if strict validation is intended, replace the test with:
  with pytest.raises(ValueError):
      GrayPaintScale().get_paint(-0.5)

Summary
The failure is caused by get_paint raising ValueError when rgb < 0 after converting the input value. The method either needs to clamp the input value into [0,1] before conversion (so -0.5 yields (0,0,0)), or the test/docstring must be changed to expect a ValueError for out-of-range inputs. The simplest fix is to clamp at the start of GrayPaintScale.get_paint as shown above.
