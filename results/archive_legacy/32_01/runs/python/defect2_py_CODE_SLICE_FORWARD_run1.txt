What happened (where)
- In get_paint (first line of the snippet) you clamp the input into v on lines:
  - v = max(value, self._lower_bound)
  - v = min(v, self._upper_bound)
- But on the next line you compute g using the original value, not the clamped v:
  - g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- Because you ignored v, g can be computed from a value outside the expected range and then fail the range check immediately afterward (the ValueError branch).

Why it fails (root cause)
- The intended clamp has no effect on the computed color because you never use v in the formula. If value < _lower_bound or value > _upper_bound, the formula can produce a g < 0 or g > 255 and trigger the error. In short: you clamped into v but then used value, so clamping is ineffective.
- There is a second potential failure: if self._upper_bound == self._lower_bound the denominator (self._upper_bound - self._lower_bound) is zero and the division will raise a ZeroDivisionError.

Where exactly (reference to lines)
- get_paint: clamping lines:
  - v = max(value, self._lower_bound)
  - v = min(v, self._upper_bound)
- wrong computation line:
  - g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- range check and raise line:
  - if g < 0 or g > 255: raise ValueError(...)

Concrete fixes (step-by-step)
1. Use the clamped variable v when computing g:
   - g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
2. Guard against zero span (upper == lower) before dividing:
   - if self._upper_bound == self._lower_bound: decide a sensible output (e.g., g = 0 or 255 or 128) or raise a clear exception.
3. Improve the ValueError message so it includes the actual out-of-range value (and which component), for example:
   - raise ValueError(f"Color parameter outside of expected range: g={g} (expected 0..255)")
4. Optionally, compute g with clamped value and then clamp g to [0,255] to be robust.

Minimal corrected implementation
- Replace the g line and add the zero-span check:

    def get_paint(self, value: float) -> Tuple[int, int, int]:
        # Clamp value to bounds
        v = max(value, self._lower_bound)
        v = min(v, self._upper_bound)

        span = self._upper_bound - self._lower_bound
        if span == 0:
            # decide policy: here we map to middle gray
            g = 128
        else:
            g = int((v - self._lower_bound) / span * 255.0)

        if g < 0 or g > 255:
            raise ValueError(f"Color parameter outside of expected range: g={g} (expected 0..255)")

        return (g, g, g)

That fixes the root cause (using value instead of v), prevents divide-by-zero, and gives a useful error message.
