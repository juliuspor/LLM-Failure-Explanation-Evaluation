What happened (where it fails)
- The failure occurs inside LocaleUtils.to_locale, in the branch after checking length when you handle the part after the language (around the lines where you do ch3 = locale_str[3] and ch4 = locale_str[4]).
- For the input "fr__POSIX" the code reaches that branch and then checks:
  if ch3 < 'A' or ch3 > 'Z' or ch4 < 'A' or ch4 > 'Z':
  This test expects the two characters after the underscore to be uppercase letters (a country code). In "fr__POSIX", ch3 is '_' (because the country is empty) and ch4 is 'P'. Because ch3 is '_' the test fails and you raise ValueError.

Why it fails (root cause)
- The code assumes the format is either:
  - "ll" (language)
  - "ll_CC" (language + country)
  - "ll_CC_V" (language + country + variant)
  but it does not handle the valid variant form where the country is empty and the format is "ll__Variant" (two consecutive underscores). In "fr__POSIX" the country is empty so there are two underscores; your validation of ch3/ch4 forces two uppercase letters and so rejects the empty-country form.

Exactly where to change it
- In to_locale, replace the strict ch3/ch4 uppercase-only check with logic that accepts either:
  - country present: ch3 and ch4 are 'A'..'Z'
  - country absent: ch3 == '_' (second underscore), in which case variant starts at index 4

Suggested concrete fix (step-by-step)
1. After checking locale_str[2] == '_' (you already do this), read ch3 = locale_str[3].
2. If ch3 == '_':
   - Treat country as empty string and set variant = locale_str[4:] (variant may be empty or non-empty).
   - Return Locale(language, "", variant) (or validate variant if desired).
3. Else:
   - Validate ch3 and ch4 are uppercase letters as you already intended.
   - If length == 5 return Locale(language, country)
   - Else ensure locale_str[5] == '_' and return Locale(language, country, locale_str[6:])
