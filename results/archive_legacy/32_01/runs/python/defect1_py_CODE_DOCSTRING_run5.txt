What happens
- The method rejects any negative minutes at this check:
  if minutes_offset < 0 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")
  That enforces minutes in the range 0..59. The docstring and examples explicitly allow minutes in -59..+59 (for example (0, -15) and (-2, -15) are valid). So passing a negative minutes value causes the method to fail here even though the API promises to accept it.

Where exactly (which lines)
- The failure is at the minutes validation line shown above (the third if-block in your method).
- The code that attempts to combine hours and minutes is in the try block starting with:
    hours_in_minutes = hours_offset * 60
    if hours_in_minutes < 0:
        minutes_offset = hours_in_minutes - minutes_offset
    else:
        minutes_offset = hours_in_minutes + minutes_offset
  That combination logic is also confusing/wrong for some sign combinations.

Why it fails (root causes)
1. Wrong validation range for minutes
   - The check forbids negative minutes; the doc and example table allow negative minutes in many combinations. So valid inputs like (0, -15) or (-2, -15) are rejected early.

2. Incorrect combination logic for sign handling
   - The code computes hours_in_minutes and then mutates minutes_offset to become the total minutes using:
       if hours_in_minutes < 0:
           minutes_offset = hours_in_minutes - minutes_offset
       else:
           minutes_offset = hours_in_minutes + minutes_offset
   - This is wrong for the case where hours are negative and minutes are negative: for hours = -2, minutes = -15 you want total = -120 - 15 = -135, but the code does hours_in_minutes - minutes_offset; since minutes_offset is negative, you end up subtracting a negative and get -120 - (-15) = -105 (wrong). The code treats minutes_offset as already signed in one place and as magnitude elsewhere; the logic mixes signs incorrectly.

What to change (clear, actionable steps)
1. Fix minutes validation
   - Allow minutes in -59..+59:
       if minutes_offset < -59 or minutes_offset > 59:
           raise ValueError(...)
2. Enforce the prohibited combination (positive hours with negative minutes)
   - After validation, add:
       if hours_offset > 0 and minutes_offset < 0:
           raise ValueError("Minutes must be non-negative when hours are positive")
   (This matches the doc's "ValueError" example for (2, -15).)

3. Compute total minutes correctly (explicit and clear)
   - Replace the current mixing-logic with an explicit formula:
       hours_in_minutes = hours_offset * 60
       if hours_offset < 0 and minutes_offset < 0:
           total_minutes = hours_in_minutes - abs(minutes_offset)
       else:
           total_minutes = hours_in_minutes + minutes_offset
   - Or more simply (and safely):
       if hours_offset < 0:
           total_minutes = hours_in_minutes - abs(minutes_offset)
       else:
           total_minutes = hours_in_minutes + minutes_offset
     This yields:
       (-2, -15) -> -120 - 15 = -135 (correct)
       (-2, 15)  -> -120 + 15 = -105 (correct)
       (0, -15)  -> 0 - 15 = -15 (correct)
       (2, 15)   -> 120 + 15 = 135 (correct)

4. Use a separate variable for total minutes (don’t overwrite minutes_offset)
   - Avoid mutating the input parameter name to hold a different meaning; use total_minutes to make the code obvious and avoid confusion before multiplying to millis.

5. Keep overflow protection
   - Continue to wrap the multiplication into millis in the try/except and call FieldUtils.safe_multiply(total_minutes, MILLIS_PER_MINUTE) as you do now.

Example final sketch (minimal):
    if minutes_offset < -59 or minutes_offset > 59:
        raise ValueError(...)
    if hours_offset > 0 and minutes_offset < 0:
        raise ValueError(...)
    hours_in_minutes = hours_offset * 60
    if hours_offset < 0:
        total_minutes = hours_in_minutes - abs(minutes_offset)
    else:
        total_minutes = hours_in_minutes + minutes_offset
    offset = FieldUtils.safe_multiply(total_minutes, DateTimeConstants.MILLIS_PER_MINUTE)
    return cls.for_offset_millis(offset)

Summary
- The immediate failure is caused by rejecting negative minutes at the validation line (minutes must be allowed in -59..59). Even if that is fixed, the existing sign-combination code is incorrect for negative hours/negative minutes (it double-negates). Fix the validation, enforce the sign rule for positive hours, and compute total minutes with an explicit, separate total_minutes variable.
