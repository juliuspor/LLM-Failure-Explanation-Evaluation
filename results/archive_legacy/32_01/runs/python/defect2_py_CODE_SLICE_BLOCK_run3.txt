What happened (where it fails)
- Method: get_paint
- Failing lines: the integer computation of g (third line in the method) and the ValueError raise at lines 76–77.
- Root cause: you clamp the input into variable v, but then compute g using the original variable value instead of v. If value is outside the [self._lower_bound, self._upper_bound] range the computed g can fall outside 0–255 and trigger the ValueError at lines 76–77.

Why that causes the exception
- Lines shown:
  - You do clamping:
    v = max(value, self._lower_bound)
    v = min(v, self._upper_bound)
  - But then compute:
    g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
  - If value < self._lower_bound or value > self._upper_bound, the fraction can be < 0 or > 1, producing g < 0 or g > 255, which makes the if check true and raises ValueError at lines 76–77.
- Additionally: if self._upper_bound == self._lower_bound the expression divides by zero and will raise a ZeroDivisionError before you reach the ValueError.

1. Use the clamped variable v when computing g:
   g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
2. Guard against division by zero (when upper == lower):
   - If upper == lower, decide the intended behavior (e.g., return 0 or 255 or a fixed gray) and handle that explicitly before computing g.
3. Clamp final g to the 0–255 integer range as a safety net (optional but recommended):
   g = max(0, min(255, int(round(...))))
4. Improve the error message to include the offending values (if you keep the explicit check).
