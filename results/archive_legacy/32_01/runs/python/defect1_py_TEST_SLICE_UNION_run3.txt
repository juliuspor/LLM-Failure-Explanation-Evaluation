What happened (where it fails)
- The failure happens inside DateTimeZone.for_offset_hours_minutes, at the check on line 400:
  if minutes_offset < 0 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")
- Calling DateTimeZone.for_offset_hours_minutes(-2, -15) triggers that check because minutes_offset is -15, so the method raises ValueError("Minutes out of range: -15"). That is why your assert never runs.

Why it fails (root cause)
- The method assumes minutes must be non‑negative (0..59). It does not accept a negative minutes argument, even though callers sometimes pass a negative minutes together with a negative hours (e.g., -2 and -15) expecting the total offset -2:15.
- Later code (lines 406–409) contains logic that tries to adjust minute values for negative hour offsets, but that code only runs when minutes are already in the accepted 0..59 range. Because the early check rejects negative minutes outright, the normalization logic never runs for inputs like (-2, -15).
- In short: the precondition on line 400 is too strict (rejects negative minutes) while the later code expects minutes to be normalized instead. The mismatch causes the exception.

Exactly where to look in the source
- Line 400: the minutes range check that throws the ValueError.
- Lines 406–409: the normalization logic that adjusts minutes for negative hours, which never runs if the check at 400 has already failed.
- The method entry is DateTimeZone.for_offset_hours_minutes (the failing call).
