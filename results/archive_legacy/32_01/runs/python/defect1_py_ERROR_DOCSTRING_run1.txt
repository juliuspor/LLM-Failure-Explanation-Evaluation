What happened
- You called the timezone factory with a positive hours value but a negative minutes value (for example: hours = 2, minutes = -15).
- The factory method enforces the rule described in the docstring: when hours is positive, minutes must be zero or positive. Passing a negative minute with a positive hour is treated as invalid and triggers the ValueError shown: "Minutes out of range: -15".

Where it occurs
- The exception is raised inside the timezone factory method shown in your docstring (e.g. DateTimeZone.forOffsetHoursMinutes or the equivalent method you are using). The code checks the arguments and raises ValueError when minutes are outside [-59, +59] or when the sign combination is invalid (positive hours with negative minutes).

Why it fails (root cause)
- The implementation expects the sign of the total offset to be expressed consistently:
  - A positive offset must have hours >= 0 and minutes >= 0.
  - A negative offset must have hours <= 0 and minutes >= 0 (the hours carry the negative sign; minutes are non‑negative).
  - Zero hours may be combined with a negative minutes to indicate a small negative offset (0, -15 => -00:15).
- Supplying (2, -15) is ambiguous/invalid under this convention; the code rejects it and raises the ValueError with the raw minutes value -15.

How to fix it (concrete steps)
1. Decide the intended total offset sign and encode it consistently:
   - If you want +02:15, pass (2, 15).
   - If you want +02:00, pass (2, 0).
   - If you want -02:15, pass (-2, 15).
   - If you want -00:15, pass (0, -15).
2. Example corrections:
   - Wrong: forOffsetHoursMinutes(2, -15)  => raises ValueError
   - Correct (positive): forOffsetHoursMinutes(2, 15)  => +02:15
   - Correct (negative): forOffsetHoursMinutes(-2, 15) => -02:15
   - Correct (small negative): forOffsetHoursMinutes(0, -15) => -00:15
3. If you receive a signed minutes value from elsewhere, normalize before calling the factory:
   - If minutes < 0 and hours > 0: subtract 1 from hours and set minutes = 60 + minutes, or better: compute total offset in minutes and split into sign/abs parts consistently. Example:
     total_minutes = hours * 60 + minutes
     sign = + or - based on total_minutes
     hours = abs(total_minutes) // 60 * (1 if total_minutes >= 0 else -1)
     minutes = abs(total_minutes) % 60
     then call factory with (hours, minutes) following the convention above.

Summary
- The error is not a bug in parsing; it’s the factory enforcing its input rule: you passed a positive hour with a negative minute. Fix by passing minutes with the same sign convention as hours (or normalize to total minutes and re-split) before calling the factory.
