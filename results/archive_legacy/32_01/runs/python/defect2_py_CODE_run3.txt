What happened (root cause)
- In get_paint you clamp the input to v (lines 2–4), but when you compute g you still use the original value variable instead of the clamped v. That means the clamping is ignored and g can be computed from an out-of-range value, triggering the ValueError check a few lines later.

Where it occurs
- Inside def get_paint(self, value: float) -> Tuple[int,int,int]:
  - Lines doing clamping:
    v = max(value, self._lower_bound)
    v = min(v, self._upper_bound)
  - The failing line:
    g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
  - The ValueError is raised immediately after if g < 0 or g > 255: ...

Why it fails
- Because g is computed from value (uncapped) rather than v (capped), an input outside the bounds produces a g outside 0–255 and triggers the ValueError. In other words, the intended clamping has no effect on the result because of the wrong variable.

Additional edge case
- If self._upper_bound == self._lower_bound, the denominator (self._upper_bound - self._lower_bound) is zero and will raise ZeroDivisionError. That should be guarded as well.

Concrete fixes (step-by-step)
1. Use the clamped variable v when computing g:
   g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
2. Guard against zero span (upper == lower). Decide expected behavior (e.g., return 0 or 255 or a midpoint), or raise a clear error. Example:
   if self._upper_bound == self._lower_bound:
       return (0, 0, 0)  # or raise ValueError("Bounds must differ")
3. Improve the ValueError message to include the actual computed g and the bounds so debugging is immediate:
   raise ValueError(f"Computed color component {g} out of range 0..255 for value {value} with bounds [{self._lower_bound}, {self._upper_bound}]")

Minimal corrected snippet
- Replace the g computation with the clamped v and handle zero span, for example:
    v = max(value, self._lower_bound)
    v = min(v, self._upper_bound)
    if self._upper_bound == self._lower_bound:
        raise ValueError("upper_bound must not equal lower_bound")
    g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

Why this fixes it
- Using v ensures the clamping actually affects g so g stays in the expected range. Checking bounds avoids division by zero. Improving the error message makes future failures obvious.
