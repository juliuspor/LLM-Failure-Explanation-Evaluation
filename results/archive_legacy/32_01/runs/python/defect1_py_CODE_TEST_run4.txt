What happened (where it fails)
- Method: for_offset_hours_minutes (the code you posted).
- Failing line(s): the branch that computes minutes when hours_in_minutes < 0:
  if hours_in_minutes < 0:
      minutes_offset = hours_in_minutes - minutes_offset
  else:
      minutes_offset = hours_in_minutes + minutes_offset
- Test: DateTimeZone.for_offset_hours_minutes(-2, -15)
  - hours_in_minutes = -2 * 60 = -120
  - minutes_offset (argument) = -15
  - code executes the if branch and computes minutes_offset = -120 - (-15) = -105
  - expected total minutes = -120 + (-15) = -135
- Result: the sign of the minutes part is handled incorrectly for negative hours, producing -105 minutes instead of -135; that produces the wrong zone id and the assertion fails.

Why it fails (root cause)
- The code wrongly subtracts minutes_offset when hours_in_minutes is negative. Subtracting a negative minutes argument turns it into addition instead of adding the negative minutes. In short: the logic assumes minutes_offset is always positive (or treats the minutes sign incorrectly), but the test passes a negative minutes_offset and the code flips its sign.
- This is a logic bug in the minutes combination, not an overflow or multiplication problem.

Exactly where to change it (concrete fix)
Replace the conditional adjustment with a single, correct computation of total minutes. For example:
- Replace
    if hours_in_minutes < 0:
        minutes_offset = hours_in_minutes - minutes_offset
    else:
        minutes_offset = hours_in_minutes + minutes_offset
  with
    minutes_total = hours_in_minutes + minutes_offset
  and then use minutes_total for the multiplication:
    offset = FieldUtils.safe_multiply(minutes_total, DateTimeConstants.MILLIS_PER_MINUTE)

Or, if you want to keep the variable name:
- Replace the two-branch assignment with:
    minutes_offset = hours_in_minutes + minutes_offset

Notes and alternatives (explicit)
- If your API intends minutes_offset to always be non-negative (e.g., caller should pass 15, not -15), enforce that by validating minutes_offset >= 0 and applying the sign from hours_offset. But given the test passes a negative minutes_offset, the simplest and correct behavior is to add the two signed values.
- Keep FieldUtils.safe_multiply as-is to handle overflow; no change needed there.

Summary
The bug is the line that subtracts minutes_offset when hours_in_minutes < 0 — it flips the minutes sign for negative hours. Compute total minutes as hours_in_minutes + minutes_offset (no conditional subtraction) to fix it.
