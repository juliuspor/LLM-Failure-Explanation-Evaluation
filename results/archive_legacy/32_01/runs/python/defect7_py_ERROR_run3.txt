What the error means (short)
- Python tried to call .upper() on a value that was None.
- .upper() is a string method. None has no .upper(), so Python raised AttributeError: 'NoneType' object has no attribute 'upper'.

How to find the exact location (do this first)
1. Look at the full traceback printed with the error. The top-most entry (the last file/line listed before the exception message) shows the file name, line number, and function where .upper() was called. Example traceback lines look like:
   File "/path/to/myfile.py", line 42, in process_name
     name_upper = name.upper()
2. Open that file and go to the listed line number. That is the exact place to examine.

Why it happened (root cause)
- On the indicated line you call something.upper(), but the variable or expression before .upper evaluated to None.
- That means some earlier code returned None instead of a str (or the variable was never set to a string).
Common sources:
- A function returned None. (e.g., value = maybe_get_name(); maybe_get_name returned None)
- You used dict.get(key) without a default and the key was absent → returns None. (e.g., name = mydict.get('name'); name.upper() fails)
- You called a method that returns None (like list.sort() or dict.update()) and then tried to call .upper() on its result.
- You assigned the variable conditionally and hit a branch that leaves it None.

Concrete examples and how to check them
- If traceback points to mymodule.py line 75 in function format_user:
  Line 75: greeting = get_username(user_id).upper()
  Then get_username returned None for that user_id.
  Check get_username implementation and any place where it returns None.

   - Don’t call string methods on the result of an in-place method that returns None:

       if user is None:

       return user.name
4. Add defensive tests and/or assertions:
   assert isinstance(name, str), f"expected str, got {type(name)}"
   or

  Change to:

When to raise instead of defaulting
- If a missing value indicates a bug or invalid input, raise a clear exception early instead of silently using a default string. That helps avoid later surprises.
