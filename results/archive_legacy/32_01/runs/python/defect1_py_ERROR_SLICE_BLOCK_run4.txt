What happened (where it failed)
- The ValueError was raised at lines 400–401 because minutes_offset held -15 and the code explicitly rejects any negative minute:  
  Line 400: if minutes_offset < 0 or minutes_offset > 59:  
  Line 401:     raise ValueError(f"Minutes out of range: {minutes_offset}")  
- The exception message shows the actual value: Minutes out of range: -15. So the immediate cause is that minutes_offset became negative (−15) and the guard treats any negative minute as invalid.

Why it happened (root cause)
- The code is validating the minutes part independently and assumes minutes_offset must be in the range 0..59. That assumption is fine only if the sign handling for the whole offset was applied before this check.
- The real bug is upstream: the sign handling or parsing of the offset applied the minus sign inconsistently (or not at all) to minutes. For example, when parsing an offset like "-1:15" the code probably turned hours into -1 but turned minutes into -15 (or left minutes negative) instead of parsing minutes as 15 and then applying the common negative sign to the combined offset. As a result minutes_offset contains a negative number and trips the check.

- The failing check is at lines 400–401 (the method that contains those lines — the timezone/offset parsing or setter).

  - How hours_offset and minutes_offset are produced from that string
  - Any code that applies the sign to hours but not to minutes (or that subtracts minutes into a negative value)
