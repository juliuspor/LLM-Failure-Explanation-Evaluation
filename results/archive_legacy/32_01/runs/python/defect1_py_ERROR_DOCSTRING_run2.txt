What happened
- You called the fixed-offset factory with a positive hours value and a negative minutes value (for example: hours = 2, minutes = -15).
- The factory rejects that combination and raises ValueError: "Minutes out of range: -15".

Why it fails (root cause)
- The documented factory (DateTimeZone.forOffsetHoursMinutes) requires the sign of minutes to match the sign of hours when hours is non‑zero. The allowed minute signs are:
  - If hours > 0, minutes must be 0..59 (non-negative).
  - If hours < 0, minutes must be 0..59 (non-negative) but the overall offset is negative.
  - If hours == 0, minutes may be negative or positive to represent -00:15 or +00:15.
- Passing hours = 2 and minutes = -15 violates that rule, so the method's validation code (near the top of DateTimeZone.forOffsetHoursMinutes) detects minutes < 0 while hours > 0 and throws the ValueError you saw.

Where it occurs in the code
- The exception is raised in the validation block inside DateTimeZone.forOffsetHoursMinutes (the factory method documented in your docstring), immediately before computing the total offset. The code checks ranges and sign consistency and raises for an out‑of‑range minute value.

How to fix it (concrete steps)
1. Decide the actual offset you meant:
   - If you wanted +02:15, call with (2, 15).
   - If you wanted +01:45 (i.e. 2 hours minus 15 minutes), normalize to (1, 45).
   - If you wanted -02:15, call with (-2, 15).
   - If you wanted -00:15, call with (0, -15).
2. Normalize before calling the factory. Example normalization code (Python-style) that converts any hours/minutes pair into the allowed form:

def normalize_offset(hours, minutes):
    total_minutes = hours * 60 + minutes
    sign = 1 if total_minutes >= 0 else -1
    total = abs(total_minutes)
    nh = (total // 60) * sign
    nm = (total % 60) * sign
    # If nh == 0, nm keeps sign of total (so 0,-15 is allowed)
    return nh, nm

Example:
- normalize_offset(2, -15) -> (1, 45) -> pass (1, 45) to DateTimeZone.forOffsetHoursMinutes
- normalize_offset(0, -15) -> (0, -15) -> pass (0, -15)

3. Alternatively, always compute the total offset in minutes/seconds and use a constructor that accepts total seconds/minutes (if available) so you avoid sign mismatch.

Summary
The error is caused by passing minutes with the wrong sign relative to hours. Fix it by normalizing the hour/minute pair (so minutes are 0..59 and have the same sign as nonzero hours) or by passing the intended total offset directly.
