What happened
- The test calls LocaleUtils.to_locale("fr__POSIX").
- LocaleUtils.to_locale parses the string and validates it against the expected locale format.
- The parser rejects "fr__POSIX" and raises ValueError("Invalid locale format: fr__POSIX").

Why it fails (root cause)
- Your parser treats the locale syntax as language[_country][_variant] and requires that any underscore-delimited component that exists is non-empty. The input "fr__POSIX" has an empty country component (there are two underscores in a row), i.e. language = "fr", country = "" (empty), variant = "POSIX".
- The code never allows the explicit empty country case — it only accepts either:
  - "language" (no underscores),
  - "language_country" (one underscore and a non-empty country),
  - "language_country_variant" (two underscores with a non-empty country between them),
  - maybe other strict forms — but not "language__variant".
- As a result the pattern match or validation check in LocaleUtils.to_locale fails and the method reaches the line that raises the ValueError. That is the exact line doing:
  raise ValueError(f"Invalid locale format: {s}")
  (this is where the exception originates in LocaleUtils.to_locale).

Why "fr__POSIX" is a valid locale in your test
- The expected Locale('fr', '', 'POSIX') means language = "fr", country = "" (empty), variant = "POSIX". The double-underscore form is a common representation for that exact situation and should be accepted by to_locale if you want to support empty country with a variant.
