What happened (where)
- The bug is in this factory method (the block shown at lines 400–412). The code rejects any negative minutes at line 400:
  - 400: if minutes_offset < 0 or minutes_offset > 59:
  - 401:     raise ValueError(f"Minutes out of range: {minutes_offset}")

Why that's wrong (root cause)
- The method's docstring explicitly allows negative minute values in some valid combinations (for example (0, -15) → +00:-15? actually -00:15, and (-2, -15) → -02:15). The unconditional check on line 400 forbids all negative minute values, so valid inputs like (0, -15) and (-2, -15) are rejected immediately.
- Because the negative values are rejected up front, the later logic that tries to combine hours and minutes (lines 406 and 408) and compute milliseconds (line 409) never gets a chance to handle the allowed negative-minute cases.
- In short: the invalid early check (line 400) prevents legitimate combinations from being processed and is the direct cause of the failure.

Where it shows up
- The failure happens at the minutes validation branch (lines 400–401). Correct combination/adjustment code at 406/408 and the final bounds check (lines 409–412) become irrelevant because they are never reached for negative minutes.
