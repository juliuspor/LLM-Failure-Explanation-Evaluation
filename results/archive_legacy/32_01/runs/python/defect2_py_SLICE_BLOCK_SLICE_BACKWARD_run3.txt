What happens (where it fails)
- The failure is in get_paint, around these lines:
  - Line 73: g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
  - Line 76–77: if g < 0 or g > 255: raise ValueError(f"Color parameter outside of expected range: Red Green Blue")
- Two distinct problems cause the code to fail here:
  1. If self._upper_bound == self._lower_bound, line 73 raises ZeroDivisionError (division by zero) before you ever get to the if on line 76.
  2. If value is outside the interval [self._lower_bound, self._upper_bound] (or if the bounds are reversed so the denominator is negative), the computed g can be <0 or >255 and you hit the ValueError at line 76. The raise tells you a color parameter is out of range, but does not say which value or what the bounds were.

Why this is the root cause (plain language)
- The formula on line 73 assumes the lower and upper bounds define a valid non-zero range and that value lies inside that range. If either assumption is false, the computation yields a division-by-zero or a number outside the valid byte range (0–255).
- The code then raises a ValueError at line 76, but the error message is unhelpful: it is a constant string "Red Green Blue" and does not include the actual offending numbers (g, value, lower_bound, upper_bound). That leaves the developer guessing which input or bound caused the problem.

2. Validate value before scaling (before computing g) or clamp after computing:
   - If value must be inside [lower, upper], check that and raise a clear error that includes value, lower_bound, upper_bound, and the method name.
   - OR if you prefer graceful behavior, clamp value to the range so g stays within 0–255.

3. Improve the error message at line 76:
   - Replace f"Color parameter outside of expected range: Red Green Blue" with something that prints the actual numbers. For example:
