What happened
- The assertion failed because get_paint(-0.5) did not return the RGB triple (0, 0, 0). In other words, GrayPaintScale.get_paint produced a value other than black when given a negative input.

Why it happened (root cause)
There are three common, concrete mistakes that produce that exact symptom; one or more of these is almost certainly present in your GrayPaintScale.get_paint implementation:

1. No clamping of the input value
   - If get_paint uses the input value directly to compute the gray level (for example gray = int(value * 255) or similar) then a negative input yields a negative channel value (e.g. -127) or a mismatched result. That will not equal (0, 0, 0).

2. Wrong return shape or extra alpha channel
   - If get_paint returns a 4-tuple (R, G, B, A) or a numpy array instead of a plain 3-tuple of ints, the assertion (0, 0, 0) will fail even if the RGB components are zero. Example: return (0, 0, 0, 1) or return np.array([0,0,0]) != (0,0,0).

3. Floating-point vs integer and rounding issues
   - If the method returns floats like (0.0, 0.0, 0.0) while your test expects ints, Python tuple equality fails. Or if the code uses an offset or incorrect scaling factor and yields (1,1,1) instead of (0,0,0).

Where to look (exact places to check)
- Open the file that defines GrayPaintScale and inspect the get_paint method. Look for:
  - Any early arithmetic using the raw input value (value * 255, value + offset, etc.).
  - Whether the input value is clamped to the [0.0, 1.0] range.
  - The return statement (does it return 3 ints, 3 floats, 4 components, or a numpy object?).
  - Any helper methods called from get_paint (e.g., _to_rgb or _to_color) and inspect them similarly.

Concrete fixes (step-by-step)
1. Clamp the input at the start of get_paint
   - Insert (at the top of the method):
     value = max(0.0, min(1.0, value))
   - This guarantees negative inputs become 0 and >1 become 1.

2. Compute integer, 3-channel RGB explicitly
   - Use a clear conversion to 0–255 integers, e.g.:
     gray = int(round(value * 255))
     return (gray, gray, gray)
   - This ensures you return a 3-tuple of ints; for value <= 0 you will return (0,0,0).

3. Ensure no extra alpha or wrong container type is returned
   - If you currently return 4 elements, remove the alpha or make the test expect a 4-tuple.
   - If you return numpy arrays or other objects, convert to a plain tuple: return (int(...), ...)

4. Add a unit test for boundary cases
   - Add tests:
     assert gps.get_paint(-0.5) == (0, 0, 0)
     assert gps.get_paint(0.0) == (0, 0, 0)
     assert gps.get_paint(1.0) == (255, 255, 255)
     assert gps.get_paint(1.5) == (255, 255, 255)

How to confirm the fix
- After applying the clamp + integer conversion, run your test:
  - If it still fails, print repr(gps.get_paint(-0.5)) to see the exact returned object and types.
  - That output will tell you whether the problem was an extra alpha channel, floats vs ints, or something else.
