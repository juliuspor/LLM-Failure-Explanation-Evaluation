What happened (where it fails)
- In get_paint, the clamp is computed into variable v (lines 2–3 of your snippet), but the greyscale value g is computed from the original value variable on line 5:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- If value is outside the bounds, using value instead of v makes the fraction go below 0 or above 1, producing g < 0 or g > 255. That triggers the ValueError on line 8.
- So the clamping on lines 2–3 has no effect because g is calculated from the un-clamped variable.

Why it happens (root cause)
- Simple bug: you clamped into v but then ignored v and used value to compute g. The logic intended to clamp the input before mapping it to 0–255, but the code continues to map the original (possibly out-of-range) value.

Where to change it (concrete fix)
1. Use v when computing g (replace value with v).
2. Also guard against division by zero if upper and lower bounds are equal.

Exact code changes (apply at the g computation line):
- Replace
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
  with
  denom = self._upper_bound - self._lower_bound
  if denom == 0:
      g = 0  # or 255, or whatever makes sense for your application
  else:
      g = int((v - self._lower_bound) / denom * 255.0)

Why this fixes it
- Using v guarantees the input used for mapping is within the intended bounds, so g will be between 0 and 255 and will not trigger the ValueError.
- Handling denom == 0 prevents a ZeroDivisionError if bounds are equal.

Summary of exact lines to edit
- Line computing g: use v instead of value and add a denom check.
- You can leave the subsequent ValueError check, but with the above fixes it should never fire for normal bounds.
