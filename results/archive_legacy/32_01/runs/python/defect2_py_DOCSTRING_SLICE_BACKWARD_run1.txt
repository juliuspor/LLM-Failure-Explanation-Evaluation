What happens
- At line 73 you compute
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
  If self._upper_bound == self._lower_bound this is a division-by-zero and the method raises a ZeroDivisionError at that line.

Why that is the root cause
- The code does not guard against the case where upper and lower bounds are equal, so the denominator can be zero. The failure happens exactly at line 73 (division), not at the later ValueError check.

Other problems you should fix while you’re here
- The range check at lines 76–77 only examines g:

        raise ValueError(f"Color parameter outside of expected range: Red Green Blue")
  That misses any out-of-range values for r or b (if you ever compute them separately), so it can let invalid RGB values through.
- The ValueError message is unhelpful: it is literal text "Red Green Blue" and does not show which component(s) or what the numeric values are.
