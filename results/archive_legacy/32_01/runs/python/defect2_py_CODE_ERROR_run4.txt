What happened (where it failed)
- The failure occurs inside get_paint. After clamping the input to v, the code still computes g using the original variable value:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- Because value can be outside the bounds, that expression can produce g < 0 or g > 255, which makes the subsequent check fail and raise the ValueError:
  if g < 0 or g > 255:
      raise ValueError(...)

Why it fails (root cause)
- You intended to use the clamped value v when mapping into the 0–255 range, but you accidentally used the un-clamped value variable (value). Using value lets the computation produce out-of-range results when the caller passes a value outside the lower/upper bounds, so g can be negative or exceed 255 and triggers the exception.

Exactly where to change (line references relative to the snippet)
- Change the g calculation line (the third statement in the method body) to use v instead of value.
- Also handle the edge case where upper_bound == lower_bound to avoid a division-by-zero.

Concrete fixes (copy-paste ready)
1) Replace the g line with the clamped variable and guard against zero range:
    v = max(value, self._lower_bound)
    v = min(v, self._upper_bound)
    range_width = self._upper_bound - self._lower_bound
    if range_width == 0:
        g = 0  # or 255, or some defined constant for identical bounds
    else:
        g = int((v - self._lower_bound) / range_width * 255.0)

2) Optionally, clamp g to [0,255] after conversion (defensive):
    g = max(0, min(255, g))

Why this fixes it
- Using v ensures you always map a value already constrained to the expected input range, so the computed g will be within 0..255 (except for the degenerate zero-range case, which you now handle).
- Clamping g afterward prevents floating-point rounding or unexpected inputs from producing an out-of-range integer.

Minimal corrected method (illustrative):
    def get_paint(self, value: float) -> Tuple[int, int, int]:
        v = max(value, self._lower_bound)
        v = min(v, self._upper_bound)
        range_width = self._upper_bound - self._lower_bound
        if range_width == 0:
            g = 0
        else:
            g = int((v - self._lower_bound) / range_width * 255.0)
        g = max(0, min(255, g))
        return (g, g, g)

That addresses the exact root cause (using value instead of v) and the potential division-by-zero.
