What happened (where it fails)
- The ValueError is raised at the minutes check on lines 400–401:
  if minutes_offset < 0 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")
- That means the method received a minutes_offset value outside 0..59 and immediately threw the error.

Why it fails (root cause)
- The method requires minutes_offset to be in the range 0..59 regardless of whether hours_offset is positive or negative, but many callers supply a negative minutes value when the overall offset is negative (for example hours_offset = -1 and minutes_offset = -30 to mean -1:30). When callers pass a negative minutes_offset the check at lines 400–401 rejects it and you get the ValueError.
- The code’s later logic assumes minutes_offset is non-negative and applies the sign only via hours_offset (it computes hours_in_minutes and then subtracts minutes for negative hours). This mismatch between the API’s expected sign convention and what some callers provide is the immediate cause.
- A secondary issue that makes this error easy to miss: the method reassigns minutes_offset later (minutes_offset = hours_in_minutes - minutes_offset or + minutes_offset). Reusing the parameter name for the combined minutes value is confusing and makes it easy to miss the requirement that the input minutes must be non-negative.

Concrete evidence in your code
- Validation that fails: lines 400–401 (the check and raise).
- The sign-handling code later (after the try:) assumes the input minutes_offset is the magnitude 0..59 and applies sign using hours_offset:
    hours_in_minutes = hours_offset * 60
    if hours_in_minutes < 0:
        minutes_offset = hours_in_minutes - minutes_offset
    else:
        minutes_offset = hours_in_minutes + minutes_offset

Summary
The method fails because callers are passing a negative minutes_offset but the code explicitly rejects negative minutes (line 400–401).
