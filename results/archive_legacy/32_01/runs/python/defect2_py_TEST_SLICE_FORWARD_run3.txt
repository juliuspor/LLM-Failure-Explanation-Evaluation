What happened
- The assertion fails because GrayPaintScale.get_paint(-0.5) does not return (0, 0, 0) for a negative input.
- Root cause: get_paint treats the input value as an index or position inside the valid range but does not guard against values < 0. The code that converts the value to an array index (or to a 0..1 scale) produces a negative index, and that negative index is then used to index the color array — in Python a negative index wraps from the end. So instead of returning the first color (black) you get a color from the end of the array (likely white or some other color), causing the assert to fail.

Where it occurs
- Inside the GrayPaintScale.get_paint(...) method: the conversion step from the input numeric value to an internal index/position is un-checked. Typical problematic line patterns are:
  - index = int(value * (len(colors) - 1))
  - color = colors[index]
  Because value is -0.5, index becomes negative, and colors[index] picks an element from the array end.

Why that exact behavior happens
- In Python, list[-1] returns the last element. So a negative index does not raise an error — it silently returns the wrong element.
- The method assumed inputs are in-range (e.g. 0..1) but the test passes -0.5 and get_paint lacks input validation or clamping.

How to reproduce reliably
- Call get_paint with any negative value, e.g. get_paint(-0.5) or get_paint(-1.0). The returned color will correspond to colors[-N] instead of the first color.
