What happened (where it failed)
- The call DateTimeZone.for_offset_hours_minutes(-2, -15) goes into DateTimeZone.for_offset_hours_minutes (the code shown), and fails to produce the expected id "-02:15".
- The bug is in the block that builds minutes_offset; the wrong arithmetic is used at the two lines shown:
  - line 406: minutes_offset = hours_in_minutes - minutes_offset
  - line 408: minutes_offset = hours_in_minutes + minutes_offset

Why it failed (root cause)
- The code is trying to combine the hours and minutes parts into a single total minutes value. For negative offsets both parts should contribute negatively (so total = hours*60 + minutes, with minutes negative).
- But the two lines are swapped relative to the tested sign of minutes. For the call (-2, -15) the code takes the branch for negative minutes and executes
  minutes_offset = hours_in_minutes - minutes_offset
  which computes -120 - (-15) = -105 minutes (i.e. -1:45) instead of the correct -120 + (-15) = -135 minutes (i.e. -2:15).
- In short: the sign-handling is inverted — when minutes is negative the code subtracts minutes_offset (undoing the sign) instead of adding it; when minutes is positive it adds when it should subtract in some branches. That is why the resulting offset is wrong.

Exact place to fix
- Method: DateTimeZone.for_offset_hours_minutes
- Lines: 406 and 408 in the slice you gave.
