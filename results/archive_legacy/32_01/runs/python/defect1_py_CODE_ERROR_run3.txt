What happened (where it failed)
- The exception ValueError: Minutes out of range: -15 is raised by this line (in your method):
  if minutes_offset < 0 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")
  That check runs immediately on method entry, so minutes_offset was -15 when the method was called.

Why it happened (root cause)
- Your method assumes minutes_offset must be between 0 and 59 regardless of hours_offset. The code raises an error when minutes_offset is negative.
- The caller, however, passed minutes_offset = -15 (presumably to represent an offset of -01:15). Because of the early validation, the method rejects that input before it gets to the code that would combine hours and minutes to a total offset.

Where in the code this is located
- The failing check is the minutes range validation (the third if) near the top of for_offset_hours_minutes:
    if minutes_offset < 0 or minutes_offset > 59:
        raise ValueError(f"Minutes out of range: {minutes_offset}")
- The later block that combines hours and minutes is not reached because the method has already raised the error.

How to fix it (concrete options and exact changes)
You have two clear choices — pick the one that matches the intended API:

1) Require callers to pass minutes as a non-negative value (0..59)
   - Document the API so callers send minutes_offset as a positive 0..59 and a negative hours_offset if the total offset is negative.
   - No code changes needed except documentation. Example: offset -1:15 should be passed as hours_offset=-1, minutes_offset=15 (not -15).

2) Allow callers to pass a negative minutes_offset and normalize inside the method
   - Change the validation and the combining logic. Replace the minutes check and the combine block with code that accepts minutes in -59..59 and computes total minutes correctly.

   Example replacement (exact lines to use):
       # accept minutes in -59..59, because callers may supply negative minutes
       if minutes_offset < -59 or minutes_offset > 59:
           raise ValueError(f"Minutes out of range: {minutes_offset}")

       hours_in_minutes = hours_offset * 60
       # normalize: total minutes is hours_in_minutes plus minutes_offset,
       # but if hours is negative and minutes provided as negative, that already works.
       total_minutes = hours_in_minutes + minutes_offset

       # then use total_minutes for the rest:
       offset = FieldUtils.safe_multiply(total_minutes, DateTimeConstants.MILLIS_PER_MINUTE)

   - This makes the method accept minutes like -15 and combine them correctly (-1 * 60 + -15 = -75).

Which to choose
- If your public API is supposed to take minutes only as 0..59, choose option (1) and fix callers (they must pass 15, not -15).
- If callers already sometimes pass negative minutes, choose option (2) so the method accepts that form and normalizes properly.

Summary
- The code fails because minutes_offset = -15 is rejected by the early minutes validation. Fix by either requiring callers to pass non-negative minutes (0..59) or by changing the validation/combining logic so negative minutes are allowed and combined correctly.
