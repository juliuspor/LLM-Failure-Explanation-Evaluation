What happened (where it fails)
- In get_paint(), you clamp the incoming number to v on these two lines:
  - v = max(value, self._lower_bound)
  - v = min(v, self._upper_bound)
- Immediately after that you compute g using the original value, not the clamped v:
  - g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

Because you used value instead of v, if value is outside the bounds the computed g can be less than 0 or greater than 255 and the code will raise the ValueError you check for. That is the root cause: the clamp has no effect on the computed color since you never use v.

Why it happened (root cause)
- The clamping lines intend to make sure the input is in range, but the calculation still references the original un-clamped variable value. So the clamp is effectively ignored for the color calculation.

Exact places to change (step-by-step fixes)
1. Use the clamped variable v when computing g:
   - Replace (value - self._lower_bound) with (v - self._lower_bound).
2. Guard against division by zero if lower and upper bounds are equal:
   - If self._upper_bound == self._lower_bound, decide a fallback (e.g., use 0 or 255 or mid gray).
3. Improve the error message to show the actual out-of-range number (optional but helpful).
4. Optionally clamp g to [0,255] after calculation to be defensive.

Corrected minimal code
- Replace the g line with something like:
  if self._upper_bound == self._lower_bound:
      g = 0  # or 127 or 255, whatever you want as fallback
  else:
      g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

- Keep or improve the range check:
  if g < 0 or g > 255:
      raise ValueError(f"Color parameter outside expected range: {g}")

Summary
- The clamp uses v but g is computed from value. Use v in the computation and handle the special case of equal bounds to prevent the failure.
