What happens
- The bug is in for_offset_hours_minutes at the two lines that compute the total minutes by mutating minutes_offset:
  - if hours_in_minutes < 0:
      minutes_offset = hours_in_minutes - minutes_offset
  - else:
      minutes_offset = hours_in_minutes + minutes_offset
- These lines treat minutes_offset as if it is always a non-negative magnitude. When minutes_offset is negative the arithmetic becomes wrong (subtracting a negative becomes addition), so certain sign combinations produce the wrong total offset.

Where it fails (concrete examples)
- Example 1 (should raise ValueError): hours_offset = 2, minutes_offset = -15
  - Expected: this combination is invalid per the docstring (positive hours with negative minutes).
  - Actual: hours_in_minutes = 120, code takes 120 + (-15) = 105 => +01:45 (wrong and should not have been accepted).
- Example 2 (wrong result): hours_offset = -2, minutes_offset = -15
  - Docstring says (-2, -15) -> -02:15 (i.e. -135 minutes).
  - Actual: hours_in_minutes = -120; code computes -120 - (-15) = -105 => -01:45 (wrong sign/amount).

Why this happens (root cause)
1. The code reuses the minutes_offset parameter as if it were always a non-negative magnitude, instead of treating minutes_offset as a signed offset or using its absolute value where appropriate.
2. The branch for negative hours subtracts minutes_offset directly: hours_in_minutes - minutes_offset. If minutes_offset is negative this becomes hours_in_minutes + abs(minutes), which is the opposite of the intended behavior.
3. The code also does not enforce the rule in the docstring that minutes must not be negative when hours is positive (so invalid inputs are accepted rather than rejected).

Where in the method
- The problematic lines are inside the try block after hours_in_minutes is computed:
  - the "if hours_in_minutes < 0: minutes_offset = hours_in_minutes - minutes_offset" line
  - the "else: minutes_offset = hours_in_minutes + minutes_offset" line
- Those lines then feed into FieldUtils.safe_multiply(...) and the wrong offset propagates.

How to reproduce quickly
- Call for_offset_hours_minutes(2, -15) and observe it returns +01:45 instead of raising ValueError.
- Call for_offset_hours_minutes(-2, -15) and observe it returns -01:45 instead of -02:15.

   - Insert this validation after the existing range checks and before doing arithmetic.

Minimal corrected outline (illustrative)

Why these steps fix it
- Using abs(minutes_offset) ensures arithmetic always treats minutes as a magnitude to be added to or subtracted from the hours component consistently.
- Enforcing the sign rule rejects invalid inputs that the docstring says should be rejected.
- Using a new variable prevents accidental reuse/overwrite of input parameters.
