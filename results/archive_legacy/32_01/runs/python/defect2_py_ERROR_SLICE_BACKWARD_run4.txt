Why it happened (root cause)
- g is derived by mapping value from the range [self._lower_bound, self._upper_bound] into 0..255. If value is outside that input range, the computed g will be <0 or >255 and the code raises the error.
- There are two common ways this happens:
  1) value is simply outside the expected input range (e.g. value = 2 while lower=0 and upper=1 → g = 510).
  2) self._upper_bound ≤ self._lower_bound (invalid bounds) which either causes a wrong g (if equal floating behavior) or a division problem. (If upper==lower a ZeroDivisionError would occur at the division, but the current stack shows the ValueError case so the typical cause is value outside the [lower, upper] interval.)
- The error message itself is unhelpful: "Red Green Blue" gives no clue about which parameter, what the numeric values were, or where in code it happened.
