What happened
- The call gps.get_paint(-0.5) caused get_paint to compute a color-channel value g that is negative.
- In get_paint, at lines 76–77, the code checks if g is outside 0..255 and raises a ValueError:
  76 |         if g < 0 or g > 255:
  77 |             raise ValueError(f"Color parameter outside of expected range: Red Green Blue")
- Because g < 0, the condition is true and the ValueError is raised. That is the direct cause of the failure.

Why it failed (root cause)
- The method get_paint accepts an input that can produce a g < 0 (for example an input of -0.5) but then immediately rejects any out-of-range g by raising an exception. The code does not clamp or otherwise handle inputs outside the expected range before converting to a color channel.
- The error message is also misleading: it does not include the actual computed value or identify which parameter failed; it just prints the fixed text "Red Green Blue", which makes debugging harder.

Where it occurred
- Inside GrayPaintScale.get_paint, lines 76–77 (the range check and the raised ValueError).

Concrete fixes (pick one depending on intended behavior)
1) If you want to clamp out-of-range values to the valid range (common for paint/color functions)
   Steps:
   - Replace the raise with a clamp operation before using g.
   - Example change at lines 76–77:
     - Before: if g < 0 or g > 255: raise ValueError(...)
     - After: g = max(0, min(255, g))
   - This makes get_paint(-0.5) produce 0 and return (0,0,0).

2) If you want to treat out-of-range inputs as programmer/user errors and keep raising, make the error informative
   Steps:
   - Raise ValueError that includes the method name, the invalid value, and the allowed range.
   - Example change at line 77:
     - raise ValueError(f"GrayPaintScale.get_paint: computed channel g={g} outside 0..255")
   - This tells the caller exactly what went wrong and where.

3) If the input domain should be validated earlier (e.g., you expect input value in [0.0, 1.0])
   Steps:
   - Validate the input parameter at the start of get_paint (or in the caller).
   - Example:
     - if not (0.0 <= value <= 1.0): raise ValueError(f"GrayPaintScale.get_paint: input value {value} not in 0.0..1.0")
   - This prevents deriving a negative g and makes the contract clear.

Which change to choose
- Clamp (option 1) if you want tolerant behavior where out-of-range inputs produce the nearest valid color.
- Informative error (option 2) if out-of-range inputs indicate a bug and should be fixed by the caller.
- Input validation (option 3) if the public API should enforce a specific input domain and fail early with a clear message.

Small improvement to the message already present
- At minimum, replace the opaque string "Color parameter outside of expected range: Red Green Blue" with one that shows the offending channel and value (see option 2), so future failures point straight to the cause.
