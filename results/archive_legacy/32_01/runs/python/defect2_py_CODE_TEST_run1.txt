What happened (where it fails)
- In get_paint(), you clamp the input value into the local variable v:
  v = max(value, self._lower_bound)
  v = min(v, self._upper_bound)
  This is lines 2–4 of the shown method.
- Immediately after that you compute g using the original value variable, not the clamped v:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
  That is the failure point.
- Because you used value instead of v, a test like gps.get_paint(-0.5) (with _lower_bound >= 0) produces a negative numerator, so g becomes < 0 and then the subsequent range check raises ValueError:
  if g < 0 or g > 255:
      raise ValueError(...)

Why it fails (root cause)
- The intent was to compute the gray level from the clamped value v. Using the original value undoes the clamping and allows g to be out of [0,255], which triggers the ValueError.
- Secondary edge case: if self._upper_bound == self._lower_bound, the denominator becomes zero and the division will raise ZeroDivisionError. That is not handled by the current code.

Precise fixes (step‑by‑step)
1. Use the clamped variable v when computing g:
   Replace
     g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
   with
     g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

2. Defend against zero range (optional but recommended):
   - At the top of get_paint(), after computing v, check:
       if self._upper_bound == self._lower_bound:
           # whatever makes sense: return (0,0,0) or (255,255,255) or raise a clearer error
           return (0, 0, 0)
     or raise ValueError("upper and lower bounds must differ")

3. Keep the existing clamping/range check if you still want it:
   - After computing g from v, you can clamp to [0,255] instead of raising, e.g.:
       g = max(0, min(255, g))

Minimal corrected implementation
- Change the g calculation to use v:
    g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

- Add a zero‑division guard if needed.

Why this will make your test pass
- With v clamped to the lower bound when value is -0.5, the numerator becomes 0 and g becomes 0, so the method returns (0, 0, 0) instead of raising ValueError.
