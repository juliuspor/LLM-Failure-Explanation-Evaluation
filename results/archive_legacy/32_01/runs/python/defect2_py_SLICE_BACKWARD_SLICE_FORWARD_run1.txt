What happens (where it fails)
- The failure happens inside get_paint (line numbers from your slice). At line 73 you compute
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
  and at line 77 you immediately check g for being inside 0..255 and raise a ValueError if it is not.

Why it fails (root causes)
1. Value outside bounds produces out-of-range g
   - If value is less than self._lower_bound or greater than self._upper_bound then the fraction on line 73 will be <0 or >1, so g will be <0 or >255.
   - That makes the if on line 76 true and causes the ValueError on line 77. The code is failing because get_paint is being called with a value outside the expected bounds.

2. Division-by-zero when bounds are equal
   - If self._upper_bound == self._lower_bound then the expression on line 73 divides by zero, raising ZeroDivisionError before you even get to the range check. That is a second, distinct way get_paint can fail.

3. Confusing error message
   - The ValueError raised at line 77 uses the fixed text "Color parameter outside of expected range: Red Green Blue". That gives no useful detail: it does not say which parameter failed (g here), what the offending value was, or what the expected range was. That makes diagnosing calls that supply an out-of-range value harder.

         raise ValueError(f"upper_bound equals lower_bound ({self._upper_bound}); cannot compute color scale")
   - This guarantees you never divide by zero.

2. Decide how to handle values outside [lower_bound, upper_bound]

     - After computing the fraction, clamp it to [0.0, 1.0] before scaling to 0..255.
     - Example steps:
