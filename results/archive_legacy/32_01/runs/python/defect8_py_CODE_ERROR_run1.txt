What happened (where it fails)
- Method: to_locale
- The failure happens in the branch after the language check, at these lines:
  - ch3 = locale_str[3]
  - ch4 = locale_str[4]
  - if ch3 < 'A' or ch3 > 'Z' or ch4 < 'A' or ch4 > 'Z':
    raise ValueError(...)
- For the input "fr__POSIX" those lines see ch3 == '_' (the second underscore). The code treats anything other than two uppercase letters in positions 3–4 as a format error and raises ValueError: Invalid locale format: fr__POSIX.

Why it fails (root cause)
- The code assumes that when a locale has more than the 2-letter language part it must have a two-letter uppercase country code at positions 3–4. It does not allow an empty country with an immediate variant.
- But "fr__POSIX" is a valid convention: language "fr", empty country (represented by a second underscore), and variant "POSIX". In that string the second underscore occupies position 3, not an uppercase country letter, so the ch3/ch4 uppercase check incorrectly rejects it.

Exactly where the check is wrong
- The incorrect assumption is enforced by the ch3/ch4 uppercase test (the country validation) immediately after checking locale_str[2] == '_'. That test does not permit the pattern "_<underscore><variant>" (i.e. empty country) which is present in "fr__POSIX".

How to fix it (concrete steps)
1. Detect the empty-country case: if locale_str[3] == '_' then treat country as "" and the variant starts at index 4.
2. Otherwise validate that locale_str[3:5] are two uppercase letters and behave as you already do.
3. Return the right Locale accordingly.

Minimal code change (replace the country/variant handling block)
- Current block (problem lines):
  ch3 = locale_str[3]
  ch4 = locale_str[4]
  if ch3 < 'A' or ch3 > 'Z' or ch4 < 'A' or ch4 > 'Z':
      raise ValueError(f"Invalid locale format: {locale_str}")
  
  if length == 5:
      return Locale(locale_str[0:2], locale_str[3:5])
  else:
      if locale_str[5] != '_':
          raise ValueError(f"Invalid locale format: {locale_str}")
      return Locale(locale_str[0:2], locale_str[3:5], locale_str[6:])

- Replace with (explicit behavior):
  # allow either a two-letter uppercase country OR an empty country indicated by another '_'
  if locale_str[3] == '_':
      # empty country, variant starts at index 4
      return Locale(locale_str[0:2], "", locale_str[4:])
  else:
      ch3 = locale_str[3]
      ch4 = locale_str[4]
      if ch3 < 'A' or ch3 > 'Z' or ch4 < 'A' or ch4 > 'Z':
          raise ValueError(f"Invalid locale format: {locale_str}")
      if length == 5:
          return Locale(locale_str[0:2], locale_str[3:5])
      else:
          if locale_str[5] != '_':
              raise ValueError(f"Invalid locale format: {locale_str}")
          return Locale(locale_str[0:2], locale_str[3:5], locale_str[6:])

Why this fix works
- It explicitly recognizes the valid "language + '_' + '_' + variant" form (empty country) and sets country to "" and variant to the substring after the second underscore.
- It keeps the existing strict validation for the normal two-letter country case.

Result
- After this change to to_locale, the input "fr__POSIX" will be accepted and parsed as Locale("fr", "", "POSIX") instead of raising ValueError.
