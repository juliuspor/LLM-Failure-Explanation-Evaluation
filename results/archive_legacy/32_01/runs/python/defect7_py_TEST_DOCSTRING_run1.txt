What happened
- The test calls ClassUtils.to_class(['Test', None, 99.0]) and expects [str, type(None), float].
- Instead of returning that list, to_class throws an AttributeError when it sees the None element. The assertion never runs because the method raised.

Why it happened (root cause)
- The docstring contains contradictory statements:
  - One line says: "If any of these objects is None, a None element will be inserted into the array."
  - Another line under Raises says: "AttributeError: If any element is None"
- The implementation follows the "Raises AttributeError" behavior. When to_class encounters the None element it either:
  - directly raises AttributeError in the None-handling branch, or
  - tries to call a member (like .__class__ or getClass()) on the element without a None check, which causes an AttributeError/exception at that point.
- Because the method raises when it encounters None, the test expecting a None entry in the returned list fails.

Where it occurs
- In the ClassUtils.to_class method: in the code path that handles elements of the input array. Specifically in the branch that processes each element and decides what Class/type to put into the result list. That is the exact place to inspect and fix:
  - the None-check branch (the code that either raises AttributeError or forgets to treat None as a valid element),
  - or the place where the code calls a member of the element without checking for None.

How to fix (clear, actionable steps)
You must choose which behavior you want and make the code and documentation consistent. Two concrete options:

Option A — Return None entries (make test pass)
1. Edit ClassUtils.to_class so that when iterating elements it explicitly handles None:
   - Pseudocode:
     for i, elem in enumerate(array):
         if elem is None:
             result[i] = None           # <- insert None, do not raise
         else:
             result[i] = elem.__class__ # or type(elem), or the current logic
2. Remove the "Raises: AttributeError: If any element is None" line from the docstring and ensure the docstring still says "If any of these objects is None, a None element will be inserted..."
3. Add or update unit tests to cover input arrays with None so the behavior is exercised.

Option B — Keep raising (change test)
1. If you want to treat None as invalid, update the docstring to state clearly that None elements cause an AttributeError and keep the implementation unchanged.
2. Change the test to expect an exception:
   - Wrap the call in a context that asserts AttributeError is raised (or adjust expected behavior accordingly).
3. Add a unit test that documents the reason for rejecting None (so future maintainers understand the design decision).

Suggested minimal code change for Option A (example in Python-like pseudocode)
- Replace the None-handling region with:
  for i, elem in enumerate(array):
      if elem is None:
          result[i] = type(None) or None  # choose type(None) if you want the class object, or None to match docstring
      else:
          result[i] = type(elem)

Notes to choose the exact value for a None entry
- Your test expects type(None) (i.e. type(None) == <class 'NoneType'>) as the element representing None. The docstring says "a None element will be inserted" which could mean a literal None value. Decide which you want:
  - To return the class object for None, use type(None).
  - To return a literal None entry, use None.
- Make the docstring and tests consistent with that choice.

Summary
- The immediate cause of the failing test is that ClassUtils.to_class raises AttributeError on None elements while the test expects a returned entry for the None input.
- Fix by picking and implementing the intended behavior (insert None/type(None) into the result or explicitly raise), and then make the implementation, docstring, and tests consistent.
