What happened (where it fails)
- The code clamps value into v on lines 3–4:
  v = max(value, self._lower_bound)
  v = min(v, self._upper_bound)
- But on line 6 you compute g using the original value variable instead of the clamped v:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- If value lies outside the bounds, g will be outside 0–255 and the subsequent check on line 9 raises the ValueError you saw:
  if g < 0 or g > 255:
      raise ValueError(...)

Why that causes the error
- The clamp is never used in the mapping calculation because you compute g from the un-clamped value. So the ValueError is thrown exactly at the range check in this method when value is out of bounds.
- Also note a secondary potential failure: if self._upper_bound == self._lower_bound the division on line 6 will raise ZeroDivisionError; that should be handled too.

How to fix it (concrete steps)
1. Use the clamped v when computing g.
2. Handle the zero-range case before dividing.
3. Optionally, keep the final safety clamp on g or use min/max to ensure 0–255.

Corrected code example (replace the body of get_paint):

    def get_paint(self, value: float) -> Tuple[int, int, int]:
        # Clamp value to bounds
        v = max(value, self._lower_bound)
        v = min(v, self._upper_bound)
        
        # Handle degenerate range
        span = self._upper_bound - self._lower_bound
        if span == 0.0:
            g = 0  # or 255, or self._default_gray depending on desired behavior
        else:
            # Use the clamped 'v' here, not 'value'
            g = int((v - self._lower_bound) / span * 255.0)
            # Ensure integer is within 0..255
            g = max(0, min(255, g))
        
        return (g, g, g)

That fixes the root cause (line 6 using value instead of v) and prevents division-by-zero.
