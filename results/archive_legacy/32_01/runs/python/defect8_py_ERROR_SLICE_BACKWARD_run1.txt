What happened (where)
- The exception is raised in Locale.to_locale at lines 242–245:
  - ch3 = locale_str[3]
  - ch4 = locale_str[4]
  - if ch3 < 'A' or ch3 > 'Z' or ch4 < 'A' or ch4 > 'Z':
      raise ValueError(f"Invalid locale format: {locale_str}")
- The input string was "fr__POSIX". Characters by index: 0='f', 1='r', 2='_', 3='_', 4='P', ...
- At line 242 the code reads ch3 = '_' and ch4 = 'P'. Because ch3 is '_' (not in 'A'..'Z') the condition is true and the method raises ValueError("Invalid locale format: fr__POSIX").

Why it failed (root cause)
- The parser assumes that after the language and first underscore there must be a two-letter country code at positions 3–4 (both uppercase A–Z). It does not handle the case where the country is omitted but a variant is present, which is expressed by a double underscore (language + "__" + variant), e.g. "fr__POSIX".
- The code treats the '_' at position 3 as an invalid country-letter and so rejects the whole string instead of interpreting the empty country and parsing the variant.

How to fix it (concrete steps and exact place to change)
1. In to_locale (around lines 236–245), add a check for the double-underscore case before validating ch3/ch4:
   - If locale_str[2] == '_' and locale_str[3] == '_':
     - treat country as empty string ""
     - set variant = locale_str[4:] (or validate variant if needed)
   - Otherwise proceed with the existing two-letter country validation for ch3 and ch4.
2. Replace the current rigid check with logic like:
   - if locale_str[2] != '_': raise as before (format error)
   - if locale_str[3] == '_':
       country = ""
       variant = locale_str[4:]
     else:
       ch3 = locale_str[3]; ch4 = locale_str[4]
       validate ch3/ch4 are A–Z
       country = ch3+ch4
       variant = locale_str[5:] (if any)
3. Add (unit) tests covering these cases:
   - "fr"                                   (language only)
   - "fr_FR"                                (language + country)
   - "fr__POSIX"                            (language + empty country + variant)
   - "fr_FR_POSIX"                          (language + country + variant)
   - malformed inputs like "fr_F" or "f_r"  (should still raise)

Why this fix works
- It explicitly recognizes the valid, standard form where the country is omitted but a variant exists (double underscore) and stops treating the underscore as an invalid country letter. That prevents the false negative ValueError for inputs like "fr__POSIX".
