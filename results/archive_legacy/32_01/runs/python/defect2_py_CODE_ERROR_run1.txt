What happened (where it failed)
- In get_paint the code clamps the incoming value into v with:
  v = max(value, self._lower_bound)
  v = min(v, self._upper_bound)
  but then computes g using the original value, not the clamped v:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- If value is outside the bounds, that division produces a result < 0 or > 255 and the next if triggers:
  raise ValueError(f"Color parameter outside of expected range: Red Green Blue")
- So the error occurs on the g = int(...) line and the raise happens immediately after because g is out of [0,255].

Why it happened (root cause)
- A simple variable mixup: you meant to use the clamped value v for the color calculation but used value instead. Using the unclamped value allows g to be out of range.
- Also the code does not handle the case where upper_bound == lower_bound (division by zero).

Exactly where to change (lines to edit)
1. Replace the use of value in the calculation with v:
   g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
2. Add a guard for equal bounds before the division:
   if self._upper_bound == self._lower_bound:
       g = 0  # or 255 or some sensible default
   else:
       g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

Suggested fixed method (concrete code)
    def get_paint(self, value: float) -> Tuple[int, int, int]:
        # Clamp value to bounds
        v = max(value, self._lower_bound)
        v = min(v, self._upper_bound)

        # Handle degenerate range to avoid division by zero
        if self._upper_bound == self._lower_bound:
            g = 0  # choose a sensible default or raise a clear error
        else:
            # Use the clamped v (not the original value)
            g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

        # Validate range
        if g < 0 or g > 255:
            raise ValueError(f"Color parameter outside of expected range: {g}")

        return (g, g, g)

Additional recommendations
- Replace the generic error message with the actual value (see above) so debugging is easier.
- Add unit tests for:
  - value < lower_bound
  - value == lower_bound
  - value between bounds
  - value == upper_bound
  - value > upper_bound
  - upper_bound == lower_bound

That fixes the root cause (using the wrong variable) and prevents division-by-zero and unclear error messages.
