What happened (where it fails)
- At lines 76–77 the code checks only the green channel (g):
  76 |         if g < 0 or g > 255:
  77 |             raise ValueError(f"Color parameter outside of expected range: Red Green Blue")
- The function's docstring promises a ValueError when any RGB component is outside 0–255, but the implementation only tests g. That means:
  - If r or b are out of range, no error will be raised (contradicting the docstring).
  - If g is out of range the code raises, but the error message is misleading and unhelpful ("Red Green Blue" rather than showing which component and what value).

Why this is the root cause
- The logic/condition is incomplete: it only inspects g instead of verifying r, g and b.
- The error text is not specific and does not include the offending values, so a developer cannot tell which channel (r, g or b) triggered the error or what the value was.
- Because only one component is validated, callers can end up with invalid RGB tuples downstream or get a confusing exception when g alone crosses the bounds.

Exact location to change
- Modify the condition at lines 76–77 (the check that now reads "if g < 0 or g > 255: ...").

Concrete fixes (step-by-step)
1. Validate all three channels (r, g, b) instead of only g. Replace lines 76–77 with a single check that covers r, g and b.
2. Make the error message specific and include the actual values so the user can see which component failed.
3. Optionally, ensure the channels are integers (if that is required by your API): either cast or reject non-integer values explicitly.

Copy-paste replacement example for lines 76–77 (apply inside the same function):
  if not (0 <= r <= 255 and 0 <= g <= 255 and 0 <= b <= 255):
      raise ValueError(f"Color parameter outside of expected range: r={r}, g={g}, b={b}")

Optional stricter variant (also rejects non-integers):
  for name, val in (("r", r), ("g", g), ("b", b)):
      if not isinstance(val, int) or not (0 <= val <= 255):
          raise ValueError(f"Color parameter out of range or wrong type: {name}={val} (expected int 0..255)")

What you will get after the fix
- The function will raise only when one or more of r, g or b is actually invalid, matching the docstring.
- The error message will tell you which component(s) and their values, so you won't have to guess the cause or location of the problem.
