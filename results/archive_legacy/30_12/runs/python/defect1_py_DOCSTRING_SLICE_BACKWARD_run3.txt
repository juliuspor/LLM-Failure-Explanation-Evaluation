What happens
- The method fails because it rejects negative minute values even though the docstring explicitly allows minutes from -59 to +59. The failure occurs at the minutes validation in this method:
  - Lines 400–401:
    if minutes_offset < 0 or minutes_offset > 59:
        raise ValueError(f"Minutes out of range: {minutes_offset}")

Why it fails (root cause)
- The condition minutes_offset < 0 forbids any negative minutes. According to the docstring the valid minutes range is -59 .. +59, so negative minutes should be accepted in some cases (for example (-2, -15) and (0, -15) are legal). The check therefore contradicts the documented API and raises ValueError for valid inputs with negative minutes.

Where it occurs
- In the same factory method described by the docstring, at lines 400–401 (the minutes validation).

How to fix it (concrete steps)
1. Allow negative minutes in the range -59 .. +59:
   - Replace the existing range check (lines 400–401) with:
     if minutes_offset < -59 or minutes_offset > 59:
         raise ValueError(f"Minutes out of range: {minutes_offset}")

2. Enforce the sign-combination rules from the docstring:
   - Immediately after validating hours and minutes ranges, add a check that disallows positive hours with negative minutes:
     if hours_offset > 0 and minutes_offset < 0:
         raise ValueError("Positive hours with negative minutes is not allowed")

   - You may also want an explicit rule for negative hours and negative minutes being allowed (no check needed), and zero hours with negative minutes should be allowed (no check needed).

3. Preserve the existing overall-offset magnitude check (the method already raises if the computed offset exceeds +/- 23:59:59.000). Make sure that check runs after the adjusted range and sign-combination validations.

Suggested tests to add
- Valid cases that must succeed:
  - (2, 15) -> +02:15
  - (0, -15) -> -00:15
  - (-2, -15) -> -02:15
  - (-2, 15) -> -02:15
- Invalid cases that must raise ValueError:
  - (2, -15)  — positive hours with negative minutes
  - (24, 0)   — hour out of range

  - (23, 59) and (-23, -59) check overall magnitude boundary if you compute seconds/milliseconds

Summary
- The immediate bug is the minutes range check at lines 400–401 incorrectly forbidding negative minutes. Change that check to allow -59..+59 and add a sign-combination check to reject the documented invalid pair (positive hours with negative minutes).
