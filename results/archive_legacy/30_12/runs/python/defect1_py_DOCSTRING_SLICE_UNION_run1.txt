What happened (where it fails)
- The unconditional minutes check at line 400 rejects any negative minutes:
  if minutes_offset < 0 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")
- That check runs before the sign-aware adjustment at lines 406–409. As a result, a call like
  for_offset_hours_minutes(-2, -15)
  (hours_offset = -2, minutes_offset = -15) is rejected immediately with ValueError at line 400.
- According to the docstring, negative minutes are allowed when hours are zero or negative (e.g. (-2, -15) should produce -02:15). The code therefore rejects valid inputs too early.

Why it fails (root cause)
- The code enforces minutes to be in [0,59] regardless of the hours sign. The validation at line 400 is too strict and runs before the code that interprets the minute sign relative to the hour (lines 406–409).
- In short: the minutes-range check is not sign-aware. That causes valid combinations (negative minutes with non-positive hours) to be rejected.

Exactly where to change it (concrete fix steps)
1. Replace the unconditional check at line 400 with one that allows negative minutes when allowed by the hours sign:
   - First validate absolute range: minutes must be between -59 and +59.
     if minutes_offset < -59 or minutes_offset > 59:
         raise ValueError(f"Minutes out of range: {minutes_offset}")
   - Then enforce the documented rule that minutes cannot be negative if hours > 0:
     if minutes_offset < 0 and hours_offset > 0:
         raise ValueError(f"Minutes must be non-negative when hours is positive: {minutes_offset}")
2. Keep the sign-aware adjustments at lines 406–409 as they are (they convert the (hours,minutes) pair to a total-minute offset). Those adjustments will now see valid negative minutes when hours is zero or negative and produce the correct offset.
3. Ensure the final "offset too large" check at line 411 remains in place.

Why this fix works
- It enforces the documented bounds for minutes (abs ≤ 59) while allowing negative minutes when permitted by the docs (hours zero or negative).
- It prevents the forbidden case (positive hours with negative minutes) explicitly.
- It lets the existing sign-handling code (lines 406–409) run on valid inputs.
