What happened (where it fails)
- In get_paint, the method clamps the input into v on these lines:
  - v = max(value, self._lower_bound)
  - v = min(v, self._upper_bound)
  These two lines create the clamped value v.
- The code then computes g using the original value instead of the clamped v on this line:
  - g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- If value is outside the bounds, that formula produces g < 0 or g > 255 and the method raises the ValueError on the next lines.
- So the root cause is: the clamped variable v is never used; the un-clamped value is used to compute g.

Other potential failure you should handle (where it will also break)
- If self._upper_bound == self._lower_bound, the denominator is zero and the expression causes a ZeroDivisionError.

How to fix it (concrete steps)
1. Use v (the clamped value) when computing g.
2. Handle the zero-range case explicitly.
3. Optionally ensure g is bounded to [0,255] after conversion (safer than relying on the exception).

A corrected implementation (drop-in change)
    def get_paint(self, value: float) -> Tuple[int, int, int]:
        # Clamp value to bounds
        v = max(value, self._lower_bound)
        v = min(v, self._upper_bound)
        
        # Handle degenerate range
        span = self._upper_bound - self._lower_bound
        if span == 0:
            g = 0  # or 255, or 127 — choose the desired fallback
        else:
            g = int((v - self._lower_bound) / span * 255.0)
        
        # Clamp g to [0,255] to be safe
        g = max(0, min(255, g))
        
        return (g, g, g)

Why this fixes it
- The formula now uses v, so values outside the bounds are first clamped and will not produce out-of-range g.
- The zero-range check avoids division by zero.
- The final clamp guarantees a valid RGB value and avoids the unexpected ValueError raised previously.
