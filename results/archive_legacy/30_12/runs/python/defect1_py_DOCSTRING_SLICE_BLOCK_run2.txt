What happened (where)
- The failure occurs in the factory that builds a fixed-offset zone. The check at lines 400–401 is:
  if minutes_offset < 0 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")
- That check is in conflict with the docstring and accepted inputs: the docstring allows negative minutes when hours are zero or negative (examples (0, -15) -> -00:15 and (-2, -15) -> -02:15). But the code rejects any negative minutes unconditionally, so calling the factory with minutes_offset = -15 raises ValueError even though the input should be valid.

Why it fails (root cause)
- The code enforces the minutes range as 0..59, while the specification requires minutes to be in -59..+59. In short: the conditional forbids negative minutes outright instead of allowing negative minutes in the permitted combinations. This mismatch between the code (lines 400–401) and the documented behavior is the direct cause.

Concrete examples that show the bug
- Input (0, -15): doc says acceptable -> should produce -00:15. The code throws ValueError at line 400 because minutes_offset < 0.
- Input (-2, -15): doc says acceptable -> should produce -02:15. The code also throws ValueError for the same reason.
- Input (2, -15): doc says invalid and should raise ValueError. The code currently also raises ValueError for this case — but it also incorrectly raises for valid negative-minute cases.

1. Allow the full negative-to-positive minute range: change the range check so minutes_offset is allowed in -59..+59, not 0..59.
2. Enforce the rule that minutes cannot be negative when hours are positive (minutes negative with positive hours is invalid).
