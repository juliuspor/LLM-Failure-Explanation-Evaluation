What happened
- The factory method you called (DateTimeZone.forOffsetHoursMinutes or the equivalent “get time zone for hours/minutes offset” method) enforces that hours and minutes have the same sign.  
- You passed hours positive and minutes negative (for example (2, -15)). The method’s input validation rejects that and raises:
  ValueError: Minutes out of range: -15

Why it fails (root cause)
- The method checks three things:
  1. hours is between -23 and +23,
  2. minutes is between -59 and +59,
  3. minutes is not negative when hours is positive (and vice‑versa for some combinations).
- A pair like (2, -15) violates rule #3. The code treats negative minutes with positive hours as ambiguous and refuses it rather than silently normalizing to +01:45. That validation is where the exception is thrown (the line that raises ValueError when minutes < 0 && hours > 0 — the message shows the invalid minutes value “-15”).

Where it occurs
- In the factory method that builds a fixed-offset DateTimeZone (commonly named DateTimeZone.forOffsetHoursMinutes or similar). The ValueError is raised during the parameter validation block at the start of that method (the code that validates hours/minutes ranges and sign consistency).

How to fix it (clear steps)
1. Pick a consistent sign for minutes and hours before calling the factory.
   - If you intended +01:45, call with (1, 45).
   - If you intended -00:15, call with (0, -15) or (-0, 15) depending on your API (the method allows zero hours with negative minutes).
2. Or normalize a total offset in minutes into a signed hours/minutes pair yourself and pass that to the factory. Example normalization (pseudocode):
   - totalMinutes = hours * 60 + minutes  // make sure minutes carries its sign in your input
   - sign = totalMinutes < 0 ? -1 : +1
   - absMinutes = abs(totalMinutes)
   - newHours = (absMinutes / 60) * sign
   - newMinutes = (absMinutes % 60) * sign
   - call factory(newHours, newMinutes)
   This produces consistent-sign pairs and avoids the ValueError.
3. Alternatively, call any factory/API that accepts a total offset in minutes or seconds (if available) so you don’t need to split into hours/minutes.

Concrete example
- Problematic call: DateTimeZone.forOffsetHoursMinutes(2, -15)  // raises ValueError: Minutes out of range: -15
- Correct calls:
  - DateTimeZone.forOffsetHoursMinutes(1, 45)  // equivalent to +01:45
  - DateTimeZone.forOffsetHoursMinutes(0, -15) // equivalent to -00:15

Summary
The method rejects mixed-sign hour/minute pairs. Convert your desired offset into a consistent-sign (hours, minutes) pair before calling the factory, or normalize using total minutes and then split into signed hours and minutes.
