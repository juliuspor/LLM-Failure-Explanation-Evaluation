What happened (where it fails)
- The failure happens in LocaleUtils.to_locale at the lines shown: 248, 250–252.
- The code reads characters by index (for example locale_str[5]) without first checking the string length. If a shorter valid-looking input like "en_US" (length 5) is passed, the access locale_str[5] raises an IndexError instead of the intended ValueError.
- Concretely:
  - At line 248 the code returns Locale(locale_str[0:2], locale_str[3:5]) but that path only makes sense if locale_str has at least 5 characters and the separator at index 2 is an underscore.
  - At line 250 the code checks locale_str[5] != '_' — but for "en_US" there is no index 5, so Python raises IndexError before your code can raise the intended ValueError for bad format.

Why it fails (root cause)
- The code assumes certain indices exist without verifying the string length and separator positions first.
- That leads to an IndexError when the input is shorter than the code expects. The method promises to "validate the input strictly" and to raise ValueError for invalid formats, but instead an unhandled IndexError appears for some inputs because length checks are missing or in the wrong order.
