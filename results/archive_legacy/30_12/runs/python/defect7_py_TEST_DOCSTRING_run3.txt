What happened
- The test calls ClassUtils.to_class(['Test', None, 99.0]) and expects [str, type(None), float].
- The method throws an AttributeError when it reaches the None element (array[1]) instead of returning a list that contains a None slot.
- Root cause: the implementation calls a method or attribute on every element without first checking for None. In other words, the code assumes every element is an object and directly does something like element.getClass() / element.__class__ / type-calling on it — that raises AttributeError (or similar) when element is None.

Where it occurs (exact spot)
- Inside ClassUtils.to_class, in the loop that converts each array element to a class object. The failure happens when processing array[1] (the None element).
- The problematic line is the one that unconditionally invokes the element’s class-retrieval operation, e.g.:
  - Java-like: clazzes[i] = array[i].getClass();
  - Python-like: classes[i] = array[i].__class__    or    classes[i] = type(array[i])
- Because array[1] is None, calling getClass / __class__ on it triggers the AttributeError at that line.

Why it contradicts the docstring
- The docstring explicitly says “If any of these objects is None, a None element will be inserted into the array,” but the Raises section says “AttributeError: If any element is None.” The implementation follows the Raises behavior (raising) rather than the earlier description (inserting None). That mismatch explains why the test fails: the test expects insertion of None but the code raises.

Concrete, actionable fixes (steps)
1. Fix the loop to check for None before trying to access a class on the element:
   - If the input array itself is None, return None (as the docstring requires).
   - For each element:
     - If element is None, append None (or type(None) if you want the actual class object; the docstring says “a None element will be inserted,” so append None).
     - Else, append the element’s class (Java: array[i].getClass(); Python: type(array[i]) or array[i].__class__).
2. Update the docstring to remove the contradictory Raises line or make it consistent with the implementation.
3. Add unit tests:
   - test None input -> returns None
   - test contains None element -> returned list contains None at same index
   - test normal elements -> returned list contains their class objects

Example fixes

- Java-like fix:
  ```
  public static Class<?>[] to_class(Object[] array) {
      if (array == null) {
          return null;
      }
      Class<?>[] classes = new Class<?>[array.length];
      for (int i = 0; i < array.length; i++) {
          Object el = array[i];
          if (el == null) {
              classes[i] = null;
          } else {
              classes[i] = el.getClass();
          }
      }
      return classes;
  }
  ```

- Python-like fix:
  ```
  def to_class(array):
      if array is None:
          return None
      classes = []
      for el in array:
          if el is None:
              classes.append(None)
          else:
              classes.append(type(el))
      return classes
  ```

What to change in tests/doc
- If you expect the returned slot to be the class object for None itself (i.e., type(None)), adjust the code to append type(None) instead of None for None elements.
- Otherwise keep docstring and tests consistent: either say “insert None” and expect None in result, or say “insert type(None)” and expect type(None).

Summary
- The code fails because it tries to access an attribute/method on a None element (array[1]) without checking for None first. Fix by checking for None before retrieving the element’s class and make the docstring consistent with the implemented behavior.
