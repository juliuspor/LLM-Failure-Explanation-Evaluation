What happened (root cause)
- The test calls GrayPaintScale.get_paint(-0.5) expecting the darkest gray (0, 0, 0).
- GrayPaintScale.get_paint currently computes RGB from the input value and then raises ValueError when any channel is outside 0–255.
- A negative value like -0.5 produces negative channel(s) (for example r = int(255 * -0.5) → -127), so the method’s validation rejects that and raises ValueError instead of returning (0,0,0). In short: the negative input creates out-of-range RGB and the method’s range check causes the test to fail.

Where it occurred
- In GrayPaintScale.get_paint: in the code path that converts value → r,g,b and then performs the range check/raise (the "if r < 0 or r > 255: raise ValueError" style check).

Why it happened (clear, concrete)
- The docstring promises "Returns a paint (RGB tuple) for the specified value" but also documents raising ValueError when RGB are outside 0–255.
- The test expects the implementation to treat values below the valid range as the boundary (clamped to 0) and return (0,0,0). The implementation instead enforces a strict range and raises an exception.
- There is therefore a mismatch between test expectations and the current behavior: the implementation validates raw computed channels rather than normalizing/clamping the input first.

Exact fixes (pick one; each step is actionable)
1) If you want get_paint to clamp inputs and return nearest legal color (recommended for a paint/scale):

     value = max(0.0, min(1.0, value))

     r = int(round(255 * value))
     r = max(0, min(255, r))
   - Repeat for g, b and return (r, g, b).
   - Where to change: inside GrayPaintScale.get_paint, add the clamp as the first lines of the method, and replace the strict range-raise with the clamping shown above.
