What happened (root cause)
- The failure comes from LocaleUtils.to_locale (the method named in your docstring).
- The input string "fr__POSIX" contains an empty country field (there are two underscores in a row). Your method treats the country code as required to be exactly two uppercase letters, so the country-check fails and the method raises ValueError("Invalid locale format: fr__POSIX").
- In other words, the parser rejects the valid Java/JDK convention for "language + empty country + variant" (which is expressed as language + "__" + variant). The code enforces "country length == 2 and uppercase" unconditionally, so an empty country is considered invalid.

Why it fails where it does
- The ValueError is raised inside LocaleUtils.to_locale during the strict validation step that checks:
  - language is lowercase of expected length,
  - separator is underscore,
  - country is uppercase and the correct length.
- Because the country substring in "fr__POSIX" is "", the country-length/case check fails and the method raises the ValueError for that input.

How to fix it (concrete steps)
1. Decide whether you want to accept the Java convention "language__variant" (double underscore) for empty country:
   - If yes, change the validation to allow country == "" only when a non-empty variant follows.
   - If no, require callers to supply a two-letter country or to use a different representation, and document that clearly.

2. Implementation approach (recommended): split on '_' and validate by number of parts rather than forcing country length unconditionally:
   - parts = locale_str.split('_')
   - if len(parts) == 1:
       - validate parts[0] is a lowercase language (or allow empty if you want None)
   - elif len(parts) == 2:
       - validate parts[0] language and parts[1] country (2 uppercase letters)
   - elif len(parts) == 3:
       - allow parts[1] to be '' (empty country) or 2 uppercase letters
       - validate parts[2] (variant) is non-empty
   - else:
       - raise ValueError

3. Minimal code fix example (pseudo/Python):
   - After splitting:
       if len(parts) == 3:
           language, country, variant = parts
           if not language.islower() or len(language) not in (2,):
               raise ValueError
           if country != "" and (len(country) != 2 or not country.isupper()):
               raise ValueError
           if variant == "":
               raise ValueError
   - This accepts "fr__POSIX" (language 'fr', empty country, variant 'POSIX').

4. Tests to add
   - Accept: "en", "en_GB", "en_GB_xxx", "fr__POSIX"
   - Reject: "EN", "en_gb", "fr__" (empty variant), "fr_GB_" (trailing underscore with empty variant)

Notes about JDK behavior
- The docstring already mentions JDK differences about variant casing. The Java convention for an empty country with a variant uses two underscores. If your code aims to mimic Java Locale parsing, you should explicitly support that form.

Summary
- The code fails because LocaleUtils.to_locale enforces a non-empty two-letter uppercase country unconditionally, so it rejects "fr__POSIX" where the country is intentionally empty. Fix by allowing an empty country when a variant is present (adjust the validation branch that raises the ValueError).
