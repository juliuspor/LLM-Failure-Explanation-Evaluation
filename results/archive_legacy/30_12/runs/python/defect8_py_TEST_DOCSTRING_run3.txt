What failed
- The test LocaleUtils.to_locale('fr__POSIX') expects Locale('fr', '', 'POSIX') — a language "fr", an empty country, and a variant "POSIX".
- The implementation in LocaleUtils.to_locale rejects that input during its country-code validation and raises ValueError instead of returning the Locale with an empty country.

Why it failed (root cause)
- to_locale splits the input on underscores and then validates the country part with a strict length check that requires a two‑letter country code whenever a second component exists.
- For the input "fr__POSIX" the split produces three components: ["fr", "", "POSIX"]. The country component is the empty string.
- The existing validation treats an empty second component as an invalid country (for example by checking len(country) != 2 or by requiring country to match [A-Z]{2}) and raises ValueError. It does not allow the valid form language + "_" + "_" + variant where country is intentionally empty.

Where it happens
- In LocaleUtils.to_locale, in the branch that validates the country component after splitting the input. Concretely: after splitting the string into parts[0] (language), parts[1] (country), parts[2] (variant), the code validates parts[1] with a strict two-character check and throws the exception when parts[1] == "".

How to fix it (clear steps)
1. Split at most into three parts: language, country, variant.
   - parts = locale_str.split('_', 2)
2. Assign with defaults:
   - language = parts[0]
   - country = parts[1] if len(parts) > 1 else ''
   - variant = parts[2] if len(parts) > 2 else ''
3. Validate language as before (lowercase letters, correct length).
4. Change the country validation to allow an empty country when a variant is present:
   - If country != '' then require country to be exactly two uppercase letters (e.g. match r'^[A-Z]{2}$')
   - Do not treat country == '' as invalid if variant is present (this is the legitimate "empty country, non-empty variant" form)
5. Validate variant as before (if there are additional rules).
