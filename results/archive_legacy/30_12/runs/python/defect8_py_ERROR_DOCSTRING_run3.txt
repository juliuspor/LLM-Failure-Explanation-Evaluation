What happened
- The error is thrown from LocaleUtils.to_locale because the input string "fr__POSIX" does not pass the method’s strict format checks.
- The code strictly accepts only these shapes:
  - "ll" (language only, 2 lowercase letters)
  - "ll_CC" (language + country, language 2 lowercase, country 2 UPPERCASE)
  - "ll_CC_VVV" (language + country + variant, with the same language/country casing rules)
- Your input "fr__POSIX" is parsed into three parts split on '_' : language = "fr", country = "" (empty), variant = "POSIX". The method treats this as the three-part form and then validates the country and fails because the country part is not two uppercase letters. That triggers the ValueError: Invalid locale format: fr__POSIX.

Where it occurs
- In the method LocaleUtils.to_locale — specifically in the branch that handles the three-part form (language_country_variant). That validation code checks:
  - language must be lowercase and the right length
  - country must be uppercase and length 2
  - variant must be non-empty
- Because country == "" it fails the country check and raises ValueError.

Why this is the root cause (concise)
- Double underscore indicates “empty country, non-empty variant” (language + "" + variant).
- The method’s validation does not allow an empty country when the input is treated as the three-part form, so the format is rejected.

Concrete fixes (choose one)

1) Fix the caller (preferred if you control the strings)
   - Use a format the method accepts. For example:
     - If you want language=fr and variant=POSIX with no country, Java’s Locale long-form uses "fr__POSIX", but this implementation rejects that. If you can, convert to a Locale programmatically instead of passing the string:
       - new Locale("fr", "", "POSIX")
     - Or pass a two-part or normal three-part value with a real country, e.g. "fr_FR_POSIX" or just "fr".

2) Relax the validation in LocaleUtils.to_locale (if you can change the method)
   - In the branch that handles three parts (language, country, variant), add an allowance for an empty country:
     - If parts length == 3:
       - allow parts[1] == "" (empty) provided parts[2] (variant) is valid
       - keep language and variant checks as-is
   - Example change (pseudocode):
     - if len(parts) == 3:
         if not valid_language(parts[0]) or not valid_variant(parts[2]):
             raise ValueError(...)
         if parts[1] != "" and not valid_country(parts[1]):
             raise ValueError(...)
   - This permits "fr__POSIX".

3) Normalize the input before calling to_locale
   - Replace the double-underscore form with an explicit empty-country constructor call in your code:
     - If input.matches("^[a-z]{2}__.+$"): call new Locale(language, "", variant) instead of LocaleUtils.to_locale(str)
   - Or transform "fr__POSIX" into an object: Locale("fr", "", "POSIX") rather than a string parsed by the strict parser.

How to reproduce
- Call LocaleUtils.to_locale("fr__POSIX") — it will split on '_' into ["fr", "", "POSIX"], then fail the country validation and raise ValueError.

Summary
- Root cause: double underscore yields an empty country; LocaleUtils.to_locale’s strict three-part validation requires a two-letter uppercase country and rejects empty country when variant is present.
- Fix by (A) providing a different input form or construct the Locale directly, or (B) update the three-part validation to allow an empty country when a variant is present.
