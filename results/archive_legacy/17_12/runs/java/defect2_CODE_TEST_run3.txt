The code fails because of an issue in the `getPaint` method within the `GrayPaintScale` class. Here's a clear and straightforward explanation of the problem:

### What Happened:
The assertion in the test is expecting the color returned for a value of `-0.5` to be `Color.black`, which means the RGB values should be `(0, 0, 0)`.

### Where It Occurred:
The issue is in the `getPaint` method, specifically in these lines:
```java
double v = Math.max(value, this.lowerBound);
v = Math.min(v, this.upperBound);
int g = (int) ((value - this.lowerBound) / (this.upperBound - this.lowerBound) * 255.0);
return new Color(g, g, g);
```

### Why It Happened:
The value `-0.5` is initially adjusted by the bounds check using `Math.max` and `Math.min`, which ensures `v` stays within the range `[lowerBound, upperBound]`. However, the calculation for `g` uses the original `value` instead of the adjusted `v`. As a result, even though `v` gets constrained correctly, the calculation proceeds with `value`, resulting in potentially incorrect values for `g`.

For a `value` of `-0.5`, `g` is calculated as:
```java
g = (int) ((-0.5 - 0.0) / (1.0 - 0.0) * 255.0);
```
This results in:
```java
g = (int) (-0.5 * 255.0);  // g = -127
```
Negative values for color components are not valid, and it gets mapped incorrectly when creating `new Color(g, g, g)`.

### Suggested Fix:
Modify the calculation of `g` to use `v` instead of `value`, which respects the bounds:
```java
public Paint getPaint(double value) {
    double v = Math.max(value, this.lowerBound);
    v = Math.min(v, this.upperBound);
    int g = (int) ((v - this.lowerBound) / (this.upperBound - this.lowerBound) * 255.0);
    return new Color(g, g, g);
}
```
This change ensures that when `value` is below `lowerBound`, it calculates `g` using `v` that is already adjusted, and hence returns `Color.black` as expected by the `assertTrue` in the test.
