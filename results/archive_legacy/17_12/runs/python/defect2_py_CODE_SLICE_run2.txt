The issue in the code arises in the `get_paint` method, particularly in the calculation and usage of the variable `g`. The relevant section is:

```python
g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
```

### What happened:
The calculation for `g` is intended to map a `value` within the `lower_bound` and `upper_bound` to a grayscale RGB value between 0 and 255. However, due to the incorrect handling of the `value`, `g` can fall outside the range of 0 to 255.

### Why it happened:
1. **Incorrect Clamping Logic:**
   The values are clamped to be within the bounds after using `value` directly in the calculation. This means that even if `value` is clamped correctly, the calculation for `g` still uses the original `value`, which might be out of bounds.
   
   Current logic:
   ```python
   v = max(value, self._lower_bound)
   v = min(v, self._upper_bound)
   ```
   But `v` is never used; `value` is directly used in the calculation, which is incorrect.

2. **Calculation Flaw:**
   If `value` is outside the bounds initially, the calculation does not correctly map it into the 0-255 range because it uses `value` instead of using the clamped `v`.

### Where it occurred:
The issue takes place on the line where `g` is calculated and directly in the method `get_paint` inside the `GrayPaintScale` class.

### Suggested Steps to Fix:
1. Use the clamped value `v` rather than `value` for calculating `g`:
   
   Modify:
   ```python
   g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
   ```
   To:
   ```python
   v = max(min(value, self._upper_bound), self._lower_bound)  # Correctly clamp `value`
   g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
   ```

2. With this change, the calculation will consistently use the correctly clamped value `v`, ensuring `g` remains within the 0-255 range and avoiding unnecessary exceptions.
