The issue with the `get_paint` method in the `GrayPaintScale` class lies in the calculation of the grayscale value `g`. The error occurs when you try to get the paint for a value that is the same as `value` without properly clamping it. Here's a breakdown:

### What Happened

In the `get_paint` method:

```python
g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
```

The calculated `g` value can sometimes exceed the bounds of 0 to 255, depending on the input `value`. This happens because the clamping logic (`v = max(value, self._lower_bound)` and `v = min(v, self._upper_bound)`) is applied to `v`, but the calculation for `g` uses the original `value`. 

### Why It Happened

The issue is that `g` is computed using `value`, which may not be clamped to stay within the bounds of `self._lower_bound` and `self._upper_bound`. Thus, when `value` exceeds these bounds, `g` can end up outside the range of 0 to 255.

### Where It Occurred

The problem is in the `get_paint` method, specifically at the line where `g` is calculated:

```python
g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
```

### Steps to Fix

1. Modify the `g` calculation to use the clamped value `v` instead of the original `value`.
   
   ```python
   v = max(value, self._lower_bound)
   v = min(v, self._upper_bound)
   g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
   ```

This change ensures that the calculation of `g` always uses a value within the specified bounds, preventing it from going outside the valid range for RGB values.
