What happened (where it fails)
- The code fails at these lines:
  ch3 = locale_str[3]
  ch4 = locale_str[4]
  if ch3 < 'A' or ch3 > 'Z' or ch4 < 'A' or ch4 > 'Z':
      raise ValueError(f"Invalid locale format: {locale_str}")
- For the input 'fr__POSIX', locale_str[3] is '_' (because the string has two underscores in a row to indicate an empty country) and locale_str[4] is 'P'. The check insists both ch3 and ch4 are uppercase letters A–Z, so ch3 == '_' fails that test and the method raises ValueError.

Why it fails (root cause)
- The method assumes that when a locale has the form language + '_' + rest, the country part is always exactly two uppercase letters (positions 3 and 4). It does not allow the valid alternative where the country is empty and the variant is present, which is encoded as two underscores in a row (language + '__' + variant).
- For 'fr__POSIX' the correct parsing is language = 'fr', country = '' (empty), variant = 'POSIX'. The code never checks for the "empty country" pattern (underscore at index 3) before validating ch3/ch4, so it rejects that valid input.

Exactly where to change it (concrete fix)
1. After verifying locale_str[2] == '_' (the existing check), add an explicit case for an empty country:
   - If locale_str[3] == '_', treat country as empty and set variant = locale_str[4:].
   - Return Locale(locale_str[0:2], '', locale_str[4:]) (or validate that there is at least one character for variant if you need).
2. Only when locale_str[3] is not '_' should you validate that ch3 and ch4 are uppercase letters and proceed with the existing country/variant handling.

Example minimal change (conceptual lines to add)

    ch3 = locale_str[3]

Other notes
- The initial length check is awkward but not the immediate cause. The critical bug is the missing handling of the double-underscore empty-country form. After applying the change above, the test result = LocaleUtils.to_locale('fr__POSIX') will return Locale('fr', '', 'POSIX') as expected.
