What happens
- The failure comes from the conditional
  if x == int(x) and not math.isinf(x) and not math.isnan(x):
  (the line near the start of add_number).
- Python evaluates that condition left-to-right. That means int(x) is called before the math.isinf/math.isnan checks.
- If x is +inf, -inf or nan, int(x) raises (OverflowError for infinities, ValueError for NaN), so the function crashes before it ever checks isinf/isnan.

Concrete example that triggers the crash
- Call add_number(float('inf')) or add_number(float('nan')).
- int(float('inf')) raises OverflowError: cannot convert float infinity to integer (or ValueError for nan).

Why this is the root cause (no guesswork)
- The code intends to avoid converting non-finite floats, but the order of the boolean terms causes the conversion to happen first.
- The offending expression is the left-most test x == int(x), which forces int(x) for all inputs.

Exact fixes (pick one)
1) Reorder and use math.isfinite (recommended):
   Replace the problematic line with:
   if math.isfinite(x) and x.is_integer():
   - math.isfinite(x) returns False for inf and NaN so int() is never called.
   - x.is_integer() tests whether the float has an integer value without converting to int.

2) Alternatively, reorder the existing checks so finite checks run before int():
   if not math.isinf(x) and not math.isnan(x) and x == int(x):
   - Because Python evaluates left-to-right, int(x) won't be called for inf/nan.

3) If you prefer to keep int conversion, guard it explicitly:
   if not math.isinf(x) and not math.isnan(x) and x == int(x):
   (same as 2), or:
   if math.isfinite(x) and x == int(x):

Recommended patch (single-line)
- Replace
    if x == int(x) and not math.isinf(x) and not math.isnan(x):
  with
    if math.isfinite(x) and x.is_integer():

Why I recommend that patch
- It's clear, safe (never calls int on inf/nan), and uses the stdlib float.is_integer which directly expresses the intent: "x is a finite integer-valued float".

Where it occurs
- The change is in add_number (the function you posted). The crash happens before add() is called (so the add() code at lines 126–138 is unrelated to this error except as the downstream method used after the integer test).
