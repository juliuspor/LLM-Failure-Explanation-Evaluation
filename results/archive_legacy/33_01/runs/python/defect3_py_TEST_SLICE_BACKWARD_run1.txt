What happened (what failed)
- The test calls translate_string with the character U+1F630 (😰), a code point above 0xFFFF that must be represented as a UTF‑16 surrogate pair (two 16‑bit code units).
- Inside to_utf16_code_units (starts at line 10; code_units is created on line 12) the code builds a list of UTF‑16 code units.
- Later code indexes into that list and checks bounds with:
  if index < 0 or index >= len(code_units):
      raise IndexError(f"String index out of range: {index}")
  (lines 35–36)
- An IndexError is raised when translate_string tries to read the second code unit of the surrogate pair, because the expected second code unit is not present in code_units. In short: to_utf16_code_units produced the wrong number of code units for the high code point, so accessing index+1 goes past len(code_units).

Why it happened (root cause)
- A character U+1F630 requires two UTF‑16 code units (a high surrogate and a low surrogate). to_utf16_code_units failed to append both 16‑bit units for characters > 0xFFFF. As a result code_units has only one entry for that character where it should have two.
- Later code that assumes surrogate pairs occupy two code units and tries to access code_units[i+1] finds i+1 == len(code_units) and triggers the IndexError at the bounds check on lines 35–36.

Where exactly (file/lines/variables)
- Function: to_utf16_code_units
- Line where list is created: 12 -> code_units = []
- Lines that raise the error: 35–36 -> the bounds check/IndexError
- Variables: code_units (the UTF‑16 unit list), index (the index being accessed), and the input string s (containing '\U0001F630').

How to fix it (concrete steps)
1. In to_utf16_code_units, convert any code point > 0xFFFF into a correct UTF‑16 surrogate pair and append both 16‑bit values to code_units. Use this exact calculation:
   - cp = ord(char)
   - if cp <= 0xFFFF: append cp
   - else:
       u = cp - 0x10000
       high = 0xD800 + (u >> 10)
       low  = 0xDC00 + (u & 0x3FF)
       append high, then append low
2. After making that change, code_units will have the correct length and any code that accesses code_units[i+1] will no longer go out of range for surrogate pairs.
3. Optionally add unit tests for characters above 0xFFFF (e.g. '\U0001F630', '\U0001F600') to verify to_utf16_code_units returns two code units for each of those characters.

Why these steps fix it
- They ensure the representation in code_units matches UTF‑16: characters > 0xFFFF become two entries. That prevents any subsequent access of index+1 from exceeding len(code_units), so the IndexError at lines 35–36 no longer occurs.
