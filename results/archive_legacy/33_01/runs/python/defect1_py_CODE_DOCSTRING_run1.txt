What happens
- The method for_offset_hours_minutes rejects any negative minutes because of this validation:
  if minutes_offset < 0 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")

  That check enforces minutes_offset ∈ [0, 59]. The docstring (and intended behavior) allow minutes to be negative when hours is zero or negative (for example (0, -15) → -00:15 and (-2, -15) → -02:15). Because of the validation above, calls like for_offset_hours_minutes(0, -15) or for_offset_hours_minutes(-2, -15) throw ValueError("Minutes out of range: -15") instead of producing the expected negative-offset zone.

Why it fails (root cause)
- The minutes validation range is wrong. The code treats minutes as only non‑negative, but the docstring and examples show minutes should be allowed in [-59, +59], with additional sign rules tying hours and minutes together.
- Because of that incorrect check the method fails early and never gets to the arithmetic that combines hours and minutes.

Where in the code it occurs
- The problem is in this line/statement inside for_offset_hours_minutes:
  if minutes_offset < 0 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")

What to change (concrete steps)
1. Allow minutes to be negative by changing the range check to [-59, +59]:
   Replace
     if minutes_offset < 0 or minutes_offset > 59:
         raise ValueError(f"Minutes out of range: {minutes_offset}")
   with
     if minutes_offset < -59 or minutes_offset > 59:
         raise ValueError(f"Minutes out of range: {minutes_offset}")

2. Enforce the disallowed combination (positive hours with negative minutes) explicitly, because the docstring says positive hours with negative minutes should raise:
   Add after the hours range check:
     if hours_offset > 0 and minutes_offset < 0:
         raise ValueError("Positive hours with negative minutes is not allowed")

3. Keep the subsequent arithmetic that combines hours and minutes. The current combination logic expects minutes_offset to be signed, so allowing negative minutes is necessary for it to work correctly:
   hours_in_minutes = hours_offset * 60
   if hours_in_minutes < 0:
       minutes_offset = hours_in_minutes - minutes_offset
   else:
       minutes_offset = hours_in_minutes + minutes_offset

Example corrected checks (exact replacement)
- Replace the two validation blocks:
    if hours_offset < -23 or hours_offset > 23:
        raise ValueError(f"Hours out of range: {hours_offset}")
    if minutes_offset < 0 or minutes_offset > 59:
        raise ValueError(f"Minutes out of range: {minutes_offset}")

  with:
    if hours_offset < -23 or hours_offset > 23:
        raise ValueError(f"Hours out of range: {hours_offset}")
    if minutes_offset < -59 or minutes_offset > 59:
        raise ValueError(f"Minutes out of range: {minutes_offset}")
    if hours_offset > 0 and minutes_offset < 0:
        raise ValueError("Minutes must not be negative when hours is positive")

Why this fixes it
- Allowing minutes in [-59, +59] accepts the negative-minute cases the docstring explicitly lists.
- Adding the explicit rule prevents invalid combinations (positive hours with negative minutes), matching the documented behavior that those combinations should produce ValueError.
- The rest of the method already expects signed minutes for correct calculation, so no other changes are required.
