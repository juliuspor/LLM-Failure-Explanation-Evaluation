What happened
- The call DateTimeZone.for_offset_hours_minutes(-2, -15) passed minutes_offset = -15.
- At lines 396–397 the code checks:
  if minutes_offset < 0 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")
  Because minutes_offset is -15, that condition is true and a ValueError is raised.

Why it failed (root cause)
- for_offset_hours_minutes expects the minutes argument to be in the range 0..59, not a negative value. The sign of the offset should be carried by the hours parameter (or by combining hours and minutes before calling).
- Passing a negative minutes value (-15) directly triggers the guard at lines 396–397, so the function rejects the input rather than producing the '-02:15' zone id.

Exactly where it happened
- Method: DateTimeZone.for_offset_hours_minutes
- Variables: minutes_offset (value -15)
- Lines: 396–397 (the range check and ValueError)

How to fix it (concrete steps)
Option A — change the call (recommended)
1. Pass non-negative minutes and keep the hour negative:
   DateTimeZone.for_offset_hours_minutes(-2, 15)
   This will produce the expected zone id '-02:15'.

Option B — change the function to accept negative minutes (if you control the implementation)
1. Remove or change the strict negative check at lines 396–397.
2. Compute a single signed total minutes first, for example:
   total_minutes = hours_offset * 60 + (1 if hours_offset >= 0 else -1) * abs(minutes_offset)
   Validate that abs(minutes_offset) is in 0..59, not that minutes_offset itself is non-negative.
3. Use total_minutes to build the zone id.
4. Add unit tests for mixed-sign inputs like (-2, 15), (-2, -15), (2, -15) to define the intended behavior.

Which to choose
