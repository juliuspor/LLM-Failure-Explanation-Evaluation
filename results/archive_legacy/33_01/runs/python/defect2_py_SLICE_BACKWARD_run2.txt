What happened (where it fails)
- The failure happens inside get_paint (line 54). The computation on line 71 divides by (self._upper_bound - self._lower_bound).
- If self._upper_bound == self._lower_bound the expression on line 71 triggers a ZeroDivisionError (division by zero). That is the root cause — the code assumes the bounds are different but does not check them.

Why it fails (root cause)
- The code computes g like this:
    g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
  If the denominator (self._upper_bound - self._lower_bound) is zero you get a ZeroDivisionError at line 71. The subsequent range check and ValueError on line 75 never run because the exception happens earlier.

How to reproduce exactly
1. Set self._lower_bound and self._upper_bound to the same number (for example both 0.0).
2. Call get_paint(value) with any value. You will get a ZeroDivisionError raised at line 71.

Precise, actionable fixes (pick one)
1) Validate bounds and raise a clear error early
   - At start of get_paint (before line 71), add:
       if self._upper_bound == self._lower_bound:
           raise ValueError(f"get_paint: upper_bound ({self._upper_bound}) equals lower_bound ({self._lower_bound}); division by zero")
   - This documents the precondition and stops the ZeroDivisionError.

2) Handle equal bounds as a special case
   - If you want to allow equal bounds, decide what g should be (for example 0 or 255) and implement:
       if self._upper_bound == self._lower_bound:
           g = 0  # or 127 or 255, depending on intended behavior
       else:
           g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

3) Clamp instead of failing after the division
   - Keep the normal formula but after computing g, clamp into [0,255]:
       g = max(0, min(255, int(...)))
   - Note: this does not avoid the ZeroDivisionError if bounds are equal; combine with (1) or (2) if needed.

Also improve the error message on line 75
- The ValueError currently says "Color parameter outside of expected range: Red Green Blue" but provides no numeric context. If you keep that check, change it to include the offending values:
    raise ValueError(f"get_paint: computed g={g} outside 0..255 for input value={value}, bounds=({self._lower_bound},{self._upper_bound})")

Summary
- Root cause: division by zero at line 71 when self._upper_bound == self._lower_bound.
- Fix by validating bounds early (preferred), or handling equal-bounds as a special case, and improve the ValueError message at line 75 to include actual numbers.
