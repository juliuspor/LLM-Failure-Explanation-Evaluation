What happens
- The if on line 3 of your snippet:
  if x == int(x) and not math.isinf(x) and not math.isnan(x):
  forces Python to call int(x) before it evaluates the math.isinf/math.isnan checks.
- If x is math.nan or math.inf (or -math.inf), calling int(x) raises an exception:
  - int(math.nan) raises ValueError: cannot convert float NaN to integer
  - int(math.inf) raises OverflowError: cannot convert float infinity to integer
- Because int(x) is evaluated first, the code raises one of those exceptions instead of safely skipping the integer branch. That is why the code "fails" for NaN/Infinity floats.

Where it occurs
- The problematic expression is at the start of add_number:
  if x == int(x) and not math.isinf(x) and not math.isnan(x):
- The int(x) call is evaluated before the math.isinf/math.isnan checks, so the exception is raised there.

How to fix it (specific, ordered steps)
1. Test for NaN/Infinity before calling int(x). For example:
   if not math.isinf(x) and not math.isnan(x) and x == int(x):
   This guarantees int(x) is only called for finite values.
2. Better: use float.is_integer() to test integer-valued floats without converting:
   if math.isfinite(x) and x.is_integer():
       value = int(x)
   (math.isfinite(x) is equivalent to not isinf and not isnan)
3. If you want minimal change, reorder the checks:
   if not math.isinf(x) and not math.isnan(x) and x == int(x):
       value = int(x)
   This preserves the rest of your logic but prevents int() from being called on NaN/Inf.

Why these fixes work
- Python evaluates operands left-to-right. By testing finiteness first, you avoid calling int() on NaN/Infinity.
- float.is_integer() is a direct, safe check for whether a finite float is an exact integer value, and it doesn't call int() until you explicitly convert.
