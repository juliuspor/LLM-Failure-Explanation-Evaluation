What happened (where it fails)
- The code raises ValueError at line 397: raise ValueError(f"Minutes out of range: {minutes_offset}").
- That check at lines 396–397 runs before the later arithmetic at lines 402 and 404 that combine hours and minutes into a single minutes_offset value.
- Because minutes_offset at the time of the check is not the raw minutes component but a value that will later be adjusted with hours (or already contains hours*60), it can be > 59 even when the input minutes were valid. The pre-check therefore rejects valid inputs.

Why it fails (root cause)
- The variable minutes_offset is being validated as if it were the minutes component (0..59) at line 396, but minutes_offset is later changed at lines 402 and 404 to include hours (hours_in_minutes ± minutes_offset). That means the check at 396–397 is using the wrong value and sometimes triggers incorrectly.
- Concretely: for an offset like "+02:30" the final minutes_offset will be 150 (2*60 + 30). If the code checks minutes_offset too early, it will see 150 (>59) and throw the ValueError, even though the original minutes component 30 is valid.

Exactly where to change it (specific lines and fixes)
Option A — Validate the raw minutes component:
- If you have a separate variable for the parsed minutes (for example parsed_minutes), change the validation to check parsed_minutes, not minutes_offset:
  - Replace lines 396–397
    - from: if minutes_offset < 0 or minutes_offset > 59: raise ValueError(f"Minutes out of range: {minutes_offset}")
    - to:   if parsed_minutes < 0 or parsed_minutes > 59: raise ValueError(f"Minutes out of range: {parsed_minutes}")
- Keep the later combination at lines 402 and 404 as-is.

Option B — Reorder the validation to run after combining hours and minutes:
- Move the minutes range check so it runs on the minutes component before you overwrite or reuse minutes_offset for the combined value. For example:
  - Parse hours and minutes into hours_in_minutes and parsed_minutes
  - Validate parsed_minutes with the 0..59 check
  - Then compute minutes_offset = hours_in_minutes ± parsed_minutes (lines 402/404)
- Do not validate minutes_offset after it has been turned into total minutes.

Why these fixes work
- They ensure the code validates the actual minutes component (0..59) rather than a value that may already include hours. That prevents false positives like rejecting "+02:30" because the intermediate variable held 150.

Summary
The code validates minutes_offset too early (when it may already hold hours*60 + minutes), causing valid minute values to be rejected. Fix by validating the raw parsed minutes value (or performing the check before you overwrite minutes_offset with the combined value).
