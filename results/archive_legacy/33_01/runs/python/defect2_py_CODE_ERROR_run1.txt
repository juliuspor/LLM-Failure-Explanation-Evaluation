What happened (where it failed)
- The code clamps the input value into variable v on lines:
  v = max(value, self._lower_bound)
  v = min(v, self._upper_bound)
- But the code then computes g using the original value instead of the clamped v:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- Because value was not clamped, g can be <0 or >255 when value is outside the bounds. That triggers the ValueError at the following check:
  if g < 0 or g > 255:
      raise ValueError(...)

Root cause
- A simple variable mix-up: you intended to use the clamped v in the normalization, but you used value. So clamping does nothing and g can fall outside the expected 0–255 range.

Exact places to change
1. Use v (the clamped value) when computing g:
   g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
2. Protect against division by zero if lower and upper bounds are equal:
   denom = (self._upper_bound - self._lower_bound)
   if denom == 0:
       g = 0  # or 255, or whatever the intended behavior is
   else:
       g = int((v - self._lower_bound) / denom * 255.0)
3. (Optional) Improve the error message or remove the runtime check by clamping g to [0,255] after calculation:
   g = max(0, min(g, 255))

Suggested corrected code
    def get_paint(self, value: float) -> Tuple[int, int, int]:
        # Clamp value to bounds
        v = max(value, self._lower_bound)
        v = min(v, self._upper_bound)

        denom = self._upper_bound - self._lower_bound
        if denom == 0:
            g = 0  # choose appropriate behavior when bounds are equal
        else:
            g = int((v - self._lower_bound) / denom * 255.0)

        # Ensure g is in range
        g = max(0, min(g, 255))

        return (g, g, g)

Why this fixes it
- Using v ensures the normalized value is within bounds.
- Checking denom avoids division-by-zero.
- Clamping g guarantees the ValueError no longer fires for small numeric errors or edge cases.
