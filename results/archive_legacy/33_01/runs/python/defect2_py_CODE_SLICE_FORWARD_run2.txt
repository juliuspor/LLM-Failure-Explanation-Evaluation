What happened (where)
- In get_paint(), you clamp the input value to v on lines:
  v = max(value, self._lower_bound)
  v = min(v, self._upper_bound)
- But on the next line you compute g using the original value instead of the clamped v:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- Because of that, when value is outside the [self._lower_bound, self._upper_bound] range g can be < 0 or > 255 and trigger your ValueError even though you intended to use the clamped value.

Why it fails (root cause)
- The root cause is a simple variable mix-up: you clamp into v but then use value for the calculation. The intention is to map the clamped value into 0–255, but the code maps the original (possibly out-of-range) value instead. That produces out-of-range g and raises the exception.
- There is also a secondary risk: if self._upper_bound == self._lower_bound the expression (self._upper_bound - self._lower_bound) is zero and you will get a ZeroDivisionError.

Exact location to change
- Change the computation to use v (the clamped value):
  g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- Add a guard for a zero span before the division, e.g. raise or handle the degenerate case:
  span = self._upper_bound - self._lower_bound
  if span == 0.0:
      raise ValueError("upper_bound equals lower_bound; cannot scale value")
  g = int((v - self._lower_bound) / span * 255.0)

Other small fixes
- The ValueError message currently says "Color parameter outside of expected range: Red Green Blue" — consider including the actual g value or bounds so the error is actionable:
  raise ValueError(f"Color parameter outside of expected range: g={g} (expected 0..255)")

Minimal corrected snippet
  v = max(value, self._lower_bound)
  v = min(v, self._upper_bound)
  span = self._upper_bound - self._lower_bound
  if span == 0.0:
      raise ValueError("upper_bound equals lower_bound; cannot scale value")
  g = int((v - self._lower_bound) / span * 255.0)
  if g < 0 or g > 255:
      raise ValueError(f"Color parameter outside of expected range: g={g} (expected 0..255)")
  return (g, g, g)

That fixes the bug (use clamped v), prevents division by zero, and gives a clearer error message.
