What happened (what failed)
- The code raises IndexError at lines 35–36 because len(code_units) is 0 when the bounds check runs.
- That happens because the variable code_units on line 12 is the empty list created inside to_utf16_code_units, and the code that does the index check (lines 35–36) runs before or without code_units ever being populated with UTF‑16 code units.

Why it fails (root cause)
- to_utf16_code_units(s: str) creates code_units = [] on line 12, but nothing in the shown slices guarantees that code_units has been filled before you use it at line 35. The index check uses len(code_units) (line 36) expecting a populated array, but code_units is still empty, so any index >= 0 fails the check and raises IndexError.
- This is either a scoping / ordering bug (the caller never calls to_utf16_code_units and instead tries to operate on the unpopulated local list), or a placement/flow bug (the code that appends to code_units is after the check or inside a different function so it never ran before the check).

   - Call code_units = to_utf16_code_units(s) in the function that does the index check / surrogate lookup, before line 35.

- They ensure len(code_units) reflects the actual number of UTF‑16 code units before you index into code_units. That removes the erroneous IndexError and allows the surrogate-pair logic at lines 38–44 to run correctly.

Summary
The IndexError is not a Unicode math bug — it's a logic/ordering bug: code_units is empty when you check and index it. Populate and/or pass the list properly (or move the check after population) so lines 35–44 operate on a filled list.
