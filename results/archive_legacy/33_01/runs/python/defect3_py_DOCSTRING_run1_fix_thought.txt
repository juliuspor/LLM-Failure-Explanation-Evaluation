The bug arises when a translator returns an unexpected value for 'consumed' (CsvEscaper returns 2 for a surrogate pair). To be robust, adjust translate to handle three cases: consumed==0 (no translation), consumed==1 (advance by the UTF-16 code unit count of the codepoint at pos), and consumed>1 (treat consumed as number of UTF-16 code units and advance pos by consumed). Keep existing guards for out and input_seq and conversion to UTF-16. This avoids skipping or double-processing code units when translators return either codepoint counts or code unit counts.
