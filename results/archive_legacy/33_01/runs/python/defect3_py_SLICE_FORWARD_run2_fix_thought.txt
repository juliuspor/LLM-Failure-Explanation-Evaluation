The bug arises from mismatch about what translate_codepoint returns. Translators (like CsvEscaper) return the number of UTF-16 code units they consumed (e.g., 2 for a surrogate pair). The existing translate() treated the returned value as a count of codepoints and advanced pos by recomputing charCount for each consumed unit, which can mis-handle consumption and lead to reprocessing the same units. The minimal, correct fix is to treat consumed as code unit count and advance pos by consumed. When consumed is 0, fall back to reading the codepoint at pos and append its UTF-16 code units. Implement translate accordingly, with proper checks for None and out parameter. Return type remains None.
