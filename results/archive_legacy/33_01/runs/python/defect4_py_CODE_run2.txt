What happened
- In the "max middle" section the code calls
  self.get_data_item(self._min_middle_index).get_period()
  instead of using self._max_middle_index. Concretely, this line is here (the start of the max-middle block):
  max_middle_period = self.get_data_item(self._min_middle_index).get_period()
- Because of that, the code computes max_middle from the period at _min_middle_index, not the period at _max_middle_index. The comparison
  if middle > max_middle:
  therefore compares the new item's middle to the wrong existing middle value.

Why it fails (root cause)
- The bug is a simple wrong variable use: _min_middle_index is used where _max_middle_index should be used. That causes the "max middle" tracking to use incorrect existing data, so _max_middle_index will never be correctly updated (or will be updated based on the wrong baseline), producing wrong results for anything that relies on the stored max-middle index.
- This error is deterministic and always occurs whenever _max_middle_index >= 0, because the code will fetch the period at the min index instead of the max index.

Where it occurs (exact location)
- In the block that starts with:
  if self._max_middle_index >= 0:
  the first line inside that block incorrectly reads:
  max_middle_period = self.get_data_item(self._min_middle_index).get_period()
  It should read:
  max_middle_period = self.get_data_item(self._max_middle_index).get_period()

How to fix (concrete steps)
1. Replace the wrong index:
   - Change
     max_middle_period = self.get_data_item(self._min_middle_index).get_period()
     to
     max_middle_period = self.get_data_item(self._max_middle_index).get_period()
2. Add a unit test that exercises updating min/max middle:
   - Create at least three periods with known start/end so their middles are ordered.
   - Call _update_bounds for each and assert that _min_middle_index and _max_middle_index point to the correct indices after each update.
3. (Optional but recommended) Reduce chances of similar copy-paste bugs:
   - Extract the repeated pattern (fetch period, convert to millis, compute start/middle/end) into small helper methods like _period_start_ms(idx), _period_middle_ms(idx), _period_end_ms(idx). Then the update code becomes clearer and less error-prone.
4. (Optional) Add an assertion or logging inside the max-middle block to catch misuse during development:
   - e.g., assert 0 <= self._max_middle_index < len(self._data), "invalid max_middle_index"

Why these steps fix it
- Step 1 corrects the immediate wrong variable usage so max comparisons use the right existing value.
- Step 2 prevents regression by verifying the intended behavior.
- Step 3 reduces copy-paste mistakes going forward.

That single wrong variable is the root cause. Fixing that line will restore correct tracking of the maximum middle index.
