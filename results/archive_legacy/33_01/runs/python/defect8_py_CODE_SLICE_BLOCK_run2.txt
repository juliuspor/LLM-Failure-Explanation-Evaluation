What fails
- The code rejects valid locale strings when the country code is lowercase.  
- The failing check is at lines 239–240 (inside to_locale): you fetch ch3 and ch4 and then require them to be between 'A' and 'Z':

  if ch3 < 'A' or ch3 > 'Z' or ch4 < 'A' or ch4 > 'Z':
      raise ValueError(f"Invalid locale format: {locale_str}")

Why that is the root cause
- Common locale forms use lowercase language and uppercase country (e.g. "en_US"), but many real inputs are lowercase for the country too (e.g. "en_us"). Your code forces the two country characters to be uppercase only, so "en_us" will hit that check and raise a ValueError even though "en_us" is a valid locale representation.
- The failure occurs specifically in to_locale when length >= 5, after you checked locale_str[2] == '_' and before you create the Locale. Variables involved: locale_str, ch3, ch4 (lines 239–240).

How to fix it (clear steps)
1. Decide which forms you want to accept. Typical options:
   - Accept either case (lower/upper) and normalize (recommended).
   - Require uppercase and document that requirement.

2. If you want to accept either case and normalize the country to uppercase, replace the uppercase-only check and the final slice usage with case-insensitive validation and normalization. Example changes:
   - Replace the ch3/ch4 bounds check with an isalpha() check for exactly two letters:
       if not (ch3.isalpha() and ch4.isalpha()):
           raise ValueError(...)
   - When constructing the Locale, normalize the country to uppercase:
       country = locale_str[3:5].upper()
       return Locale(locale_str[0:2], country)   # or include variant as needed
   - Optionally normalize the language to lowercase:
       language = locale_str[0:2].lower()

3. Minimal code patch (where lines 239–240 are):
   - Remove the 'A'..'Z' range test and use isalpha(); normalize before returning.

What this change fixes
- The function will accept "en_us", "en_US", "EN_us" etc., normalize them (e.g. language to lowercase and country to uppercase) and no longer raise the ValueError at lines 239–240 for otherwise valid locales.
