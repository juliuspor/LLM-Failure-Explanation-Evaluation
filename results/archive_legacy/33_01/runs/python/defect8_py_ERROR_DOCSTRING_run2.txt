What happened
- The exception comes from LocaleUtils.to_locale(locale_str).
- You passed "fr__POSIX" as locale_str.
- The method splits locale_str into tokens and assigns them to language, country, variant.
  - For "fr__POSIX" the split result is ["fr", "", "POSIX"] so:
    - language = "fr"
    - country = "" (empty string)
    - variant = "POSIX"
- The code then runs the country validation (something like: if country and (len(country) != 2 or not country.isupper()): raise ValueError(...)).
- Because country is the empty string, the code treats it as an invalid country and raises ValueError("Invalid locale format: fr__POSIX").

Why this is surprising
- The Java Locale convention accepts a locale with an empty country and a variant (language + empty country + variant) — "fr__POSIX" is a legitimate Java-style locale string.
- LocaleUtils.to_locale implements stricter validation and does not accept an empty country token, so it rejects that valid Java form.

Where it occurs (variables and method names)
- Method: LocaleUtils.to_locale(locale_str)
- Input variable: locale_str == "fr__POSIX"
- Local variables after split: language == "fr", country == "", variant == "POSIX"
- Failing check: the country validation branch that enforces 2-letter uppercase country codes (it rejects the empty country string)

How to fix it (concrete steps)
Option A — change the input (quick, no code change)
1. If you intend language + variant with no country, supply a form that this implementation accepts. For example "fr_POSIX" (one underscore) will be parsed as language="fr", country="POSIX" by naive split but that is not a valid country either — so only use this if your library expects that form. Safer: avoid variant-only strings with empty country; instead construct the Locale object directly in code.

Option B — change the validation to allow empty country (recommended)
1. In LocaleUtils.to_locale, after splitting locale_str into language, country, variant, add a branch that treats an empty country as valid when a variant exists:
   - If len(parts) == 3 and parts[1] == "":
       - set country = "" (keep empty)
       - set variant = parts[2]
       - skip the strict country-length-and-case check for this case (allow empty country)
2. Keep existing checks for non-empty country values (verify len == 2 and isupper()).
3. Add a unit test: to_locale("fr__POSIX") should return Locale("fr", "", "POSIX").

Option C — normalize input before validation
1. If you want to canonicalize "fr__POSIX" to something your validator already accepts, transform the token list into language + variant form handled by the code (but this risks losing standard form and is brittle).
