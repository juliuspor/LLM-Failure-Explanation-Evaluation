What happened (where and what failed)
- The test calls cc.add_number(-0.0). That path ends up calling add(self, newcode) (the function shown).
- At line 132 you do c = newcode[0]. That assumes indexing newcode returns a one-character string.
- In Python 3, if newcode is a bytes object (or some non-str sequence), newcode[0] is an int (not a one-character str). Using that int with your is_word_char and the logical expression at line 133 produces the wrong branch decision so a space is inserted (line 136) before the number. The produced result is " -0.0" not "-0.0", so the assert fails.

Why this root cause explains the failure
- The code later appends newcode (line 138) unchanged. If newcode was bytes or otherwise not the expected str, the logic at lines 132–136 miscomputes whether to insert a separating space because c is not a character but an int.
- The symptom (leading space before "-0.0") matches this exact problem: the code thinks the first character requires separation and inserts a space, so result != "-0.0".

Concrete fixes (exact changes you can make)
1) Normalize newcode to a str at the start of add:
   - Insert near line 126/129:
     newcode = str(newcode)
   - Pros: simple and ensures newcode[0] is a str.
   - Cons: if callers expect bytes to be preserved, that will convert them.

2) Safely get the first character without depending on element type:
   - Replace line 132 with:
     c = newcode[:1]
     # then use c in is_word_char (make sure is_word_char accepts single-char strings)
   - Pros: newcode[:1] will be a one-character string if newcode is str, and the same type slice for many sequence types; avoids int-from-bytes issue.
   - If you need always a str, use c = str(newcode)[:1].

3) Explicitly handle bytes vs str:
   - Add at top of add:
     if isinstance(newcode, bytes):
         newcode = newcode.decode('utf-8')    # or appropriate encoding
   - Then proceed normally.

Pick one of these based on whether add should accept bytes. The simplest and least invasive is to convert newcode to str (option 1) or to use newcode[:1] for c (option 2).
