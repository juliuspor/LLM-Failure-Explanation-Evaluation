What happened (where it failed)
- The exception was raised by this line in for_offset_hours_minutes:
  if minutes_offset < 0 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")
- The ValueError message shows minutes_offset = -15, so the function rejected a negative minutes value before any later math ran.

Why it failed (root cause)
- The method expects minutes_offset to be in the range 0..59 (non‑negative) and treats the sign of the offset via hours_offset. You passed minutes_offset = -15, which is outside the validated range, so the method correctly raised "Minutes out of range: -15".
- In short: the caller supplied a negative minutes value, but the function forbids negative minutes.

Why you might have been confused
- Many callers think of a combined signed offset (e.g. "-1 hour 15 minutes" as hours=-1, minutes=-15). This API instead expects hours to carry the sign and minutes to be a non‑negative 0..59 component that is added or subtracted depending on hours' sign. That mismatch is the source of the error.

- If you intended -1 hour 15 minutes, call:

  not for_offset_hours_minutes(-1, -15)

2) Change the method to accept signed minutes (if you want minutes to be allowed negative)

  - validate absolute range: if minutes_offset < -59 or minutes_offset > 59: raise ValueError(...)
  - compute total minutes as hours_offset*60 + minutes_offset
  - multiply by MILLIS_PER_MINUTE and continue

- Update docstring/signature to state "minutes_offset must be 0..59 and hours_offset carries the sign" so callers won't pass negative minutes.

Summary
- The method rejects negative minutes (minutes_offset < 0). The exception happened because you passed -15 for minutes_offset. Either pass minutes as a non‑negative value (and use the sign on hours) or change the method to accept signed minutes and compute total_minutes = hours*60 + minutes.
