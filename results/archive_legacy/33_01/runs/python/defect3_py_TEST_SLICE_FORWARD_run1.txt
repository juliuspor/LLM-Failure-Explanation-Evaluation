Why that assumption is wrong here (root cause)
- In Python 3 a str stores Unicode code points, not UTF‑16 code units. When you iterate or index a Python str you get complete characters (code points). For the input '\U0001F630' a single index yields the integer ord(character) == 0x1F630, not a 16‑bit high surrogate in the 0xD800..0xDBFF range.
- Because high becomes 0x1F630, the surrogate check at line 39 fails, so the code takes the "no surrogate" path (line 44). The function is written for a different internal representation (UTF‑16 code units), so the logic that expects surrogate pairs never runs for a Python str and later processing (outside the slice you showed) ends up producing the wrong output.
