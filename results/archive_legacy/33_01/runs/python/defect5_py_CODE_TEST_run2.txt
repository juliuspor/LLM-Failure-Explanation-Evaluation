What happens
- With the call ArrayUtils.add(None, None, expected_type=str) both array and element are None, so the code sets inferred_type = object (the else branch).
- Later the code reaches the check:
  if expected_type is not None:
      if inferred_type == object and expected_type != object:
          raise TypeError(...)

  Because inferred_type == object and expected_type == str, the function raises that TypeError and fails instead of returning [None].

Why that check is wrong (root cause)
- The code treats "inferred_type == object" as a sign it cannot safely return a list of the requested expected_type. But when both array and element are None, there is no real element type to conflict with expected_type. A list containing a single None value is compatible with any element type; there is nothing to cast.
- So the function mistakes "unknown element type" (inferred_type == object) for "incompatible element type", and raises when it should accept expected_type and produce the list.

Where exactly this happens in your code
- inferred_type is set by the branch:
    if array is not None:
        inferred_type = type(array)
    elif element is not None:
        inferred_type = type(element)
    else:
        inferred_type = object
- The incorrect error is raised in the later block that begins:
    if expected_type is not None:
        if inferred_type == object and expected_type != object:
            raise TypeError(...)

How to fix it (concrete steps & code)
1. If both array and element are None, use expected_type (when provided) as the inferred element type instead of object. That avoids the wrongful TypeError.
2. Keep the existing incompatibility check for true conflicts (for example, when you have a concrete inferred_type that cannot be cast to expected_type).

Minimum change (replace the inference block):

    # current block:
    if array is not None:
        inferred_type = type(array)
    elif element is not None:
        inferred_type = type(element)
    else:
        inferred_type = object

Replace with:

    if array is not None:
        inferred_type = type(array)
    elif element is not None:
        inferred_type = type(element)
    elif expected_type is not None:
        # no array and no element, but caller specified expected_type:
        inferred_type = expected_type
    else:
        inferred_type = object

Or alternatively, change the post-check to accept unknown types when element is None:

    if expected_type is not None:
        # only raise if we have a concrete inferred type that conflicts
        if inferred_type is not object and inferred_type != expected_type:
            raise TypeError(...)

Either change fixes your failing test. The first option is preferable because it makes inferred_type reflect the caller's intent (expected_type) when there is no other type information.
