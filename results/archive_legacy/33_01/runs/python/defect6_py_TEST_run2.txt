What happened
- The failing test calls cc.add_number(-0.0) on your MockCodeConsumer and asserts the output is '-0.0'.
- Your add_number implementation is treating -0.0 as non-negative so it outputs '0.0' (or '0') instead of '-0.0'.

Why it happened (root cause)
- In IEEE 754 floats there is a signed zero: -0.0 is distinct from +0.0 at the bit level, but numerical comparisons treat them as equal.
- Most code that detects sign uses a test like if value < 0: or if value >= 0:. For -0.0, value < 0 is False, so that test says “not negative” even though the bit-sign is negative. That is why add_number(–0.0) loses the minus and produces '0.0'.
- Concretely: inside add_number you likely do something like:
  - if number < 0:
      sign = '-'
      number = -number
    else:
      sign = ''
    buf.append(sign + str(number))
  Because -0.0 < 0 is False, the code takes the else branch and prints a non-negative representation.

Where it occurred (places to inspect)
- The failing behavior happens inside MockCodeConsumer.add_number (the method you call).
- The variable checked is the input float (often named number, value, or n) inside add_number.
- The place to look is the conditional that decides whether to prepend '-' (the "if number < 0" style check) and the subsequent conversion to string.

Correct fixes (concrete steps)

  - repr(-0.0) yields '-0.0' in Python, so the minus is preserved.

3) Inspect the sign bit directly (low-level option)
- If you need exact control, inspect the float bits:

  - bits = struct.unpack('!Q', struct.pack('!d', number))[0]

  - Then format as you need.
  - Use this only if you need a bitwise approach.
