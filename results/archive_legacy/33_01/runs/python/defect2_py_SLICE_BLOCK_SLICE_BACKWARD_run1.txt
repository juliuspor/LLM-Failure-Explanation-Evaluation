What fails
- The failure happens inside get_paint at line 71:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- If self._upper_bound == self._lower_bound that division is division by zero and Python raises ZeroDivisionError before you ever reach the check at lines 74–75.

Why it fails (root cause)
- The code assumes _upper_bound and _lower_bound are different. When they are equal the denominator (_upper_bound - _lower_bound) is 0, so the expression raises ZeroDivisionError at line 71.
- The later test (lines 74–75) for g < 0 or g > 255 is unreachable in that case because the exception occurs earlier.
- Additional issue (not the immediate crash): the error message at line 75 uses a f-string but contains no variables:
    raise ValueError(f"Color parameter outside of expected range: Red Green Blue")
  That message does not show which value (g, value, _lower_bound, _upper_bound) caused the problem; it is unhelpful for debugging.

Where it occurs
- Method: get_paint
- File/lines shown: line 71 (division), then 74–75 (range check & ValueError). The ZeroDivisionError originates at line 71.

How to fix (concrete steps)
1. Prevent division by zero by handling the equal-bounds case before doing the division. Decide what get_paint should return when _upper_bound == _lower_bound (example options below).
   Example fix inside get_paint before line 71:
   - If bounds equal, return a fixed color or clamp:
     if self._upper_bound == self._lower_bound:
         # return middle color or clamp to 0 or 255 as appropriate
         return (127, 127, 127)   # or (0,0,0) or (255,255,255)
   - Or raise a specific error explaining the invalid bounds:
     if self._upper_bound == self._lower_bound:
         raise ValueError(f"_upper_bound ({self._upper_bound}) equals _lower_bound ({self._lower_bound}) in get_paint")

2. After computing g, clamp it to [0, 255] rather than only checking and raising. That avoids unexpected ValueError for small float rounding:
   g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
   g = max(0, min(255, g))

   If you prefer to raise an error when out of range, include the actual values in the message:
   if g < 0 or g > 255:
       raise ValueError(f"Computed g={g} from value={value} out of range for bounds ({self._lower_bound}, {self._upper_bound})")

3. Improve the error message on line 75 so it shows relevant variables (see above). That makes future debugging easy.

Summary of exact changes to make
- Add a guard for _upper_bound == _lower_bound before the division (either return a default color or raise a clear ValueError mentioning _upper_bound and _lower_bound).
- After computing g, clamp or raise an informative error that prints g, value, _lower_bound, and _upper_bound.
- Replace the current f-string text with one that includes variable values.

These fixes eliminate the ZeroDivisionError at line 71 and make the range error at lines 74–75 informative and actionable.
