What went wrong (concise summary)
- On line 337 you set inferred_type = type(array). That takes the Python type of the list object itself (e.g. list, ArrayList), not the type of the elements inside the list.
- _copy_list_grow1 expects a component/element type, not the container type. Passing the container type makes the copied list use the wrong component type at runtime.
- On line 346 you then assign new_list[len(new_list) - 1] = element. Because new_list was created with the wrong component type, that assignment fails (a runtime type error / ClassCast-like failure), or the function behaves incorrectly when expected_type checks are done later.
- The check on lines 346–347 only detects the special case inferred_type == object; it does not catch the real bug where inferred_type is a list type. So the code neither constructs the list correctly nor raises the informative error you intended.

Exactly where it happens
- Line 337: inferred_type = type(array) — wrong value. You want the element/component type, not the list's type.
- Line 340: the branch inferred_type = type(element) is fine when array is None.
- Line 346: new_list[...] = element — this assignment is where the incorrect inferred_type surfaces as a runtime failure.
- Lines 346–347: the later check only handles the object-vs-expected_type mismatch; it won't detect that inferred_type is e.g. list instead of the actual element type.

Why that causes the failure (root cause)
- Passing type(array) (the list's type) to ArrayUtils._copy_list_grow1 causes that helper to allocate a list/array whose element/component type is wrong. When you later write the element into that new structure, the runtime rejects the assignment (or you get a ClassCast-like failure), because the underlying structure expects a different element type than the element you provide.
