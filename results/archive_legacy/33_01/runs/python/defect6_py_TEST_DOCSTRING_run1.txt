What happened
- The test calls MockCodeConsumer.add_number(-0.0) and expects get_code() to return "-0.0".
- The test fails because add_number turned the -0.0 input into "0.0" (or "0") before storing it, so get_code() returned "0.0" instead of "-0.0".

Why it happened (root cause)
- In Python (and most languages) -0.0 compares equal to 0.0: (-0.0 == 0.0) is True. A branch that tests the numeric value for zero therefore cannot distinguish positive zero from negative zero.
- I suspect the code in MockCodeConsumer.add_number contains a zero-check like:
  if value == 0:
      token = "0.0"
  else:
      token = str(value)
  That branch is what loses the sign bit. The loss occurs exactly in MockCodeConsumer.add_number, at the conditional that treats zero specially (the line with "if value == 0:" or any equivalent check).
- Any code that canonicalizes zeros using equality or converts via arithmetic that collapses the sign (for example multiplying by 0, taking abs, or formatting that normalizes zero) will produce a positive zero string and drop the minus sign.

Where it occurred
- In MockCodeConsumer.add_number, at the branch that detects zero (the line that compares the number to 0 or otherwise special-cases zero). After that branch the stored token no longer contains the negative sign, so get_code() later returns "0.0" (or "0") instead of "-0.0".

- Removing the special-case avoids the equality-based collapse of sign altogether.
