The bug relates to incorrect advancing of the input position after a translator consumes characters. Translators currently return a count that may represent UTF-16 code units consumed (e.g., CsvEscaper returning 2 for a surrogate pair). The existing logic advances the position by recomputing char counts per codepoint which can mis-handle surrogate pairs. Simpler robust fix: when a translator returns a non-zero consumed value, advance pos by that consumed value (treating it as UTF-16 code units consumed). Keep original behavior for consumed==0. This avoids incorrect iteration over surrogate pairs and aligns with translator implementations that may indicate units consumed. Implement the translate method accordingly, with same inputs/outputs and input validation.
