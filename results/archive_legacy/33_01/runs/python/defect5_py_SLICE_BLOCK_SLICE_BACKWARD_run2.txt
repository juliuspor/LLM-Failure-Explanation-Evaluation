What happened (where it fails)
- At line 340 your code sets inferred_type = object (slice shows: "elif element is not None: inferred_type = object").
- Immediately after, at lines 346–351 you check:
  if inferred_type == object and expected_type != object:
      raise TypeError(...)

Because inferred_type was forced to object on line 340, that check is true whenever expected_type is a concrete type (not object). The function then raises the TypeError claiming "Cannot cast object list to X list" even when the list actually contains elements of a concrete type. The exception is raised at lines 346–351.

Why this happens (root cause)
- The code overwrites inferred_type with object regardless of the actual element value (line 340). The intent appears to be to record the element's runtime type, but instead you always assign object there. That makes the later test believe you have an Object[] (java-style) when you may have, for example, Foo[].
- In short: inferred_type is computed incorrectly (wrong value assigned), so the provenance check at lines 346–351 misfires.

Exactly where to change it
- Fix the assignment at line 340. Replace inferred_type = object with something that reflects the element's actual class, e.g.:

- If your intent was to detect mixed-type lists and fall back to object, implement logic that:
  - On first non-None element, set inferred_type = type(element).
  - On subsequent non-None elements, if type(element) != inferred_type, set inferred_type = object (or compute common supertype), then break or continue accordingly.
