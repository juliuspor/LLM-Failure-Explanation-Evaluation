The bug is that translate() manipulates UTF-16 code units and advances pos incorrectly by using char_count(code_point_at(code_units, pos)) in a loop over consumed translators. If a translator consumed n codepoints, we must advance pos by the sum of char_count() for each codepoint consumed, but the current code calls code_point_at with the same pos repeatedly while pos is being advanced in the loop, which is okay. However the real bug described is index-out-of-range in CsvEscaper.translate_codepoint when callers pass code_units converted from Python string; but the diagnosis says bounds checks when peeking ahead in input string. To fix, rewrite translate to iterate safely over code_units and when a translator returns consumed, advance by the correct number of UTF-16 code units without causing out-of-range lookups. Implement loop: while pos < length: for translators call translate_codepoint; if consumed>0: for i in range(consumed): cp = code_point_at(code_units, pos); pos += char_count(cp); break; else: cp = code_point_at(...); out.extend(to_chars(cp)); pos += char_count(cp)
