What happened
- The test calls cc.add_number(-0.0).
- That method (add_number) passed a numeric value into add, but add assumes its argument is a string.
- At add, the code at line 129 calls len(newcode) and line 132 does newcode[0]. Those operations require newcode to be a sequence (a string), not a float.
- A float is not subscriptable in Python, so newcode[0] raises TypeError and the test fails before get_code() can return "-0.0".

Where it occurred
- Failure occurs inside MockCodeConsumer.add, specifically:
  - line 129: if len(newcode) == 0:
  - line 132: c = newcode[0]
  These lines assume newcode is a string.

Why it happened (root cause)
- add_number passed a numeric (float) value to add instead of a string. add has no conversion or guard for non-string input, so indexing/len on a float raises TypeError. The code never reaches the string append path, so result != '-0.0'.

Concrete fixes (pick one)

1) Convert to string in add_number (recommended)
- Change the caller so add_number converts the number to a string before calling add.
- Example change inside add_number:
    self.add(str(value))
- Reason: keeps add's contract simple (it expects strings) and makes add_number responsible for formatting numbers.

2) Make add accept non-strings by coercing at start
- Modify add to coerce newcode to a string before using it:
  - Insert at top of add (just after maybe_end_statement()):
      newcode = str(newcode)
  - Then the rest of the method can use len/newcode[0] safely.
- Reason: more forgiving API; callers can pass non-strings.

3) Defensive check + clear error
- At the top of add, validate type and raise a clear error:
      if not isinstance(newcode, str):
          raise TypeError("add expects a string; got %s" % type(newcode).__name__)
- Reason: explicit failure mode makes the violation clear during development.

Steps to implement the recommended fix (1)
1. Open the implementation of add_number.
2. Replace the line that calls add(value) with:
     self.add(str(value))
3. Run tests again.

This will ensure add receives "-0.0" (a string), so lines 129 and 132 operate correctly and get_code() will return '-0.0'.
