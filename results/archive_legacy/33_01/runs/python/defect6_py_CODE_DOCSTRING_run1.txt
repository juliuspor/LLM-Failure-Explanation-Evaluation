What goes wrong
- Where: inside add_number(), in the while loop whose condition is
      while mantissa % 10 == 0:
  (the loop starts at the line after "if abs(x) >= 100:")
- What: the code fails for negative integers with trailing zeros (for example x = -1200.0). For those values the loop stops too early because mantissa % 10 does not behave as intended for negative mantissa values.
- Why (root cause): Python’s % and // for negative integers use floor semantics, so:
    - For mantissa = -120, mantissa % 10 == 8, not 0, even though -120 ends with a zero digit. So `mantissa % 10 == 0` is false and the loop exits.
    - Meanwhile the code also relies on // to compute new mantissa and then recompute check_val:
          check_val = (mantissa // 10) * (10 ** (exp + 1))
      That computation can still match value for negative numbers (because // on exact multiples will give the expected quotient), but the % check prevents entering the branch that would reduce mantissa further.
  The net effect: numbers like -1200.0 are not reduced to a compact mantissa+exponent form correctly (and the code may produce the wrong string or skip intended reductions).

Reproduce with a concrete example
- x = -1200.0
- value = int(x) -> -1200
- mantissa = -1200, exp = 0
- mantissa % 10 == 0 is True for -1200, so first iteration runs:
    check_val = (mantissa // 10) * (10 ** (exp + 1))
              = (-120) * 10 = -1200 == value → reduce mantissa -> -120, exp -> 1
- next loop: mantissa % 10 == (-120) % 10 == 8 → not equal to 0 → loop breaks
- But (-120 // 10) * (10 ** 2) = (-12) * 100 = -1200 equals value, so a further reduction was valid and should have happened. The % test prevented it.

Exact spot to change
- Change the while condition line:
      while mantissa % 10 == 0:
  to use the absolute value of mantissa, e.g.
      while abs(mantissa) % 10 == 0:
  This makes the trailing-zero test correct for negative numbers as well.

Additional robustness recommendations (concrete steps)
1. Replace the while condition:
     while mantissa % 10 == 0:
   with:
     while abs(mantissa) % 10 == 0:
2. Prefer computing the candidate new mantissa explicitly to avoid confusion with floor division on negatives:
     new_mantissa = mantissa // 10
     check_val = new_mantissa * (10 ** (exp + 1))
     if check_val == value:
         mantissa = new_mantissa
         exp += 1
     else:
         break
   This keeps the same logic but makes the intent clear and avoids relying on % behaviour for negatives.
3. (Optional) If you want identical behavior for non-integer floats that are huge, ensure x == int(x) is the right test for your use case (it compares floats, but you already check for inf/nan).

Why this fix works
- abs(mantissa) % 10 checks the last decimal digit regardless of sign, so trailing zeros are detected for both positive and negative mantissa values.
- Using a separate new_mantissa variable avoids accidental misinterpretation of // for negative numbers and keeps the check/assignment order explicit.

This pinpoints the exact failing line (the while condition) and gives precise, minimal changes to fix the bug.
