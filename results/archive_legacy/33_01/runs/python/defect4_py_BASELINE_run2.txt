What happened
- In the "max middle" branch you fetch the wrong period object before computing max_middle:
  max_middle_period = self.get_data_item(self._min_middle_index).get_period()
  This uses self._min_middle_index instead of self._max_middle_index.
- As a result you are comparing the new period's middle value against the middle of the minimum-middle period, not the current maximum-middle period. That makes the max-middle tracking logic wrong and lets a later item (index 3) become the recorded max even though index 1 should be the max. That is why the assertion Expected max_middle_index=1, got 3 fails.

Where it occurs
- In the block that begins with:
    if self._max_middle_index >= 0:
  the line fetching max_middle_period mistakenly indexes self._min_middle_index.

Why that causes the test to fail
- The test adds periods whose middles are:
    index 0: middle = 50
    index 1: middle = 150
    index 2: middle = 25
    index 3: middle = 75
  The correct max_middle_index should be 1 (middle 150). Because the code compares new middles to the period at self._min_middle_index instead of self._max_middle_index, the comparison is invalid and later items can overwrite the max index incorrectly (producing 3 in your run).

Exact fix (code change)
- Replace the incorrect line with the correct index:
  Change:
    max_middle_period = self.get_data_item(self._min_middle_index).get_period()
  To:
    max_middle_period = self.get_data_item(self._max_middle_index).get_period()

What to do and verify
1. Edit the code as shown above.
2. Run the test:
   s = TimePeriodValues('Test')
   s.add(SimpleTimePeriod(0, 100), 1.0)
   s.add(SimpleTimePeriod(100, 200), 2.0)
   s.add(SimpleTimePeriod(0, 50), 3.0)
   s.add(SimpleTimePeriod(50, 100), 4.0)
   assert s.get_max_middle_index() == 1
3. Optionally add a unit test that constructs several periods with known middles to prevent regressions.

Why this is safe
- The only logical intent of that branch is to compute the current max middle using self._max_middle_index. Using self._min_middle_index is clearly a copy-paste error and does not match any other logic in the method. Replacing it with self._max_middle_index restores the intended comparison.
