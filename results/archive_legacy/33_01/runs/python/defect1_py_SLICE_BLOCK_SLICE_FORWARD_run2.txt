What happened (where it failed)
- The code raises ValueError at line 396:
  if minutes_offset < 0 or minutes_offset > 59:
  raise ValueError(f"Minutes out of range: {minutes_offset}")
  That exception is triggered because minutes_offset is greater than 59.

Why it failed (root cause)
- The variable minutes_offset is being used for two different meanings:
  - once as the minute-component (should be in 0..59), and
  - later as the combined total minutes (hours_in_minutes ± minutes_offset), which is usually > 59 when hours > 0.
- At lines 402 and 404 the code does:
  minutes_offset = hours_in_minutes - minutes_offset
  minutes_offset = hours_in_minutes + minutes_offset
  where hours_in_minutes = hours * 60. Those assignments make minutes_offset a total-minute value (e.g. 90 for 1h30m).
- But the validation at line 396 checks minutes_offset against 0..59. Because the same variable was already or later used to represent total minutes, the check sees values > 59 and throws "Minutes out of range" even though the original minute component (the one that should be checked) may have been valid.

Precise location to inspect
- Validation: line 396 (minutes_offset range check).
- Reassignment to total minutes: lines 402 and 404 (minutes_offset = hours_in_minutes ± minutes_offset).
- Conversion to milliseconds after that: line 405 (offset = ...MILLIS_PER_MINUTE).

How to fix it (concrete steps)
1. Do not reuse minutes_offset for both meanings. Introduce a separate variable for the minute component, e.g. minute_part.
   - After parsing, validate minute_part with:
     if minute_part < 0 or minute_part > 59: raise ValueError(f"Minutes out of range: {minute_part}")
2. Compute the combined total minutes into a different variable, e.g. total_minutes:
   - total_minutes = hours_in_minutes + minute_part  (or subtract when needed)
3. Use total_minutes when converting to milliseconds:
   - offset = FieldUtils.safe_multiply(total_minutes, DateTimeConstants.MILLIS_PER_MINUTE)
4. Remove or update the existing check at line 396 so it validates the minute component variable (minute_part), not the combined total.

Why these steps fix it
- The minute component stays validated against 0..59.
- The total-minutes variable is allowed to exceed 59 (as it should when hours > 0) and won't trigger the minute-range check.
- This prevents the spurious ValueError while preserving correct validation and conversion at lines 405+.

No other behavior needs changing: the error was purely caused by variable reuse and unit confusion (minutes component vs total minutes).
