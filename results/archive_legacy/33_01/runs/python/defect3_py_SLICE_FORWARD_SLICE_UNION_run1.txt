What happened (in plain terms)
- The function assumes code_units is a list of UTF‑16 code units (16‑bit values where non‑BMP characters are represented as surrogate pairs). It then checks for a high surrogate at code_units[index] (line 39) and, if present, reads the next unit to form a full code point (lines 40–43).
- But the code that built code_units (the to_utf16_code_units function; lines around 10–12) did not actually produce UTF‑16 code units. It appended Python code points (ord(c)) directly. For characters outside the Basic Multilingual Plane (ord(c) > 0xFFFF), ord(c) is a single integer > 0xFFFF, not a pair of 16‑bit surrogate values.
- Because code_units contains raw code points instead of 16‑bit units, the surrogate tests fail unpredictably: either the ranges 0xD800..0xDBFF/0xDC00..0xDFFF are never matched and you return a wrong (too large) value at line 44, or the function tries to form a surrogate pair from values that are not UTF‑16 units and produces incorrect results or later errors elsewhere.

Where it fails (specific lines & variables)
- The wrong assumption starts where code_units is built (around line 12). That list must contain UTF‑16 code units, but the implementation appended ord(c) directly.
- The incorrect behavior appears at:
  - line 39: the check 0xD800 <= high <= 0xDBFF fails for non‑BMP characters because high is a value > 0xFFFF (ord(c)), not a high surrogate.
  - line 44: the function returns high (a raw code point > 0xFFFF) instead of combining two UTF‑16 units into a code point, so callers that expect one UTF‑16 unit may get an out‑of‑range integer.
  - If another part of the code assumes code_units are 16‑bit units and uses index arithmetic (consuming two units for a surrogate pair), those assumptions break because to_utf16_code_units produced one element instead of two.

Why it happened (root cause)
- Root cause: mismatch between what to_utf16_code_units creates and what the downstream decoding code expects. The decoder expects a list of 16‑bit UTF‑16 code units (with surrogate pairs split across two list elements). The builder instead stored full Unicode code points (ord(c)), so non‑BMP characters are not split into high/low surrogates.
