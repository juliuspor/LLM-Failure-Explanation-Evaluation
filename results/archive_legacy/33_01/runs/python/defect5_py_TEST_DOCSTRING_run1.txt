What happened
- The failing call is: result = ArrayUtils.add(None, None, expected_type=str)
- Your docstring promises that when array is None and element is None the function returns a one-element list whose component type is List[object]. The test expects the returned value to be [None].
- The actual implementation is performing the expected_type/type check against element even when element is None (and array is None). Because expected_type is str and the code checks something like isinstance(element, expected_type) or compares element.__class__ to expected_type, the check fails and the method raises TypeError instead of returning [None].

Why it fails (root cause, plainly)
- The function checks the parameter expected_type against element unconditionally.
- element is None, so the type check either:
  - treats None as not an instance of str (isinstance(None, str) is False), or
  - compares NoneType to expected_type and finds a mismatch.
- As a result, ArrayUtils.add raises TypeError instead of returning [None] when array is None and element is None.

Where to change it (specific names and place)
- Method: ArrayUtils.add
- Parameters involved: array, element, expected_type
- Faulty behavior happens before the early-return for array is None (or inside that branch but after the unconditional type check). Make the change early in the method, before you enforce expected_type on element.

Concrete fixes (step-by-step)
1. At the start of ArrayUtils.add, handle the array is None case first:
   - If array is None:
     a) If element is None:
        - Return [None] immediately (component type object). Do NOT apply expected_type/type checks.
     b) Else (element is not None):
        - If expected_type is provided, check isinstance(element, expected_type); if it fails, raise TypeError.
        - Return [element].
2. For the non-None array path keep the existing behavior: check expected_type against the array element type or element as intended (but still allow element==None if your docstring permits None elements; that is, treat None as acceptable element if it’s allowed).
3. Make sure any type-check code uses isinstance (or proper typing-aware checks) and only runs when element is not None (unless your design explicitly disallows None elements).

Minimal example patch (pseudo-Python)
Replace the early/type-check portion with:

def add(array, element, expected_type=None):
    # handle None input array first
    if array is None:
        if element is None:
            return [None]          # docstring: List[object] when element is None
        if expected_type is not None and not isinstance(element, expected_type):
            raise TypeError("element type mismatch")
        return [element]

    # existing code for non-None array follows...
    ...

Why this fixes it
- It prevents the function from applying the expected_type check to a None element in the special case array is None, which is exactly what your docstring specifies should be allowed and what your test expects.

Extra note
- If your design intends that expected_type should forbid None values even when element is None, then either the docstring or the test is wrong. But based on the docstring and the failing test, the implementation is wrong and should be changed as above.
