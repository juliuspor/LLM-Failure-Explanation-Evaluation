What happened
- In the "update max middle" block you fetch the wrong period. Instead of getting the period for self._max_middle_index you call:
  max_middle_period = self.get_data_item(self._min_middle_index).get_period()
  (you use self._min_middle_index there).
- That means the later comparison
  if middle > max_middle:
uses the middle value from the currently tracked minimum-middle item, not the current maximum-middle item. This corrupts the logic and produces an incorrect _max_middle_index.

Where it occurs
- In the block that begins "if self._max_middle_index >= 0:" (the one that should update the max-middle index). The single mistaken name is the root cause.

Why it fails (concrete sequence)
- Suppose you have two items and _max_middle_index is set to 1. When a third item is added, the code computes its middle and should compare it against the middle of the item currently at _max_middle_index.
- Because of the typo it instead compares against the middle of the item at _min_middle_index. If that yields a different outcome, the code may wrongly leave _max_middle_index unchanged or set it incorrectly.
- That is exactly why the test using four SimpleTimePeriod items (centers 50, 150, 25, 75) fails to end up with _max_middle_index == 1.

Exact fix (line-level)
- Replace the incorrect line:
  max_middle_period = self.get_data_item(self._min_middle_index).get_period()
  with:
  max_middle_period = self.get_data_item(self._max_middle_index).get_period()

Steps to apply and verify
1. Edit the "max middle" block to use self._max_middle_index when fetching the period (as shown above).
2. Run the existing test:
   s = TimePeriodValues('Test')
   s.add(SimpleTimePeriod(0, 100), 1.0)
   s.add(SimpleTimePeriod(100, 200), 2.0)
   s.add(SimpleTimePeriod(0, 50), 3.0)
   s.add(SimpleTimePeriod(50, 100), 4.0)
   assert s.get_max_middle_index() == 1
3. Add an extra unit test that exercises a few more orderings of intervals to catch future regressions.

Why this fix is safe
- The only incorrect use was a wrong variable name. Using the correct index restores the intended comparison (compare the new middle to the current max-middle item's middle) without changing surrounding logic.

No other changes needed unless you want to harden the routine:
- Optionally add an assertion or test coverage for all four index-tracking code paths (min/max start, min/max middle, min/max end).
