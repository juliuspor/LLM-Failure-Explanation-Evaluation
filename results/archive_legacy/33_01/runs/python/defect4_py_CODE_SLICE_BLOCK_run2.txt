What happened (root cause)
- In _update_bounds, inside the block that updates self._max_middle_index you call:
  max_middle_period = self.get_data_item(self._min_middle_index).get_period()
  That uses self._min_middle_index instead of self._max_middle_index.
- Because of that wrong variable:
  - You compare the new item's middle against the wrong existing period's middle.
  - self._max_middle_index will not be updated correctly (it may stay stale).
  - If _max_middle_index >= 0 but _min_middle_index is -1 you can get an IndexError from get_data_item(-1).
- This bug is in the method _update_bounds (the "max middle" branch) and explains why get_max_middle_index() (lines 449–451 in your slice) returns wrong or invalid values: _max_middle_index is never reliably computed.

Exactly where to change
- In _update_bounds, replace the call that uses self._min_middle_index with self._max_middle_index. Concretely, change:
    max_middle_period = self.get_data_item(self._min_middle_index).get_period()
  to:
    max_middle_period = self.get_data_item(self._max_middle_index).get_period()

Why this fixes it
- The max-middle comparison must examine the current stored max-middle period (indexed by _max_middle_index). Using _min_middle_index examines a different period, so the comparison and any assignment to _max_middle_index are meaningless/wrong. Using the correct index makes the logic consistent and prevents wrong updates or index errors.

1. Edit _update_bounds: in the "if self._max_middle_index >= 0:" branch, replace self._min_middle_index with self._max_middle_index as shown above.

3. Add a defensive assertion (optional) at the start of the branch to make future mistakes fail fast:
