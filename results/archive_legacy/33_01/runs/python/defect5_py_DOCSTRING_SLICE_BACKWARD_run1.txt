What happened (where)
- At line 337 the code enters the branch for a non-None element: "elif element is not None:".
- At line 340 the code sets inferred_type = object.
- At line 346–347 the code checks if inferred_type == object and expected_type != object and raises TypeError when that's true.

Why that is wrong (root cause)
- The branch "element is not None" is exactly the place where the code should infer the element's concrete type (e.g. type(element)).
- Instead, the code unconditionally sets inferred_type to object (line 340). That makes inferred_type == object even when element actually has a more specific type.
- Because inferred_type is object, the subsequent check at line 346–347 thinks the function could not determine a concrete type and therefore raises TypeError whenever the caller provided an expected_type other than object — even though element is valid for that expected_type.
- In short: the code is using object instead of the actual type of element, so valid inputs get incorrectly rejected.

What you will see (symptoms)
- Calling the function with array=None, element not None, and expected_type set to some concrete type (e.g. int) causes a TypeError from the check at line 347.
- Example: add_to_list(None, 3, expected_type=int) will raise TypeError even though 3 is an int.

Exact location to change
- Replace the assignment at line 340:
  inferred_type = object
  with:
  inferred_type = type(element)
  (or element.__class__ — both are equivalent for normal objects)

Why that fix works
- When element is not None, inferred_type becomes the actual runtime type of element.
- The check at lines 346–347 then correctly compares the actual inferred type to expected_type and will not raise if they match.

Summary
- Bug: line 340 sets inferred_type to object inside the "element is not None" branch.
- Consequence: legitimate inputs appear to have unknown type and trigger the TypeError at line 347.
- Fix: set inferred_type = type(element) at line 340 so the later type check works correctly.
