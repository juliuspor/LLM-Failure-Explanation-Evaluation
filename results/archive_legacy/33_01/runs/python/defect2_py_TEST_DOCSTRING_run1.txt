What happened
- The test calls GrayPaintScale().get_paint(-0.5) expecting (0, 0, 0).
- Instead the method raised ValueError. That happens because get_paint computes RGB values from the input value and then rejects out-of-range RGBs instead of returning a clamped black for negative inputs.

Why it happened (root cause)
- get_paint converts the scalar value into r/g/b numbers. For value = -0.5 those numbers become negative.
- The method contains a check that raises ValueError when any channel is outside 0–255. Because the computed channels are negative, that check triggers and the method raises instead of returning (0,0,0).
- In short: the code enforces an RGB range after conversion, but does not handle input values outside the expected domain (negative values). The code is doing validation on the derived r/g/b and raising, rather than clamping or validating the input domain earlier.

Where it occurred
- Inside GrayPaintScale.get_paint, at the line that performs the out-of-range test and raises ValueError (the line similar to: if r < 0 or r > 255 or g < 0 or g > 255 or b < 0 or b > 255: raise ValueError(...)). That check runs after converting value to r,g,b and is what causes the failure for value = -0.5.

Concrete fixes (choose one depending on intended behavior)
1) If you want negative values to map to black (0,0,0) — clamp channels:
- Replace the raise with clamping code. Example change inside get_paint:
  - Before (problematic):
    if r < 0 or r > 255 or g < 0 or g > 255 or b < 0 or b > 255:
        raise ValueError("RGB out of range")
    return (r, g, b)
  - After (clamp channels to 0..255):
    r = max(0, min(255, int(round(r))))
    g = max(0, min(255, int(round(g))))
    b = max(0, min(255, int(round(b))))
    return (r, g, b)
- This returns (0,0,0) for -0.5 as your test expects.

2) If you want to enforce that input value must be in a particular range (for example 0..1) — validate input first and raise a clear error:
- At the start of get_paint add:
    if value < 0 or value > 1:
        raise ValueError("value must be between 0 and 1")
- Keep the existing RGB range check or remove it because input validation ensures channels will be inside bounds.
- Update your test to expect a ValueError for -0.5 (if that is desired behavior).

3) If you want to preserve the current strictness but make the failure clearer
- Change the error message and validate earlier:
    if value < 0 or value > 1:
        raise ValueError(f"value {value} out of domain [0,1]")
- This points to the real cause (bad input) rather than an after-the-fact RGB failure.

Which to pick
- If the docstring (or API contract) expects values outside 0..1 to be accepted or clamped, implement option 1.
- If the API intends only a limited domain, implement option 2 and update tests to assert an error for -0.5.

Summary
- Root cause: get_paint computes negative RGB from value = -0.5 and then raises ValueError because the channels are out of 0–255 bounds.
- Fix by either clamping the computed channels before returning, or by validating the input value up front and updating the test accordingly.
