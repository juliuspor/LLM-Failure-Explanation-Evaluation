What happens
- The code sets inferred_type to object on line 6 (the else branch) when both array and element are None.
- Immediately after building the new list, the code checks:
  if inferred_type == object and expected_type != object: raise TypeError(...)
  That is line 13–18 in your snippet.
- In your test you call ArrayUtils.add(None, None, expected_type=str). Because both array and element are None, inferred_type becomes object, expected_type is str, so the condition is true and the method raises the TypeError. The test expected [None] but the method raised an exception instead.

Why that is the root cause
- The logic assumes that inferred_type == object means “we have no useful type information,” but then treats that as an automatic conflict with any non-object expected_type and raises a cast error. That makes no allowance for the caller providing expected_type to define the element type when both array and element are None.
- The code never uses expected_type to construct or grow the list; it only uses expected_type to decide whether to throw. So the method refuses to produce a list of the requested expected_type when it has no other information.

Where it occurs (specific lines / variables)
- inferred_type assignment:
  - if array is not None: inferred_type = type(array) (line 2–3)
  - elif element is not None: inferred_type = type(element) (line 4–5)
  - else: inferred_type = object (line 6)
- Type check that raises: the block starting with if expected_type is not None: if inferred_type == object and expected_type != object: raise TypeError(...) (lines 13–18)
- Test call that triggers it: ArrayUtils.add(None, None, expected_type=str) — array is None, element is None, expected_type is str.

Concrete fixes (step-by-step)
Option A — prefer expected_type when provided (simple and correct for your test)
1. Change the inferred_type computation to use expected_type when expected_type is given:
   - Replace the whole inferred_type block with:
     if expected_type is not None:
         inferred_type = expected_type
     elif array is not None:
         inferred_type = type(array)
     elif element is not None:
         inferred_type = type(element)
     else:
         inferred_type = object
2. Keep the rest of the logic but remove the TypeError raise that treats inferred_type == object as incompatible. If you still want to validate mismatches, compare actual element or array component types to expected_type explicitly.

Option B — only raise when there is conflicting concrete information
1. Keep inferred_type as-is, but change the check to raise only when you have concrete type information that conflicts with expected_type:
   - if expected_type is not None:
         if inferred_type is not object and inferred_type != expected_type:
             raise TypeError(...)
   This allows the expected_type provided by the caller to be used when inferred_type == object.

Notes about type(array)
- As written, inferred_type = type(array) gives <class 'list'> when array is a list; that is probably not what you want if you mean “list element type.” If you need the element type from an existing list, you must inspect the list contents (e.g., non-None elements) or track element type metadata. Using type(array) to represent element type is misleading and can produce incorrect comparisons to expected_type.

Minimal change to make your test pass
- Implement Option A or Option B. For example, Option B is the smallest change:

    if expected_type is not None:
        if inferred_type is not object and inferred_type != expected_type:
            raise TypeError(...)

This stops raising when inferred_type == object and expected_type is non-object, allowing the method to return [None] for your test.

Summary
- The code raises because inferred_type becomes object when both inputs are None, and the method treats that as incompatible with any provided expected_type. The fix is to let expected_type set the element type when no other information exists, or only raise when inferred_type conflicts with expected_type rather than when it is unknown.
