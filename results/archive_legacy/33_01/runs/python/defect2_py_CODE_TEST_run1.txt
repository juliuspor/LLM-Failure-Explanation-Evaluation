What happened (where & why)
- In get_paint you clamp the input into v on lines 3–4:
  v = max(value, self._lower_bound)
  v = min(v, self._upper_bound)
  This ensures v is inside the allowed range.

- You then compute g using the original value (line 6), not the clamped v:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

- Because the test calls get_paint(-0.5) and -0.5 < self._lower_bound, the unclamped value produces a negative numerator and thus g < 0. The subsequent range check (lines 8–10) raises a ValueError:
  if g < 0 or g > 255:
      raise ValueError(...)

So the root cause is that you clamped to v but never used v in the grayscale calculation; you used value instead, which lets values outside the bounds produce g outside 0–255 and trigger the exception.

Other things to watch for
- If self._upper_bound == self._lower_bound the division (line computing g) will raise ZeroDivisionError. You should guard against that.

Clear fixes (exact steps)
1. Use the clamped v in the computation:
   g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

2. Add a check for zero range to avoid division by zero:
   if self._upper_bound == self._lower_bound:
       g = 0  # or 255, or some defined behavior

3. Improve the ValueError message to include the actual g and bounds if you keep the check:
   raise ValueError(f"Color parameter outside expected range: g={g} (expected 0..255)")

Minimal corrected version
    v = max(value, self._lower_bound)
    v = min(v, self._upper_bound)
    if self._upper_bound == self._lower_bound:
        g = 0
    else:
        g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
    if g < 0 or g > 255:
        raise ValueError(f"Color parameter outside expected range: g={g} (0..255)")
    return (g, g, g)

That change fixes the failing test (get_paint(-0.5) will use v, produce g == 0, and return (0, 0, 0)).
