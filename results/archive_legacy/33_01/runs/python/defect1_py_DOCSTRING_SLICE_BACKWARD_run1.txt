What happened (where it fails)
- The failure occurs at lines 396–397:
  if minutes_offset < 0 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")

- That unconditional check rejects any negative minutes_offset. It therefore raises a ValueError for inputs that the docstring says are valid, for example:
  - (hours_offset=0, minutes_offset=-15) should produce +00:15? (doc says 0,-15 => -00:15) — valid per doc but rejected.
  - (hours_offset=-2, minutes_offset=-15) should be treated as -02:15 (valid) but is rejected.

Why it fails (root cause)
- The code tests minutes_offset in isolation and does not consider hours_offset. The doc allows negative minutes in some cases (hours zero or negative). The condition minutes_offset < 0 is therefore too strict: it forbids all negative minute values even when they are legitimate per the method contract.

Exact variables and lines involved
- minutes_offset is the offending variable (checked at lines 396–397).
- hours_offset is not consulted here but must be for correct validation.

Concrete fixes (step-by-step)
1. Allow minutes_offset to be negative when the hours_offset is non-positive (hours_offset <= 0). Reject negative minutes only when hours_offset > 0.
2. Always enforce the absolute minute bound of 0..59 (i.e., abs(minutes_offset) <= 59).
3. Keep or add the later check that the combined offset does not exceed +/- 23:59:59.000.

Recommended replacement logic (exact replacement for lines 396–397)
- Option A (clear and explicit):
  if abs(minutes_offset) > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")
  if minutes_offset < 0 and hours_offset > 0:
      raise ValueError("Minutes must not be negative when hours is positive")

- Option B (equivalent, two-condition form):
  if minutes_offset < 0 and hours_offset > 0:
      raise ValueError(f"Minutes out of range for positive hours: {minutes_offset}")
  if minutes_offset > 59 or minutes_offset < -59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")

Tests to add after the fix
- Valid cases that must not raise:
  (0, -15)  -> -00:15
  (-2, -15) -> -02:15
  (-2, 15)  -> -02:15
  (2, 15)   -> +02:15
- Invalid cases that must raise:
  (2, -15)  -> should raise (minutes negative with positive hours)
  (0, -60)  -> should raise (abs(minutes) > 59)
  (24, 0)   -> should raise later if you also check hours range

Why this fix matches the docstring
- It aligns validation with the documented allowed sign combinations: negative minutes are permitted only when hours are zero or negative; positive hours cannot be paired with negative minutes.
