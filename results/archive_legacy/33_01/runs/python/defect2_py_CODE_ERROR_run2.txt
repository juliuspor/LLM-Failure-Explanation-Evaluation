What happened (root cause)
- After you clamp the input into v:
    v = max(value, self._lower_bound)
    v = min(v, self._upper_bound)
  you never use v. The next line still uses value:
    g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
  If value was outside the bounds, that expression can produce g < 0 or g > 255, which triggers the subsequent check and raises the ValueError you saw.

Where it occurred
- The problem is on the line that computes g (the third code line inside the method). The validation if g < 0 or g > 255 is working correctly — it’s just catching the out-of-range result caused by using value instead of the clamped v.

Why it fails for your run
- You passed a value outside the [self._lower_bound, self._upper_bound] range. Because the clamped variable v was ignored, g was computed from the original out-of-range value and fell outside 0..255, so the method raised the ValueError.

How to fix it (concrete steps)
1. Use the clamped variable v when computing g:
    g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
2. Guard against division-by-zero when upper == lower:
    denom = (self._upper_bound - self._lower_bound)
    if denom == 0:
        return (0, 0, 0)   # or choose an appropriate constant color
    g = int((v - self._lower_bound) / denom * 255.0)
3. Ensure final g stays inside 0..255 (defensive):
    g = max(0, min(255, g))
4. Optionally use round() instead of int() if you want conventional rounding:
    g = int(round((v - self._lower_bound) / denom * 255.0))

Fixed example
    def get_paint(self, value: float) -> Tuple[int, int, int]:
        v = max(value, self._lower_bound)
        v = min(v, self._upper_bound)
        denom = self._upper_bound - self._lower_bound
        if denom == 0:
            return (0, 0, 0)
        g = int((v - self._lower_bound) / denom * 255.0)
        g = max(0, min(255, g))
        return (g, g, g)
