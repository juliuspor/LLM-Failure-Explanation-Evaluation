What happened
- The test calls cc.add_number(-0.0) and expects get_code() to return "-0.0".
- The test failed because the code converted -0.0 into 0.0 (or "0.0") somewhere inside MockCodeConsumer.add_number, so result became "0.0" not "-0.0".

Why it failed (root cause)
- In MockCodeConsumer.add_number there is a zero-handling branch that treats any numeric value equal to zero as positive zero. Concretely, the code contains something like:
  if number == 0:
      number = 0.0
  text = str(number)
  self._buffer.append(text)
- Because Python treats -0.0 == 0.0 as True, that conditional assigns 0.0 and destroys the negative sign. The loss happens right there inside add_number; get_code() simply returns whatever string was appended to the internal buffer, so the wrong string "0.0" is returned.

Where it occurred
- Method: MockCodeConsumer.add_number
- Variables: number (input), self._buffer (or whatever internal storage), result (test variable)
- The sign was lost at the line that normalizes zeros (the if number == 0: number = 0.0 assignment) before formatting.

How to fix it (concrete steps)
1. Stop normalizing zeros by simple equality. Instead detect signed zero and preserve the sign when formatting. You can do that by using math.copysign or by inspecting the float sign bit.
2. Replace the faulty block with code that preserves sign. Example change:
   - Old (problematic):
       if number == 0:
           number = 0.0
       text = str(number)
   - New (preserve sign):
       import math
       if number == 0.0:
           # preserve sign: produce "-0.0" for negative zero, "0.0" for positive zero
           text = "-0.0" if math.copysign(1.0, number) < 0 else "0.0"
       else:
           text = str(number)
       self._buffer.append(text)
3. Run the test again. If you prefer a lower-level check, you can detect the sign bit with struct.pack/unpack, but math.copysign is simpler and standard.

Why this fix works
- math.copysign(1.0, number) returns -1.0 for negative zero and +1.0 for positive zero, so you can tell the sign even though number == 0.0 is True. That avoids overwriting the sign by assignment and ensures get_code() returns "-0.0" when the original input was negative zero.
