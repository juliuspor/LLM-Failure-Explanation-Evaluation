(at line 71) and immediately tests

Why it failed (root cause):
- The method assumes value lies between self._lower_bound and self._upper_bound and that those bounds are different. If either assumption is false, g can be outside 0..255 (or the division can be invalid). That triggers the ValueError at line 75.
- The code does not:
  1) check for a zero-width range (self._upper_bound == self._lower_bound) before doing the division, so a ZeroDivisionError is possible earlier or the division produces an unexpected number; and
  2) validate or clamp value before mapping, so value outside the configured range produces g < 0 or g > 255.
- The raised ValueError message (line 75) is unhelpful: it does not show which variable caused the problem (value, g, lower_bound, upper_bound) so the developer cannot see the actual numeric cause.

- If self._upper_bound == self._lower_bound, the expression at line 71 will cause a division-by-zero failure or undefined behavior.
- If value < self._lower_bound or value > self._upper_bound, g becomes <0 or >255 and you hit the ValueError at line 75 without actionable diagnostics.
