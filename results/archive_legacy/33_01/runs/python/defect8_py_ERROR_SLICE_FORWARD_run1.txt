What happened (where & when)
- The code raised ValueError("Invalid locale format: fr__POSIX") at the check on line 246 (the raise at line 246).
- That check executed while parsing the input string locale_str == "fr__POSIX".

Why it failed (exact root cause)
- The parser expects fixed positions for language, country and an optional variant (it slices locale_str[0:2] and locale_str[3:5] and then examines locale_str[5]).
- The input "fr__POSIX" has two underscores in a row, which means the country part is empty (language="fr", country="").
- For that input:
  - locale_str[0:2] == "fr"
  - locale_str[3:5] == "_P" (not a two-letter country code)
  - locale_str[5] == "O"
- The code reaches the test if locale_str[5] != '_' on line 245–246 and finds 'O' (not '_'), so it raises ValueError. In short: the parser assumes a non-empty two-letter country at positions 3–4 and an underscore at position 5 when a variant is present; it does not handle an empty country (double underscore), so it mis-parses and rejects "fr__POSIX".

Why the current checks are brittle
- The code relies on fixed index positions (0:2, 3:5, index 5) instead of splitting the string into parts. That makes the code fail for valid locale strings that allow an empty country with a variant (the common form "language__variant", e.g. "fr__POSIX").

   - country = parts[1] if len(parts) > 1 and parts[1] != "" else None
   - variant = parts[2] if len(parts) > 2 else None
   - Validate language is two letters; if country is not None validate two letters; then return Locale(language, country, variant)

   - After extracting language = locale_str[0:2], country = locale_str[3:5], treat country == "__" or country == "" as country = None.
