What happened (where it fails)
- In get_paint the code clamps the incoming value into variable v (lines shown above), but the grayscale index g is computed from the original parameter value, not from v:
  - v = max(value, self._lower_bound)
  - v = min(v, self._upper_bound)
  - g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- Because g is computed from value, not v, it can be outside 0..255 when value is out of bounds. That triggers the ValueError check at lines 74–75:
  - if g < 0 or g > 255:
  -     raise ValueError(...)

Why it fails (root cause)
- The intention of the clamps was to keep the input inside [self._lower_bound, self._upper_bound] before mapping to 0–255. But the mapping uses the raw value, so the clamp has no effect on g. If value < lower_bound or value > upper_bound, the computed g will be <0 or >255 and the ValueError is raised.
- There is also another potential failure left unhandled: if self._upper_bound == self._lower_bound the expression divides by zero.

Exactly where to change (concrete steps)
1. Use the clamped variable v when computing g:
   - Replace the g assignment with:
     g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
2. Protect against division by zero if bounds are equal:
   - If self._upper_bound == self._lower_bound, decide what g should be (e.g., 0 or 255 or 127) and return that; or raise a clear error before dividing.
   Example:
     if self._upper_bound == self._lower_bound:
         g = 0  # or some defined value
     else:
         g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
3. Clamp g into the integer range to avoid off-by-one/rounding surprises:
   - g = max(0, min(255, g))
4. Improve the exception message (if you keep it):
   - Make it specific about which parameter was out of range and show the offending value.
