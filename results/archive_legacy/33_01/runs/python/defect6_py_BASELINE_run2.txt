What happened
- Root cause: negative zero (-0.0) is being treated as the integer 0 and its sign is lost.
- Why: in Python -0.0 == 0 is True and -0.0 < 0 is False. Your code uses both of those facts:
  - The test if x == int(x) (lines in add_number) is True for x == -0.0, so the code enters the integer-handling branch.
  - value = int(x) converts -0.0 to the integer 0 (value = 0).
  - Later you call self.add(str(value)) which produces "0" (no sign).
  - The earlier sign-protection check if x < 0 and prev == '-' (top of add_number) does not trigger for -0.0 because x < 0 is False for -0.0, so you never insert the minus or a space.

Where it occurs (lines you gave)
- The incorrect decision happens in add_number:
  - The sign check at the top: prev = self.get_last_char(); if x < 0 and prev == '-': ... — this fails for -0.0 because x < 0 is False.
  - The integer branch: if x == int(x) and not math.isinf(x) and not math.isnan(x): value = int(x) ... self.add(str(value)) — this converts -0.0 to 0 and adds "0".

Why that gives the test failure
- Test calls cc.add_number(-0.0). Because of the above, result becomes "0" instead of "-0.0", so the assertion Expected -0.0, got 0 fails.

How to fix it (concrete steps and code)
Option A — special-case negative zero early (simple, explicit)
1. Detect negative zero using math.copysign, which distinguishes sign for zero:
   is_neg_zero = x == 0.0 and math.copysign(1.0, x) < 0
2. If is_neg_zero, add "-0.0" (or otherwise handle sign) and return before the integer branch.

Patch snippet:
    import math
    ...
    def add_number(self, x: float):
        # handle negative zero explicitly
        if x == 0.0 and math.copysign(1.0, x) < 0:
            # preserve the negative zero representation
            self.add("-0.0")
            return

    (rest of your code unchanged)

Option B — preserve sign when converting to integer (if you want same formatting rules)
1. Compute sign with math.copysign:
   neg = math.copysign(1.0, x) < 0
2. When you would add str(value) and value == 0 and neg is True, add "-0" or "-0.0" per your formatting rules.

Patch snippet:
    neg = math.copysign(1.0, x) < 0
    if x == int(x) and not math.isinf(x) and not math.isnan(x):
        value = int(x)
        ...

Which to choose
- If you specifically want the string "-0.0" for -0.0 (your test expects '-0.0'), use Option A and return "-0.0" immediately.
- If you prefer to keep your integer-branch logic and only need the sign preserved when value==0, use Option B and choose the exact "-0" vs "-0.0" formatting to match your expectations.

Summary
- The bug is caused by treating -0.0 as equal to 0 (x == int(x)) and using x < 0 to detect negativity — both of which fail for negative zero. Use math.copysign to detect negative zero and preserve the minus sign before converting to int or when formatting the zero.
