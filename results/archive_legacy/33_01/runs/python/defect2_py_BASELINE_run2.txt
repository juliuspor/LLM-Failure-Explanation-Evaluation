What happened (where it fails)
- At line 71 you compute g using the original parameter value:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- But earlier (lines 67–69) you clamped value into v:
  v = max(value, self._lower_bound)
  v = min(v, self._upper_bound)
- You never use v in the formula. So when the caller passes value outside the bounds (for example -0.5 in the test), g is calculated from the unclamped value and can be < 0 or > 255. That causes the check at lines 74–75 to raise the ValueError you see.

Why it happened (root cause)
- A clamped variable (v) is created but not used. The code still uses the original value, so the clamping has no effect on g.
- This is the direct reason the test gps.get_paint(-0.5) fails: value = -0.5, self._lower_bound (presumably 0.0) → formula gives negative g → raises ValueError.

Exact places to change and steps to fix
1. Use the clamped variable v in the calculation (change line 71):
   Replace
     g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
   with
     g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

2. Also guard against zero division if lower and upper bounds are equal (add before computing g):
   if self._upper_bound == self._lower_bound:
       g = 0  # or 255 or some defined behavior; choose what the class should return

3. (Optional but recommended) After converting to int, ensure g is in [0, 255] rather than raising unexpectedly:
   g = max(0, min(255, g))
