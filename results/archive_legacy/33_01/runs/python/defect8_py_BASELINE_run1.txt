What happened (where)
- The exception is raised at lines 237–240 (the ch3/ch4 check). For input "fr__POSIX", ch3 == '_' and ch4 == 'P', so the test
  if ch3 < 'A' or ch3 > 'Z' or ch4 < 'A' or ch4 > 'Z':
  is true and a ValueError is thrown ("Invalid locale format: fr__POSIX").

Why it happens (root cause)
- The code assumes that when length != 2 the country code always occupies characters at indices 3 and 4 (i.e. two uppercase letters). It does not handle the legal case where the country is empty and the variant follows immediately after two underscores (language + "__" + variant), e.g. "fr__POSIX".
- In "fr__POSIX" the characters are:
  index 0='f', 1='r', 2='_' (sep), 3='_' (sep for empty country -> variant begins next), 4='P',...
  The code treats index 3 as the first letter of the country and rejects '_' because it is not between 'A' and 'Z'.

What to change (concrete fixes)
1. Detect the empty-country form (locale_str[3] == '_') and handle it as language + empty country + variant:
   - After verifying locale_str[2] == '_', add:
     - if locale_str[3] == '_': return Locale(locale_str[0:2], "", locale_str[4:])
2. Otherwise keep the current validation that locale_str[3:5] are uppercase letters and, if there is a variant, require locale_str[5] == '_' and return variant from index 6.
3. (Optional) Make the length checks clearer — the current length condition is awkward. Prefer explicit branches:
   - length == 2 -> language only
   - length == 5 -> language + country
   - length >= 7 -> either language + country + variant, or language + "" + variant (double underscore case)

Exact places to edit

  if locale_str[3] == '_':
      return Locale(locale_str[0:2], "", locale_str[4:])
- Keep the existing ch3/ch4 uppercase check as the else branch (for non-empty country).
