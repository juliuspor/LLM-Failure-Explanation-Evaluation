What happened (where it fails)
- At lines 35–36 the code evaluates this condition:
  if index < 0 or index >= len(code_units):
      raise IndexError(f"String index out of range: {index}")
- If the caller passed out = None (which the docstring says should raise ValueError), evaluating len(code_units) fails first because code_units is None. That produces a TypeError ("object of type 'NoneType' has no len()") before any explicit check or the intended ValueError is raised.
- If out is not None but index is outside the valid range, the code raises IndexError with the message "String index out of range: {index}". That both uses the wrong exception type for the situation described in the docstring and gives a misleading message (it talks about a string index while the check is against the output/code_units length).

Why it fails (root cause)
1. Missing early None check for out: the implementation never checks out is None before using len(code_units). That means callers who pass out=None get a TypeError during len(...) rather than the ValueError promised by the docstring.
2. Wrong/misaligned error reporting at the bounds check: the code raises IndexError with a message about a string index. That contradicts the docstring (which says the method should raise ValueError for out=None) and is misleading about what went wrong (it implies a string indexing error rather than an output/list bounds error).

- Failure modes:

  - If out is provided but index invalid → code raises IndexError with a misleading message.
