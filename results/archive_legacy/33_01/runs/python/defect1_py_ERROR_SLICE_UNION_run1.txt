What happened (where it fails)
- The ValueError is raised at line 396:
  if minutes_offset < 0 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")
- Your code rejects a minutes_offset of -15, so you get "Minutes out of range: -15".

Why it fails (root cause)
- The code is trying to accept signed timezone offsets (hours and minutes, positive or negative), but the validation at line 396 forbids any negative minutes.
- Later code (lines 402–405) treats negative minutes as meaningful and tries to combine hours and minutes into a total offset:
  - line 402: minutes_offset = hours_in_minutes - minutes_offset
  - line 404: minutes_offset = hours_in_minutes + minutes_offset
  That logic only makes sense if minutes can be negative or if the sign is handled explicitly.
- Because the check at line 396 denies negative minutes up front, valid inputs like hours = -5, minutes = -15 (meaning -5:15) are rejected even though later code expects to handle them. In short: the sign-handling is inconsistent — validation forbids negatives, computation expects them.

How to fix it (concrete steps and code changes)
Option A — simplest: validate absolute minutes (if minutes should be signed separately)
- Replace the failing check at line 396 with an absolute-range check:
  if minutes_offset < -59 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")
  (This allows negative minutes.)
- Or, more correct and clearer, normalize sign explicitly (recommended below).

Option B — recommended: treat minutes as a magnitude and handle sign based on the hours (clear and robust)
1. At the start, capture the sign from the hours argument (or a separate sign parameter if present). Example:
     sign = -1 if hours_offset < 0 else 1
2. Validate minutes as a magnitude 0..59:
     if minutes_offset < 0 or minutes_offset > 59:
         raise ValueError(f"Minutes out of range: {minutes_offset}")
3. Convert minutes to absolute and compute total minutes then apply sign:
     total_minutes = abs(hours_offset) * 60 + minutes_offset
     total_minutes *= sign
4. Convert to millis and continue:
     offset = FieldUtils.safe_multiply(total_minutes, DateTimeConstants.MILLIS_PER_MINUTE)

Concrete patch sketch (replace lines 396–405)
- Remove the existing inconsistent check and replace with:
  # determine sign from hours, treat minutes as magnitude
  sign = -1 if hours_offset < 0 else 1
  if minutes_offset < 0 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")
  total_minutes = abs(hours_offset) * 60 + minutes_offset
  minutes_offset = sign * total_minutes
  offset = FieldUtils.safe_multiply(minutes_offset, DateTimeConstants.MILLIS_PER_MINUTE)

Why this fixes it
- It explicitly separates sign (from hours) from minute magnitude, so -5 hours and 15 minutes becomes -((5*60)+15) = -315 minutes, while +5 hours and 15 minutes becomes +315.
- It avoids rejecting valid negative offsets like -5:-15 (or rejecting positive minutes with negative hours).
- It keeps the minutes validation strict (0..59) while allowing a negative total offset when appropriate.

If you prefer minimal change: change the single if to allow negative minutes (if your later logic already expects negative minutes). But the recommended change above makes the sign handling explicit and eliminates ambiguity.
