What happened (where it fails)
- In ArrayUtils.add you call with two None values and expected_type=str.
- The code fragment shows these lines around the failure:
  - line 337: elif element is not None:
  - line 340: inferred_type = object
  - line 346: if inferred_type == object and expected_type != object:
  - line 347: raise TypeError(...)
- Because every input element is None, the branch at line 337 never runs, so inferred_type never gets set from any real element.
- inferred_type therefore remains the initial value object (line 340 is the assignment used when a non-None element is found in the normal path, but here the code path leaves inferred_type as object).
- At line 346 the code sees inferred_type == object and expected_type != object (expected_type is str) and raises a TypeError. That is why your call ArrayUtils.add(None, None, expected_type=str) fails even though returning [None] would be reasonable.

Why this is the root cause (plain language)
- The function uses inferred_type to decide whether the actual items match expected_type.
- inferred_type being object is treated as “unknown/wildcard” and the code currently interprets that as a mismatch with a non-object expected_type.
- But inferred_type ends up as object simply because there were no non-None elements to infer a real type from. In other words, the code conflates “we saw no values to infer a type” with “we inferred the generic object type” and then rejects the expected_type when it is not object.

Concrete fixes (specific steps you can apply)
1. Make inferred_type start as None (not object), so you can distinguish “no values seen” from “inferred object”.
   - Change initialization from inferred_type = object to inferred_type = None (wherever inferred_type is initialized).
2. Only set inferred_type to object when you actually encounter a non-None element whose type is object (if that is meaningful), otherwise set it to the element’s real type when element is not None.
   - In the branch at line 337, where you handle non-None elements, set inferred_type = type(element) (or update logic that merges types).
3. Change the check at line 346 to treat None as “no data seen” and accept expected_type in that case:
   - Replace
       if inferred_type == object and expected_type != object:
           raise TypeError(...)
     with something like
       if inferred_type is not None and inferred_type == object and expected_type != object:
           raise TypeError(...)
     or
       if inferred_type == object and expected_type != object and seen_non_none_elements:
           raise TypeError(...)
4. Alternatively, when no non-None elements were seen (inferred_type is None), set inferred_type = expected_type before the final compatibility check so expected_type is accepted.

Minimal patch example (conceptual)
- Before:
    inferred_type = object
    ...
    elif element is not None:
        ...
- After:
    inferred_type = None
    ...
    elif element is not None:
        inferred_type = type(element)
    ...
    if inferred_type is not None and inferred_type == object and expected_type != object:
        raise TypeError(...)

Why these fixes work
- They separate the case “we never saw a value to infer from” from “we inferred a generic object type”, so the function no longer incorrectly raises when all inputs are None and an expected_type is provided.
