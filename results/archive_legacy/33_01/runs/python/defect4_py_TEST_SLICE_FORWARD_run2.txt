What happened
- You create a TimePeriodValues object s and add four time periods in this order:
  1) SimpleTimePeriod(0, 100) -> value 1.0
  2) SimpleTimePeriod(100, 200) -> value 2.0
  3) SimpleTimePeriod(0, 50) -> value 3.0
  4) SimpleTimePeriod(50, 100) -> value 4.0
- You then assert s.get_max_middle_index() == 1. That assertion assumes the item with midpoint 150 (the second added period, value 2.0) is still at index 1.

Why it fails (root cause)
- TimePeriodValues.add(...) reorders the stored entries so that they are sorted by time period start (and/or normalized). It does not preserve the original insertion index order.
- After all four adds, the internal order becomes something like:
  index 0: SimpleTimePeriod(0, 50)   -> midpoint 25  (value 3.0)
  index 1: SimpleTimePeriod(0, 100)  -> midpoint 50  (value 1.0)
  index 2: SimpleTimePeriod(50, 100) -> midpoint 75  (value 4.0)
  index 3: SimpleTimePeriod(100, 200)-> midpoint 150 (value 2.0)
- get_max_middle_index() returns the index of the item with the largest midpoint within the current internal ordering, so it returns 3, not 1. Your test expects the original insertion index (1), which no longer applies.

Where this occurs (relevant names)
- Method that changed the order: TimePeriodValues.add(TimePeriod, double) (the add implementation that inserts/sorts)
- Method used to compute result: TimePeriodValues.get_max_middle_index()
- Objects involved: s (the TimePeriodValues), SimpleTimePeriod(0,100), SimpleTimePeriod(100,200), SimpleTimePeriod(0,50), SimpleTimePeriod(50,100)

Summary
