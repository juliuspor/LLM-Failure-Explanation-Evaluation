What happened (root cause)
- The code treats -0.0 as an integer and then prints int(x), which discards the negative sign.
- Specifically:
  - The test calls add_number(-0.0).
  - The check if x < 0 and prev == '-' (lines near the top) does not run for -0.0 because x < 0 is False for -0.0 (Python considers -0.0 == 0.0 and not less than 0).
  - Next, the code hits if x == int(x) ... which is True for -0.0 because -0.0 == 0 and int(-0.0) == 0.
  - value = int(x) sets value to 0 (losing the negative sign).
  - Later the code does self.add(str(value)) and writes "0" instead of "-0.0".
- Result: the negative zero sign is lost and the test expecting "-0.0" fails.

Where it occurs (specific names/places)
- The failing logic is in add_number:
  - The negative check using x < 0 (the first if) does not detect negative zero.
  - The integer branch (if x == int(x) ...) converts x to value = int(x) and later self.add(str(value)), which serializes 0 rather than preserving "-0.0".

How to fix it (concrete steps)
Option A — Special-case negative zero early (recommended minimal change)
1. Import math if not already imported.
2. Immediately before the integer branch (before if x == int(x) ...), detect negative zero and handle it explicitly:

    if x == 0.0 and math.copysign(1.0, x) < 0:
        # preserve "-0.0"
        self.add("-0.0")
        return

This ensures -0.0 is emitted correctly and avoids the integer path that drops the sign.

Option B — Use sign-aware checks instead of x < 0
1. Replace uses of x < 0 that are intended to detect negative numbers (including -0.0) with a sign test, e.g.:

    is_negative = math.copysign(1.0, x) < 0

2. Use is_negative where appropriate (for the space-prevention check and anywhere you need to know the sign).

Option C — Serialize using str(x) for zeros
1. Avoid the int conversion branch for exact zeros (x == 0.0). For example, add a check:

    if x == 0.0:
        # preserve sign of zero
        self.add(str(x))
        return

and keep the integer-optimization path for other integers.

Why these fixes work
- math.copysign(1.0, x) < 0 reliably detects negative zero while x < 0 does not.
- Handling -0.0 before int(x) is taken prevents losing the sign when value = int(x) converts -0.0 to 0.

Minimal code patch (option A)
- Add at top of add_number (after prev = ...):

    if x == 0.0 and math.copysign(1.0, x) < 0:
        self.add("-0.0")
        return

This directly fixes the failing test while changing as little else as possible.
