What happened (symptom)
- The call DateTimeZone.for_offset_hours_minutes(-2, -15) fails the assert that zone.id == "-02:15". Either the method throws ValueError or it returns the wrong offset string (for example "-01:45" or "+02:15"), depending on your implementation. The test expects a zone representing minus two hours and minus fifteen minutes.

Where the bug is (root cause)
- The problem is in the input sign/validation logic inside for_offset_hours_minutes. The method takes two parameters (hours_offset and minutes_offset) and then validates and normalizes them before producing the zone.
- The code treats the sign of the minutes_offset inconsistently with the hours_offset. Concretely:
  - It validates minutes_offset against range [-59, +59] but also has a conditional that rejects negative minutes when hours_offset is negative (or otherwise applies minutes_offset as if its sign must match hours_offset). That check contradicts the documented table that explicitly allows (hours negative, minutes negative) => -02:15.
  - Or alternatively, the code applies the minutes_offset sign twice (for example by applying hours_offset sign to an already-negative minutes_offset), so the calculated total offset is wrong.
- In short: the code either rejects minutes_offset < 0 when hours_offset < 0, or it mis-applies sign when building total minutes. The result is either an exception or a wrong offset value for the (-2, -15) case.

Why that causes the failure (simple math)
- Desired behaviour for (-2, -15): resulting offset in minutes = - (2*60 + 15) = -135 minutes => "-02:15".
- A buggy implementation that treats minutes_offset as positive when hours_offset is negative would compute - (2*60) + 15 = -105 minutes => "-01:45" (wrong).
- A buggy implementation that rejects minutes_offset < 0 when hours_offset < 0 will throw ValueError and the assert cannot run.

Exactly where to fix it (what to change)
- Locate for_offset_hours_minutes(hours_offset, minutes_offset).
- Replace the current sign/validation code with this clear procedure:
  1. Validate numeric ranges:
     - if hours_offset < -23 or hours_offset > 23: raise ValueError
     - if minutes_offset < -59 or minutes_offset > 59: raise ValueError
  2. Decide overall sign:
     - If hours_offset != 0: sign = sign_of(hours_offset)
     - Else: sign = sign_of(minutes_offset) (so (0, -15) produces negative)
  3. Compute absolute components:
     - abs_hours = abs(hours_offset)
     - abs_minutes = abs(minutes_offset)
  4. Compute total offset in minutes (or milliseconds) as:
     - total_minutes = sign * (abs_hours * 60 + abs_minutes)
  5. If total_minutes is outside allowed total-range (e.g. > 23*60+59 or < -(23*60+59)) raise ValueError.
  6. Build zone from total_minutes (or milliseconds) and format id as sign + two-digit hours + ":" + two-digit minutes.
- This makes (-2, -15) produce sign = -1, abs_hours = 2, abs_minutes = 15 → total_minutes = -135 → id "-02:15".

Concrete pseudocode to replace the buggy block
- (using your parameter names hours_offset and minutes_offset)

  if hours_offset < -23 or hours_offset > 23:
      raise ValueError("hours_offset out of range")
  if minutes_offset < -59 or minutes_offset > 59:
      raise ValueError("minutes_offset out of range")

  if hours_offset > 0:
      sign = 1
  elif hours_offset < 0:
      sign = -1
  else:
      sign = 1 if minutes_offset >= 0 else -1

  total_minutes = sign * (abs(hours_offset) * 60 + abs(minutes_offset))

  if abs(total_minutes) > (23 * 60 + 59):
      raise ValueError("total offset exceeds allowed range")

  // proceed to create DateTimeZone from total_minutes

Why this exact change fixes it
- It separates range checks (allowed magnitudes) from sign determination, so negative minutes are allowed when hours are negative (and zero hours + negative minutes becomes negative minutes zone).
- It avoids double-applying or losing a sign by always taking abs() of components and applying a single, well-defined sign to the summed minutes.
