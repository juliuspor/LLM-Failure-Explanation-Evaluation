What happened (where it failed)
- The call was DateTimeZone.for_offset_hours_minutes(-2, -15).
- Inside for_offset_hours_minutes the parameter minutes_offset is -15.
- At line 396 the code checks:
  if minutes_offset < 0 or minutes_offset > 59:
and at line 397 it raises ValueError("Minutes out of range: {minutes_offset}").
- Because minutes_offset is -15 the check is true and the ValueError is raised. The code rejects negative minutes even though the caller passed a negative minutes to indicate a negative offset.

Why it failed (root cause)
- The method treats minutes_offset as if it must always be non‑negative (0..59). The API call, however, passed minutes with the same sign as hours (both negative), so minutes_offset was -15 and triggered the "out of range" check.
- In short: the validation at lines 396–397 assumes minutes_offset >= 0, but the caller provided negative minutes to indicate sign. The sign handling is inconsistent between hours and minutes.

- The offending code is the validation of minutes_offset at lines 396–397 inside DateTimeZone.for_offset_hours_minutes.
- Variable names: minutes_offset (the parameter being validated) and hours_offset (or similar — the hours parameter passed in) are relevant.

Which to choose
- If you want the API to accept callers like DateTimeZone.for_offset_hours_minutes(-2, -15) without changing all call sites, implement Option B.

Summary
- The ValueError is raised because minutes_offset is negative and the code at lines 396–397 requires 0..59.
- Fix by either normalizing signed minutes inside for_offset_hours_minutes (preferred if you want backward-compatible behavior) or requiring callers to pass nonnegative minutes and taking sign from the hours parameter.
