What happened (where it failed)
- The code raises the TypeError at the check starting on the lines:
  ```
  if expected_type is not None:
      if inferred_type == object and expected_type != object:
          raise TypeError(...)
  ```
  That check runs after you computed inferred_type and after calling ArrayUtils._copy_list_grow1(...).

Why it failed (root cause)
- The variable inferred_type is computed incorrectly:
  ```
  if array is not None:
      inferred_type = type(array)
  elif element is not None:
      inferred_type = type(element)
  else:
      inferred_type = object
  ```
- type(array) returns the type of the list object (e.g. list), not the element type. If array is empty or otherwise cannot tell the element type, inferred_type ends up being a generic container type (or object in your code path), so inferred_type == object becomes true.
- Later you pass that inferred_type into ArrayUtils._copy_list_grow1, which creates a new list with an "object" element type. Because expected_type was supplied as str, the code then treats this as an attempt to cast an object-list to a str-list and raises the TypeError:
  "Cannot cast object list to str list ..."

Concisely: the function tries to infer the element type from the container itself (type(array)) instead of from the contents (the element or elements inside the array). That wrong inference leads to inferred_type == object and triggers your explicit TypeError when expected_type is more specific than object.

What to change (concrete fixes)
1. Prefer the element type when available:
   - If array is not None and non-empty, use type(array[0]) (or examine the actual elements to decide a common type).
   - Else if element is not None, use type(element).
   - Else fall back to expected_type if provided, otherwise to object.
   Example ordering:
   ```
   if array is not None and len(array) > 0:
       inferred_type = type(array[0])
   elif element is not None:
       inferred_type = type(element)
   elif expected_type is not None:
       inferred_type = expected_type
   else:
       inferred_type = object
   ```
2. If array may contain mixed element types, decide and document what you want to do (pick the first element's type, pick a common base, or require expected_type).
3. Use expected_type as authoritative when appropriate: if expected_type is provided and array is empty and element is None, use expected_type instead of object so you do not raise unnecessarily.
4. Add a clear unit test for the case: array == [] and expected_type == str (and element is None) to ensure your fix prevents the TypeError.

Lines to change
