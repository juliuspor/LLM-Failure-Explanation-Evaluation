What happened
- At lines 396–397 the code always rejects negative minute offsets:
  ```
  396 |         if minutes_offset < 0 or minutes_offset > 59:
  397 |             raise ValueError(f"Minutes out of range: {minutes_offset}")
  ```
- That condition violates the docstring: negative minute values are valid in some combinations (for example (0, -15) → -00:15 and (-2, -15) → -02:15). Because of the check on line 396, the factory raises ValueError for any negative minutes_offset, causing valid inputs to fail.

Why it fails (root cause)
- The code conflates two different checks:
  1. the absolute allowed range for minutes (should be -59..+59),
  2. a sign-consistency rule (positive hours MUST NOT be combined with negative minutes).
- The implemented check enforces "minutes must be non‑negative", which is stricter than the docstring and rejects valid inputs like minutes_offset = -15 when hours_offset is 0 or negative.

Where to change it
- File: the code containing lines 396–397 (the factory method described in the docstring).
- Change the validation logic for minutes and add the sign-consistency check against hours_offset.

Concrete fixes (patch-style guidance)
1. Allow minutes in the range -59..+59:
   Replace the current range check with:
   ```
   if minutes_offset < -59 or minutes_offset > 59:
       raise ValueError(f"Minutes out of range: {minutes_offset}")
   ```
2. Enforce the docstring rule that a positive hours_offset cannot be combined with negative minutes_offset:
   ```
   if hours_offset > 0 and minutes_offset < 0:
       raise ValueError("Positive hours cannot be combined with negative minutes")
   ```
3. (Optional but required by the docstring) check resulting total offset does not exceed ±23:59:59:
   ```
   total_seconds = hours_offset * 3600 + minutes_offset * 60
   max_seconds = 23 * 3600 + 59 * 60 + 59
   if abs(total_seconds) > max_seconds:
       raise ValueError("Resulting offset exceeds +/- 23:59:59.000")
   ```

Suggested replacement snippet (complete)
```
# minutes allowed to be negative for some combinations
if minutes_offset < -59 or minutes_offset > 59:
    raise ValueError(f"Minutes out of range: {minutes_offset}")

# disallow +hours with -minutes per docstring
if hours_offset > 0 and minutes_offset < 0:
    raise ValueError("Positive hours cannot be combined with negative minutes")

# ensure overall offset does not exceed 23:59:59
total_seconds = hours_offset * 3600 + minutes_offset * 60
max_seconds = 23 * 3600 + 59 * 60 + 59
if abs(total_seconds) > max_seconds:
    raise ValueError("Resulting offset exceeds +/- 23:59:59.000")
```

Test cases to verify behavior
- Should succeed:
  - (2, 15) → +02:15
  - (0, -15) → -00:15
  - (-2, 15) → -02:15
  - (-2, -15) → -02:15
- Should fail:
  - (2, -15) → ValueError (positive hours with negative minutes)
  - (24, 0) or (0, 60) or (0, -60) → Minutes/hours out of range
  - (23, 59) is OK; (23, 60) should fail; offsets whose absolute seconds > 23:59:59 fail

Summary
- The failing line is 396–397: it rejects negative minutes unconditionally. Change the minutes range check to allow -59..+59 and add a separate sign-consistency check (hours>0 and minutes<0) plus the overall offset magnitude check. This brings the implementation into agreement with the docstring.
