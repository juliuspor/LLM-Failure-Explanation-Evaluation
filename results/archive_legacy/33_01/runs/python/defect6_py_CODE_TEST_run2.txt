What happened (where it fails)
- The code enters the integer branch at the line
  if x == int(x) and not math.isinf(x) and not math.isnan(x):
  for the value x = -0.0, because -0.0 == 0 is True.
- Then value = int(x) sets value to 0 and loses the negative sign.
- Earlier you tried to treat negative numbers specially with if x < 0 and prev == '-': … but for -0.0 the test x < 0 is False (in Python -0.0 is not less than 0), so that code path never runs either.
- As a result the function prints the unsigned integer representation ("0" or "0E…") instead of "-0.0", so the assertion result == '-0.0' fails.

Why precisely (root cause)
- Two things combine:
  1) The integer-detection check x == int(x) treats -0.0 as an integer-equal value and forces integer conversion (value = int(x)) which discards the sign bit.
  2) The sign check uses x < 0 which is False for -0.0, so you don’t restore the negative sign later.
- The sign bit of negative zero is special in floating-point; equality and int() do not preserve it, so the current logic loses it.

Exact places to change (what to do, step-by-step)
Option A — simplest, explicit negative-zero early-return
1. After prev = self.get_last_char() (before the integer branch), add a special-case check for negative zero:

2. That immediately prints "-0.0" and exits; no other code will strip the sign.

Which to pick
- Option A is simplest and most explicit if the only special case you need is negative zero formatting.
- Option B integrates with the integer formatting logic if you prefer to keep one path for integers.

Why math.copysign
- math.copysign(1.0, x) returns -1.0 for negative-zero and +1.0 for positive-zero, so it reliably detects the sign bit even when x == 0.0.
