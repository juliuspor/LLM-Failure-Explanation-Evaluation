What happened
- The method rejects negative minutes unconditionally at line 396:
  if minutes_offset < 0 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")
- Your test calls DateTimeZone.for_offset_hours_minutes(-2, -15). Because minutes_offset is -15 that check raises ValueError("Minutes out of range: -15") before any further logic runs.

Why that's wrong
- The docstring (and the tests) allow negative minutes when the hours are negative (and also allow negative minutes when hours == 0). The current check disallows negative minutes in all cases, so it contradicts the documented/expected behaviour.
- There is additional code below that expects negative minutes to be possible in some hour-sign cases (the code tries to combine hours and minutes differently depending on hours_in_minutes < 0), so the unconditional rejection is the root cause.

Where to change (exact places)
1. Replace the unconditional minutes range check at line 396 with two checks:
   - Validate absolute range: minutes must be between -59 and 59.
   - Disallow negative minutes only when hours_offset is positive (hours > 0). Hours == 0 and hours < 0 may have negative minutes.
   Concretely:
   if minutes_offset < -59 or minutes_offset > 59:
       raise ValueError(f"Minutes out of range: {minutes_offset}")
   if hours_offset > 0 and minutes_offset < 0:
       raise ValueError(f"Minutes must be non-negative when hours are positive: {minutes_offset}")

2. Replace the hours+minutes composition logic (lines around 401–405) with a clear formula that yields the correct total minutes for every allowed sign combination:
   - If hours_offset >= 0: total_minutes = hours_offset * 60 + minutes_offset
   - If hours_offset < 0: total_minutes = hours_offset * 60 - abs(minutes_offset)
   This ensures both (-2, 15) and (-2, -15) produce -135 minutes (i.e. -02:15), while (2, -15) is rejected by the second validation above.

Suggested fixed code (only the changed block)
    if minutes_offset < -59 or minutes_offset > 59:
        raise ValueError(f"Minutes out of range: {minutes_offset}")
    if hours_offset > 0 and minutes_offset < 0:
        raise ValueError(f"Minutes must be non-negative when hours are positive: {minutes_offset}")
    offset = 0
    try:
        hours_in_minutes = hours_offset * 60
        if hours_offset < 0:
            total_minutes = hours_in_minutes - abs(minutes_offset)
        else:
            total_minutes = hours_in_minutes + minutes_offset
        offset = FieldUtils.safe_multiply(total_minutes, DateTimeConstants.MILLIS_PER_MINUTE)
    except ArithmeticError:
        raise ValueError("Offset is too large")

Why this fixes the failing test
- For (-2, -15):
  - passes the new validations (minutes in [-59,59], hours_offset not > 0)
  - hours_offset < 0 so total_minutes = -120 - 15 = -135
  - offset becomes -135 * MILLIS_PER_MINUTE and for_offset_millis(-135*...) yields "-02:15" as expected.

Summary
- Root cause: an unconditional minutes < 0 check (line 396) rejects inputs that the doc/test expect to allow.
- Fix: validate absolute minutes range, reject negative minutes only when hours are positive, and compute total minutes using hours_in_minutes ± abs(minutes_offset) for negative hours so sign combinations produce the documented results.
