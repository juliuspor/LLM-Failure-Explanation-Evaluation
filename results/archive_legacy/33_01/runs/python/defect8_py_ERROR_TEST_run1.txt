What happened
- The call LocaleUtils.to_locale("fr__POSIX") fails because the parser rejects an empty country field.
- Inside LocaleUtils.to_locale(...) the code splits the input string on '_' into parts (usually language, country, variant). For "fr__POSIX" that split yields parts = ["fr", "", "POSIX"] — an empty string for the country is correct and expected.
- The to_locale implementation then validates parts[1] (the country) and treats an empty string as invalid, raising ValueError("Invalid locale format: " + input). That is why you get ValueError: Invalid locale format: fr__POSIX instead of a Locale("fr", "", "POSIX").

Where it occurred (exact places to look)
- In the method LocaleUtils.to_locale(String) — specifically the input-parsing/validation block that:
  1) does input.split('_') or tokenizes on '_' into a parts array/list, and
  2) immediately rejects empty parts[1] (the country) rather than allowing an empty string.
- Look for code like:

    if (parts.length == 3) {
      if (parts[1].isEmpty()) { throw new ValueError(...); } // this is the failing check

Why that check is wrong
- The Java/Unicode locale format allows an empty country with an explicit variant separated by two underscores: language + "__" + variant. In other words, "fr__POSIX" means language="fr", country="", variant="POSIX".
- By rejecting an empty parts[1], the code treats a valid locale form as invalid.

2) Remove or change the check that rejects an empty country token. Instead accept parts[1] == "" and pass it through to the Locale constructor.
   Example replacement logic:
   - After splitting:
       String language = parts[0];
       String country  = parts.length > 1 ? parts[1] : "";
       String variant  = parts.length > 2 ? parts[2] : "";
