What happened (where)
- Lines 69–70 clamp the input into the local variable v:
  v = max(value, self._lower_bound)
  v = min(v, self._upper_bound)
- But line 71 computes g using the original value, not the clamped v:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- Because value (−0.5 in your test) is outside the bounds, g becomes negative and the check on lines 74–75 raises ValueError.

Why it happens (root cause)
- The code clamps into v but then ignores v and uses value for the normalized computation. That lets out-of-range inputs produce out-of-range g even though you intended to clamp them first.
- Also note: if self._upper_bound == self._lower_bound, the expression on line 71 would divide by zero; the current code has no guard for that.

Concrete fix (step-by-step)
1. Use the clamped variable v in the calculation on line 71:
   g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
2. Add a guard for a zero range (upper == lower) to avoid division by zero:
   if self._upper_bound == self._lower_bound:
       g = 0  # or 255, or whatever makes sense for your class
3. Optionally, clamp g to [0, 255] after computation to be extra-safe:
   g = max(0, min(255, g))

Minimal corrected example
    v = max(value, self._lower_bound)
    v = min(v, self._upper_bound)
    if self._upper_bound == self._lower_bound:
        g = 0

        g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

Why this fixes it
- Using v ensures out-of-range inputs are mapped inside the bounds before computing g.
- The zero-range guard prevents division-by-zero errors.
- The final clamp guarantees g stays in [0,255], so the ValueError won’t be raised for expected inputs.
