What happened
- The docstring promises "Raises: ValueError: if out is None".
- The implementation never checks out for None. There is no guard that raises ValueError when out is None.
- Later code (not shown in the slice) calls out.append(...) and fails with AttributeError: 'NoneType' object has no attribute 'append' (or raises a TypeError from using out as a list). The real runtime exception is therefore not ValueError and is less informative than the docstring promised.

Where it occurred (lines and variables)
- The method's contract mentions the parameter out, but there is no None-check before the code that uses out.
- The only shown checks are the index bounds check at lines 35–36 (raises IndexError for an invalid index) and the surrogate-pair handling at lines 38–44 (reads code_units[index] and code_units[index+1]).
- The failure happens when out is None and the code later tries to call out.append(...) (that call is not in your slice, but it is the place that will crash because out is None).

Why it fails (root cause)
- Mismatch between contract and implementation: the code promises a ValueError for a None out, but it never raises one. Instead the code proceeds and an AttributeError (or similar) occurs when the method uses out as if it were a list.
- Because the function does not validate out at entry, callers who pass None get an unexpected exception type and location; that makes the bug harder to diagnose.

2. Optionally: make the error message specific and include the parameter name (example above).
3. Add a unit test that calls the method with out=None and asserts that ValueError is raised.

   - Implement it explicitly, e.g. if out is None: out = [] and return it (and update return type/documentation).
