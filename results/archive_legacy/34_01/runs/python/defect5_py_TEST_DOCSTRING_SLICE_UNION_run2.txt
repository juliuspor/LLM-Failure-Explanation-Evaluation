What happened
- Test call: ArrayUtils.add(None, None, expected_type=str)
- The code reaches the branch where it must decide an element type to use for the returned list.
- At line 340 the code sets inferred_type = object (because element is None).
- At lines 346–347 the code checks:
    if inferred_type == object and expected_type != object:
        raise TypeError(...)
  Because inferred_type == object and expected_type == str, it raises TypeError instead of returning [None].
- That is why the test fails: the implementation treats "inferred_type == object" as a type conflict with any non-object expected_type and raises, even when the element is None and the docstring says the function should return List[object] in that case.

Why this is wrong (root cause)
- The logic assumes inferred_type == object always represents a real type mismatch. But when element is None (and array is None), inferred_type is correctly object because there is no concrete element to infer from.
- The docstring says: if the element is None, return type should be List[object] (unless element itself supplies a type). The code instead refuses when an expected_type other than object is provided, so it enforces the expected_type even though there is no element to validate against.
- In short, the code conflates "we couldn't infer a type because element is None" (acceptable) with "we inferred object and therefore must conflict with expected_type" (not acceptable).
