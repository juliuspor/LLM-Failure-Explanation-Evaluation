What happened (root cause, where)

- At the branch starting on the line
  if x == int(x) and not math.isinf(x) and not math.isnan(x):
  you convert x to an integer with
  value = int(x)
  (same block as your exp/mantissa logic).
- For x == -0.0, int(x) becomes 0 (so value == 0) and the equality x == int(x) is True because -0.0 == 0.0 in Python.
- Converting -0.0 to int loses the negative sign, so later you call
  self.add(str(value))
  which appends "0" (line shown in the snippet), not "-0.0".
- The failing assertion expected '-0.0' but got '0'.

Why it mattered here (why the sign was lost)

- IEEE floats have a negative zero that compares equal to zero. int(-0.0) -> 0 and x == int(x) holds, so the code enters the integer-handling branch and prints the integer value (0), dropping the sign.
- The earlier check that tried to avoid printing e.g. "x--4" uses x < 0 to decide whether to insert a space:
  prev = self.get_last_char()
  if x < 0 and prev == '-':
      self.add(" ")
  But x < 0 is False for -0.0, so that check also does not treat -0.0 as negative. In short, using numeric comparisons and int() loses the negative-zero information.

Exactly where to fix it (concrete steps)

1) Preserve negative zero before converting to int. Add a check right after you compute prev (before the integer branch):

   import math
   # immediately after prev = self.get_last_char()
   if x == 0.0 and math.copysign(1.0, x) < 0:
       # x is negative zero; print '-0.0' and return
       if prev == '-':
           self.add(" ")
       self.add("-0.0")
       return

This uses math.copysign(1.0, x) < 0 to detect the sign bit of 0.0 reliably.

2) Alternatively, if you want to preserve the exact float text in all non-integer cases, you can use repr(x) or format(x) for floats; repr(-0.0) yields '-0.0'. But because you special-case integer-valued floats here, the explicit negative-zero check is the safest minimal change.

Why that exact code fixes it

- The math.copysign test detects the negative sign on zero even though x == 0.0 and x < 0 are both False.
- Printing "-0.0" (and returning) avoids entering the integer branch that would convert the value to 0 and lose sign.
- The extra prev == '-' spacing check mirrors your existing spacing logic so you don't produce accidental "x--0.0".

Summary

The failure is because int(-0.0) becomes 0 and the code prints that 0. Detect the negative zero via math.copysign (or 1.0/x == -math.inf) before converting to int and handle it explicitly (print "-0.0" and return).
