What happened
- Call: ArrayUtils.add(None, None, expected_type=str)
- At the top of add (lines before 346) both array and element are None, so inferred_type is set to object.
- new_list is created and element None is inserted successfully.
- Then the guard at lines 346–351 runs:
  - inferred_type == object is true
  - expected_type != object is true (expected_type is str)
  - so the method raises the TypeError. The test expects [None] but the function raises instead.

Why it fails (root cause)
- The code treats inferred_type == object as a sign that the existing array is an object-typed list and therefore forbids producing a list of the requested expected_type. That makes sense when you have a concrete array with element type object that you try to cast to a different element type.
- But when both array and element are None, inferred_type becomes object only because there was no information. In that case the caller provided expected_type=str to tell the function what type to create; the function should honor expected_type instead of treating the unknown inferred_type as a blocking cast.
- In short: the code conflates "unknown type" (both inputs None) with "actual object-typed array", so it wrongly raises at lines 346–351.

Where to change it (concrete steps)
1. Decide that an explicit expected_type parameter should override the unknown inferred_type. That is, if expected_type is provided and neither array nor element gave a type, use expected_type as inferred_type.
2. Implement one of these fixes:
   - Prefer expected_type when it is passed:
     - After computing inferred_type, if expected_type is not None and inferred_type is object: set inferred_type = expected_type (so the created list uses expected_type).
   - Or change the final guard so it only raises when the original array existed and had element type object (i.e., when array is not None and inferred_type == object and expected_type != object).
3. Update the code at lines 346–351 accordingly.
