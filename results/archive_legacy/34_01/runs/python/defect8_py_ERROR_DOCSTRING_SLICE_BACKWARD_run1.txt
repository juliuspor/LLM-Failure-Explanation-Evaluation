What happened (where)
- The ValueError is raised in LocaleUtils.to_locale at lines 231–240.
- For the input "fr__POSIX", length = 9, so code reaches the branch that expects a country code at indices 3 and 4:
  - ch3 = locale_str[3]  (this is '_' for "fr__POSIX")
  - ch4 = locale_str[4]  (this is 'P')
- The code checks
    if ch3 < 'A' or ch3 > 'Z' or ch4 < 'A' or ch4 > 'Z':
        raise ValueError(f"Invalid locale format: {locale_str}")
  Because ch3 == '_' is not between 'A' and 'Z', that conditional is true and the method raises the ValueError shown.

Why it failed (root cause)
- The code strictly requires a two-letter uppercase country code at positions 3–4 whenever it sees a third character (index 2) that is '_'. It does not handle the valid locale form where the country is empty and the variant is present, which is written with a double underscore: language + "__" + variant (e.g. "fr__POSIX").
- In "fr__POSIX" the double underscore means: language = "fr", country = "" (empty), variant = "POSIX". The implementation instead treats the character at index 3 as the first country letter and so fails the uppercase-letter check.

Exact spots to change
- The failing check is at lines 231–240 (variables ch3, ch4 and the uppercase test). The parser must detect and accept the special case where locale_str[2] == '_' and locale_str[3] == '_' (double underscore) and treat the substring after the second underscore as the variant, without requiring an uppercase country code.

       country = ""
       variant = locale_str[4:]   # variant can be any case/length per JDK rules

Alternative (clean) approach
- Replace ad-hoc index checks with a small regex to parse the three parts allowing empty country:
  - pattern: r'^([a-z]{2})(?:_([A-Z]{2})|_)?(?:_(.+))?$' (adjust to exact rules you want)
  - Extract language, country (may be empty), variant.

Why this fix is safe
- It directly handles the documented, valid format "language__variant" (double underscore) that the current code incorrectly rejects.
- It keeps the existing strict checks for the normal language_country and language formats.

Summary
The code treats the first character after the initial underscore as the first letter of a country and enforces uppercase. For "fr__POSIX" that character is '_' (the second underscore), so the uppercase check fails and ValueError is raised. Add a branch to detect and handle the double-underscore (empty country) case before enforcing the uppercase-country check.
