What happened
- The test calls GrayPaintScale.get_paint(-0.5).
- Inside get_paint you pass the computed color components directly to your Color constructor (e.g. Color(r, g, b)).
- The Color constructor checks that each channel is in its allowed numeric range, and raises:
  ValueError: Color parameter outside of expected range: Red Green Blue
  because at least one of r/g/b is negative (result of multiplying -0.5 by the scale), so the check fails.

Why it failed (root cause)
- The input value -0.5 is outside the allowed domain for color generation (you expected values in [0,1] or [0,255]).
- get_paint did not clamp or validate the input before converting to the Color constructor, so negative numbers were passed to Color and triggered its range check.
- This manifests specifically at the line where Color(...) is called inside GrayPaintScale.get_paint.

Where it occurred (which names to look at)
- Method: GrayPaintScale.get_paint(...)
- Variable: the incoming parameter (the negative value passed to get_paint) and the computed channel values passed into Color(...) inside that method
- The exception originates from the Color constructor that validates Red, Green, Blue ranges

Concrete fixes (step-by-step)
1. Validate or clamp the input value at the start of get_paint. If the allowed input domain is [0.0, 1.0], do:
   value = max(0.0, min(1.0, value))
   This ensures negative inputs become 0.0 (black), matching the test expectation.

2. After clamping, convert to the expected channel range and type before calling Color. Example for 0–255 integer channels:
   channel = int(round(value * 255))
   Color(channel, channel, channel)

3. Optionally, make the behavior explicit in docstring and/or raise a clear error for totally invalid inputs if you prefer not to clamp.

Minimal corrected code example (replace the Color call inside get_paint):
- If Color expects integer 0–255:
    def get_paint(self, value):
        # clamp input to [0.0, 1.0]
        v = max(0.0, min(1.0, value))
        c = int(round(v * 255))
        return Color(c, c, c)

- If Color expects floats 0.0–1.0:
    def get_paint(self, value):
        v = max(0.0, min(1.0, value))
        return Color(v, v, v)

Why this fixes the test
- The test expects get_paint(-0.5) to yield (0, 0, 0). Clamping -0.5 to 0.0 produces channel 0, so Color receives valid in-range values and no ValueError is raised.

If you want instead to detect invalid use rather than silently clamp:
- Replace the clamp with an explicit check and raise a clearer error at the start of get_paint:
    if not (0.0 <= value <= 1.0):
        raise ValueError(f"get_paint: value {value!r} out of range [0.0,1.0]")

Summary
- Root cause: negative input passed through to Color, which rejects out-of-range channel values.
- Fix: clamp/validate the input inside GrayPaintScale.get_paint before converting to color channels and calling Color.
