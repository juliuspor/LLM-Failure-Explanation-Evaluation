What happened (what you saw)
- The test calls cc.add_number(-0.0).
- add_number constructs the sign and the digits separately (it calls add("-") then add("0.0")).
- At add(newcode) (lines shown), when adding the "0.0" piece the code decides it needs to insert a separating space and produces "- 0.0" instead of "-0.0".
- The assertion fails because result is "- 0.0" (has a space) not the expected "-0.0".

Why it happened (root cause, exact place)
- The decision to insert a space comes from line 133:
  if (self.is_word_char(c) or c == '\\') and self.is_word_char(self.get_last_char()):
- When add("0.0") runs, c = "0" so (self.is_word_char(c) or ...) is True.
- The second part self.is_word_char(self.get_last_char()) is called with the last character being "-" (from the earlier add("-")).
- Because is_word_char("-") returns True in the current implementation, the whole condition becomes True and the code executes line 136 self.append(" "), inserting a space between "-" and "0.0".
- So the bug is that is_word_char incorrectly classifies the minus sign "-" as a word character, triggering an unwanted space when a sign and a number are added in two calls. The problematic call site is line 133 (and lines 136–138 where append(" ") and append(newcode) happen).

How to fix it (clear concrete steps)
1. Make is_word_char return True only for actual word characters (letters, digits, underscore), not for punctuation like "-". For example, change its test to something like:
   - return c.isalnum() or c == "_"
   (or the equivalent in your codebase) so that "-" is not treated as a word char.
2. Alternatively, if you cannot change is_word_char for other reasons, change the condition at line 133 to explicitly exclude "-" as a previous character:
   - replace self.is_word_char(self.get_last_char()) with something like:
       last = self.get_last_char()
       if last == "-" or last == "":
           last_is_word = False
       else:
           last_is_word = self.is_word_char(last)
     and use last_is_word in the if.
3. Add/adjust unit tests:
   - Keep the failing test cc.add_number(-0.0) to ensure no space is introduced.
   - Add a test where add("-",) then add("0.0") are called separately to prevent regressions.

Why these steps fix it
- The space insertion logic is intended to prevent token sticking when two consecutive word characters would blend (e.g., "returnfoo"). The minus sign is not a word character and should not trigger that spacing. Ensuring is_word_char does not treat "-" as a word character (or explicitly excluding "-") prevents the condition on line 133 from being true in this case, so no space gets inserted and you get "-0.0" as expected.
