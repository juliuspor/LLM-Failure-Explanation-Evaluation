What happened (root cause)
- At line 346 you compare inferred_type to object, but inferred_type is set on line 342 with inferred_type = type(array). For a Python list, type(array) is list, not the element type. That means inferred_type will almost always be list, never object, so the conditional logic that should detect an "object element type" never runs and the function makes the wrong kind of new list/array.
- The call new_list = ArrayUtils._copy_list_grow1(array, inferred_type) is therefore being given the wrong type: you pass the list class (list) instead of the element class. That causes the created array/list to have the wrong component type and leads to the ClassCast-like failure you detect (the raised TypeError at lines 346–351 is only reached in a different branch, so the real failure occurs earlier when the array is created or later when elements are assigned).
- Practically: type(array) returns <class 'list'>. You wanted the element class (e.g., int, str, MyClass), not the container class. Because of that mismatch the code creates/casts arrays incorrectly and errors out.

Where it occurs
- Line 342: inferred_type = type(array) is wrong when array is a list — this returns the list type.
- Line 344: inferred_type = type(element) is OK only when array is None.
- Line 346–351: The check inferred_type == object is never true when the inferred_type was set from the list; thus the specific TypeError you try to raise is not the primary protection point.

What goes wrong (concrete example)
- Suppose array = [MyClass()] and element = MyClass(). type(array) == list, so inferred_type becomes list.
- _copy_list_grow1(array, inferred_type) receives list instead of MyClass; it produces a list/array with wrong component type.
- Assigning element into new_list may fail or later code will behave as if the container contains the wrong element type.

How to fix it (step-by-step, exact changes)
1. Decide what "inferred_type" must represent: the element/component class, not the container class.
2. Replace inferred_type = type(array) with logic that picks the element type from the array contents (or falls back to expected_type or object if not available).

Minimal, correct replacement (exact code you can apply)
- Replace the beginning of the function with:

    if array is not None and len(array) > 0:
        inferred_type = type(array[0])
    elif element is not None:
        inferred_type = type(element)
    else:
        inferred_type = object

Notes and edge cases:
- If array can contain None elements, use the first non-None element to infer type:
    next_elem = next((x for x in array if x is not None), None)
    inferred_type = type(next_elem) if next_elem is not None else (type(element) if element is not None else object)
- If array can be empty and element is None but expected_type is provided, prefer expected_type as the component type:
    inferred_type = expected_type if expected_type is not None else object
- Make sure ArrayUtils._copy_list_grow1 expects and handles the element type (not the container type). If it expects a component class name/string, pass the component class accordingly.

Why these exact steps fix it
- Using type(array[0]) or the first non-None element yields the element class (e.g., MyClass) so _copy_list_grow1 gets the correct component type and the check inferred_type == object (line 346) will behave as intended when the component type truly is object.

Summary (short)
- The bug is that inferred_type is set to type(array) (the list class) instead of the element/component class. Change the inference to use an element from the list (or expected_type) so the rest of the logic and array creation/casting work correctly.
