What happened (root cause)
- In ArrayUtils.add the code decides the element type in a variable named inferred_type.
- The branch that sets inferred_type runs only when element is not None (you can see that on line 337: "elif element is not None:" and line 340: "inferred_type = object").
- When you call ArrayUtils.add(None, None, expected_type=str) both old/target and element are None, so the "element is not None" branch is skipped. inferred_type is left at the default value object.
- Immediately afterwards (lines 346–347) there is this check:
  if inferred_type == object and expected_type != object:
      raise TypeError(...)
  Because inferred_type stayed object and expected_type is str (not object), the code raises TypeError and the test fails.

Where it fails (exact lines and variables)
- The failure originates between lines 337–347.
  - element is the argument passed as None, so line 337's "elif element is not None:" block is not taken.
  - inferred_type remains object (the default).
  - At line 346 the code checks inferred_type == object and expected_type != object.
  - At line 347 it raises TypeError because expected_type is str.
- Variables involved: element (None), inferred_type (object), expected_type (str).

Why that is wrong for your test
- Your test expects adding a None element to a collection typed as str to produce [None].
- The code treats an unknown inferred_type (object) as incompatible with a specified expected_type and rejects the operation, rather than allowing None as a valid value for the expected_type.

Concrete fixes (specific steps and places to change)
Option A — Use expected_type when element is None
- Before the type check at line 346, set inferred_type to expected_type when element is None and expected_type is specified:
  - Insert around line 345:
      if element is None and expected_type != object:
          inferred_type = expected_type
  - This makes the later check pass, letting None be added for a str-typed array.

Option B — Allow None values explicitly
- Change the TypeError condition at line 346 to skip raising when the element is None:
    if inferred_type == object and expected_type != object and element is not None:
        raise TypeError(...)
- This allows adding None regardless of inferred_type, which matches your test expectation.

Option C — Treat None as having no type but allow it when expected_type is set
- If you want to be explicit, add a branch earlier (around line 337) that handles element is None:
    if element is None:
        # accept None if expected_type is provided or if array element type allows None
        if expected_type == object:
            inferred_type = object
        else:
            inferred_type = expected_type
- Then the existing check at line 346 will not raise.

Which to pick
- If the intent is "None is allowed as a value for any expected element type" pick Option B.
- If the intent is "when element is None, use the caller-provided expected_type as the inferred type" pick Option A or C.

Summary
The code raises TypeError because inferred_type stays object when element is None, and the later check rejects object vs. the caller's expected_type (str). Fix by setting inferred_type from expected_type when element is None or by skipping the TypeError for None elements (see concrete code snippets above).
