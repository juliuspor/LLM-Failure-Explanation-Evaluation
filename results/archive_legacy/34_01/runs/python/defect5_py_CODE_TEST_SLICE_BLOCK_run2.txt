What happened (where it fails)
- In add(...), when both array and element are None the code sets inferred_type = object (line where inferred_type is assigned in the else branch).
- Later, at the check on line 346 you test if inferred_type == object and expected_type != object, and if true you raise a TypeError.
- In your test call ArrayUtils.add(None, None, expected_type=str) inferred_type is object and expected_type is str, so the condition is true and the function raises the TypeError instead of returning [None]. That is why the test fails.

Why this is the root cause (variables and lines)
- inferred_type is decided at:
  - if array is not None: inferred_type = type(array)
  - elif element is not None: inferred_type = type(element)
  - else: inferred_type = object  <-- this branch runs for add(None, None, expected_type=str)
- The raising logic at line 346 uses inferred_type and expected_type:
  - if inferred_type == object and expected_type != object: raise TypeError(...)
- The intent appears to be to prevent silently casting a list whose element type really is object to a list of a more specific expected_type. But when both array and element are None there is no real inferred element type yet — using object here and then immediately treating it as an illegal cast is wrong.

How to fix it (concrete steps)
Choose one of these simple fixes depending on intended behavior:

1) If expected_type should determine the list element type when both array and element are None, set inferred_type from expected_type:
   - Replace the else branch with:
     inferred_type = expected_type if expected_type is not None else object
   - That way add(None, None, expected_type=str) will use str as inferred_type and not raise.

2) If you want to allow None values when no real type can be inferred (i.e., don't raise when both array and element are None), change the raise condition to ensure element or array actually determined the type:
   - Replace the if on line 346 with something like:
     if inferred_type == object and expected_type is not None and (array is not None or element is not None) and expected_type != object:
         raise TypeError(...)
   - Or simpler: only raise when inferred_type came from array or element (not from the default object case for both None).

3) If you want to require expected_type to be used whenever provided, explicitly set inferred_type = expected_type at the start when expected_type is provided:
   - At top of function after parameters:
     if expected_type is not None:
         inferred_type = expected_type
     else:
         (existing inference logic)

Pick the change that matches your intended semantic. The current code raises because inferred_type is set to object when nothing can be inferred, and the code treats that as an error when expected_type is more specific.
