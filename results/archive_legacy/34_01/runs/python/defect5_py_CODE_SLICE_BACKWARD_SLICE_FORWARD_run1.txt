What went wrong (root cause)
- On line 337 you set inferred_type = type(array). That returns the Python type of the container (e.g. list), not the element type stored inside the list.
- ArrayUtils._copy_list_grow1 is being called with inferred_type (line 342). The copy/grow implementation expects the element class/type, not the container class. Passing the container type means the new list is created with the wrong underlying element type information.
- Later, on lines 346–347 you check if inferred_type == object to decide whether to raise a TypeError. Because inferred_type is the list class (e.g. <class 'list'>) when array was passed, that equality check never matches even when you intended to treat it as an "unknown element type". That makes the branch for the TypeError incorrect and allows invalid casts to proceed or misreports the error.
- Concretely: when array is not None you end up using inferred_type == list (or whatever the container class is). That breaks both the copy/grow behavior and the runtime type-check logic that follows.

Why that causes the failure (plain)
- You should be telling _copy_list_grow1 the element type (the type of items inside the list), not the list's class. Passing the list class confuses the copy/grow routine and the later cast check, so either you create an array that can't accept the new element or you don't raise the expected TypeError when you should.

   - Else if element is not None: use type(element).
   - Else if expected_type is not None: use expected_type.

- The condition inferred_type == object will correctly detect the genuinely unknown-element-type case, allowing the TypeError branch to behave as intended.

If your lists can be heterogenous (mixed element types) and you need a deterministic element type choice, pick a policy (first element's type, element argument's type, or expected_type) and document it.
