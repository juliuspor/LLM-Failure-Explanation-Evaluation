What happened
- The test calls cc.add_number(-0.0) and expects cc.get_code() to return the string '-0.0'.
- The assertion failed because cc.get_code() returned '0' (a positive-looking zero) instead of '-0.0'.

Root cause (exact place to look)
- The bug is in MockCodeConsumer.add_number, specifically on the line that converts the passed float to a string and appends it to the internal buffer (for example a variable named self._parts, self._buffer, or self._code).
- That conversion is using a formatting path that collapses negative zero to the plain "0" (examples that do that: int(value), "{:g}".format(value), format(value, 'g'), or any formatter that trims sign for zero). Those formatters treat -0.0 and 0.0 as identical and produce "0" (or "0.0") rather than preserving the negative sign.

Why negative zero was lost
- IEEE-754 floating-point has a distinct negative zero value. Python's float preserves the sign internally, so -0.0 is a float that will stringify as '-0.0' with some conversions (repr(-0.0) and str(-0.0) preserve the negative sign).
- But certain formatting options (notably the general format 'g' and some custom logic that coerces to int) normalize -0.0 to 0 because they consider sign irrelevant for a zero value. If add_number uses one of those, the negative sign is dropped and you get "0" in the buffer, which makes the test fail.

How to find the exact line
1. Open MockCodeConsumer.add_number. Find the line that turns the incoming number into a string. It will look like one of:
   - self._parts.append(str(value))
   - self._buffer += "{:g}".format(value)
   - s = format(value, 'g')
   - self._code.append(int(value)) or s = str(int(value))
2. That line is where -0.0 is turned into "0". That's the line to change.
