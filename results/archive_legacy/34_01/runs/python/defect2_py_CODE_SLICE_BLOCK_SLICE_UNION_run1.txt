What happened (where and what went wrong)
- At line 71 you compute g using the original parameter value:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- But at lines 54–59 you already clamped value into the local variable v:
  v = max(value, self._lower_bound)
  v = min(v, self._upper_bound)
- Because the formula uses value (not the clamped v), g can be outside 0–255 even though you intended to clamp. That causes the check at line 74–75 to raise ValueError:
  if g < 0 or g > 255:
      raise ValueError(...)

Why it happens (root cause)
- You intended to clamp the input before mapping to 0–255, but you forgot to use the clamped variable v in the mapping formula. The mapping therefore still uses the original, unclamped value and can produce values below 0 or above 255.
- There is also a secondary failure mode: if self._upper_bound == self._lower_bound the division will raise ZeroDivisionError. That is not handled here.

Exact fixes (step-by-step)
1. Use the clamped value v in the mapping (change line 71):
   g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
2. Protect against zero range (add a check before computing g):
   if self._upper_bound == self._lower_bound:
       g = 0  # or 255, or choose desired behavior for a zero span
   else:
       g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
3. Ensure final g is within [0, 255] to avoid floating-point rounding pushing it outside:
   g = max(0, min(255, int(round(g))))
   (or clamp after int: g = max(0, min(255, g)))
4. Improve the error message or remove the ValueError if you clamp instead:
   if g < 0 or g > 255:
       raise ValueError(f"Color parameter outside of expected range: g={g}, lower={self._lower_bound}, upper={self._upper_bound}")
