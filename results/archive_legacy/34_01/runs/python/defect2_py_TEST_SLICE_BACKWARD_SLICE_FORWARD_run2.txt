What happened (where)
- In GrayPaintScale.get_paint, at line 71 you compute g:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- For your test call value = -0.5 this produces a negative g (for typical bounds like _lower_bound=0.0, _upper_bound=1.0 g becomes int(-127.5) → -127).
- Immediately after, lines 74–75 check:
  if g < 0 or g > 255:
      raise ValueError(...)
  That raises a ValueError instead of returning (0, 0, 0). The test expects the function to clamp out-of-range inputs to black, but the implementation rejects them.

Why it fails (root cause)
- The code unconditionally raises when the computed channel value is outside [0,255] (line 74), so negative inputs (value < _lower_bound) trigger an exception instead of being clamped to 0.
- Converting the floating result to int before range handling (line 71) is not the root issue by itself, but it makes the check at lines 74–75 detect out-of-range values that should often be clamped.

Exactly where to change it
- Modify get_paint so it clamps the computed channel(s) into [0,255] and then returns (g, g, g) instead of raising for out-of-range colors.
- Example patch (replace lines 71–75):

  # compute floating gray level
  g_float = (value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0
  # clamp into [0.0, 255.0]
  g_float = max(0.0, min(255.0, g_float))
  # convert to int after clamping
  g = int(g_float)

- Or, if you prefer a one-liner:

  g = int(max(0, min(255, (value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)))

Two alternative fixes (choose one)
1. Clamp and return black/white for out-of-range values (recommended if tests expect clamping):
   - Use the code above, then return (g, g, g).

2. If out-of-range input should be an error, change the test to expect a ValueError:
   - Use pytest.raises(ValueError) or update assertion to match intended behavior.

Why this fix works
- Clamping prevents negative or >255 channel values so the function returns a valid RGB tuple (0..255 each) rather than raising an exception. That matches the test expectation assert color == (0, 0, 0) for -0.5.

Small note about the error message
- The raised ValueError currently uses a fixed string "Color parameter outside of expected range: Red Green Blue" with no numeric context. If you keep the exception behavior, include the numeric g and value in the message for better debugging.
