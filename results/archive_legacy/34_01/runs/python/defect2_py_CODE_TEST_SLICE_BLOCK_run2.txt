What happened
- In get_paint you clamp the input into v:
  v = max(value, self._lower_bound)
  v = min(v, self._upper_bound)
- But when you compute g you use the original value instead of the clamped v:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- For the test value -0.5 (and typical bounds lower_bound = 0.0, upper_bound = 1.0) that produces a negative g (≈ -127), so the if check at lines 74–75 raises ValueError.

Why it fails (root cause)
- The clamp is calculated into variable v but never used in the mapping formula. Using value allows numbers outside the bounds to produce g outside 0–255 and trigger the error at lines 74–75.

Where it occurs
- Method: get_paint
- Variables: value (original), v (clamped), g (computed color component)
- Faulty line: the g assignment (uses value instead of v)
- Failure check: the if at lines 74–75 raises ValueError when g < 0 or g > 255

How to fix (clear steps)
1. Use the clamped v in the formula instead of value.
2. Protect against a zero denominator (upper_bound == lower_bound).
3. (Optional) Clamp g into [0,255] as a final safeguard.

Concrete replacement (minimal change)
Replace the g assignment with:
    denom = (self._upper_bound - self._lower_bound)
    if denom == 0:
        g = 0
    else:
        g = int((v - self._lower_bound) / denom * 255.0)

Or more compactly:
    denom = (self._upper_bound - self._lower_bound) or 1.0
    g = int((v - self._lower_bound) / denom * 255.0)

And you can keep the existing range check or replace it with a clamp:
    g = max(0, min(255, g))

After these changes, get_paint(-0.5) will produce (0, 0, 0) as your test expects.
