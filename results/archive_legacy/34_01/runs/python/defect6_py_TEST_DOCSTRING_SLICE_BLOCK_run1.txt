What happened (what you saw)
- The test expected the exact string '-0.0'.
- The code produced a string with an extra space between the minus and the digits (for example "- 0.0"), so the assertion failed.

Where it happened (exact place)
- The unwanted space is inserted in add() at lines 133–136:
  - Line 132: c = newcode[0]
  - Line 133: if (self.is_word_char(c) or c == '\\') and self.is_word_char(self.get_last_char()):
  - Line 136:     self.append(" ")

Why it happened (root cause)
- add() decides to insert a separating space when the new token starts with a "word" character and the previously appended character is also a "word" character.
- In this case add_number() produced the number in two pieces: first the '-' sign, then the digits "0.0". When add() is called to append the second piece, newcode[0] is '0' (a word char), and get_last_char() returns the previously appended '-'.
- is_word_char(self.get_last_char()) returned True for '-' (incorrectly treating '-' as a word character). Because both sides of the condition were true, add() appended a space at line 136, producing "- 0.0" instead of "-0.0".

Why that is the real bug
- The decision to add a space should only happen when both the previous char and the new token's first char are true "word" characters (letters, digits, underscore, etc.). A minus sign '-' is not a word character and should not make add() insert a separating space before digits.
- So the incorrect behavior comes from is_word_char treating '-' as a word char (or otherwise returning True for the last char when it shouldn't), triggering the space insertion at lines 133–136.
