What happened (root cause)
- At line 337 you enter the branch for "elif element is not None" (i.e. you do have an element).
- At line 340 you set inferred_type = object. That is wrong: when element is not None you should infer the element's actual type, not object.
- Later, at line 346 you test if inferred_type == object and expected_type != object, and at line 347 you raise TypeError because you think the element's type is unknown (object) while an expected_type was supplied.
- Because inferred_type was incorrectly set to object even though element is present, the function raises TypeError incorrectly whenever an expected_type other than object is given.

Why that causes failure
- The code treats a present element as if its type were unknown. That triggers the "unknown type vs expected type" check and raises TypeError even when the element actually matches expected_type.
- The bug is located in the branch guarded by "elif element is not None" (around lines 337–347). The wrong assignment is at line 340; the wrong exception check is at lines 346–347.

How to fix it (concrete steps)
1. In the branch where element is not None (the block that starts at the line marked 337), set inferred_type to the element's runtime type:
   - Replace inferred_type = object (line 340) with inferred_type = type(element)
2. Keep the existing check at lines 346–347 (the TypeError raise), which will now only fire when the element is actually of unknown/ambiguous type (object) while an expected_type was supplied.
3. If you have a separate branch for element is None, keep inferred_type = object there (that branch should remain as the only place that sets inferred_type to object).

Minimal code patch example
- Change from:
  inferred_type = object
- To:
  inferred_type = type(element)

What to verify after the fix
- Call the function with element not None and expected_type matching type(element): should not raise.
- Call with element None and expected_type not object: should still raise (if that was intended).
- Call with element not None but expected_type incompatible: should raise TypeError.

Why this explanation is definitive
- The wrong value assignment (inferred_type = object) is the single point that causes the later TypeError check to misfire. Fixing that one line corrects the type inference logic and prevents the spurious TypeError.
