What happened (root cause)
- In to_utf16_code_units(s: str) the code checks the validity of the variable named index like this (lines 35–36):
  if index < 0 or index >= len(code_units):
      raise IndexError(f"String index out of range: {index}")
- index is an index into the input string s, but the code is comparing it to len(code_units) instead of len(s).
- Because code_units starts empty (code_units = [] at line 12) or is shorter than s while you still need to read s, index >= len(code_units) becomes true and the function raises IndexError for a perfectly valid string index (for example index == 2).

Where it happened
- Function: to_utf16_code_units
- File lines shown: the check at lines 35–36 causes the raise; code_units is defined at line 12.
- The erroneous comparison is the immediate cause.

Why that causes the error
- The intent is to ensure index is within the bounds of the input string s. Comparing index to len(code_units) tests against the wrong list (the growing output) rather than the input string length, so valid string indices appear out of range whenever code_units is shorter than s.

How to fix it (explicit steps)
1. Replace len(code_units) with len(s) in the boundary check. For example:
   if index < 0 or index >= len(s):
       raise IndexError(f"String index out of range: {index}")
2. Confirm that index is indeed meant to index into s. If index is supposed to index code_units for some reason, rename variables to avoid confusion (e.g., input_index vs output_index) and keep separate bounds checks for each.
3. Run tests covering:
   - small strings (length < code_units growth)
   - empty string s == ""
   - negative index
   to ensure the corrected check behaves as expected.
