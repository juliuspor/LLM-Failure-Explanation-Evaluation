Why this is wrong (root cause)
- inferred_type becomes object in the common case where there is no concrete element type to infer. That happens for at least two scenarios:
  1. The input list (array) is None and the new element is None. According to the docstring the function should return a one-element List[object] in that case — not throw.
  2. The input list exists but all elements (and the new element) are None, so there is nothing to infer beyond object.
- The current code treats "inferred_type == object" as an error any time expected_type is more specific. But object here is the absence of a concrete element type, not a real incompatible element type. Raising a cast error in that situation is incorrect: there is nothing concrete to cast, only an unknown / None element. The code therefore throws a TypeError at lines 346–351 even though the correct behaviour (per the docstring) would be to accept None as compatible with any expected_type or to return a List[object] when the element is None and no array type exists.

Where it occurs
- The exception is raised at lines 346–351, i.e. inside the block:

        raise TypeError(...)

- The variables involved are inferred_type (the type the function inferred from array/element) and expected_type (the caller-provided expected element type). The function incorrectly uses inferred_type == object as evidence of an incompatible list instead of as "no information".

   - Only raise if there is a concrete type mismatch between inferred_type (derived from non-None elements) and expected_type.
