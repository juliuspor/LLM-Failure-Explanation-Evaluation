What happened
- In GrayPaintScale.get_paint you pass RGB values that are negative when value is negative. For value = -0.5 the code computes an intensity below 0, produces an rgb triple like (-127, -127, -127) (or otherwise passes negative components) and then calls whatever color constructor/validator you use. That validator rejects components outside 0–255 and raises ValueError: "Color parameter outside of expected range: Red Green Blue" before get_paint can return (0, 0, 0). The assertion never runs because the exception is raised inside get_paint.

Why it happened (root cause, tied to names in your code)
- Variable: value = -0.5 (from the test).
- Method: GrayPaintScale.get_paint computes a gray intensity from value (e.g. intensity = int(value * 255) or similar) and then builds rgb = (intensity, intensity, intensity).
- The rgb tuple contains negative numbers (e.g. intensity = -127).
- The code then either constructs a color object or passes rgb to a validator (the line that creates/validates the color inside get_paint). That validation enforces each channel be 0..255 and raises the ValueError you see.

Where it occurred
- Inside GrayPaintScale.get_paint, at the line where you convert the scaled/rounded intensity into an rgb triple or where you hand rgb to the color constructor/validator. (This is the line that executes after computing intensity and before returning the result.)

How to fix it (concrete steps)
1. Clamp the input value into the allowed range before scaling. For example:
   - clamped = max(0.0, min(1.0, value))
   This ensures negative values become 0 and values >1 become 1.

2. Convert clamped to an integer 0–255 safely:
   - intensity = int(round(clamped * 255))
   - rgb = (intensity, intensity, intensity)

3. Return rgb directly (or pass it to your color constructor if required). Example replacement body for get_paint:
   - clamped = max(0.0, min(1.0, value))
   - intensity = int(round(clamped * 255))
   - return (intensity, intensity, intensity)

4. (Optional) If you prefer to keep raising for out-of-range inputs, validate value at the start and raise a clear ValueError:
   - if not (0.0 <= value <= 1.0):
         raise ValueError(f"value {value} out of range 0.0..1.0")
   This makes the failure explicit and avoids the confusing color-validator error.

Why these steps fix it
- Clamping prevents negative intensities and keeps each channel within 0–255, so the validator no longer raises the ValueError and get_paint can return the expected (0, 0, 0) for value = -0.5.
