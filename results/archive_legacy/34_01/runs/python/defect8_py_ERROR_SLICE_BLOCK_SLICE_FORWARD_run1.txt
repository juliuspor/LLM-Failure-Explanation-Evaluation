What happened (where it failed)
- The ValueError is raised at line 240: the code checks ch3 and ch4 and does
  raise ValueError(f"Invalid locale format: {locale_str}") when they are not uppercase A–Z.
- The failing input is "fr__POSIX". At index positions:
  - locale_str[0:2] = "fr"
  - locale_str[2] = '_' (the separator)
  - locale_str[3] = '_' (there is an empty country code, so ch3 is '_')
  - locale_str[4] = 'P'
- Because ch3 is '_' (not between 'A' and 'Z'), the check on line 239–240 fails and the ValueError is thrown with the message shown.

Why it failed (root cause)
- The code assumes that after the initial language and '_' there will always be a two-letter uppercase country code (chars ch3 and ch4).
- The input "fr__POSIX" uses the established POSIX form where the country code is empty and a variant follows: language + "__" + variant. The code does not recognize that pattern and treats the second underscore as an invalid country-letter, so it rejects the whole string.

Exactly where to change it (specific variables/lines)
- Lines involved: 239–247. The offending check is the guard that tests ch3 and ch4 are in 'A'..'Z' and raises on failure (line 240).
- The variant-with-empty-country case is not handled; the code later expects variant to start at index 6 (line 247) for language_country_variant, but for "fr__POSIX" the variant actually starts at index 4.
