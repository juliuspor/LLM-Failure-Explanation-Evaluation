What happened (where, when)
- In get_paint the code clamps the incoming value into local variable v (lines 3–5):
  v = max(value, self._lower_bound)
  v = min(v, self._upper_bound)
- Immediately after that the code calculates g using the original value (line 7):
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- Because the test calls get_paint(-0.5) and the lower bound is higher than -0.5, g becomes negative and the subsequent range check raises ValueError. That is why gps.get_paint(-0.5) does not return (0,0,0) but instead raises.

Why it fails (root cause)
- The clamped variable v is never used when computing g. The code clamps value into v but then computes g from the un-clamped value variable. So the clamping has no effect and out-of-range inputs still produce out-of-range g (and trigger the ValueError).

Exact location to change
- Replace the line that computes g to use v instead of value:
  g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

Additional edge case to handle
- If self._upper_bound == self._lower_bound you'll divide by zero. Add a guard, for example treat that case as middle (or return 0 or 255 as appropriate).

Concrete corrected function
- Minimal corrected version:

    def get_paint(self, value: float) -> Tuple[int, int, int]:
        # Clamp value to bounds
        v = max(value, self._lower_bound)
        v = min(v, self._upper_bound)
        
        # Use the clamped value v, not the original value
        denom = (self._upper_bound - self._lower_bound)
        if denom == 0:
            g = 0  # or choose an appropriate fallback
        else:
            g = int((v - self._lower_bound) / denom * 255.0)
        
        if g < 0 or g > 255:
            raise ValueError(f"Color parameter outside of expected range: {g}")
        
        return (g, g, g)

That change fixes the failing test gps.get_paint(-0.5) returning (0,0,0).
