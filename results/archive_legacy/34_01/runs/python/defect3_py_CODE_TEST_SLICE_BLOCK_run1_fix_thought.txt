The bug arises from a mismatch between what translate_codepoint returns and how translate() advances the position. Translator implementations return the number of UTF-16 code units consumed, but translate() treated the return as a count of codepoints and iterated calling code_point_at(), which can index past the end. The simplest consistent fix is to advance pos by the consumed number of UTF-16 code units when consumed != 0. I will return the updated translate implementation that makes this change and preserves previous behaviors for None and validation.
