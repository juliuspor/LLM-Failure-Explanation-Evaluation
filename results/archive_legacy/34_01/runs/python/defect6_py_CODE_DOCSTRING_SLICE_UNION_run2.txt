What happens
- The code can enter an infinite loop in the while loop inside add_number. That makes the program hang (and can prevent further calls to add or append).
- The infinite loop happens when value == 0 (i.e. x == 0.0). In that case mantissa starts as 0, mantissa % 10 == 0 is true, and check_val is computed as 0 every iteration, so the loop body always runs and never changes the condition in a way that breaks the loop.

Where it occurs (exact spot)
- In add_number, inside:
  while mantissa % 10 == 0:
    check_val = (mantissa // 10) * (10 ** (exp + 1))
    if check_val == value:
      mantissa //= 10
      exp += 1
    else:
      break
- For x == 0.0, value == int(x) == 0 and mantissa == 0, so:
  - mantissa % 10 == 0 is true,
  - check_val == 0 so check_val == value is true,
  - mantissa //= 10 leaves mantissa == 0, exp increments,
  - repeat forever.

Why this is the root cause (plain language)
- The loop expects mantissa to shrink toward a non-multiple-of-10 value, but when mantissa is 0 it never changes to break the loop; exp keeps increasing and the loop never exits. The condition mantissa % 10 == 0 is satisfied forever for mantissa == 0.

How to reproduce quickly
- Call add_number(0.0) or add_number(0). The program will hang inside add_number.

Concrete fixes (step-by-step)
1) Simple guard for zero before the loop:
   - After value = int(x), add:
     if value == 0:
         self.add("0")
         return
   This prevents entering the while loop when the integer value is zero.

2) Or tighten the while condition to avoid the mantissa == 0 case:
   - Replace
       while mantissa % 10 == 0:
     with
       while mantissa != 0 and mantissa % 10 == 0:
   This stops the loop when mantissa becomes 0.

3) (Optional) Prefer an absolute check if negative values are possible:
   - Use while abs(mantissa) != 0 and mantissa % 10 == 0: if you want to be explicit about sign handling (not required for the zero bug).

One of (1) or (2) is sufficient. Example patch (minimal):
    if value == 0:
        self.add("0")
        return
    # ... existing code follows

Why these fixes are safe
- They prevent the infinite loop for x == 0 while preserving the intended trimming of trailing zeros for other integers (e.g., 1000 -> mantissa 1, exp 3).
