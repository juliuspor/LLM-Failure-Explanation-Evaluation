What happened
- The test calls GrayPaintScale().get_paint(-0.5) and expects black: (0, 0, 0).
- The implementation of GrayPaintScale.get_paint instead produces RGB values outside the 0–255 range for negative input and raises ValueError (per the docstring) or returns a negative channel tuple. That makes the assertion fail.

Why it happened (root cause)
- The code that converts value -> RGB does a direct scale like r = int(value * 255) (or equivalent) without clamping the input to the valid domain [0, 1]. For value = -0.5 that yields a negative channel (e.g. -127) which either:
  - triggers the method’s ValueError check (because the computed channel is outside 0–255), or
  - returns a tuple with negative components, causing the assert color == (0, 0, 0) to fail.
- In short: get_paint enforces/returns raw scaled RGB directly from value instead of clamping negative inputs to the minimum grayscale (0).

Where it occurs
- In GrayPaintScale.get_paint, in the mapping/calculation block that computes the channel value(s) from the input value (the lines that compute r/g/b from value, e.g. r = int(value * 255) or similar). This is the code path documented by the docstring "Returns a paint (RGB tuple) for the specified value" and the Raise behavior in the docstring.

Minimal example patch (clamping approach)

        v = max(0.0, min(1.0, float(value)))

Why this fix resolves the failure
- Clamping prevents negative values from generating negative channel components or triggering the method’s "outside 0–255" check. For value = -0.5, v becomes 0.0, k becomes 0, and get_paint returns (0, 0, 0), making the test pass.
