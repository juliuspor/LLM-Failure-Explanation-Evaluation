What happened (root cause)
- In get_paint you clamp the incoming value into variable v on lines 2–4:
  v = max(value, self._lower_bound)
  v = min(v, self._upper_bound)
- But on line 6 you compute g using the original value variable instead of the clamped v:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- As a result, when value is below self._lower_bound (your test uses value=-0.5), the subtraction (value - self._lower_bound) is negative. That produces a negative g, which then fails the range check and raises ValueError. The clamping you did has no effect because g is computed from value, not v.

Where it occurred
- In the get_paint method:
  - clamping lines (creating v): lines 2–4 of the snippet
  - incorrect use of value in the grayscale calculation: the line that assigns g
  - the ValueError is raised on the range-check immediately after that g calculation

Why it fails for your test
- Test: gps.get_paint(-0.5)
- Suppose self._lower_bound == 0.0. After clamping v becomes 0.0, but you compute g from value (-0.5) so g < 0 and the function raises ValueError instead of returning (0,0,0).

Also note a second potential failure (edge case)
- If self._upper_bound == self._lower_bound you will get a division-by-zero in the g calculation. You should guard against that.

Concrete fixes (step-by-step)
1. Use the clamped variable v when computing g:
   g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
2. Add a guard against zero range (upper == lower) before computing g:
   if self._upper_bound == self._lower_bound:
       raise ValueError("upper_bound and lower_bound must differ")
3. (Optional) Improve the ValueError message to include the actual g or bounds for easier debugging.

Corrected version
Replace the g assignment and add the guard. For example:

    def get_paint(self, value: float) -> Tuple[int, int, int]:
        # Clamp value to bounds
        v = max(value, self._lower_bound)
        v = min(v, self._upper_bound)
        
        # Guard against division by zero
        if self._upper_bound == self._lower_bound:
            raise ValueError("upper_bound and lower_bound must differ")
        
        # Use the clamped value 'v' here
        g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
        
        if g < 0 or g > 255:
            raise ValueError(f"Color parameter outside of expected range: {g}")
        
        return (g, g, g)

How this fixes your test
- For value=-0.5 and a lower_bound of 0.0, v becomes 0.0, so g becomes 0 and the method returns (0, 0, 0) as your assertion expects.
