What happened (root cause)
- The code treats -0.0 as an “integer” because the condition if x == int(x) is True for x == -0.0.
- You then do value = int(x), which produces 0 and loses the negative sign (int(-0.0) == 0).
- Because the integer branch runs, you call self.add(str(value)) and add "0" (no sign, no decimal), so the output becomes "0" instead of "-0.0".
- The earlier sign-handling (if x < 0 and prev == '-') does not help because in Python -0.0 < 0 is False (so that branch is skipped).

Where it occurs in the code
- The integer-detection line: if x == int(x) and not math.isinf(x) and not math.isnan(x):
- The sign-loss line: value = int(x) which later leads to self.add(str(value))

Why that comparison behaves badly for -0.0
- IEEE-754 floats represent negative zero; numeric equality x == 0.0 is True for -0.0, so x == int(x) is True.
- int(-0.0) converts to 0 (no sign information), so you lose the sign.

Concrete fixes (clear steps)
Option A — special-case negative zero early (recommended, simplest):
1. At the top of add_number, detect negative zero using math.copysign:
   is_neg_zero = (x == 0.0 and math.copysign(1.0, x) < 0)
2. If is_neg_zero:
   - Add the intended text directly and return, e.g.
     self.add("-0.0")
     return
This guarantees -0.0 is printed exactly as you expect and avoids the integer branch.

Option B — keep zero sign through integer branch:
1. Compute is_neg_zero as above before converting to int:
   is_neg_zero = (x == 0.0 and math.copysign(1.0, x) < 0)
2. Change the integer-branch condition to exclude negative zero:
   if x == int(x) and not math.isinf(x) and not math.isnan(x) and not is_neg_zero:
3. If you still end up in the integer branch for zero (positive zero), handle normally; for negative zero the code will fall through to the non-integer branch or be handled separately.

Minimal code example to insert at top of method:
    is_neg_zero = (x == 0.0 and math.copysign(1.0, x) < 0)
    if is_neg_zero:
        self.add("-0.0")
        return

Why this fixes it
- math.copysign(1.0, x) preserves the sign bit; for -0.0 it returns -1.0. That lets you detect negative zero and avoid converting it to int (which drops the sign).

No other behavior changes
- These changes only special-case the distinct -0.0 representation; all other numbers follow your existing formatting logic.
