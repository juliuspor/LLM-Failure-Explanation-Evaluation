What happened (where it fails)
- At line 54 you clamp the input into local variable v:
  v = max(value, self._lower_bound)
  v = min(v, self._upper_bound)
- But at line 71 you compute g using the original variable value, not the clamped v:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- Because you used value instead of v, a value below self._lower_bound (e.g. -0.5 in your test) produces a negative numerator and so g becomes negative. That triggers the ValueError at line 74 instead of returning (0, 0, 0). The test fails for exactly this reason.

Why this is the root cause
- The purpose of the first two lines is to clamp the input into the allowed range and then use that clamped value for computation. Using the un-clamped value bypasses that intention and produces out-of-range g. The error occurs specifically at the expression on line 71.

How to fix it (concrete steps)
1. Use the clamped variable v in the computation of g (replace value with v).
2. Also guard against a zero denominator (if upper and lower bounds are equal) to avoid ZeroDivisionError.
3. Optionally ensure g is clamped to [0, 255] after conversion to int to be defensive.

Exact corrected code snippet (replace lines 71–75):
    # compute using clamped v and protect against zero range
    range_span = self._upper_bound - self._lower_bound
    if range_span == 0:
        g = 0
    else:
        g = int((v - self._lower_bound) / range_span * 255.0)
    # clamp g to valid byte range
    g = max(0, min(255, g))

This change fixes the test: get_paint(-0.5) will use v (clamped to lower_bound) and return (0, 0, 0) instead of raising ValueError.
