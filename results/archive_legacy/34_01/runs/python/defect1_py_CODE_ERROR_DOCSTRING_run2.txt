What happened (where it failed)
- The failure happens on this validation line:
  if minutes_offset < 0 or minutes_offset > 59:
- The exception raised was ValueError: Minutes out of range: -15
- That means minutes_offset was -15 when the code reached that line (so the method raised because it rejects any negative minutes).

Why it failed (root cause)
- The docstring says minutes may be negative (range -59 to +59). The code, however, rejects any negative minutes because the minutes check uses < 0 instead of < -59.
- In other words, the intended allowed range is -59..+59, but the code enforces 0..59. So a valid input like (hours_offset = 0, minutes_offset = -15) (allowed per docstring) is rejected at that check.
- Additionally, the method intends to forbid combinations where hours is positive and minutes is negative (docstring states (+ve, -ve) should raise ValueError). That specific rule is not enforced at this point; instead the code incorrectly rejects all negative minutes rather than only those that conflict with a positive hour.

Where in the code to change (exact fixes)
1. Fix the minutes range check:
   - Replace
     if minutes_offset < 0 or minutes_offset > 59:
   - With
     if minutes_offset < -59 or minutes_offset > 59:
   This allows negative minutes within the documented range.

2. Enforce the "hours positive + minutes negative" prohibition:
   - Add, after the hours range check and before combining minutes:
     if hours_offset > 0 and minutes_offset < 0:
         raise ValueError(f"Minutes must be positive when hours is positive: {minutes_offset}")
   This implements the docstring rule that (positive hour, negative minute) is invalid.

3. Avoid reusing minutes_offset for total minutes (clarity / safety):
   - Instead of overwriting minutes_offset with the combined value:
       if hours_in_minutes < 0:
           minutes_offset = hours_in_minutes - minutes_offset
       else:
           minutes_offset = hours_in_minutes + minutes_offset
     create a new local variable, e.g. total_minutes, and assign that:
       if hours_in_minutes < 0:
           total_minutes = hours_in_minutes - minutes_offset
       else:
           total_minutes = hours_in_minutes + minutes_offset
     then multiply total_minutes to compute offset:
       offset = FieldUtils.safe_multiply(total_minutes, DateTimeConstants.MILLIS_PER_MINUTE)

Why these fixes solve it
- Changing the lower bound to -59 matches the documented allowed range and stops false positives like -15.
- Adding the explicit check for (hours > 0 and minutes < 0) enforces the documented invalid sign combination.
- Using a separate total_minutes variable avoids confusing the original minutes_offset and makes the code easier to read and reason about.

Optional note
- The try/except around FieldUtils.safe_multiply is fine to catch overflow, but be sure safe_multiply raises an ArithmeticError (or change to catch the actual exception type it raises).
