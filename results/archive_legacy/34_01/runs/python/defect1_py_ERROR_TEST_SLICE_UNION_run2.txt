What happened (exact place)
- At line 396 the code checks: if minutes_offset < 0 or minutes_offset > 59: raise ValueError(f"Minutes out of range: {minutes_offset}").
- Your test called DateTimeZone.for_offset_hours_minutes(-2, -15). minutes_offset is -15, so that check fails and the ValueError is raised.

Why it happens (root cause)
- The method validates minutes_offset for being in 0..59 before it applies the sign/normalization logic.
- The sign-handling/combining logic that should accept a negative minute when hours are negative appears later (lines 402–405):
  - line 402 and 404 adjust minutes_offset depending on the sign of the hours,
  - line 405 computes offset = minutes_offset * MILLIS_PER_MINUTE.
- Because the negative-minute check runs first, a negative minute is rejected outright and never reaches the normalization code. So passing (-2, -15) fails even though the intended meaning is -2 hours and -15 minutes.

Where to change it (concrete)
You have two straightforward options:

1) Call site fix (quick)
- Change your test to pass a positive minute value and let the hour carry the sign:
  zone = DateTimeZone.for_offset_hours_minutes(-2, 15)
  This produces the intended '-02:15' id because the method expects minutes in 0..59.

2) Library fix (correct behavior)
- Move the sign/normalization logic so it runs before the minutes range check, or normalize minutes first and then validate the absolute minutes. Concretely:
  - Before line 396, compute a combined minutes value using the hour sign. Example logic:
    total_minutes = hours_offset * 60 + (minutes_offset if hours_offset >= 0 else -abs(minutes_offset))
  - Then validate abs(total_minutes % 60) is within 0..59 (or validate abs(minutes_part) <= 59).
  - Replace later uses of minutes_offset with total_minutes and compute offset = total_minutes * MILLIS_PER_MINUTE.
- This prevents rejecting negative minutes like -15 when the hour is negative.

Exactly which variables to change
- minutes_offset is the variable being validated at line 396 (causing the error).
- The normalization code at lines 402 and 404 should be applied to minutes_offset (or produce a new total_minutes) before the validation at line 396 or the validation should be applied to the normalized value.

Summary
- The method rejects negative minutes too early (line 396). Either pass a positive minutes argument (15) with a negative hour (-2), or fix the method so sign normalization happens before the minutes range check (use total_minutes = hours*60 +/- minutes then compute offset at line 405).
