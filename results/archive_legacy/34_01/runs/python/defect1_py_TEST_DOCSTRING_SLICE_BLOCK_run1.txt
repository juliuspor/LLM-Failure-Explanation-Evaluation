What happened (where)
- The call DateTimeZone.for_offset_hours_minutes(-2, -15) fails because of the validation at lines 396–397:
  - Line 396 checks: if minutes_offset < 0 or minutes_offset > 59:
  - Line 397 raises ValueError("Minutes out of range: {minutes_offset}")
- That check rejects any negative minutes_offset, regardless of hours_offset, so passing minutes_offset = -15 always triggers the ValueError.

Why it fails (root cause)
- The docstring explicitly allows negative minutes when the hours value is negative (example: (-2, -15) → -02:15).
- The implementation does not implement that sign rule; it enforces minutes to be non‑negative unconditionally.
- In other words, the validation only looks at minutes_offset in isolation and ignores the required relationship between hours_offset and minutes_offset.

      raise ValueError(f"Minutes out of range: {minutes_offset}")
  if minutes_offset < 0 and hours_offset > 0:
      raise ValueError("Minutes must be non-negative when hours_offset is positive")

     total_minutes = hours_offset * 60 + (minutes_offset if hours_offset >= 0 else -abs(minutes_offset))

2. Validate absolute ranges:
     if abs(minutes_offset) > 59:
