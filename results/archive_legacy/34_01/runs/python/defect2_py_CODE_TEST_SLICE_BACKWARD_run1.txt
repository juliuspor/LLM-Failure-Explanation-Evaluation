What happened (where it fails)
- At line 54 get_paint clamps the incoming value into variable v:
  - v = max(value, self._lower_bound)
  - v = min(v, self._upper_bound)
- At line 71 you calculate g using the original value variable, not the clamped v:
  - g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- If value is less than self._lower_bound (your test uses value = -0.5), that formula produces a negative g, which triggers the ValueError at line 74.
- So the code fails because the clamp is computed and stored in v but never used in the brightness calculation (you use value instead). The failure occurs in get_paint at the g calculation (line 71) and the subsequent range check (lines 74–75).

Why this caused the test to fail
- Your test calls gps.get_paint(-0.5) expecting (0,0,0).
- Because you used value instead of v in the formula, g becomes negative instead of 0 and the function raises ValueError instead of returning (0,0,0).

How to fix it (explicit steps)
1. Use the clamped variable v in the gray calculation:
   - Replace the line at 71 with:
     g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
2. Also guard against zero division if lower and upper bounds are equal:
   - If self._upper_bound == self._lower_bound, handle that case (for example, set g = 0 or 255 depending on convention).
3. Optionally, you can keep the subsequent range check, or clamp g after calculation to 0..255.

    v = max(value, self._lower_bound)
    v = min(v, self._upper_bound)
