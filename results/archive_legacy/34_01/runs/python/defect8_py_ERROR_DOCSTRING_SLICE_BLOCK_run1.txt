What happened (what failed)
- The call LocaleUtils.to_locale("fr__POSIX") raised ValueError("Invalid locale format: fr__POSIX").
- The check that raised the error is on lines 239–240:
  - it tests ch3 and ch4 are between 'A' and 'Z' and raises if not.
- For the input "fr__POSIX" the characters are:
  - index 0='f', 1='r', 2='_', 3='_', 4='P', ...
  - so ch3 is '_' (not an uppercase letter), the validation fails and the ValueError is raised.

Why it failed (root cause)
- The method strictly expects the country code to be exactly two uppercase letters at the positions it checks (ch3 and ch4).
- The string "fr__POSIX" uses an empty country (two underscores) and a variant "POSIX". Some locale formats allow language + empty country + variant (language + "__" + variant), but the current code does not accept an empty country: it treats the first underscore as the separator then insists the next two chars are uppercase letters, which is not true here (ch3 is '_').
- Thus the code rejects a valid-ish input pattern because the country-part validation is too strict and does not recognize the "language__variant" form.

- In LocaleUtils.to_locale, at the country-code validation, lines 239–240:

  - raise ValueError(f"Invalid locale format: {locale_str}")
