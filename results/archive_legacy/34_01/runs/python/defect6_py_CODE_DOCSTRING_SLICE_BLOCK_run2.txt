What happens
- The condition on the if at the start of add_number calls int(x) before checking whether x is infinite or NaN:
  if x == int(x) and not math.isinf(x) and not math.isnan(x):
- If x is math.inf, -math.inf, or math.nan, int(x) raises (OverflowError or ValueError) and your function crashes before the isinf/isnan checks run.
- That is because Python evaluates x == int(x) first (left-to-right), so the conversion is attempted even for non-finite floats.

Where it occurs
- In add_number, on the line beginning with:
  if x == int(x) and not math.isinf(x) and not math.isnan(x):
  The call to int(x) is the immediate cause of the exception when x is infinite or NaN.

Why it fails (concise)
- Ordering of operations: you test equality with int(x) before guarding against non-finite floats, so int(...) is called on inputs it cannot convert.

How to fix it (clear steps)
1. Check finiteness first, before converting to int. Example:
   if not math.isinf(x) and not math.isnan(x) and x == int(x):
       ...
   or better:
   if math.isfinite(x) and x == int(x):
       ...

2. Only call int(x) after you know x is finite:
   if math.isfinite(x) and x == int(x):
       value = int(x)
       ...

3. (Optional) Use math.isfinite(x) for a single check instead of two checks:
   if math.isfinite(x) and x == int(x):

Minimal corrected snippet
    if math.isfinite(x) and x == int(x):
        value = int(x)
        mantissa = value

That guarantees int(x) is only called for finite floats and prevents the OverflowError/ValueError when x is inf or NaN.
