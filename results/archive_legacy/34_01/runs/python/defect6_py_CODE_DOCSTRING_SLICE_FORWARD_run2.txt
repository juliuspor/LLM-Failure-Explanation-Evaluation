What happened (root cause)
- The loop that strips trailing zeros (starting at "while mantissa % 10 == 0:") never terminates when value == 0 (or when mantissa reaches 0).  
- For x == 0.0, value = int(x) == 0, mantissa starts at 0, mantissa % 10 == 0 is True, and inside the loop mantissa //= 10 leaves mantissa == 0. The loop condition stays True forever, so the code enters an infinite loop.

Where it occurs
- In the block that begins:
  if x == int(x) and not math.isinf(x) and not math.isnan(x):
    value = int(x)
    mantissa = value
    exp = 0
    if abs(x) >= 100:
        while mantissa % 10 == 0:
            ...
- Specifically the infinite loop is caused by mantissa becoming 0 and never changing, so the while condition (mantissa % 10 == 0) never becomes False.

Why this happens (brief)
- The loop assumes mantissa will shrink to a non-multiple-of-10, but 0 is a multiple of 10 and dividing 0 by 10 stays 0. The check inside the loop (check_val == value) will also be True for value == 0, so the code keeps reducing mantissa (stays 0) and incrementing exp forever.

How to fix it (concrete steps)
1. Handle zero explicitly before the loop:
   - If value == 0 (or mantissa == 0), skip the trimming loop and just add "0" (or format as needed).
2. Or change the loop condition so it stops once mantissa < 10:
   - Replace while mantissa % 10 == 0: with while mantissa >= 10 and mantissa % 10 == 0:
     - This prevents the loop from running when mantissa is 0..9.
3. Example minimal change (replace the if/while section):

    if abs(x) >= 100:
        # avoid infinite loop when value == 0
        while mantissa >= 10 and mantissa % 10 == 0:
            check_val = (mantissa // 10) * (10 ** (exp + 1))
            if check_val == value:
                mantissa //= 10
                exp += 1
            else:
                break

- Or explicitly handle zero above:

    if value == 0:
        self.add("0")
        return

What to expect after the fix
- The function will no longer hang for x == 0.0 (or integers that reduce to mantissa == 0) and will correctly either output "0" or perform the intended trailing-zero trimming for other integers.

Additional note
- The initial get_last_char() / negative-number spacing logic is unrelated to the infinite loop; the bug is entirely in the trailing-zero trimming loop.
