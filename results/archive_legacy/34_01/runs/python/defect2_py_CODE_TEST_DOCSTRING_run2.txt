What happened (root cause)
- You clamp the input into v on lines 3–4:
  v = max(value, self._lower_bound)
  v = min(v, self._upper_bound)
- But on line 6 you compute g using the original value, not the clamped v:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- If value is below self._lower_bound (your test uses value = -0.5), the unclamped expression produces a negative fraction, so g becomes negative and triggers the ValueError check on line 9.
- In short: the clamp is computed into v but never used. That is why gps.get_paint(-0.5) fails.

Exactly where it fails
- The bug is on the line that computes g (the line showing g = int(...)). The calculation uses value instead of v. The ValueError is raised immediately after that if g < 0 or g > 255.

Other issues you should address
1. Division by zero: if self._upper_bound == self._lower_bound the denominator is zero; the current code will raise a ZeroDivisionError.
2. Rounding/truncation: int(...) truncates toward zero. Depending on how you want to map endpoints, consider round(...) or clamping the final integer to [0,255].
3. Error message: the raised ValueError prints a generic string; include the offending g and bounds for easier debugging.

Concrete fixes (step-by-step)
1. Use the clamped v when computing g:
   g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
2. Handle equal bounds:
   if self._upper_bound == self._lower_bound:
       g = 0  # or 255 or some defined behavior
   else:
       compute g as above
3. Ensure g is within [0,255] after rounding/truncation:
   g = max(0, min(255, int(round(...))))
4. Improve the error message (if you keep the exception path) to include g and bounds.

Minimal corrected example
    def get_paint(self, value: float) -> Tuple[int, int, int]:
        v = max(value, self._lower_bound)
        v = min(v, self._upper_bound)

        if self._upper_bound == self._lower_bound:
            g = 0
        else:
            frac = (v - self._lower_bound) / (self._upper_bound - self._lower_bound)
            g = int(round(frac * 255.0))
            g = max(0, min(255, g))

        return (g, g, g)

This fixes the failing test (gps.get_paint(-0.5) will use the clamped value and return (0, 0, 0)) and addresses the other concerns above.
