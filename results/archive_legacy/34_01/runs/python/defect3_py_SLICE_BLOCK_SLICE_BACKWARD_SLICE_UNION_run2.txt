What fails
- The IndexError is raised at line 36:
  if index < 0 or index >= len(code_units):
      raise IndexError(f"String index out of range: {index}")
  That means len(code_units) <= index (or code_units is empty) when that check runs.

Why it happens (root cause, tied to your lines)
- At line 12 you create code_units = [] inside to_utf16_code_units(s).
- Later, at line 38 you immediately use code_units[index] (after the bounds check).
- The exception shows up because code_units has not been populated before you do the bounds check/use. In other words code_units is still an empty list (or shorter than you expect) at lines 35–39, so index is out of range.

How that usually happens (concrete, actionable possibilities)
- You never filled code_units: the routine that should append the UTF‑16 code units to code_units is missing, not executed, or incorrectly indented so the loop that populates code_units never runs.
- Or you built code_units in a different function (to_utf16_code_units), but when you later run the bounds check you are not using that returned list. For example:
  - calling to_utf16_code_units(s) but not assigning its return to a local variable named code_units, and then trying to use code_units — so code_units is empty or shadowed.
- Or you are mixing two different index spaces: the caller passed an index in Python code points (string character index) while this code expects an index in UTF‑16 code units (or vice versa). That mismatch can make index invalid for the code_units list produced by to_utf16_code_units.

Exact places to look and fix
1) If the population is missing:
   - Inspect the code between line 12 and line 35. There must be logic that appends values into code_units. If that logic is accidentaly removed or mis-indented, restore it so code_units is populated before the bounds check at line 35.

2) If you intended to use the return value of to_utf16_code_units(s):
   - Make sure you call and assign it, e.g.
     code_units = to_utf16_code_units(s)
     (then do the bounds check)
   - Don’t rely on a local variable inside to_utf16_code_units being visible elsewhere.

3) If the index semantics are mixed:
   - Decide whether index is meant to index Python code points (s[i]) or UTF‑16 code units.
   - If callers provide code point indexes, convert them to UTF‑16 index before comparing to len(code_units) (or operate on code points instead).
   - If callers provide UTF‑16 indexes, make sure you actually created the UTF‑16 code_units list and measured length against that.

Concrete minimal fixes
- Populate the list before the check:
  - Move/execute the code that fills code_units so that by line 35 it contains the UTF‑16 units.
- Or use the returned list:
  - Replace whatever call you have with:

      if index < 0 or index >= len(code_units): ...
- Or normalize index:
  - If index is a Python string index, convert s -> code_units and translate index into a code‑unit index, or operate consistently on s instead of code_units.

Summary
The code raises IndexError because len(code_units) is smaller than the index at the time of the check. That happens because code_units was never filled (or the wrong list/namespace was used), or because the caller uses a different indexing scheme than the code expects. Fix by ensuring code_units is actually populated before the check or by using the correct list/converted index (see the three concrete fixes above).
