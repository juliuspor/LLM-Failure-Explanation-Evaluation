What happened (where it fails)
- In get_paint you clamp the input to v on lines 3–4:
  v = max(value, self._lower_bound)
  v = min(v, self._upper_bound)
  but on line 6 you compute g from the original value, not from the clamped v:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- Because your test calls get_paint(-0.5) and value is below self._lower_bound, the expression (value - self._lower_bound) is negative and produces g < 0. That triggers the check on line 9 and raises the ValueError you saw.

Why it fails (root cause)
- The root cause is a simple variable mix-up: you intended to use the clamped value v when computing the gray level, but you used value instead. The clamped v prevents out-of-range intermediate values; using value allows a negative or >upper result to produce g outside 0..255, which then raises the error.

Exactly where to change (concrete fix)
- Replace the computation of g (line 6) to use v, and also guard against division by zero if lower and upper bounds are equal:
  old:
    g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
  new:
    if self._upper_bound == self._lower_bound:
        g = 0  # or 255, depending on desired behavior
    else:
        g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

Other small improvements (optional but recommended)
- Improve the ValueError message to include the actual g and bounds to make future debugging easier:
  raise ValueError(f"Color parameter outside expected range: g={g} (expected 0..255) for bounds {self._lower_bound}..{self._upper_bound}")
- Alternatively, clamp g to 0..255 instead of raising, depending on desired behavior.

How this fixes your test
- With g computed from v (the clamped value), get_paint(-0.5) will first clamp v to self._lower_bound, producing g == 0, and will no longer raise the ValueError. The assertion color == (0, 0, 0) will pass.
