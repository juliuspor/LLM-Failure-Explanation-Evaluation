What happened (where it failed)
- The code raises IndexError at line 35:

  raise IndexError(f"String index out of range: {index}")
  The error message shows index == 2 and len(code_units) <= 2, so the condition triggered and the exception was raised.

Why it happened (root cause)
- index is being treated inconsistently: sometimes it refers to a position in the original string (s), and sometimes it is used to index into code_units.
- code_units is built at line 12 from s, but its length can differ from len(s) because UTF-16 code units represent some Unicode scalar values with two code units (surrogate pairs). If index is an index into s (characters/code points) you must compare it to len(s). If index is an index into the code_units array you must ensure the caller supplied a code_units index, not a string index.
- The check at line 35 compares index against len(code_units) but the caller or other code is passing a string index. That mismatch causes the check to fail and raise "String index out of range" even when index is valid for s.

Exactly where to look
- Line 12: code_units = [] (code_units is populated from s)
- Line 35: the bound check using len(code_units)
- Line 38: high = code_units[index] (this assumes index is an index into code_units)
- Lines 39–41: index+1 < len(code_units) and reading the next code unit for a surrogate pair

How to fix it (concrete steps)
Pick one consistent model and apply one of these fixes:

A) If index is meant to be an index into the original string s
- Change the bounds check at line 35 to compare against len(s):
  if index < 0 or index >= len(s):
- When you need the corresponding code_unit index, convert the string index into the appropriate position in code_units before reading code_units[index]. That requires mapping characters to their code unit offsets (because some characters map to two code units). Implement a small function to compute the UTF-16 offset from the string index, or build code_units first and use a mapping array.

B) If index is supposed to index code_units
- Ensure callers pass the code_units index (not a string index). Update caller code to compute and pass the offset into code_units.
- Optionally, rename the parameter from index to code_unit_index to make intent clear.

Additional defensive improvements
- Improve the exception message to say which index space it expects (e.g., "code unit index out of range" vs "string index out of range").
- Add unit tests that exercise characters that map to surrogate pairs (e.g., emoji) so you catch mismatches between string and code_unit indexing.

Summary
The failure is caused by mixing two different index spaces: a string index vs. a UTF-16 code unit index. Fix by choosing which index you mean and making the bounds check and any caller code use that same index space (or convert between them explicitly).
