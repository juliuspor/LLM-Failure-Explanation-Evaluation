What happened (where it failed)
- The call LocaleUtils.to_locale('fr__POSIX') raised ValueError at the check on lines 237–240:
  - ch3 = locale_str[3]
  - ch4 = locale_str[4]
  - if ch3 < 'A' or ch3 > 'Z' or ch4 < 'A' or ch4 > 'Z':
      raise ValueError(f"Invalid locale format: {locale_str}")
- For the input 'fr__POSIX' the characters (by index) are:
  - 0='f', 1='r', 2='_', 3='_', 4='P', ...
- ch3 is '_' which is not between 'A' and 'Z', so the code raises the ValueError "Invalid locale format: fr__POSIX".

Why it happened (root cause)
- The method assumes that when a locale string has the form language_country_variant it will always have a two-letter country at indices 3 and 4. It therefore always checks indices 3 and 4 for uppercase letters.
- The input 'fr__POSIX' uses the valid JVM-style syntax for "language + empty country + variant": the double underscore means "empty country, then variant". In that case the country is empty and the variant begins at index 4 (after the second underscore).
- The code does not handle the double-underscore case, so it performs the country-letter check against '_' (index 3) and fails incorrectly.

   - If locale_str[2] == '_' and locale_str[3] == '_' (i.e. double underscore), parse as:
     - language = locale_str[0:2]
     - country = '' (empty)
     - variant = locale_str[4:]
   - Return Locale(language, '', variant) (and validate variant non-empty if needed).
2. Only after that branch, perform the existing two-letter country checks that inspect ch3/ch4.
