What happened
- The test calls cc.add_number(-0.0) on MockCodeConsumer and then checks result == '-0.0'.
- The assertion fails because result is '0.0' (or otherwise loses the negative sign) instead of '-0.0'.

Why it happened (root cause)
- MockCodeConsumer.add_number is deciding whether to prefix a '-' by testing the numeric value with a comparison like if value < 0: ….
- For IEEE‑754 floats, -0.0 compares equal to 0.0, so (-0.0) < 0 is False. That means add_number does not detect the sign bit on -0.0 and emits "0.0" instead of "-0.0".
- In addition, code that negates the value (value = -value) before formatting will also lose the negative‑zero sign bit (so even if you tried other tests afterward, the sign is gone).
- In short: the code uses value < 0 (or value == 0 checks) to detect negativity, which fails for negative zero because the sign bit is not visible via ordinary numeric comparisons.

Where in the code to look
- MockCodeConsumer.add_number(value): find the branch that decides whether to add a '-' (the if value < 0: or similar).
- Also check any line that does value = -value or uses abs(value) before formatting, and the line that converts the number to string (the call inside add_number that produces the string appended to the internal buffer that get_code later returns).

How to fix it (concrete steps)
1. Import math at the top of the module:
   - add: import math
2. Replace the comparison that detects negativity. For example, change:
   - if value < 0:
     to:
   - if math.copysign(1.0, value) < 0.0:
   This detects the sign bit (including -0.0) correctly.
3. Do not remove the sign by negating the value before formatting. If you need the absolute magnitude for formatting, use math.copysign or format the magnitude separately but keep the detected sign:
   - sign = '-' if math.copysign(1.0, value) < 0.0 else ''
   - magnitude = abs(value)   # OK to use abs() for the printed magnitude

4. Alternatively, avoid special‑casing negative zero entirely by using a formatting routine that preserves the sign, for example:

   - or s = format(value, 'g') but verify it keeps '-0.0' in your Python version
   Use whichever preserves the exact output you expect; repr(-0.0) -> '-0.0' reliably.
5. Add a unit test for negative zero specifically (the test you wrote is good). After the change, run the test suite.

Minimal example to verify detection:

- In contrast, (-0.0) < 0.0 is False.

Why these steps fix the problem
- math.copysign reads the sign bit from the float rather than relying on numeric ordering, so it correctly reports negativity for -0.0.
- Avoiding negation before formatting preserves the sign bit until you decide how to emit it.
- Using repr(value) (or a formatter that preserves sign) guarantees that the textual representation includes the negative zero sign if present.
