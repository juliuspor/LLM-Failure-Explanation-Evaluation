What happened (where it fails)
- The failure happens in the branch that handles a None input list and a non-None element. See the fragment around lines 337–347:
  - Line 337: control enters the branch for "elif element is not None".
  - Line 340: inferred_type is set to object.
  - Line 346–347: the code then checks
        if inferred_type == object and expected_type != object:
            raise TypeError(...)
  Because inferred_type was set to object on line 340, that condition becomes true whenever the caller passed an expected_type other than object, and the code raises TypeError immediately.

Why it fails (root cause)
- The code sets inferred_type to object when element is not None (line 340). That is wrong: with a concrete element available you should infer its actual type (e.g. type(element)), not object.
- As a result, later logic treats the element as having "unknown" type and rejects callers who supplied an expected_type, even though the concrete element could have satisfied that expected_type. In short: inferred_type is computed incorrectly, which causes the type-mismatch check at line 346 to raise erroneously.

Exactly what to change (step-by-step fixes)
1. Replace the incorrect inference at line 340 with the element's actual type:
   - Change inferred_type = object  ->  inferred_type = type(element)
   This ensures the code records the real type of the provided element.

2. Update the type-check logic (line 346) to validate the element against expected_type rather than testing inferred_type == object:
   - Instead of:
         if inferred_type == object and expected_type != object:
             raise TypeError(...)
     do:
         if expected_type is not None and not isinstance(element, expected_type):
             raise TypeError(f"element of type {type(element)!r} is not compatible with expected_type {expected_type!r}")
   This makes the intent explicit: if the caller supplied an expected_type, verify the actual element matches it.

3. If the rest of the function relies on inferred_type for constructing a new list of a specific component type, keep inferred_type = type(element) so subsequent code sees the right component type.

Why these fixes fix the problem
- Using type(element) gives the correct component type, avoiding the false assumption that the element is "unknown".
- Checking element against expected_type (via isinstance) enforces the contract without conflating "unknown inferred type" with "mismatch with expected_type".
