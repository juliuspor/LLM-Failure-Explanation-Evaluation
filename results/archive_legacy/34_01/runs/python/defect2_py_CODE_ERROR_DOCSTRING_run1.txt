What happened (where it failed)
- In get_paint, you clamp the input into v:
  - v = max(value, self._lower_bound)
  - v = min(v, self._upper_bound)
- Immediately afterwards you compute g using the original un-clamped variable value:
  - g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- Because you used value instead of v, g can be < 0 or > 255 when value is outside the bounds, so the subsequent check
  - if g < 0 or g > 255: raise ValueError(...)
  raises the ValueError you saw.

Why that is the root cause
- The purpose of v (the clamped value) is to ensure the value used to compute the color falls inside [self._lower_bound, self._upper_bound]. By ignoring v and using value, the clamp is effectively skipped for the color computation.
- Also, if self._upper_bound == self._lower_bound you will divide by zero in that same line; that will raise a different error if it happens.

Exact fixes (step-by-step)
1. Use the clamped variable v when computing g (replace value with v).
2. Protect against division by zero when upper_bound == lower_bound.
3. Optionally clamp g after conversion to int to guarantee it is inside 0..255.
4. Improve the error message to include the out-of-range value if you keep the check.

Corrected code (minimal change)
    def get_paint(self, value: float) -> Tuple[int, int, int]:
        # Clamp value to bounds
        v = max(value, self._lower_bound)
        v = min(v, self._upper_bound)
        
        # Avoid division by zero
        span = (self._upper_bound - self._lower_bound)
        if span == 0:
            g = 0  # or 255, or some defined behavior when bounds are equal
        else:
            g = int((v - self._lower_bound) / span * 255.0)
        
        # Ensure integer is within 0-255
        g = max(0, min(255, g))
        
        return (g, g, g)

If you prefer to keep the ValueError behavior instead of silently clamping g, change the check to test the computed g and include g in the message:
    if g < 0 or g > 255:
        raise ValueError(f"Computed grayscale value out of range: g={g}")

Summary
- The immediate cause is computing g from value instead of the clamped v. Fix that and also handle the zero-span case to avoid divide-by-zero.
