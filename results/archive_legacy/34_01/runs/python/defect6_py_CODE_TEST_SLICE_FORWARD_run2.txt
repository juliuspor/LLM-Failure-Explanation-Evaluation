What happened (root cause)
- The code treats -0.0 as an integer value because x == int(x) is True for x = -0.0. That causes the function to take the integer branch and call value = int(x), which becomes 0. Converting that 0 to string produces "0", losing the negative-zero sign.
- The earlier sign check if x < 0 (used to avoid sequences like x- -4) does not detect -0.0 because -0.0 < 0 is False in Python (it compares equal to 0.0). So no special handling occurs before the integer branch.
- Result: add(str(value)) on line where value is 0 produces "0" instead of the expected "-0.0".

Exactly where it fails in your code
- The branch condition if x == int(x) (start of the integer-handling block) catches -0.0.
- value = int(x) and mantissa = value convert the negative zero to the integer 0 and lose the sign.
- The test cc.add_number(-0.0) therefore reaches self.add(str(value)) and writes "0".

How to fix it (clear steps)
Option A — special-case negative zero before the integer branch (recommended)
1. At the top of add_number, detect negative zero using math.copysign:
   - if x == 0.0 and math.copysign(1.0, x) < 0:
       self.add("-0.0")
       return
2. Leave the rest of the function unchanged.

Option B — preserve sign inside the integer branch
1. Before converting to int, detect negative zero and format appropriately:
   - if x == 0.0 and math.copysign(1.0, x) < 0:
       self.add("-0.0")
       return
   (or set a flag and use "-0.0" instead of str(value) later)

Option C — use a float-to-string routine that preserves sign for -0.0
1. Use repr(x) or format(x, ".17g") for the non-integer branch, but you still need to special-case -0.0 similarly if you want the exact "-0.0" output when x looks like an integer.

Minimal code change (drop-in)
Insert at top of add_number (before the if x < 0 and prev == '-' check):
    if x == 0.0 and math.copysign(1.0, x) < 0:
        self.add("-0.0")
        return

Why this fixes it
- math.copysign(1.0, x) returns -1.0 for negative-zero values, so the check detects -0.0 even though x < 0 is False. Handling it first prevents the integer branch from converting -0.0 to int( -0.0 ) == 0 and losing the sign.

No other behavior changes
- This change only affects the special case x == -0.0. All other numbers follow the same existing paths.
