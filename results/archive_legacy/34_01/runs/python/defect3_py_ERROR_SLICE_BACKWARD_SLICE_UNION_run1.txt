What happened (where)
- The IndexError is raised at lines 35–36 in get_utf32_at (the check that does: if index < 0 or index >= len(code_units): raise IndexError(...)).
- The code is calling get_utf32_at(index) with index == 2 while len(code_units) == 2, so the check correctly treats index 2 as out of range and raises "String index out of range: 2".

Why it happened (root cause)
- The function to_utf16_code_units(s) (line 10, code_units on line 12) produces a list of UTF‑16 code units. That list length is the number of UTF‑16 units, not necessarily the number of Python string characters.
- The caller is supplying an index measured in Python string code points (character indices) rather than a UTF‑16 code‑unit index. When the string contains a surrogate pair (one Unicode character represented by two UTF‑16 code units) the code unit list is longer than, or shorter than, the caller’s index expectation. In this run the supplied index equals the number of code units (2), so it is out of range and the explicit bounds check at lines 35–36 raises the error.
- The later lines (38–44) show correct handling of surrogate pairs when looking ahead (index+1), but that logic is only reached if the initial index is inside the code_units bounds. The exception occurs before that because the index itself was invalid for the code_units list.

2) If callers should pass a UTF‑16 code‑unit index:
