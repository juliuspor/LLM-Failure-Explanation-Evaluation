The bug arises because after a translator returns a nonzero 'consumed' value, the original code tried to advance 'pos' by iterating consumed times and calling code_point_at at each step, which can call code_point_at with pos == length and raise IndexError. The translator contract treats 'consumed' as the number of UTF-16 code units consumed, so we should advance pos by consumed directly. Also validate consumed to be within the remaining range to fail fast on bad translators. Implemented translate to check 'out' and 'input_seq' argument validity, convert input to UTF-16 code units, loop calling translate_codepoint, handle consumed == 0 path by copying codepoint chars, validate consumed, and advance pos by consumed. Return None.
