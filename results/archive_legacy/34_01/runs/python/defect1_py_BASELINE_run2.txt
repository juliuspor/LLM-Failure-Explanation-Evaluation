What happened (where and what failed)
- The call DateTimeZone.for_offset_hours_minutes(-2, -15) throws ValueError at lines 396–397:
  if minutes_offset < 0 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset})
- minutes_offset is -15, so the unconditional check treats any negative minutes as out-of-range and raises "Minutes out of range: -15".
- That check is at the start of the method (before any combination of hours and minutes). Because the docstring and tests allow negative minutes when hours are negative or when hours are zero, the validation at lines 396–397 is too strict and rejects valid inputs.

Why that check is wrong (root cause)
- The docstring describes allowed combinations:
  - hours positive: minutes must be zero or positive (positive minutes with positive hours is allowed; positive hours with negative minutes should be ValueError).
  - hours zero: minutes may be positive or negative.
  - hours negative: minutes may be positive or negative; both (-2, 15) and (-2, -15) should produce -02:15.
- The code currently enforces minutes_offset >= 0 unconditionally (minutes_offset < 0 triggers an error). That contradicts the documented behavior and the test which passes negative minutes for a negative hour.
- So the immediate failure is caused by the unconditional minutes sign check at lines 396–397.

Secondary logic issues you should be aware of
- After the (wrong) validation the code mutates minutes_offset:
    hours_in_minutes = hours_offset * 60
    if hours_in_minutes < 0:
        minutes_offset = hours_in_minutes - minutes_offset
    else:
        minutes_offset = hours_in_minutes + minutes_offset
  This is confusing and error-prone because:
  - It reuses minutes_offset for a different quantity (total minutes), mixing concepts.
  - If minutes_offset were negative when hours_in_minutes < 0, the expression hours_in_minutes - minutes_offset will add the absolute value (because subtracting a negative adds), which is not the intended total offset calculation. The intended rule is:
    total_minutes = hours_offset*60 + (if hours_offset < 0 then -abs(minutes) else +minutes)
  So the correct arithmetic should explicitly compute total_minutes rather than mutating minutes_offset with a subtraction that flips signs unintentionally.

Exact fixes (concrete steps, with line references)
1. Change minutes validation (lines 396–397). Allow minutes in [-59, 59], but reject negative minutes when hours are positive:
   - Replace
       if minutes_offset < 0 or minutes_offset > 59:
           raise ValueError(f"Minutes out of range: {minutes_offset}")
     with something like:
       if minutes_offset < -59 or minutes_offset > 59:
           raise ValueError(f"Minutes out of range: {minutes_offset}")
       if hours_offset > 0 and minutes_offset < 0:
           # positive hours must not have negative minutes
           raise ValueError("Minutes must be non-negative when hours are positive")

2. Compute total minutes explicitly and clearly (around lines 402–405). Replace the current block with:
   hours_in_minutes = hours_offset * 60
   if hours_offset < 0:
       total_minutes = hours_in_minutes - abs(minutes_offset)
   else:
       total_minutes = hours_in_minutes + minutes_offset
   offset = FieldUtils.safe_multiply(total_minutes, DateTimeConstants.MILLIS_PER_MINUTE)

   (Then return cls.for_offset_millis(offset) as before.)

Why this fix works
- Validation now matches the docstring: minutes in [-59,59], but disallows the documented invalid case (positive hours + negative minutes).
- total_minutes calculation explicitly applies the correct sign rule:
  - For (-2, -15): hours_in_minutes = -120; total_minutes = -120 - 15 = -135 → "-02:15"
  - For (-2, 15): hours_in_minutes = -120; total_minutes = -120 - 15 = -135 → "-02:15"
  - For (2, 15): total_minutes = 120 + 15 = 135 → "+02:15"
  - For (0, -15): total_minutes = 0 + (-15) = -15 → "-00:15"

Small note about naming
- Use a separate variable total_minutes (do not overwrite minutes_offset) to avoid mixing the meaning of minutes and total minutes.

That change will make the test zone = DateTimeZone.for_offset_hours_minutes(-2, -15); assert zone.id == '-02:15' pass, and will align the code with the documented behavior.
