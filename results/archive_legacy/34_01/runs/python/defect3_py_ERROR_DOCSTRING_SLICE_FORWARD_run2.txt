What happened
- At line 38 the code does: high = code_units[index]
- The IndexError "String index out of range: 2" means index had the value 2 but code_units has length <= 2, so code_units[index] tried to read past the end of the sequence.
- The bounds check for the next element (index + 1 < len(code_units)) is on line 39, but it comes after the access to code_units[index]. That check protects the access to low (index+1) but not the initial access to high.

Why it happened (root cause)
- The method assumes index is a valid index into code_units, but it is not guaranteed by the caller. The code reads code_units[index] before verifying index < len(code_units), so a caller passing index == len(code_units) (or any out-of-range index) causes the exception.
- In short: you access high before checking bounds, so an out-of-range index crashes here (line 38).

Where exactly (lines & variables)
- Line 38: high = code_units[index] — this is where the exception is raised.
- Line 39: if 0xD800 <= high <= 0xDBFF and index + 1 < len(code_units): — this check is too late for protecting the high access.
- Variables involved: code_units (the sequence), index (value 2 in the reported error), high (attempted read).

   - Add a check before line 38 to ensure index is in range:
     - if index >= len(code_units): raise IndexError("index out of range")
     - then proceed with high = code_units[index]
   - This makes the failure explicit and avoids the surprising crash at the assignment.

           high = code_units[index]
           if 0xD800 <= high <= 0xDBFF and index + 1 < len(code_units):
               low = code_units[index + 1]

3. Option C — Fix the caller
   - Find the code that calls this method with index equal to len(code_units) (or otherwise out of range) and stop calling it with an out-of-range index. Add a caller-side guard such as: if index >= len(code_units): do not call / adjust index.

Why these fixes help
