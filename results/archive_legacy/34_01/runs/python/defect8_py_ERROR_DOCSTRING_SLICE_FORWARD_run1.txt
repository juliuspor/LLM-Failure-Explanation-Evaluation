What happened (where)
- The ValueError is raised at line 246:
  raise ValueError(f"Invalid locale format: {locale_str}")
  with locale_str == "fr__POSIX".
- The failing code around lines 243–247 assumes a variant form of "xx_CC_variant" where:
  - language = locale_str[0:2]
  - country  = locale_str[3:5]
  - variant separator must be at locale_str[5] (an underscore)
- That assumption is wrong for inputs where the country is empty but a variant is present (the form "ll__VARIANT", i.e. two consecutive underscores), which is exactly "fr__POSIX".

Why it fails (root cause)
- For "fr__POSIX" the character positions are:
  0:'f' 1:'r' 2:'_' 3:'_' 4:'P' 5:'O' ...
- The code expects the country to occupy locale_str[3:5]. In this string locale_str[3:5] == "_P", not two uppercase letters.
- Then the code checks locale_str[5] != '_' (line 245–246). For "en_GB_xxx" that check is correct because the variant separator is at index 5, but for "fr__POSIX" the variant separator is at index 2→3 (two underscores in a row) and there is no underscore at index 5 — locale_str[5] == 'O' — so the code treats the string as malformed and raises the ValueError.
- In short: the method does not handle the valid locale form with an empty country and a variant ("ll__VARIANT"), and so it mis-parses "fr__POSIX" and raises the error.

Exact location to fix
- The problem is in the branch that handles the 3-component form (lines 243–247). It needs an extra branch to detect the "empty country + variant" case (double underscore after the language).

Concrete fix (code-level steps)
1. Before the existing check at line 245, add a branch for the double-underscore case:
   if locale_str[2] == '_' and locale_str[3] == '_':
       # language present, empty country, variant is remainder
       return Locale(locale_str[0:2], "", locale_str[4:])
2. Keep the existing branch for the normal language+country+variant case:
   - Validate that locale_str[3:5] are uppercase letters
   - Validate that locale_str[5] == '_' before taking variant from locale_str[6:]
3. Add or update unit tests to cover:
   - "fr__POSIX" (should produce Locale("fr", "", "POSIX"))
   - "en_GB_xxx" (existing case)
   - Invalid forms like "f__POSIX", "en_gb_xxx" (language lowercase / country uppercase checks)

Why this fix works
- It explicitly supports the valid locale syntax where country is empty and the variant follows two underscores, so "fr__POSIX" is parsed as language="fr", country="" and variant="POSIX" instead of being rejected.
