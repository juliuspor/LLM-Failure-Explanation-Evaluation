What happened (root cause)
- The code receives x == -0.0.
- The check x == int(x) is True for -0.0, so the path that converts to an int runs.
- value = int(x) converts -0.0 to 0 and loses the negative sign (int(-0.0) == 0).
- The earlier test if x < 0 and prev == '-': used to decide whether to add a space does not fire for -0.0 because (-0.0 < 0) is False in Python.
- As a result the sign is lost and you output "0" instead of preserving the negative zero ("-0.0" or "-0").

Where it occurs (exact lines/variables)
- The sign test is at the top of add_number: if x < 0 and prev == '-': (this fails to detect -0.0).
- The loss of sign happens at value = int(x) (value becomes 0 for -0.0).
- Later self.add(str(value)) produces "0" instead of a signed zero, which triggers your assertion (Expected -0.0, got 0).

Why this is a bug
- Floating-point negative zero is a distinct value for formatting/sign purposes even though numeric comparisons treat it equal to 0. You must detect its sign separately before you convert to int.

Concrete fixes (steps you can apply)
1. Detect negative zero using math.copysign before converting to int. For example:
   - sign_negative = math.copysign(1.0, x) < 0
   - Use sign_negative instead of x < 0 when deciding spacing and when formatting the output.

2. Preserve the sign when x is an integer-valued float:
   - If sign_negative and value == 0, format explicitly with a leading '-' (e.g. "-0" or "-0.0") instead of relying on int(x).
