What went wrong (root cause)
- On the first branch you do:
  inferred_type = type(array)
  but type(array) returns the Python list class (i.e. list), not the element/component type of the list. This is the wrong thing to pass to ArrayUtils._copy_list_grow1 and the wrong thing to compare to object in the later type-check.
- Consequences:
  - _copy_list_grow1(array, inferred_type) receives list instead of the element type. That function expects a component/element type, so it will create or cast the copied array incorrectly and later assignment or checks will fail or produce a misleading TypeError.
  - The conditional if inferred_type == object and expected_type != object will rarely behave as intended because inferred_type will usually be list, not the element type or object, so the check doesn't detect the real mismatch.
- Where it occurs (lines referenced to your snippet):
  - Line "inferred_type = type(array)" is the wrong expression.
  - Line "new_list = ArrayUtils._copy_list_grow1(array, inferred_type)" receives the wrong type and causes the failure.
  - The subsequent type check (if inferred_type == object ...) uses the wrong inferred_type value and therefore can produce incorrect behavior or miss a mismatch.

Why this fails in plain terms
- You wanted the element's class (the component type), but you grabbed the container's class (list). The copy helper and the type-checking logic assume they are working with element/component types, so they break when given the list class.

1. When array is not None, infer the element type from the array's elements, not from the list object:

   - If the array is empty, fall back to using element (if provided) or expected_type or object.
2. Make the inferred_type value consistent with what ArrayUtils._copy_list_grow1 expects (an element/component type).
3. Adjust the later check to compare component types (and handle empty-array cases explicitly).
