What happens
- At lines 396–397 the code rejects any negative minutes:
  if minutes_offset < 0 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")
  This makes calls with negative minutes (e.g. minutes_offset = -15) raise ValueError even though the docstring explicitly allows negative minutes for zero or negative hours (examples (0, -15) and (-2, -15)).

- Even if you removed that range check, the arithmetic that computes the total offset is wrong. The block:
    hours_in_minutes = hours_offset * 60
    if hours_in_minutes < 0:
        minutes_offset = hours_in_minutes - minutes_offset
    else:
        minutes_offset = hours_in_minutes + minutes_offset
  is incorrect when both hours_offset and minutes_offset are negative. Example:
    hours_offset = -2  -> hours_in_minutes = -120
    minutes_offset = -15
  The code computes minutes_offset = -120 - (-15) = -105 minutes, which is -1:45. The correct result for (-2, -15) is -135 minutes (-2:15). The bug is that the code subtracts minutes_offset even when minutes_offset is already negative, causing addition instead of further subtraction.

Why that causes failure
- Because the minutes validation at lines 396–397 prevents valid negative-minute inputs from reaching the arithmetic, callers that expect the documented behavior get a ValueError there.
- If you fix the validation but keep the arithmetic as-is, negative-hour + negative-minute combinations will produce wrong offsets (incorrect sign and magnitude).

Where to change (concrete steps)
1) Fix the minutes range check (lines 396–397):
   - Allow minutes_offset in the range -59 .. +59 (inclusive). Do not reject negative minutes unconditionally.
   - Add an explicit check to reject the disallowed combination: positive hours with negative minutes (hours_offset > 0 and minutes_offset < 0), because the doc says (2, -15) is invalid.

   Example replacement logic:
     if minutes_offset < -59 or minutes_offset > 59:
         raise ValueError(f"Minutes out of range: {minutes_offset}")
     if hours_offset > 0 and minutes_offset < 0:
         raise ValueError("Positive hours with negative minutes is not allowed")

2) Fix the minutes arithmetic (the block that computes the total minute offset):
   - Compute total_minutes in a way that respects signs. Use hours_in_minutes plus minutes_offset for same-sign or zero hours, and subtract absolute minutes when hours are negative.
   - A correct and clear implementation:
       hours_in_minutes = hours_offset * 60
       if hours_in_minutes < 0:
           total_minutes = hours_in_minutes - abs(minutes_offset)
       else:
           total_minutes = hours_in_minutes + minutes_offset
   - Note: treat hours_offset == 0 specially if you want both signs for minutes: when hours_in_minutes == 0, the else branch above adds minutes_offset (which may be negative), producing the correct +00:15 or -00:15 behavior.

3) Continue to convert minutes to milliseconds and check overflow:
   offset = FieldUtils.safe_multiply(total_minutes, DateTimeConstants.MILLIS_PER_MINUTE)

Summary of correct behavior
- Allow minutes between -59 and +59.
- Reject (hours > 0 and minutes < 0).
- For hours < 0 and minutes < 0, subtract the absolute minutes from hours_in_minutes (so (-2, -15) -> -120 - 15 = -135).
- For hours >= 0, add minutes_offset to hours_in_minutes (so (2, 15) -> 120 + 15 = 135; (0, -15) -> 0 + (-15) = -15).

This fixes the ValueError at lines 396–397 and the wrong-sign/wrong-magnitude result from the minutes arithmetic.
