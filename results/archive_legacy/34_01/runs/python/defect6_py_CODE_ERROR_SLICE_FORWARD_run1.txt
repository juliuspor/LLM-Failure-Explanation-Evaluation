What happened
- The code treats any float that compares equal to an integer as an integer branch:
  if x == int(x) and not math.isinf(x) and not math.isnan(x):
- For x == -0.0 this condition is true (because -0.0 == 0 and int(-0.0) == 0).
- Then value = int(x) sets value to 0, losing the negative-zero sign.
- Later you call self.add(str(value)) and get "0" instead of the expected "-0.0".

Where it occurs (lines / variables)
- The problematic checks and conversion are here:
  if x == int(x) and not math.isinf(x) and not math.isnan(x):
      value = int(x)
  ...
      self.add(str(value))
- The loss of sign happens on the assignment value = int(x) when x is -0.0.

Why this is the root cause
- Python's int(-0.0) returns 0 (no sign) and float equality x == int(x) is true for -0.0.
- IEEE-754 negative zero is distinct as a float but becomes indistinguishable after int(...) or str(int(...)).

Concrete fixes (exact steps and snippets)

Option A — detect negative zero first and handle it explicitly (recommended)
1. Right before converting to int, check for negative zero:
   import math
   if x == 0.0 and math.copysign(1.0, x) < 0:
       self.add("-0.0")
       return
2. Then keep the existing integer branch unchanged.

Example insertion:
   # handle negative zero explicitly
   if x == 0.0 and math.copysign(1.0, x) < 0:
       self.add("-0.0")
       return

Option B — treat zero as non-integer so you preserve float formatting
1. Change the integer-detection condition to avoid treating negative zero as integer:
   if (x == int(x) and not math.isinf(x) and not math.isnan(x)
       and not (x == 0.0 and math.copysign(1.0, x) < 0)):
       ...

Option C — format from the float instead of int and try to strip .0 while preserving sign
1. Use str(x) for integers too, but post-process to shorten "123.0" to "123" while preserving "-0.0".
   s = str(x)
   if s.endswith(".0") and not s.startswith("-0.0"):
       s = s[:-2]
   self.add(s)

Which to pick
- Option A is simplest and explicit: handle negative zero first so you never lose the sign.
- Option C is robust if you want consistent float-to-string formatting and must keep float-sign semantics.

Summary
The failure is because int(-0.0) becomes 0 and your code takes the integer branch, producing "0". Detect negative zero (x == 0.0 and math.copysign(1.0, x) < 0) before converting to int, and output "-0.0" (or otherwise avoid converting negative zero to int).
