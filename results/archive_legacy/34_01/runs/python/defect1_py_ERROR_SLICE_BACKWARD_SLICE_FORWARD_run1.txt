What happened (where it failed)
- The code raised ValueError("Minutes out of range: -15") at the check on lines 396–397:
  - 396: if minutes_offset < 0 or minutes_offset > 59:
  - 397:     raise ValueError(f"Minutes out of range: {minutes_offset}")
- That means minutes_offset was -15 when that check executed.

Why this is the root cause (why minutes_offset is negative)
- Earlier code computes minutes_offset from parsed hour/minute parts and the sign of the offset, but the negative sign can be applied to the hours only while minutes stay positive (for input like "-1:15"), or the sign can be applied to the combined value incorrectly.
- The code later tries to adjust minutes with these lines (402 and 404):
  - 402: minutes_offset = hours_in_minutes - minutes_offset
  - 404: minutes_offset = hours_in_minutes + minutes_offset
  These adjustments indicate the code expects minutes_offset to be a non-negative minute-part (0..59) and then combine it with hours_in_minutes, but the range check on 396 runs too early against minutes_offset before that normalization.
- Concretely: for an input that represents -1 hour and 15 minutes, minutes_offset may be computed as -15 (minutes * sign). The early check at line 396 treats -15 as invalid and raises, instead of treating it as the 15-minute part of a negative offset.

Where to look in the code
- The failing variable: minutes_offset
- The failing check: lines 396–397 (range validation)
- The later normalization/combination that expects a non-negative minute part: lines 402 and 404
- The eventual use: offset = FieldUtils.safe_multiply(minutes_offset, DateTimeConstants.MILLIS_PER_MINUTE) (line 405)
