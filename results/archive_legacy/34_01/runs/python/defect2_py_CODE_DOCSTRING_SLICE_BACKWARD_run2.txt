What happened (where and why)
- At get_paint (lines shown around 54–75) you try to clamp the input into v:
  - Line ~56: v = max(value, self._lower_bound)
  - Line ~57: v = min(v, self._upper_bound)
- But when you compute g you use the original value, not the clamped v:
  - Line 71: g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
  Because of that, clamping has no effect. If value is outside the bounds, g can fall outside 0..255 and you hit the ValueError at line 74 even though you intended to clamp it first.

Additional bug that will crash in some cases
- If self._upper_bound == self._lower_bound then the denominator (self._upper_bound - self._lower_bound) is zero and the expression on line 71 raises ZeroDivisionError before your ValueError check.

Where to change (precise fixes)
1. Use the clamped variable v when computing g:
   - Replace (value - self._lower_bound) with (v - self._lower_bound) on line 71.
2. Protect against equal bounds (zero denominator) before computing g:
   - If upper_bound == lower_bound, decide a behavior (e.g., return mid-gray or 0/255) and handle it explicitly.
3. Optionally improve the error reporting to include the actual g and the bounds so debugging is easier.

Suggested corrected code (minimal, explicit)
    def get_paint(self, value: float) -> Tuple[int, int, int]:
        # Clamp value to bounds
        v = max(value, self._lower_bound)
        v = min(v, self._upper_bound)
        
        denom = self._upper_bound - self._lower_bound
        if denom == 0.0:
            # Decide behavior when bounds are equal; here return middle gray
            g = 128
        else:
            # Use the clamped value v, not the original value
            g = int((v - self._lower_bound) / denom * 255.0)
        
        if g < 0 or g > 255:
            raise ValueError(f"Color parameter outside of expected range: g={g}, bounds=({self._lower_bound},{self._upper_bound})")
        
        return (g, g, g)

Why this fixes it
- Using v ensures out-of-range inputs are clamped before conversion to color, so you won't raise ValueError for inputs you intended to accept.
- Checking denom == 0 prevents division-by-zero crashes.
- The improved error message gives the exact g and bounds when an error still occurs.

That pinpoints the root cause: you clamped into v but then ignored v and used value in the calculation (line 71).
