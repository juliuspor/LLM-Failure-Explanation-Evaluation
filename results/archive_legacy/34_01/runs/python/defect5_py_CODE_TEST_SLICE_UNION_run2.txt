What happened (summary)
- The test calls ArrayUtils.add(None, None, expected_type=str) and expects [None].
- The function sets inferred_type = object (line 340) because both array and element are None.
- After constructing the new list, the code checks:
  if inferred_type == object and expected_type != object: raise TypeError (line 346).
- Because inferred_type is object and expected_type is str, the function raises a TypeError instead of returning [None].

Why that happens (root cause)
- inferred_type is computed as object whenever both array and element are None (line 340). That makes the code treat the new list as an "object list".
- The later check (lines 346–347) treats an "object list" as incompatible with any expected_type other than object and raises a TypeError. That check runs even when the element value is None, which is a valid value for a list of str in Python.
- In short: inferred_type == object triggers a strict incompatibility check against expected_type, so providing expected_type=str causes the early error when both inputs are None.

Where it occurs (specific lines/variables)
- inferred_type assigned to object: line 340 (elif element is not None branch falls through to inferred_type = object).
- The failing check: lines 346–347:
    if inferred_type == object and expected_type != object:
        raise TypeError(...)

Why that check is wrong here
- None is a valid list element for any element type (e.g., a list[str] can contain None), so raising because the inferred_type is object is incorrect when the only element is None.
- The code also conflates type(array) with element type (type(array) is list, not the list element type). That makes inferred_type unreliable in other cases too.

Concrete fixes (ordered steps you can apply)
1. If expected_type is provided and both array and element are None, use expected_type as inferred_type:
   - Replace the branch around line 340 with:

     elif element is not None:
         inferred_type = type(element)
     elif expected_type is not None:
         inferred_type = expected_type
     else:
         inferred_type = object
   - This prevents the erroneous object vs expected_type mismatch when expected_type is known.

2. Don’t raise just because inferred_type == object:
   - Change the check at lines 346–347 to allow None values:
     if inferred_type == object and expected_type is not None and element is not None:
         if expected_type != object:
             raise TypeError(...)
   - Or more simply, only raise when there is an actual non-None element that cannot be cast to expected_type.

3. Fix type(array) usage (important for correctness elsewhere):
   - type(array) returns list, not the element type. If you intended to infer element type from an existing list, inspect an element (if any) instead:
     if array is not None and len(array) > 0:

     elif array is not None:
         # empty list — fall back to expected_type or object

   - This prevents mis-inference when a non-empty list is passed.
