What happened (where it fails)
- The failure is in add_number at the very first condition:
  if x == int(x) and not math.isinf(x) and not math.isnan(x):
  Python evaluates that left-to-right. That means int(x) is executed before math.isinf(x) and math.isnan(x).
- If x is float('inf') or float('nan'), calling int(x) raises an exception (OverflowError for inf, ValueError for nan). So the code crashes while trying to evaluate int(x) instead of guarding against non-finite floats first.

Why this is the root cause (clear, concrete)
- The code intended to skip integer-conversion work for infinities/NaNs, but due to evaluation order the conversion happens first.
- The variables/methods involved: x, int(x) (line in the condition), math.isinf(x), math.isnan(x), and the function add_number itself (the error occurs inside that function before any call to self.add).

Exact place: the condition at the top of add_number (the very first if) — int(x) runs too early.

How to fix it (step-by-step, explicit)
1. Check finiteness first, before calling int(x). Replace the condition with one of these safe options:

   Option A (minimal change):
   if not math.isinf(x) and not math.isnan(x) and x == int(x):

   This reorders the checks so math.isinf/math.isnan run before int(x).

   Option B (better and clearer):
   if math.isfinite(x) and x == int(x):
   - math.isfinite(x) covers both isinf and isnan.

   Option C (best for readability + avoids int conversion unless needed):
   if math.isfinite(x) and isinstance(x, float) and x.is_integer():
   - float.is_integer() expresses the intent directly without converting to int.

2. (Optional) If you still want an int value, compute it after confirming finiteness and integrality:
   val = int(x)
   then use val instead of recomputing int(x) elsewhere.

3. Add a test case to prevent regression:
   - Call add_number(float('inf')) and add_number(float('nan')) and verify no exception.
   - Call add_number(1e308) (very large finite float) to ensure behavior remains correct.

Why these fixes work
- They avoid calling int(x) for non-finite floats, preventing OverflowError/ValueError.
- math.isfinite or float.is_integer are explicit and robust guards.

No other changes are required; the rest of the method can remain the same once the initial check is fixed.
