What happened (what you saw)
- The test called cc.add_number(-0.0) expecting the consumer to emit the string '-0.0'.
- The actual code produced '0' (the minus sign was lost).

Where it happened (which code and lines)
- The symptom appears while calling MockCodeConsumer.add(newcode) (slice shown). Relevant lines:
  - Line 129–133: we check newcode length and set c = newcode[0]
  - Lines 133–136: a conditional decides whether to append a separating space
  - Line 138: self.append(newcode) writes the token into the consumer

Why it happened (root cause)
- add_number() is delivering the negative number in two pieces (a '-' token and a '0.0' token) rather than as one string "-0.0". Because add() treats tokens separately, the minus and the digits are appended in separate calls.
- The conditional at lines 133–136 is intended to decide whether to insert a space between tokens. Its current logic causes the minus token and the numeric token to be combined or separated in the wrong way, which lets the minus sign be dropped when the second piece (the digits) is appended.
  - Concretely: one call appended just '-' (previous token), then a second call appended '0.0' (newcode). Because the code that formats or normalizes those tokens elsewhere canonicalized the numeric piece to '0' (dropping the negative sign when the float value is -0.0), the final result lost the sign.
- In short: the negative zero never stayed attached to the numeric characters in a single newcode string, and the logic that splits/joins tokens plus the numeric formatter produced '0' instead of '-0.0'.

Why the minus sign was lost instead of producing '-0.0'
- Floating negative zero exists (Python str(-0.0) is '-0.0'), but if code splits the minus into its own token then downstream formatting/normalization of the numeric token can canonicalize its absolute value and produce '0' (no sign). The sign is only preserved when the minus is part of the same string as the numeric characters or the code specially preserves sign for zero values.
- The shown add() method never repairs that split: it simply appends tokens and only inserts a space when certain word-character conditions are met, so the separated '-' ended up not being preserved in the joined output.
