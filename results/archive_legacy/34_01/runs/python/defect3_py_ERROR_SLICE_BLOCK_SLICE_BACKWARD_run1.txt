Why it happened (root cause)
- The function to_utf16_code_units(s: str) builds a list named code_units containing UTF‑16 code units for the Python string s.
- The caller computed or passed an index that was intended to count Unicode characters (Python code points) — e.g., "character index 2" — but to_utf16_code_units is checking against the number of UTF‑16 code units instead.
- For strings that contain characters outside the BMP (for example an emoji), one Python character (one code point) becomes two UTF‑16 code units. That makes the mapping between "character index" and "code unit index" different.
- Concretely: if the caller wanted the 3rd character (index 2) of s ("A", "😊", "B"), that is code point index 2, but the UTF‑16 code units list for "A😊B" is length 4 (A=1, emoji=2, B=1). If the caller instead computed an index in code points but passed it to code that expects a code‑unit index, they can end up with a value that is out of range for code_units — producing exactly the IndexError you saw.

- The mismatch is between how index was computed/meaningful in the caller and what to_utf16_code_units expects.
