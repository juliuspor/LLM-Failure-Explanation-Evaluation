What happened
- The code that updates _max_middle_index compares the current period’s middle against the wrong reference period.
- Specifically, in this block:

      if self._max_middle_index >= 0:
          max_middle_period = self.get_data_item(self._min_middle_index).get_period()
          ...

  the call uses self._min_middle_index instead of self._max_middle_index. That means the algorithm sometimes compares the current period's middle to the middle of the item at _min_middle_index, not the item at _max_middle_index. As a result the branch that should update _max_middle_index uses an incorrect baseline and can set _max_middle_index to the wrong value — which is why your assertion failed (expected 1, got 3).

Where it occurs (line reference)
- In the provided snippet, the incorrect line is the one assigning max_middle_period:
  max_middle_period = self.get_data_item(self._min_middle_index).get_period()
- It should read:
  max_middle_period = self.get_data_item(self._max_middle_index).get_period()

Why this causes the failure
- The purpose of the block is: if this item’s middle is greater than the current maximum middle, make this item the new maximum.
- Because the code retrieved the period at _min_middle_index instead of _max_middle_index, the comparison used the wrong middle value. That leads to wrongly skipping or accepting updates and yields an incorrect _max_middle_index (3 instead of 1 in your test).

Exact fix
- Replace the mistaken reference:
  Change
    max_middle_period = self.get_data_item(self._min_middle_index).get_period()
  to
    max_middle_period = self.get_data_item(self._max_middle_index).get_period()
