The bug arises because translate converts input_seq to UTF-16 code units but some other parts assume Python code points; however earlier code already has to_utf16_code_units which converts properly. The reported issue was that translate_string passed a Python string and translate used to_utf16_code_units — that is fine. The root cause per diagnosis was that code_point_at raises IndexError if index out of range; but in translate we only call code_point_at when consumed==0, at pos which should be < length. The specific failing check was code_point_at's index check triggered when code_units index+1 was accessed elsewhere; but the diagnosis suggests translate should ensure proper bounds when incrementing pos after consumed>0: the loop currently does for pt in range(consumed): pos += char_count(code_point_at(code_units, pos)). If consumed==2 and code_units has only one code unit, code_point_at(pos) will raise. That can happen if a translator incorrectly returns consumed>0 claiming it consumed two codepoints but input lacks second. To be robust, ensure pos < length before calling code_point_at and handle truncated surrogate by advancing by 1. Implement safer advance: for _ in range(consumed): if pos < length: cp = code_point_at(code_units, pos); pos += char_count(cp) else: pos +=1 break. Also ensure consumed can't be negative.
