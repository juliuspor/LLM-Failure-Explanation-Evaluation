What happened
- The failing assertion is because the code produced "- 0.0" (with a space) instead of the expected "-0.0".
- The space is inserted by the branch at lines 132–136 in MockCodeConsumer.add:

  - At line 133 the code tests:
    if (self.is_word_char(c) or c == '\\') and self.is_word_char(self.get_last_char()):
      self.append(" ")

  - When adding the number -0.0 your code path splits the token so add() is called for "0.0" when the buffer already ends with "-".
  - c == '0' -> self.is_word_char(c) is True.
  - self.get_last_char() returns '-' and self.is_word_char('-') also returns True (incorrectly).
  - Because both sides of the and are True, the code inserts a space between '-' and '0', producing "- 0.0".

Root cause
- is_word_char(...) incorrectly classifies the minus '-' character as a "word" character. The space-insertion logic at line 133 is intended only when both the new token and the previous final character are ordinary word characters (letters/digits/underscore), so they need a separating space. Because '-' is misclassified as a word character, the condition fires and a space gets inserted before numbers that follow a minus.

Exactly where it occurred
- Method: MockCodeConsumer.add
- Lines: 132–136 (condition and append(" ") call)
- Variables: newcode (first char c), self.get_last_char() (returns '-'), and self.is_word_char(...) (returns True for '-').

How to fix (clear, actionable steps)
1. Fix is_word_char so it only returns True for real word characters (letters, digits, underscore). Example change (inside is_word_char):
   - Replace whatever current logic is with a check equivalent to:
     return ch.isalnum() or ch == '_'
   - Make sure it returns False for '-' and other punctuation.

2. Alternatively (if you prefer a localized change), change the condition in add() to test last_char with a stricter predicate rather than self.is_word_char:
   - last = self.get_last_char()
   - if (self.is_word_char(c) or c == '\\') and (last is not None and (last.isalnum() or last == '_')):
         self.append(" ")

3. Add a unit test that checks a negative zero / negative number scenario:
   - cc = MockCodeConsumer(); cc.add_number(-0.0); assert cc.get_code() == '-0.0'
   - And tests where previous char is punctuation to ensure no space inserted unexpectedly.

Why these fixes solve it
- They prevent '-' from being treated as a word character, so the condition that inserts a separating space will not trigger when the previous character is a minus. That stops the extra space between '-' and the numeric literal.

No-other-guess note
- The visible symptom (extra space) maps directly to the condition at lines 133–136 and the misclassification in is_word_char; no other parts of add() or append() need changing.
