What happened (symptom)
- The test expects s.get_max_middle_index() == 1 but the method returned 3 (AssertionError).
- get_max_middle_index simply returns self._max_middle_index (see slice: def get_max_middle_index -> return self._max_middle_index at lines 449–451). That means _max_middle_index was set to 3 earlier, not computed on the fly.

Where it went wrong (root cause)
- The bug is in the code that updates self._max_middle_index when you call s.add(...). The value 3 equals the index of the last added item (the fourth add call), so the updater stores the index of the newly added interval instead of the "middle index" for the overlapping group.
- In other words, when adding the fourth interval (index 3), the code sets self._max_middle_index = 3 rather than computing the group middle and keeping the maximum of previously computed middles.
- Concretely: after the four adds you showed (indices 0..3), the correct middle index for the maximal-overlap group should be 1, but the updater overwrote _max_middle_index with the newly appended index 3. That is why get_max_middle_index() returns 3.

Why that is incorrect for your test case
- Your test sequence:
  - add index 0: (0,100) value 1.0
  - add index 1: (100,200) value 2.0
  - add index 2: (0,50) value 3.0

- The intervals that produce the expected "max middle index = 1" are the overlapping group(s) computed across existing intervals (the logic that decides a "middle index" should find index 1, not the newest append index 3).
- Because the updater stores the append index unconditionally (or uses the wrong endpoints to compute the middle), it loses the correct previously computed middle = 1.
