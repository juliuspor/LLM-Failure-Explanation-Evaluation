What happened
- The test calls ArrayUtils.add(None, None, expected_type=str).
- In add(), the code sets inferred_type = object because both array and element are None (lines shown in your snippet).
- new_list is created and element None is placed into it — that part is fine.
- Right after that, this check runs (lines in your snippet):

    if expected_type is not None:
        if inferred_type == object and expected_type != object:
            raise TypeError(...)

  Because inferred_type == object and expected_type == str, the code raises a TypeError and the test fails.

Why it fails (root cause)
- The method treats inferred_type == object as “unknown element type” and then refuses to allow converting/declaring that unknown list to an expected_type that is not object. In other words, when both array and element are None the code treats the element-type as “object” and then always rejects any non-object expected_type (line with raise TypeError).
- That rejection is incorrect in this situation: asking for a list of str containing None (expected_type=str) is reasonable and should not throw. The code uses inferred_type == object as a hard block instead of deferring to expected_type.

Exactly where it occurs
- inferred_type is set to object in the else branch of this block:

        if array is not None:
            inferred_type = type(array)
        elif element is not None:
            inferred_type = type(element)
        else:
            inferred_type = object

- The exception is raised in this block (same snippet):

        if expected_type is not None:
            if inferred_type == object and expected_type != object:
                raise TypeError(...)

How to fix it (concrete steps)
Option A — prefer expected_type when provided (recommended):
1. When expected_type is passed, use it as the inferred element type before creating the new list. Replace the inference block with:

        if expected_type is not None:
            inferred_type = expected_type
        elif array is not None:
            inferred_type = type(array)
        elif element is not None:
            inferred_type = type(element)
        else:
            inferred_type = object

2. Remove or relax the later check that raises TypeError, because expected_type is now already honored. At minimum, remove the branch that raises when inferred_type == object and expected_type != object.

Option B — only raise when you have conflicting actual information:
1. Keep inferred_type as before but change the TypeError condition so you only raise when inferred_type != object and expected_type is incompatible with that inferred_type. For example:

        if expected_type is not None and inferred_type is not object and inferred_type != expected_type:
            raise TypeError(...)

This allows expected_type to be applied when inferred_type was unknown (object) or equal to expected_type.

Why these fixes work
- Option A gives expected_type precedence: when the caller explicitly asks for a str list, you create a list that declares that element type rather than refusing it.
- Option B avoids rejecting a caller who supplied None for both array and element; you only reject when you actually know the concrete inferred_type and it conflicts with expected_type.

One-line patch example (Option A):
- Replace the three-way inference block with:

        if expected_type is not None:
            inferred_type = expected_type
        elif array is not None:
            inferred_type = type(array)
        elif element is not None:
            inferred_type = type(element)
        else:
            inferred_type = object

and remove the later raise branch.

That will make ArrayUtils.add(None, None, expected_type=str) succeed and return [None] as the test expects.
