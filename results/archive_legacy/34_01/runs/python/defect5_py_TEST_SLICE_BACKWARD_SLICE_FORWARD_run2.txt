What happened (where it failed)
- The test calls ArrayUtils.add(None, None, expected_type=str).
- In the code shown, the loop never executes the branch at line 337 because every element is None (the condition is "elif element is not None").
- As a result, inferred_type stays at its default value object (line 340).
- At line 346 the code checks:
  if inferred_type == object and expected_type != object:
and then raises a TypeError at line 347.
- Because inferred_type is object (no non-None value was found) and expected_type is str, the function raises the TypeError instead of returning [None]. That is why the test fails.

Why this is the root cause (simple)
- The function assumes that if no element provides a concrete type, inferred_type == object represents an ambiguous type and should not be combined with a non-object expected_type.
- But passing expected_type=str while all elements are None is a valid call: None is compatible with any element type, so the function should accept expected_type=str and return [None].
- The code never distinguishes “no non-None elements” from “inferred element type is the generic object” and therefore rejects the call.

         inferred_type = expected_type
   - Or skip raising when all elements are None:
     if inferred_type == object and expected_type != object and any_non_none:
         raise TypeError(...)
   - Or treat None as compatible with expected_type by simply not raising when expected_type is provided and all elements are None.
3. Ensure any follow-up code that relies on inferred_type uses the updated value.
