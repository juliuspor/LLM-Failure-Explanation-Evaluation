What went wrong, where it happened, and why
- What happened: get_paint(-0.5) raised ValueError instead of returning (0,0,0).
- Where it happened: the computation of g on this line:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- Why it happened: you clamp the input into v on lines:
  v = max(value, self._lower_bound)
  v = min(v, self._upper_bound)
  but you then ignore v and compute g from the original value variable. For a test value below the lower bound (value = -0.5) v becomes the lower bound (presumably 0), but (value - self._lower_bound) is still negative, producing a negative g. The subsequent range check fails and raises ValueError.

Additional failure mode to be aware of
- If self._upper_bound == self._lower_bound you will get a division by zero at the same computation line. That should be handled explicitly.

Fix (concrete steps)
1. Use the clamped variable v when computing g.
2. Check for zero range before dividing.
3. Optionally use round() or clamp g into [0,255] after conversion.

Example corrected code (replace the failing lines):
    # Clamp value to bounds
    v = max(value, self._lower_bound)
    v = min(v, self._upper_bound)

    # Avoid division by zero if bounds are equal
    range_span = self._upper_bound - self._lower_bound
    if range_span == 0:
        g = 0  # or 255, or some defined value for degenerate range
    else:
        g = int((v - self._lower_bound) / range_span * 255.0)

    # validate range
    if g < 0 or g > 255:
        raise ValueError(f"Color parameter outside of expected range: Red Green Blue")

    return (g, g, g)

Why this fixes the test
- For value = -0.5, v will be clamped to self._lower_bound (0), the numerator becomes 0, g becomes 0, and the function returns (0,0,0) as expected.
