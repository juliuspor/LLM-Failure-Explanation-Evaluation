What happened (where): at lines 396–397 the code unconditionally rejects any negative minutes:

 396 |         if minutes_offset < 0 or minutes_offset > 59:
 397 |             raise ValueError(f"Minutes out of range: {minutes_offset}")

Why it fails (root cause): the docstring explicitly allows negative minute offsets in some combinations (for example (0, -15) -> -00:15 and (-2, -15) -> -02:15). The current check treats any negative minutes as invalid, so calls like make_zone(0, -15) or make_zone(-2, -15) will raise ValueError even though the doc says they are valid. In short: the code enforces minutes ∈ [0,59] rather than the intended minutes ∈ [-59,59] with extra rules about sign combinations.

Why that happens (mechanics): minutes_offset < 0 is part of the condition, so any negative minutes short-circuit to the raise, before the code can apply the allowed sign-combination rules tied to hours_offset.

Exactly what to change (step-by-step):

1. Validate hours_offset first (ensure -23 <= hours_offset <= 23).
2. Allow minutes in the full allowed numeric range: -59 <= minutes_offset <= 59.
   Replace the two-line check with a bounds check for -59..59.
3. Enforce the sign-combination rule: if hours_offset > 0 and minutes_offset < 0, raise ValueError("Minutes must be non-negative when hours is positive").
4. Compute the total offset from hours and minutes consistently (use absolute minutes added to absolute hours with sign determined by hours_offset or, when hours is zero, by minutes_offset sign).
5. Final check: ensure the resulting total offset does not exceed ±23:59:59.000.

Finally, check abs(total_minutes) <= 23*60 + 59 (and if you allow seconds/milliseconds, account for them).

This change matches the docstring: it allows negative minutes when hours are zero or negative, disallows negative minutes when hours are positive, and keeps minutes within [-59,59].
