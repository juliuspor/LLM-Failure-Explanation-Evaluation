What happened (what failed)
- The call is ArrayUtils.add(None, None, expected_type=str).
- In your code, the inferred_type calculation (lines 2–8) runs like this:
  - array is None, so the first branch is skipped.
  - element is None, so the second branch is skipped.
  - inferred_type becomes object.
- Later you hit the explicit check (lines 16–22 in the snippet):
  - if inferred_type == object and expected_type != object: raise TypeError(...)
- Because inferred_type is object and expected_type is str, the method raises the TypeError shown. That is exactly why the test fails instead of returning [None].

Why it fails (root cause)
- The function assumes it can infer the list element type from array or element. When both are None you fall back to object, then you treat object as a signal that a cast to the requested expected_type would be invalid and raise.
- But in the test the caller explicitly asked for expected_type=str and passed element None. When both array and element are None, it's valid to create a list of the expected_type and put None into it. Your code does not prefer expected_type when both array and element are None, so it incorrectly decides “object cannot be cast to str” and throws.

- inferred_type selection (your lines):

        inferred_type = type(array)
    elif element is not None:
        inferred_type = type(element)
    else:
        inferred_type = object
  This sets inferred_type = object for the failing test.
- The guard that throws (your lines):
    if expected_type is not None:

How to fix it (concrete steps and example patches)
You need to prefer expected_type when array and element give no concrete type, or generally let expected_type take precedence. Two simple, explicit fixes:

Option A — Prefer expected_type when provided
Replace the inferred_type block with:
    if expected_type is not None:
        inferred_type = expected_type
    elif array is not None:
        inferred_type = type(array)
    elif element is not None:
        inferred_type = type(element)
    else:
        inferred_type = object

This makes add(None, None, expected_type=str) set inferred_type to str and avoid the TypeError.

Option B — Only raise when actual runtime array type prevented the cast
If your intention was to reject a requested expected_type only when you actually created an untyped/object array from an existing array value, change the raising condition so it only triggers when array was provided (not when both array and element are None). For example:
    if expected_type is not None:
        if array is not None and inferred_type == object and expected_type != object:
            raise TypeError(...)

This lets callers create a new list of expected_type when they passed None for array and/or element.

Which to choose
- If expected_type is meant to be a hint or requested element type for the new list, use Option A.
- If expected_type is only a safety check against converting an existing array of object to a more specific type, use Option B.

Summary
Your code raises because inferred_type becomes object when both array and element are None, then your later check treats that as an illegal cast to expected_type=str and raises. Fix by preferring expected_type when available, or by only raising when there was an actual existing array that made inferred_type == object.
