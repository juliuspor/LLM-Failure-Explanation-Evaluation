What happened (root cause)
- The code treats -0.0 as an integer because x == int(x) is True for x == -0.0 (Python treats -0.0 equal to 0.0).
- At the start of add_number you compute value = int(x) and later call self.add(str(value)). For -0.0 that makes value == 0, so you add "0" (losing both the minus sign and the decimal part).
- The earlier guard if x < 0 and prev == '-' also does not run for -0.0 because -0.0 < 0 is False (again -0.0 compares equal to 0.0). So you neither preserve the sign there nor later.

Where this happens (lines / variables)
- In add_number(self, x: float):
  - The integer-branch check at line "if x == int(x) and not math.isinf(x) and not math.isnan(x):" catches -0.0.
  - value = int(x) sets value to 0.
  - self.add(str(value)) (the last line of that branch) writes "0" instead of "-0.0".
- The conditional "if x < 0 and prev == '-':" (just above) fails to catch -0.0 because x < 0 is False for -0.0.

Why this is the root cause (concise)
- Negative zero exists in IEEE floating-point; equality checks (==) and int() collapse -0.0 to 0.0/0, so formatting logic that relies on x == int(x) and x < 0 will lose the sign on -0.0.

Concrete fixes (step-by-step)
Option A — special-case negative zero early (recommended, simplest):
1. At the top of add_number, after prev = self.get_last_char(), add:
   if x == 0.0 and math.copysign(1.0, x) < 0:
       self.add("-0.0")
       return
   This uses math.copysign to detect the sign bit without changing x.

Option B — preserve sign when converting to integer:
1. Replace the integer branch final add with:
   if value == 0 and math.copysign(1.0, x) < 0:
       self.add("-0")
   else:
       self.add(str(value))
   (or use "-0.0" depending on desired exact output)
2. Also handle the earlier space-insertion: change x < 0 to math.copysign(1.0, x) < 0 so the "space before negative number" logic applies to -0.0.

Option C — avoid x == int(x) for deciding formatting:
1. Use explicit checks for exact integerness without collapsing sign, or prefer formatting with str(x) except when you need the integer shortening rules — but always detect negative zero with math.copysign if you want to preserve the sign.

Which to choose
- Use Option A if you only need to ensure "-0.0" appears exactly as in your test.
- Use Option B if you want the existing integer-formatting behavior but also preserve sign for zero.
- In all cases use math.copysign(1.0, x) < 0 to detect a negative sign on zero; don't rely on x < 0 or x == int(x) for that.
