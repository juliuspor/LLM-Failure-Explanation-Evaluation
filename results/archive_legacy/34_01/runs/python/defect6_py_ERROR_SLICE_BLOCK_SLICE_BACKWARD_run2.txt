What happened
- The code inserted an extra space between a minus sign and the numeric literal, so the output lost the negative sign when later processed. The break happens in add() at line 133.

Where it happened (exact lines)
- add(self, newcode) at lines 126–138; the decision to insert a space is on line 133:
  if (self.is_word_char(c) or c == '\\') and self.is_word_char(self.get_last_char()):
      self.append(" ")

Why that removes the minus sign
- Suppose the buffer currently ends with '-' and you call add("0.0") to append the number -0.0.
  - newcode[0] is '0' so c is a word character (self.is_word_char(c) is True).
  - get_last_char() returns '-', and is_word_char('-') also returns True (incorrectly).
  - Both sides of the condition are True, so the code inserts a space between '-' and '0' and then appends "0.0".
  - The result in the buffer becomes "- 0.0" (minus separated by a space). Downstream code that canonicalizes or prints the token sequence then treats '-' as a separate token (or removes an isolated unary minus) and you end up with "0" instead of "-0.0".
- In short: the code treats '-' as a "word character" so it splits '-' from the numeric token, losing the negative sign.

Root cause
- is_word_char('-') returns True. The condition at line 133 assumes is_word_char(last_char) means the last character is something that must be separated from the next token (letters/digits/underscore), but '-' is not such a character and should not be classified as a word char here.
