What happened (where): The code at add(newcode) tests whether the last character in the buffer is a "word char" before inserting a separating space (lines 132–136). The problematic call is self.is_word_char(self.get_last_char()) on line 133. get_last_char() can return the integer 0 rather than a one-character string when the buffer is empty or in some states.

Why that causes the failure (root cause): is_word_char expects a character (string) and treats falsy values like 0 as "not a word char". Because get_last_char returned the integer 0, the condition on line 133 evaluates false and the space that should separate tokens is not inserted. Two tokens are concatenated; that token merging changes how the numeric literal is parsed downstream and turns "-0.0" (the expected token) into 0 (the observed token). The assertion message "Expected -0.0, got 0" is the symptom of that mis-parsed/merged token.

Precise lines/variables involved:
- newcode (line 126–131): first character c = newcode[0] (line 132)
- check using self.is_word_char(self.get_last_char()) (line 133)
- when get_last_char() returns the integer 0, the check is wrong and no space is appended (line 136), causing token concatenation and the wrong numeric value later.
