Why that is incorrect / why it throws in normal use
- You tried to add element = None to a list that is supposed to contain strings (expected_type == str). Adding None must be allowed for a list of strings (None is a valid element value); the presence of None does not mean the list's element type is unknown in the sense that you must forbid a str list. But your code treats element is None as "no type information" and then refuses to create a typed list when expected_type is provided.
- The immediate cause of the thrown exception is the line that sets inferred_type = object when element is None, combined with the strict check that raises whenever inferred_type is object but expected_type is not object. That logic turns a None element into an error instead of accepting it as a valid (nullable) member of the expected_type list.

How to fix it (concrete steps)
1. Decide what inferred_type should represent. If you mean "element type", change how you compute it:
   - If element is not None: inferred_type = type(element)
   - Else if array is not None: infer the element type from array contents (for example, inspect first non-None element: next((type(e) for e in array if e is not None), object))
   - Else fall back to expected_type if provided, else object.

2. Do not treat element == None as a reason to forbid creating a typed list. If expected_type is provided, prefer expected_type when element is None.

3. Replace the current raise condition with a stricter check that only raises if you genuinely cannot satisfy expected_type:
   - For example, only raise if you discovered the array’s element type (from actual non-None members) is incompatible with expected_type.
   - Or, if you create a new list using expected_type (when expected_type provided), do not raise merely because element is None.

Why this fixes it
- Using type(element) when element is present avoids confusing list type with element type.
- Falling back to expected_type when you cannot infer from element or array avoids rejecting valid nullable additions (element None) to a typed list.
- Only raising on actual incompatibility (not mere absence of element type info) prevents false positives like the exception you saw.
