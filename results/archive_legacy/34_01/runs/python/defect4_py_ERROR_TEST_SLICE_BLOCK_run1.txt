What happened (what failed)
- The test expected get_max_middle_index() to return 1 (the period with middle = 150 ms).
- The code returned 3 (the last-added period with middle = 75 ms).

Why this happened (root cause)
- The stored index _max_middle_index is being overwritten when later items are added even though their middle time is smaller than the current maximum.
- That points to the comparison used when updating the max being inverted or non-strict. In other words, the code that updates _max_middle_index is using a condition that is true when it shouldn’t be (e.g. using <= or otherwise treating a smaller middle as if it were larger).
- Concretely, when SimpleTimePeriod(50,100) (middle = 75) was added, the update logic executed and set _max_middle_index = 3 despite the current recorded maximum middle being 150 from index 1.

Where it occurs (what to inspect)
- get_max_middle_index (lines 449–451) just returns self._max_middle_index, so the bug is in the code that sets self._max_middle_index (most likely the add(...) method).
- Look for code like:
    if middle <= self._max_middle_millis:
        self._max_middle_index = new_index
  or the equivalent reversed comparison. The condition should be middle > self._max_middle_millis (or a proper initialization check when no max exists).

Exact fix (step-by-step)
1. Find the add(...) method where you compute the period middle and update the stored max. It will compute something like:
       middle = (period.start + period.end) / 2
   and then compare middle to the current max.
2. Change the update condition to strictly greater:
       if self._max_middle_index is None or middle > self._max_middle_millis:
           self._max_middle_index = new_index
           self._max_middle_millis = middle
   - Use ">" not "<=", "<", or "<=".
   - Ensure you initialize _max_middle_index to None (or -1) and _max_middle_millis to None (or -inf) so the first item sets them.
3. Re-run the test:
       s = TimePeriodValues('Test')
       s.add(SimpleTimePeriod(0, 100), 1.0)
       s.add(SimpleTimePeriod(100, 200), 2.0)
       s.add(SimpleTimePeriod(0, 50), 3.0)
       s.add(SimpleTimePeriod(50, 100), 4.0)
       assert s.get_max_middle_index() == 1
   It should pass.

Why this exact change fixes it
- Using middle > current_max only updates the stored index when you encounter a strictly larger middle. That preserves the earlier index (1, middle=150) when later added middles are smaller (25 and 75), so get_max_middle_index() returns 1 as expected.

What to watch for
- If you want a deterministic tie-breaker when middles are equal, decide and document whether to keep the first or the last occurrence; implement >= only if you intentionally want to prefer a newly added equal value (but your test expects to keep the earlier max, so use >).
- Make sure both the index and the stored max middle value variables are updated together and correctly initialized.
