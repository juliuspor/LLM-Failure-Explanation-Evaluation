What happened (where it failed)
- The ValueError was raised at lines 396–397:
  ```
  396 |         if minutes_offset < 0 or minutes_offset > 59:
  397 |             raise ValueError(f"Minutes out of range: {minutes_offset}")
  ```
- That code rejects any negative minutes (minutes_offset < 0). The docstring, however, allows negative minutes in certain cases (for example (0, -15) -> -00:15 and (-2, -15) -> -02:15). So the check is too strict and incorrectly rejects valid inputs such as minutes_offset == -15 when hours_offset is zero or negative.

Why it failed (root cause)
- The check at lines 396–397 enforces minutes_offset to be non‑negative unconditionally. The docstring requires minutes_offset to be in the range -59..+59 (and additionally enforces sign rules that depend on hours_offset). Because the code tests only minutes_offset < 0, it throws "Minutes out of range: -15" even though -15 can be valid depending on hours_offset. In short: the guard conflates "out of absolute range" with "invalid sign combination".

Concrete fixes (step‑by‑step)
1. Enforce the absolute minutes limit, not an unconditional non‑negativity check:
   - Replace the unconditional check with one that tests the absolute range:
     if minutes_offset < -59 or minutes_offset > 59:
         raise ValueError(f"Minutes out of range: {minutes_offset}")
   - This ensures minutes are within -59..+59.

2. Enforce the sign combination rules the docstring describes:
   - After validating the absolute minute range, add the rule that positive hours cannot be paired with negative minutes:
     if hours_offset > 0 and minutes_offset < 0:
         raise ValueError("Minutes negative when hours positive: "
                          f"{hours_offset}, {minutes_offset}")
   - (No error for hours_offset == 0 and minutes_offset < 0 — that is allowed.)
   - You may also want a clear error if hours_offset is out of its own allowed range (-23..+23).

3. Ensure the final total offset does not exceed +/-23:59:59.000:
   - Compute total_seconds = hours_offset * 3600 + minutes_offset * 60
   - If abs(total_seconds) > (23 * 3600 + 59 * 60 + 59): raise ValueError(...)

Example corrected fragment (replace lines 396–397 and add the sign/total checks):
  if minutes_offset < -59 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")
  if hours_offset > 0 and minutes_offset < 0:
      raise ValueError(f"Minutes negative when hours positive: {hours_offset}, {minutes_offset}")
  if hours_offset < -23 or hours_offset > 23:
      raise ValueError(f"Hours out of range: {hours_offset}")
  total_seconds = hours_offset * 3600 + minutes_offset * 60
  if abs(total_seconds) > (23*3600 + 59*60 + 59):
      raise ValueError("Resulting offset exceeds +/-23:59:59.000")

Why this exact change fixes it
- It stops rejecting all negative minutes (fixing the -15 case) while still enforcing that minutes are within -59..+59.
- It enforces the docstring rule that minutes cannot be negative when hours are positive.
- It prevents producing offsets outside the allowed +/- 23:59:59 limit.

Where to change it
