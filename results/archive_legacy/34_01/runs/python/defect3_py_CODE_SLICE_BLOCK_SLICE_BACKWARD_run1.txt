What happened (where it failed)
- The exception is raised in code_point_at(code_units, pos), which checks the index and throws IndexError when index < 0 or index >= len(code_units). The stack fragment shows that check at lines 35–36 inside to_utf16_code_units / code_point_at.
- That call happens on this line in translate (near the end of your snippet):
  for pt in range(consumed):
      pos += char_count(code_point_at(code_units, pos))
- The immediate symptom: code_point_at was called with pos == len(code_units) (or otherwise out of range), so the index check failed.

Why it happened (root cause)
- The loop that advances pos after a translator returned consumed uses code_point_at(code_units, pos) repeatedly for each value in range(consumed).
- The contract between translate and translate_codepoint is inconsistent: translate treats consumed as “how many code points the translator consumed” and then tries to advance pos by re-reading code points starting at the current pos. But translate_codepoint (your translators) actually report a consumed value measured in code units (UTF-16 code units / char[] positions), not in code points — or vice versa. Because of that mismatch the loop runs the wrong number of iterations or advances pos incorrectly.
- Concretely, if consumed is measured in code units but the loop treats it as count of code points, the loop will call code_point_at more times than there are code units left, eventually calling code_point_at with pos == len(code_units) and triggering the IndexError at lines 35–36.
- Or the opposite: if consumed is in code points but you advance pos by char_count for each and the translator actually already advanced over code units differently, you can skip ahead incorrectly and end up with pos out-of-range on the next code_point_at call. Either way the mismatch between the meaning of consumed and the way you advance pos causes pos to move incorrectly and to be used out of bounds.

Summary
The crash is caused by a mismatch between what translate_codepoint reports in consumed and how translate advances pos. That mismatch leads to code_point_at being called with pos == len(code_units), triggering the IndexError at lines 35–36. Fix by deciding whether consumed counts code units or code points, make that explicit, and advance pos consistently (recommended: if consumed is code units then do pos += consumed).
