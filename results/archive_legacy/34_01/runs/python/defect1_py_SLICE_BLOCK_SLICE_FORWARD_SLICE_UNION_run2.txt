What fails
- The code raises ValueError("Minutes out of range: {minutes_offset}") at lines 396–397 because minutes_offset has a value outside the allowed range when it is validated.

Why it fails (root cause)
- The code applies the sign/combination logic that makes minutes_offset negative/adjusted after the range check. In other words, the sign adjustment that happens later at lines 402 and 404 (minutes_offset = hours_in_minutes - minutes_offset or minutes_offset = hours_in_minutes + minutes_offset) is not performed before the validation at lines 396–397.
- As a result, minutes_offset can already be negative (or otherwise outside 0..59) by the time the check runs, and the check only allows 0..59. That mismatch between when the sign is applied and when the range is enforced causes legitimate inputs to be rejected.

Where it occurs
- Validation happens at lines 396–397:
  - 396: if minutes_offset < 0 or minutes_offset > 59:
  - 397:     raise ValueError(f"Minutes out of range: {minutes_offset}")
- Sign/combination logic that should affect minutes_offset is later at lines 402 and 404:
  - 402: minutes_offset = hours_in_minutes - minutes_offset
  - 404: minutes_offset = hours_in_minutes + minutes_offset

What happened in practice
- Suppose the parsed minutes value has already been assigned a sign (negative for negative offsets) or will be combined with hours to produce a negative total. Because the code checks for minutes_offset < 0 too early, it rejects a valid negative minutes component (or otherwise adjusted minutes) instead of letting the later combination produce a valid total offset.

How to fix it (explicit, step-by-step)
Option A — validate magnitude before sign application
1. Change the range check to validate the magnitude of minutes without sign. Replace the condition at line 396 with:
   - if abs(minutes_offset) > 59:
       raise ValueError(f"Minutes out of range: {minutes_offset}")
   This accepts minute values in -59..59 and prevents rejecting a negative minute value that will be combined later.

Option B — apply sign/combination before validating minutes
1. Move the sign/combination lines (402 and 404) so they run before the range check at lines 396–397.
2. Keep the existing check but it will then validate the final minutes_offset value (which will be in the expected 0..59 after combination or produce a correctly rejected case).

Which to choose
- If minutes_offset is supposed to be a signed component (e.g., parsed as -30 for "-01:30"), use Option A (abs check).
- If minutes_offset should be an unsigned 0..59 until combined with hours, use Option B (move the sign/combination earlier) so the check validates the raw minutes and the sign is applied afterward.

Be explicit about the change location
- Edit the code around line 396 (the range check) and lines 402/404 (the lines that adjust minutes_offset). Either change the condition at 396 to use abs(...) or move 402/404 to before 396 so the validation occurs against the final intended minutes_offset. After that, offset calculation at line 405 and subsequent steps will operate on a correctly validated value.
