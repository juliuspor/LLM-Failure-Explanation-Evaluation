What happened (where it fails)
- In GrayPaintScale.get_paint (method header at line 54), g is computed at line 71:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- Immediately after (lines 74–75) the code checks
  if g < 0 or g > 255:
      raise ValueError(...)
- For value = -0.5 the computed g is negative, so g < 0 is true and the method raises ValueError instead of returning a black color. That is why the test
  gps.get_paint(-0.5); assert color == (0, 0, 0)
  fails — you get an exception, not (0,0,0).

Why it fails (root cause)
- The implementation raises a ValueError whenever the computed intensity g is outside [0,255].
- The test expects out-of-range inputs to be clamped to the nearest valid grayscale value (i.e., values below the lower bound should produce (0,0,0)), but the code enforces a strict error instead of clamping. The mismatch between expected behavior and the code path at lines 71–75 is the direct cause.

Exact places to change and how (step-by-step fixes)
Option A — Clamp the value (recommended if behavior should return nearest valid color)
1. Replace the raise check at lines 74–75 with clamping logic. For example:
   g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
   g = max(0, min(255, g))
2. Then return (g, g, g).
3. This will make get_paint(-0.5) produce (0,0,0) instead of raising.

Option B — Only raise on invalid numeric input (if you still want errors for non-finite results)
1. After computing g, check for non-finite results (NaN/inf) instead of simple out-of-range:
   if not math.isfinite(g):
       raise ValueError("computed color is not finite")
2. Otherwise clamp as in Option A.
