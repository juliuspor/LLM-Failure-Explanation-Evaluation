The bug is double-counting pos advancement after a translator consumes code units. Simplest fix: when consumed != 0, advance pos by the number of UTF-16 code units consumed (consumed) rather than looping and calling code_point_at/char_count. Also keep other behavior unchanged: on consumed == 0, copy current codepoint to out and advance by char_count of that codepoint. Implemented by replacing the for-loop with pos += consumed.
