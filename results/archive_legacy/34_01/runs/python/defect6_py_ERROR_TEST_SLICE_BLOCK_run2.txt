What happened
- Your test calls cc.add_number(-0.0). The code that turns a numeric value into text treats zero by numeric equality (e.g. if number == 0: newcode = '0').
- In Python (and most languages that follow IEEE-754), -0.0 == 0 is True, so that branch runs for -0.0 and replaces the representation with "0".
- As a result cc.get_code() returns "0" (no minus sign, no decimal) instead of the original textual "-0.0". That is why the assertion fails: expected "-0.0", got "0".

Where it occurred
- The call came from add_number (not shown in the quoted slice). The shown add(self, newcode) method at lines 126–138 then appends the string created by add_number. Because add_number already produced "0" for -0.0, add(...) appends "0" and get_code() yields "0".
- Lines to look at: the add_number implementation (the branch that special-cases zero) and the usage of newcode inside add at line 138 (self.append(newcode)). The bug originates in add_number’s normalization of numeric zero, not in add().

Why that code path causes the bug
- The code used a numeric equality test (number == 0) to decide how to represent zero. That test cannot distinguish +0.0 from -0.0, so the sign is lost before add() ever sees the value.

How to fix it (concrete steps)
1. Locate add_number (the method that converts the float to a string). Find the branch that looks like:
   if number == 0:
       newcode = '0'
   (or similar)
2. Remove or refine that special-case. Options:
   - Use a string conversion that preserves the sign by default, e.g. newcode = repr(number) (in Python repr(-0.0) -> '-0.0') or str(number) which also preserves '-0.0'.
   - If you must special-case zeros, detect sign explicitly using math.copysign or 1.0/number:
       import math
       if number == 0.0:
           if math.copysign(1.0, number) < 0.0:
               newcode = '-0.0'
           else:
               newcode = '0.0'  # or '0' if you want plain zero
   - Alternatively use format that always shows the sign when needed, e.g. format(number, '.17g') or repr(number) to avoid accidental normalization.
3. Add a unit test for negative zero:
   cc = MockCodeConsumer()
   cc.add_number(-0.0)
   assert cc.get_code() == '-0.0'
   and optionally for +0.0.

Why these fixes work
- They avoid using equality alone to decide representation and either use a text conversion that preserves the original sign or explicitly inspect the sign bit (math.copysign or reciprocal test). That keeps "-0.0" as "-0.0" instead of collapsing it to "0".

Summary
The root cause is the numeric equality check (number == 0) in add_number, which treats -0.0 the same as 0. Fix by using a string conversion that preserves sign (repr or str) or by checking the sign bit (math.copysign) before normalizing zero.
