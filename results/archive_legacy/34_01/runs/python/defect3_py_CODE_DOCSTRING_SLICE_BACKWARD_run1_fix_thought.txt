The bug arises in CharSequenceTranslator.translate when consumed != 0; the loop advancing pos calls code_point_at(pos) without checking bounds leading to IndexError. Simpler fix: interpret consumed as number of UTF-16 code units consumed and advance pos by consumed. That is safe because translate_codepoint functions in code (AggregateTranslator and CsvEscaper) return counts in code units (CsvEscaper returned 2 for surrogate pair, 1 otherwise). So replace the for-loop with pos += consumed. Implement updated function accordingly.
