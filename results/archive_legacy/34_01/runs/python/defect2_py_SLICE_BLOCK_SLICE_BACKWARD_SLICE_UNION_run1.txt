What happens (where it fails)
- The failure occurs inside get_paint (line 71) on this expression:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- If self._upper_bound == self._lower_bound, the denominator is zero and Python raises a ZeroDivisionError before you ever reach the range check at line 74.

Why it happens (root cause)
- The code assumes upper and lower bounds are different. When they are equal, the division (self._upper_bound - self._lower_bound) is 0 and you get a ZeroDivisionError at line 71.
- The subsequent check (lines 74–75) and the ValueError message ("Color parameter outside of expected range: Red Green Blue") never run because the exception is raised earlier.
- The error message is also misleading: it mentions "Red Green Blue" but the computed variable here is g (green), and it does not report the offending value, bounds, or which channel failed.

How to reproduce
- Call get_paint with any value when self._upper_bound == self._lower_bound (for example both 0). The code will raise ZeroDivisionError at line 71.

Concrete fixes (exact steps and code snippets)
1) Prevent division by zero by checking bounds before computing g:
   - At the start of get_paint, add:
     if self._upper_bound == self._lower_bound:
         raise ValueError(f"Invalid bounds in get_paint: lower_bound ({self._lower_bound}) == upper_bound ({self._upper_bound})")

   This makes the failure explicit and occurs at the right place.

2) Optionally, if equal bounds are an expected situation with a defined behavior (e.g., treat everything as 0 or 255), handle it instead of raising:
   - Example to map all values to 0:
     if self._upper_bound == self._lower_bound:
         g = 0
     else:
         g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

3) After computing g, clamp it and give a clear error message (replace the misleading message):
     g = int(...)
     if g < 0 or g > 255:
         raise ValueError(f"Computed green channel out of range: g={g}, value={value}, bounds=({self._lower_bound},{self._upper_bound})")
   Or simply clamp instead of raising:
     g = max(0, min(255, g))

Summary
- Root cause: division by zero at line 71 when upper_bound == lower_bound.
- Fix: check or handle equal bounds before doing the division and improve the error message to include g, value, and the bounds.
