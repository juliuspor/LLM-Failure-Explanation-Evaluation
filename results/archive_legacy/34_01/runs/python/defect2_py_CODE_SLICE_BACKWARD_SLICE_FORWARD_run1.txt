What happened (where it fails)
- At line 54 you clamp the incoming value into local variable v:
  - v = max(value, self._lower_bound)
  - v = min(v, self._upper_bound)
- At line 71 you compute g using the original value variable, not the clamped v:
  - g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- Because you used value instead of v, a value outside the bounds can produce g < 0 or g > 255 and trigger the ValueError at line 74.

Why it happened (root cause)
- The clamp was applied to v but never used. The calculation uses the un-clamped value, so the check that follows can legitimately fail even though you intended to clamp first.
- There is a secondary edge-case: if self._upper_bound == self._lower_bound, the denominator is zero and the division will raise a ZeroDivisionError. That error is separate from the clamping bug but should be handled.

Minimal corrected example

        raise ValueError("upper_bound equals lower_bound; cannot normalize value")

Why this fix works
- Using v ensures the normalized value is inside the intended bounds and g will be within [0,255] unless the bounds or inputs are invalid.
- The zero-division check prevents a runtime crash when bounds are equal and makes the failure explicit.
