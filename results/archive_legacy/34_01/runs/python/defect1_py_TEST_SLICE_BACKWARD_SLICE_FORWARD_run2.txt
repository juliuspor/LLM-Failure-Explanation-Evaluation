What happened
- The test calls DateTimeZone.for_offset_hours_minutes(-2, -15) expecting an id of "-02:15".
- The code immediately validates minutes_offset at lines 396–397:
  - 396: if minutes_offset < 0 or minutes_offset > 59:
  - 397:     raise ValueError(f"Minutes out of range: {minutes_offset}")
- Because minutes_offset is -15, the condition minutes_offset < 0 is true and a ValueError is raised. The method never reaches the later lines (402–405) that adjust minutes based on a negative hour.

Why this is the root cause
- The code treats a negative minutes argument as invalid, while the test supplies negative minutes to indicate a negative offset (hours = -2, minutes = -15).
- Later code (lines 402–405) already contains logic that expects to adjust minutes when hours are negative, but that logic never runs because negative minutes are rejected earlier at lines 396–397.
- In short: the validation at lines 396–397 rejects a legitimate representation (-2, -15) before the sign-handling code can apply.

Where it occurs (exact references)
- Validation rejecting negative minutes: lines 396–397.
- Sign/adjustment logic that would create -02:15 if minutes were allowed: lines 402–405.
- Test call: DateTimeZone.for_offset_hours_minutes(-2, -15) (from [TEST]).

How to fix it (concrete steps)
1. Accept negative minutes by normalizing minutes to a non-negative value before validating the 0–59 range. For example, immediately after reading minutes_offset, replace it with its absolute value:
   - Insert: minutes_offset = abs(minutes_offset)
2. Keep the range check on the normalized minutes:
   - if minutes_offset > 59:
       raise ValueError(f"Minutes out of range: {minutes_offset}")
3. Leave the later sign-handling code (lines 402–405) as-is; it will perform the correct combination of hours and minutes when hours is negative.
