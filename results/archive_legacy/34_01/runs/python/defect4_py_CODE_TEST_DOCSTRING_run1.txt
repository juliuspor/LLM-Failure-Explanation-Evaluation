What happened (what you saw)
- The test expects the maximum-middle index to be 1 (the period [100,200]).
- The code did not produce that result because the max-middle comparison used the wrong previously-stored index, so the wrong period was compared and _max_middle_index was not updated as expected.

Where it happens (exact lines / variables)
- In the block that updates _max_middle_index you call:
  max_middle_period = self.get_data_item(self._min_middle_index).get_period()
  (this is just after the comment for the max-middle block).
- That line uses self._min_middle_index instead of self._max_middle_index. As a result:
  - s and e are derived from the period at index self._min_middle_index, not the period at self._max_middle_index.
  - max_middle is computed from the wrong period.
  - The comparison if middle > max_middle: may be comparing against an unrelated value, so self._max_middle_index is not set to the correct index (1) when it should be.

Why it happens (root cause)
- Simple: a copy-paste / typo error. Every other block uses the corresponding stored index (e.g. _min_start_index vs _max_start_index), but here the max block still references the min index variable. That causes the algorithm to compare the new period's middle to the wrong existing period's middle and therefore incorrectly decide whether to update _max_middle_index.

How to fix it (specific, step-by-step)
1. Change the single incorrect line:
   - Replace
     max_middle_period = self.get_data_item(self._min_middle_index).get_period()
   - With
     max_middle_period = self.get_data_item(self._max_middle_index).get_period()

2. Run the unit test again:
   - s = TimePeriodValues('Test')
     s.add(SimpleTimePeriod(0, 100), 1.0)
     s.add(SimpleTimePeriod(100, 200), 2.0)
     s.add(SimpleTimePeriod(0, 50), 3.0)
     s.add(SimpleTimePeriod(50, 100), 4.0)
     assert s.get_max_middle_index() == 1

3. Optionally run your full test suite to ensure no other blocks have a similar copy-paste error.

Why this fix resolves the problem
- After the change, the code compares the new period's middle to the current maximum-middle period (the one stored at _max_middle_index), so _max_middle_index will be updated correctly when the new period's middle is larger.

Notes / sanity checks
- All other blocks in the method already use the correct _min_/ _max_ index variables; only the max-middle block had the wrong variable.
- Ensure _max_middle_index is initialized (presumably to -1) so the else branch sets it on first use — your current logic already handles that.
