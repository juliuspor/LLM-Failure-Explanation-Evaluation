What happened (where it failed)
- The call DateTimeZone.for_offset_hours_minutes(-2, -15) passed minutes_offset = -15.
- At lines 396–397 the code checks:
  ```
  if minutes_offset < 0 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")
  ```
  That check raised ValueError: "Minutes out of range: -15".

Why it failed (root cause)
- The method expects minutes_offset to be in the range 0..59. It does not accept a negative minutes value.
- You gave -15 because you intended the whole offset to be negative (-2 hours, 15 minutes). The API expects the sign to be carried in the hours parameter (hours = -2) and the minutes parameter to be non-negative (minutes = 15). Passing negative minutes breaks the minutes validation and triggers the exception at line 396.

How to fix it (concrete steps)
You have two clear options:

Option A — fix the caller (recommended)
- Call the method with a non-negative minutes argument:
  DateTimeZone.for_offset_hours_minutes(-2, 15)
- Rationale: hours = -2 already marks the offset negative; minutes must be 0..59.

Option B — change the method to accept negative minutes and normalize (if you want the API to accept both signs)
- Replace the current validation and normalization at/around lines 396–397 with logic that:
  1. Compute sign = -1 if hours < 0 or minutes_offset < 0, else +1.
  2. Use abs_minutes = abs(minutes_offset).
  3. Validate abs_minutes <= 59 (raise if > 59).
  4. Compute total_minutes = hours * 60 + sign * abs_minutes (or better: total_minutes = hours*60 + sign*abs_minutes AFTER making hours the absolute number if necessary).
  5. Build the zone from total_minutes.
- Example patch sketch:
  ```
  sign = -1 if hours_offset < 0 or minutes_offset < 0 else 1
  abs_minutes = abs(minutes_offset)
  if abs_minutes > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")
  total_minutes = hours_offset * 60 + sign * abs_minutes
  # then use total_minutes to construct the zone id and value
  ```
- This change lets callers pass either (-2, -15) or (-2, 15) and produces the same result, but it is a behavior change and should be covered by tests.

Summary
- The exception was raised at lines 396–397 because minutes_offset = -15 is outside the accepted 0..59 range.
- Fix easiest: call DateTimeZone.for_offset_hours_minutes(-2, 15).
- If you want to accept negative minutes at the API level, implement the normalization steps shown above and update tests.
