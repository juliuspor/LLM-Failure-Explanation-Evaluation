Why it fails (root cause)
- translate_codepoint(...) returns a non-zero consumed value to tell the caller how many codepoints the translator handled.
- Your loop then calls code_point_at(code_units, pos) to compute how many UTF-16 code units to advance (char_count result). But code_point_at may itself need to read a low surrogate at pos+1 when the codepoint is a surrogate pair.
- If consumed > 0 and pos points to the final code unit (the last index), code_point_at(code_units, pos) tries to inspect pos+1 and raises IndexError because pos+1 == len(code_units) is out of bounds. That is exactly what the error message (index 2 out of range) indicates.
- In short: you asked code_point_at to read a codepoint at a position where there is no next code unit available, because you rely on translate_codepoint to already have consumed the pair. The loop then redundantly attempts to inspect the same code units and steps past the array boundary.

How to fix it (concrete, specific steps)

Option A — If translate_codepoint returns the number of UTF-16 code units consumed (1 or 2):
- Replace the entire for-loop with:
    pos += consumed
- Why: consumed already tells you how many code_units were consumed, so you should advance pos by that number. This avoids calling code_point_at at the boundary.

Option B — If translate_codepoint returns the number of codepoints consumed (not code units) and you must compute code units to advance:
- Change the loop so it advances by single code units without calling code_point_at on a possibly incomplete surrogate pair. For example:

- If you actually need to skip full codepoints (i.e., skip combined surrogate pairs correctly), compute the code unit count only when you know a full codepoint is available. A safe approach:
