What happened
- The ValueError is raised at lines 396–397:
  - 396: if minutes_offset < 0 or minutes_offset > 59:
  - 397:     raise ValueError(f"Minutes out of range: {minutes_offset}")
- That check rejects any negative minutes_offset (e.g. -15) immediately, so the call fails with "Minutes out of range: -15".

Why this is wrong (root cause)
- The docstring explicitly allows negative minutes in some combinations:
  - (0, -15) should be allowed (yielding -00:15).
  - (-2, -15) should be allowed (yielding -02:15).
  - Only the combination (positive hours, negative minutes) is invalid.
- The code’s validation at lines 396–397 is too strict: it disallows negative minutes in all cases instead of only disallowing them when hours_offset is positive. Because of that, legitimate inputs such as hours=0, minutes=-15 or hours=-2, minutes=-15 are rejected before the sign-combining logic at lines 402/404 can run.

Where the allowed combinations would be handled
- Lines 402 and 404 appear intended to adjust minutes_offset based on hours_in_minutes and the sign of minutes_offset:
  - 402: minutes_offset = hours_in_minutes - minutes_offset
  - 404: minutes_offset = hours_in_minutes + minutes_offset
- Those lines are never reached for negative minutes because of the early rejection at 396–397.

Concrete fixes (step-by-step)
1. Broaden the numeric range check so negative minutes are allowed within [-59, 59]:
   - Replace:
     if minutes_offset < 0 or minutes_offset > 59:
         raise ValueError(f"Minutes out of range: {minutes_offset}")
   - With:
     if minutes_offset < -59 or minutes_offset > 59:
         raise ValueError(f"Minutes out of range: {minutes_offset}")

2. Add a specific check that forbids negative minutes when hours_offset is positive (per docstring):
   - After the updated range check, add:
     if hours_offset > 0 and minutes_offset < 0:
         raise ValueError("Positive hours with negative minutes are not allowed")

3. Keep or ensure the later sign-combining logic (lines 402/404) runs for the remaining cases:
   - If hours_offset < 0 and minutes_offset < 0, the code should produce the combined negative offset (use line 402/404 as intended).
   - If hours_offset == 0 and minutes_offset < 0, treat that as a negative offset (combine accordingly).

Why this fixes it
- Step 1 prevents rejecting valid negative-minute inputs (like 0, -15 or -2, -15).
- Step 2 enforces the one invalid combination described in the docstring (positive hours with negative minutes).
- Steps 1–3 let the later adjustment logic at lines 402/404 compute the correct total minutes and offset without being short-circuited by the overly strict early check.

Example expectations after the fix
- (2, 15) => +02:15 (allowed)
- (2, -15) => raises ValueError (explicitly checked in step 2)
- (0, -15) => -00:15 (allowed)
- (-2, -15) => -02:15 (allowed)

Exact places to change
- Modify the condition at lines 396–397 as described.
- Insert the hours-positive/minutes-negative check immediately after that range check, before line 402.
