What happened
- The input string is "fr__POSIX".
- to_locale (in this file at/around lines 231–240) treats underscores as separators and expects the country code to occupy locale_str[3] and locale_str[4].
- At lines 237–240 the code does:
  - ch3 = locale_str[3]
  - ch4 = locale_str[4]
  - if ch3 < 'A' or ch3 > 'Z' or ch4 < 'A' or ch4 > 'Z': raise ValueError(...)
- For "fr__POSIX", locale_str[3] is '_' (because the string has two underscores in a row), so the test fails and you get ValueError: Invalid locale format: fr__POSIX.

Why it fails (root cause)
- The implementation assumes either:
  - language only ("fr"), or
  - language + '_' + 2-letter COUNTRY ("fr_FR"), or
  - language + '_' + COUNTRY + '_' + variant ("fr_FR_POSIX").
- It does not handle the legal variant form where the country is empty and the variant follows a double underscore ("fr__POSIX").
- In other words, the code requires uppercase letters at positions 3 and 4 when a separator exists, but in the double-underscore form position 3 is '_' (empty country) and position 4 is the first variant character. The code therefore misidentifies a valid "language__variant" as invalid.

Exactly where this occurs

- File/lines referenced in your slice: around lines 231–240
- Problematic lines: 237–240 (ch3/ch4 extraction and the uppercase letter check)
