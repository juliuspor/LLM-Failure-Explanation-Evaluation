What happened (where it fails)
- The check if x is an integer (the line "if x == int(x) and not math.isinf(x) and not math.isnan(x):") treats -0.0 as an integer-equal value because -0.0 == 0 is True.
- You then do value = int(x) (line that assigns value = int(x)). int(-0.0) produces 0, losing the negative sign.
- Later you call self.add(str(value)) (the branch for small exponents), which adds "0" instead of "-0.0".
- The assertion fails because the caller expected the negative zero sign ("-0.0") but the code produced "0".

Why it happens (root cause)
- IEEE floats have a signed zero: -0.0 is distinct from +0.0, but numeric comparisons treat them as equal. The code uses x == int(x) and int(x) to decide integer form, which erases the sign for -0.0.
- The earlier guard "if x < 0 and prev == '-':" also does not detect -0.0 because x < 0 is False for -0.0, so that path doesn't fix the output either.

Exact variables and lines involved
- x: the input float (example -0.0)
- Line: "if x == int(x) and not math.isinf(x) and not math.isnan(x):" — detects integer-like floats including -0.0
- Line: "value = int(x)" — converts -0.0 to 0, losing the sign
- Line: "self.add(str(value))" — emits "0" instead of "-0.0"

How to fix it (concrete steps)
1. Detect negative zero before converting to int. Use math.copysign or compare x == 0.0 and check sign:
   - if x == 0.0 and math.copysign(1.0, x) < 0: treat as negative zero.
2. When you detect negative zero, output the correct representation ("-0.0" if that is what you expect) and return early from add_number.
3. Only convert to int and drop sign for non-zero values.

Minimal patch example (insert near top of method, before integer branch):

    if x == 0.0 and math.copysign(1.0, x) < 0:
        self.add("-0.0")

Summary
