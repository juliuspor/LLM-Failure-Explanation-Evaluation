What happened
- The call DateTimeZone.for_offset_hours_minutes(-2, -15) failed because the method treats -15 as an invalid minute value.
- The thrown error is ValueError: Minutes out of range: -15.

Why it failed (root cause)
- Inside DateTimeZone.for_offset_hours_minutes the code validates minutes with a check that rejects any negative minute value unconditionally. For example it likely does:
  if minutes_offset < 0 or minutes_offset > 59:
      raise ValueError("Minutes out of range: " + str(minutes_offset))
- That check treats -15 as out of range even though the docstring explicitly allows negative minutes when hours is negative (see the table: (-2, -15) should produce -02:15).
- In short: the minute-range validation is too strict — it forbids negative minutes regardless of the hours sign, so the valid case (-2, -15) is rejected.

Where it occurs
- In the factory method DateTimeZone.for_offset_hours_minutes (the input-validation section that checks minutes_offset).

How to fix it (concrete steps)
1. Replace the unconditional negative-minute rejection with a validation that accepts negative minutes when allowed. Two correct approaches:

  A. Check absolute range and then enforce sign consistency:
     - Validate minutes magnitude:
         if abs(minutes_offset) > 59:
             raise ValueError("Minutes out of range: " + str(minutes_offset))
     - Then disallow the documented invalid combination (positive hours with negative minutes):
         if hours_offset > 0 and minutes_offset < 0:
             raise ValueError("Minutes must be positive when hours are positive")
     - (Other combinations: hours < 0 with minutes < 0 is allowed; hours == 0 can have either sign.)

  B. Or explicitly check the allowed interval depending on hours:
     - if hours_offset > 0 and not (0 <= minutes_offset <= 59): raise
     - if hours_offset < 0 and not (-59 <= minutes_offset <= 0): raise
     - if hours_offset == 0 and not (-59 <= minutes_offset <= 59): raise

2. When computing the total offset seconds, combine hours and minutes with their intended signs. For example:
     total_seconds = hours_offset * 3600 + minutes_offset * 60
   (This works once minutes_offset is allowed to be negative for negative hours.)

3. Add tests for the edge cases in the docstring:
   - for_offset_hours_minutes(-2, -15) -> id == "-02:15"
   - for_offset_hours_minutes(2, -15) -> raises ValueError
   - for_offset_hours_minutes(0, -15) -> id == "-00:15"

Why these steps fix it
- They stop rejecting valid negative minute values that were intended to represent negative offsets, while still catching true out-of-range inputs (minutes magnitude > 59) and the documented invalid sign combos (positive hour with negative minutes).
