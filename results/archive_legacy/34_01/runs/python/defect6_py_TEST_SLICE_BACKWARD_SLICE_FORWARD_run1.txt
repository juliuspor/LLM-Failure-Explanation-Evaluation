- The test calls cc.add_number(-0.0).
- The code path that turns that float into text loses the negative sign, so cc.get_code() returns "0.0" instead of "-0.0".
- The loss of the sign happens before add() and append() — those methods (add at line 126 and append at line 138) only accept and concatenate the string you give them; they do not change sign. That means the root cause is the float→string conversion inside add_number (the place that constructs newcode).

Where it occurred (file/lines/variables/methods)
- add_number(value) is the method that produces newcode for the consumer.
- add(self, newcode) at line 126 simply forwards the string.

Why it happened (root cause)
- -0.0 is a distinct IEEE-754 value that carries a negative sign bit but compares equal to 0.0. Many common formatting functions or custom format code normalize zero values to "0.0" (or "0") and drop the sign bit.
- add_number uses a formatting/conversion routine that normalizes negative zero to the same text as positive zero. Because add/append only append that text, the final result loses the negative sign.
