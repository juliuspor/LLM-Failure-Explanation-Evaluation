What happened
- The getter get_max_middle_index() (lines 449–451) simply returns the stored attribute self._max_middle_index.
- That attribute is maintained by the add(...) logic when you insert new TimePeriod entries.
- In your sequence you first call:
  1) add(SimpleTimePeriod(0, 100), 1.0)
  2) add(SimpleTimePeriod(100, 200), 2.0)
  After these two entries the internal index tracking (self._max_middle_index) is correct for the two-item list.
- Then you insert two earlier periods:
  3) add(SimpleTimePeriod(0, 50), 3.0)
  4) add(SimpleTimePeriod(50, 100), 4.0)
  Those two additions are inserted before the existing entries (they are earlier in time), so they shift the positions of existing items in the internal list.
- The bug: the add(...) code updates or increments self._max_middle_index without taking into account that items can be inserted at an earlier index (list.insert). As a result self._max_middle_index becomes inconsistent with the actual list order and points to the wrong entry by index. get_max_middle_index() simply returns that stale/wrong number, so the assert fails.

Why this is the root cause (plain language)
- The code assumes adding always appends at the end (so indexes do not change) or it updates the index incorrectly for insert-at-front cases.
- But you did inserts that change the indices of elements that were already present. The code that maintains self._max_middle_index does not adjust correctly for that shift.
- Because get_max_middle_index() returns the stored attribute (line 451), you see the wrong index instead of the correct current index.

Where to look in the code
- The getter (lines 449–451): return self._max_middle_index — this just exposes the stored value.
- The add(...) method (the place that sets or mutates self._max_middle_index). Review the branch that handles insertion (list.insert or bisect insertion). That is where the index bookkeeping is wrong.

1. Short-term patch (minimal change)

     - If insert_idx <= self._max_middle_index: increment self._max_middle_index by 1 (or otherwise adjust by number of inserted items before that index).

     - if hasattr(self, "_max_middle_index") and insert_idx <= self._max_middle_index:
