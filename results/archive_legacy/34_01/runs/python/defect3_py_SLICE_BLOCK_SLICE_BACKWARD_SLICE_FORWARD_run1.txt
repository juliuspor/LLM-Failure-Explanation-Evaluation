Why it fails (root cause)
- to_utf16_code_units(s: str) is producing the wrong kind of list in code_units. Instead of a list of UTF‑16 16‑bit code units (values in 0x0000..0xFFFF, with surrogate pairs split into two entries), it produces entries that are full Unicode code points (ord(c)). For characters above U+FFFF that should be represented by two 16‑bit code units, your code_units ends up with a single integer > 0xFFFF.
- As a result:
  - len(code_units) is smaller than the number of UTF‑16 code units the rest of the logic expects, so index checks (line 35) can fail incorrectly (IndexError), or later code that looks for a low surrogate at index+1 finds there is no following 16‑bit unit to read.
  - The subsequent surrogate logic (lines 38–44) expects two 16‑bit units (high and low). Because code_units contains single full code points instead, you either hit the index check (line 35) or skip the surrogate handling and return a wrong value.

- The mismatch occurs inside to_utf16_code_units (lines shown in the backward slice). The check and the surrogate-handling code (lines 35–44) operate on the list returned by to_utf16_code_units, but that function is not actually producing UTF‑16 code units.
