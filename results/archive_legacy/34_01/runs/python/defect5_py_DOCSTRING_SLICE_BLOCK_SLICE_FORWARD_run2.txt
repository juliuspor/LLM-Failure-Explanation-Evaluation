What happened (where it fails)
- The failure comes from this check at lines 346–351:
  - Condition: inferred_type == object and expected_type != object
  - Action: raise TypeError("Cannot cast object list to {expected_type} list ...")
- That means: if the code infers the element type as object but the caller asked for a more specific expected_type, the function throws a TypeError instead of returning a list.

Why it happens (root cause)
- inferred_type becomes object whenever the code cannot determine a more specific element type. That happens in common situations such as:
  - array is None and element is None (no information to infer type)
  - array contains only None values (no concrete element to infer)
  - element is None but expected_type was provided
- The code then treats "inferred_type == object" as an unrecoverable mismatch and refuses to produce a list of the requested expected_type, raising the TypeError. In other words, the function confuses "unknown element type" with "incompatible element type" and rejects valid calls where the element is None or type inference is impossible.

Why this is wrong / surprising
- The docstring promises: "If the input list is None, a new one element list is returned whose component type is the same as the element, unless the element itself is None, in which case the return type is List[object]."
- But if the caller supplies expected_type explicitly (for example expected_type=MyClass) and the element is None, the implementation throws instead of honoring expected_type or returning a List[object] as described. The code treats unknown/inferred==object as a cast error even when expected_type was supplied or when None is a valid element.
