Why it happened (root cause)
- to_utf16_code_units(s) builds code_units as a list of UTF‑16 code units.
- The function that then reads code_units (slice_forward) expects the caller to pass an index measured in UTF‑16 code units (0..len(code_units)-1).
- The caller passed index == 2 but len(code_units) == 2, so the passed index was out of range for the UTF‑16 code units list.
- Most likely the caller computed index as a Python string character index (codepoint index) or as a byte/character offset, not as a UTF‑16 code‑unit index. For strings that contain characters outside the Basic Multilingual Plane (BMP), a single Python character (a single codepoint with ord() > 0xFFFF) maps to two UTF‑16 code units. That mismatch between “character index” and “UTF‑16 code‑unit index” leads to passing an index that is invalid for code_units.
