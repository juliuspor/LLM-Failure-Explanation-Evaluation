What went wrong (root cause)
- At line 337 the code goes into the branch "elif element is not None".
- At line 340 it sets inferred_type = object unconditionally for that branch.
- At line 346 the code checks:
  if inferred_type == object and expected_type != object:
      raise TypeError(...)
- Because inferred_type was forced to object even when element != None, the check at line 346 evaluates true whenever the caller supplied a non-object expected_type (e.g. expected_type is int). That triggers the TypeError even though there is a concrete element available to infer the type from.

Where in the code this happens
- Branch entry: the "elif element is not None" branch (around line 337).
- Wrong assignment: line 340 assigns inferred_type = object.
- Failure check: line 346 compares inferred_type == object and raised the TypeError on line 347.

Why that causes the failure (concrete example)
- Suppose you call the function with array=None, element=5, expected_type=int.
- Execution reaches the element-not-None branch and sets inferred_type = object (line 340).
- The code then sees inferred_type == object and expected_type != object and raises TypeError (line 346–347).
- The intended behavior (per the docstring) would be to return [5] because element is present and matches expected_type=int, but instead the function raises.

How to fix it (clear steps)
1. Replace the incorrect assignment at line 340 with an inference from the actual element:
   - Use inferred_type = type(element)
   - Or, if you want to allow subclasses, use inferred_type = element.__class__
   Example replacement for line 340:
     inferred_type = type(element)
2. Ensure the TypeError check still makes sense: keep the check that raises when you cannot infer a type (inferred_type is object) but do not trigger it when element provided.
   - With the change above, inferred_type will only be object if you intentionally set it so (for example when element is None and you cannot infer).
3. Add or adjust tests:
   - Test case that currently fails: add(None, 5, expected_type=int) should return [5], not raise.
   - Test element None case: add(None, None, expected_type=int) should raise or follow intended behavior.

One-line patch
- Change line 340 from:
    inferred_type = object
  to:
    inferred_type = type(element)

Why this is safe and correct
- The docstring says when input list is None and element is not None, the new list component type should match the element. Using type(element) implements that intent. The current code incorrectly treats any non-None element as if its type were unknown (object), which causes the later type-check to fail when expected_type is more specific.
