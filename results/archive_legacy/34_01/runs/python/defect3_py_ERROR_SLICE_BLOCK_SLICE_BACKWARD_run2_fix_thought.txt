The bug arises from mixing code-point indices and UTF-16 code-unit indices. translate() currently treats pos as index into code_units and advances by char_count(code_point_at(code_units,pos)) or len(c) when no translator consumed. The bug likely when translate_codepoint returns consumed representing code points count, not code units. To fix, ensure pos advances by code units corresponding to consumed code points. When consumed==0, we already compute codepoint and to_chars; we advance by len(c) which is code units. When consumed>0, callers' translate_codepoint must return number of code units consumed, but contract says number of codepoints consumed. So need to advance pos by sum of char_count of each code point consumed by iterating consumed times using code_point_at each step. But current implementation uses code_point_at(code_units,pos) inside loop but increments pos inside loop causing code_point_at to be called with pos after increment which is wrong. Should compute next codepoint before incrementing. Specifically, for pt in range(consumed): cp = code_point_at(code_units,pos); pos += char_count(cp). This avoids IndexError. Also handle case where surrogate pair boundaries may cause index out of range; code_point_at raises IndexError if pos out of range. So loop should check pos < length before calling.
