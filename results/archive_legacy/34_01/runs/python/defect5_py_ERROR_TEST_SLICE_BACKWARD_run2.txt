What happened (root cause)
- You called ArrayUtils.add(None, None, expected_type=str).
- Inside ArrayUtils.add the method walks elements and sets inferred_type = object at line 340 because every element it inspected was None (the code reaches the branch on lines 337–340: "elif element is not None: ... inferred_type = object" is not updated when element is None).
- Later, at lines 346–347 the code checks:
  if inferred_type == object and expected_type != object:
      raise TypeError(...)
  That check fires because inferred_type stayed object while expected_type is str. The code therefore tries to produce a str[] but it actually built an Object[] and then attempts a runtime cast (Object[] -> String[]), which JVM cannot do, causing the ClassCastException and the TypeError you saw: Cannot cast object list to str list.

Why that causes the specific exception you saw
- The implementation creates an Object[] for the items because it cannot infer a more specific element type when all elements are None.
- Later code expects to return an array-of-expected_type (String[]), so it either attempts to cast the Object[] to String[] or to copy elements into a String[] and fails when doing an unsafe cast. In Java you cannot cast Object[] to String[] at runtime, hence the ClassCastException reported inside the TypeError.

Where it occurs (exact lines & variables)
- element variable is checked at line 337: the branch that would set a specific type only runs when element is not None; when element is None nothing updates inferred_type.
- inferred_type is set to object at line 340 and remains object.
- The guard at lines 346–347:
    if inferred_type == object and expected_type != object:
        raise TypeError(...)
  triggers because inferred_type == object and expected_type == str.
- The attempt to produce/return the expected-type array causes Object[] -> String[] cast failure (the ClassCastException you saw).

1) Treat "all-None" arrays as compatible with expected_type

         if inferred_type == object and expected_type is not None:
             inferred_type = expected_type
     - Then the code will construct a String[] (or appropriate typed array) instead of an Object[], avoiding the cast.

2) Avoid casting Object[] to Typed[]; always allocate the final array with the expected type

Why these fixes work
- They ensure the code does not build an Object[] and then cast it to a typed array. Instead it either:
  - uses the expected_type to create the correct typed array when all elements are None, or
  - allocates the correctly typed array up front and copies elements into it.
