What happened (where it fails)
- In get_paint the code clamps the input into variable v (lines 56–59), but the grayscale calculation still uses the original value variable (line 71). Because value can be outside the bounds, g can end up < 0 or > 255, so the check on line 74 fails and you get the ValueError shown.

Why it happens (root cause)
- You intended to use the clamped value but accidentally used the un-clamped value in the computation:
  - Clamping: v = max(value, self._lower_bound); v = min(v, self._upper_bound)
  - Calculation (bug): g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
  - The calculation should use v, not value. Because of that mismatch g can be outside [0,255] and trigger the exception at line 74.

Also note a secondary issue to guard against
- If self._upper_bound == self._lower_bound you will get a division-by-zero when computing g. That should be handled explicitly.

Concrete fixes (exact changes)
1) Use the clamped variable v:
   Replace line 71 with:
   g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

2) Add a guard for equal bounds to avoid ZeroDivisionError, for example:
   if self._upper_bound == self._lower_bound:
       g = 0  # or 255 or some defined default
   else:
       g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

3) (Optional) Ensure integer is clipped to [0,255] to be extra safe:
   g = max(0, min(255, g))

Minimal corrected function:
    def get_paint(self, value: float) -> Tuple[int, int, int]:
        # Clamp value
        v = max(value, self._lower_bound)
        v = min(v, self._upper_bound)

        # Handle equal bounds
        if self._upper_bound == self._lower_bound:
            g = 0
        else:
            g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

        g = max(0, min(255, g))

        return (g, g, g)

This fixes the immediate ValueError by using the clamped value and prevents division-by-zero.
