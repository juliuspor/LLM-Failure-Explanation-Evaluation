What happened
- The failure occurs at line 71 inside get_paint:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- If self._upper_bound == self._lower_bound, the denominator is zero and Python raises ZeroDivisionError. That happens before the code reaches the bounds check at lines 74–75, so you never get the custom ValueError.

Why it happened (root cause)
- The code assumes _upper_bound and _lower_bound are different. When they are equal, the expression (self._upper_bound - self._lower_bound) is 0 and division by zero occurs. This is the direct cause of the crash.

Where it occurred
- Method: get_paint
- File/lines shown: line 71 (calculation), lines 74–75 (bounds check that does not protect against zero denominator)

Explicit fixes (step-by-step)
1. Validate bounds at object construction or before compute:
   - When setting _upper_bound and _lower_bound, ensure they are not equal. If they can be equal by design, decide how to map values in that case.
   - Example check (on construction or setter):
     if self._upper_bound == self._lower_bound:
         raise ValueError(f"Invalid bounds: upper ({self._upper_bound}) equals lower ({self._lower_bound})")

2. Guard inside get_paint before dividing (defensive):
   - Check denominator and handle it explicitly so you avoid ZeroDivisionError and give a clear error or behavior:
     denom = self._upper_bound - self._lower_bound
     if denom == 0:
         raise ValueError(f"Cannot compute color: upper_bound ({self._upper_bound}) equals lower_bound ({self._lower_bound})")
     g = int((value - self._lower_bound) / denom * 255.0)

3. Optionally clamp or handle out-of-range values rather than raising later:
   - After computing g, clamp to [0,255] before converting/using, or raise a ValueError with the offending value:
     g = int(round((value - self._lower_bound) / denom * 255.0))
     if g < 0 or g > 255:
         raise ValueError(f"Color parameter outside of expected range: computed green {g} from value {value}, bounds ({self._lower_bound},{self._upper_bound})")

Why these fixes solve it
- The explicit denom check prevents ZeroDivisionError and gives a clear error message including the actual bound values.
- Validating bounds up front prevents the method from ever facing a zero denominator.
- Clamping or giving a clear ValueError for out-of-range g makes the failure mode explicit and debuggable.

Minimum change recommendation
- Add the denom check at start of get_paint (step 2). It’s the smallest localized change and makes the failure explicit with the exact variables that caused it.
