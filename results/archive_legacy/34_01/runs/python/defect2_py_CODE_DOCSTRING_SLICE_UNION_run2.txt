- At line 71 you compute g using the original parameter value:

- But at lines 56–57 you had already clamped value into v:

  v = min(v, self._upper_bound)
- Because you used value instead of v, values outside the bounds can produce g < 0 or g > 255. That triggers the ValueError at lines 74–75:

Why it happens (root cause)
- The intent of the clamp is to make out-of-range inputs safe. The calculation still uses the un-clamped variable value, so the clamp is ignored.
- If value < self._lower_bound or value > self._upper_bound, the normalized result is outside [0,1] and the multiplied result is outside [0,255], so g can be negative or greater than 255 and the code raises the error.
- There is a second potential failure if self._upper_bound == self._lower_bound: division by zero in the denominator (self._upper_bound - self._lower_bound).

How to fix it (concrete steps)
1. Use the clamped variable v in the g calculation (replace value with v).
2. Guard against zero span (upper == lower) before dividing.
3. (Optional) Use round() then clamp the final integer to [0,255] to avoid tiny floating-point edge cases.
