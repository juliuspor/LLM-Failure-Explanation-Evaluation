What happens
- At line 340 the code sets inferred_type = object when element is not None.
- Immediately afterward (line 346) the code checks if inferred_type == object and expected_type != object and raises a TypeError.
- Because inferred_type was just set to object whenever element is not None, any non-None element causes the code to always take the error path when expected_type is not object. In other words: every time element is present but expected_type is a concrete class, the code raises the exception on lines 346–351.

Why this is wrong (root cause)
- The condition that assigns inferred_type = object is wrong. The code treats "element is not None" as the situation where the element’s type cannot be inferred, and so it sets inferred_type to object. That is the opposite of what you want: when element is not None you usually can infer its type from the element value, not mark it as object. Because inferred_type is forced to object in that branch, the subsequent guard (line 346) always fails for concrete expected_type values.

- The variables involved are element (the value being inspected), inferred_type (what the code thinks the list element type is), and expected_type (the list element type that was required).

        inferred_type = object

        inferred_type = object

- If elements may be None and you want to treat that as unknown, option 1 is the minimal fix.
