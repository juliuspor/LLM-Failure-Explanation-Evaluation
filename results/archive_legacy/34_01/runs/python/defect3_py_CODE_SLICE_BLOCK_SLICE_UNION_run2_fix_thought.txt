The bug is that after a translator consumes characters, the code advances pos by iterating over 'consumed' and calling code_point_at which can be called past the end. Translators in this codebase (e.g., CsvEscaper) return the number of UTF-16 code units consumed, so simply advancing pos by 'consumed' is correct. I'll rewrite CharSequenceTranslator.translate to advance pos by consumed directly and keep the rest of the logic intact.
