What happened
- Calling gps.get_paint(-0.5) raised a ValueError instead of returning (0, 0, 0). The test fails because the method did not return a color tuple; it raised an exception.

Why it happened (root cause, exactly where)
- In get_paint() the method computes a single component named g and then immediately checks only g for being out of the 0–255 range:
  - At lines 74–75: 
    - 74 |         if g < 0 or g > 255:
    - 75 |             raise ValueError(f"Color parameter outside of expected range: Red Green Blue")
- Passing value = -0.5 makes g negative (for example g = -0.5 * 255 ≈ -127.5), so the condition g < 0 is true and the code raises ValueError.
- The docstring and test expect negative inputs to map to black (0,0,0) or at least to return a tuple, but the code raises instead.
- Two additional issues visible in the same slice:
  - Only g is checked; r and b are not being validated (so the check is incomplete if you really want to validate).
  - The error message is unhelpful and the f-string is pointless (f"..." with no interpolations).

What to change (concrete fixes, step-by-step)
Option A — Clamp inputs/outputs (recommended for a gray scale that saturates at 0 and 255)
1. Compute the component(s) as currently done.
2. Clamp each final component into [0, 255] instead of raising on negatives. Example replacement for lines 74–75:
   - r = int(round(r))
   - g = int(round(g))
   - b = int(round(b))
   - r = max(0, min(255, r))
   - g = max(0, min(255, g))
   - b = max(0, min(255, b))
   - return (r, g, b)
3. This will return (0,0,0) for value = -0.5.

Option B — If you want to keep strict validation, validate all three components and give a clear message
1. After computing r, g, b, validate all three:
   - if not (0 <= r <= 255) or not (0 <= g <= 255) or not (0 <= b <= 255):
       raise ValueError(f"RGB component out of range: r={r}, g={g}, b={b}")
2. This prevents a false-positive check that only considered g.

Minimal code example (clamping approach)

Why this fixes the failing test
- With value = -0.5 the negative component(s) become 0 after clamping, so get_paint(-0.5) returns (0, 0, 0) and the assertion passes.
