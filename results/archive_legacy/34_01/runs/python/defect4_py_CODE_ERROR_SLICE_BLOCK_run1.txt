What happened (root cause)
- In the block that updates self._max_middle_index you call get_data_item(self._min_middle_index) instead of get_data_item(self._max_middle_index). Concretely, inside:
  if self._max_middle_index >= 0:
      max_middle_period = self.get_data_item(self._min_middle_index).get_period()
you are reading the period for the minimum-middle index (min_middle_index) rather than the current maximum-middle index. That means you compare the new period's middle against the wrong reference value (min_middle) and may update _max_middle_index incorrectly. This explains the failing assertion "Expected max_middle_index=1, got 3".

Where it occurs
- The bug is in the if self._max_middle_index >= 0: block (the "max middle" section). The incorrect variable is self._min_middle_index used inside get_data_item(...) instead of self._max_middle_index.

Why that produces the observed failure
- Because you compare the incoming middle to the middle of the min_middle entry (not the max_middle entry), the update condition (if middle > max_middle) is evaluated against the wrong number. That allows the wrong index to be stored into _max_middle_index (index 3 in your test) even though the true maximum-middle entry is at index 1.

Exact code change to fix it
- Replace self._min_middle_index with self._max_middle_index in that block. For example:

    if self._max_middle_index >= 0:
        max_middle_period = self.get_data_item(self._max_middle_index).get_period()
        if isinstance(max_middle_period, SimpleTimePeriod):
            s = max_middle_period.get_start_millis()
            e = max_middle_period.get_end_millis()

            s = int(max_middle_period.get_start().timestamp() * 1000)
            e = int(max_middle_period.get_end().timestamp() * 1000)
        max_middle = s + (e - s) // 2
        if middle > max_middle:
