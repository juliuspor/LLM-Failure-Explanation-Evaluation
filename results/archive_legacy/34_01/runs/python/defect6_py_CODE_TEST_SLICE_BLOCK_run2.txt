What happened (root cause)
- The code treats any float that equals an integer with the branch starting at "if x == int(x) ..." and then does "value = int(x)".
- For x == -0.0 that equality is True (because -0.0 == 0.0), so the code converts it to int( -0.0 ) => 0 and then adds str(value) => "0".
- That loses the negative-zero sign. The test expects the string "-0.0", but the function produced "0".
- The loss happens specifically at the line:
  value = int(x)
  and in the logic that follows (the integer-only formatting), so the negative sign on -0.0 is thrown away.

Why the early branch is taken
- The condition x == int(x) holds for -0.0, so the integer-formatting path runs when it shouldn't for a value that must preserve the "-0.0" representation.

Exactly where to change (concrete fix)
- Before converting to int or entering the integer formatting block, detect negative zero and handle it as a special case. One simple reliable test is math.copysign(1.0, x) < 0 and x == 0.0.

Suggested concrete patch (replace/add these lines in add_number)
1) At the top of the integer-handling branch, add:

    # preserve negative zero: -0.0 should print "-0.0"
    if x == 0.0 and math.copysign(1.0, x) < 0:
        self.add(str(x))
        return

2) Then keep the rest of the integer-handling code unchanged.

Why this fix works
- math.copysign(1.0, x) preserves the sign bit even for signed zero. For -0.0 it returns -1.0, so the condition detects negative zero and prints it with str(x) (which yields "-0.0") without applying the int-conversion path that strips the sign.

Alternative minimal fixes
- Use x.is_integer() instead of x == int(x) doesn't avoid the problem (is_integer() is True for -0.0).
- Instead of returning str(value) for integer cases, special-case zero like above or check sign and add a "-" prefix when appropriate. But checking sign with math.copysign is essential for distinguishing -0.0 from +0.0.

Where it occurred
- The loss happens inside add_number, at the int conversion line (value = int(x)) and the integer branch that follows. The add(...) method (lines 126–138 shown) later just appends whatever string you produced; it does not restore the sign.
