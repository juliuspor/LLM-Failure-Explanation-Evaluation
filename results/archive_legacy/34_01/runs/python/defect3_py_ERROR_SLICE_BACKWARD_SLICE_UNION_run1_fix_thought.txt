The bug arises from mixing Python string character indices and UTF-16 code-unit indices. The translate method should iterate over UTF-16 code units, but when a translator returns consumed count it should be measured in codepoints. The existing code increments pos by char_count(code_point_at(code_units, pos)) inside a for loop over consumed; that double-counts when consumed >1. Simpler approach: when consumed==0 advance by char_count of codepoint at pos; when consumed>0 advance pos by sum of char_count for each codepoint consumed by repeatedly computing code_point_at at current pos and adding char_count. But consumed from translate_codepoint represents number of codepoints consumed, not code units; so we should advance pos by sum of char_count of each consumed codepoint. Implement loop for i in range(consumed): cp=code_point_at(code_units,pos); pos+=char_count(cp). That fixes index mismatches. Ensure function signature unchanged and no comments in output.
