What happened
- At line 12 you create code_units = [] and never fill it. The function never converts the input string s into UTF‑16 code units, so code_units stays empty.
- Later (line 35) the code checks index against len(code_units). When translate_string or any caller asks for index 0 (the first code unit of the emoji U+1F630), len(code_units) is 0 and the check at line 35 fails, raising IndexError("String index out of range: 0"). That is why the test that calls escaper.translate_string('\U0001F630') fails.

Why that causes the observed failure
- The test input is a non‑BMP code point (U+1F630). translate_string (via this helper) expects to work with UTF‑16 code units and to detect surrogate pairs (lines 38–44). But because code_units was never populated, the function cannot read high and low surrogates and crashes earlier with IndexError at line 35.

Where to fix it (concrete steps)
1. Populate code_units right after line 12. You must convert the Python string s into a list of UTF‑16 code units (integers).
2. Implement the conversion loop so that:
   - For code points <= 0xFFFF, append the code point as a single code unit.
   - For code points > 0xFFFF, compute the surrogate pair and append high then low:
     - cp = ord(ch)
     - if cp <= 0xFFFF: code_units.append(cp)
     - else:
         cp -= 0x10000
         high = 0xD800 + (cp >> 10)
         low  = 0xDC00 + (cp & 0x3FF)
         code_units.append(high); code_units.append(low)
3. Alternatively, you can produce UTF‑16 code units by encoding to UTF‑16 little endian and splitting bytes into 16‑bit words, but be careful with BOM; the loop above is straightforward and explicit.

Minimal example to insert after line 12:
- for ch in s:

        code_units.append(0xDC00 + (cp & 0x3FF))

Once code_units is populated, the existing checks at lines 35–44 will correctly handle surrogate pairs and the test will stop raising IndexError and can return the expected result '\U0001F630'.
