What happened (where and why)
- Location: add_number, lines starting at the equality check: `if x == int(x) and not math.isinf(x) and not math.isnan(x):` and the next line `value = int(x)`.
- Root cause: -0.0 compares equal to 0.0 in Python, so `x == int(x)` is True for x == -0.0. `int(-0.0)` yields 0 (no sign information), so you lose the negative sign and later `self.add(str(value))` writes "0" instead of "-0.0". That is why the assertion "Expected -0.0, got 0" fails.
- Where it manifests: you enter the integer branch and convert to an int, thereby discarding the negative-zero sign before formatting.

Precise fix (step-by-step)
1. Detect negative zero before you convert to int or otherwise fold sign. Use math.copysign because it preserves sign for zero:
   - check: `if x == 0.0 and math.copysign(1.0, x) < 0:`
2. If that condition is true, write the desired negative-zero representation and return (or skip the integer branch). For example:
   - `self.add("-0.0")`
   - `return`
3. Then leave the rest of the existing logic unchanged.

Exact code change (minimal patch)
Replace the start of the integer-handling block with:

    # detect negative zero before converting to int
    if x == 0.0 and math.copysign(1.0, x) < 0:
        self.add("-0.0")
        return

    if x == int(x) and not math.isinf(x) and not math.isnan(x):
        value = int(x)
        ...

Why this works
- math.copysign(1.0, x) returns -1.0 for -0.0, so you can distinguish -0.0 from +0.0 even though they compare equal numerically.
- Handling the negative-zero case before `int(x)` avoids losing the sign.

Alternative approaches (brief)
- Use formatting routines that preserve sign for negative zero (for example using format specifiers on floats without converting to int), or use `math.copysign` to choose a sign when building the string. But the simplest, safest change is to detect -0.0 early and format it explicitly as shown.
