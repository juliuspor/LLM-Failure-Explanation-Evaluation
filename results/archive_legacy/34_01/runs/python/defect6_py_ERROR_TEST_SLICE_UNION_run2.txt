Why it happened (root cause)
- Somewhere between add_number and the final output the negative sign on the floating-point zero was dropped.
- From your slice, the relevant methods are add_number (the caller), add (defined at line 126) and append (called at line 138). That tells us the value is being transformed or coerced inside one of those steps.
- A very common and specific cause is normalization code that treats any number equal to  numeric zero as plain zero. For example:
  - if value == 0: value = 0
  - or using a truthiness check like if not value: ...
  Both of those discard the sign bit of -0.0 because in IEEE floats -0.0 == 0.0 is true. That makes -0.0 indistinguishable from +0.0 via equality or truthiness tests, so the code replaces it with a plain 0 or string "0".
- Another possible spot is formatting code that uses an integer conversion (int(value)) or a formatter that collapses the sign for zero. Any code that normalizes or coerces zero values will remove the negative-zero sign.
