What happened (root cause)
- At line 400 the code checks minutes_offset and rejects any negative values:
  if minutes_offset < 0 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")
- Later (lines 406–408) the code intentionally adjusts minutes_offset to represent a negative total offset in minutes when the hour sign is negative:
  minutes_offset = hours_in_minutes - minutes_offset
  minutes_offset = hours_in_minutes + minutes_offset
  offset = FieldUtils.safe_multiply(minutes_offset, DateTimeConstants.MILLIS_PER_MINUTE)
- Because the validation at line 400 runs before the adjustment, a valid parsed minutes component of -15 (for an offset like -1:-15 or an intermediate negative minutes value produced by parsing) is rejected immediately with ValueError: Minutes out of range: -15, even though the later logic expects to handle negative minute values.

Where it occurred
- The failure happens in the method that builds an offset from hour/minute parts (the method that eventually returns cls.for_offset_millis(offset)).
- Specific lines: the pre-adjustment validation at 400–401 rejects minutes_offset = -15; the adjustment logic that would compute the final minutes-based offset is at 406–408; the code then multiplies minutes_offset by MILLIS_PER_MINUTE at 409 and returns at 412.

Why this is wrong
- The validation is done too early. The code intends to accept minute values that will be combined with hour sign logic (lines 406–408), but the early check treats negative minutes as invalid before that combination happens.
- As a result, valid offsets that require negative minute-handling are incorrectly rejected.
