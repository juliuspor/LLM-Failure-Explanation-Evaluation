What happened (what failed)
- The ValueError "Minutes out of range: -15" was raised by this guard at line 400–401:
  - Line 400: if minutes_offset < 0 or minutes_offset > 59:
  - Line 401:     raise ValueError(f"Minutes out of range: {minutes_offset}")
- The code reached that check with minutes_offset == -15, so the check correctly rejected it. The real problem is that minutes_offset should have been normalized before this check, not rejected immediately.

Why it happened (root cause)
- The code validates minutes_offset for being in 0..59 before it performs the normalization steps at lines 406–409. Those later lines are clearly intended to turn a negative-minute value into a positive offset within the hour:
  - Line 406:     minutes_offset = hours_in_minutes - minutes_offset
  - Line 408:     minutes_offset = hours_in_minutes + minutes_offset
  - Line 409:     offset = FieldUtils.safe_multiply(minutes_offset, DateTimeConstants.MILLIS_PER_MINUTE)
- Because the validation runs too early, a valid logical input that uses a negative minutes component (for example, an input representing an offset like -0:15 or a negative hour with positive minutes) gets rejected before those normalization lines can convert it into an acceptable minute count. In short: the order of operations is wrong — you check minutes before you finish computing/normalizing minutes.

Where it occurred (exact places to look)
- The rejection occurs at lines 400–401.
- The normalization logic that would make minutes_offset acceptable occurs at lines 406–409.
- The offending flow is therefore: validate (line 400) → fail → never reach normalization (lines 406–409).

   - This keeps the intent of rejecting out-of-range minute values but allows legitimate negative-minute inputs to be normalized first.
