What happened (root cause)
- The code clamps the input into v, but then calculates g using the original value instead of the clamped value. The clamping is done just after the def (lines immediately after line 56), but line 73 computes:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
  so any out-of-range value still produces an out-of-range g.
- If value is below self._lower_bound or above self._upper_bound, g can be <0 or >255 and the check at lines 76–77 raises the ValueError you see.
- There is a secondary issue: if self._upper_bound == self._lower_bound, the denominator at line 73 is zero and you get a ZeroDivisionError.

Where it happens
- get_paint (method starting at line 56)
- Clamping is performed just after the method starts.
- Wrong variable used at line 73 (use of value instead of v).
- Range validation raises at lines 76–77 when g is out of [0,255].

2. Guard against identical bounds to avoid division by zero:
   - Either return a constant color when lower==upper, or raise a clear error. For example, before computing g add:

         return (255, 255, 255)   # or some defined behavior
3. Improve the error/message if you keep the check (optional): include the offending g value and bounds so failures are easier to debug.
