- So the code fails because the computed component g is outside the expected 0–255 range, and the method intentionally raises that ValueError.

Why it happened (root cause)
- If value < self._lower_bound then (value - lower) is negative → g < 0 → triggers the ValueError.
- If value > self._upper_bound then (value - lower) / (upper - lower) > 1 → g > 255 → triggers the ValueError.
- There is also a related risk: if self._upper_bound == self._lower_bound then the expression on line 73 will divide by zero (ZeroDivisionError); that would fail earlier than the g-range check, but your observed failure shows the range check triggered, so in this run upper != lower.
- The code neither validates the incoming value nor explains which input caused the failure; it simply raises a generic message "Color parameter outside of expected range: Red Green Blue" (line 77) with no context.
