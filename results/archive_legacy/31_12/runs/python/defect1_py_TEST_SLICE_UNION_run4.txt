What happened (where, exactly)
- The failure occurs inside DateTimeZone.for_offset_hours_minutes when you call DateTimeZone.for_offset_hours_minutes(-2, -15).
- At line 400 the code checks:
  if minutes_offset < 0 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")
  That check throws ValueError("Minutes out of range: -15") because minutes_offset is -15.

Why it fails (root cause)
- The method expects the minutes argument to be an unsigned number between 0 and 59. Sign is taken only from the hours parameter. Passing a negative minutes value (-15) violates the check at line 400, so the method fails immediately.
- The later code at lines 406–409 attempts adjustments (it looks like it tries to combine hours and minutes into a signed offset), but that never runs because the negative minutes are rejected earlier by the validation at line 400.

How to fix it (concrete steps)
Two valid fixes depending on intent:

1) Caller-side fix (recommended, simple)
- Pass minutes as a non-negative value and use the sign on hours. Example:
  DateTimeZone.for_offset_hours_minutes(-2, 15)
  This produces offset -02:15. (Replace -15 with 15.)

2) Library-side fix (if you want to accept signed minutes)
- Change the validation and combination logic so minutes may be negative but are interpreted with the hours sign. For example:
  - Replace the check at line 400 with something like:
      if abs(minutes_offset) > 59:
          raise ValueError(f"Minutes out of range: {minutes_offset}")
  - Compute a single signed minutes value using hours’ sign (at or before lines 406–409), e.g.:
      sign = -1 if hours_offset < 0 else 1
      total_minutes = hours_offset * 60 + sign * abs(minutes_offset)
  - Then multiply total_minutes by MILLIS_PER_MINUTE and proceed as the existing code does (line 410 onward).
- Ensure you still reject inconsistent pairs (for example hours = 0 and minutes negative should be clearly defined) and add unit tests for combinations (-2,-15), (-2,15), (2,-15), etc.

Summary
- The immediate cause is the guard at line 400 rejecting negative minutes. Either call the method with minutes = 15 (and hours = -2) or change the validation to accept signed minutes and combine them using the hours sign as shown above.
