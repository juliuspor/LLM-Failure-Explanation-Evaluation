Why that happened (root cause)
- Your code calculated or parsed a minute value that became negative (−15) and passed it directly into a time-construction or setter API that expects 0..59.
- Common causes:

  - A parsing routine read a negative sign or malformed input.
  - A timezone or offset computation produced a negative minute field and you tried to assign it directly.
  - You assumed the API would wrap/normalize negative minutes automatically; it does not.
