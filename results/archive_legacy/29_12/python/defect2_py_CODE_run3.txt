What happened (root cause)
- In get_paint you clamp the input into variable v:
  - v = max(value, self._lower_bound)
  - v = min(v, self._upper_bound)
- But on the next line you compute g using the original value variable, not the clamped v:
  - g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- So when value is outside the bounds the clamp does nothing for the computed g. That leads to g being out of 0–255 and the ValueError being raised even though you intended to clamp first.

Where it occurs
- The bug is on the line that computes g (the line immediately after the two clamping lines).
- The two clamping lines are correct; the mistake is that the computation uses value instead of the clamped v.

Why it fails
- Using value instead of v means the normalized fraction can be < 0 or > 1. After scaling by 255 it produces g < 0 or > 255, which triggers the ValueError check that follows.

Concrete fixes (step-by-step)
1. Use the clamped variable v when computing g:
   - Change the g computation to:
     g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
2. Improve the error message so it shows the offending component and numeric value (optional but helpful):
   - Replace the generic message with something like:
     raise ValueError(f"Color component out of range: g={g}")
3. (Optional) Handle zero range to avoid division by zero:
   - If self._upper_bound == self._lower_bound, decide a behavior (e.g., return 0 or 255) and guard the division.

Corrected method
- Replace the g line with the one that uses v, and update the error message and optional guard. For example:

    def get_paint(self, value: float) -> Tuple[int, int, int]:
        # Clamp value to bounds
        v = max(value, self._lower_bound)
        v = min(v, self._upper_bound)
        
        # Use the clamped v here
        g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
        
        if g < 0 or g > 255:
            raise ValueError(f"Color component out of range: g={g}")
        
        return (g, g, g)

Summary
- The code fails because the clamped value v is not used when computing g; replacing value with v on the g calculation line fixes the bug. Optionally add a division-by-zero guard and a clearer error message.
