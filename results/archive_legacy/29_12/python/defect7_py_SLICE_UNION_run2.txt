What happened (where and what error you get)
- At line 769 the code calls array[i].upper(). If array[i] is not a string (for example an int, None, list, etc.) Python raises an AttributeError: "'<type>' object has no attribute 'upper'". So the code fails on that line and never reaches line 770 in those cases.

Why it happened (root cause)
- .upper() is a string method. It exists only on str objects. The code assumes every element of array is a str, but array contains at least one element that is not a str. That assumption is false, so calling .upper() on a non-str object raises the error.

Exactly where this occurs
- Line 769: _ = array[i].upper() — this is the failing call.
- Line 770: classes.append(type(array[i])) would only run after 769 succeeds; the exception prevents reaching 770 when a non-str element is encountered.

Concrete ways to fix it (each a clear step you can apply)
1) Skip non-strings
   - Replace line 769–770 with:
     for i in range(len(array)):
         if isinstance(array[i], str):
             _ = array[i].upper()
             classes.append(type(array[i]))
   - Behavior: only calls .upper() for strings and only appends types for strings.

2) Convert everything to strings before calling .upper()
   - If your intent is to treat every element as text:
     for i in range(len(array)):
         s = str(array[i]).upper()
         classes.append(type(array[i]))
   - Behavior: never raises AttributeError; you still record the original element type.

3) Handle the exception explicitly
   - If you want to try .upper() but tolerate failure:
     for i in range(len(array)):
         try:
             _ = array[i].upper()
         except AttributeError:
             # handle or skip non-strings
             continue
         classes.append(type(array[i]))
   - Behavior: catches the specific error and continues.

4) Validate input earlier
   - If array is expected to contain only strings, validate once when you receive it:
     if not all(isinstance(x, str) for x in array):
         raise TypeError("array must contain only strings")
   - Behavior: fail early with a clear message instead of an AttributeError at line 769.

Which option to pick
- If array legitimately contains mixed types and you only want to process strings: use option 1 or 3.
- If every element should be a string and non-strings indicate a bug upstream: use option 4 to surface that issue earlier.
- If you want to coerce everything to text: use option 2.

Any of these fixes remove the root cause: calling a string-only method on non-string objects.
