### Problem Identification

**What happened:**  
The test `LocaleUtils.to_locale('fr__POSIX')` raises a `ValueError` with the message `"Invalid locale format: fr__POSIX"`.

**Why it happened:**  
The code does not correctly parse locale strings with an empty country code followed by a variant (e.g., `"language__variant"` format like `'fr__POSIX'`).

**Where it occurred:**  
The issue is caused by a validation check in the `LocaleUtils.to_locale` method at **line 124**. This line checks if characters at positions 3 and 4 of the string are uppercase letters. It fails because, in the string `'fr__POSIX'`, the third character is an underscore (`_`), not an uppercase letter.

### Explanation Clarity

The `LocaleUtils.to_locale` method expects:
- A locale string in the format like `"language_country_variant"`.
- The third and fourth characters (expected to be the country code) must be uppercase letters if present.

However, in cases like `'fr__POSIX'`, the country is empty and is represented by a double underscore. The current validation logic incorrectly flags this valid format as invalid because it doesn't allow an empty country code followed by a variant.

### Actionability

**Fix the bug by modifying the validation logic:**

1. **Change** the validation condition on line 124 to allow for an empty country code:
   
    ```python
    # Original problematic code
    if not ('A' <= ch3 <= 'Z') or not ('A' <= ch4 <= 'Z'):
        raise ValueError(f"Invalid locale format: {locale_str}")
    ```

2. **Replace** it with:

    ```python
    # Allow the sequence to skip country if there's a double underscore
    if ch3 == '_':

    ```
   
This change accommodates formats like `'fr__POSIX'`, properly handling locale strings with an empty country code and a defined variant.
