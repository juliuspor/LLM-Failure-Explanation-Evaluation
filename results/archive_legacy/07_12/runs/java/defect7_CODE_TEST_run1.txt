### Problem Analysis

**WHAT happened:**
The test assertion fails when comparing two arrays: one generated by `ClassUtils.toClass` and the other hard-coded as `{ String.class, null, Double.class }`.

**WHY it happened:**
Inside the `ClassUtils.toClass` method, there is a line where it extracts the class type for each element in the input array. However, this logic does not adequately handle null values, which causes the resulting array to not match the expected output array when a null is present.

**WHERE it occurred:**
In the `ClassUtils.toClass` method, particularly on line 674, in this section of code:

```java
for (int i = 0; i < array.length; i++) {
    classes[i] = array[i].getClass(); // This line throws a NullPointerException if array[i] is null
}
```

### Solutions

**1. Fix the NullPointerException:**

Modify the logic to handle `null` values properly:

- **Action:** Add an `if` check to handle `null` values before attempting to call `getClass()`. Specifically, update the loop as follows:

```java
for (int i = 0; i < array.length; i++) {
    if (array[i] != null) {
        classes[i] = array[i].getClass();
    } else {
        classes[i] = null; // Explicitly set null for null array elements
    }
}
```

**2. Verify Correctness:**

- **Action:** Re-run the problematic assertion after applying the fix to confirm the behavior is now correct.

```java
assertTrue(Arrays.equals(new Class[] { String.class, null, Double.class }, ClassUtils.toClass(new Object[] { "Test", null, 99d })));
```

### Summary

To fix the issue, ensure the `toClass` method correctly handles `null` entries and returns `null` where appropriate in the resulting class array. By adding a conditional check for `null`, the method becomes robust against `null` values, thus aligning the test's expectation with the method's output. Make these changes to ensure the assertion passes successfully.
