What happens
- The loop that scans for the next match can finish with no match found, leaving text_index (and replace_index) at their initial "no match" value (-1 or None).
- Immediately after that scan you unconditionally do:
  - parts.append(replacement_list[replace_index]) (line 92)
  - start = text_index + len(search_list[replace_index]) (line 94)
  Those lines assume replace_index and text_index refer to a valid match. If no match was found, replace_index is invalid (or -1) and text_index is -1. That produces an IndexError or a wrong slice (start becomes len(text)-1) or otherwise corrupts behavior.

Where it occurs (lines)
- search_length = len(search_list) (line 52) — code assumes search_list and replacement_list align but you never guard against differing lengths; this can cause index errors later.
- The inner search loop (lines 108–114) sets temp_index and may leave text_index unchanged (still -1) if there are no remaining matches.
- Immediately following, without checking the result of that search, the code does:
  - parts.append(replacement_list[replace_index]) (line 92)
  - start = text_index + len(search_list[replace_index]) (line 94)
  These are the exact lines that fail when no match was found.
- Additionally, the code that marks "no more matches" (lines 102–111) can set all entries true; nothing in the caller checks for that before lines 92–94 run.

Why this is the root cause (plain language)
- The code assumes a successful match before it verifies that a match actually exists. When there are no more matches, it still tries to use the nonexistent match index and length, which causes the failure.

Exact fixes (concrete steps)
1. Ensure replace_index / text_index are checked after the search loop and before using them.

   This prevents using an invalid replace_index and ensures you append the remainder of text.

3. Validate list lengths up front (optional but recommended):

   - This prevents out-of-range accesses like replacement_list[i] when i >= len(replacement_list).

4. (Minor) When computing start, only do start = text_index + len(search_list[replace_index]) when you know text_index >= 0 and replace_index is valid.

Why these fixes fix it
- They stop the code from using an invalid index (-1 or None) and from computing a bad slice start. They also make the end-of-search case explicit: append the remaining text and exit the loop instead of attempting a replacement that doesn't exist.
