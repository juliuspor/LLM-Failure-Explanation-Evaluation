What happened (root cause)
- In get_paint you correctly clamp the incoming value into local variable v:
  - Line ~56–60: v = max(value, self._lower_bound); v = min(v, self._upper_bound)
- But when you compute the gray channel you use the original value instead of the clamped v:
  - Line 71: g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- If value is outside the bounds (e.g. -0.5 or 1.5 in the tests) that formula produces g < 0 or g > 255, so the check on lines 74–75 raises the ValueError you see.

Where it occurs
- The bug is the mismatch between the clamped variable v and the variable used in the interpolation (value) at line 71.

Why that causes the test failure
- Tests call get_paint(-0.5) and get_paint(1.5). value is out of bounds, the code sets v to a clamped value, but then computes g from the unclamped value, producing an out-of-range g which triggers the ValueError.

Concrete fixes (step‑by‑step)
1. Use the clamped variable v in the interpolation:
   - Replace line 71 with:
     g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
2. Guard against division by zero when lower_bound == upper_bound:
   - Example:
     if self._upper_bound == self._lower_bound:
         g = 0
     else:
         g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
3. (Optional) Decide how to round fractional results: int(...) truncates; if you want the test's expected 0.5→127, truncation is fine. If you prefer nearest integer, use round(...) then int(...).

Minimal corrected get_paint body (illustrative)
- Clamp to v (keep your existing two lines)
- Compute g from v (use v instead of value)
- Check bounds and return (r, g, b)

That change fixes the ValueError for out-of-range inputs because g will always be computed from the clamped v and therefore be within 0–255.
