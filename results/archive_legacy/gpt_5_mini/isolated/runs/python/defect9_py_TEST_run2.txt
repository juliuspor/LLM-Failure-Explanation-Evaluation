What happened
- The tests fail when one element of the replacement list is None (Python None). The failure is raised from your replace_each implementation when it tries to use that None as a replacement string.
- Root cause: your code calls Python's str.replace(...) (or performs an operation that expects a string) using the replacement variable directly. When replacement is None this produces a TypeError (replace() argument 1 must be str, not NoneType) or turns the None into the string "None", neither of which matches the Java StringUtils behavior the tests expect.

Where it occurs
- Open hit09_stringutils.py and find the function/method named replace_each (or replaceEach). Look for a line that looks like:
  s = s.replace(search, replace)
  or any code that passes the variable named replace (or repl, replacement, rep) directly into a string operation.
- That exact line is where the None replacement value causes the failure.

Why it fails (step-by-step)
1. The test calls: StringUtils.replace_each("aba", ["a"], [None]).
2. Your function iterates over the search/replacement pairs and for the pair ("a", None) calls Python string replace with None as the replacement argument.
3. Python's str.replace expects both arguments to be strings. Passing None raises a TypeError (or produces "None" if you convert to str), so the produced output is not the expected unchanged string "aba".
4. The tests expect Java StringUtils semantics: if a replacement entry is null, that entry should be treated as "do not replace" (leave occurrences of the corresponding search token unchanged).

How to fix it (concrete steps)
1. Edit hit09_stringutils.py, find replace_each.
2. Before calling str.replace (or any string operation using the replacement), check whether the replacement value is None.
3. If replacement is None, skip that pair entirely (do not call replace). That will leave occurrences of the search token as-is.
   Example change (conceptual — adapt variable names to your implementation):
   - current line (problematic)
     s = s.replace(search, replacement)
   - change to
     if replacement is None:
         # skip this pair; do not modify s for this search token
         continue
     s = s.replace(search, replacement)
4. If your implementation does multiple simultaneous replacements (building a mapping or scanning for the earliest match), apply the same guard at the point where you would apply the replacement: skip when replacement is None.
5. Run the tests again.

Extra notes (common pitfalls)
- Don’t convert None to the string "None"; that will make tests fail and produce wrong output.
- If your code tries to coerce all replacements to strings up front (e.g., with str(replacement)), remove that coercion or guard it so None is treated as a signal to skip.
- If you have code that zips search_list and replace_list, ensure you iterate pairs correctly so you can test replacement is None for that pair.

Summary
The failing line is the place where the code uses the replacement value in a string operation without checking for None first. Fix by skipping any pair where the replacement is None so the search token is left unchanged.
