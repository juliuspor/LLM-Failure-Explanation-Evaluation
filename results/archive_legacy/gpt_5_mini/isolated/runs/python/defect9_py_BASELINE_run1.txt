What happened (where it fails)
- The TypeError occurs at the loop starting on line 81:
  greater = len(replacement_list[i]) - len(search_list[i])
- That loop assumes both replacement_list[i] and search_list[i] are non-None and that search_list[i] has length > 0.
- In the failing test you call replace_each("aba", ["a"], [None]). replacement_list[0] is None, so len(replacement_list[0]) raises TypeError: object of type 'NoneType' has no len().

Why it happens (root cause)
- Earlier, the code already handled null/empty entries when searching for the first match (lines ~59–76) by checking:
    if no_more_matches_for_repl_index[i] or search_list[i] is None or len(search_list[i]) == 0 or replacement_list[i] is None: continue
  That prevents calling find() on invalid entries.
- The loop at lines 81–84 (the "get a good guess on the size" loop) forgot the same null/empty safety checks. It unconditionally calls len(...) on elements of search_list and replacement_list, so a None element causes the TypeError.
- The code comment even points this out: "# BUG: this loop is not null-safe for elements inside the arrays".

Concrete example from the tests
- For test_null_safety_inside_arrays_single:
  search_list = ["a"], replacement_list = [None]
  In the size-estimate loop, i = 0:

How to fix (clear steps)
1. Add the same null/empty checks used elsewhere before using len() in that loop. Replace the loop with:

    for i in range(search_length):
        if (
            search_list[i] is None
            or len(search_list[i]) == 0
            or replacement_list[i] is None

        greater = len(replacement_list[i]) - len(search_list[i])

Why that fix works
- It prevents calling len() on None or an empty search string.
- It keeps the size-estimate loop consistent with the earlier match-finding loops, so entries that are supposed to be ignored (None replacements or empty searches) are ignored everywhere.
