What happened (where and why)

- Why it fails: that if condition will raise a ValueError whenever the green channel variable g is outside 0..255 — or if g is not a number, the comparisons g < 0 or g > 255 will raise a TypeError. Once raised, the ValueError message is unhelpful because the f-string contains no interpolated variable names or values; it just emits the literal text "Red Green Blue". In short:
  - The condition only checks g (not r or b), so failures tied to r or b will be missed.
  - If g is not an int/float (e.g. a string or None), the comparisons will error before you get a clear message.
  - The raised ValueError does not tell you which channel failed or what its value was because the f-string has no expressions.

- If g == -1, the code raises ValueError with message "Color parameter outside of expected range: Red Green Blue" — you can't see that g == -1 or which channel is bad.

- If r or b is out of range, this check will not catch it at all.

1. Validate types first: ensure r, g, b are integers (or convertible).
   - Example: if you accept strings, convert with int(...) and handle ValueError from conversion.
2. Check each channel separately and raise a clear error that names the channel and shows its value.
3. Use properly interpolated f-strings to include the variable and its value.

Suggested replacement (replace lines 74–75 with this):
- Validate type and range for each channel:
  - try:
      g = int(g)
    except (TypeError, ValueError):
      raise TypeError(f"Green channel must be an integer, got {g!r} (type {type(g).__name__})")
  - if g < 0 or g > 255:
      raise ValueError(f"Green channel out of range: g={g} (expected 0..255)")

- Or validate all three channels at once:
  - def validate_channel(name, val):
        try:
            v = int(val)
        except (TypeError, ValueError):
            raise TypeError(f"{name} channel must be an integer, got {val!r} (type {type(val).__name__})")
        if v < 0 or v > 255:
            raise ValueError(f"{name} channel out of range: {name}={v} (expected 0..255)")
        return v
    r = validate_channel("Red", r)
    g = validate_channel("Green", g)
    b = validate_channel("Blue", b)

Why this fix addresses the root cause
- It prevents TypeError from invalid comparisons by ensuring correct types first.
- It checks every color channel, not just g.
- It gives explicit, informative error messages including which channel failed and the offending value.
