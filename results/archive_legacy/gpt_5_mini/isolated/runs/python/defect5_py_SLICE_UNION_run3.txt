What happened (root cause)
- At line 336 you set inferred_type = type(array) when array is provided.
- At line 337 you start an elif to set inferred_type = type(element) when element is provided.
- But at line 340 you set inferred_type = object unconditionally (the code as shown puts that assignment outside the if/elif block).
- As a result inferred_type is always object by the time you reach line 346, so the check at line 346:
  if inferred_type == object and expected_type != object:
    raise TypeError(...)
  always sees inferred_type == object and raises the TypeError whenever expected_type is not object.

Why that causes the failure
- The intent was clearly: use the array element type if available, otherwise the single element type, otherwise use object as a fallback.
- Because inferred_type is overwritten at line 340 regardless of the earlier assignments, the fallback replaces a valid inferred type (from lines 336 or 338) and triggers the error check on line 346, producing an incorrect TypeError.

Where it occurs (precise variables/lines)
- Variables: array, element, inferred_type, expected_type
- Lines: 336 (inferred_type = type(array)), 338 (inferred_type = type(element)), 340 (inferred_type = object — misplaced), 346 (the comparison that raises TypeError)

How to fix it (explicit steps)
1. Move the fallback assignment so it only runs when neither array nor element is provided. Replace the current structure with an if/elif/else block. Concretely:
   - Keep line 336 as: if array is not None: inferred_type = type(array)
   - Keep line 337 as: elif element is not None: inferred_type = type(element)
   - Change line 340 to be the else branch: else: inferred_type = object
2. Confirm indentation and block structure so inferred_type = object is not executed unconditionally.
3. Add a unit test for three cases:
   - array provided (non-None) — inferred_type should be type(array) and no TypeError if expected_type matches.
   - element provided (array is None) — inferred_type should be type(element).
   - neither provided — inferred_type should be object and the original TypeError branch should behave as intended.
4. Run the tests to verify the TypeError is only raised when inferred_type is object while expected_type != object and when no other type could be inferred.

Optional small improvement
- Use isinstance checks for array vs element presence only if you need to accept falsy values; otherwise the is not None checks you already have are fine.
- Consider storing the actual type object (e.g., inferred_type = array.__class__) only when you really need it; the main bug is the overwritten assignment, not how you obtain the type.

Summary
Line 340 overwrites inferred_type unconditionally because of incorrect control flow (indentation/placement). Move that assignment into an else branch so inferred_type is only set to object when both array and element are None; then the TypeError on line 346 will behave correctly.
