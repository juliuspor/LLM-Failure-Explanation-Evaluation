What fails
- The failing line is in the test test_add_null_to_null_with_expected_type:
  result = ArrayUtils.add(None, None, expected_type=str)
- The test passes a keyword argument expected_type, but the failing error you see when you run the test is most likely:
  TypeError: add() got an unexpected keyword argument 'expected_type'

Why this happens (root cause)
- The ArrayUtils.add implementation (in hit05_arrayutils.py) does not declare an expected_type parameter in its signature, so Python raises a TypeError immediately when the test calls add(..., expected_type=str).
- This is not a runtime bug inside the body of add; it is a mismatch between the test's expected API and the actual function signature.
- The test exercises a Java-origin bug (LANG-35) about type inference when adding nulls. To reproduce the intended check the Python API must accept the expected_type hint. Without that parameter, the test cannot even reach the logic that should handle the (None, None) case.

Minimal example patch (conceptual)

- With

      # otherwise copy and append

      new.append(element)
      return new

Why this fix addresses the failure
- Adding the optional parameter expected_type removes the TypeError raised at the call site so the test can run.
- Returning [None] for (None, None, expected_type=str) yields the exact value the test asserts.
- This approach mirrors Python's dynamic typing (no need to construct typed arrays as in Java), but still honors the test's intent to let callers pass a type hint without crashing.
