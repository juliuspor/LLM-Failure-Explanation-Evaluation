What happened
- When you call create_number("l") or create_number("L"), val is non-empty and last_char == "l", so numeric = val[:-1] becomes the empty string "".
- The code then calls char_at(numeric, 0) without checking whether numeric is non-empty.
- char_at checks bounds and raises IndexError when index >= len(s). In this case index == 0 and len(numeric) == 0, so char_at raises IndexError("String index out of range: 0").
- That IndexError bubbles out of create_number instead of the expected ValueError, causing the test failure.

Exactly where it fails
- In create_number, inside the branch:
  if last_char in ("l", "L"):
      # BUG: missing guard for len(val) == 1 and non-digit input.
      if (char_at(numeric, 0) == "-" and numeric[1:].isdigit()) or numeric.isdigit():
          return int(numeric)
- The call char_at(numeric, 0) triggers the IndexError. The char_at implementation raises that error at lines 30–32 (the index bounds check and return).

Why this is the root cause
- The code assumes numeric has at least one character when stripping a trailing 'l'/'L'. That assumption is false for inputs that are just "l" or "L". The guard for numeric being empty is missing, so you attempt to inspect numeric[0] (via char_at) and numeric[1:] when there is no character 0 or 1.

Concrete fixes (step-by-step)
1. Add an explicit check for the empty numeric string before calling char_at or accessing numeric[1:].
   Example replacement of the broken block:
   if last_char in ("l", "L"):
       if not numeric:
           raise ValueError(f"{val} is not a valid number.")
       if (numeric[0] == "-" and numeric[1:].isdigit()) or numeric.isdigit():
           return int(numeric)
       raise ValueError(f"{val} is not a valid number.")

2. Or use safer Python string methods and length checks:
   if last_char in ("l", "L"):
       if numeric and ((numeric.startswith("-") and numeric[1:].isdigit()) or numeric.isdigit()):
           return int(numeric)
       raise ValueError(f"{val} is not a valid number.")

3. Add unit tests to cover the failing cases: "l", "L" (already in your test suite), and also "-l" if you intend to allow "-1L"-style inputs.

Why these fixes work
- They prevent any attempt to read numeric[0] or numeric[1:] when numeric == "", so char_at will not be called with an out-of-range index and the function will raise ValueError as intended for invalid inputs.

Summary
- Root cause: calling char_at(numeric, 0) when numeric is the empty string (inputs "l"/"L").
- Fix: check len(numeric) (or numeric truthiness) before accessing numeric[0] or numeric[1:], and raise ValueError for empty numeric.
