What happens
- Root cause: upper can end up larger than the string length after the line
    if upper < lower:
        upper = lower
  and then substring_java(text, 0, upper) is called. substring_java (Java's substring) throws an index-out-of-range error if the end index is greater than text.length(), so the function fails when lower > len(text).

Where it happens (by code lines / variables)
- The problematic sequence is:
  1) You clamp upper based on itself and len(text):
     if upper == -1 or upper > len(text):
         upper = len(text)
     (this sets upper = len(text) only when upper > len(text) at that moment)
  2) Then you do:
     if upper < lower:
         upper = lower
     (this can increase upper)
  3) Later you call substring_java(text, 0, upper) — if lower > len(text) that call will pass an end index greater than len(text) and throw.

Why it fails (plain explanation)
- If the supplied lower parameter is greater than the input text length, step (2) sets upper = lower which becomes > len(text). You never re-clamp upper to the text length after that assignment, so substring_java is asked to extract up to an index beyond the string end. substring_java (Java String.substring) requires end ≤ length and so throws StringIndexOutOfBoundsException (or in Python a similar IndexError).

How to reproduce quickly
- Call abbreviate("short", lower=10, upper=5, append_to_end="..."):
  - After the first clamp upper becomes len(text)=5
  - Then because upper < lower, upper = 10
  - substring_java(text, 0, 10) -> error

Precise fixes (step-by-step)
Option A — simplest, re-clamp upper after adjusting for lower:
- Right after
    if upper < lower:
        upper = lower
  add:
    if upper > len(text):
        upper = len(text)
  This guarantees upper is never > len(text) before any substring_java calls.

Option B — change order to clamp lower against length first:
- Do:
    if lower > len(text):
        lower = len(text)
    if upper == -1 or upper > len(text):
        upper = len(text)
    if upper < lower:
        upper = lower
  This keeps both bounds inside [0, len(text)].

Also check index_of behavior:
- index = StringUtils.index_of(text, " ", lower)
  If lower > len(text) many index implementations return -1; code then goes to the branch that calls substring_java(0, upper) — so the re-clamp above fixes that too.

One-line recommended patch (Option A):
- After the existing if upper < lower: upper = lower line, add:
    if upper > len(text):
        upper = len(text)

That fixes the out-of-bounds call without other changes.
