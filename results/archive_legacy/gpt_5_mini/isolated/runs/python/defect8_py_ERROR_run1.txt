What happened (root cause)
- The process raised ValueError: Invalid locale format: fr__POSIX because the program passed the string "fr__POSIX" to a locale-parsing function that rejects that format.
- "fr__POSIX" is invalid because it has an empty territory/encoding slot between two underscores. Locale formats expected by functions such as Python’s locale.setlocale, locale.normalize, or many third-party libraries follow patterns like:
  - language[_territory][.encoding][@modifier], e.g. "fr_FR.UTF-8" or "fr_FR@POSIX"
  - or language[_territory] or language_modifier, but never language__modifier (double underscore with an empty middle part).
- In short: code built a locale string by concatenating three pieces and didn’t skip the empty piece, producing "fr__POSIX". The locale parser then rejected it.

Concrete example of the likely code that caused it
- Faulty assembly (example of what you probably have):
  - language = "fr"
  - territory = ""          # empty string when not set
  - modifier = "POSIX"
  - locale_str = f"{language}_{territory}_{modifier}"   # => "fr__POSIX"
  - locale.setlocale(locale.LC_ALL, locale_str)        # raises ValueError

Why that exact string fails
- The double underscore signals an empty territory field; the parser treats that as malformed and raises ValueError saying "Invalid locale format: fr__POSIX".

Exact fixes (step-by-step, with code)
A. Fix the string assembly so you never leave an empty middle part:
- Replace the simple f-string concatenation with logic that only includes non-empty parts.

Example patch (apply at the line where you build locale_str; assume the function is build_locale_string in config.py line 27):

Old (faulty)
```
# config.py:27
locale_str = f"{language}_{territory}_{modifier}"
```

New (robust)
```
# config.py:27
parts = []
if language:
    parts.append(language)
if territory:
    parts.append(territory)
locale_base = "_".join(parts)          # e.g. "fr" or "fr_FR"
if modifier:
    # use @ or . depending on what your target parser expects; @ is common for modifiers
    locale_str = f"{locale_base}@{modifier}"
else:
    locale_str = locale_base
```
- This produces "fr@POSIX" or "fr_FR@POSIX" (both valid depending on the consumer), never "fr__POSIX".

B. Or use a library function that builds/parses locales rather than hand-joining:
- If using Babel:
  - from babel.core import Locale
  - locale_obj = Locale.parse(f"{language}_{territory}" if territory else language)
  - if modifier: apply modifier with a known, supported pattern or handle separately.
- If you must produce a POSIX modifier, prefer "fr@POSIX" or "fr_FR@POSIX" rather than inserting an empty piece.

C. Quick defensive patch to avoid the error if you only want to guard the call site:
```
# before calling locale.normalize() or locale.setlocale()
locale_str = locale_str.replace("__", "_")   # simple guard (but prefer building correctly)
```
- Note: This is a stopgap. Better to avoid creating the bad string in the first place and to choose the correct separator for modifier (@ vs .) depending on the library.
