What happened (root cause)
- The loop at lines 81–84 computes
  greater = len(replacement_list[i]) - len(search_list[i])
  but it doesn't check whether replacement_list[i] or search_list[i] is None.
- When a test passes replacement_list = [None] (see test_null_safety_inside_arrays_single), replacement_list[0] is None. Calling len(None) raises TypeError: "object of type 'NoneType' has no len()".
- You already guard against None/empty elements in the two match-finding loops (lines ~59 and ~102), but you forgot to make the same check in the buffer-size estimation loop (lines 81–84). That is exactly where the crash occurs.

Where it occurs
- File: replace_each function
- Faulty lines: 81–84 (the "BUG" comment in the code points to it)
- Trigger: replacement_list[i] is None (or search_list[i] is None)

How to reproduce quickly
- Call StringUtils.replace_each("aba", ["a"], [None]) — this is the failing unit test. The function will hit the size-estimation loop and call len(None).

Precise fix (two safe options)
1) Skip None/empty elements in the size-estimation loop, same as the other loops:
   Replace the loop at lines 81–84 with:

   for i in range(search_length):
       if (
           no_more_matches_for_repl_index[i]
           or search_list[i] is None
           or len(search_list[i]) == 0
           or replacement_list[i] is None
       ):
           continue
       greater = len(replacement_list[i]) - len(search_list[i])
       if greater > 0:
           increase += 3 * greater

   This mirrors the null-safety checks used elsewhere and prevents len(None) from being evaluated.

2) Alternatively, treat None as empty string for length computation:
   greater = (len(replacement_list[i]) if replacement_list[i] is not None else 0) - (len(search_list[i]) if search_list[i] is not None else 0)

   But option 1 is preferable because the function intentionally ignores None entries elsewhere.

Why this fix is correct
- It preserves the original intent: ignore search/replacement pairs that are None or empty.
- It prevents len() from being called on None and thus removes the TypeError while keeping the size-estimation optimization.

Summary
- The crash is caused by calling len() on None in the size-estimation loop (lines 81–84). Add the same null/empty guards used elsewhere (or treat None as length 0) to fix it.
