What happened (what failed)
- The TypeError occurs at the estimate-size loop (lines 81–84): the code calls len(replacement_list[i]) and len(search_list[i]) without checking whether those elements are None.
- When replacement_list[i] is None (or search_list[i] is None), calling len(...) raises TypeError: "object of type 'NoneType' has no len()".
- This is exactly the failing test test_null_safety_inside_arrays_single where replacement_list = [None].

Where it occurs in the code
- The problematic loop is the one that tries to estimate the output buffer growth:
  - Line 81: for i in range(search_length):
  - Line 82: greater = len(replacement_list[i]) - len(search_list[i])
  - There is no None check here, so if either replacement_list[i] or search_list[i] is None, len(...) fails.
- Note that later loops (the match-finding loops at lines ~59–75 and ~102–113) do correctly skip entries when search_list[i] is None or replacement_list[i] is None. The initial estimation loop was overlooked and not null-safe.

Why it happened (root cause)
- The code intended to treat None entries in search_list or replacement_list as "ignore this pair", but only the match-finding loops enforce that rule.
- The precomputation loop incorrectly assumes every pair is a valid string and performs len() on each element. That assumption is false when callers pass None inside the arrays.

Concrete fix (exact steps)
- Skip pairs where search_list[i] is None or empty, or replacement_list[i] is None, before calling len():
  - Replace the body of that loop with something like:

    for i in range(search_length):
        if (
            search_list[i] is None
            or len(search_list[i]) == 0
            or replacement_list[i] is None
        ):
            continue
        greater = len(replacement_list[i]) - len(search_list[i])
        if greater > 0:
            increase += 3 * greater

- Alternatively, compute lengths with a safe helper that handles None, but the simplest is to add the same checks already used in the matching loops.

Why this fix works
- It makes the estimate loop consistent with the matching loops: pairs that are supposed to be ignored are also ignored during buffer-size estimation, preventing any len(None) calls and fixing the TypeError.

Summary
- Root cause: calling len() on None in the size-estimation loop (lines 81–84).
- Fix: skip entries where search_list[i] is None or empty, or replacement_list[i] is None before using len().
