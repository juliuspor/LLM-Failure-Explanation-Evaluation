What happened (where it fails)
- In get_paint, you clamp the input to v on lines 3–4:
  v = max(value, self._lower_bound)
  v = min(v, self._upper_bound)
- On line 6 you compute g using the original value variable instead of the clamped v:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- If value was outside the bounds, that calculation can produce g < 0 or g > 255, so the subsequent range check on line 9 raises ValueError.
- Also: if self._upper_bound == self._lower_bound the division will raise ZeroDivisionError (or produce an invalid result).

Why it fails (root cause)
- You intended to use the clamped input v for the color calculation but accidentally used the un-clamped value variable. That makes the clamp ineffective for the g calculation, leading to out-of-range g and the ValueError (or a division by zero when bounds are equal).

2. Guard against equal bounds to avoid division by zero. Either raise a clear error or handle it explicitly. Example (raise):
   if self._upper_bound == self._lower_bound:
       raise ValueError("upper_bound must be different from lower_bound")
   Or (map to mid-gray):
       if self._upper_bound == self._lower_bound:
           g = 128
3. Optionally ensure g is exactly in [0,255] after rounding (defensive clamping):
   g = max(0, min(255, int(...)))
4. Improve the error message to show the actual offending g or bounds if you still want to raise.
