What happened (where it failed)
- The IndexError is raised on these lines inside to_utf16_code_units:
  - Line 12: code_units = []
  - Line 35–36:
    if index < 0 or index >= len(code_units):
        raise IndexError(f"String index out of range: {index}")
- That check ran when code_units was empty (len(code_units) == 0), so any index >= 0 triggered the IndexError.

Why it failed (root cause)
- The function is supposed to return or use a list of UTF‑16 code units, but the code checks bounds against code_units before that list has been filled.
- More precisely: the code computes or receives an index (variable name index) that represents a position in the string, then compares it to len(code_units). However code_units is still empty at that point because the function either:
  - has not yet appended any UTF‑16 code units to code_units, or
  - built code_units under a conditional branch that did not run for this input.
- There is therefore a timing/ordering bug: the bounds check is performed against an unpopulated list instead of the populated list (or instead of the correct length unit for that index).

Why this is the *real* problem (no guesswork)
- The code shows code_units is explicitly set to [] at line 12 and the bounds check uses len(code_units) at line 35. If code_units were populated earlier, len(code_units) would be > 0 and the check would not fail for normal indexes. Since it fails, the only possibility consistent with these lines is that code_units was not populated before running the check.

Concrete fixes (step-by-step)
1. Ensure code_units is fully populated before any use:
   - Move any code that computes or checks index to after the loop/logic that appends to code_units.
   - Example step: if you currently build code_units later, reorder so building happens immediately after line 12 and before line 35.

2. If index refers to Unicode code points (Python str indexes) but you are checking against UTF‑16 code units:
   - Convert the code-point index to a UTF‑16 code-unit index before the bounds check, or
   - Use the same unit (code points vs code units) consistently for both index computation and bounds checking.
   - Concrete step: iterate through the string and count code units (count +2 for characters that become surrogate pairs) until you reach the requested code-point index, then use that code-unit count for comparison.

3. Add defensive checks and clear error messages:
   - After building code_units, re-check:

           raise IndexError(f"String index out of range: {index} (0..{len(code_units)-1})")
   - This guarantees the list length used in the message matches the actual list.

4. Add tests that cover:
   - Empty string s == ""
   - Strings with only BMP characters
   - Strings containing non-BMP characters (surrogate pairs when encoded as UTF‑16)
   - Negative index values

Follow those steps and the IndexError will only be raised when index truly is out of range for the populated code_units list, and not because the list was still empty.
