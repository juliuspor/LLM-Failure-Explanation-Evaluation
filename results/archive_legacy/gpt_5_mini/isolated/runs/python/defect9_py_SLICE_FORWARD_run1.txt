What happened (root cause)
- The inner search loop (lines 102–115) can finish without finding any remaining match; in that case text_index and replace_index are left at -1 (they are only set when a match is found).
- The code does not check for that condition before using replace_index and text_index at lines 92 and 94.
- As a result parts.append(replacement_list[replace_index]) (line 92) and start = text_index + len(search_list[replace_index]) (line 94) execute with replace_index == -1 (and text_index == -1). That indexes the last element of replacement_list/search_list and computes an invalid start, which produces incorrect output or an infinite loop depending on the data.

Exactly where it happens
- The problem arises immediately after the inner loop at lines 102–115. The code assumes a match was found and uses replace_index at line 92 and text_index at line 94, but nothing guarantees a match was found.
- You can see the sign: the inner loop sets no_more_matches_for_repl_index[i] when temp_index == -1 (line 110), but never exits or checks “no match anywhere” before continuing to use replace_index.

Why this fails (concrete effect)
- If no match is found in that pass, replace_index remains -1 and the code:
  - appends replacement_list[-1] (the last replacement) instead of stopping,
  - computes start = (-1) + len(search_list[-1]) producing a wrong start (often negative or too small),
  - then continues the outer loop with bad indices, causing wrong replacements or an infinite loop.

How to fix it (clear steps)
1. After the inner loop (lines 102–115) add an explicit check for “no match found”:
   - if replace_index == -1 (or text_index == -1):
       - append the remainder of text (parts.append(text[start:]) — line 115 already does this at the loop end) or simply break out of the outer replacement loop.
2. Do not use replace_index or text_index unless you know they were set by a found match.
3. Example placement:
   - Immediately before line 92:
       if replace_index == -1:
           parts.append(text[start:])   # no more matches; finish
           break
   - Then proceed to lines 92 and 94 safely (replace_index and text_index are guaranteed valid).

Other notes
- Do not rely on Python’s negative indexing to indicate “no match”; use an explicit sentinel (e.g. None or -1 checked) and test it before using indices.
- This fix stops the incorrect indexing and prevents the loop from continuing when there are no more matches.
