What happened
- The tests fail because add_number loses the sign on -0.0. The code that decides how to format a numeric value treats the value as “zero” or “an integer” and formats it using the integer/zero branch (which produces "0" or "0.0" without a sign). That branch checks value == 0.0 or value.is_integer() and therefore collapses -0.0 to "0", so the negative sign is dropped.

Where it occurs
- In CodeConsumer.add_number, when the method branches to format integers or zero values (the code path that produces "0" or "42"), the test case negative zero (value = -0.0) follows that path and ends up appended as "0". The loss of the sign happens before the append call, inside the integer/zero formatting branch. The relevant variables and methods are:
  - value (the float passed to add_number)
  - CodeConsumer.is_negative_zero(value) — the class already has a utility for negative-zero detection
  - the branch that checks value == 0.0 or value.is_integer() and then converts/prints as an integer/zero string

Why it happened (root cause)
- Python representations and common formatting drop the sign bit for zero. A direct numeric test like value == 0.0 or value.is_integer() is true for both +0.0 and -0.0, so the code treats them the same.
- The integer/zero formatting branch does not check for the negative-zero special case, so -0.0 is printed as "0" (or "0.0") rather than preserving the sign as "-0.0".
- The test expects negative zero to be preserved as "-0.0" to keep the sign information.

Exact, actionable fix (step-by-step)
1. Open hit06_codeconsumer.py and find add_number(self, value). Locate the branch where the code formats zeros and integer values — the branch that runs when value == 0.0 or value.is_integer().
2. At the very start of add_number (or at least before any integer/zero formatting), add an explicit negative-zero check using the existing utility:
   - if CodeConsumer.is_negative_zero(value):
         self.append("-0.0")
         return
   This makes sure -0.0 is handled before it can fall through to the integer/zero formatting branch.
3. Leave the rest of the formatting logic unchanged for other values (positive zero, integers, floats, scientific notation, inf, NaN).
4. Run the tests.

Concrete patch example (insert into add_number before integer/zero formatting):
- Before:
    def add_number(self, value):
        # existing code that checks value == 0.0 or value.is_integer() ...
- After:
    def add_number(self, value):
        # Preserve negative zero sign first
        if CodeConsumer.is_negative_zero(value):
            self.append("-0.0")
            return
        # existing code that checks value == 0.0 or value.is_integer() ...

Why this fix works
- It preserves the sign bit for -0.0 by catching that special case before the branch that collapses values into integer/zero text.
- It uses the class's is_negative_zero utility, so no low-level bit fiddling is required and behaviour stays consistent with other uses of that utility.

Notes and pitfalls
- Do not change general integer formatting (int/value.is_integer() branches) to try to handle -0.0 there — handle negative zero as a distinct special case before those branches.
- Make sure you append exactly "-0.0" (tests expect that string) not "-0" or "-0.00".
- Keep handling for positive 0.0 unchanged (it should still produce "0" per tests).
