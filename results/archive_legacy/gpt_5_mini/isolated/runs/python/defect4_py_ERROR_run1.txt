Three concrete, common root causes and how to confirm each one
1) Off-by-one in middle calculation
- Symptom: returned index is 2 or 3 instead of 1 (common with mid = (left + right) / 2 vs (left + right) // 2 or with integer rounding toward the wrong side).
- What to check:
  - Find the line where mid is computed. Example wrong lines:
      mid = (left + right) / 2     # non-integer division in some languages
      mid = (left + right + 1) // 2 # forces right bias
  - Confirm you are using integer division and the intended bias (left vs right).
- How to reproduce and log:
  - Add a debug print immediately after mid is computed:
      print("DEBUG: left=%d right=%d mid=%d" % (left, right, mid))
- Fixes to try:
  - Use mid = left + (right - left) // 2 (avoids overflow and gives left bias).
  - If you deliberately want the right-biased mid, do mid = (left + right + 1) // 2 — but be consistent with tests.

2) Wrong initialization or update of max_middle_index / max_value
- Symptom: You never update the best index properly, or you start with a bad initial value so a later larger value is accepted when it shouldn’t be.
- What to check:
  - Initialization lines. Example problematic patterns:
      max_value = 0  # but values might be negative
      max_middle_index = 0  # but valid best index could be -1 if none found
  - Update condition lines. Look for:
      if value >= max_value:
          max_value = value
          max_middle_index = mid
    This will prefer later indices on ties. Tests may expect earlier index on ties.
- How to reproduce and log:
  - Add prints around the update:
      print("DEBUG: mid=%d value=%r current_best_value=%r current_best_index=%r" %
            (mid, value, max_value, max_middle_index))
- Fixes to try:
  - Initialize max_value to -inf (or a sentinel lower than any possible value).
  - Use if value > max_value: (not >=) if the spec requires choosing the first occurrence.
  - Make sure max_middle_index is initialized to an impossible index (like -1) if no candidate yet.

3) Wrong loop/scan range (you inspect the wrong set of candidates)
- Symptom: You check indices 0..N instead of 0..N-1, or you shift the index when storing it (e.g., storing i+1 instead of i).
- What to check:
  - The for/while loop boundaries:
      for i in range(1, n+1):  # but tests expect 0-based indices
  - Any place you store/return index: return i, return i+1, return i-1
- How to reproduce and log:
  - Print the list of candidate indices you considered:
      print("DEBUG: candidates =", candidates)  # or print loop values each iteration
- Fixes to try:
  - Ensure loop uses correct inclusive/exclusive bounds.
  - Ensure index arithmetic matches the API (0-based vs 1-based).
