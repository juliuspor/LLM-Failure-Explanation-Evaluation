What happens
- Root cause: when lower is greater than text length, the code sets upper = lower (line: if upper < lower: upper = lower). That makes upper > len(text). Later calls to substring_java(text, 0, upper) pass an end index bigger than text.length(), which causes a StringIndexOutOfBoundsException (or equivalent) inside substring_java.
- Where it occurs: the problematic path is the branch that reaches substring_java(text, 0, upper) (lines in the block after index == -1, and in the block when index > upper). The bad assignment is the line if upper < lower: upper = lower (just after the upper bounds adjustment).
- Why it happens: the code first clamps upper down to len(text) only if upper == -1 or upper > len(text). Then, without checking lower against text length, it forces upper to be at least lower. If lower was already > len(text), this restores upper to a value > len(text), undoing the earlier clamp and creating an invalid end index for substring_java.

Concrete example to reproduce
- Call abbreviate("short", lower = 10, upper = -1, append_to_end = "...").
- Execution:
  - upper == -1 so upper becomes len("short") = 5
  - upper < lower (5 < 10) so upper = lower -> upper = 10
  - substring_java(text, 0, upper) => substring_java("short", 0, 10) -> throws out-of-bounds

Exact lines to fix
- The problematic lines:
  1) if upper == -1 or upper > len(text): upper = len(text)
  2) if upper < lower: upper = lower
  3) later calls like substring_java(text, 0, upper)

How to fix (explicit steps)
1. Ensure lower is not allowed to be greater than len(text), or ensure upper is clamped after any change that can make it > len(text).
2. Two simple, explicit fixes you can apply (pick one):

Option A — clamp lower to text length:
- Insert after the first if-block:
    if lower > len(text):
        lower = len(text)
  Keep the existing "if upper < lower: upper = lower". This guarantees upper will never exceed len(text).

Option B — enforce upper <= len(text) after adjusting for lower:
- Replace the two lines:
    if upper == -1 or upper > len(text):
        upper = len(text)
    if upper < lower:
        upper = lower
  with:
    if upper == -1:
        upper = len(text)
    if lower > len(text):
        lower = len(text)
    if upper > len(text):
        upper = len(text)
    if upper < lower:
        upper = lower
  (or more compactly: upper = min(upper == -1 ? len(text) : upper, len(text)); lower = min(lower, len(text)); if upper < lower: upper = lower)

Option C — validate inputs and fail fast:
- At function start, return text (or throw IllegalArgumentException) if lower > len(text) — make it explicit so you don't get hidden exceptions.

Which I recommend
- Use Option A: clamp lower to len(text) right after you clamp upper. It's minimal, keeps existing behavior, and prevents invalid substring bounds.

One-line test to confirm fix
- Before fix: abbreviate("short", 10, -1, "...") -> throws
- After Option A: abbreviate("short", 10, -1, "...") -> behaves like abbreviate("short", 5, 5, "...") and returns "short" (or "short" + append depending on your desired behavior).

Summary
- Cause: upper gets increased to lower even when lower > text length, producing an out-of-range end index for substring_java.
- Fix: ensure lower is clamped to text length (or re-clamp upper after comparing to lower), so substring_java never receives an end index > len(text).
