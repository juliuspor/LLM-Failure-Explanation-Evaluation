What happened (root cause)
- In the "max middle" branch you fetch the wrong existing period to compare against.
- At the start of the max-middle block you wrote:
  max_middle_period = self.get_data_item(self._min_middle_index).get_period()
  That uses self._min_middle_index instead of self._max_middle_index.
- Because you compare the new middle to the period at _min_middle_index, the comparison can be wrong and you may overwrite _max_middle_index with the wrong index (this matches the failing test: expected 1, got 3).

Where it occurs (code location)
- The error is in the block beginning with:
    if self._max_middle_index >= 0:
  Inside that block you call get_data_item(self._min_middle_index) (the wrong variable).

Why that causes the test failure
- For the provided sequence of midpoints (indices 0=50, 1=150, 2=25, 3=75), when adding index 3 the code compares its middle (75) to the middle computed from the period at _min_middle_index (which could be 25) instead of the current maximum (150). That makes 75 appear larger than the compared value (25) and incorrectly sets _max_middle_index to 3. The intended comparison was against the current maximum (150 at index 1), so index 1 should have remained the max.

How to fix (concrete steps)
1. Change the get_data_item call in the max-middle branch to use self._max_middle_index:
   Replace:
     max_middle_period = self.get_data_item(self._min_middle_index).get_period()
   With:
     max_middle_period = self.get_data_item(self._max_middle_index).get_period()
2. Run the test suite. The failing test should now pass.
3. Optionally grep for similar copy-paste mistakes (e.g., using _min_... where _max_... is intended) and add/keep unit tests for min/max indices to prevent regressions.
