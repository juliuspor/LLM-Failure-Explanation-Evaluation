What happened (where it fails)
- The failure happens at the line that computes g:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- You intended to clamp the input into v (lines above), but the formula still uses value instead of the clamped v.
- If value is outside [self._lower_bound, self._upper_bound], the computed g can fall outside 0..255 and the subsequent if check raises ValueError.

Why it fails (root cause)
- You compute v correctly:
    v = max(value, self._lower_bound)
    v = min(v, self._upper_bound)
  but then ignore v and use the original value in the normalization. That means the clamping had no effect on g.
- Also, if self._upper_bound == self._lower_bound you will divide by zero in that same line.

Exact fixes (step-by-step)
1. Use the clamped variable v in the normalization:
   Replace the g line with:
     g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
2. Guard against zero-width range (avoid division by zero). For example:
   if self._upper_bound == self._lower_bound:
       g = 0  # or 255 or some defined value for this degenerate case
   else:
       g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
3. Optionally improve the ValueError message to include the numeric g and range to aid debugging. For example:
   if g < 0 or g > 255:
       raise ValueError(f"Computed g={g} outside 0..255 for value={value}, bounds=({self._lower_bound},{self._upper_bound})")
