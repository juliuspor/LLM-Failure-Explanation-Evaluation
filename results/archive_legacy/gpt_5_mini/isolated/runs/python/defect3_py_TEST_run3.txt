What failed
- The tests fail with an IndexError when the translator sees a character represented as a UTF-16 surrogate pair (an emoji like "\U0001F630") followed by another character. The failure happens while advancing the character index inside the translator loop.

Where it happens (methods and variables)
- The bug is in the translate method (CharSequenceTranslator.translate / CsvEscaper.translate_string) that iterates the input string using an integer index variable named index (or i).
- The code calls helper(s) like code_point_at and/or to_utf16_code_units to inspect characters, but still advances index by 1 unconditionally (index += 1).
- When the code encounters a surrogate pair (two UTF-16 code units), the translator treats it as a single code point in some places but advances only one UTF-16 code unit in others. That mismatch leads to an attempt to read beyond the string end (IndexError) or to skip/duplicate characters.

Why this happens (root cause)
- Python strings are sequences of Unicode code points. But the code you have is written around UTF-16 code-unit semantics (helpers named to_utf16_code_units and code_point_at reveal that).
- A Unicode character outside the Basic Multilingual Plane (BMP), such as U+1F630 (😰), is represented in UTF-16 as two code units (a surrogate pair). If the translator mixes "advance by code point" and "advance by UTF-16 code unit" logic, the index becomes inconsistent.
- Specifically:
  - The code_point_at (or equivalent) looks ahead to consume a surrogate pair as a single code point.
  - The loop that advances index always does index += 1 (one code unit) instead of advancing past both code units when a surrogate pair was consumed.
  - As a result, after processing the high surrogate, the loop tries to read the low surrogate as if it were a separate character; if the surrogate pair was at the end, reading the next code unit causes an IndexError, or the next step tries to consume a non-existent next character.
- In short: the translator reads the input with a mix of "one code point" and "one code unit" semantics, causing the index to be advanced incorrectly when it encounters surrogate pairs.

Exact places to look in your code
- The while loop that uses index (e.g., while index < len(seq):) and the line that does index += 1 (or index = index + 1). That single unconditional increment is the culprit.
- The code that calls code_point_at or to_utf16_code_units immediately before or after that increment—those helpers imply the code expects to handle surrogate pairs but the loop increment doesn't.

Concrete fixes (step-by-step)
Pick one of these two approaches:

A. Treat Python strings as sequences of code points (recommended, simplest)
1. Stop handling surrogate pairs specially. Iterate characters directly:
   - Replace the index-based loop with "for ch in s:" (or use range(len(s)) but operate on ch = s[i]).
2. Use ord(ch) to get the code point. That always returns the full Unicode code point (e.g., ord("😰") == 0x1F630).
3. Remove UTF-16 helpers (to_utf16_code_units, code_point_at) or rewrite them to operate on Python code points.
4. Make sure any logic that expected UTF-16 units is rewritten to use code points instead.

B. If you must work in UTF-16 code units
1. Encode the string to UTF-16 code units explicitly and iterate units:
   - For example, get a list of 16-bit code units by encoding to 'utf-16-le' and splitting into 2-byte units, or use the 'surrogates' behavior explicitly.
2. Ensure that when you detect a high surrogate (0xD800..0xDBFF) you consume two units (advance index by 2), and when you detect a BMP character advance by 1.
3. Where you use code_point_at, make sure index values refer to UTF-16 code units consistently.
4. Add bounds checks before trying to read the low surrogate: if high-surrogate at last unit, treat it as replacement or raise a controlled error.

Concrete code patch suggestions
- If your loop currently looks like:
    while index < len(seq):
        cp = code_point_at(seq, index)
        ... process cp ...
        index += 1
  change it to:
    while index < len(seq):
        cp = code_point_at(seq, index)
        ... process cp ...
        # cp may be a surrogate pair: advance by number of UTF-16 code units consumed
        index += 2 if cp > 0xFFFF else 1
  or better (Pythonic):
    for ch in seq:
        cp = ord(ch)
        ... process cp ...

Tests to run after fix
- Run the provided test suite (test_translate_surrogate_pair_with_suffix and the others).
- Add an explicit test where a surrogate pair is at the last position (already present) and where multiple surrogate pairs and ASCII are mixed (already present).

Summary
- Root cause: index advancement is inconsistent for surrogate pairs — the loop advances by 1 code unit even when a character consumed was a UTF-16 surrogate pair (2 code units).
- Where: the while loop that uses index and the line index += 1 (or equivalent) inside CharSequenceTranslator.translate / CsvEscaper.translate_string.
- Fix: either iterate by Python characters (for ch in s) and use ord(ch), or when working with UTF-16 code units advance index by 2 for surrogate pairs and always check bounds before reading the low surrogate.
