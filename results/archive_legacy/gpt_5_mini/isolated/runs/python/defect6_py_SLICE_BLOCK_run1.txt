What fails
- Python raises an indentation error at the start of the inner branch (around lines 183–186). You will see either
  - TabError: inconsistent use of tabs and spaces in indentation
  - or IndentationError: unexpected indent
  pointing at the line with mantissa //= 10 (line 184 in your snippet).

Why it fails (root cause)
- The logic block starting at line 181 ("while mantissa % 10 == 0:") contains an if (line 182) and then two lines intended to run when that if is true (the mantissa //= 10 and exp += 1 on lines 184–185).
- Those two lines are indented with a different kind of indentation than the surrounding code (a different number of spaces or tabs). Python requires consistent indentation (same mixture of tabs/spaces) within a block. Because those two lines use inconsistent indentation, the parser treats them as an unexpected indent and raises the error before any of the numerical logic runs.
- This is purely an indentation/tokenization problem — not an arithmetic or algorithm bug — so fixing indentation is sufficient.

Exactly where it occurs
- The problem shows up when the interpreter reaches the inner if at line 183 ("if check_val == value:"). The following line (mantissa //= 10) is where the interpreter complains (line 184 in your listing).

How to fix it (clear, actionable steps)
1. Open the source file in an editor that can show tabs vs spaces (or run a tool).
2. Make the indentation consistent throughout the function (and file). The usual and recommended convention is 4 spaces per indent level and no tabs.
   - Replace any tabs with 4 spaces.
   - Make sure the lines inside the if (mantissa //= 10 and exp += 1) are indented one additional level relative to the if line, matching the indentation style used in the rest of the file.
3. Verify with python -m tabnanny path/to/file.py — it will flag mixed indentation problems.
4. Run the code again; the indentation error should be gone. After fixing indentation, the numeric logic (removing trailing zeros and adjusting exp) will execute as intended.
