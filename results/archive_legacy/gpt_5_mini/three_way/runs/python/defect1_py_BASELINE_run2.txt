What happened (where it fails)
- The failure occurs at lines 396–397:
  if minutes_offset < 0 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")
- The test calls DateTimeZone.for_offset_hours_minutes(0, -15). minutes_offset is -15, which is a valid value according to the docstring (minutes must be in -59..+59). But the code rejects any negative minutes unconditionally and raises "Minutes out of range: -15". So the code fails because it enforces the wrong minutes range.

Why it happened (root cause)
- The code incorrectly checks minutes_offset against 0..59 instead of the required -59..+59. That unconditional check rejects all negative minute inputs even when the docstring and tests expect negative minutes to be accepted in certain cases (e.g., hours == 0 or hours < 0).
- Because that check happens before any sign-dependent logic, the function never reaches the later code that combines hours and minutes into a total offset.

Where in the code to fix it
- Lines 396–397: change the minutes validation to allow negative minutes within -59..+59.
- Also add the sign-consistency validation the docstring requires:
  - Positive hours with negative minutes should be invalid and raise ValueError.
  - Zero hours may have negative minutes (e.g., (0, -15) -> -00:15).
  - Negative hours with positive or negative minutes should be accepted and treated as increasing the negative magnitude.

Concrete steps and suggested code
1. Replace the current minutes check at lines 396–397 with:
   if minutes_offset < -59 or minutes_offset > 59:
       raise ValueError(f"Minutes out of range: {minutes_offset}")

2. Add a check after that to reject mixed-sign positive-hours/negative-minutes:
   if hours_offset > 0 and minutes_offset < 0:
       raise ValueError("Positive hours with negative minutes is not permitted")

3. Compute total minutes in a clear, correct way (example replacement for the try block):
   try:
       total_minutes = hours_offset * 60
       if hours_offset < 0:
           # For negative hours, minutes should reduce total further (both (-2,15) and (-2,-15) -> -135)
           total_minutes -= abs(minutes_offset)
       else:
           # For zero or positive hours, minutes add or (zero hours with negative minutes) subtract
           total_minutes += minutes_offset
       offset = FieldUtils.safe_multiply(total_minutes, DateTimeConstants.MILLIS_PER_MINUTE)
   except ArithmeticError:
       raise ValueError("Offset is too large")

Why this fixes the failure
- Step 1 allows -15 to pass the validation so the (0, -15) test reaches calculation.
- Step 2 enforces the rule that (2, -15) is invalid (test expects ValueError).
- Step 3 makes (-2, 15) and (-2, -15) produce the same negative total offset, matching the docstring and tests.
