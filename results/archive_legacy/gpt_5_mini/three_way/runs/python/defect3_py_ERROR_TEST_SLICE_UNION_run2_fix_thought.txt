The bug arises because translate assumes translator returned 'consumed' in codepoints, while some translators (e.g., CsvEscaper) return counts in UTF-16 code units. Simplest robust fix is to treat 'consumed' as the number of UTF-16 code units consumed and advance pos by that amount. Keep existing logic for consumed==0 where we copy the current codepoint and advance by its UTF-16 length (char_count). Also validate consumed is non-negative and doesn't advance past length to avoid infinite loops or out-of-range. Return only the function code.
