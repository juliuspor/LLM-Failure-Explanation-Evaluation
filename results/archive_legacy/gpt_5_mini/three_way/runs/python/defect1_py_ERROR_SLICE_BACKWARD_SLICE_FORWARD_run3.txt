What happened (root cause)
- The ValueError is raised at line 396 because minutes_offset is negative: minutes_offset == -15. The check on lines 396–397 does not allow values below 0, so it throws "Minutes out of range: -15".
- The negative value is produced earlier in this method while parsing/normalizing the offset (you can see later adjustments at lines 402 and 404 that expect to correct minutes_offset). In other words, minutes_offset can be temporarily negative until the code at 402/404 flips or adds hours_in_minutes. But the range check at 396 runs before that normalization, so valid inputs that generate a temporary negative minutes_offset (for example a -0:15 offset) fail prematurely.

Where it occurs
- Failure line: 396–397:
  if minutes_offset < 0 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")
- Normalization that would fix minutes_offset is at 402 and 404:
  minutes_offset = hours_in_minutes - minutes_offset
  minutes_offset = hours_in_minutes + minutes_offset

Why it fails now
- The code enforces minutes must be 0..59 before it has applied the adjustments that make minutes_offset positive and within that range. Because minutes_offset is -15 at the time of the check, the method raises the error even though later code intends to handle that case.

Concrete fixes (specific steps and exact places to change)
Option A — Move the range check after normalization
1. Remove lines 396–397 (or comment them out).
2. Place the minutes range validation after lines 402/404, once minutes_offset has been normalized to a non-negative minute value.
   - New location: immediately after the block that sets minutes_offset at 402/404 and before the multiplication at line 405.
   - Use the same check: if minutes_offset < 0 or minutes_offset > 59: raise ValueError(...)

Option B — Validate against absolute minutes using the sign variable
1. If the code computes and stores a sign (e.g., sign = -1 or +1) before computing minutes_offset, change the check to validate absolute minutes instead:
   - Replace the check at 396–397 with:
     if abs(minutes_offset) > 59:
         raise ValueError(f"Minutes out of range: {minutes_offset}")
   - This lets temporary negative minute values (like -15) pass, but still rejects invalid minute magnitudes such as -75 or 100.
2. Keep the later normalization at 402/404 unchanged.

Why these fixes work
- Option A ensures validation happens only after minutes_offset has been put into its final, normalized form (so -15 will become 45 or similar as intended).
- Option B accepts negative temporary minute values but still rejects values whose magnitude is invalid, preventing the premature exception.

Which to choose
- If the negative minutes are expected only as a temporary intermediate during parsing/normalization, Option A (move the check) is the clearest and least intrusive.
- If other code paths may produce negative minutes whose magnitude you still want to reject early, Option B is acceptable.

Example minimal change (Option A)
- Delete lines 396–397.
- Insert after line 404 (before line 405):
  if minutes_offset < 0 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")

This prevents the "Minutes out of range: -15" error caused by the premature check and keeps the intended validation in place.
