What happened (root cause)
- At __init__ you initialize self._min_start_index = -1 (line 253). Good.
- When adding/inspecting items you try to set or update _min_start_index by comparing the new item's start against the current minimum (lines 331–346 and 340–349).
- But line 349 unconditionally assigns self._min_start_index = index regardless of the comparison. Because of that unconditional assignment the “minimum start” index gets overwritten with whatever item was last processed, not the item with the smallest start time.
- The unconditional overwrite defeats the intent of the conditional block at lines 340–347 that should only update _min_start_index when start < min_start. As a result the code reports the wrong index later (your test expected max_middle_index = 1 but got 3), because the min/max tracking logic was corrupted by this overwrite.
