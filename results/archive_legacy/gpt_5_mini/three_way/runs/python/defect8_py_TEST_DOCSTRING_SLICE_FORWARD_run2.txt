What happened (where it fails)
- The failure occurs in LocaleUtils.to_locale when parsing strings like "fr__POSIX".
- The relevant code is at the slice you provided (roughly lines 243–247). It assumes a 2-letter language followed by a single '_' then a 2-letter country:
  - It returns Locale(locale_str[0:2], locale_str[3:5]) (line 243)
  - Then it checks locale_str[5] and returns Locale(locale_str[0:2], locale_str[3:5], locale_str[6:]) (lines 245–247)

Why that fails for "fr__POSIX" (root cause)
- "fr__POSIX" has a double underscore to indicate an empty country and a variant: language = "fr", country = "" (empty), variant = "POSIX".
- The code still tries to read a 2-character country from locale_str[3:5]. For "fr__POSIX" those characters are "_" and "P" (locale_str[3] == '_' , locale_str[4] == 'P'), not two uppercase letters. The parser therefore treats the country as invalid.
- Next the code checks locale_str[5] expecting that position 5 is '_' (the separator before variant). For "fr__POSIX", locale_str[5] == 'O', so the check fails and a ValueError is raised.
- In short: the parser never checks for the double-underscore case (empty country). It always assumes a country exists at indices 3–4 and that the variant separator is at index 5, so language__variant input triggers the "Invalid locale format" path.

Precisely where to change it (specific variables/lines)
- In LocaleUtils.to_locale, around lines 243–247:
  - Before you slice out locale_str[3:5] as the country, check whether locale_str[3] is '_' (i.e. there's a second underscore) meaning the country is empty.
  - If locale_str[3] == '_' treat country = "" and variant = locale_str[4:] (or locale_str[4:] depending on exact index logic) and return Locale(locale_str[0:2], "", variant).
  - Only if locale_str[3] is not '_' should you read the country as locale_str[3:5] and continue with the existing checks for a variant separator at locale_str[5].

Suggested concrete fix (small patch)
Replace the current block (lines ~243–247):

    return Locale(locale_str[0:2], locale_str[3:5])
    if locale_str[5] != '_':
        raise ValueError(f"Invalid locale format: {locale_str}")
    return Locale(locale_str[0:2], locale_str[3:5], locale_str[6:])

With logic like:

    # handle empty country with double underscore: "fr__POSIX"
    if locale_str[3] == '_':
        # country is empty, variant starts at index 4
        return Locale(locale_str[0:2], "", locale_str[4:])
    # otherwise expect a two-letter country at 3:5
    country = locale_str[3:5]
    # if there's no variant part, return language+country
    if len(locale_str) == 5:
        return Locale(locale_str[0:2], country)
    # if there is a variant, require underscore at index 5
    if locale_str[5] != '_':
        raise ValueError(f"Invalid locale format: {locale_str}")
    return Locale(locale_str[0:2], country, locale_str[6:])

Why this fixes it
- For "fr__POSIX" the new check sees locale_str[3] == '_' and returns Locale("fr", "", "POSIX"), matching the intended behavior.
- For regular "en_US" and "en_US_WIN" cases the code continues to parse country from locale_str[3:5] as before.
- This explicitly implements the language__variant case that the tests expect.

Summary
- The bug is a missing branch for the language__variant form (empty country). The parser always assumes a 2-letter country at indices 3–4 and checks index 5 for the variant separator, which is wrong when the country is empty. Add an explicit check for locale_str[3] == '_' and handle the empty-country variant case before attempting to read locale_str[3:5].
