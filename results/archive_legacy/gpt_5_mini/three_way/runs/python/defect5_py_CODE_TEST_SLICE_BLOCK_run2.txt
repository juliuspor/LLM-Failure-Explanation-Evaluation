What happened (root cause)
- At line 346 you set inferred_type = type(array). That returns the Python type of the list object (i.e. list), not the element/component type stored in the list. Example: if arr = ["a","b"], type(arr) is list, not str.
- The code later treats inferred_type as the array element type (used by ArrayUtils._copy_list_grow1 and by the check at lines 346–351). Because inferred_type becomes list instead of the element type, the type logic and the conditional that raises TypeError are incorrect. Concretely:
  - When array is non-None you should be inferring the element type from the array contents (e.g. str for ["a"]), not using type(array) (which is list).
  - When array is None and element is None you set inferred_type = object (reasonable), but when array is non-None yet contains only None values you still produce wrong inferred_type (list), and you never fall into the expected_type check correctly.
- The TypeError-raising branch (slice lines 346–351) expects inferred_type to be object when the component type is unknown, but because inferred_type is list in many cases the branch behavior is wrong (it either raises when it shouldn’t or fails to raise when it should). That is why the test case ArrayUtils.add(None, None, expected_type=str) fails: the code is using the wrong source for the inferred type and therefore the subsequent logic is invalid.

Exactly where (lines / variables)
- Line with inferred_type = type(array) (the first branch) is wrong — you are using type(array) (the list type) instead of the element/component type.
- The conditional at lines 346–351 checks inferred_type == object to decide whether to raise a TypeError when expected_type is provided. Because inferred_type is often list, that comparison does not reflect whether element/component type is actually unknown.

Why this produces the observed problem
- _copy_list_grow1 and the TypeError check assume inferred_type represents element type (or object when unknown). Because inferred_type is list, those components behave incorrectly:
  - Wrong array element type passed to copy function (it expects the element type/class, not the list class).
  - Wrong decision about whether a cast to expected_type is safe, leading to either a false-positive TypeError or failing to detect a real mismatch.

Concrete fixes (step-by-step)
1. Infer element type from the array contents, not from type(array):
   - If array is not None:
     - Find the first non-None element in array and use type(that element).
     - If all elements are None, fall back to expected_type (if provided) or object.
   - Else if array is None:
     - If element is not None -> inferred_type = type(element)
     - Else -> inferred_type = expected_type if provided else object
2. Use inferred element type when calling _copy_list_grow1 (or better, pass expected_type explicitly if your copy function needs it).
3. Keep the TypeError check but compare the correct conditions: raise only when inferred_type is object (unknown) and expected_type is supplied and not object.

Example replacement logic (pseudo-Python, replace the current inferred_type block):
    if array is not None:
        # find first non-None element
        first_non_none = next((x for x in array if x is not None), None)
        if first_non_none is not None:
            inferred_type = type(first_non_none)
        else:
            inferred_type = expected_type if expected_type is not None else object
    elif element is not None:
        inferred_type = type(element)
    else:
        inferred_type = expected_type if expected_type is not None else object

Then call:
    new_list = ArrayUtils._copy_list_grow1(array, inferred_type)

And keep the TypeError check as:
    if expected_type is not None and inferred_type is object and expected_type is not object:
        raise TypeError(...)

Why this fix works
- When array has elements you get the actual element type (not list).
- When array is None and both element and expected_type are None you still get object.
- When array elements are all None you use expected_type if caller supplied it (so ArrayUtils.add(None, None, expected_type=str) will infer str and not produce a cast error).
- The _copy_list_grow1 and the TypeError check then receive the correct element/component type and behave as intended.

No other changes are required unless _copy_list_grow1 itself expects a different form of type argument — if it expects a class name or a factory, pass/convert accordingly.
