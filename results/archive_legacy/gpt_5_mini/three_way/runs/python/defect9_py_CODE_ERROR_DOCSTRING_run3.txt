What happened
- The code crashes in this loop that tries to estimate the result buffer size (the "increase" loop).
- That loop calls len(replacement_list[i]) and len(search_list[i]) without checking whether either element is None.
- If either replacement_list[i] or search_list[i] is None, Python raises TypeError: object of type 'NoneType' has no len().

Where it occurs (exact spot)
- The failing loop starts at the comment "# BUG: this loop is not null-safe for elements inside the arrays" and contains:
  greater = len(replacement_list[i]) - len(search_list[i])
- Because this loop does not perform the same None/empty checks used later in the function, a None element will cause the TypeError there.

Why it happens (root cause)
- The function intends to ignore None and empty search/replacement entries (you already check for those in the later matching loops), but the initial sizing loop assumes every pair is a non-None string.
- Therefore inputs like search_list = ["a", None] or replacement_list = [None, "b"] will fail even though the rest of the function would have skipped those entries.

Minimal failing example
- replace_each("x", ["a", None], ["A", "B"]) raises TypeError in the sizing loop.

Exactly how to fix it
1. Apply the same guards used elsewhere before calculating lengths. Replace the sizing loop with one that skips None/empty search strings and None replacements. For example (line numbers relative to your snippet):
   - At the loop with comment "# BUG: this loop ...", change it to:
     for i in range(search_length):
         if search_list[i] is None or len(search_list[i]) == 0 or replacement_list[i] is None:
             continue
         greater = len(replacement_list[i]) - len(search_list[i])
         if greater > 0:
             increase += 3 * greater

2. Alternative (cleaner): iterate pairs with zip and check each pair:
     for s, r in zip(search_list, replacement_list):
         if s is None or len(s) == 0 or r is None:
             continue
         greater = len(r) - len(s)
         if greater > 0:
             increase += 3 * greater

Why this fix works
- It prevents calling len() on None and keeps behavior consistent with the later matching code, which already ignores None/empty entries.

That’s it — the cause is the one loop that calls len(...) without guarding for None; add the same None/empty checks and the crash goes away.
