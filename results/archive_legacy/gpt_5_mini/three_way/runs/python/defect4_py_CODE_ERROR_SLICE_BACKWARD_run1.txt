What happened
- In the "max middle" section you compare the new period's middle against the wrong existing period. Instead of retrieving the period at self._max_middle_index you call get_data_item(self._min_middle_index).get_period(). That means you are comparing against the min-middle period, not the current max-middle period, so _max_middle_index can be updated incorrectly.
- That explains the test failure: the code used the min-middle value for the comparison and ended up setting _max_middle_index to 3 instead of keeping it at 1.

Where it occurs (in your snippet)
- The bug is in this block:

      if self._max_middle_index >= 0:
          max_middle_period = self.get_data_item(self._min_middle_index).get_period()
                                      ^^^^^^^^^^^^^^^^  <-- wrong variable
          ...

- You should be using self._max_middle_index there.

Why this causes the wrong result
- When computing max_middle you must compare the new middle against the period currently recorded as the maximum. By fetching the min-middle period instead, the comparison uses the wrong reference value. If the min-middle value is smaller (as it usually is), the comparison will often succeed and overwrite _max_middle_index even though the new period is not actually the largest, producing the observed mismatch.

How to fix (exact change)
- Replace the mistaken line with:

      max_middle_period = self.get_data_item(self._max_middle_index).get_period()

- No other logic in that block needs to change.

Suggested quick checks after fix
1. Run the failing test to confirm max_middle_index becomes 1.
2. Add a small unit test that inserts periods in different orders and asserts min/max start/middle/end indices to catch regressions.
3. Optionally add an assertion or a short comment at that block to make the intent explicit (compare against self._max_middle_index).
