What happened (where it failed)
- The IndexError comes from to_utf16_code_units (lines you showed) when code_point_at(code_units, pos) is called with pos == len(code_units). That check happens at:
  - slice lines 35–36: if index < 0 or index >= len(code_units): raise IndexError(...)
- In translate(), that out-of-range call happens inside the for pt in range(consumed): loop near the end. After the first iteration the code increments pos and on a later iteration pos can equal length, and the next call to code_point_at(code_units, pos) raises the IndexError.

Why it failed (root cause)
- translate_codepoint(...) returns consumed (the number of codepoints it handled), but the loop uses that number to repeatedly call code_point_at(code_units, pos) and advance pos by char_count(...). The code assumes there are always enough remaining code units for all consumed codepoints, but it does not guarantee that before calling code_point_at each time.
- Concretely:
  - Suppose pos starts near the end and consumed > 1 (translate_codepoint handled a surrogate pair or multiple codepoints). The loop does:
    for pt in range(consumed):
        pos += char_count(code_point_at(code_units, pos))
  - After some iterations pos can reach len(code_units). The loop still tries another iteration and calls code_point_at(code_units, pos) with pos == len(code_units), which triggers the index check in to_utf16_code_units (index >= len -> IndexError).
- In short: the loop calls code_point_at at positions that are not verified to be in-bounds, so it can read past the end of code_units.

Exactly where to change (concrete fixes)
Pick one of these explicit fixes depending on how translate_codepoint is supposed to behave:

1) If translate_codepoint is supposed to return UTF-16 code unit count (1 or 2) instead of "codepoint count"
   - Change translate_codepoint to return the number of UTF-16 code units consumed (units_consumed).
   - Replace the for-loop with a single increment:
        pos += units_consumed
   - This avoids any extra code_point_at calls and cannot overshoot because translate_codepoint already processed those units.

2) If translate_codepoint must return number of codepoints consumed and you need to advance pos by the actual UTF-16 units:
   - Compute the code points/units safely before the loop and stop if pos would reach the end. For example:
        for pt in range(consumed):
            if pos >= length:
                break   # or raise a clearer error
            cp = code_point_at(code_units, pos)   # safe now
            pos += char_count(cp)
   - This prevents calling code_point_at when pos == length.

3) Alternatively, compute the total units to skip without multiple calls:
   - Save start = pos
   - For pt in range(consumed):
        cp = code_point_at(code_units, pos)
        pos += char_count(cp)
     but check pos < length before each call as in (2). Then you know exactly how many units were advanced and won't call out-of-bounds.

What to do now (recommended)
- Prefer option (1): change translate_codepoint to return the number of UTF-16 code units consumed (1 or 2) and replace the loop by pos += consumed. That is the simplest and least error-prone if translators already work in UTF-16 units.
- If you cannot change translate_codepoint's contract, implement the bounds check shown in (2) to avoid calling code_point_at at pos == length, and surface a clear error if translate_codepoint promised to consume more codepoints than remain.

Short summary
The loop that advances pos after a non-zero consumed value calls code_point_at without verifying pos is still inside code_units. That lets pos reach len(code_units) and then call code_point_at(len), causing the IndexError. Fix by making translate_codepoint return the number of UTF-16 code units consumed (so you can do pos += consumed) or add a bounds check before each code_point_at call.
