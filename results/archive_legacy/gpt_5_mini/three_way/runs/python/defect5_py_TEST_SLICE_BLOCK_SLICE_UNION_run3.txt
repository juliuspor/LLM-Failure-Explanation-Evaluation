Why it fails (root cause)
- Both array and element are None in the failing call. That causes the inference branch to set inferred_type = object (line 340).
- expected_type is provided as str, so expected_type != object is true.
- The code treats inferred_type == object as an incompatible "object list" and unconditionally raises a TypeError (lines 346–351).
- In short: when both inputs are None, the function infers object and then refuses to produce a list of the requested expected_type, raising instead. The exception message even mirrors the original Java ClassCastException mentioned in the test.

Why this is wrong
- The function should accept expected_type as authoritative when array and element give no type information. When caller passes expected_type=str, returning [None] as a list of strings (or at least not erroring) is correct.
- The code both uses wrong inference (type(array) is the type of the container, not the component type) and treats inferred_type == object as a fatal incompatibility even when expected_type is available and meaningful.
