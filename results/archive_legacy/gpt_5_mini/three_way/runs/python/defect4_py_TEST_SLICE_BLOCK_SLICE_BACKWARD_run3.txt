- Concretely: when testing whether the new period's middle is greater than the currently recorded maximum-middle period, the code compares against (or assigns) the stored index variable instead of the stored middle value — and/or it assigns the wrong index variable when updating.     # compute middle for the incoming period
    if isinstance(period, SimpleTimePeriod):
        middle = (period.get_start_millis() + period.get_end_millis()) // 2
    else:
        middle = int((period.get_start().timestamp() * 1000 + period.get_end().timestamp() * 1000) // 2)

    if self._max_middle_index >= 0:
        max_middle_period = self.get_data_item(self._max_middle_index).get_period()
        if isinstance(max_middle_period, SimpleTimePeriod):
            max_middle = (max_middle_period.get_start_millis() + max_middle_period.get_end_millis()) // 2
        else:
            max_middle = int((max_middle_period.get_start().timestamp() * 1000 + max_middle_period.get_end().timestamp() * 1000) // 2)
        if middle > max_middle:
            self._max_middle_index = index
    else:
        self._max_middle_index = index

Why this fix works

- It compares numeric "middle" values, so only a larger middle can replace the current max.
- It assigns the index of the larger element (index) to self._max_middle_index.
- That prevents later smaller middles from overwriting the true maximum's index.
