What happened (where it failed)
- At line 176 you test if x == int(x) and then convert value = int(x) (lines 176–177).
- For x == -0.0, Python treats -0.0 == 0 as True, and int(-0.0) returns 0.
- That causes the code path that formats integers to run, and later self.add(str(value)) (line 192) produces "0".
- The test expected "-0.0", but the code printed "0", so the AssertionError arises.

Why it happens (root cause)
- The check x == int(x) loses the sign for negative zero because IEEE-754 negative zero compares equal to positive zero and int(-0.0) == 0. There is no preservation of the negative zero sign in value after converting to int, so you output "0" instead of preserving "-0.0".

Exactly where to change (concrete fixes)
Option A — keep current integer formatting but preserve negative zero as a float:
- Replace the condition on line 176 with an extra check that excludes negative zero:
  - From:
    if x == int(x) and not math.isinf(x) and not math.isnan(x):
  - To:
    if x == int(x) and not math.isinf(x) and not math.isnan(x) and not (x == 0.0 and math.copysign(1.0, x) < 0):
- This makes negative zero fall through to the else branch and use self.add(str(x)), which yields "-0.0".

Option B — detect negative zero early and explicitly add "-0.0":
- Add a check immediately before line 176:
  if x == 0.0 and math.copysign(1.0, x) < 0:
      self.add(str(x))   # or self.add("-0.0")
      return
- Then the existing integer logic can remain unchanged.

Why those fixes work
- math.copysign(1.0, x) < 0 reliably detects negative zero without changing other behavior. Either approach ensures "-0.0" is not converted to int(0) and lost.

Minimal patch (recommended, exact lines)
- Modify line 176 as:
  if x == int(x) and not math.isinf(x) and not math.isnan(x) and not (x == 0.0 and math.copysign(1.0, x) < 0):
  (no other changes required)

This addresses the root cause (x == int(x) treating -0.0 as 0) and will stop converting negative zero into the integer branch that prints "0".
