- The bug is in TimePeriodValues._update_bounds. The method both compares and updates "min/max" index fields, but it contains a stray unconditional assignment that overwrites the result of the comparison logic.
- In the code slice you provided, look at these lines inside _update_bounds:

  - Line 347:     self._min_start_index = index
  - Line 349: self._min_start_index = index

  The assignment at line 349 executes every time _update_bounds runs and therefore always sets _min_start_index to the most recently added index, regardless of the comparison that should determine the true minimum start. That explains tests like test_min_start_index failing: the "min" index is overwritten with the last added item.

- The intended flow: compute start (or middle/end), compare it with the currently saved min/max (min_start, max_middle, etc.), and only update the corresponding index (e.g. _min_start_index, _max_middle_index) when the new value is more extreme.

Even if the comparison on line 346 correctly detects that the new start is not smaller, line 349 still sets _min_start_index to index, so the saved index becomes the last one added. This results in wrong indices returned by get_min_start_index(), get_max_middle_index(), etc.
