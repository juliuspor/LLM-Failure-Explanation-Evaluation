What happened
- For the failing test you call WordUtils.abbreviate("0123456789", 15, 20, None). Here:
  - text length = 10
  - lower = 15
  - upper = 20
- Execution in your code:
  1. if upper == -1 or upper > len(text): upper = len(text)  —> upper becomes 10
  2. if upper < lower: upper = lower  —> upper becomes 15
  3. index = StringUtils.index_of(text, " ", lower) —> index is -1 (no space starting at pos 15)
  4. result = substring_java(text, 0, upper) —> calls substring_java(text, 0, 15)

substring_java then performs bounds checking and raises IndexError because end (15) > len(text) (10). The slice check in substring_java (from SLICE_BACKWARD) explicitly raises when end > len(s). So the code sets upper to a value larger than the string length and then calls substring_java with that invalid end index.

Why it failed (root cause)
- You allowed upper to be increased to lower (upper = lower) after you had already clamped upper to len(text). If lower is greater than len(text), upper becomes > len(text) and you never clamp it back before calling substring_java(…, 0, upper). That produces an out-of-range end index and triggers the IndexError.

Where it occurs
- The problem occurs around these lines in abbreviate:
  - the two ifs that set upper:
    - if upper == -1 or upper > len(text): upper = len(text)
    - if upper < lower: upper = lower
  - then later the call:
    - result = substring_java(text, 0, upper)

How to fix it (concrete steps)
Option A — short, closest to Java behavior (recommended)
- If lower is beyond the end of the string, return the original text immediately. Insert this check before any substring calls:
  - if lower >= len(text): return text
This matches the intended behavior for the test case that expects the original string when lower > length.

Option B — clamp upper after adjustment
- Ensure upper is never larger than len(text) before calling substring_java. After you set upper = lower (when upper < lower), clamp again:
  - if upper > len(text): upper = len(text)

Either option prevents substring_java from being called with end > len(text).

Exact places to change
- Right after the existing upper adjustments add one of the following:

  1) Return early (Option A):
     if lower >= len(text):
         return text

  2) Clamp (Option B):
     if upper > len(text):
         upper = len(text)

Either fix will stop substring_java(text, 0, upper) from being called with an invalid end index and will make the failing test pass.
