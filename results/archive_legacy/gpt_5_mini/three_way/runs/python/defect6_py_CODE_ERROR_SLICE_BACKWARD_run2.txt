What happened (where)
- The problem happens on the conditional at line 176:
  if x == int(x) and not math.isinf(x) and not math.isnan(x):
- When x is -0.0, Python treats -0.0 == 0.0 as True and int(-0.0) returns 0. So the code enters that branch, then sets value = int(x) which becomes 0 (losing the negative sign), and later calls self.add(str(value)) which produces "0".
- The failing test expected "-0.0", but the code produced "0" because the sign of negative zero was lost by int(x).

Why it happens (root cause)
- Floating point -0.0 compares equal to 0.0 and int(-0.0) == 0. There is no way to tell from the int result that the original float was negative zero. The code relies on int(x) for the integer-path formatting and never checks for negative zero first, so the sign is discarded.

How to fix it (specific steps and code)
Two straightforward fixes; pick one.

1) Quick explicit negative-zero check before the integer branch
- Add a check for negative zero and handle it before line 176:

    # detect negative zero and print it specially
    if x == 0.0 and math.copysign(1.0, x) < 0:
        self.add("-0.0")
        return

  Place this immediately before the existing "if x == int(x) ..." line. This ensures -0.0 is printed as "-0.0" and avoids the integer branch that loses the sign.

2) Preserve sign when using int(x)
- If you want to keep the rest of the integer-optimizing code but still preserve -0.0, change the condition so it does not treat negative zero as an integer case:

    if not (x == 0.0 and math.copysign(1.0, x) < 0) and x == int(x) and not math.isinf(x) and not math.isnan(x):

  Or equivalently capture a boolean:

    is_neg_zero = (x == 0.0 and math.copysign(1.0, x) < 0)
    if is_neg_zero:
        self.add("-0.0")
        return
    if x == int(x) and not math.isinf(x) and not math.isnan(x):
        ...

Why these fixes work
- math.copysign(1.0, x) < 0 reliably detects negative zero without false positives.
- Handling -0.0 before converting to int prevents losing the sign when int(-0.0) returns 0.

Notes
- Don’t rely on int(x) to preserve sign information for zeros.
- The explicit negative-zero check is small, obvious, and keeps the rest of your formatting logic unchanged.
