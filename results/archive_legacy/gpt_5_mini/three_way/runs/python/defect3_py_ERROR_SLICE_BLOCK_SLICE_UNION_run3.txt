What happened (where it failed)
- The function raised IndexError on these lines:
  - Line 35–36: the guard `if index < 0 or index >= len(code_units): raise IndexError(f"String index out of range: {index}")`.
- The error means `index` was 2 (per the message) but `len(code_units)` was ≤ 2, so the check detected an out-of-range access and raised the error.

Why it happened (root cause)
- The code uses `code_units`, a list of UTF-16 code units produced by `to_utf16_code_units(s)`.
- Whoever called the code passed an `index` value that is not a valid position inside `code_units`. Common ways this can happen:
  1. The caller computed `index` as a Python string (character) index instead of a UTF-16 code-unit index. In UTF-16, one Unicode character can produce two code units (a surrogate pair). That makes `len(code_units)` larger than `len(s)` in some cases, and conversely a character index can be out of range for `code_units`.
  2. The caller assumed the string was longer than it actually is (off-by-one or wrong loop bounds) and passed a too-large index.
  3. The conversion `to_utf16_code_units` returned fewer code units than the caller expected (e.g. empty string or truncated input).

Where in the code to inspect
- to_utf16_code_units: lines 12 (where `code_units` is built) and 35–41 (the index guard and subsequent access).
- The caller that passes `index` into this function (the code that supplies `index` is the immediate source of the wrong value).

     - Example: print(f"index={index}, len(code_units)={len(code_units)}, code_units={code_units[:5]}...")
   - This shows whether `index` is a character index vs a code-unit index and whether the list is shorter than expected.

     - Use count as the code-unit index.

3. If the caller simply computed the wrong numeric index (off-by-one or loop bug):

4. Harden the function if appropriate:

Summary

The fix is to ensure the index you pass refers to code-unit positions (not Python character indices) or to correct the caller’s index arithmetic so it stays < len(code_units).
