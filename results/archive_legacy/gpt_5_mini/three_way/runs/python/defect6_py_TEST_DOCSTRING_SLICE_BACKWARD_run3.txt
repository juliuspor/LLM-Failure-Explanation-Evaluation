What happened (where it fails)
- The failure happens inside CodeConsumer.add_number at the check shown on line 176:
  if x == int(x) and not math.isinf(x) and not math.isnan(x):
- For x == -0.0 that condition evaluates True because Python treats -0.0 == 0 as True. The code therefore takes the "integer" branch and prints the integer form (which becomes "0"), losing the negative sign.
- The test test_negative_zero expects "-0.0" but gets "0". So the bug is that the integer-detection branch runs before any check for negative zero, and nothing prevents -0.0 from being considered an integer.

Why it happens (root cause)
- The check uses x == int(x) to detect whole numbers. For -0.0, x == int(x) is True (since -0.0 == 0). That causes the formatter to print the integer representation.
- The code never checks for negative zero (or calls is_negative_zero) before deciding to print the integer form, so the sign-bit of negative zero is discarded.
- In short: wrong ordering/logic — the integer branch runs without excluding negative zero.

Concrete place to change
- File: hit06_codeconsumer.py
- Line: 176, the condition shown in the slice.

Exact fixes (clear steps)
1. Ensure negative-zero detection runs before the integer branch (or exclude negative zero from that branch).
2. Change the condition on line 176 so it does not treat -0.0 as an integer. Example replacement:

    # before (current)
    if x == int(x) and not math.isinf(x) and not math.isnan(x):

    # recommended: do not treat negative zero as integer
    if (x == int(x)
        and not math.isinf(x)
        and not math.isnan(x)
        and not self.is_negative_zero(x)):

   This keeps the integer branch but explicitly excludes negative zero.

3. Alternatively, detect negative zero first and handle it explicitly:

    if self.is_negative_zero(x):
        self.append("-0.0")
        return
    if x == int(x) and not math.isinf(x) and not math.isnan(x):
        ...

4. Implement/verify is_negative_zero if missing or incorrect. A correct implementation:

    @staticmethod
    def is_negative_zero(x):
        return x == 0.0 and math.copysign(1.0, x) < 0.0

   (math.copysign is the reliable way to check the sign bit for zeros.)

Why these fixes work
- They prevent the integer-formatting branch from swallowing the negative sign bit in -0.0.
- If you handle negative zero explicitly (or exclude it from integer detection), the code will print "-0.0" as the test expects, while still printing "0" for positive zero and integer values.

Summary
- Root cause: the condition on line 176 treats -0.0 as an integer because x == int(x) is True for -0.0, so the negative sign is lost.
- Fix: exclude negative zero from that branch (or handle negative zero first) — either add "and not self.is_negative_zero(x)" to the condition, or explicitly check self.is_negative_zero(x) and append "-0.0" before the integer check.
