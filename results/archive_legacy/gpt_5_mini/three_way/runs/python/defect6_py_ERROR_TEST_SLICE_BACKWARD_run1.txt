What happened (where and why)
- The failing test is test_negative_zero which calls CodeConsumer.add_number(-0.0).
- In add_number there is an integer-check at line 176:
  if x == int(x) and not math.isinf(x) and not math.isnan(x):
- For x = -0.0, the comparison x == int(x) evaluates True because -0.0 == 0.
- That branch treats the value as an integer and converts it with int(x) (or otherwise formats it as an integer), producing the string "0".
- Converting to int loses the sign bit of negative zero, so the output becomes "0" instead of the required "-0.0".
- The utility method is_negative_zero exists (used in tests) but add_number does not check it before the integer branch, so the special negative-zero case is never handled.

Why that causes the exact failure
- The test expects "-0.0". The code returns "0". That is the AssertionError "Expected -0.0, got 0".
- The root is the order of checks in add_number: treating -0.0 as an integer happens before any negative-zero detection, so the negative-zero sign is dropped.

How to fix (concrete steps)
1. Detect negative zero before the integer branch. Use the existing is_negative_zero() or an equivalent (math.copysign or bit-level check).
2. If is_negative_zero(x) is True, output "-0.0" immediately (or format to preserve sign and decimal).
3. Only if not negative zero continue with the existing integer check and other formatting.

Example patch (conceptual; adjust to your function names/formatting):
- At or just before line 176, insert:

        self.append("-0.0")
        return
- Keep the rest of the method unchanged after that.
