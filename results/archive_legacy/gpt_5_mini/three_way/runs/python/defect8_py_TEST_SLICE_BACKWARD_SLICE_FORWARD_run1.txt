What happened
- The failure happens inside LocaleUtils.to_locale around lines 231–247 (the branch that handles 5+ character locale strings).
- For input "fr__POSIX" the code does:
  - ch3 = locale_str[3] (line 237)
  - ch4 = locale_str[4] (line 238)
  - then it checks that ch3 and ch4 are uppercase letters (lines 239–240)
- For "fr__POSIX", locale_str[3] is '_' (indexing: 0='f',1='r',2='_',3='_',4='P'...), so the uppercase-letter check fails and the method raises ValueError("Invalid locale format: fr__POSIX").

Why it fails (root cause)
- The implementation assumes that when length >= 5 the form must be language + '_' + two-letter country (i.e. positions 3 and 4 are both uppercase letters). It never considers the valid format language__variant (empty country, double underscore).
- In other words, the code never checks for the special case where locale_str[2] == '_' and locale_str[3] == '_' (empty country). Because of that, the code treats the second '_' as an invalid country letter and raises an exception (lines 237–240).

Where to change (exact places referenced)
- Modify to_locale in the same area shown: between line 231 (start of longer-string handling) and the checks at lines 237–240.
- Add a branch that detects a double underscore at positions 2 and 3 and returns Locale(language, "", variant) (variant begins at index 4).

Concrete fix (exact logic to add)
- Before reading ch3/ch4, add:

  if length >= 4 and locale_str[2] == '_' and locale_str[3] == '_':
      # language__variant => language = locale_str[0:2], country = "", variant = locale_str[4:]
      return Locale(locale_str[0:2], "", locale_str[4:])

Notes and edge cases
- Use length >= 4 because "fr__" (no variant) would have length 4; decide whether "fr__" should be valid — current tests expect variants after the double underscore, so if empty variant should be rejected, you can check length > 4 and raise if not.
- Keep the existing uppercase-country checks for the normal language_country and language_country_variant forms.
- This change directly prevents the ValueError raised at lines 237–240 for inputs like "fr__POSIX" and allows correct creation of Locale("fr", "", "POSIX").
