What happened (where the exception was raised)
- The IndexError is raised at line 35 in to_utf16_code_units when this check fails:
  if index < 0 or index >= len(code_units):
    raise IndexError(f"String index out of range: {index}")
- The exception name and message show index == 2 and len(code_units) <= 2, so index >= len(code_units) triggered the error.

Why it failed (root cause)
- The code is validating an index against len(code_units), but code_units is the list of UTF-16 code units derived from the input string s. The index that reaches this check is a string/character index (or otherwise a different index space) — not necessarily an index into the code_units list.
- That mismatch between index spaces is the direct cause: the caller passed an index counted in characters (or in a different sequence), while the function checks it against the length of code_units (which can be different because characters outside the BMP become two UTF‑16 code units). As a result, index can be equal to or larger than len(code_units) even when it looks valid for the original string, and you get "String index out of range: 2".
- Put simply: the code checks the wrong length against the provided index.

Where exactly to look in the code
- to_utf16_code_units:
  - code_units is created at line 12: code_units = []
  - the index bounds check at line 35 uses len(code_units)
  - subsequent logic at lines 38–44 reads code_units[index] and code_units[index + 1]

Concrete fixes (choose one based on intended behavior)
1) If the function expects a string-character index (index into s)
   - Validate against len(s), not len(code_units). Change line 35 to:
       if index < 0 or index >= len(s):
           raise IndexError(f"String index out of range: {index}")
   - Then convert the character index to the corresponding code-unit index before accessing code_units (you must build code_units first and map character positions to code-unit positions).
   - Steps:
     a. Build code_units from s (as the function name implies).
     b. Create a mapping from character index -> code_unit index (advance by 1 for BMP chars, by 2 for surrogate pairs).
     c. Use that mapped index to read code_units[index_in_code_units].

2) If the function expects a code-unit index (index into code_units)
   - Ensure the caller passes a code-unit index (not a character index). Document that the index is into code_units.
   - Or build code_units before validating and ensure index is computed in the same unit-space.
   - Change comments/docstring to clearly state "index is a UTF‑16 code-unit index".

3) Quick defensive fix (if uncertain which index space should be used)
   - Add clearer validation and an explanatory error message. For example:
       if index < 0:
           raise IndexError(...)
       if index >= len(code_units):
           raise IndexError(f"Index {index} out of range for {len(code_units)} UTF-16 code units; did you mean a string character index?")
   - This doesn't fix the logic mismatch, but makes the cause obvious in future failures.

Which you should pick
- If the function name to_utf16_code_units suggests it accepts a Python string and returns code units, it should not be indexing into code_units itself. Likely the index passed is a character index and validation should use len(s) or map the character index to a code-unit index. That is the most probable intended behavior.

Summary
- The error comes from validating the wrong length (len(code_units)) against an index that is in a different coordinate space (character index). Fix by either validating against len(s) and mapping indexes to code units, or require caller to pass a code-unit index and document that requirement.
