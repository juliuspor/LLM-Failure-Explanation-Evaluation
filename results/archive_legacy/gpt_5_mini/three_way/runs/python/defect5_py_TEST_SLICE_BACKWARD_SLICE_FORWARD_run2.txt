Why this is wrong (root causes)
1. Wrong source for array element type:
   - When array is not None, the code uses type(array) (line 335). For a Python list this returns list (the container type), not the element type. The intention is to infer the array element type, but type(array) is the wrong value to use.
2. Incorrect handling of both-None case:
   - When both array and element are None the code unconditionally sets inferred_type = object (line 340) and then rejects any caller-provided expected_type != object by raising TypeError (lines 346–347). But the caller passed expected_type=str and expected that a String-typed result (in Java terms) or at least a list containing None would be created. Raising here is unjustified: there is no need to error when both inputs are None but an expected_type is provided.
3. Overstrict equality test against object:
   - The check inferred_type == object is brittle and mixes container vs element type semantics. It causes the function to fail for a valid call pattern (both None + expected_type specified).

Exactly where to change (concrete steps)
1. Infer element type from the array contents, not from the container:
   - Replace inferred_type = type(array) (line 335) with logic like:
     - if array is not None and len(array) > 0: inferred_type = type(array[0])
     - elif array is not None and len(array) == 0: inferred_type = None (or use expected_type if provided)
2. When element is non-None use its type (line 337) — that part is OK.
3. When both array and element are None:
   - Do not set inferred_type = object and then raise when expected_type is provided.
   - Instead set inferred_type = expected_type if expected_type is given; otherwise set inferred_type to object (or None) but do not raise.
