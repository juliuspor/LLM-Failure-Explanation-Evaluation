What happened
- For inputs like "L" or "l" the code raises IndexError instead of ValueError.
- Why: when val ends with "l" or "L" you set numeric = val[:-1]. For "L" that makes numeric == "" (empty string).
- You then call char_at(numeric, 0) inside the condition:
  if (char_at(numeric, 0) == "-" and numeric[1:].isdigit()) or numeric.isdigit():
  That calls char_at with index 0 on an empty string. char_at checks bounds and raises IndexError (see lines 30–32), so the function crashes there.

Where it occurs (exact place)
- In the branch that handles last_char in ("l", "L") (the if at about line 17–25 of the snippet).
- The IndexError originates in char_at at the code shown in [SLICE_BACKWARD]/[SLICE_FORWARD] (lines 30–32).

Root cause
- Missing guard for the case len(val) == 1 (so numeric == "") before calling char_at or indexing numeric[1:].
- The logic calls char_at unconditionally, so an empty numeric causes an out-of-range access.

Concrete fixes (pick one)

1) Add a length check before calling char_at
- Replace the l/L branch with:
    numeric = val[:-1]
    if len(numeric) == 0:
        raise ValueError(f"{val} is not a valid number.")
    if (char_at(numeric, 0) == "-" and numeric[1:].isdigit()) or numeric.isdigit():
        return int(numeric)
    raise ValueError(f"{val} is not a valid number.")

2) Avoid char_at and use safe string methods (recommended)
- Use startswith and isdigit which are safe on empty strings:
    numeric = val[:-1]
    if (numeric.startswith("-") and numeric[1:].isdigit()) or numeric.isdigit():
        return int(numeric)
    raise ValueError(f"{val} is not a valid number.")
This covers the empty numeric case because numeric.isdigit() will be False for "" and numeric.startswith("-") will be False for "".

3) Explicitly handle the single-character case at top
- Add a guard after len(val) == 0 check:
    if len(val) == 1 and not val.isdigit():
        raise ValueError(f"{val} is not a valid number.")
This prevents entering the l/L branch for single-character non-digits.

Which to use
- Option 2 is simplest and robust: it removes the unsafe char_at access and expresses the intent directly.
- Option 1 is minimal if you must keep char_at.

Example failing input
- val = "L" → numeric == "" → char_at(numeric, 0) raises IndexError.
- After the fix, val = "L" will raise ValueError("L is not a valid number.") as intended.
