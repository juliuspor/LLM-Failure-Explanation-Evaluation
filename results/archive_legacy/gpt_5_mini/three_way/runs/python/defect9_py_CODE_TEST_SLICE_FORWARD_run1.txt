What happened (root cause)
- The bug is a null-safety error in the loop that estimates the output buffer growth (the one marked with "BUG: this loop is not null-safe for elements inside the arrays").
- On line 83 you do: greater = len(replacement_list[i]) - len(search_list[i]) without checking whether replacement_list[i] or search_list[i] is None.
- If either element is None (exactly the case in your tests: replacement_list contains None), len(None) raises TypeError. That exception occurs before the rest of the method runs, so the tests fail.

Where it occurs
- The failing code is in the "increase" calculation loop (slice shows lines 83–85):
  - line 83: greater = len(replacement_list[i]) - len(search_list[i])
  - line 84: if greater > 0:
  - line 85:     increase += 3 * greater
- The earlier match-safety checks (inside the main replacement loops at lines ~102–108) do check for None, but this pre-sizing loop does not, so the code crashes there.

Why the tests trigger it
- test_null_safety_inside_arrays_single calls replace_each("aba", ["a"], [None]).
- That passes a None in replacement_list; the pre-sizing loop tries len(None) and raises TypeError.

Concrete, actionable fixes (step-by-step)
1. Guard the pre-sizing loop the same way you guard the other loops. Replace the body of the increase loop with a null/empty check before calling len():
   - if search_list[i] is None or len(search_list[i]) == 0 or replacement_list[i] is None:
       continue
   - else compute greater = len(replacement_list[i]) - len(search_list[i]) and update increase.
2. Alternatively, use safe length expressions but be careful to treat empty-string search entries as excluded:
   - search_len = len(search_list[i]) if search_list[i] is not None else 0
   - repl_len = len(replacement_list[i]) if replacement_list[i] is not None else 0
   - if search_len == 0 or repl_len == 0: continue
   - greater = repl_len - search_len
3. Run the unit tests; the TypeError should disappear and the null-safety tests should pass.

Minimal patch example
Replace the increase loop (around lines 79–85) with:

for i in range(search_length):
    if search_list[i] is None or len(search_list[i]) == 0 or replacement_list[i] is None:
        continue
    greater = len(replacement_list[i]) - len(search_list[i])
    if greater > 0:
        increase += 3 * greater

Why this fix is correct
- It mirrors the existing null/empty checks used later when finding matches (lines ~102–108). That prevents calling len(...) on None and keeps behavior consistent: entries with None replacement or empty/null search strings are ignored for both sizing and replacement logic.
