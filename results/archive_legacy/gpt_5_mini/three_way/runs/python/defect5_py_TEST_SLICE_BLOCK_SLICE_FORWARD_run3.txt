Why this check is wrong (root cause)
- The check treats "inferred_type == object" as an error whenever an expected element type is supplied. But inferred_type==object here simply means "no concrete type could be inferred" (both array and element are None). It is valid for the method to create a list of the requested expected_type (a String[] in the original Java example) or at least to return a Python list containing None and not raise.
- In short: the code confuses "unknown/inferred object type" with a genuine type mismatch and aborts instead of honoring the caller's expected_type when inference failed.
