What happened (where it fails)
- At line 71 you compute g using the original parameter value:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- But earlier you clamped the value into variable v:
  v = max(value, self._lower_bound)
  v = min(v, self._upper_bound)
- Because g uses value instead of v, if value is outside the [self._lower_bound, self._upper_bound] range g can fall outside 0..255 and trigger the ValueError at line 74.
- Also, if self._upper_bound == self._lower_bound the division will raise ZeroDivisionError before the range check happens.

Why it happened (root cause)
- A simple variable mixup: you computed the clamped value into v but then ignored v and used the original value when computing g. That lets out-of-range inputs produce out-of-range g.
- There is no guard against an equal lower/upper bound, so the denominator can be zero.
