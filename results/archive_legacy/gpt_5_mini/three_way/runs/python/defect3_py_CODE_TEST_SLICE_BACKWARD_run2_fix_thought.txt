The bug arises because after a translator returns consumed>0 the loop advanced pos by repeatedly calling code_point_at, which can call code_point_at with pos==len(code_units) and raise IndexError. The translators in this codebase return the number of UTF-16 code units consumed (e.g., surrogate pair -> 2). The simplest robust fix is to advance pos by consumed code units directly. I'll replace the inner advancement logic with pos += consumed. This avoids out-of-range calls and matches the translators' behavior.
