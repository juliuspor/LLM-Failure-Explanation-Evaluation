What happened (root cause)
- The error occurs in the "estimate result buffer" loop at lines 79–85 (shown in the slice as the loop with comment "BUG: this loop is not null-safe for elements inside the arrays").
- That loop does:
  greater = len(replacement_list[i]) - len(search_list[i])
  and raises TypeError: object of type 'NoneType' has no len() when either replacement_list[i] or search_list[i] is None.
- The code does check for None/empty/skip conditions elsewhere (in the first match-search loop and inside the main while loop), but it does NOT guard the short "increase" loop. So if any element in search_list or replacement_list is None, calling len(...) on it will crash before the function reaches the later loops that would skip those entries.

Where it happens (specific variables/lines)
- Variable(s): search_list[i] or replacement_list[i] is None.
- Line(s): the loop just above "increase = min(increase, len(text) // 5)" — the calculation using len(replacement_list[i]) and len(search_list[i]).
- The failure happens before the first real replacement search completes because that estimation loop runs earlier and assumes non-None elements.

Why the assumption fails
- The code does allow None entries in search_list and replacement_list (it checks for them later), so the early size-estimate loop must also account for them. It currently does not, causing a TypeError whenever a None element appears in either list.

Concrete fixes (step-by-step)
1. Skip None or empty search strings and None replacements in the increase loop, just like you do in the match-finding loops.
2. Replace the buggy loop with a null-safe version.

Example replacement for the increase loop:

    # old buggy loop:
    # for i in range(search_length):
    #     greater = len(replacement_list[i]) - len(search_list[i])
    #     if greater > 0:
    #         increase += 3 * greater  # assume 3 matches

    # null-safe loop:
    for i in range(search_length):
        if (
            search_list[i] is None
            or replacement_list[i] is None
            or len(search_list[i]) == 0
        ):
            continue
        greater = len(replacement_list[i]) - len(search_list[i])
        if greater > 0:
            increase += 3 * greater  # assume 3 matches

3. (Optional) If you also want to mirror the boolean no_more_matches_for_repl_index skip (not necessary for correctness of the estimation), you can include that condition too:
        if no_more_matches_for_repl_index[i]:
            continue

Why this fix is safe
- It uses exactly the same null/empty checks you already use later in the function, so the estimate loop no longer tries to compute len(...) on None and won't crash.
- The rest of the function still performs the real filtering and replacement logic unchanged.

Other notes
