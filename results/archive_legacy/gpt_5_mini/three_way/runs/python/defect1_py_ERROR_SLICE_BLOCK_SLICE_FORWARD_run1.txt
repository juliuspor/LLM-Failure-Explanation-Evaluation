What happened (where it failed)
- The ValueError "Minutes out of range: -15" was raised at the check on lines 396–397:
  - 396: if minutes_offset < 0 or minutes_offset > 59:
  - 397:     raise ValueError(f"Minutes out of range: {minutes_offset}")
- The offending value was minutes_offset == -15.

Why it happened (root cause)
- minutes_offset is calculated earlier, but the code validates it too early — before the sign/normalization logic on lines 402 and 404 runs.
- The code later expects to adjust a negative minutes_offset (see lines 402 and 404):
  - 402: minutes_offset = hours_in_minutes - minutes_offset
  - 404: minutes_offset = hours_in_minutes + minutes_offset
  Those lines will convert a negative remainder (like -15) into the intended positive minutes component (for example converting -15 to 45 minutes).
- Because the negative minutes_offset is checked on lines 396–397 before those adjustments, a perfectly fixable negative remainder (−15) triggers the "out of range" error instead of being normalized.

Exactly where to change it
- Move the validation (lines 396–397) to after the normalization/adjustment code (after line 404). That way you validate the final, normalized minutes value.
  OR
- Change the validation to accept the intermediate negative remainder and only reject values whose absolute magnitude is invalid:
  - Replace the check with something like: if minutes_offset < -59 or minutes_offset > 59: raise ValueError(...)
  then let the existing normalization logic turn a negative remainder into the correct positive minutes before final use.

Concrete fixes (pick one)
1) Move the check
   - Remove lines 396–397 from their current place.
   - Insert them after line 404 (after the minutes_offset adjustments), so you validate the normalized minutes.

2) Broaden the check and keep normalization where it is
   - Change line 396 to:
     if minutes_offset < -59 or minutes_offset > 59:
         raise ValueError(f"Minutes out of range: {minutes_offset}")
   - This prevents rejecting valid negative remainders like -15 that are meant to be normalized.

Tests to add
- Test input that produced the failure (the case that generated minutes_offset == -15).
- Cases around the boundaries: minutes_offset == -59, -1, 0, 1, 59, and a clearly invalid value like -60 or 60.

Why these fixes work
- The normalization code on lines 402/404 expects to receive possibly-negative remainders and convert them. Either validating after normalization or allowing negative remainders in the validation prevents premature rejection of valid inputs and avoids the ValueError shown.
