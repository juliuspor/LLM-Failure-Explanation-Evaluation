What happened (summary)
- The code raises IndexError for inputs where lower > len(text) (e.g. text="0123456789", lower=15, upper=20) because the line "if upper < lower: upper = lower" can increase upper to a value greater than the string length after you already forced upper to text length earlier. That larger upper is later passed to substring_java(..., 0, upper), which then fails when it tries to create a slice/substring beyond the string bounds.

Exactly where it occurs (lines / variables)
1. You set upper to text length here:
   if upper == -1 or upper > len(text):
       upper = len(text)
   (upper becomes 10 for "0123456789".)

2. Immediately after you do this:
   if upper < lower:
       upper = lower
   Because lower = 15, this sets upper = 15. Now upper > len(text).

3. Later you call substring_java(text, 0, upper) (in the index == -1 branch), with upper == 15. substring_java is not prepared to accept an end index larger than text length and raises IndexError.

Why that logic is wrong
- The code enforces upper = len(text) then undoes that by setting upper = lower when lower is larger than len(text). The code never brings upper back into range before calling substring_java. No check for lower > len(text) is made, so lower being beyond the string length propagates into upper and causes the out-of-range substring call.

Concrete ways to fix it (clear steps)
Choose one of these explicit fixes:

Option A — Handle lower > len(text) early (closest to original Java behavior):
- Add at the top, right after you check for text == None / empty:
    if lower > len(text):
        return text
  This matches the intended behavior for the test case (abbreviate("0123456789", 15, 20, None) should return the original string).

Option B — Ensure upper is limited after you possibly set it to lower:
- Replace the existing two lines:
    if upper == -1 or upper > len(text):
        upper = len(text)
    if upper < lower:
        upper = lower
  with:
    if upper == -1 or upper > len(text):
        upper = len(text)
    if upper < lower:
        upper = lower
    if upper > len(text):
        upper = len(text)
  (or more simply set upper = min(upper, len(text)) after those adjustments)

Option C — Clamp lower before using it:
- At the start (after empty checks) do:
    lower = min(lower, len(text))
  That prevents lower from ever exceeding len(text), so upper = lower won’t create an out-of-range upper.

Which to prefer
- Option A is the simplest and directly matches the Java/expected behavior described by the test (if lower is beyond the end, return the original text).
- Option B and C are also safe; Option B keeps the current flow but ensures upper never exceeds len(text). Option C is semantically different (it treats lower as clipped) but avoids the error.

Where to change in your code (reference lines)
- Change right after these lines:
    if len(text) == 0:
        return StringUtils.EMPTY
  Insert Option A there, or follow Option B/C by changing the block that starts:
    if upper == -1 or upper > len(text):
