What happened
- The input locale_str "fr__POSIX" triggers the ValueError at line 239: the code reads
  ch3 = locale_str[3]
  ch4 = locale_str[4]
  if ch3 < 'A' or ch3 > 'Z' or ch4 < 'A' or ch4 > 'Z':
      raise ValueError(f"Invalid locale format: {locale_str}")

- For "fr__POSIX", locale_str[3] is '_' (because the string has two underscores for an empty country), so ch3 < 'A' is true and the method raises ValueError("Invalid locale format: fr__POSIX").

Why this is wrong (root cause)
- The code assumes that if there are characters after the language and an underscore, the country code must be two uppercase letters at positions 3 and 4. It does not account for the valid Java-style form where the country is empty and a variant follows, which uses a double underscore: language + "_" + "" + "_" + variant (e.g. "fr__POSIX").
- Specifically, the method never handles the "empty country, non-empty variant" case; it treats the '_' at index 3 as invalid instead of recognizing it as the marker of an empty country and a variant starting at index 4.

Where it occurs (lines / variables)
- In to_locale(cls, locale_str) (slice shown).
- The failing check is at lines 237–239: it uses ch3 = locale_str[3], ch4 = locale_str[4] and then strictly requires ch3 and ch4 to be 'A'..'Z'.
- The raise is executed at line 239, producing ValueError(f"Invalid locale format: {locale_str}") with the exact message you saw.

How to fix it (clear steps)
1. Detect the double-underscore case (empty country, variant present) before or as part of the country validation.
2. If locale_str[3] == '_' then treat country = "" and variant = locale_str[4:], returning Locale(locale_str[0:2], "", locale_str[4:]).
3. Otherwise, keep the existing validation that ch3 and ch4 are uppercase letters and return the two-letter country as before.
