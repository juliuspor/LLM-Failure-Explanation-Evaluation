What happened (what you see)
- The test call WordUtils.abbreviate("0123456789", 15, 20, None) raises IndexError.
- That IndexError happens inside the call to StringUtils.index_of(text, " ", lower) because lower (15) is greater than the length of text (10).

Where it occurs (specific line/variable)
- The failure occurs at the line:
  index = StringUtils.index_of(text, " ", lower)
  (the first place the code uses lower for anything other than comparisons).
- At that point lower can be > len(text) because the function never reduces or guards lower. Only upper is adjusted earlier:
  if upper == -1 or upper > len(text):
      upper = len(text)
  if upper < lower:
      upper = lower
- There is no equivalent adjustment for lower. Passing start > len(text) into StringUtils.index_of triggers the IndexError inside index_of (the underlying implementation likely attempts to index into the string starting at that position).

Why it happens (root cause)
- The code assumes it is safe to call index_of(text, " ", lower) for any lower, but many index implementations (or helpers wrapped by StringUtils.index_of) do not accept a start index beyond the string length — they raise instead of returning "not found".
- Because lower is not clamped or handled when it is larger than len(text), the call uses an invalid start and crashes.

How to fix it (concrete steps)
You have two clear and equivalent fixes. Apply one of these before calling index_of:

1) Early-return when lower >= len(text)
- Add this check after handling null/empty and upper adjustments:
  if lower >= len(text):
      # nothing to abbreviate, return the whole text unchanged
      return text
- This mirrors the Java behaviour expected by the test: with lower beyond length, return the original whole string.

2) Clamp lower to len(text) before calling index_of
- Change lower to min(lower, len(text)) before index_of:
  if lower > len(text):
      lower = len(text)
- This ensures index_of is called with a valid start; index_of will then behave as if there is no space after that point.

Recommended minimal patch (replace into your code)
Insert right before the index_of call (or right after the upper adjustments):
    if lower >= len(text):
        return text

Why that is simplest
- It directly implements the intended semantics (if lower is beyond the string, return the string unchanged).
- It avoids relying on StringUtils.index_of to tolerate out-of-range start values and fixes the specific case demonstrated by the failing test.

Summary
- Root cause: calling StringUtils.index_of(text, " ", lower) with lower > len(text) leads to an IndexError because lower is never clamped or handled.
- Fix: check for lower >= len(text) and return text (or clamp lower to len(text)) before calling index_of.
