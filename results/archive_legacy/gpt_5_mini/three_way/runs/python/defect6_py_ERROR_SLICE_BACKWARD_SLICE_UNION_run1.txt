What happened (where it failed)
- At line 176 the code tests if x == int(x). For x == -0.0 that comparison is True because in Python -0.0 == 0 evaluates to True.
- On line 177 you then do value = int(x). int(-0.0) becomes 0 and loses the negative-zero sign bit.
- Later (lines 190/192/194) you output value or use the integer-derived mantissa/value path, which produces "0" (or "0E..." etc.) instead of "-0.0".
- The test expected "-0.0" but the code returned "0" because the negative sign was lost as soon as you used int(x).

Why this happens (root cause)
- Floating-point negative zero compares equal to positive zero, so the equality test x == int(x) is not sufficient to detect and preserve the special -0.0 value.
- Converting -0.0 to int removes the sign bit; there is no further check to restore or preserve sign, so the output becomes positive zero.

   This uses math.copysign to detect the negative sign on zero and outputs the original "-0.0" string immediately.

Pick one of these and add it just before the integer-handling block (line 176) so you never lose the negative-zero sign.

Summary
- Root cause: the equality test and int() conversion collapse -0.0 to 0, losing sign.
- Fix: detect negative zero (math.copysign or bit test) before converting and handle/output it directly.
