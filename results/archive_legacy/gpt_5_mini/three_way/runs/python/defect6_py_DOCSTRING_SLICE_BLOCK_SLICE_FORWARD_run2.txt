What fails
- You get an UnboundLocalError (or the equivalent “referenced before assignment”) when the code reaches line 182:
  check_val = (mantissa // 10) * (10 ** (exp + 1))
  The variable exp is used there but has not been initialized on that execution path.

Why it happens (exact location and cause)
- On line 182 the code uses exp in the power expression (exp + 1).
- exp is assigned at line 179: exp = 0.
- But the execution path that hits the while at line 181 can occur when exp has never been set, because of the current indentation/control-flow. In other words, exp = 0 is not guaranteed to run before the while uses exp.
- Concretely: if the code reaches the abs(x) >= 100 branch and then enters the while mantissa % 10 == 0 loop, there are cases where exp was not defined in that activation (the initialization at line 179 is skipped on that path), so the use of exp at line 182 fails.

How to fix it (concrete steps)
1. Ensure exp is initialized before any use. Move or add the initialization so it executes before the abs(x) >= 100 branch and before the while. For example:
   - Place exp = 0 immediately after mantissa = value (line 178), so exp is always defined:
     177 |     value = int(x)
     178 |     mantissa = value
     179 |     exp = 0
     180 |     if abs(x) >= 100:
     181 |         while mantissa % 10 == 0:
     182 |             check_val = (mantissa // 10) * (10 ** (exp + 1))
     ...
2. Verify indentation so exp = 0 is inside the same outer block where the while can run (i.e., not accidentally moved under a different conditional).
3. Optionally add an assert or explicit default to make the intent clear:
   - assert isinstance(exp, int) or explicitly set exp = exp if 'exp' in locals() else 0 before the loop (but simply initializing exp = 0 as in step 1 is preferable).

Why this fix works
- Initializing exp before the while guarantees (exp + 1) is valid and avoids the UnboundLocalError. It also preserves the intended behavior of accumulating trailing-zero reductions into mantissa and exp.
