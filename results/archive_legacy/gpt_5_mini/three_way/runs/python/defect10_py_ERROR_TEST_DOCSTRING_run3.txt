What happened
- The failing code is NumberUtils.create_number(s).
- For inputs "l" or "L" the code treats the last character as a long qualifier and removes it (e.g. s = s[:-1]).
- After that removal the code immediately does an indexed access of the (now-empty) string, e.g. checking s[0] or doing s[0] == '.'.
- Because s is "", attempting s[0] raises IndexError: "String index out of range: 0" instead of the intended ValueError/NumberFormatException.

Why it fails (root cause)
- The code assumes that after stripping a trailing qualifier there is still at least one character left to inspect. That assumption is false when the original input was a single-character qualifier ("l" or "L").
- No guard exists to detect the empty-string case before executing s[0] (or similar indexed access), so Python raises IndexError.

Where it occurs (what to change)
- In NumberUtils.create_number, immediately after you strip a trailing qualifier (the line like s = s[:-1] or mant = mant[:-1]), add a check for empty string before any further character indexing or parsing.
  Example check to add:
    if s == "":
        raise ValueError("Invalid number: missing digits before/after qualifier")

Concrete fixes (pick one)
1) Raise ValueError when qualifier-only input is detected
   - After removing the qualifier: if s == "" (or len(s) == 0), raise ValueError("Invalid number: no digits")
   - This preserves the intended behavior that "l"/"L" are invalid numeric inputs.

2) Validate earlier and avoid stripping a lone qualifier
   - Before stripping, verify the original string has length > 1 when its last char is a qualifier:
       if len(s) == 1 and s[-1] in ("l", "L", "f", "F"):
           raise ValueError("Invalid number: missing digits")
       if s[-1] in ("l","L","f","F"):
           s = s[:-1]
   - This prevents ever creating an empty string and stops further indexing errors.

3) Defensive access
   - After stripping, replace any direct s[0] uses with checks like:
       if not s or s[0] not in allowed_chars: ...
   - But prefer (1) or (2) because they directly encode the intended error for bad inputs.

Why this fix is correct
- It stops the code from indexing into an empty string (which causes IndexError).
- It returns the expected ValueError for bad inputs like "l"/"L", matching the test's intent and Java equivalent behavior.

Minimal example patch (conceptual)
- Current fragment (problematic):
    if s[-1] in ("l","L"):
        s = s[:-1]
    if s[0] == '.':   # <-- IndexError when s == ""
        ...
- Fixed:
    if s[-1] in ("l","L"):
        if len(s) == 1:
            raise ValueError("Invalid number: no digits before qualifier")
        s = s[:-1]
    if s[0] == '.':
        ...

This exact check is all that's needed to make the test for ["l","L","f","F","junk","bobL"] raise ValueError instead of causing IndexError.
