What failed
- The code raises ValueError("bound must be positive") at line 52: "if gap <= 0: raise ValueError('bound must be positive')".
- That happens when gap = end - start is 0 or negative (i.e. end <= start). Example: calling random(5, 10, 10, False, False) produces gap == 0 and triggers the exception at line 52.

Why it fails (root cause)
- The function never validates that the caller supplied a valid half-open character range before using it. The comment at line 50 even notes the omission: "BUG: missing validation when end <= start".
- The code computes gap = end - start (line 49). When end <= start, gap <= 0 and _RANDOM.randrange(gap) (used later) would be invalid. The code detects that only at line 52 and raises a generic "bound must be positive" error that does not reference the offending parameters.
- So the real problem is either:
  - callers can pass end <= start (invalid inputs), and the function does not guard/handle that early enough or with a helpful message, or
  - the function should accept equal start/end or automatically normalize an inverted range, but it currently does neither.

Where it occurs (exact lines)
- Function header: line 35 defines random(...)
- Range adjustment for default zero values: lines 42–47 set start/end when both are zero
- gap computed: line 49 ("gap = end - start")
- Exception thrown: line 52 ("if gap <= 0: raise ValueError('bound must be positive')")

How to fix (clear, actionable steps)
Choose one of these fixes depending on intended behavior:

1) Reject invalid ranges with a clear message (recommended if end is meant to be strictly greater than start)
   - Replace the line 52 check with an explicit validation that includes the offending values:
     - if end <= start: raise ValueError(f"end ({end}) must be greater than start ({start})")
   - This makes the cause obvious to callers and prevents the vague "bound must be positive" message.

2) Accept a single-character range when start == end (if you want to allow that)
   - If you want start==end to produce that single character repeatedly, change the code to handle gap == 0 before calling randrange:
     - if gap == 0:
         ch = chr(start)
         while len(buffer) < count:
             if (letters and ch.isalpha()) or (numbers and ch.isdigit()) or (not letters and not numbers):

3) Normalize swapped arguments (if you want to be permissive and handle start > end)
   - Swap start and end if start > end:
     - if end <= start:
         start, end = min(start, end), max(start, end)
         gap = end - start
     - Then proceed; but still keep the check for gap == 0 if you don’t accept single-value ranges.

Notes about messages and safety
- Use a clear error message that includes start and end (see step 1). The current "bound must be positive" message is copied from Random#nextInt and hides which parameters were bad.
- Do the validation immediately after computing gap (right where line 49–52 are), so you fail fast and with a clear message before any use of _RANDOM.randrange(gap).
