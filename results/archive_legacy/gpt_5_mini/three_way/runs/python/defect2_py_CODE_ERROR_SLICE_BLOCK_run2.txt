What happened (where it failed)
- In get_paint the code clamps the input to v on lines 2–4:
  ```
  v = max(value, self._lower_bound)
  v = min(v, self._upper_bound)
  ```
  but when computing g the code still uses value instead of v:
  ```
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
  ```
- Because value was outside the configured bounds, g computed from value can be < 0 or > 255. That triggers the ValueError on lines 74–75:
  ```
  if g < 0 or g > 255:
      raise ValueError(...)
  ```

Why this caused the test to fail
- The intended behavior is to use the clamped input (v) when mapping to 0–255. Using the original value ignores the clamp, so an out-of-range input produces an out-of-range g and the ValueError is raised.

How to fix it (concrete steps and exact code)
1. Use the clamped variable v when computing g.
2. Prevent division by zero if upper and lower bounds are equal (handle that case explicitly).

Replace the g calculation with something like:

```
# use the clamped value v
denom = self._upper_bound - self._lower_bound
if denom == 0:
    # bounds equal: pick a sensible default (e.g. middle gray) or raise a clear error
    g = 128
else:
    g = int((v - self._lower_bound) / denom * 255.0)
```

Or, if you prefer to map an equal-range to 0 or 255, replace g = 128 with g = 0 or g = 255 and document the choice.

Summary
- Root cause: computed g uses value instead of the clamped v (and no check for zero denominator).
- Location: calculation line (uses value) and the ValueError check at lines 74–75.
- Fix: compute g from v and handle upper==lower.
