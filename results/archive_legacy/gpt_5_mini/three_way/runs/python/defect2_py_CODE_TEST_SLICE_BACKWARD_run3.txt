What happened (root cause)
- In get_paint you clamp the incoming value into v (lines 64–66), but then you ignore v and compute g using the original value variable (line 71).
- Because you use value instead of v, an out-of-range input like -0.5 produces a negative g. That triggers the ValueError check (line 74–75) instead of returning the clamped black or white color.

Where it occurs (exact lines / variables)
- Clamping is done into v at lines ~64–66:
  v = max(value, self._lower_bound)
  v = min(v, self._upper_bound)
- The grayscale index g is computed from value at line ~71:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- The exception is raised at lines ~74–75 when g < 0 or g > 255:
  if g < 0 or g > 255:
      raise ValueError(...)

Why the tests fail
- test_lower_bound_clamping calls get_paint(-0.5). You clamped to v = self._lower_bound (0.0), but used value = -0.5 in the calculation. That yields a negative g and raises the ValueError instead of returning (0,0,0). The same happens for values > upper bound.

Exact fix (step-by-step)
1. Use the clamped variable v when computing g (replace value with v):
   g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
2. Also guard against division by zero if upper and lower bounds are equal. For example:
   if self._upper_bound == self._lower_bound:
       g = 0  # or 255 or 127 depending on desired behavior; commonly return 0 or raise
   else:
       g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
3. Optionally clamp g to [0,255] before returning, instead of raising, if you prefer tolerant behavior:
   g = max(0, min(255, g))
