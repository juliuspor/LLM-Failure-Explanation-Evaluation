What happened (where it fails)
- The IndexError is raised at the bounds check in the slice block: lines 35–36
  if index < 0 or index >= len(code_units):
      raise IndexError(f"String index out of range: {index}")
- The failing value is index == len(code_units) (for the test input "\U0001F630A" this becomes a UTF‑16 code_units sequence of length 2). The code later tries to access code_units[index] when index equals the length, so the bounds check triggers.

Why it happens (root cause)
- The translator iterates over UTF‑16 code units (variable code_units) using an index variable.
- When it encounters a surrogate pair (a supplementary character like the emoji \U0001F630), the code_point_at function reads two UTF‑16 code units (high and low surrogate) and returns the code point.
- After processing that surrogate pair, the loop advances index by 1 instead of by 2. That leaves index pointing one past the end of the array (index == len(code_units)).
- On the next loop iteration the code tries to call code_point_at or otherwise access code_units[index], and the bounds check at lines 35–36 raises IndexError because index is out of range.

Where in the code the bug is introduced
- In the translate loop inside CsvEscaper.translate_string (or whichever routine iterates code_units): the index increment is wrong after processing a surrogate pair.
- The check that raised the error is at lines 35–36, but the real mistake is the incorrect index update after handling a supplementary code point.

How to fix it (concrete steps)
1. When you detect a surrogate pair (i.e., code_point_at consumed a high + low surrogate and returned a code point > 0xFFFF), advance the index by 2, not 1.
   - Example logic in the main loop:
     - cp = code_point_at(code_units, index)
     - consumed = 2 if cp > 0xFFFF else 1
     - index += consumed
2. Alternatively, change code_point_at to return both the code point and the number of code units consumed (tuple (cp, consumed)), so the caller can always advance the index correctly.
3. Make sure code_point_at itself checks index+1 < len(code_units) before accessing the low surrogate and handles an unmatched high surrogate gracefully (treat as a single code unit or handle as an error).
4. Add unit tests for:
   - surrogate pair followed by another character (the failing test you already have)
   - lone high surrogate at end of string (should not cause out-of-range access)

Why these steps solve it
- Advancing by the correct number of UTF‑16 code units prevents index from moving one past the array and avoids the bounds check failure at lines 35–36.
- Returning the consumed count from code_point_at makes the contract explicit and prevents repeated off‑by‑one errors.

Summary in one sentence
The loop advances index by 1 after reading a surrogate pair instead of by 2, so index becomes equal to len(code_units) and the subsequent access triggers the "String index out of range" check at lines 35–36; fix by advancing index by the number of UTF‑16 code units consumed (2 for surrogate pairs, 1 otherwise) and ensure code_point_at guards access to index+1.
