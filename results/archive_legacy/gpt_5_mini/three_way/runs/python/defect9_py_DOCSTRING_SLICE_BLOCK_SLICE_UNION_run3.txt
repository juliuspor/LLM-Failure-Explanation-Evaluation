What happens
- The method replace_each first computes search_length and then runs the preflight loop at lines 81–85 to estimate how big the result may become.
- That loop (line 82) calls len(replacement_list[i]) and len(search_list[i]) for every index i:
  82 |             greater = len(replacement_list[i]) - len(search_list[i])
- If either replacement_list[i] or search_list[i] is None, calling len(...) raises TypeError: 'NoneType' object has no len(), so the function crashes there.

Why the crash is surprising
- Later in the method there is explicit logic intended to ignore None entries (see lines 102–105):
  102–105 |                     no_more_matches_for_repl_index[i]
           |                     or search_list[i] is None
           |                     or len(search_list[i]) == 0
           |                     or replacement_list[i] is None
- Those checks would correctly skip None entries when searching/matching, but they run only after the preflight loop. The code therefore tries to measure lengths of items that should have been ignored, causing the exception before the ignore logic ever runs.

Where the bug occurs (specific lines)
- Crash point: line 82 (len(...) calls) inside the preflight loop at 81–85.
- Intended guard that would have prevented this: lines 102–105, but those are executed later and too late.

Why this fix works
- It prevents len(...) from being called on None by performing the same None/empty checks that the search phase uses, but earlier — before any length-accessing code runs.

Summary
- Root cause: len(...) is called on None in the preflight loop (line 82) before the code that is supposed to ignore None entries (lines 102–105) runs.
- Fix: skip None/empty entries in the preflight loop (add the same guards used later) so no len(None) calls occur.
