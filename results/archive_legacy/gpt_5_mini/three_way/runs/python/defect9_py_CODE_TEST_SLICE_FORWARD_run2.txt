What happens
- The code crashes when either search_list[i] or replacement_list[i] is None because the code calls len(...) on them in the "increase" sizing loop (the loop starting just before line 83).
- That loop runs before any of the later null/empty checks, so None entries are not filtered out and you get a TypeError: object of type 'NoneType' has no len().

Where it fails (exact lines / variables)
- The problematic loop is at lines shown in your SLICE_FORWARD (around lines 77–85 in the original file). The key lines are:
  - greater = len(replacement_list[i]) - len(search_list[i])  (line ~83)
- If replacement_list[i] is None or search_list[i] is None, len(...) raises a TypeError. Tests that pass None inside the arrays (e.g. test_null_safety_inside_arrays_single and test_null_safety_inside_arrays_mixed) trigger this.

Why it fails (root cause)
- Earlier in the function you correctly guard against None/empty values during the initial match search and in the main replacement loop:
    if (
        no_more_matches_for_repl_index[i]
        or search_list[i] is None
        or len(search_list[i]) == 0
        or replacement_list[i] is None
    ):
        continue
  But the "increase" loop that tries to estimate result buffer size does not perform those same guards. It unconditionally uses len(search_list[i]) and len(replacement_list[i]). That makes the function not null-safe for elements inside the arrays, as your inline comment already notes.

Concrete fix (step-by-step)
1. In the "increase" loop, skip any i where search_list[i] is None, len(search_list[i]) == 0, or replacement_list[i] is None — exactly the same conditions you use elsewhere.
2. Then compute the length difference only for those i that remain.

Exact replacement code for the "increase" loop:

    increase = 0

    # Skip null/empty search strings and null replacements (null-safe).

            or replacement_list[i] is None
