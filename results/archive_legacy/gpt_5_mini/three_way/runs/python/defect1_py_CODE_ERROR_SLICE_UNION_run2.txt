What happened (where it failed)
- The failure occurs at lines 396–397:
  if minutes_offset < 0 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")
  The test passed minutes_offset = -15, so the code raises "Minutes out of range: -15" immediately.

Why it happened (root cause)
- The code later assumes minutes_offset can be signed when combining it with hours, but it rejects negative minutes up front.
  - At line 402 you compute:
      hours_in_minutes = hours_offset * 60
      if hours_in_minutes < 0:
          minutes_offset = hours_in_minutes - minutes_offset
    That branch expects to subtract a positive minutes_offset from a negative hours_in_minutes (i.e. treat minutes_offset as positive magnitude). But because you rejected negative minutes earlier, callers trying to represent e.g. "-0 hours and -15 minutes" (hours_offset = 0 or negative, minutes_offset = -15) are blocked.
- In short: the validation at lines 396–397 forbids negative minutes, but the later logic (lines 402–405) is written as if minutes_offset can be negative or must be combined with sign of hours. The code is inconsistent about the sign of minutes_offset and reuses the variable minutes_offset for two different meanings (raw input and computed total minutes), which causes the error.

Exact places to change (steps to fix)
1. Stop rejecting negative minutes unconditionally. Replace the current minutes validation (lines 396–397) with a check that the magnitude of minutes is 0..59, and handle sign explicitly when computing total minutes.
2. Do not reuse minutes_offset to hold total minutes; compute a new variable total_minutes to avoid confusion.

Concrete fix (drop-in replacement for the middle section)
- Validate magnitude:
    if minutes_offset < -59 or minutes_offset > 59:
        raise ValueError(f"Minutes out of range: {minutes_offset}")
  or better, require minutes_offset be 0..59 and let hours_offset carry the sign; then compute total minutes as:
    hours_in_minutes = hours_offset * 60
    if hours_offset < 0:

Recommended clear implementation
- Make minutes_offset an unsigned magnitude in the interface, or treat sign only on hours and combine like this:
    # require minutes magnitude between 0 and 59
    if minutes_offset < 0 or minutes_offset > 59:
        raise ValueError(...)
    total_minutes = hours_offset * 60
    if hours_offset < 0:
        total_minutes -= minutes_offset
    else:
        total_minutes += minutes_offset
- Or accept signed minutes and compute total in one line (no special-case):
    if minutes_offset < -59 or minutes_offset > 59:
        raise ValueError(...)
    total_minutes = hours_offset * 60 + minutes_offset

Why this fix works
- It removes the contradictory validation that forbids negative minutes while later treating minutes as if they could be negative.
- Using total_minutes (a new variable) avoids reusing minutes_offset for two meanings, so the logic is clear and the ValueError won't be raised incorrectly.

Summary
- The code fails because it rejects negative minutes (lines 396–397) while later treating minutes as possibly negative when computing the combined offset (lines 402–405). Fix by validating the magnitude properly and computing total_minutes explicitly (don't reuse minutes_offset).
