What happened (root cause)
- The loop that computes increase (lines 81–86 in your slice) calls len(replacement_list[i]) and len(search_list[i]) without checking whether those elements are None.
- But your function signature allows None inside the lists (search_list: List[Optional[str]], replacement_list: List[Optional[str]]). The rest of the code already treats individual elements as possibly None (see the checks at lines 59–76 and again 102–109).
- If any replacement_list[i] or search_list[i] is None, len(...) raises a TypeError and the function crashes before doing any replacements.

Where exactly it fails
- The problematic code is this loop (around line numbers shown in the slice):
  for i in range(search_length):
      greater = len(replacement_list[i]) - len(search_list[i])
      if greater > 0:
          increase += 3 * greater
- If replacement_list[i] is None, calling len(replacement_list[i]) triggers TypeError: object of type 'NoneType' has no len().
- If search_list[i] is None, calling len(search_list[i]) triggers the same error.

A concrete example that causes the crash
- Inputs:
  text = "hello"
  search_list = ["he", None]
  replacement_list = ["hi", None]
- On the loop computing increase, the iteration with i==1 will call len(None) and raise TypeError.

Why other parts don't crash
- Later loops (the match-finding loops) already guard each element with:
    if no_more_matches_for_repl_index[i] or search_list[i] is None or len(search_list[i]) == 0 or replacement_list[i] is None:
        continue
  so those sections skip None entries. The compute-increase loop misses that same guard.

How to fix it (concrete steps)
Option A — Skip null/empty elements (recommended, consistent with the rest of the function)
- Replace the increase loop with the same guard used elsewhere:

    for i in range(search_length):
        if search_list[i] is None or len(search_list[i]) == 0 or replacement_list[i] is None:
            continue
        greater = len(replacement_list[i]) - len(search_list[i])
        if greater > 0:
            increase += 3 * greater

Option B — Treat None as empty string (if that matches desired semantics)
- Convert None elements to "" before computing lengths (but be careful: elsewhere you treat None replacements as "skip", so this changes behavior).

Option C — Validate inputs early and raise
- At the start, reject lists that contain None (raise ValueError), e.g.:

    if any(x is None for x in search_list) or any(x is None for x in replacement_list):
        raise ValueError("search_list and replacement_list must not contain None")

Which to choose
- Given the rest of the function skips None elements, Option A is the least intrusive and consistent: skip None or empty search strings when sizing the buffer. It prevents the TypeError and preserves current behavior of ignoring nulls.

One-line patch you can apply
- Change the increase loop to:

    for i in range(search_length):
        if search_list[i] is None or len(search_list[i]) == 0 or replacement_list[i] is None:
            continue
        greater = len(replacement_list[i]) - len(search_list[i])
        if greater > 0:
            increase += 3 * greater

That fixes the crash by making the loop null-safe in the same way the rest of the function is.
