What happened (root cause)
- You clamp the input into variable v (lines ~56–58):
  v = max(value, self._lower_bound)
  v = min(v, self._upper_bound)
- But when computing g you accidentally use the original value instead of the clamped v (line 71):
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- Because of that, values outside the bounds (for example value = -0.5) produce a g < 0 (or > 255) and trigger the check at lines 74–75 which raises the ValueError you saw.

Where it fails
- get_paint: the clamping is done into v, but the interpolation uses value (line 71). The error is thrown at the validation check (lines 74–75).

Why the test fails
- test_lower_bound_clamping calls get_paint(-0.5). v becomes self._lower_bound (0.0) by clamping, but g is computed from -0.5, yielding a negative g and raising the ValueError instead of returning (0,0,0).

Exact fix (clear steps)
1. Use the clamped variable v when computing g. Replace line 71 with:
   g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
2. Add a guard against zero division if lower and upper bounds are equal (optional but recommended). For example, at the top of get_paint:
   if self._upper_bound == self._lower_bound:
       return (0, 0, 0)  # or raise a clear ValueError during init
3. (Optional) If you want symmetric rounding instead of truncation, use round(...) before int, but this is not required for the current tests.

This change ensures the clamped value is used for interpolation and prevents the false ValueError on out-of-range inputs.
