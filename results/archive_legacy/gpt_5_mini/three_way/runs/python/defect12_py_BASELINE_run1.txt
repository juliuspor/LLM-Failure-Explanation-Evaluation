What happened (root cause)
- When lower is greater than len(text) (e.g. lower == 15, len(text) == 10), the function ends up setting upper to a value larger than the string length and then calls substring_java(text, 0, upper). That slice checks end against len(s) and raises IndexError: "String index out of range: 15".
- The immediate cause is the sequence of assignments:
  1. if upper == -1 or upper > len(text): upper = len(text)
  2. if upper < lower: upper = lower
  After step (1) upper is clamped to len(text). If lower > len(text), step (2) sets upper = lower (which is > len(text)). Later substring_java(text, 0, upper) uses this too-large end index and throws.

Where it happens in your code
- The problematic lines are the two updates to upper (the first clamps upper to len(text); the second forces upper = lower when lower is bigger). Because you do the clamp before comparing to lower, upper can wind up > len(text).
- The exception is raised by substring_java(text, 0, upper) (the call that slices the string with an end index > len(text)).

Why this is incorrect behavior
- The function should not attempt to slice beyond the end of the string. If lower is beyond the end, the original Java behavior (and tests) expect the function to return the original text, not to create an out-of-range slice.

- Right after handling the None/empty checks, add:

      return text
  (Use >= because lower equal to len(text) cannot start a word and should return the whole string per the test example.)

- Ensure lower is reduced to at most len(text) before you set upper = lower:

Either option will prevent upper from becoming > len(text) and avoid the IndexError.
