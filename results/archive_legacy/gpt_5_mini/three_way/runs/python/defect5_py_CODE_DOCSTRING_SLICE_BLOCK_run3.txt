What happened (root cause)
- On entry the code tries to decide the "component type" of the array using:
  if array is not None:
      inferred_type = type(array)
  This returns the runtime class of the container itself (for a Python list, <class 'list'>), not the element type stored inside the list.
- ArrayUtils._copy_list_grow1(...) is being called with that wrong value (a container type instead of an element type). That is the wrong thing to pass and is the immediate cause of the failure you're seeing.
- The later type check (lines 346–351) tests inferred_type == object to decide whether to raise. Because inferred_type is <class 'list'>, that test doesn't behave as intended. In short: inferred_type is the wrong concept and value from the start.

Where it occurs (specific lines / variables)
- The error originates at the top of the function where inferred_type is set from array (the code quoted above).
- As a result, new_list = ArrayUtils._copy_list_grow1(array, inferred_type) receives <class 'list'> (variable inferred_type) instead of an element/component type.
- The later conditional at lines 346–351 (the slice block you included) uses inferred_type to decide whether to raise a TypeError; because inferred_type is the list class that conditional never expresses the intended semantics.

Why that causes failure (concrete example)
- Example: add([1, 2], 3, expected_type=int)

  - ArrayUtils._copy_list_grow1(array, inferred_type) is called with the list class where that method expects an element/component type (or at least the code that follows assumes inferred_type represents the element type).

Because inferred_type is <class 'list'>, the comparison is wrong and the function can either skip an intended error or raise the wrong one (depending on _copy_list_grow1 implementation), producing the failure you saw.
