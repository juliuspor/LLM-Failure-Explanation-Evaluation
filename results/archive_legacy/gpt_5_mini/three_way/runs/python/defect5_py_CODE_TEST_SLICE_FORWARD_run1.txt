What happened (root cause)

- At the top of add() you compute inferred_type like this:
  - if array is not None: inferred_type = type(array)
  - elif element is not None: inferred_type = type(element)
  - else: inferred_type = object

  That is wrong for two reasons:
  1. type(array) returns the Python list type, not the element type. If array is ["a","b"], type(array) is list, not str. So inferred_type becomes list, which is not the intended element type used later.
  2. When both array and element are None you fall back to object. Immediately afterwards you raise a TypeError whenever inferred_type == object and expected_type != object. So calling add(None, None, expected_type=str) triggers that raise. That is exactly the failing test test_add_null_to_null_with_expected_type.

Where it happens (specific lines / variables)

- Line(s) around:
    if array is not None:
        inferred_type = type(array)
    elif element is not None:
        inferred_type = type(element)
    else:
        inferred_type = object

  This sets inferred_type incorrectly.

- The TypeError is thrown in this block:
    if expected_type is not None:
        if inferred_type == object and expected_type != object:
            raise TypeError(...)

  Because inferred_type == object when both inputs are None, the function rejects the expected_type path instead of using expected_type to decide the element type.

Why that causes the test failure

- The test calls ArrayUtils.add(None, None, expected_type=str). The code sets inferred_type = object, then sees expected_type is not None and inferred_type == object and expected_type != object, so it raises TypeError with the ClassCastException message. The test expects [None] to be returned, not an exception.

How to fix it (clear, concrete steps)

1. Decide precedence for determining the element type. A correct, simple precedence:
   - If expected_type is provided, use that for element type (highest priority).
   - Else, if element is not None, use type(element).
   - Else, if array is not None, inspect the array for the element type (e.g. first non-None element or a provided stored element type); do not use type(array).
   - Else default to object (or None) and do not raise just because expected_type exists (because expected_type would have been used already).

2. Change the inferred_type assignment accordingly. Example replacement for the three-line block:

    if expected_type is not None:
        inferred_type = expected_type
    elif element is not None:
        inferred_type = type(element)
    elif array:
        # pick first non-None element's type if present, else object
        first_non_none = next((x for x in array if x is not None), None)
        inferred_type = type(first_non_none) if first_non_none is not None else object
    else:
        inferred_type = object

3. Remove or change the later TypeError check. If you follow step 2 and give expected_type priority, you no longer need to raise TypeError when inferred_type == object and expected_type != object (that was enforcing a cast that should instead be satisfied by choosing expected_type earlier). Either remove that raise entirely or change it to a real mismatch check (e.g. when inferred_type is known and incompatible with expected_type).

4. Also fix using type(array): if any code downstream expects an element class, make sure you pass an element type, not list. If ArrayUtils._copy_list_grow1 expects the element type, pass inferred_type as above; if it expects the container type, adjust accordingly. Based on the tests, _copy_list_grow1 should create a new list of elements; it should be given the element type, not list.

Minimal code patch (illustrative)

- Replace the inferred_type block and the TypeError block with:

    # choose element type: prefer expected_type, then element, then array contents
    if expected_type is not None:
        inferred_type = expected_type
    elif element is not None:
        inferred_type = type(element)
    elif array:
        first_non_none = next((x for x in array if x is not None), None)
        inferred_type = type(first_non_none) if first_non_none is not None else object
    else:
        inferred_type = object

    # no immediate TypeError if expected_type provided — we already used expected_type if present

Why this will fix the tests

- For add(None, None, expected_type=str): inferred_type becomes str, so _copy_list_grow1 will create a list consistent with expected_type and no TypeError is raised. For typical calls with a non-null element, element's type is used. For a non-empty array, we inspect element types instead of using type(array) which prevented the code from treating the element type as "list".

Additional note

- If you need stronger runtime checks (e.g. prevent adding an int to a str-typed array), implement an explicit compatibility check: if expected_type is set and element is not None and not isinstance(element, expected_type): raise TypeError. The current code attempted a Java-style ClassCastException message — in Python prefer isinstance checks and meaningful Python exceptions.
