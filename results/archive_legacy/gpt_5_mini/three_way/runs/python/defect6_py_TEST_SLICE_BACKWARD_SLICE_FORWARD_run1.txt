What happened (root cause)
- The code treats any float x that equals int(x) as an integer at line 176:
  if x == int(x) and not math.isinf(x) and not math.isnan(x):
- For -0.0, x == int(x) is True because -0.0 == 0.0, and int(x) becomes 0. That loses the sign of negative zero immediately (value = int(x) at line 177).
- Later the integer branch writes out value (or otherwise uses the integer-form result) and ends up producing "0" (or sometimes str(x) after other branches), not "-0.0". The negative sign is gone because int(x) cannot represent negative zero.
- Therefore add_number(-0.0) prints "0" instead of the expected "-0.0". The bug is in the integer-detection branch in add_number (around lines 176–194).

Where exactly it occurs
- The faulty logic is in add_number, lines shown in your slice:
  - Line 176: if x == int(x) and not math.isinf(x) and not math.isnan(x):
  - Line 177: value = int(x)  # loses negative zero sign for -0.0
  - Later at line 192: self.add(str(value))  # writes "0"
  - Or at line 194: self.add(str(x)) may be used in other paths, but the integer branch is what discards the sign.

Why that loses the sign
- Casting -0.0 to int yields 0 (no sign information).
- The code uses that integer value to form the output, so the original negative-zero sign can no longer be recovered.

Concrete, actionable fixes (exact steps)
1. Detect negative zero before converting to int. Add an explicit check at the start of the integer branch:
   - Use the existing utility CodeConsumer.is_negative_zero(x) (tests show it exists).
2. If is_negative_zero(x) is True, output "-0.0" immediately and return; do not enter the integer conversion logic.

Patch example (apply in add_number near line 176)
- Immediately before or at the start of the integer branch:
  if CodeConsumer.is_negative_zero(x):
      self.add("-0.0")
      return
  # then continue with the existing 'if x == int(x) ...' code

Why this fix works
- It preserves the negative sign for the special -0.0 case before any int() conversion can remove it.
- It keeps the integer-path optimization for true integers while handling the sign-sensitive special case separately.

Alternate approaches (if you prefer)
- Instead of early return, preserve sign by formatting based on math.copysign or inspect float bits (struct.pack/unpack) before converting to int. But using is_negative_zero and an early "-0.0" output is the simplest, least invasive fix.

What to test after fixing
- The provided test_negative_zero should pass (add_number(-0.0) -> "-0.0").
- All other tests (positive zero, integers, floats, scientific notation) should be re-run to confirm no regressions.
