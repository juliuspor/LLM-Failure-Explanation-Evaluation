What happened (where it fails)
- At line 335–336 the code does: if array is not None: inferred_type = type(array)
- That sets inferred_type to the Python type of the container object (usually list), not the type of the elements inside the list.
- Later (line 346) the code checks if inferred_type == object and expected_type != object and raises TypeError.
- Because inferred_type was set incorrectly, the function can end up treating the array as an "object array" and raising the error even when the elements are of the expected type.

Why this is wrong (root cause)
- type(array) returns the class of the container (e.g. list), not the class of items in the container. The code intends to infer the element type but is inferring the container type. That bad inference triggers the guard at line 346 and raises the TypeError incorrectly.
- The elif branch (line 337–338) handles element when passed directly, but when array is provided the code never looks at its elements to determine their types.
- If the array is empty, you do need a fallback (object or expected_type), but the current flow conflates "unknown element type" with "container type is object".
