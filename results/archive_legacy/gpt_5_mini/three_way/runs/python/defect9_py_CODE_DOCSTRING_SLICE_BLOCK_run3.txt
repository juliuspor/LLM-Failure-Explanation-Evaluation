What happened (what fails)
- At lines 81–84 the code does:
  greater = len(replacement_list[i]) - len(search_list[i])
  This calls len(...) on replacement_list[i] and search_list[i] without checking whether either is None.
- If replacement_list[i] is None or search_list[i] is None this raises TypeError: object of type 'NoneType' has no len().
- The rest of the function clearly intends to ignore None or empty search/replacement entries (see the null-safe checks used in the loops that scan the text), so the crash occurs because this one sizing loop is not null-safe.

Why it fails (root cause)
- Root cause: the "increase" sizing loop (starting at line 81) assumes every element in search_list and replacement_list is a non-None string. It does not apply the same guards that the matching loops do (the checks for no_more_matches_for_repl_index[i], search_list[i] is None, len(search_list[i]) == 0, or replacement_list[i] is None).
- Concrete variable names & location: replacement_list[i] or search_list[i] being None at line 82 triggers the TypeError. This is the exact mismatch between intended behavior (ignore None entries) and this loop's implementation.

Minimal example that reproduces the crash
- text = "a"
- search_list = ["a", None]
- replacement_list = ["b", None]
- On the sizing loop the code executes len(replacement_list[1]) which raises TypeError.

How to fix it (step-by-step, explicit code change)
Option A — skip None/empty entries (recommended):
Replace the loop at lines 81–84 with a guarded version that mirrors the checks used when searching:

    for i in range(search_length):
        if (
            no_more_matches_for_repl_index[i]
            or search_list[i] is None
            or len(search_list[i]) == 0
            or replacement_list[i] is None
        ):
            continue
        greater = len(replacement_list[i]) - len(search_list[i])
        if greater > 0:
            increase += 3 * greater  # assume 3 matches

This keeps behavior consistent with the search loops and prevents len(None).

Option B — defensive length calculation:
If you prefer a compact check, compute lengths only if both elements are strings:

    for i in range(search_length):
        s = search_list[i]
        r = replacement_list[i]
        if not (isinstance(s, str) and isinstance(r, str) and len(s) > 0):
            continue
        greater = len(r) - len(s)
        if greater > 0:
            increase += 3 * greater

Option C — safe coercion (less explicit, not recommended because it hides data problems):
Treat None as empty string:

    for i in range(search_length):
        s_len = len(search_list[i]) if search_list[i] is not None else 0
        r_len = len(replacement_list[i]) if replacement_list[i] is not None else 0
        if s_len == 0 or r_len == 0:
            continue
        greater = r_len - s_len
        if greater > 0:
            increase += 3 * greater

Which to choose
- Use Option A (mirror existing checks) so the sizing loop behaves exactly like the matching loops: ignore entries that are None or empty or already marked in no_more_matches_for_repl_index. That is the least surprising and safest change.

Summary
- The crash originates at line 82 where len(...) is called on a None element. Fix by adding the same null/empty/replacement None checks used elsewhere (see lines where matching uses those checks) before computing lengths.
