What happened (root cause)
- At lines 346–349 you intended to update _min_start_index only when the new period starts earlier than the currently recorded minimum, or to set it when no minimum exists yet.
- But line 349 unconditionally sets self._min_start_index = index, which overwrites the conditional update done at line 347. As a result the code always sets _min_start_index to index whenever execution reaches that section, so the “minimum” tracking never actually keeps the earliest start time.
- Concretely:
  - Line 340 checks whether a minimum index already exists (self._min_start_index >= 0).
  - If it does, you compare start < min_start and at line 347 set self._min_start_index = index only when start is earlier.
  - Immediately after (line 349) you set self._min_start_index = index unconditionally, undoing the check.

Where it occurs
- Problem appears in the block shown around lines 340–349:
 340 |         if self._min_start_index >= 0:
 346 |             if start < min_start:
 347 |                 self._min_start_index = index
 349 |             self._min_start_index = index

Why this causes failure
- Because the unconditional assignment means the stored “minimum start index” is always the most recently processed index (or simply overwritten), not the index of the earliest start. That breaks any code that relies on get_min_start_index (line 437–439) to return the true minimum-start item.

How to fix it (step-by-step, exact change)
Option A — Preserve the conditional logic (most likely intended)
1. Remove the unconditional assignment at line 349 so the index is only set in the correct places:
   Replace the block with:
   if self._min_start_index >= 0:
       # compute min_start...
       if start < min_start:
           self._min_start_index = index
   else:
       # no current minimum, so set it
       self._min_start_index = index

2. Ensure the else branch exists so you set _min_start_index when it was previously -1.

Option B — Equivalent explicit patch
Replace the lines 340–349 with:
if self._min_start_index >= 0:
    min_start_period = self.get_data_item(self._min_start_index).get_period()
    if isinstance(min_start_period, SimpleTimePeriod):
        min_start = min_start_period.get_start_millis()
    else:
        min_start = int(min_start_period.get_start().timestamp() * 1000)
    if start < min_start:
        self._min_start_index = index
else:
    self._min_start_index = index

Notes and small cleanups

- Make sure min_start is always defined before comparing (the code above ensures that).

Summary
- The bug is a stray unconditional assignment at line 349 that overwrites the conditional update at line 347. Remove that unconditional assignment and add an else branch so _min_start_index is set only when appropriate.
