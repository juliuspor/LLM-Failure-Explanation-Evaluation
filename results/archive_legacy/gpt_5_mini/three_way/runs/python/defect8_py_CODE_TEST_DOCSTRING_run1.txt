What happened (where it fails)
- The failure occurs in LocaleUtils.to_locale when parsing strings like "fr__POSIX" (language + empty country + variant).
- After verifying the first underscore at locale_str[2], the code always reads and validates ch3 = locale_str[3] and ch4 = locale_str[4] as uppercase letters:
  - ch3 = locale_str[3]
  - ch4 = locale_str[4]

- For "fr__POSIX", locale_str[3] is '_' (the second underscore). That character is not between 'A' and 'Z', so the code raises ValueError("Invalid locale format: fr__POSIX").

Why it happens (root cause)
- The method assumes that whenever length > 2 there must be a country code occupying positions 3 and 4. It does not account for the valid format language__variant (empty country, indicated by two consecutive underscores).
- In other words, the code enforces a country code unconditionally after the first underscore, but a double underscore is a legal case meaning country == "" and a variant follows.
