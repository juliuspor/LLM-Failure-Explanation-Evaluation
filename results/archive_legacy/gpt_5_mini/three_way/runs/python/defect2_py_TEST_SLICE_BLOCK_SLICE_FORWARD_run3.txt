What happened (where): In GrayPaintScale.get_paint the code checks the computed color component g and, if it's out of [0,255], raises a ValueError at lines 74–75:

  74 |         if g < 0 or g > 255:
  75 |             raise ValueError(f"Color parameter outside of expected range: Red Green Blue")

Why it fails (root cause):
- The method is supposed to clamp input values to the scale and return a gray RGB tuple. Instead it raises an exception when the computed gray component g falls outside 0..255. Because the test calls get_paint(-0.5) (and the Java original had the same case), g becomes negative and the code raises ValueError. The tests expect clamping to black/white, not an exception.
- The check uses only variable g (line 74). If you intended to validate r, g, b you are only checking g here.
- The error message on line 75 is an f-string but contains no variables — that’s misleading but not the test failure cause. The actual failure comes from raising an exception instead of clamping.

Exactly what to change (concrete steps):
1. Clamp the scaled gray value to the 0..255 range instead of raising an exception. Replace the if + raise on lines 74–75 with something like:
   g = max(0, min(255, g))
   (and if you have r and b variables, apply the same clamp to them).
2. Ensure input value is clamped to the configured lower and upper bounds before mapping to 0..255. Example at start of get_paint(value):
   if value < self.lower: value = self.lower
   if value > self.upper: value = self.upper
3. Convert to integer the same way the tests expect (they expect 0 -> 0, 255 -> 255 and 0.5 -> 127). Use int(...) or math.floor consistently when converting float to integer.

Minimal patch example (replace lines 74–75):
  # clamp computed component to valid byte range instead of raising
  g = max(0, min(255, g))

If the function computes r,g,b, do this for each:
  r = max(0, min(255, r))
  g = max(0, min(255, g))
  b = max(0, min(255, b))

And earlier, before mapping:
  # clamp input to configured bounds
  if value < self.lower: value = self.lower
  if value > self.upper: value = self.upper

Why this fixes the tests:
- Clamping input before mapping guarantees negative inputs map to the lower bound (black) and >upper map to white.
- Clamping color components prevents exceptions from being raised for values slightly out of range due to arithmetic, matching the tests' expectations.

Notes:
- If you still want to raise errors for programming mistakes (e.g., r/g/b not numbers), raise a specific error earlier with useful info. Do not raise for values that should be clamped as part of normal behavior.
