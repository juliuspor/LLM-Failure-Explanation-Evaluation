What happened (where it failed)
- The test called DateTimeZone.for_offset_hours_minutes(0, -15).
- Inside for_offset_hours_minutes the variable minutes_offset is checked at lines 396–397:

  397 |             raise ValueError(f"Minutes out of range: {minutes_offset}")
- Because minutes_offset is -15, the condition minutes_offset < 0 is true and the code raises ValueError("Minutes out of range: -15").
- So the immediate root cause is that negative minute values (e.g. -15) are being rejected by that range check.

- The test suite (and Joda-Time behavior referenced by the tests) expects minutes to be allowed in the range -59..+59, not only 0..59.
- The current check only allows 0..59. It therefore incorrectly treats valid negative minute inputs as out-of-range.
- This is a validation bug in for_offset_hours_minutes: it uses the wrong lower bound for minutes.
