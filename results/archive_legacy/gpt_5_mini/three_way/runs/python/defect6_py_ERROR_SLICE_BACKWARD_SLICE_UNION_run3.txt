What happened (where it fails)
- At line 176 the code enters the integer-handling branch for any float x that equals int(x):
  - 176: if x == int(x) and not math.isinf(x) and not math.isnan(x):
- For x == -0.0 that condition is True because -0.0 == 0 is True.
- At line 177 you do value = int(x). int(-0.0) is 0, so the negative sign is lost.
- Later (line 192) you call self.add(str(value)), which produces "0", not "-0.0".
- The test expected the negative zero string "-0.0", so you get the assertion: Expected -0.0, got 0.

Why it happens (root cause)
- Floating-point negative zero compares equal to positive zero (x == int(x) is True), but converting to int removes the sign. The code assumes equality to an integer means it can safely convert to int and use that integer representation — that assumption fails for -0.0 because the sign carries meaning for the expected string output.

Exactly where to change it and how
Option A — avoid entering integer branch for negative zero:
- Change the if at line 176 to skip the integer branch when x is negative zero:
  if x == int(x) and not math.isinf(x) and not math.isnan(x) and not (x == 0.0 and math.copysign(1.0, x) < 0):
  This keeps -0.0 out of the int-handling code so later you can output str(x) (which yields "-0.0").

Option B — preserve the sign when value==0:
- Keep the branch but detect negative zero after computing value and use the proper string:
  value = int(x)
  if value == 0 and math.copysign(1.0, x) < 0:
      self.add("-0.0")
  else:
      self.add(str(value))
  This ensures you emit "-0.0" instead of "0".

Which function to use
- Use math.copysign(1.0, x) < 0 to detect a negative sign even when x == 0.0. (You could also use struct.pack/unpack, but copysign is simple and clear.)

Summary
- The bug is that the code converts -0.0 to int and loses the sign, producing "0" instead of "-0.0".
- Fix by detecting negative zero (math.copysign) and either avoid the int conversion for it or format it explicitly as "-0.0".
