- substring_java(s, 0, 15) runs the slice check shown in [SLICE_UNION]. Because end (15) > len(s) (10), it raises IndexError: "String index out of range: 15".

Why it fails (root cause)
- The code first clamps upper down to len(text) and then, if upper < lower, sets upper = lower. When lower is already greater than len(text), that sets upper to a value > len(text). There is no check afterwards to prevent calling substring_java with end > len(text).
- In short: the line "if upper < lower: upper = lower" can reintroduce an upper bound that is larger than the string length, so substring_java is called with an invalid end index and raises IndexError.

Where exactly (functions/variables/lines)
- The problematic sequence is the two checks that touch upper:
  - "if upper == -1 or upper > len(text): upper = len(text)" (caps upper to the string length)
  - "if upper < lower: upper = lower" (may increase upper again)
- The exception is thrown in substring_java(text, 0, upper) when upper > len(text). The slice check that raises the error is the one in [SLICE_UNION] lines 32–34 (start/end validation).

Concrete fixes (clear steps you can apply)
Pick one of these options depending on intended behavior:

Option A — refuse/short-circuit when lower is beyond the string:
1. After computing len(text), add:
     if lower > len(text):
         return text  # or return StringUtils.EMPTY or None depending on intended contract
2. Leave the rest unchanged.
This prevents ever calling substring_java with start/end > len(text).

Option B — clamp lower instead of raising upper:
1. Replace

         upper = lower
   with

         lower = min(lower, len(text))
         upper = lower

         upper = lower if lower <= len(text) else len(text)
This ensures upper is never set above len(text).

1. Replace the two-step logic with a single safe computation, for example:

This guarantees upper is always in [lower, len(text)] and never > len(text).
