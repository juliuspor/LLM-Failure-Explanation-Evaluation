What happened
- The bug is in the try block where the code converts hours and minutes to a single minutes value:

      hours_in_minutes = hours_offset * 60
      if hours_in_minutes < 0:
          minutes_offset = hours_in_minutes - minutes_offset
      else:
          minutes_offset = hours_in_minutes + minutes_offset

- That logic wrongly handles negative hours combined with negative minutes. Example: hours_offset = -2, minutes_offset = -15:
  - hours_in_minutes = -120
  - the code executes minutes_offset = hours_in_minutes - minutes_offset → -120 - (-15) = -105
  - expected total minutes is -120 - 15 = -135 (i.e. -02:15), but the code produces -105 (i.e. -01:45).

Why it fails (root cause)
- The code reuses the minutes_offset variable to hold the computed total and performs subtraction of minutes_offset when hours are negative without taking absolute value. Subtracting a negative minutes_offset effectively adds instead of increasing the magnitude.
- In short: the sign handling is wrong for negative hours combined with negative minutes (and also for negative hours combined with positive minutes the intended behavior requires subtracting the magnitude of minutes, not adding/minus-negating blindly).
- This error occurs in the try block while calculating the combined minutes (the two lines above), before safe_multiply is called.

Where it occurs (specific names/lines)
- Inside for_offset_hours_minutes, in the try block:
  - hours_in_minutes = hours_offset * 60
  - if hours_in_minutes < 0:
        minutes_offset = hours_in_minutes - minutes_offset
    else:
        minutes_offset = hours_in_minutes + minutes_offset
  - Then offset = FieldUtils.safe_multiply(minutes_offset, DateTimeConstants.MILLIS_PER_MINUTE)

Concrete example showing the wrong arithmetic:
- Input (-2, -15):
  - Expected total minutes: -2*60 - 15 = -135 → offset -135 * MILLIS_PER_MINUTE
  - Actual computed by code: -120 - (-15) = -105 → wrong offset

How to fix it (step-by-step, exact changes)
1. Compute a separate total_minutes variable instead of overwriting minutes_offset.
2. Enforce the sign rules from the docstring:
   - If hours > 0 and minutes < 0 → raise ValueError
   - If hours < 0 → total_minutes = hours_in_minutes - abs(minutes_offset)
   - If hours == 0 → total_minutes = minutes_offset
   - If hours > 0 → total_minutes = hours_in_minutes + minutes_offset
3. Multiply total_minutes by MILLIS_PER_MINUTE (use FieldUtils.safe_multiply) and continue.

Exact replacement code for the try block (drop-in):

    try:
        hours_in_minutes = hours_offset * 60
        # invalid: positive hours with negative minutes
        if hours_in_minutes > 0 and minutes_offset < 0:
            raise ValueError("Minutes must not be negative when hours is positive")
        if hours_in_minutes < 0:
            total_minutes = hours_in_minutes - abs(minutes_offset)
        else:
            total_minutes = hours_in_minutes + minutes_offset
        offset = FieldUtils.safe_multiply(total_minutes, DateTimeConstants.MILLIS_PER_MINUTE)
    except ArithmeticError:
        raise ValueError("Offset is too large")

Why this fix works
- It computes total_minutes deterministically and handles signs according to the documented behavior:
  - For negative hours, both (-2, 15) and (-2, -15) produce -135 minutes.
  - For positive hours, (2, 15) → +135; (2, -15) correctly raises ValueError.
  - For zero hours, minutes keep their own sign.

Summary
- Root cause: incorrect sign arithmetic when combining negative hours with negative minutes (the line minutes_offset = hours_in_minutes - minutes_offset causes double-negation).
- Fix: compute a new total_minutes using abs(minutes_offset) for negative hours and validate the illegal combination (positive hours with negative minutes).
