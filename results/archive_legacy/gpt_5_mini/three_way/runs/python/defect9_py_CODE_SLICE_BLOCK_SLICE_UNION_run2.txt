What happened (what fails)
- The function throws a TypeError at lines 81–84 when it tries to call len(replacement_list[i]) or len(search_list[i]) for an element that is None.
- Exact symptom: Python raises "TypeError: object of type 'NoneType' has no len()" at line 82 (the len(replacement_list[i]) call).

Why it happens (root cause)
- Earlier in the function you allow None elements inside search_list and replacement_list and you explicitly skip them in the matching loops (lines 59–70 and lines 101–108) with checks like:
  or search_list[i] is None
  or len(search_list[i]) == 0
  or replacement_list[i] is None
- However, the sizing/estimate loop at lines 81–84 does not perform those same None checks. It unconditionally does:
  greater = len(replacement_list[i]) - len(search_list[i])
- If either search_list[i] or replacement_list[i] is None, calling len(...) raises the TypeError. So the code works when all elements are strings, but fails whenever any array element is None.

Where it occurs (specific lines / variables)
- Failure occurs inside the loop starting at line 81:

  82 |     greater = len(replacement_list[i]) - len(search_list[i])

  84 |         increase += 3 * greater
- Variables involved: replacement_list[i], search_list[i]. The len() calls are the immediate cause.

How to fix it (concrete, actionable steps)
1. Add the same null/empty checks used elsewhere before calling len() in that loop. For example:
   for i in range(search_length):
       if search_list[i] is None or len(search_list[i]) == 0 or replacement_list[i] is None:
           continue
       greater = len(replacement_list[i]) - len(search_list[i])
       if greater > 0:

2. Alternatively, compute safe lengths using conditional expressions to avoid calling len on None:

       greater = r_len - s_len
       if s_len == 0 or replacement_list[i] is None:
           continue
       if greater > 0:

3. (Optional) Move the estimate loop after the initial match-scan so you know which indices are valid and you can base the estimate only on those indices.

Any of these fixes will stop len(...) being called on None and prevent the TypeError.
