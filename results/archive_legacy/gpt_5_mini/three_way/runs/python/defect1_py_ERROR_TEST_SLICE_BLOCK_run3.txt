What happened (where it failed)
- The test called DateTimeZone.for_offset_hours_minutes(0, -15).
- Inside that method, at lines 396–397 the code does:
  if minutes_offset < 0 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")
- minutes_offset is -15 for this call, so minutes_offset < 0 is true and a ValueError is raised with the message "Minutes out of range: -15". That is exactly what the failing test saw.

Why it failed (root cause)
- The code is incorrectly treating any negative minute value as out of range. The condition minutes_offset < 0 is wrong for the intended behavior. The tests (and the usual API contracts referenced in your test suite) expect minutes to be allowed in the range -59..+59, so -15 should be valid.
- Concretely: the variable minutes_offset contains the raw minutes argument (which can be negative). The check at line 396 should allow negative minutes in range -59..59, but instead it rejects all negative values.
