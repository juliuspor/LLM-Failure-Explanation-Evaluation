What happened (where it fails)
- The TypeError occurs at the loop that computes variable increase (your commented "BUG" block). Specifically at
  greater = len(replacement_list[i]) - len(search_list[i])
  (your slice points to lines ~81–82).
- Either replacement_list[i] or search_list[i] is None, so calling len(None) raises TypeError: "object of type 'NoneType' has no len()".

Why it happens (root cause)
- The function allows elements of search_list and replacement_list to be Optional[str] (you declared both lists as List[Optional[str]]).
- Earlier logic that searches the text correctly skips elements that are None or empty:
  if no_more_matches_for_repl_index[i] or search_list[i] is None or len(search_list[i]) == 0 or replacement_list[i] is None: continue
- But the "increase" loop (the buffer-size guess) does not perform the same null checks. It unconditionally calls len(...) on both search_list[i] and replacement_list[i]. If either element is None, len(None) throws the TypeError you saw.

Where in code to change
- The problematic block starts at:
  # BUG: this loop is not null-safe for elements inside the arrays
  for i in range(search_length):
      greater = len(replacement_list[i]) - len(search_list[i])
      if greater > 0:
          increase += 3 * greater  # assume 3 matches

How to fix it (concrete steps)
1. Make the loop null-safe: skip entries where search_list[i] is None or empty or replacement_list[i] is None before calling len().
   Example replacement for that loop:

   for i in range(search_length):
       if search_list[i] is None or len(search_list[i]) == 0 or replacement_list[i] is None:
           continue
       greater = len(replacement_list[i]) - len(search_list[i])
       if greater > 0:
           increase += 3 * greater

2. (Alternative) Normalize the lists up front so neither contains None (if that matches desired semantics). For example convert None replacements to "" or filter out pairs where search is None/empty or replacement is None:
   safe_pairs = [(s, r) for s, r in zip(search_list, replacement_list) if s and r is not None]
   then use safe_pairs everywhere (update search_length, loops, lookups).

Recommendation
- Prefer the small change in step 1: add the same null/empty checks to the "increase" loop so the function behaves consistently and avoids len(None) errors.
