- In get_paint (around lines 54–75) you compute g on line 71:

- Immediately after (lines 74–75) you check if g < 0 or g > 255 and raise ValueError:

Why it fails (root cause)
- The tests expect values outside the configured bounds to be clamped to the nearest endpoint (black for below lower_bound, white for above upper_bound). Instead your code raises a ValueError whenever the computed g is outside 0..255.
- For example test_lower_bound_clamping calls get_paint(-0.5). That makes the fraction negative, so g < 0 and the code raises ValueError at the check on lines 74–75. The exception is why the test fails.
- You also compute g directly from the raw value before clamping; that calculation can produce values outside 0..255 which you then treat as exceptional rather than clamping.
- (Also note: the error message is unhelpful and does not mention which value/bounds caused the problem.)

Exactly where to change it
- Replace the post-computation raise on lines 74–75 with proper clamping logic.
- Also guard the division by zero case: if self._upper_bound == self.__lower_bound you should handle or raise a clear error.

Concrete fix (code to replace around lines 71–75)
Replace:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
  if g < 0 or g > 255:
      raise ValueError(f"Color parameter outside of expected range: Red Green Blue")

With something like:
  # avoid division by zero
  if self._upper_bound == self._lower_bound:
      raise ValueError("upper_bound must be different from lower_bound")
  # compute normalized fraction and clamp to [0.0, 1.0]
  frac = (value - self._lower_bound) / (self._upper_bound - self._lower_bound)
  if frac < 0.0:
      frac = 0.0
  elif frac > 1.0:
      frac = 1.0
  # convert to 0..255 integer
  g = int(frac * 255.0)
  # ensure integer stays within byte range (defensive)
  g = max(0, min(255, g))

Return (g, g, g).

Why this fixes the tests
- Values below lower_bound produce frac <= 0.0 and get clamped to 0.0 -> g = 0 -> (0,0,0).
- Values above upper_bound produce frac >= 1.0 and get clamped to 1.0 -> g = 255 -> (255,255,255).
- Middle values interpolate and map 0.5 -> 127 as expected.
- Division-by-zero is handled with a clear error message rather than an obscure crash.
