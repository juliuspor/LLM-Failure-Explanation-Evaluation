What failed
- The test that uses negative minutes (e.g. DateTimeZone.for_offset_hours_minutes(-2, -15) or (0, -15)) fails because the function rejects negative minute values before it ever computes the total offset.
- The rejection happens at the minutes range check on lines 396–397:

  396 |         if minutes_offset < 0 or minutes_offset > 59:
  397 |             raise ValueError(f"Minutes out of range: {minutes_offset}")

Why it fails (root cause)
- That condition treats any negative minute value as invalid (minutes_offset < 0 triggers the ValueError).
- The test suite expects minutes to be allowed in the range -59..+59 (negative minute values are valid input in the code under test). By rejecting minutes < 0 the implementation contradicts the expected API and the docstring behavior.
- As a result, calls such as DateTimeZone.for_offset_hours_minutes(-2, -15) and DateTimeZone.for_offset_hours_minutes(0, -15) raise ValueError from that check even though the tests expect those inputs to produce offsets (-02:15 and -00:15).

Where it occurs
- File: hit01_timezone.py
- Lines: 396–397 (the minutes bounds check shown above)
- Variable: minutes_offset

What to change (clear steps)
1. Allow minutes_offset to be negative in the accepted range -59..+59. Replace the condition with one of these correct checks:
   - if minutes_offset < -59 or minutes_offset > 59: raise ValueError(...)
   - or if abs(minutes_offset) > 59: raise ValueError(...)

2. Decide and implement consistent sign-handling between hours and minutes:
   - If you want the API to accept independently signed hours and minutes and compute the total offset additive (as the tests expect for negative hours/minutes), compute total_millis using:
     total_minutes = hours * 60 + minutes
     total_millis = total_minutes * 60 * 1000
     (This will naturally handle negative minutes when hours is negative or zero.)
   - If you instead want to enforce that hour and minute signs must match, validate that sign(hours) == sign(minutes) (with special-case zero) and raise ValueError for mixed signs. The current tests expect mixed inputs like (-2, 15) to produce -02:15, so prefer additive behavior or the sign-normalizing behavior described in the docstring.
