What happened (what failed)
- The test calls DateTimeZone.for_offset_hours_minutes with a negative minutes value (for example (0, -15) and (-2, -15)).
- The method immediately rejects any negative minutes at the check on lines 396–397:

      raise ValueError(f"Minutes out of range: {minutes_offset}")
- Because that check requires minutes_offset >= 0, any negative minutes cause a ValueError and the tests fail.

Why it happens (root cause)
- The minutes range check is wrong: it restricts minutes to [0, 59] while the expected/allowed range is [-59, 59]. The test suite and docstring expect negative minute values to be accepted (e.g. -15) in at least some contexts.
- This check occurs before any logic that would combine hour and minute signs, so the method never gets to handle negative minutes correctly.
