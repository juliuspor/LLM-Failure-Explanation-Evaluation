What happened (where it failed)
- The failure happens in LocaleUtils.to_locale at the block that parses the country code (around lines 231–240).
- For input "fr__POSIX" the code computes:
  - locale_str = "fr__POSIX"
  - ch3 = locale_str[3] which is '_' and ch4 = locale_str[4] which is 'P'.
- At line 239 the code checks:
  if ch3 < 'A' or ch3 > 'Z' or ch4 < 'A' or ch4 > 'Z':
    raise ValueError(f"Invalid locale format: {locale_str}")
- Because ch3 == '_' the check fails and a ValueError is raised with "Invalid locale format: fr__POSIX".

Why this is wrong (root cause)
- The implementation assumes that when a variant is present there will always be a two-letter country code at positions 3 and 4.
- It does not handle the valid form language + "__" + variant (i.e. an empty country, two underscores, then variant). In "fr__POSIX" the country is intentionally empty and the variant begins after the second underscore. The code never checks for the double-underscore case, so it treats the first underscore as an invalid country letter.

- Modify the country-parsing logic in to_locale (the code around lines 231–240) to explicitly allow the double-underscore pattern and extract the variant correctly.

  2. If locale_str[3] == '_' (i.e. the double-underscore case), set country = "" and set variant = locale_str[4:].
  3. Otherwise validate ch3 and ch4 as uppercase letters (existing checks) and set country = locale_str[3:5] and variant = locale_str[6:] (or whatever the existing indexing uses).
  4. Only raise the ValueError if neither valid uppercase country letters nor the double-underscore pattern match.
