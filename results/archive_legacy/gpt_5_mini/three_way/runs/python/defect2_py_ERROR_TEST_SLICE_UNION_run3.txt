What happened (where and why)
- In GrayPaintScale.get_paint (line 71) you compute g directly from the input value:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- Immediately afterward (lines 74–75) you test whether g is outside 0..255 and raise:
  if g < 0 or g > 255:
      raise ValueError("Color parameter outside of expected range: Red Green Blue")
- For inputs outside the configured bounds (for example value = -0.5 in the failing test), the formula produces g = int(-127.5) = -127, which triggers g < 0 and causes the ValueError. The tests expect values outside the bounds to be clamped to black (0) or white (255), not to raise an exception.

Root cause
- You calculate g from the raw value and then raise whenever g is outside 0..255. The code never clamps the input (or g) before this check, so out-of-range inputs lead to an exception instead of the expected clamping behavior.

Concrete fixes (exact changes to make)
1. Clamp the input value to the [self._lower_bound, self._upper_bound] range before computing g:
   - Replace the computation at line 71 with:
     value_clamped = min(max(value, self._lower_bound), self._upper_bound)
     g = int((value_clamped - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
2. Remove the raise and ensure g itself is clamped to 0..255 (defensive):
   - Replace lines 74–75 with:
     g = min(max(g, 0), 255)
   - Do not raise an exception here.
3. Keep the rest of the method returning (g, g, g).

Why this will fix the tests
- Clamping the input ensures negative values (like -0.5) become lower_bound (0.0) so g computes to 0 -> (0,0,0).
- Clamping high values ensures values > upper_bound produce 255 -> (255,255,255).
- Using the same int behavior preserves the test expectation that 0.5 yields int(127.5) -> 127.

Minimal example replacement for the relevant block
- Before (problematic):
