What happened
- The failing test called CodeConsumer.add_number(-0.0) and expected the output "-0.0", but the implementation produced "0".
- Root cause: the method converts a float that is an integer-valued float into an integer representation (e.g. by calling int(value) or using formatting that drops the fractional part). For -0.0, float.is_integer() is True and int(-0.0) yields 0 (no sign information), so the code writes "0" and the negative sign is lost.

Where it occurs
- Inside CodeConsumer.add_number (the method under test). The loss of sign happens in the branch that handles integer-valued floats (the code path that does something like if value.is_integer(): append(str(int(value))) or uses formatting that produces "0" for -0.0).
- The sign needs to be preserved for the special case negative zero; the test suite includes CodeConsumer.is_negative_zero and tests it, so that helper should be used.

Why that loses the negative sign
- Negative zero in IEEE-754 floats is equal to 0.0 by numeric equality, and converting to an int or formatting as an integer drops the negative sign because integers don't represent signed zero.
- The implementation checks “is integer” and then converts the float to an integer representation before appending, but it checks that after the numeric test that cannot distinguish -0.0 from +0.0.

Concrete fix (step-by-step)
1. In add_number(self, num) handle negative zero before any integer-conversion/formatting branch. Use the existing utility or an equivalent check:
   - The repository already has CodeConsumer.is_negative_zero(value) (tests call it), so use that.
2. If is_negative_zero(num) is True, append "-0.0" exactly and return (or skip the integer branch).
3. Otherwise proceed with the existing logic (integer-detection, scientific notation, etc.).

Minimal patch example
- Assume the method signature is add_number(self, num) and current code begins with integer handling. Insert this at the top of the method:

    # Handle negative zero explicitly before any integer conversion
    if self.is_negative_zero(num):
        self.append("-0.0")
        return

- Or, if the method uses a local variable name like value or n, apply the same check to that variable.

Why this fixes it
- It prevents the integer-conversion branch from running for -0.0, preserving the correct printed representation "-0.0".
- It’s a targeted change: it only affects the special case negative zero and leaves all other formatting logic unchanged.

Notes and alternatives
- Another way is to check sign with math.copysign(1.0, num) < 0 and num == 0.0, but since a utility is_negative_zero already exists in the class and is covered by tests, prefer using that to keep behavior centralized.
- Do not fix this by printing "-0" or "-0.00"; the test expects "-0.0". Use that exact string unless you update the tests.
