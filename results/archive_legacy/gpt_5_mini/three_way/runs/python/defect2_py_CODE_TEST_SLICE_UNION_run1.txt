What happened (root cause)
- In get_paint you clamp the input into a local variable v:
  - v = max(value, self._lower_bound)
  - v = min(v, self._upper_bound)
  (these clamping lines are just after def get_paint, around lines 54–57)
- But when you compute the gray component you use the original value variable instead of v:
  - g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
  (this is at line 71)
- Because of that, clamping has no effect. If value is outside the bounds (e.g. -0.5 or 1.5), the formula yields g outside [0,255] and the code raises the ValueError at lines 74–75. For example:
  - value = -0.5, v becomes 0.0 (clamped), but formula uses value = -0.5 → g < 0 → exception
  - value = 1.5, v becomes 1.0 (clamped), but formula uses value = 1.5 → g > 255 → exception

Where it fails in your tests
- test_lower_bound_clamping and test_upper_bound_clamping fail because the clamping is computed into v but never used in the interpolation that computes g.

Precise fix (step-by-step)
1. Use the clamped variable v when computing g:
   - change line 71 to:
     g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
2. Add a guard against division by zero in case upper_bound == lower_bound:
   - e.g.
     if self._upper_bound == self._lower_bound:
         g = 0  # or 255, or whatever makes sense for your semantics
     else:
         g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
3. (Optional) If you want different rounding behavior, consider round(...) instead of int(...). Current tests expect truncation (int), so keep as is.

Why this fix works
- Using v ensures the interpolation uses the clamped value, so g will always be within 0..255 for inputs outside the bounds and will match the test expectations. Adding the division-by-zero guard prevents a crash if someone constructs the scale with equal bounds.
