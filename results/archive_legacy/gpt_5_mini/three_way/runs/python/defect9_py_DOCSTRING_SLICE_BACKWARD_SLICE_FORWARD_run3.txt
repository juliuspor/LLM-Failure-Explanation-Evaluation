What happened (root cause)
- The code assumes elements of search_list and replacement_list are strings and calls len(...) and text.find(...) on them before it checks for None. That causes a TypeError when an element is None.
- Two places show the problem:

  1) Lines 81–82 (inside the loop starting at line 59) compute
     greater = len(replacement_list[i]) - len(search_list[i])
     If replacement_list[i] or search_list[i] is None this raises:
       TypeError: object of type 'NoneType' has no len()

  2) Line 108 calls
     temp_index = text.find(search_list[i], start)
     If search_list[i] is None this raises:
       TypeError: must be str, not NoneType

- There is a guard against bad entries at lines 102–105:
     no_more_matches_for_repl_index[i]
     or search_list[i] is None
     or len(search_list[i]) == 0
     or replacement_list[i] is None
  but that guard runs later, inside the matching loop. The earlier length calculations (lines 81–82) and any use of find (line 108) happen before the guard, so None values already cause the crash.

Where it occurs (exact places)
- Length calculation crash: inside the loop starting at line 59; specifically lines 81–82.
- find() crash: inside the match-finding loop at line 108.
- The intended protective check is at lines 102–105 but is applied too late to prevent the failures.

How to fix it (concrete steps)
- Ensure you skip any None entries (or zero-length search entries) before ever calling len(...) or text.find(...). Do this both when you pre-compute the expected increase (lines ~59–86) and when you search for matches (lines ~102–112).
