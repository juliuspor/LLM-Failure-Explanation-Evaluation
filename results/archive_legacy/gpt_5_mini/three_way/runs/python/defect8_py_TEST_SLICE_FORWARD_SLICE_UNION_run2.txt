What happened
- The failure happens in LocaleUtils.to_locale when parsing "fr__POSIX".
- At lines 237–246 the code assumes characters at indices 3 and 4 (variables ch3 and ch4) are uppercase letters for a country code:
  - lines shown:
    - 237: ch3 = locale_str[3]
    - 238: ch4 = locale_str[4]
    - 239: if ch3 < 'A' or ch3 > 'Z' or ch4 < 'A' or ch4 > 'Z':
    - 240:     raise ValueError(f"Invalid locale format: {locale_str}")
- For "fr__POSIX" the bytes/indices are:
  - index 0 = 'f', 1 = 'r' (language)
  - index 2 = '_' (separator)
  - index 3 = '_' (second underscore, indicating empty country)
  - index 4 = 'P' (start of variant)
- Because ch3 (locale_str[3]) is '_' not an uppercase letter, the check on line 239 fails and raises ValueError. That is exactly the Java-reported bug: the code does not handle the language__variant (empty country) format.

Why it happened (root cause)
- The code only has a parsing path for:
  - 2-char language only (length == 2),
  - language_country where both country letters exist,
  - language_country_variant where country exists and variant follows.
- It never handles the specific pattern where the country is empty and the input uses two underscores in a row (language__variant). The validation on ch3 and ch4 rejects '_' as an invalid country character instead of interpreting the second underscore as "empty country" and treating the remainder as the variant.

Where exactly (file and lines)
- Method: LocaleUtils.to_locale
- Problem lines: 237–240 (the ch3/ch4 extraction and validation) and the ValueError raise at line 240.
- The bad behavior leads to raising ValueError for inputs like "fr__POSIX" (the failing test test_locale_with_empty_country_variant).
