What happened (symptom)
- Tests expected get_min_start_index() to return 1 but the method returned 3. That means _min_start_index ended up holding the last processed index instead of the index with the smallest period start.

Where the bug is (file/lines you showed)
- In the code you pasted, the logic that updates self._min_start_index is wrong around lines 340–349.
  - Line 340 checks if self._min_start_index >= 0 (i.e. a minimum has already been recorded).
  - Lines 341–347 compute the current recorded minimum start (min_start) and, if the new start is earlier, set self._min_start_index = index.
  - Line 349 then executes self._min_start_index = index unconditionally.

Why it fails (root cause)
- The unconditional assignment on line 349 clobbers any previous value of self._min_start_index every time you process a data item. That means after processing all items, _min_start_index will be the last index processed (3 in your failing run) rather than the index that had the smallest start time (1).
- In short: the code tries to do a conditional update when a minimum exists, but then always overwrites it immediately afterwards.

Exactly where and how to fix it (concrete steps and patch)
- Remove the unconditional assignment or make it the else branch for the "no previous minimum" case.
- Two valid fixes:

Option A — keep the unconditional assignment only when there was no previous minimum (move it into an else):
Replace
  if self._min_start_index >= 0:
      ...
      if start < min_start:
          self._min_start_index = index
      self._min_start_index = index
with
  if self._min_start_index >= 0:
      ...
      if start < min_start:
          self._min_start_index = index
  else:
      # no previous min recorded; set it now
      self._min_start_index = index

Option B — remove the extra assignment and ensure you set _min_start_index in the correct places:
- If you already set _min_start_index when start < min_start, and you set it when there was no previous value, keep only those two assignments and delete the stray line 349.

What to check after the fix
- Run the failing test that asserted max_middle_index / get_min_start_index; it should now return the index of the earliest start (1 in your case).
- Add or update unit tests that insert several time periods in non-sorted order and assert get_min_start_index() returns the index of the earliest period.
