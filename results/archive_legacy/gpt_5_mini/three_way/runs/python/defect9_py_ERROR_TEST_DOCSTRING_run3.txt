What happened
- The tests pass None values inside the replacement list (e.g. ["c", None]) and expect the code to ignore those replacement entries.
- The implementation of StringUtils.replace_each assumes every entry in search_list / replacement_list is a str and calls len(...) (or otherwise treats it like a string) without checking for None.
- When the code hits a None replacement it executes len(replacement) (or len(search) for a None search) and Python raises TypeError: object of type 'NoneType' has no len().

Where it occurs
- Inside StringUtils.replace_each, in the loop that processes the paired items from search_list and replacement_list. Specifically the line(s) that do something like:
  - search_len = len(search_list[i])
  - replacement_len = len(replacement_list[i])
  or any operation like replacement.lower(), replacement.__contains__(), etc.
  One of those calls is executed on a None value and triggers the TypeError.

Why it fails (root cause)
- The code does not treat None as a special case. The tests intend None to mean “ignore this replacement pair” (mirror of the Java Lang-39 behavior). Instead the code treats None like a string and immediately calls len() (or other string methods) on it, which is invalid.

Concrete steps to fix (exact changes to make)
1. In StringUtils.replace_each, find the loop that iterates over the search_list / replacement_list pairs (for example an index-based loop like for i in range(len(search_list)) or a zip loop).
2. Before any use of search_list[i] or replacement_list[i], add an explicit check for None and skip that pair. Example logic:
   - repl = replacement_list[i]
   - if repl is None:
       continue
   - search = search_list[i]
   - if search is None:
       continue
   This ensures you do not call len(...) or any string methods on None.
3. If you compute lengths ahead of time, compute them only after confirming the value is not None:
   - if search is not None: search_len = len(search)
   - if repl is not None: repl_len = len(repl)
4. Ensure the skip semantics match the tests: when a replacement is None the corresponding search term must be ignored (i.e., do not alter input for that pair). Do not treat None as the empty string unless you specifically want that behavior.

Minimal example patch (pseudo-Python) — replace the risky lines with checks:
- Before:
    for i in range(len(search_list)):
        search = search_list[i]
        repl = replacement_list[i]
        search_len = len(search)            # <-- raises TypeError if search is None
        repl_len = len(repl)               # <-- raises TypeError if repl is None
        ...
- After:
    for i in range(len(search_list)):
        search = search_list[i]
        repl = replacement_list[i]
        if search is None or repl is None:
            continue                       # skip this pair
        search_len = len(search)
        repl_len = len(repl)
        ...

Why this fix meets the tests
- Skipping pairs with None replacement matches the Java behavior the tests expect (None means “ignore this replacement”), so the TypeError is avoided and the resulting string is correct.
