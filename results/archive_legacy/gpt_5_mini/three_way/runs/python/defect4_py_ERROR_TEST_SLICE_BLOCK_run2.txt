What happened (root cause)
- The failure is in TimePeriodValues._update_bounds. When updating the "max middle" tracking it uses the wrong index variable when storing the index of the new maximum.
- Concretely, the code computes the current period's middle value (middle) and correctly compares it to the stored maximum middle value, but then sets self._max_middle_index to the wrong variable (an unrelated index like self._max_start_index or some other index variable) instead of the current index (usually named i or index).
- As a result, when a later element becomes the new maximum middle the stored index is set to the wrong index, so s.get_max_middle_index() can return 3 even though the maximum middle value is at index 1.

Why the test fails (walk-through)
- Test items (indices and middles):
  - idx 0: mid=50
  - idx 1: mid=150  <- true max
  - idx 2: mid=25
  - idx 3: mid=75
- _update_bounds finds mid=150 at index 1 and updates the stored maximum middle value to 150.
- Later it processes index 3. Because of the bug it assigns the max-middle index from the wrong variable (for example reusing self._max_start_index or index of the last start comparison) so the stored index becomes 3 even though the stored maximum middle value is still 150. get_max_middle_index() therefore returns 3 (wrong).

Where it occurs in code
- Method: TimePeriodValues._update_bounds
- Variables involved: middle (computed middle value), self._max_middle (or equivalent), self._max_middle_index (should hold index of period with the max middle), and the index loop variable (commonly named i or index).
- The bad assignment is something like:
    self._max_middle_index = self._max_start_index
  or
    self._max_middle_index = some_other_index_var
  instead of:
    self._max_middle_index = index

Exact fix (step-by-step)
1. Open hit04_timeperiod.py and locate the _update_bounds method.
2. Find the branch that updates the maximum middle. It will look similar to:
     if middle > self._max_middle:
         self._max_middle = middle
         self._max_middle_index = <wrong-variable>
3. Change the assignment so the index for the new max middle is set to the current loop index (the variable used to iterate elements). For example:
     # assume the loop variable is i
     if middle > self._max_middle:
         self._max_middle = middle
         self._max_middle_index = i
4. Do the same check for the minimum-middle update: ensure you assign self._min_middle_index = i (not an unrelated index).
5. Run tests.

Minimal example patch (illustrative)
- Replace the bad line:
    self._max_middle_index = self._max_start_index
  with:
    self._max_middle_index = i
- And similarly for min:
    self._min_middle_index = i

Why this fix works
- The middle value comparison already correctly identifies whether the current period's middle is larger/smaller than the stored max/min. The only bug was storing the wrong index. Using the current loop index (i or index) records the index that produced that middle value, so get_max_middle_index() and get_min_middle_index() will return the correct indices.
