What happened (short): _update_bounds compares the computed middle against the wrong variable — the code compares middle to self.max_middle_index (an index), not to self.max_middle_value (the middle value). That makes later middles like 75 be treated as "larger than 1" (the stored index), so the method overwrites max_middle_index with the last added index (3) instead of keeping the true max (1).

Where it occurs: in TimePeriodValues._update_bounds(period, index). The bug is in the comparison/assignment logic that maintains the "max middle" bound.

Why it fails (step-by-step with variables):
- For each period you compute middle = (start + end) / 2.
- The code should compare middle to a stored numeric value self.max_middle_value and, if larger, set:
  self.max_middle_value = middle
  self.max_middle_index = index
- Instead, the buggy code compares middle to self.max_middle_index (an integer index). Example run from your test:
  - After the second addition (index=1), self.max_middle_index becomes 1 and self.max_middle_value becomes 150 (expected).
  - When index=3 has middle=75, the code checks "if middle > self.max_middle_index" → 75 > 1 is True, so it sets self.max_middle_index = 3. That overwrites the correct index 1 even though 75 < 150.
- Result: get_max_middle_index() returns 3 instead of 1, causing the AssertionError.
