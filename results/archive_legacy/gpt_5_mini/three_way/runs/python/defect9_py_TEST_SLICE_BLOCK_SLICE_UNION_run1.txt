What fails
- When you call StringUtils.replace_each("aba", ["a"], [None]) the function raises a TypeError in the initial "increase" estimation loop:
  TypeError: object of type 'NoneType' has no len()

Where it happens (exact lines)
- The exception is raised at line 82:
  81 |         for i in range(search_length):
  82 |             greater = len(replacement_list[i]) - len(search_list[i])
  83 |             if greater > 0:
  84 |                 increase += 3 * greater  # assume 3 matches

Why it happens (root cause)
- replacement_list[i] can be None (the tests pass None on purpose). The code calls len(replacement_list[i]) without checking for None first, so len(None) raises TypeError.
- The code later checks for None (see lines 102–108) when scanning the text, but that check comes after the initial loop that computes "increase". Therefore the function crashes before it reaches the null-safety logic.

Which tests trigger it
- test_null_safety_inside_arrays_single:
  result = StringUtils.replace_each("aba", ["a"], [None])
  This calls the function and hits the len(None) call at line 82.
- test_null_safety_inside_arrays_mixed can also fail for the same reason if replacement_list contains None.

What to change (concrete fix)
- Skip length calculations when either replacement_list[i] is None or search_list[i] is None or search_list[i] is empty.
- Replace the problematic block with a guarded check. For example:

  for i in range(search_length):
      if replacement_list[i] is None or search_list[i] is None or len(search_list[i]) == 0:
          continue
      greater = len(replacement_list[i]) - len(search_list[i])
      if greater > 0:
          increase += 3 * greater  # assume 3 matches

- Alternatively, treat None as an empty replacement by using safe lengths:
      repl_len = 0 if replacement_list[i] is None else len(replacement_list[i])
      search_len = 0 if search_list[i] is None else len(search_list[i])
      if search_len == 0:
          continue
      greater = repl_len - search_len
      ...

Why this fix works
- It prevents calling len(None), so the function no longer throws a TypeError during the pre-scan. It preserves the later null-safety logic (which decides not to attempt replacements for None entries) and keeps the increase estimate only for valid search/replacement pairs.

Where to apply it
- Modify the loop starting at line 81 (the "increase" estimation loop) in replace_each as shown above.
