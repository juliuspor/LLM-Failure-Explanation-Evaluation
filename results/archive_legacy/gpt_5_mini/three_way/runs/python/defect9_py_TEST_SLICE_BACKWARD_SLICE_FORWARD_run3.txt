What happened (what failed)
- The code raises an exception when a replacement entry is None. The tests call StringUtils.replace_each(..., ["a"], [None]) and the implementation tries to call len(replacement_list[i]) while replacement_list[i] is None. That triggers a TypeError: object of type 'NoneType' has no len().

Where it happens (exact lines / variables)
- At lines 81–84 (the "increase" estimation loop):
  - line 82: greater = len(replacement_list[i]) - len(search_list[i])
  - This calls len(replacement_list[i]) without checking replacement_list[i] for None.
- Later in the code (lines 102–105) there is a check that correctly looks at replacement_list[i] is None:
  - The code checks "or replacement_list[i] is None" to mark that replacement index as unavailable.
- The root cause is that the code uses replacement_list[i] before the check that handles replacement_list[i] being None.

- For test_null_safety_inside_arrays_single the replacement_list contains [None]. When the increase-estimation loop reaches i = 0 it executes len(None) → TypeError. The code never reaches the later logic that would have skipped that replacement index.
- The result is a crash instead of the intended behavior (treat None as "no replacement" / ignore it).

2. Make the checks consistent: the same None / empty checks that exist in the main search loop (lines 102–105) must be applied earlier wherever you access search_list[i] or replacement_list[i] (especially before len()).
