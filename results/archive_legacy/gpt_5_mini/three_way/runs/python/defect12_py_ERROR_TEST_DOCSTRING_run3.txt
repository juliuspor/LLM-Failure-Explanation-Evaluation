What happened
- The test calls WordUtils.abbreviate("0123456789", 15, 20, None). Here text = "0123456789" (length 10) and lower = 15.
- The implementation uses lower to index into the string without first checking whether lower is inside the string bounds. The code hits something like text[lower] or text[lower:...] and Python raises IndexError: String index out of range: 15.

Root cause (why it happens)
- There is no guard against the case lower >= len(text). The code assumes lower is always a valid index and proceeds to read text[lower] (or perform a slice that internally tries to access that position). When lower is larger than the string length, Python raises IndexError. In short: the code uses the variable lower to access string positions before validating it.

Where it occurs
- In method WordUtils.abbreviate (the function under test), at the line(s) that perform indexing or slicing using lower (for example the line that does text[lower] or uses text[lower:lower+something]). Those lines must be executed before any bounds check, so when lower == 15 for a 10-character string you get the exception.

What to change (concrete steps)
1. Validate inputs early:
   - If text is None: return None (your tests expect this).
   - If lower <= 0 and upper == -1: treat as "no limit" (existing behavior).
2. Guard against lower beyond the end of the string:
   - If lower >= len(text): return text (nothing to abbreviate).
   - Alternatively, clamp lower to len(text) if the algorithm expects an index you can still slice from (but returning the original string is the intended behavior in the test).
3. Ensure other uses of lower/upper are safe (check negative values, handle upper == -1 meaning no upper bound).
