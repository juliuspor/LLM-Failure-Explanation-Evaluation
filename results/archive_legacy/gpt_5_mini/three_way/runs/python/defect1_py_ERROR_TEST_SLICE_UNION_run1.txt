What happened (where)
- The error is raised at lines 396–397:
  ```
  396 |         if minutes_offset < 0 or minutes_offset > 59:
  397 |             raise ValueError(f"Minutes out of range: {minutes_offset}")
  ```
  That code rejects any negative minutes input immediately. The failing test called DateTimeZone.for_offset_hours_minutes(0, -15) (or other calls with negative minutes) and got ValueError: Minutes out of range: -15.

Why it fails (root cause)
- The code enforces minutes_offset to be in [0, 59] only (minutes_offset < 0 triggers the error). Tests (and Joda-Time behavior described in the docstring) expect minutes to be allowed in the range -59..+59. By rejecting negative minutes unconditionally the function throws for valid inputs like -15. The minutes-sign validation is too strict and uses the wrong bounds.

Exactly where to change
- Change the range check at line 396 so negative minutes within -59..-1 are permitted:
  - Replace the current check:
    ```
    if minutes_offset < 0 or minutes_offset > 59:
        raise ValueError(f"Minutes out of range: {minutes_offset}")
    ```
  - With a check that enforces -59 <= minutes_offset <= 59:
    ```
    if minutes_offset < -59 or minutes_offset > 59:
        raise ValueError(f"Minutes out of range: {minutes_offset}")
    ```

Additional notes about sign handling (why this alone is correct)
- The later code (lines 402–405) adjusts minutes according to the hours sign:
  ```
  402 |                 minutes_offset = hours_in_minutes - minutes_offset
  404 |                 minutes_offset = hours_in_minutes + minutes_offset
  ```
  Those branches expect minutes_offset can be negative or positive and perform the proper arithmetic to compute total offset in minutes. Allowing minutes in -59..59 therefore lets those branches compute correct results for cases like (0, -15), (-2, 15), and (-2, -15).

Suggested concrete fix (patch)
- Replace the check at line 396 with:
  ```
  if minutes_offset < -59 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")
  ```
- Run the test suite. If any mixed-sign rules should be stricter (e.g., forbidding (2, -15) per docstring), ensure those are implemented separately after validating the minutes bounds.

Summary
- Root cause: minutes validation used the wrong bounds and rejected negative minutes.
- Fix: allow minutes in range -59..59 by changing the if condition at line 396 to use -59 as the lower bound.
