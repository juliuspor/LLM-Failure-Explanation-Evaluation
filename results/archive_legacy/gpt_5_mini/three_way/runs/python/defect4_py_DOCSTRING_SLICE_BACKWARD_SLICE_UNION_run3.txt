What happened (root cause)
- At lines 346–349 you try to update self._min_start_index only when the new period’s start is earlier than the currently stored minimum. That check is:
  - if start < min_start:
      self._min_start_index = index
- But immediately after that block there is an unconditional assignment at line 349:
  - self._min_start_index = index

Because of that unconditional assignment, self._min_start_index is always set to index regardless of the comparison. In other words, the conditional update on line 346 is negated by the unconditional assignment on line 349. The result: the “minimum start index” never stays as the smallest — it always ends up pointing to the last processed index.

Where it occurs (exact lines and variables)
- Read the lines around 340–349:
  - 340: if self._min_start_index >= 0:
  - 341:     min_start_period = self.get_data_item(self._min_start_index).get_period()
  - 342:     if isinstance(min_start_period, SimpleTimePeriod):
  - 343:         min_start = min_start_period.get_start_millis()
  - 345:         min_start = int(min_start_period.get_start().timestamp() * 1000)
  - 346:     if start < min_start:
  - 347:         self._min_start_index = index
  - 349:     self._min_start_index = index
- Variables/methods involved: self._min_start_index, start, min_start, get_data_item(...).get_period(), SimpleTimePeriod.get_start_millis(), datetime.timestamp() conversion.

Why that is incorrect
- The unconditional assignment at line 349 overwrites any decision made by the comparison at lines 346–347. That makes the comparison pointless and causes incorrect behavior: the min index tracks the most recent index rather than the earliest start.

Other suspicious/duplicated code to clean up
- start is computed twice (lines 331 and 335) with different methods:
  - 331–332: if isinstance(period, SimpleTimePeriod): start = period.get_start_millis()
  - 335: start = int(period.get_start().timestamp() * 1000)
  Decide on one consistent method to get the milliseconds and remove the other duplicate.
- Similarly min_start is assigned in two ways (lines 343 and 345). Keep one.

Concrete fixes (step-by-step)
1. Remove or move the stray unconditional assignment at line 349.
   - Option A (likely intended): Move line 349 into the else branch that runs when no previous minimum exists:
     - if self._min_start_index >= 0:
         ...compare and maybe set...
       else:
         self._min_start_index = index
   - Option B: Remove line 349 entirely if the earlier logic already handles the case when _min_start_index is unset.
2. Remove duplicate start/min_start computations and keep a single, clear conversion:
   - Choose either get_start_millis() (if SimpleTimePeriod provides it reliably) or do int(period.get_start().timestamp() * 1000) for datetime objects.
   - Example: if isinstance(period, SimpleTimePeriod): start = period.get_start_millis() else: start = int(period.get_start().timestamp() * 1000)
   - Same for min_start_period.
3. Add a unit test that:
   - Inserts two items where the second has a later start, then a third with an earlier start, and asserts get_min_start_index() points to the earliest-start item.
   - This will catch regressions where index is always overwritten.

Summary
- The immediate bug is the unconditional assignment self._min_start_index = index at line 349 which overrides the conditional update at line 346–347. Fix that by removing or moving the assignment into the intended else branch. While fixing, eliminate duplicate start/min_start calculations so the logic is clear and correct.
