What happened
- Function: create_number(val: Optional[str])
- Failure trigger: calling create_number("l") or create_number("L")
- Where it fails: inside the branch that handles trailing long-qualifiers (the if last_char in ("l", "L") branch).
- Why it fails: when val is "l" or "L", numeric = val[:-1] becomes "" (empty string). The code then calls char_at(numeric, 0) without checking that numeric is non-empty. char_at (see the slice block) checks index bounds and raises IndexError when the string is empty (lines 30–31: "if index < 0 or index >= len(s): raise IndexError(...)"). That IndexError bubbles out, so the function raises IndexError ("String index out of range: 0") instead of the intended ValueError for invalid numbers.

Exactly where in the code
- last_char = val[-1] — OK for "l" / "L".
- numeric = val[:-1] — yields "" when val is one character.
- The code enters:
    if last_char in ("l", "L"):
        # ...
        if (char_at(numeric, 0) == "-" and numeric[1:].isdigit()) or numeric.isdigit():
            return int(numeric)
Because numeric is "", char_at(numeric, 0) triggers the IndexError (the char_at implementation raises on empty input at lines 30–31 shown in the slice block).

How to fix it (explicit steps)
Option A — explicitly reject single-character qualifiers before calling char_at:
- Immediately inside the "l"/"L" branch add a guard:
    if len(numeric) == 0:
        raise ValueError(f"{val} is not a valid number.")
This stops the IndexError and returns the intended ValueError.

Option B — check numeric is non-empty in the condition:
- Replace the condition with:
    if (numeric and ((char_at(numeric, 0) == "-" and numeric[1:].isdigit()) or numeric.isdigit())):
        return int(numeric)
This avoids calling char_at on an empty string.

Option C — prefer safe string checks without char_at:
- Use Python slicing / methods:
    if numeric.isdigit() or (numeric.startswith("-") and numeric[1:].isdigit()):
        return int(numeric)
This does the same validation without any explicit char_at/indexing.

Any of these fixes ensure that for inputs "l"/"L" the code raises ValueError(f"{val} is not a valid number.") instead of raising IndexError.
