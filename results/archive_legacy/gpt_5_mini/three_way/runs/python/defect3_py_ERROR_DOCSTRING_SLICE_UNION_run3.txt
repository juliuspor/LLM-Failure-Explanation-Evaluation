What happened (root cause)
- At line 35 the code checks the index against len(code_units):
  if index < 0 or index >= len(code_units):
- But code_units is created at line 12 as an empty list and has not been populated before that check (len(code_units) == 0). The caller passed index == 2, so the test raises IndexError: String index out of range: 2.
- In short: the bounds check is performed against the wrong variable (code_units) instead of the sequence that the index is meant to index, so even valid indexes for the input string look out of range.

Exactly where it occurs
- to_utf16_code_units(s: str) defines code_units = [] at line 12.
- The bounds check at line 35 uses len(code_units) and therefore uses 0 for the length.
- The IndexError is raised at line 36.

Why that is incorrect
- The index being validated is meant to refer to the input string / its UTF-16 code-unit sequence, but the code checks code_units before that list has been filled.
- Checking against an empty list produces a false out-of-range result and raises the error even when the input string has enough characters.

How to fix it (step-by-step, concrete)
Option A — if index is an index into the input string s:
1. Replace the check at line 35 with:
   if index < 0 or index >= len(s):
       raise IndexError(f"String index out of range: {index}")
2. Leave the remainder of the function that reads code units from s (or populate code_units before using it).

Option B — if index is supposed to index code_units (the function must build code_units first):
1. Move the code that populates code_units (convert s -> code_units) to run before the bounds check at line 35.
2. Keep the check but it will now use the populated code_units length:
   if index < 0 or index >= len(code_units):
       raise IndexError(f"String index out of range: {index}")

Tests to run after the fix
- Call the function with a short string and index 0 and 1 (expect no IndexError if valid).
- Call with index equal to len(s) (expect IndexError).
- Call with a string containing surrogate pairs and test an index that points to a high-surrogate to confirm the surrogate-pair handling at lines 38–44 still works.

Why this explanation is definitive
- The code shows code_units is empty at declaration (line 12) and the bounds check at line 35 therefore always treats any non-negative index as out-of-range. The fix is to check against the actual sequence length (s) or populate code_units before checking; either change directly addresses the mistaken variable used in the length test.
