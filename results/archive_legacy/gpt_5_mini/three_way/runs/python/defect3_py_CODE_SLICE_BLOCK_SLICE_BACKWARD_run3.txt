What failed
- The code throws IndexError when you run the translate loop. The exception is triggered by the bounds check
  if index < 0 or index >= len(code_units):
  raise IndexError(...)
  which is hit when code_point_at(code_units, pos) is called with pos == len(code_units).

Where it fails (exact place)
- In translate(), inside the else branch after a translator consumed codepoints you run:
  for pt in range(consumed):
      pos += char_count(code_point_at(code_units, pos))
  The call to code_point_at(code_units, pos) is what raises the IndexError when pos is equal to length (pos >= length).

Why it fails (root cause, plain)
- The loop increments pos as it iterates. On the last iteration pos can become equal to length, and then the next attempt to call code_point_at(...) uses that out-of-range pos and triggers the IndexError check.
- In short: you call code_point_at(...) without first ensuring pos < length. The code assumes there is always another code unit at pos, but that assumption is false for the last step and causes an out-of-bounds access.

Concrete fixes (step-by-step, with precise changes)
1) Decide what translate_codepoint(...) returns:
   - If it returns the number of code units consumed, you should advance pos by that many units directly.
   - If it returns the number of codepoints consumed, you still must advance pos by the equivalent number of UTF-16 code units (1 for BMP codepoints, 2 for surrogate pairs).
   Pick one of the two following concrete fixes depending on the intended contract.

2a) If translate_codepoint returns code unit count (preferred if possible):
   Replace the entire for-loop with a single addition:
     # consumed is number of code units consumed
     pos += consumed
   This avoids repeated calls to code_point_at and cannot step past the end unless consumed itself is wrong.

2b) If translate_codepoint returns codepoint count (current code's apparent intent):
   Replace the loop with code that advances pos without calling code_point_at on an out-of-range index. For example:
     for pt in range(consumed):
         if pos >= length:
             break   # or raise a clearer error if this should never happen
         cu = code_units[pos]
         # high surrogate range 0xD800..0xDBFF
         if 0xD800 <= cu <= 0xDBFF and pos + 1 < length:
             pos += 2
         else:
             pos += 1
   This computes the number of code units using the code_units array itself (no out-of-range lookup) and avoids calling code_point_at(pos) when pos==length.

3) Add defensive checks and clearer error messages
   - If consumed claims to have consumed more codepoints than remain in the input, treat that as an error in translate_codepoint and raise a clear ValueError with the offending pos and consumed value. Example:
     if pos + expected_code_units > length:
         raise ValueError(f"translate_codepoint reported consuming {consumed} codepoints starting at pos {pos}, but only {length-pos} code units remain")

Why these fixes are safe and what they prevent
- They prevent calling code_point_at(...) with pos == len(code_units), which is the immediate cause of IndexError.
- They make the advancement of pos explicit and consistent with whether consumed counts codepoints or code units.
- They add clear validation so a misbehaving translate_codepoint implementation fails with a clear message instead of an IndexError deep in a helper.

Summary (short)
- The loop calls code_point_at(code_units, pos) when pos can equal length, causing IndexError.
- Fix by advancing pos by code unit count directly (pos += consumed) if consumed is in code units, or by inspecting code_units[pos] safely (check pos < length before calling code_point_at) if consumed is in codepoints. Add validation to catch mismatch between consumed and remaining input.
