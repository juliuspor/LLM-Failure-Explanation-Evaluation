What happened (where and why)
- The exception is raised at line 396:

  -     raise ValueError(f"Minutes out of range: {minutes_offset}")
- The value printed in the error is minutes_offset = -15, so the check fails because the code sees a negative minutes value and throws the error.
- The negative minutes value was already produced earlier, before the range check. The following lines show the code later attempts to combine hours and minutes:
  - 402: minutes_offset = hours_in_minutes - minutes_offset
  - 404: minutes_offset = hours_in_minutes + minutes_offset
  - 405: offset = FieldUtils.safe_multiply(minutes_offset, DateTimeConstants.MILLIS_PER_MINUTE)
- Root cause: minutes_offset contains a signed value (negative when the parsed minutes component had a minus sign) but the validation at line 396 expects minutes_offset to be a non-negative minute quantity (0–59). The code validates minutes_offset before normalizing/combining it with the hours part, so a legitimately negative minute token like "-15" (from a string such as "-01:15") triggers the "Minutes out of range" error instead of being treated as a signed minute component.

- Input timezone string: "-01:15" (or any parsed minutes token that is negative).
- Parsed values: hours = -1, minutes = -15 (minutes_offset = -15).
- At line 396 the code checks minutes_offset < 0 and raises ValueError("Minutes out of range: -15").

Why these fixes work
