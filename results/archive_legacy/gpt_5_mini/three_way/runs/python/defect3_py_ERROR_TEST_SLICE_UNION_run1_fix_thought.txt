The bug arises from mismatched index increments when operating on UTF-16 code units. Simplest fix: iterate over UTF-16 code_units with a single while loop and carefully advance pos by the number of UTF-16 code units consumed. Translate_codepoint returns count in codepoints, so after a translator consumes n codepoints we should advance pos by the sum of char_count for each codepoint starting at current pos. However original code attempted that but used code_point_at(pos) inside loop while incrementing pos in the loop, which can lead to passing index == len. Instead compute advances without calling code_point_at with out-of-range: loop for pt in range(consumed): get cp = code_point_at(code_units, pos) then pos += char_count(cp). This is what original did; bug maybe from translate_codepoint in CsvEscaper returning 2 when consuming a surrogate pair but outer loop then increments again. Safer approach: when consumed>0, advance using code_point_at but do not call code_point_at with pos==len; ensure consumed doesn't cause overrun by breaking when pos>=length.
