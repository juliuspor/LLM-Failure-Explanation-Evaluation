- The if condition lumps several different error cases together (start < 0, end < 0, start > end, start > len(s), end > len(s)). The code always reports the value of end in the error message regardless of which sub-condition actually triggered.
- In your failing test the real problem was not necessarily that end was out of range. For example:

- In short: the conditional is correct to detect an error, but the error message points only at end (f"{end}") so it can lie about the true cause. That mismatch is why the test output looks wrong.
