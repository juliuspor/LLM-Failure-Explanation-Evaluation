What happened (what you saw)
- The code crashes when running the tests that pass None in replacement_list (e.g. ["c", None]). The failure is a TypeError coming from trying to call len() on None.

Where it happens (exact spot)
- In the loop at lines 81–84:
  81 |         for i in range(search_length):
  82 |             greater = len(replacement_list[i]) - len(search_list[i])

  84 |                 increase += 3 * greater  # assume 3 matches

Why it happens (root cause)
- The code assumes every entry replacement_list[i] (and implicitly search_list[i]) is a string and calls len(...) on it.
- When replacement_list[i] is None, len(replacement_list[i]) raises TypeError: object of type 'NoneType' has no len().
- The tests expect that None in the replacement_list means "no replacement / ignore this mapping", so the code should skip such entries instead of trying to measure their length.

What to change (concrete fix steps)
1. Before calling len(...) on either list element, check for None. Skip the mapping if search_list[i] is None or replacement_list[i] is None.
2. Keep the existing behavior for non-None entries unchanged.

Minimal patch (example)
Replace the block at lines 81–84 with:

  for i in range(search_length):
      # ignore null entries in either list (None means "no replacement")
      if search_list[i] is None or replacement_list[i] is None:
          continue
      greater = len(replacement_list[i]) - len(search_list[i])
      if greater > 0:
          increase += 3 * greater  # assume 3 matches

Why this fix works
- It prevents len(None) from being called, so no TypeError.
- It implements the intended null-safety: None entries are treated as "ignore this mapping", matching the test expectations (e.g. "aba" -> ["a"], [None] leaves "aba" unchanged; ["a","b"] -> ["c", None] yields "cbc").

Notes
- Also ensure any other places in the function that access replacement_list[i] or search_list[i] (for comparisons, slicing, concatenation) are protected the same way. The same None-safety check should be applied consistently wherever elements are used.
