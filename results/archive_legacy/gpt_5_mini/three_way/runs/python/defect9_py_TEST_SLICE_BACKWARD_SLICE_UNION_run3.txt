What failed
- The test passed replacement_list containing None (replacement_list = ["c", None]). The code calls len(replacement_list[i]) at line 82 without checking for None first, causing a TypeError: "object of type 'NoneType' has no len()".
- In short: the code tries to take len() of a None replacement too early.

Where it happens (exact lines / variables)
- At the start of replace_each, search_length = len(search_list) (line 52) is fine.
- In the loop beginning at line 81 the code does:
  greater = len(replacement_list[i]) - len(search_list[i])   (line 82)
  if greater > 0: increase += 3 * greater  (line 84)
- If replacement_list[i] is None (or search_list[i] is None), len(...) raises TypeError. The test case that fails uses replacement_list[1] == None, so the failure is triggered on i == 1.

Why the checks that appear later don't help
- Later (lines 102–108) the code already has guards that skip search items whose replacement is None or whose search string is None or empty:
    no_more_matches_for_repl_index[i]
    or search_list[i] is None
    or len(search_list[i]) == 0
    or replacement_list[i] is None
  Those checks are intended to avoid processing such items during the main search loop, but they occur after the pre-sizing loop (the loop that computes increase at lines 81–85). The code therefore fails before it reaches those later guards.

What happened (concise root cause)
- Root cause: the code assumes replacement_list[i] and search_list[i] are non-None when precomputing length differences (line 82). That assumption is false for inputs that contain None replacements (the tests). The pre-sizing code should have the same None/empty-string guards used later.

How to fix it (specific steps and exact change)
1. In the pre-sizing loop (around lines 81–85), skip items where replacement_list[i] is None or search_list[i] is None or len(search_list[i]) == 0 before calling len(...). Example change:
   - Replace the body of that loop with:
     if replacement_list[i] is None or search_list[i] is None or len(search_list[i]) == 0:
         continue
     greater = len(replacement_list[i]) - len(search_list[i])
     if greater > 0:
         increase += 3 * greater

2. Alternatively, if you prefer explicit checks:
   - Before computing greater, do:
     if replacement_list[i] is None:
         continue
     if search_list[i] is None or len(search_list[i]) == 0:
         continue

3. (Optional) If the pre-sizing logic intends to estimate space for matches, you could more accurately count actual matches in text instead of assuming 3 matches; but that's a separate improvement. The immediate fix is to avoid calling len() on None.

What to expect after the fix
- The TypeError will be avoided because you will not call len(None).
- The later guards (lines 102–108) will still prevent processing None/empty entries in the main search loop, so behavior will match the tests (for example, replace_each("aba", ["a","b"], ["c", None]) should return "cbc").
