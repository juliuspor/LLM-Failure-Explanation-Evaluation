What happened (what failed)
- The test fails with IndexError: String index out of range: 2 raised inside to_utf16_code_units at the check on line 35–36:
  if index < 0 or index >= len(code_units):
      raise IndexError(f"String index out of range: {index}")
- The index value passed in is 2, but len(code_units) is 2, so index >= len(code_units) triggers the exception.

Why it happened (root cause)
- to_utf16_code_units returns a list of UTF‑16 code units. In UTF‑16 a single Unicode code point outside the BMP (an emoji like \U0001F630) is represented as a surrogate pair — two code units.
- The translator code that iterates the string (the caller of to_utf16_code_units / code_point_at) is mixing two different kinds of indexing:
  - sometimes it treats positions as code points (Python str indexes, where that emoji is a single position),
  - other times it treats positions as UTF‑16 code unit indexes (where that emoji occupies two positions).
- Concretely: when the input contains a surrogate pair followed by another character ("😰A"), the loop advances the index incorrectly after processing the surrogate pair. That incorrect advance produces an index value (2) that is interpreted by to_utf16_code_units as a code unit index, but the code_units list length is only 2, so index == len(code_units) is out of range and raises the IndexError.
- In short: index arithmetic is inconsistent (off-by-one / off-by-one-unit) because the code confuses code point indexing with UTF‑16 code unit indexing when it handles surrogate pairs.

- Failure thrown inside to_utf16_code_units (slice shows lines ~10–12 and the check at 35–36).
- The problematic index value comes from the translator loop that calls code_point_at / to_utf16_code_units and then updates index. That loop fails to advance index consistently after encountering a surrogate pair.
