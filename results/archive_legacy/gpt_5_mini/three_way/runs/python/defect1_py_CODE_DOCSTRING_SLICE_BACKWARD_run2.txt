What happened (where it fails)
- The failure happens at lines 396–397 in for_offset_hours_minutes:
  if minutes_offset < 0 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")
- That code always rejects negative minutes. But the docstring and intended behavior allow negative minutes in many valid cases (for example (0, -15) should produce -00:15, and (-2, -15) should produce -02:15).
- Because negative minutes are rejected unconditionally, calls such as for_offset_hours_minutes(0, -15) or for_offset_hours_minutes(-2, -15) incorrectly raise "Minutes out of range".

Why it happens (root cause)
- The minutes validation is too strict: it enforces minutes_offset >= 0 always.
- The method must permit negative minutes in some combinations (hours zero or negative), and must forbid negative minutes when hours are positive. The current code does not implement that conditional rule.
- The later arithmetic (lines that compute hours_in_minutes and then add/subtract minutes_offset) expects minutes_offset to be negative in some flows; since negatives are blocked earlier, those flows never occur or behave incorrectly.

Precise fix (what to change, step-by-step)
1. Replace the single unconditional minutes check (lines 396–397) with a conditional that:
   - always rejects minutes_offset > 59,
   - rejects minutes_offset < 0 only when hours_offset > 0,
   - allows negative minutes when hours_offset <= 0.
2. Keep the rest of the method as-is so the later code can combine hours_in_minutes and minutes_offset correctly.

Exact replacement snippet (apply inside for_offset_hours_minutes, replacing the two lines at 396–397):

    if minutes_offset > 59:
        raise ValueError(f"Minutes out of range: {minutes_offset}")
    if minutes_offset < 0 and hours_offset > 0:
        # minutes negative while hours positive is explicitly invalid
        raise ValueError(f"Minutes out of range for positive hours: {minutes_offset}")

Why this fix works
- It implements the rule described in the docstring: "The minutes value must be in the range -59 to +59" and "the minutes are negative when the hours are positive" is invalid.
- It allows cases like (0, -15) and (-2, -15) that should produce negative offsets, while still rejecting invalid combos like (2, -15).

Optional extra check to consider
- The method already tries to detect overflow via FieldUtils.safe_multiply and catches ArithmeticError. If you want to be extra cautious, you can also validate that the computed total minutes magnitude does not exceed 23*60+59 before multiplying to make the error message clearer, but that's optional because the existing overflow handling is present.
