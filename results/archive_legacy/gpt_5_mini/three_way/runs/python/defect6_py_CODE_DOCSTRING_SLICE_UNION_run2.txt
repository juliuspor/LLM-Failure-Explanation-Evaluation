What happens
- On line 176 your code evaluates the whole expression left-to-right:
  if x == int(x) and not math.isinf(x) and not math.isnan(x):
  Python evaluates x == int(x) first. If x is NaN or ±Inf, the call int(x) raises (ValueError for NaN, OverflowError for ±Inf) before the math.isinf/math.isnan checks run. That is why the function crashes.

Why this is the root cause
- The intent is clearly to skip integer-conversion for NaN/Inf, but because Python evaluates the operands left-to-right and short-circuits only after an operand completes, int(x) gets called even when x is NaN/Inf. So the code never reaches the safe guards for those special floats.

Where it happens (lines from your slice)
- Line 176: the conditional order causes int(x) to run too early.
- The exception originates from int(x) at line 177 when x is NaN/Inf (ValueError or OverflowError).

How to fix it (concrete steps)
1. Check for NaN/Inf (or use math.isfinite) before calling int(x). For example replace the condition on line 176 with either:
   - if not (math.isinf(x) or math.isnan(x)) and x == int(x):
     (put the safety checks first)
   - or better, use math.isfinite:
     if math.isfinite(x) and x == int(x):
2. Optionally wrap int(x) in try/except to be defensive:
   try:
       iv = int(x)
   except (ValueError, OverflowError):
       self.add(str(x))
       return
   if x == iv:
       ...  # rest of integer handling

Minimal corrected example (replace the single if at line 176):
   if math.isfinite(x) and x == int(x):
       value = int(x)
       ...

Why that works
- math.isfinite(x) (or checking not isinf and not isnan) will be evaluated first, preventing int(x) from being called on NaN/Inf, so no exception is raised and the intended integer-path runs only for finite floats.

This directly addresses the crash (int(x) on NaN/Inf) and shows exactly where to change the code.
