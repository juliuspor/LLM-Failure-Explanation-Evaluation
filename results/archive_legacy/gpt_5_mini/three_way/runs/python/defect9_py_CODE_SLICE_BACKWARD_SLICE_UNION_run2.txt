What happened (where it fails)
- The failure happens in the loop that computes the initial "increase" estimate at lines 81–85:
  > greater = len(replacement_list[i]) - len(search_list[i])
- If any search_list[i] or replacement_list[i] is None, calling len(...) raises a TypeError ("object of type 'NoneType' has no len()").
- This happens even though later search loops (lines 59–76 and 102–113) explicitly skip None or empty entries. The increase loop does not perform the same null checks, so the function can crash early.

Why it fails (root cause)
- The code assumes every element in search_list and replacement_list is a non-None string when computing length differences, but callers are allowed to pass None elements (and the function already handles those elsewhere).
- The mismatch of checks: earlier matching code guards against None/empty entries, but the "increase" estimation does not, so len(None) is called and raises.

Exactly where (line references)
- Problem line: 82 (in your slice) — the len(...) calls:

- Context lines:
