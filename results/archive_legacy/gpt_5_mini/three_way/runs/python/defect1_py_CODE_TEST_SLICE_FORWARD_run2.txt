What happened (where it fails)
- The bug is in the try block that computes the total offset in minutes (lines shown in the slice: 402–405). Specifically these lines:
  - hours_in_minutes = hours_offset * 60
  - if hours_in_minutes < 0:
        minutes_offset = hours_in_minutes - minutes_offset
    else:
        minutes_offset = hours_in_minutes + minutes_offset
  - offset = FieldUtils.safe_multiply(minutes_offset, DateTimeConstants.MILLIS_PER_MINUTE)

- Root cause: the code reuses the parameter minutes_offset to hold the total minutes, and the sign logic is wrong when hours are negative and minutes are negative. Example:
  - Input: hours_offset = -2, minutes_offset = -15
  - hours_in_minutes = -120
  - Because hours_in_minutes < 0 the code executes minutes_offset = hours_in_minutes - minutes_offset
    -> minutes_offset = -120 - (-15) = -105
  - That yields -105 minutes (i.e. -1:45) instead of the expected -135 minutes (i.e. -2:15).
- So the method produces the wrong offset for (-2, -15). The subtraction of minutes_offset when hours are negative is incorrect: it cancels the negative sign on minutes instead of combining magnitudes.

Why that happened (why the logic is wrong)
- The code tries to handle sign by subtracting when hours are negative, but it treats the minutes input as already signed and subtracts it, which flips sign for negative minutes. For negative hours the correct effect is to extend the negative magnitude by the magnitude of minutes (i.e. subtract the absolute value of minutes), not subtract the signed minutes value.
- The code also lacks explicit validation for mixed-sign inputs (e.g. hours > 0 and minutes < 0), so some invalid combinations may be silently accepted.

Exactly what to change (concrete fixes)
1. Stop reusing minutes_offset for the total; compute a new variable total_minutes for clarity.
2. For negative hours, combine magnitudes: total_minutes = hours_in_minutes - abs(minutes_offset)
   For non-negative hours, total_minutes = hours_in_minutes + minutes_offset
3. Validate and reject incompatible mixed-sign inputs if that is required by your API. For example, if hours_offset > 0 and minutes_offset < 0 should be invalid, raise ValueError early.

Example patch (replace the try block with the following):
    try:
        hours_in_minutes = hours_offset * 60
        # Reject mixed sign where positive hours and negative minutes are not allowed
        if hours_offset > 0 and minutes_offset < 0:
            raise ValueError("Minutes must not be negative when hours are positive")
        # Compute total minutes correctly
        if hours_in_minutes < 0:
            total_minutes = hours_in_minutes - abs(minutes_offset)
        else:
            total_minutes = hours_in_minutes + minutes_offset
        offset = FieldUtils.safe_multiply(total_minutes, DateTimeConstants.MILLIS_PER_MINUTE)
    except ArithmeticError:
        raise ValueError("Offset is too large")

Why this fixes it
- For (-2, -15): hours_in_minutes = -120, total_minutes = -120 - abs(-15) = -135 (correct).
- For (-2, 15): hours_in_minutes = -120, total_minutes = -120 - abs(15) = -135 (matches the documented behavior where -2 and +15 yield -02:15).
- For (2, 0) or (2, 15): else branch gives correct positive totals.
- The explicit mixed-sign check prevents incorrect acceptance of (2, -15) if your API intends to forbid that.

Lines to update
- Update the code at the lines shown in the slice (402–405) and the associated conditional that sets minutes_offset. Add the mixed-sign validation before computing totals.
