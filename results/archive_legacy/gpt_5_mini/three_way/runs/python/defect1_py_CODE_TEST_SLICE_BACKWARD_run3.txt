What happened (where & why)
- At lines 396–397 the code rejects any negative minutes:
  if minutes_offset < 0 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")
  That makes calls like for_offset_hours_minutes(0, -15), for_offset_hours_minutes(-2, -15), and for_offset_hours_minutes(-2, 15) fail unexpectedly because the tests expect negative minute values to be allowed in some combinations.

- The arithmetic that combines hours and minutes is wrong (around the block after hours_in_minutes = hours_offset * 60). The code does:
  if hours_in_minutes < 0:
      minutes_offset = hours_in_minutes - minutes_offset
  else:
      minutes_offset = hours_in_minutes + minutes_offset
  offset = FieldUtils.safe_multiply(minutes_offset, DateTimeConstants.MILLIS_PER_MINUTE)

  Two problems here:
  1) It reuses minutes_offset as a temporary for total minutes, which is confusing.
  2) The math is incorrect for negative hours with negative minutes. Example:
     hours_offset = -2 -> hours_in_minutes = -120
     minutes_offset = -15
     Code does minutes_offset = -120 - (-15) = -105
     Expected total minutes = -120 - 15 = -135
     So subtracting minutes_offset (which can itself be negative) produces the wrong sign/magnitude.

- Even if negative minutes were allowed, the combination logic computes the wrong total when hours are negative (wrong sign handling), so offsets like (-2, -15) and (-2, 15) would produce incorrect millisecond offsets.

    if minutes_offset < -59 or minutes_offset > 59:
