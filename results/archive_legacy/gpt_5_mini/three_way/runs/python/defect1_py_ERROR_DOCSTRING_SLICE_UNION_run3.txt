- That means a call that passes minutes_offset = -15 hits this unconditional range check and fails immediately.

Why it fails (root cause)
- The docstring and later code expect negative minutes to be allowed in some cases (for example (0, -15) → -00:15, or (-2, -15) → -02:15). The code later (lines 402 and 404) adjusts minutes based on the sign of hours:

  - line 404: minutes_offset = hours_in_minutes + minutes_offset
  So the method intends to accept negative minutes in some hour-sign combinations and only reject invalid absolute values later.
- But the validation at line 396 unconditionally rejects any negative minutes (minutes_offset < 0). That validation happens before the sign-aware adjustments, so valid cases like minutes_offset = -15 are rejected too early.

Where to look in the code
- Line 396: unconditional minutes range check (minutes_offset < 0 or > 59) — this is the immediate cause of the error.
- Lines 402 and 404: sign-aware logic that expects to be able to handle negative minutes.
- Line 405 onward: computes offset and checks full offset magnitude.
