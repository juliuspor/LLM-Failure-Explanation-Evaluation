What happened (what failed)
- The add_number method treats -0.0 as an integer zero and prints "0" instead of preserving the negative sign. The failing behavior comes from the integer branch starting at line 176:
  - Line 176: `if x == int(x) and not math.isinf(x) and not math.isnan(x):`
  - Line 177–178: `value = int(x)` and `mantissa = value`
  - Later (line 192): `self.add(str(value))` (or other integer formatting) is used to output the number.
- Because -0.0 == 0 is True, `value = int(x)` becomes 0 (losing the negative sign) and the code outputs `"0"` instead of `"-0.0"`. The special-case for negative zero exists as a utility (CodeConsumer.is_negative_zero) but is not consulted here, so the sign is lost.

Why it happened (root cause)
- The condition `x == int(x)` at line 176 groups all values that compare equal to an integer into the integer-output path. That includes -0.0, because IEEE -0.0 == 0.0 yields True in Python.
- Converting to int with `value = int(x)` converts -0.0 to the integer 0 without sign information, and subsequent formatting uses `str(value)` (or other integer-based formatting) which cannot represent the negative zero sign.
- There is no early check for negative zero (for example, using CodeConsumer.is_negative_zero or math.copysign) before entering the integer branch, so -0.0 is incorrectly handled.
