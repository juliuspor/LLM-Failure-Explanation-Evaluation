What happened
- The test called DateTimeZone.for_offset_hours_minutes(0, -15) (minutes_offset = -15).
- The implementation raised ValueError: "Minutes out of range: -15".
- That error indicates the method is rejecting any negative minutes value instead of allowing minutes in the range -59..+59 per the docstring and tests.

Where it occurs
- In DateTimeZone.for_offset_hours_minutes (the factory method).
- The failing check is the minutes validation that raises when minutes_offset < 0, producing the message "Minutes out of range: %d" % minutes_offset.

Why it fails (root cause)
- The code currently treats minutes as invalid whenever minutes_offset is negative. In other words, it enforces something like:
    if minutes_offset < 0 or minutes_offset > 59:
        raise ValueError("Minutes out of range: %d" % minutes_offset)
  That unconditional reject of negative minutes contradicts the docstring and tests which allow negative minutes in many cases (for example hours == 0 and minutes < 0 should produce "-00:15"; hours < 0 and minutes < 0 or minutes > 0 are both allowed and produce a negative offset).
- The method is therefore too strict: it validates sign of minutes independently of hours instead of validating minutes in the numeric range -59..+59 and enforcing the allowed sign combinations described in the docstring.

What to change (concrete steps)
1. Replace the current minutes validation with a numeric range check that accepts -59..+59:
    if minutes_offset < -59 or minutes_offset > 59:
        raise ValueError("Minutes out of range: %d" % minutes_offset)

2. Enforce the sign combination rule from the docstring:
    - If hours_offset > 0 and minutes_offset < 0: raise ValueError (positive hours cannot have negative minutes).
    - Otherwise allow the combination (hours == 0 may have negative minutes; hours < 0 may have positive or negative minutes).

   Concretely:
    if hours_offset > 0 and minutes_offset < 0:
        raise ValueError("Positive hours with negative minutes not allowed")

3. After validation, compute the total offset in minutes (or milliseconds) correctly, making sure to combine signs as intended by the docstring:
    # Treat the sign of the hours as the primary sign; minutes augment magnitude.
    total_minutes = abs(hours_offset) * 60 + abs(minutes_offset)
    if hours_offset < 0 or (hours_offset == 0 and minutes_offset < 0):
        total_minutes = -total_minutes

   Or equivalently compute:
    sign = 1
    if hours_offset < 0 or (hours_offset == 0 and minutes_offset < 0):
        sign = -1
    total_minutes = sign * (abs(hours_offset) * 60 + abs(minutes_offset))

4. Validate the combined offset does not exceed 23:59:
    if abs(total_minutes) > (23 * 60 + 59):
        raise ValueError("Offset exceeds allowed maximum")

5. Convert to milliseconds and build the DateTimeZone instance and id string (e.g. "+02:15", "-00:15") from sign, hours, minutes.

Example of the minimal validation block to replace the failing code:
    # validate ranges
    if hours_offset < -23 or hours_offset > 23:
        raise ValueError("Hours out of range: %d" % hours_offset)
    if minutes_offset < -59 or minutes_offset > 59:
        raise ValueError("Minutes out of range: %d" % minutes_offset)

    # disallow positive hours with negative minutes
    if hours_offset > 0 and minutes_offset < 0:
        raise ValueError("Positive hours with negative minutes not allowed")

    # compute signed total minutes
    sign_negative = (hours_offset < 0) or (hours_offset == 0 and minutes_offset < 0)
    total_minutes = abs(hours_offset) * 60 + abs(minutes_offset)
    if sign_negative:
        total_minutes = -total_minutes

    if abs(total_minutes) > (23 * 60 + 59):
        raise ValueError("Resulting offset out of range")

Why this fixes the test failure
- It allows minutes_offset = -15 (within -59..59) and treats hours_offset = 0 and minutes_offset = -15 as a -00:15 offset, matching the docstring and tests.
- It still rejects invalid combinations such as (2, -15) and out-of-range values like minutes = -60.
