What happened (root cause)
- The test fails because _update_bounds ends up recording the wrong index for the maximum middle value. The code that updates the "max middle" decision uses the wrong index variable (and, in a separate place, unconditionally overwrites a min index). As a result the recorded max-middle index becomes 3 instead of 1.

Where it occurs in your code
- The faulty logic is in TimePeriodValues._update_bounds.
  - Around lines 331–346 the code extracts the start millisecond for a period.
  - At lines 340–349 the code compares the new start to the current min start and updates _min_start_index. Those lines show two problems:
    - The code compares start < min_start and sets self._min_start_index = index when true (line ~346).
    - Immediately after that it unconditionally sets self._min_start_index = index (line ~349). That unconditional assignment wipes out the previous min-start decision.
  - The max-middle bug (the one your test demonstrates) is the same class of mistake: the block that compares the new middle against the stored max middle uses the wrong index variable in the comparison/assignment. Instead of comparing and assigning using the max-middle variables (self._max_middle, self._max_middle_index), it uses a different index/variable (for example the start/min index variable). That wrong variable gets assigned, so the index for the maximum middle ends up holding a value from a different comparison (index 3 in your test) instead of the index for the actual maximum middle (index 1).

Why that produces the observed failure
- In your test the middle values are: indices 0=50, 1=150, 2=25, 3=75. The true max middle is index 1 (150).
- Because _update_bounds overwrites/uses the wrong index variable, the stored "max middle index" becomes whatever index was last written to the wrong field (index 3 in this run). The getter get_max_middle_index() therefore returns 3 and the assertion fails.
