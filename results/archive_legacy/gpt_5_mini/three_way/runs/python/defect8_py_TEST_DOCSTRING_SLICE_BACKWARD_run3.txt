What happened (where it fails)
- The failure occurs inside LocaleUtils.to_locale at the checks around lines 237–240 in your slice:
  - ch3 = locale_str[3]
  - ch4 = locale_str[4]
  - if ch3 < 'A' or ch3 > 'Z' or ch4 < 'A' or ch4 > 'Z': raise ValueError(...)
- For the input "fr__POSIX" the character at index 3 is '_' (because the format is language + '_' + '_' + variant). The code expects ch3 and ch4 to be uppercase letters for a country code and therefore fails the test ch3 < 'A' or ch3 > 'Z' (since '_' < 'A') and raises ValueError.
- In short: the method unconditionally validates positions 3 and 4 as an uppercase country code, so it rejects the "language__variant" form (empty country, double underscore) instead of recognizing that the empty country is allowed and the rest is the variant.

Why this happens (root cause)
- The code assumes that whenever a variant is present, there must be a 2-letter country code in positions 3 and 4. It never checks for the special case where the country is empty (two consecutive underscores). Because of that, the check on ch3/ch4 treats the second underscore as an invalid country letter and raises an error.
- Variable names and lines: locale_str is indexed at [3] and [4] (lines 237–239), and the ValueError is raised at line 240 when those characters are not uppercase letters.

1. Detect the empty-country case explicitly before validating ch3/ch4:
   - If locale_str[2] == '_' and locale_str[3] == '_' then:
     - set country = ""
     - set variant = locale_str[4:] (allowing variant to be any non-empty string)
   - Otherwise, proceed to validate a 2-letter country at locale_str[3:5] as the existing code intends.
2. Keep the existing validations for:
   - language length and lower-case check (unchanged)
   - country two-letter upper-case check (only when country is present)
   - separator being underscore
3. Example pseudocode patch:

       country = ""
       variant = locale_str[4:]
     else:
       ch3 = locale_str[3]
