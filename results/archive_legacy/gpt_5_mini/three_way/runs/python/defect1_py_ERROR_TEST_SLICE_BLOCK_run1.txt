What happened
- The test calls DateTimeZone.for_offset_hours_minutes(-2, -15) (and other cases that pass negative minutes).
- In the implementation, at lines 396–397 you have:

  if minutes_offset < 0 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")

- That condition rejects any negative minutes (minutes_offset < 0), so passing -15 triggers the ValueError "Minutes out of range: -15".

Why this is wrong (root cause)
- The code is using the wrong validation range for minutes. The minutes argument should be allowed in the range -59 .. +59 (or at least the absolute value should be <= 59) depending on the intended sign rules.
- The current check enforces 0..59 only, so valid negative minute inputs are incorrectly rejected.

Where it occurs
- Method: DateTimeZone.for_offset_hours_minutes
- Variable: minutes_offset
- Lines: 396–397 (the shown slice block)
