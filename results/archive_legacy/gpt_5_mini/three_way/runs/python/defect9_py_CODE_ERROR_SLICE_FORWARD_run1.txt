What happened (root cause)
- At line 76–85 you try to compute "increase" using len(replacement_list[i]) and len(search_list[i]) (your traceback shows this as the failing expression).
- If any element replacement_list[i] or search_list[i] is None, calling len(None) raises TypeError: object of type 'NoneType' has no len().
- The code later (lines 98–106 and 102–110 in the slice) already guards against None/empty elements when searching and replacing, but the earlier "increase" loop does not. In short: the "increase" loop is not null-safe while the rest of the algorithm is.

Where it fails (exact variables and lines)
- The failing expression is in the loop starting at "for i in range(search_length):" that computes:
    greater = len(replacement_list[i]) - len(search_list[i])
  (shown around line 83 in your slice).
- If replacement_list[i] is None or search_list[i] is None this causes the TypeError.

Why it happens (sequence)
1. You validate the outer inputs (text/search_list/replacement_list) are not None and that lengths match.
2. You compute search_length and allocate no_more_matches_for_repl_index.
3. You run the "increase" loop to estimate buffer size without checking for None elements inside search_list/replacement_list.
4. If any element inside those lists is None, len(...) is called on None and Python raises TypeError immediately.

How to fix it (concrete steps and exact code to change)
Make the "increase" loop apply the same null/empty checks you use later. Replace the loop at lines ~76–85 with:

    for i in range(search_length):
        s = search_list[i]
        r = replacement_list[i]
        if s is None or r is None or len(s) == 0:
            continue
        greater = len(r) - len(s)
        if greater > 0:
            increase += 3 * greater

Or equivalently, add a guard before using len(...):

    if replacement_list[i] is None or search_list[i] is None or len(search_list[i]) == 0:
        continue

Why this fix works
- It prevents calling len(...) on None by skipping any pairs where either the search string is None/empty or the replacement is None — exactly the same conditions you already use in the matching loops.
- It keeps the "increase" estimate consistent with the elements that will actually be used for replacement.

Optional (small additional suggestion)
- Consider centralizing the element validation (e.g., building a filtered list of valid (search, replacement) pairs) so every loop uses the same prefiltered data and you don’t repeat the checks in multiple places. This reduces the chance of the same bug recurring.
