What happened (root cause)
- At line 82 the code calls len(replacement_list[i]). When replacement_list contains a None entry, replacement_list[i] is None and len(None) raises TypeError: "object of type 'NoneType' has no len()".
- The loop that runs at lines 81–84 iterates from 0 to search_length and never checks whether search_list[i] or replacement_list[i] is None before using them, so the code hits None values even though the docstring says those replacements should be ignored.

Where it occurred
- In the slice you gave:
  - Line 81: for i in range(search_length):
  - Line 82: greater = len(replacement_list[i]) - len(search_list[i])
  - If replacement_list[i] is None (or search_list[i] is None), len(...) fails here and raises the TypeError.

Why this violates the intended behavior
- The docstring says "replacements should be ignored" when search_list or replacement_list contain None, but the code uses len(...) and arithmetic on those entries without exclusion. That causes a crash instead of ignoring the None pair.

Concrete fixes (explicit steps)
1. Short-term minimal fix: skip None entries immediately inside the loop before any use:
   - Add at the start of the loop (before line 82):
     if search_list[i] is None or replacement_list[i] is None:
         continue
   - This prevents calling len(...) on None and respects the "ignore None" behavior.

2. Preferred clearer fix: pre-filter pairs so the loop only sees valid (non-None) pairs:

   - Then run the existing loop over range(search_length). This keeps logic simple and ensures all downstream code can assume no None entries.

3. Additional defensive checks (optional):
