What happened (root cause)
- In get_paint (defined at line 54) you clamp the input into variable v:
  - v = max(value, self._lower_bound)
  - v = min(v, self._upper_bound)
- But when you compute g at line 71 you use the original value variable instead of the clamped v:
  - g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- If value is outside the bounds, that calculation can produce g < 0 or g > 255, which triggers the ValueError at line 74–75.
- Also, if self._upper_bound == self._lower_bound the division will raise ZeroDivisionError.

Why it fails (concise)
- You clamp into v but never use v in the normalized computation; the normalization still uses the unclamped value. That makes the clamping step ineffective and leads to out-of-range g or division-by-zero when bounds are equal.

Exact locations
- Clamping happens at lines 54–57 (v defined and clamped).
- The bad calculation happens at line 71 (uses value instead of v).
- The check and exception are at lines 74–75.
- lower/upper fields at lines 43–44 are involved in the denominator.

Concrete fixes (step-by-step)
1. Use the clamped variable v when computing g.
2. Protect against upper == lower to avoid ZeroDivisionError.
3. (Optional) Improve the exception message to include the bad g and the bounds for easier debugging.

Minimal corrected code (replace the body of get_paint):
    def get_paint(self, value: float) -> Tuple[int, int, int]:
        # Clamp value to bounds
        v = max(value, self._lower_bound)
        v = min(v, self._upper_bound)
        
        # Avoid division by zero if bounds are equal
        span = self._upper_bound - self._lower_bound
        if span == 0.0:
            g = 0  # or 255, or whatever is appropriate when bounds are degenerate
        else:
            g = int((v - self._lower_bound) / span * 255.0)
        
        if g < 0 or g > 255:
            raise ValueError(f"Color parameter outside of expected range: {g} not in [0,255]")
        
        return (g, g, g)

Why this fixes it
- The normalized value now uses v (the clamped value), so g will lie within 0–255 when bounds are valid.
- The span check prevents a ZeroDivisionError when lower_bound == upper_bound.
- The improved error message shows the actual bad value.
