What happened (where)
- The bug is in add_number(), in the block starting at line 176:
  - 176: if x == int(x) and not math.isinf(x) and not math.isnan(x):
  - 177:     value = int(x)
  - ...
  - 192:     self.add(str(value))
  - 194: else: self.add(str(x))

Why it fails (root cause)
- For x == -0.0:
  - x == int(x) is True because -0.0 == 0
  - value = int(x) sets value to 0 (an integer), losing the negative sign bit
  - execution follows the integer branch and eventually does self.add(str(value)) (line 192), producing "0"
- The code never preserves the sign bit for zero in that integer branch, so negative zero becomes "0" instead of "-0.0".
- Note: str(-0.0) would produce "-0.0", but that path (the else branch at line 194) is not taken because -0.0 satisfies x == int(x).

Exactly where to change and how (concrete fixes)
1) Preferred simple fix: detect negative zero before the integer branch and handle it explicitly.
   - Insert at the top of add_number (before line 176):
     if CodeConsumer.is_negative_zero(x):
         self.add("-0.0")
         return
   - Rationale: this preserves negative zero immediately and prevents falling into the int-path that erases sign.

2) Alternative minimal change to the integer-branch condition:
   - Change line 176 from:
       if x == int(x) and not math.isinf(x) and not math.isnan(x):
     to:
       if x == int(x) and not math.isinf(x) and not math.isnan(x) and not CodeConsumer.is_negative_zero(x):
   - Rationale: keeps all other behavior but excludes negative zero from integer formatting.

3) If you prefer using math.copysign directly (no utility):
   - Before the integer branch:
       if x == 0.0 and math.copysign(1.0, x) < 0:
           self.add("-0.0")
           return
   - Or in the condition add: and not (x == 0.0 and math.copysign(1.0, x) < 0)

Why these fixes work
- They detect the special case of negative zero (which compares equal to 0) and format it explicitly as "-0.0" instead of converting to int(0) and losing the sign.
- They are small, localized changes that preserve current behavior for all other numbers.

Extra note
- The code already has a utility CodeConsumer.is_negative_zero; use it (as suggested) to keep intent clear and avoid introducing ad-hoc sign checks.
