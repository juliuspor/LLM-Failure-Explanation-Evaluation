What happened (where)
- At line 335 the code sets inferred_type = type(array). For a Python list this makes inferred_type be the list class (builtins.list), not the element type. This is the wrong thing to infer when you want the element/component type of the array.
- At lines 337–340, when both array and element are None the code sets inferred_type = object.
- At lines 346–347 the code treats inferred_type == object as an error if expected_type != object and raises TypeError.

Why the tests fail (root cause)
- In the failing test you call ArrayUtils.add(None, None, expected_type=str).
  - array is None and element is None, so inferred_type becomes object (line 340).
  - Because inferred_type == object and expected_type != object, the code raises TypeError (lines 346–347). The test expects a list [None] without an error, so the code incorrectly rejects this valid call.
- The second bug (related) is that when array is not None the code uses type(array) (line 335). That yields the list type (list) instead of the element type (e.g., str, int). That can lead to wrong behavior in other cases where you want to infer the element type from an existing array (you should inspect the array’s element types, not the list object itself).

What went wrong conceptually
- The function tries to infer the component/element type but:
  - Confuses the container type (type(array) → list) with the element/component type.
  - Treats an unknown element type (object) as an immediate violation of expected_type instead of using expected_type as a valid inference when provided.
- As a result, valid calls like add(None, None, expected_type=str) raise when they should succeed.

Concrete steps to fix (specific lines and changes)
1. Infer element type from actual elements, not the container (change line 335):
   - If array is not None:
     - If array has any non-None elements, set inferred_type = type(first_non_none_element).
     - Else (empty array or only None elements), prefer expected_type if given; otherwise keep inferred_type = object.

2. When both array and element are None (lines 337–340):
   - If expected_type is provided, set inferred_type = expected_type instead of object.
   - Only use object as last resort.

3. Remove the error-raising behavior at lines 346–347:
   - Don’t raise TypeError simply because inferred_type is object and expected_type is not object.
   - Instead, if inferred_type is object and expected_type is given, use expected_type for allocation/semantics.
   - If you must enforce a mismatch, compare concrete inferred element type to expected_type and raise only on a real incompatible mismatch.

Suggested replacement logic (pseudocode you can paste in place of the current block around lines 335–347):

    if array is not None:
        # find first non-None element to infer element type
        inferred_type = None
        for el in array:
            if el is not None:
                inferred_type = type(el)
                break
        if inferred_type is None:
            # empty array or all-None: use expected_type if provided
            inferred_type = expected_type if expected_type is not None else object
    elif element is not None:
        inferred_type = type(element)
    else:
        # both array and element None: prefer expected_type
        inferred_type = expected_type if expected_type is not None else object

    # Do not raise just because inferred_type == object; prefer expected_type
    if inferred_type == object and expected_type is not None:
        inferred_type = expected_type

Why this fixes the failing test
- For add(None, None, expected_type=str): both array and element are None, so inferred_type becomes expected_type (str) instead of object, so no TypeError is raised and a [None] is returned as expected.
- For non-empty arrays the code now inspects elements to get the true element type instead of using type(array) which was the list class.

Notes
- In Python the runtime typing is loose, so the code mostly matters for any Java-emulation behavior (the original bug refers to ClassCastException). The changes above preserve the intended type-inference semantics used by the tests: prefer element types, fall back to expected_type when available, and only use object as a last resort.
