What happened (root cause)
- The code calls len() on items of search_list and replacement_list before it checks whether those items are None.
- Specifically, in the "increase" loop (lines shown in the slice):
  - Line 82: greater = len(replacement_list[i]) - len(search_list[i])
  - If either replacement_list[i] or search_list[i] is None, len(...) raises TypeError: object of type 'NoneType' has no len().
- Later checks at lines 102–105 correctly guard against None, but they run after the "increase" loop. Because the code measures lengths earlier without guards, the function crashes when any list contains None even though those pairs should be ignored.

Where it occurs (precise references)
- The failing call is at line 82 (inside the loop that computes "increase").
- The cause is that the code only ignores None entries later (lines 102–105), so the earlier length computation has no guard and will throw len(None).

Why this is wrong
- The function must treat None entries as "ignored replacements" everywhere it needs to examine those entries. Calling len(...) on a None breaks that contract.
- The later guard happens too late — after the code has already assumed the entries are strings and used len on them.

How to fix it (concrete steps)
1. In the loop that computes "increase" (around lines 59–85), skip any i where either search_list[i] or replacement_list[i] is None, or where search_list[i] is the empty string:
   - Example change (pseudo-replacement for the body of that loop):
     - if search_list[i] is None or replacement_list[i] is None or len(search_list[i]) == 0:
         continue
     - greater = len(replacement_list[i]) - len(search_list[i])
     - if greater > 0:
         increase += 3 * greater
2. Alternatively, if you want those entries marked as "no more matches" so other logic treats them as inactive, set no_more_matches_for_repl_index[i] = True for None entries right away.
3. Review any other code that uses len(search_list[i]) or len(replacement_list[i]) before the guard at lines 102–105 and add the same guard there to avoid future similar crashes.
4. Add unit tests that include None in search_list and replacement_list to prevent regression (the bug report already points to this case).

Why this fixes it
- By checking for None (and empty search strings) before calling len(), the function no longer tries to get the length of None objects and will correctly treat those entries as ignored, matching the intended behavior described in the docstring.
