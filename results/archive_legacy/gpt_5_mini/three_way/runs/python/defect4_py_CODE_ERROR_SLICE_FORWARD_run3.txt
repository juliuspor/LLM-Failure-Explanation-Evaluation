What happened (root cause)
- In the "max middle" branch you retrieve the wrong reference period. At the line
  max_middle_period = self.get_data_item(self._min_middle_index).get_period()
  you use self._min_middle_index instead of self._max_middle_index.
- As a result you compare the new middle value against the minimum-middle period, not the current maximum-middle period. That makes the comparison wrong and lets _max_middle_index become the new index when it shouldn’t, so tests expecting max_middle_index=1 get 3.

Where it occurs (exact location)
- Inside the block beginning with:
  if self._max_middle_index >= 0:
- The erroneous line is the first line of that block (the get_data_item(...) call described above).

Why that causes failure
- The code intends to compute max_middle from the period at _max_middle_index and update _max_middle_index only if the new middle > max_middle.
- Because you fetched the min-middle period, max_middle is computed from the wrong period and the comparison is invalid, causing incorrect updates of _max_middle_index.

How to fix (concrete steps)
1. Replace the incorrect lookup with the max index:
   Change
     max_middle_period = self.get_data_item(self._min_middle_index).get_period()
   to
     max_middle_period = self.get_data_item(self._max_middle_index).get_period()
2. Run tests to confirm the behavior is fixed.

Optional extra safety (recommended)
- Add an assertion or unit test that exercises both min and max branches for start/middle/end to prevent regressions.
- Consider extracting a helper to get start/middle/end milliseconds from a period to avoid repeating the same logic (less error-prone).

One-line patch
- Replace _min_middle_index with _max_middle_index in that get_data_item(...) call.
