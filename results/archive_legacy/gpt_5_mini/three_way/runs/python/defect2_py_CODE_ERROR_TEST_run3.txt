What happened (where, exactly)
- In get_paint you clamp the input to v on lines:
  v = max(value, self._lower_bound)
  v = min(v, self._upper_bound)
- But on the next line you compute g using the original variable value instead of the clamped v:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- So when an out-of-range input (e.g. value = -0.5 or 1.5) is passed, the formula produces g < 0 or g > 255 and your range check raises the ValueError. The clamp you intended is never used in that calculation.

Why it failed (root cause)
- The root cause is a simple variable mistake: you clamped into v but then used value. The exception is raised on the g calculation / validation line because g is computed from the un-clamped value and can fall outside 0–255.

Secondary potential issue to be aware of
- If self._upper_bound == self._lower_bound the division (self._upper_bound - self._lower_bound) would be zero and cause a ZeroDivisionError. Your code should guard against or document that case.

Exact fix (concrete steps)
1. Use the clamped variable v when computing g:
   g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
2. Add a guard for equal bounds (optional but recommended):
   if self._upper_bound == self._lower_bound:
       g = 0  # or 255, or raise a clear error depending on intended behavior
3. Keep the range check the same (or clamp g into 0..255 after converting).

Minimal corrected function
    def get_paint(self, value: float) -> Tuple[int, int, int]:
        # Clamp value to bounds
        v = max(value, self._lower_bound)
        v = min(v, self._upper_bound)
        
        # Use the clamped value v, not the original value
        denom = (self._upper_bound - self._lower_bound)
        if denom == 0:
            # choose behavior appropriate for your class; example: return black
            g = 0
        else:
            g = int((v - self._lower_bound) / denom * 255.0)
        
        if g < 0 or g > 255:
            raise ValueError("Color parameter outside of expected range: Red Green Blue")
        
        return (g, g, g)

This change makes the clamping effective and prevents the ValueError for inputs like -0.5 and 1.5.
