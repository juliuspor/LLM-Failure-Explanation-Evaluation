The bug arises because after a translator reports it consumed N codepoints/units, the loop advanced pos by re-reading codepoints at pos which may be out-of-range. The translators in this code return the number of UTF-16 code units consumed (CsvEscaper returns 2 for surrogate pair, 1 for BMP), so the simplest fix is to advance pos by consumed directly. I will replace the for-loop that incremented pos per codepoint with pos += consumed. I will preserve the initial argument checks and conversion to UTF-16 code units and the fallback when consumed == 0. No comments in the output code per instructions.
