What failed
- Root cause: get_paint raises a ValueError for values outside the bounds instead of clamping them to 0..255.
- Where it happens: get_paint at lines shown in [SLICE_UNION]:
  - Line 71 computes g:
    g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
  - Lines 74–75 then do:
    if g < 0 or g > 255:
        raise ValueError(f"Color parameter outside of expected range: Red Green Blue")
- Why that fails the test:
  - The unit tests expect out-of-range inputs to be clamped to black/white (e.g. gps.get_paint(-0.5) -> (0,0,0) and gps.get_paint(1.5) -> (255,255,255)).
  - Instead, the code raises ValueError whenever the computed g is outside 0..255, so tests that pass values < lower_bound or > upper_bound get an exception rather than the expected clamped color.

Other notes (explicit and actionable)
1. The error is not caused by the arithmetic for in-range values — the computation produces 127 for 0.5 as expected.
2. The current ValueError message is generic and not helpful (it literally contains "Red Green Blue"). If you keep raising errors, include the offending value and computed RGB in the message.
3. Also consider handling the case upper_bound == lower_bound (division by zero). The tests do not hit that, but it’s good to guard it.

How to fix it (exact steps)
Option A — clamp after computing g (recommended minimal change)
- Replace the raise at lines 74–75 with clamping code, then return the tuple:

    # compute g (existing)
    g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

    # clamp to 0..255 instead of raising
    if g < 0:
        g = 0
    elif g > 255:
        g = 255

    return (g, g, g)

Option B — clamp the input value before scaling (equivalent)
- First clamp value into [self._lower_bound, self._upper_bound], then compute g and return.

    v = min(max(value, self._lower_bound), self._upper_bound)
    g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
    return (g, g, g)

Extra safety (handle division-by-zero)
- Add an early guard in get_paint:

    if self._upper_bound == self._lower_bound:
        # arbitrarily treat equal bounds as black (or choose 255 if you prefer)
        return (0, 0, 0)

Or raise a clearer ValueError at initialization if bounds are invalid.

Summary (concise)
- Problem: Lines 74–75 raise ValueError when g is outside [0,255], but tests expect values outside bounds to be clamped to black or white.
- Fix: replace the raise with clamping (set g = 0 if g < 0; g = 255 if g > 255) or clamp the input value before computing g. Also add a guard for upper_bound == lower_bound.
