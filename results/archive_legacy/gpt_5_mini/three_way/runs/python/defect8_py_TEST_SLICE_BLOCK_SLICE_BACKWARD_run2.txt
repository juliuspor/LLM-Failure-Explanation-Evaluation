What happened (where it fails)
- The failure occurs inside LocaleUtils.to_locale, around lines 237–240. The code at those lines does:
  - ch3 = locale_str[3]
  - ch4 = locale_str[4]
  - if ch3 < 'A' or ch3 > 'Z' or ch4 < 'A' or ch4 > 'Z': raise ValueError(...)
- For the input "fr__POSIX" the characters are:
  - index 0 = 'f'
  - index 1 = 'r'
  - index 2 = '_'
  - index 3 = '_'   <-- ch3 = '_'
  - index 4 = 'P'   <-- ch4 = 'P'
- Because ch3 is '_' (not an uppercase letter), the check at line 239 triggers and a ValueError is raised ("Invalid locale format: fr__POSIX").

Why that check is wrong (root cause)
- The code assumes that if there is more than a language, positions 3 and 4 must always be uppercase country letters. That assumption does not hold for the "language__variant" form (language, empty country, variant), where there are two underscores in a row at indices 2 and 3 and the variant begins at index 4.
- In other words, the code fails to detect and handle the empty-country case (double underscore). It treats the second underscore as an invalid country letter and raises an error.

Exact spot to fix
- The problematic validation is at lines 237–240 (the ch3/ch4 read and the uppercase check). The code needs an explicit branch for the empty-country-with-variant case before enforcing that ch3/ch4 are uppercase letters.
