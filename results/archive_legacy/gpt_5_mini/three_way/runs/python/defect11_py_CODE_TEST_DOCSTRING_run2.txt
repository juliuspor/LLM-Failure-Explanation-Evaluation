What fails
- The test calls RandomStringUtils.random(3, 5, 5, False, False). In the function you compute gap = end - start and then raise ValueError("bound must be positive") when gap <= 0.
- That ValueError message does not mention start or end, so the test that asserts the exception message contains "start" and "end" fails.

Where it happens (lines / variables)
- In random(...): after computing gap = end - start you have:
  if gap <= 0: raise ValueError("bound must be positive")
- For the failing input start == 5 and end == 5, gap == 0 and that branch is taken.

Why it happens (root cause)
- The function correctly detects an invalid range when end <= start, but it raises a generic message ("bound must be positive") that mirrors Random#nextInt instead of describing the parameters of this function.
- The test expects the exception message to include the names/values of the parameters ("start" and "end"), so the message content is the root cause of the failure — not the presence/absence of the exception itself.

How to fix it (exact steps)
1. Replace the generic raise line with a message that includes start and end. For example:
   if gap <= 0:
       raise ValueError(f"end ({end}) must be greater than start ({start})")
   This produces a message containing both "start" and "end".
2. Alternatively, if you want to keep wording closer to the Java test expectation, include the words "start" and "end" explicitly:
   if gap <= 0:
       raise ValueError(f"start ({start}) and end ({end}) must define a non-empty range")
3. Run the test suite; the test that checks the exception message will pass because the message now contains the substrings "start" and "end".

Notes
- The special-case block that sets start/end when both are zero does not run for start=5,end=5, so it is irrelevant to this failing case.
- No other logic needs to change; only the exception message wording should be adjusted so it mentions start and end.
