What happened (where it failed)
- The ValueError comes from the minutes-range check in for_offset_hours_minutes:
  if minutes_offset < 0 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset})
- That check runs before any combination of hours and minutes. So calling for_offset_hours_minutes(0, -15) fails immediately with "Minutes out of range: -15" even though the docstring explicitly allows negative minutes when hours == 0.

Why it fails (root cause)
- The code only allows minutes in 0..59, but the docstring and intended behavior permit minutes in -59..+59 (some combinations are later rejected). The unconditional minutes_offset < 0 test is too strict and rejects valid inputs such as (0, -15) or (-2, -15).
- The later logic already expects minutes_offset to be able to be negative (it adds or subtracts minutes_offset from hours_in_minutes). So rejecting negative minutes up-front contradicts the later combination logic.

Exactly where to change (lines / variables)
1. Change the minutes range check to allow negative minutes within -59..59:
   - Replace:
       if minutes_offset < 0 or minutes_offset > 59:
           raise ValueError(...)
     With:
       if minutes_offset < -59 or minutes_offset > 59:
           raise ValueError(...)
   This fixes the immediate rejection of e.g. minutes_offset == -15.

2. Enforce the sign-combination rule the docstring requires (positive hours with negative minutes is invalid):
   - After the revised range check, add:
       if hours_offset > 0 and minutes_offset < 0:
           raise ValueError("Minutes must be non-negative when hours is positive")
   - Optionally also enforce that when hours_offset < 0 and minutes_offset > 0 that the result is interpreted as negative; the existing combining logic already handles (-2, 15) correctly, so no extra check is strictly required for that case unless you want to explicitly forbid mismatched signs beyond what's documented.

Why those changes fix it
- Allowing minutes in -59..59 prevents valid inputs like (0, -15) from being rejected too early.
- Adding the explicit check for (hours > 0 and minutes < 0) enforces the docstring rule that combinations like (2, -15) are invalid, while leaving valid combinations (0, -15), (-2, -15), and (-2, 15) intact. The subsequent combination code (hours_in_minutes +/- minutes_offset) will then produce the intended offset in minutes.

Minimal corrected code fragment
- Replace the two checks with:

    if minutes_offset < -59 or minutes_offset > 59:
        raise ValueError(f"Minutes out of range: {minutes_offset}")
    if hours_offset > 0 and minutes_offset < 0:
        raise ValueError("Minutes must be non-negative when hours is positive")

This is the specific, minimal set of changes required to stop the ValueError for (0, -15) while still enforcing the docstring's sign rules.
