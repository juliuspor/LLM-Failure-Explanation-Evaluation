What happened (root cause)
- The code crashes inside the "guess result buffer size" loop (lines 83–85). That loop does:
  > greater = len(replacement_list[i]) - len(search_list[i])

  If either replacement_list[i] or search_list[i] is None, calling len(...) raises a TypeError. The docstring says None entries should be ignored, but this loop does not ignore them.

Where it occurs (exact spots)
- The unsafe loop is at lines 83–85:
  - It indexes replacement_list[i] and search_list[i] and calls len(...) without any None checks.
- The code does perform proper null/empty checks later when searching for matches (lines 102–105 and 108–113), so the crash happens before those checks are used to skip None entries.
- There are other places that rely on replace_index pointing to a non-None entry (lines 92 and 94: parts.append(replacement_list[replace_index]) and start = text_index + len(search_list[replace_index])). Those are safe only because earlier search skipped None entries. The real bug is the earlier increase loop not being null-safe.

Why this causes the observed failure
- If either list contains None anywhere, the initial increase loop attempts len(None) and Python raises TypeError: object of type 'NoneType' has no len(). That matches the reported behaviour: "crashes if search_list/replacement_list contain None, even though those replacements should be ignored."

How to fix it (clear actionable steps)
1. Make the initial "increase" loop null-safe by skipping entries that the later code already skips. Replace the loop body (lines 83–85) with the same guard used elsewhere:
   - for i in range(search_length):
       if (
         no_more_matches_for_repl_index[i]
         or search_list[i] is None
         or len(search_list[i]) == 0
         or replacement_list[i] is None
       ):
         continue
       greater = len(replacement_list[i]) - len(search_list[i])
       if greater > 0:
         increase += 3 * greater
2. Alternatively, explicitly check for None before calling len():
   - if search_list[i] is None or replacement_list[i] is None: continue
   - if len(search_list[i]) == 0: continue
3. (Optional) Since increase is only a heuristic, you can simplify: compute increase using only entries where both strings are non-None and non-empty, or remove the heuristic entirely if not needed.

Why this fix is correct
- It makes the initial sizing heuristic use exactly the same criteria as the match-finding logic (so entries intended to be ignored are ignored everywhere).
- It prevents calling len(...) on None and thus prevents the TypeError crash.
