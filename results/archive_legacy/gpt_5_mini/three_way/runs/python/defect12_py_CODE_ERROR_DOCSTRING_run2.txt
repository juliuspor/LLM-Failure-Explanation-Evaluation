What happened (what failed and where)
- The call that throws is:
  index = StringUtils.index_of(text, " ", lower)
  with lower = 15 and text = "0123456789" (length 10).
- StringUtils.index_of delegates to a string-index routine that uses the given start position; passing start > len(text) makes that routine raise IndexError (String index out of range: 15).
- In other words, the function never checks that lower is within the valid range before calling index_of, so index_of gets an out-of-range start index and throws.

Why it happened (root cause)
- The code adjusts upper but does not validate or clamp lower.
- The specific lines involved:
  - the clamping of upper:
      if upper == -1 or upper > len(text):
          upper = len(text)
      if upper < lower:
          upper = lower
    These do not protect the later index_of call against lower > len(text).
  - the failing line:
      index = StringUtils.index_of(text, " ", lower)
    If lower > len(text), index_of raises an IndexError.
- The bug manifests for inputs where lower > len(text) (example from the docstring: abbreviate("0123456789", 15, 20, None)).

How to fix it (concrete steps)
Option A — clamp lower before calling index_of (preferred minimal change)
- Right before the index_of call, ensure lower is at most len(text):
    if lower > len(text):
        lower = len(text)
  This prevents index_of from being called with an out-of-range start.

Option B — return early when lower is beyond the string (explicit behavior)
- If you want to treat lower > len(text) as "nothing to abbreviate", return a sensible result. For example:
    if lower >= len(text):
        # no word break possible after `lower`; return text truncated to upper (if needed)
        result = substring_java(text, 0, upper)
        if upper != len(text):
            result += StringUtils.default_string(append_to_end)
        return result
  This makes behavior explicit and avoids calling index_of at all.

Exact small patch (insert these lines before index_of):
    # ensure `lower` is within the string bounds before searching for a space
    if lower >= len(text):
        # nothing to search beyond the end; truncate and append if needed
        result = substring_java(text, 0, upper)
        if upper != len(text):
            result += StringUtils.default_string(append_to_end)
        return result

Why this fix works
- It prevents passing an out-of-range start to StringUtils.index_of, so no IndexError is raised.
- It also produces a deterministic, non-throwing result for the documented problematic case (lower > len(text)).

Optional note on behavior choice
- If you'd rather treat lower values beyond the string as "use the end of the string" for word-boundary search, use the clamp lower = min(lower, len(text)) approach instead of returning early. Either fix prevents the IndexError; choose the one that matches the intended contract for abbreviate.
