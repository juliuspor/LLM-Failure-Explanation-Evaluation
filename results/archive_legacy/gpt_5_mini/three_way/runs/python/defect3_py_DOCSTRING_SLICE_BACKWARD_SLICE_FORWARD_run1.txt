What happens
- At line 12 you create an empty list: code_units = [].
- At line 35 you check index against len(code_units): if index < 0 or index >= len(code_units): raise IndexError(...)
- The code then reads code_units[index] at line 38.

Because code_units is still empty at the time of the check, len(code_units) == 0. Any non‑negative index will be >= 0 and thus trigger the IndexError on line 35, even when the index is valid for the input string s. In short: the function is validating an index against the wrong, still-empty list (code_units) instead of the data that should be there.

Why this is the root cause
- The check uses code_units (line 35) but code_units hasn't been populated yet (line 12 shows it was just created).
- The later code (lines 38–44) expects code_units to contain UTF‑16 code units so it can read high and low surrogates. Because code_units was never built before the check, the check always fails and raises IndexError prematurely.

Where it occurs
- Initialization: line 12 (code_units = []).
- Faulty index validation: line 35 (if index < 0 or index >= len(code_units): ...).
- Subsequent access that assumes code_units is populated: line 38 (high = code_units[index]) and following lines (39–44).

How to fix it (concrete steps)
Choose one of these fixes depending on intended behavior:

1) If the index should be validated against the UTF‑16 code units list:
   - Build/populate code_units before doing the index check.
   - Move the index validation (lines ~35) to after the code that fills code_units.
   Example:
     - populate code_units from s
     - then run: if index < 0 or index >= len(code_units): raise IndexError(...)

2) If the index should be validated against the input string s (not code_units):
   - Replace len(code_units) with len(s) in the check.
   - Ensure the rest of the function consistently uses the same representation.
   Example change at line 35:
     if index < 0 or index >= len(s):
         raise IndexError(f"String index out of range: {index}")

3) If both lists exist and you accidentally used the wrong variable name:
   - Rename variables or use the correct one (e.g., code_units_built vs code_units_in_progress) so the check references the list you expect.
