What failed
- At lines 396–397 the code rejects any negative minutes:
  if minutes_offset < 0 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")
  This contradicts the docstring which allows minutes to be negative when hours are zero or negative (range should be -59..+59). As written, calls such as DateTimeZone.for_offset_hours_minutes(-2, -15) will be rejected at this check even though the doc says (-2, -15) should produce -02:15.

- The logic that builds the total minute offset is wrong when hours are negative. The code does:
    hours_in_minutes = hours_offset * 60
    if hours_in_minutes < 0:
        minutes_offset = hours_in_minutes - minutes_offset
    else:
        minutes_offset = hours_in_minutes + minutes_offset
  Suppose hours_offset == -2 and minutes_offset == -15:
    hours_in_minutes = -120
    minutes_offset = -120 - (-15) = -105
  That yields -01:45 (-105 minutes), but the expected combined offset is -02:15 (-135 minutes). In short, for negative hours the code subtracts minutes_offset, which has the wrong sign handling. The combination formula is incorrect for the negative-hours cases shown in the docstring.

Where it occurs (lines / variables)
- Lines 396–397: minutes validation using minutes_offset rejects negative values.
- The combination block (lines after hours_in_minutes assignment): incorrect use of minutes_offset and hours_in_minutes produces wrong total minutes when hours_in_minutes < 0.
- Variables involved: hours_offset, minutes_offset, hours_in_minutes, offset.

Why it happens (root cause)
1) Validation mistake: minutes_offset is constrained to [0,59] instead of [-59,59], so valid negative-minute inputs (per doc) are rejected early.
2) Sign-handling mistake: the code tries to compose total minutes by subtracting minutes_offset when hours are negative, but that subtraction double-negates negative minute inputs and leads to the wrong total. The code does not implement the intention captured by the doc's table (for negative hours the minutes should always add to the magnitude of the negative offset).

Concrete fixes (step-by-step)
1) Fix the minutes range check:
   - Replace
       if minutes_offset < 0 or minutes_offset > 59:
   - With a check that allows -59..+59, e.g.
       if minutes_offset < -59 or minutes_offset > 59:
   (This lets negative minutes through when appropriate.)

2) Enforce the rule "positive hours cannot be combined with negative minutes" (per the doc):
   - After the hours range check add:
       if hours_offset > 0 and minutes_offset < 0:
           raise ValueError("Minutes must be non-negative when hours are positive")

3) Correct the composition of total minutes (clear, correct logic):
   - Compute hours_in_minutes = hours_offset * 60
   - Then:
       if hours_offset < 0:
           total_minutes = hours_in_minutes - abs(minutes_offset)
       else:
           total_minutes = hours_in_minutes + minutes_offset
     Note: If hours_offset == 0, this correctly yields total_minutes == minutes_offset (including negative minutes).
   - Then compute offset = FieldUtils.safe_multiply(total_minutes, DateTimeConstants.MILLIS_PER_MINUTE)

4) Keep the existing try/except for overflow around safe_multiply.

Why these fixes work
- Changing the minutes range to [-59,59] matches the doc and allows negative minute inputs where permitted.
- Explicitly rejecting (hours > 0 and minutes < 0) enforces the doc's disallowed combination.
- Using hours_in_minutes - abs(minutes_offset) for negative hours handles both (-2, 15) and (-2, -15) correctly:
    - (-2, 15): -120 - 15 = -135 => -02:15
    - (-2, -15): -120 - 15 = -135 => -02:15
  And for positive hours the simple addition gives the expected +02:15 for (2,15).
- This eliminates the double-negation bug and matches every example in the docstring.

Example corrected code sketch (replace the problematic block)
    if minutes_offset < -59 or minutes_offset > 59:
        raise ValueError(f"Minutes out of range: {minutes_offset}")
    if hours_offset > 0 and minutes_offset < 0:
        raise ValueError("Minutes must be non-negative when hours are positive")
    try:
        hours_in_minutes = hours_offset * 60
        if hours_offset < 0:
            total_minutes = hours_in_minutes - abs(minutes_offset)
        else:
            total_minutes = hours_in_minutes + minutes_offset
        offset = FieldUtils.safe_multiply(total_minutes, DateTimeConstants.MILLIS_PER_MINUTE)
    except ArithmeticError:
        raise ValueError("Offset is too large")

This resolves both the validation and the incorrect arithmetic causing the failure.
