What went wrong (in plain terms)

1) Wrong thing inferred from array:
- On line
      if array is not None:
          inferred_type = type(array)
  the code uses type(array). For a Python list this returns the list class (list), not the element/component type.
  So whenever array is non-None you end up with inferred_type == list (the container type) instead of a useful element type (e.g. str or int). That is the root source of many wrong behaviors later.

2) Bad handling of the fully-unknown case (array is None and element is None):
- When both array and element are None the code sets
      inferred_type = object
  and later, on these lines:
      if expected_type is not None:
          if inferred_type == object and expected_type != object:
              raise TypeError(...)
  it always raises for the test case ArrayUtils.add(None, None, expected_type=str). The test expects you to return [None] (a list containing a None) without raising. The code instead treats "object" as a hard failure when an expected_type was given, which is incorrect: if no information is available you should use expected_type (or allow creation of an Object/None list), not raise immediately.

Where it happens (lines/variables)
- inferred_type is set incorrectly at the top (the if/elif/else that assigns inferred_type).
- new_list = ArrayUtils._copy_list_grow1(array, inferred_type) is passed the wrong inferred_type (list or object) instead of an element type.
- The TypeError is raised in the expected_type check block (the nested if that raises when inferred_type == object and expected_type != object).

Why that causes the failing test
- test_add_null_to_null_with_expected_type calls ArrayUtils.add(None, None, expected_type=str). Because both array and element are None, inferred_type becomes object. The code sees expected_type is str and inferred_type == object and raises TypeError (ClassCastException message), so the test fails instead of returning [None].

Correct/clear fixes (step by step)

1) Infer element type properly from a non-None array:
- If array is not None, do NOT use type(array). Instead scan the array for the first non-None element and use its type:
    - for elt in array: if elt is not None: inferred_type = type(elt); break
    - if no non-None element found, leave inferred_type unknown (None or object) and fall through to use element or expected_type.

2) Use expected_type to resolve unknowns instead of raising immediately:
- If both array and element give no information (all None), but expected_type is provided, treat inferred_type = expected_type (or at least do not raise on that basis).
- Only raise a TypeError if you have a concrete inferred_type (from array elements or element) that is incompatible with expected_type (e.g., inferred_type is not a subclass of expected_type). For Python, you can check compatibility with issubclass(inferred_type, expected_type) when inferred_type is a class.

3) Order of precedence for deciding component type (explicit steps):
- If array is not None:
    - find first non-None element in array; if found, inferred_type = type(that_element)
    - else if expected_type is not None: inferred_type = expected_type
    - else if element is not None: inferred_type = type(element)
    - else inferred_type = object
- Else (array is None):
    - if element is not None: inferred_type = type(element)
    - elif expected_type is not None: inferred_type = expected_type
    - else inferred_type = object

4) When checking expected_type, only raise when you actually have a concrete inferred_type that contradicts expected_type:
- if expected_type is not None and inferred_type is not object:
    - if not issubclass(inferred_type, expected_type):
         raise TypeError(...)
  (If inferred_type == object you should accept expected_type and proceed; do not preemptively raise.)
