What happened
- The bug is in the pre-sizing loop at lines 81–85:
  greater = len(replacement_list[i]) - len(search_list[i])
  if greater > 0:
      increase += 3 * greater  # assume 3 matches
- That code calls len(...) on both replacement_list[i] and search_list[i] without checking for None.
- If either array contains None at index i, calling len(None) raises TypeError: "'NoneType' object has no len()".
- The rest of the function explicitly skips None or empty entries when searching (see the checks at lines 59–68 and again at 102–109), but this pre-sizing loop does not, so the function crashes before the matching logic runs.

Where it occurs
- Method: replace_each
- Faulty lines: the loop starting at line 81 (the "BUG" comment), specifically the len(...) calls on replacement_list[i] and search_list[i].
- Symptoms: calling replace_each with search_list or replacement_list containing None entries causes a TypeError.

Why it happens (root cause)
- The function intends to ignore None entries (and empty search strings), but the pre-sizing step assumes all entries are non-null and indexable. That inconsistent null-safety causes the crash.

How to fix it (concrete steps)
1. In the pre-sizing loop, skip indices where either entry is None or the search string is empty, using the same condition used elsewhere:
   if (no_more_matches_for_repl_index[i]
       or search_list[i] is None
       or len(search_list[i]) == 0
       or replacement_list[i] is None):
       continue
2. Only compute lengths when both values are non-None:
   greater = len(replacement_list[i]) - len(search_list[i])
   if greater > 0:
       increase += 3 * greater
3. Optionally remove the unused/inaccurate "increase" estimate entirely (it's only a guess), or make it robust by using safe length extraction:
   s_len = len(search_list[i]) if search_list[i] is not None else 0
   r_len = len(replacement_list[i]) if replacement_list[i] is not None else 0
   greater = r_len - s_len
