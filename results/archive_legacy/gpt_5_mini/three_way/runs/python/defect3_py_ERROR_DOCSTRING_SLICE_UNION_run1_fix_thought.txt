The bug arises because translate treats pos and consumed in terms of codepoints vs UTF-16 code units. We should compute consumed as number of codepoints (as contract says) but advance pos by number of code units consumed. The current loop after consumed!=0 incorrectly increments pos by char_count(code_point_at(code_units,pos)) inside loop for pt in range(consumed), repeatedly recalculating at same pos leading to infinite or wrong increments and possibly index errors. Instead, when consumed>0, for i in range(consumed): cp = code_point_at(code_units,pos); pos += char_count(cp). Also need to ensure when translate_codepoint returns 1 or 2 that it may have already appended output; if consumed==0 we handle as before. Implemented accordingly.
