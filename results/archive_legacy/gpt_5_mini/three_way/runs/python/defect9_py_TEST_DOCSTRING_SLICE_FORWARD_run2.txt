What happened (what failed)
- At line 92 the code does: parts.append(replacement_list[replace_index]).
- If replacement_list[replace_index] is None this appends a Python None into parts.
- Later the code concatenates the pieces (parts) back into a string. Joining a list that contains None raises a TypeError (or otherwise produces wrong output), so the test crashes instead of treating the None replacement as “ignore this replacement”.

Why it happened (root cause)
- The check that tries to treat None replacements as “no more matches” appears in the search loop (lines 102–105) where the code sets no_more_matches_for_repl_index[i] when replacement_list[i] is None.
- That check comes too late relative to where matches are used: once replace_index is selected, the code immediately uses replacement_list[replace_index] at line 92 without checking whether that element is None.
- In short: the algorithm can select a replace_index whose replacement is None and then unconditionally append that None to parts. The “treat None as ignore” logic does not prevent using that index early enough.

Where it occurs (exact spots)
- Selection/use mismatch:
  - Use: parts.append(replacement_list[replace_index]) at line 92.
  - Update start: start = text_index + len(search_list[replace_index]) at line 94 (also assumes search_list[replace_index] is valid).
  - The guard that should have prevented selecting indices with None is at lines 102–105:
    - it checks "or replacement_list[i] is None" there, but only inside the loop that finds the next match. That is too late to prevent the append already performed.
- Because parts now contains None, the final parts.append(text[start:]) at line 115 and the subsequent join fail.

How to fix it (concrete steps)
1. Prevent selecting a replace_index with a None replacement before using it:
   - When choosing the next replace_index (the loop at lines ~102–113), ensure you skip any i where replacement_list[i] is None (i.e., set no_more_matches_for_repl_index[i] = True before comparing temp_index with text_index). That way replace_index will never point at an element whose replacement is None.
   - Concretely: move the "or replacement_list[i] is None" condition so it is evaluated before considering temp_index, or set no_more_matches_for_repl_index[i] = True when replacement_list[i] is None as part of initialization.

2. Or, guard at use time (line 92):
   - Before appending, check if replacement_list[replace_index] is None. If it is, append the original matched substring (search_list[replace_index]) instead of None — this matches the intended semantics of “ignore None replacements” (leave original text).
   - Example behavior:
     - if replacement_list[replace_index] is None:
         parts.append(search_list[replace_index])
       else:
         parts.append(replacement_list[replace_index])

Either approach works; the first prevents ever choosing an index with a None replacement, the second ensures you never put None into parts. The important thing is to stop appending None (line 92) and to ensure the search/selection logic and the use of replace_index agree about which replacements are valid.

Why this fix matches intention
- The docstring and tests expect that a None in replacement_list is ignored (treats the original substring as left in place), not that the code crashes. The above fixes make the code either never choose None replacements or substitute the original substring at use time, restoring the intended behavior.
