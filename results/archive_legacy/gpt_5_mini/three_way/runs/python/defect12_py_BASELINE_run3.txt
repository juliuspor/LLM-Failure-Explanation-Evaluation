What happened (what failed)
- The function raises IndexError because substring_java(text, 0, upper) is called with upper > len(text).
- In your failing test, text = "0123456789" (length 10), lower = 15, upper = 20. The exception message shows end = 15 (the bad value passed to substring_java).

Why it happens (root cause, with line references)
- You first clamp upper to the text length:
  if upper == -1 or upper > len(text):
      upper = len(text)
  After this, upper == 10 (len(text)).
- Immediately after you run:
  if upper < lower:
      upper = lower
  This sets upper = 15 because upper (10) < lower (15). That reintroduces a value larger than the string length.
- Later you call substring_java(text, 0, upper) (in the branch where index == -1 or index > upper), which calls your slice function with end == 15. The slice helper checks end > len(s) and raises IndexError(f"String index out of range: {end}") — exactly the error seen.

Where it occurs
- The faulty logic is the pair of checks that adjust upper relative to len(text) and lower:
  - the first block (lines shown) clamps upper down to len(text)
  - the next block (if upper < lower: upper = lower) undoes that when lower > len(text)
- So the problematic line is the one that sets upper = lower without ensuring lower is within the text bounds.

How to fix (concrete options and exact changes)
You must ensure you never pass an end index greater than len(text). Two clear fixes:

Option A — Return the original text if lower >= len(text)
- Replace the "if upper < lower" block with:
  if lower >= len(text):
      return text
  if upper < lower:
      upper = lower
- This matches the intended behavior in the Java bug report: if the starting lower position is beyond the end, just return the original string.

Option B — Clamp lower to the text length before comparing
- At the top, after handling text == None / empty, add:
  if lower > len(text):
      lower = len(text)
- Then your existing logic can stay:
  if upper == -1 or upper > len(text):
      upper = len(text)
  if upper < lower:
      upper = lower
- This prevents upper being set to a value > len(text).

Which to choose
- Option A mirrors the documented expected behavior for the Java bug case (test expects original text when lower is beyond length).
- Option B is safer if you prefer to treat lower as "clamped" instead of "out of range".

Minimal concrete patch (Option A, replace the two lines that follow upper clamping)
Change:
  if upper < lower:
      upper = lower
To:
  if lower >= len(text):
      return text
  if upper < lower:
      upper = lower

That prevents upper being set to 15 in your failing test and fixes the IndexError.
