What happened (root cause)
- On line "if array is not None: inferred_type = type(array)" you call type(array). That returns the list class (built-in list), not the element/component type of the list.
- ArrayUtils._copy_list_grow1 expects a component/element type (for example str, int, or object), not the list class. Passing the list class or object when you actually meant an element type makes the later type-check raise the TypeError you saw.
- In the failing test inferred_type ends up as object (the code falls back to object when both array and element are None), and expected_type is str. The code then reaches the check:
    if inferred_type == object and expected_type != object:
        raise TypeError(...)
  which raises the TypeError you saw: "Cannot cast object list to str list ..."

Exactly where it went wrong (lines in your snippet)
1. inferred_type = type(array) — this uses the list object's type (list) instead of the element type.
2. The fallback inferred_type = object only happens when both array and element are None; that leads to the explicit TypeError later if an expected_type was supplied.
3. new_list = ArrayUtils._copy_list_grow1(array, inferred_type) — this receives the wrong type value because of (1)/(2), so the produced list's component type is incorrect and the check raises in the subsequent if block.

Why that produces the observed error
- If both array and element are None (or if array is empty and element is None), your code sets inferred_type = object. If expected_type is provided and not object (e.g. str), you explicitly raise TypeError because you claim you cannot convert an object-typed list to a str-typed list. That exactly matches the test error message.
- If array is not None but you used type(array), you won't be passing the element type to _copy_list_grow1; that will also produce wrong behavior even when array contains elements.

Concrete fixes (step-by-step)
1. When array is not None and not empty, infer the element type from the array contents, not from the array object:
   - Use type(array[0]) (or better: find a common element type or use expected_type if elements are None/mixed).
   - Example: if array and len(array) > 0: inferred_type = type(array[0])
2. If array is not None but empty, and element is None, prefer expected_type (if supplied) over object:
   - elif element is None and expected_type is not None: inferred_type = expected_type
   - else: inferred_type = object
3. If element is not None and array is None, use type(element) as you already do.
4. Remove or adjust the final TypeError condition: only raise when you truly cannot produce a list of the expected_type. For example, if inferred_type is object but expected_type is provided, you can set inferred_type = expected_type before creating the new list (if that behavior matches your API), or keep the raise but document that element==None and array==None with expected_type triggers a TypeError.
