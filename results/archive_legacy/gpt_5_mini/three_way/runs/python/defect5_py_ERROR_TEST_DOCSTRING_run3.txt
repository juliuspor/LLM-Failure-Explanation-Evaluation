What happened (root cause)
- The failing test calls ArrayUtils.add(None, None, expected_type=str).
- The implementation handles the "array is None" case by allocating a new array with one slot and placing None into it, but it allocates an untyped/generic array (Object[] / list-of-objects) instead of an array whose component type is the requested expected_type.
- Later the caller expects a String[] (because expected_type=str), so the runtime tries to cast the returned Object[] to String[] and fails with:
  ClassCastException: [Ljava.lang.Object; cannot be cast to [Ljava.lang.String;
- In short: when both array and element are None, the code falls back to creating an Object[] rather than creating an array whose component type matches expected_type.

Where it occurs in the code
- In the branch inside ArrayUtils.add that handles array is None:
  - If element is not None the code infers the component type from element and creates the right typed array.
  - If element is None the code currently creates an Object[] (e.g. new Object[] { null } or its Python/interop equivalent). That allocation is the wrong place: it must honor expected_type when provided.
- The same kind of bug can affect add_at_index when it needs to create a new array for a null original array and a null element.

Why that produces ClassCastException
- Java arrays are covariant and carry component-type information at runtime. An Object[] is not a String[], so casting fails.
- The test/consumer expects an array with component type equal to expected_type (String.class), so returning Object[] breaks that expectation at the JVM type-check/cast boundary.

Exactly what to change (concrete fixes and steps)
1. In ArrayUtils.add, find the branch that runs when array is null (or length == 0):
   - Currently it probably does something like: return new Object[] { element }  (or in Python-Java interop returns an Object[]).
2. Replace that generic allocation with an allocation that uses expected_type when available:
   - If expected_type is provided (non-null), create an array with that component type:
     - In Java:
       - Use java.lang.reflect.Array.newInstance(expectedType, 1) to create a T[1].
       - Do: T[] result = (T[]) Array.newInstance(expectedType, 1);
       - result[0] = (T) element; // element may be null — fine
       - return result;
     - Or use Arrays.copyOf(original, newLength, expectedType) if copying.
   - If expected_type is not provided but element != null, infer component type from element.getClass() as before.
   - If expected_type is not provided and element == null, then returning Object[] is acceptable (no type expectation).
3. Apply the same fix to add_at_index where a new single-element array must be created when array is null.
4. When copying an existing array to a bigger array, preserve the component type:
   - Do not always copy into Object[]. Use Arrays.copyOf(original, newLength, original.getClass()) or Array.newInstance(original.getClass().getComponentType(), newLength) followed by System.arraycopy.
   - If expected_type is provided and differs from original component type, validate and choose the appropriate component type or raise TypeError/IllegalArgumentException.

Minimal example patch (Java-like pseudocode)
- Replace:
    if (array == null) {
        return new Object[] { element };
    }
  With:
    if (array == null) {
        Class<?> compType = (expectedType != null) ? expectedType : (element != null ? element.getClass() : Object.class);
        Object newArray = Array.newInstance(compType, 1);
        Array.set(newArray, 0, element);
        return (T[]) newArray;
    }

Checks you should add / keep
- If expected_type is supplied but the existing array has a different component type, raise TypeError (or the Java equivalent) rather than silently returning an incompatible array.
- Add unit tests for:
  - add(None, None, expected_type=SomeClass)
  - add(None, non_null_element, expected_type=SomeClass)
  - add(existingArrayOfTypeX, elementOfTypeY, expected_type=Z) to verify validation behavior.

Summary
- Bug: when array and element are both null, the code returns an Object[] regardless of expected_type.
- Why: Object[] cannot be cast to String[] (or other concrete array types), so the cast fails at the caller.
- Fix: when creating a new array for a None input, use Array.newInstance(expected_type, 1) (or the equivalent) whenever expected_type is provided; otherwise infer type from element or fall back to Object. Apply same logic when resizing/copying arrays.
