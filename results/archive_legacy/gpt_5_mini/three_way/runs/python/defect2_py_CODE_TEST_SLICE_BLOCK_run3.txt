What happened (root cause), where it happened, and why it fails
- What happened: values outside the configured bounds are not being clamped before the grayscale value is computed. That lets g fall outside 0–255 and triggers the ValueError on lines 74–75.
- Where it happened: in get_paint(), you compute and use v for clamping at the top (lines 3–7), but on line 9 you compute g from value instead of from v:
  - Lines 3–7: v = max(value, self._lower_bound); v = min(v, self._upper_bound)
  - Line 9: g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
  Because you used value there, the clamped v is never used for the conversion.
- Why it fails: when value is out of bounds (e.g. -0.5 or 1.5 in your tests), the formula produces g < 0 or g > 255 and the code raises the ValueError on lines 74–75. That is exactly the test failure you saw for the negative/overbound cases.

Additional possible edge case
- If self._upper_bound == self._lower_bound, the denominator (self._upper_bound - self._lower_bound) is zero and you'll get a ZeroDivisionError. Your current code does not handle that.

Concrete fixes (specific changes to make)
1. Use the clamped variable v when computing g (replace value with v):
   - Change
     g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
     to
     g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

2. Guard against zero-length range (upper == lower). Choose behavior:
   - Option A — treat as middle gray or black/white as appropriate; or
   - Option B — raise a clear ValueError early.
   Example check:
     if self._upper_bound == self._lower_bound:

3. (Optional) Ensure final clamping of g to 0..255 after rounding:
   - After computing g, do:

Minimal corrected get_paint example
- Replace the problematic lines with:

    # clamp value
    v = max(value, self._lower_bound)
    v = min(v, self._upper_bound)

    # guard zero-length range
    if self._upper_bound == self._lower_bound:
        raise ValueError("upper_bound must differ from lower_bound")

    # compute using clamped v
    g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

    # final clamp to avoid out-of-range due to rounding
    if g < 0:
        g = 0
    elif g > 255:
        g = 255

    return (g, g, g)

Why this fixes it
- Using v ensures values outside the bounds are mapped to the bound before converting, so g will be 0..255 instead of outside that range (prevents the ValueError on lines 74–75).
- The zero-range check prevents division-by-zero crashes.
- The final clamp guarantees robustness against tiny floating-point rounding errors.

That directly addresses the bug at line 9 (using value instead of v) and prevents the exception raised on lines 74–75.
