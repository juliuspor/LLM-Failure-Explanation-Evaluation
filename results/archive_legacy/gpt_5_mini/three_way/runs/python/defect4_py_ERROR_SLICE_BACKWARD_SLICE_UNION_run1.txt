What happened (root cause)
- At __init__ you set self._min_start_index = -1 (line 253) so the first time you see a data item you must record its index.
- In the code that updates the minimum-start tracking (lines 340–349) you have two assignments to self._min_start_index:
  - One inside the conditional that compares starts (line 347): if start < min_start: self._min_start_index = index
  - One unconditional assignment immediately after (line 349): self._min_start_index = index
- Because of the unconditional assignment on line 349 the _min_start_index is always set to the most recently processed index, overriding the conditional result. That makes _min_start_index end up as the last index processed (3 in your failing run) instead of the true minimum-start index (1). The downstream code that computes max_middle_index expects the correct min start index, so the test fails with Expected max_middle_index=1, got 3.

- Update block: lines 340–349
  - self._min_start_index >= 0 check: line 340

- Accessor used elsewhere: get_min_start_index (lines 437–439) returns the wrong value because of the overwrite.

Why this produces the specific failure
- For the dataset that produced the test failure the earliest-start item was at index 1. Your code overwrote that with index 3 (the last processed item), so later logic computed max_middle_index based on the wrong min-start index and the assertion failed.

Concrete fix (exact change to make)
Replace the update block so you only set _min_start_index when:
- it was not set yet (initial case), or
- the new item's start is strictly earlier than the recorded min_start.

Option A — simplest replacement (modify lines 340–349):

Replace that whole block with:

if self._min_start_index < 0:
    self._min_start_index = index
else:
    min_start_period = self.get_data_item(self._min_start_index).get_period()
    if isinstance(min_start_period, SimpleTimePeriod):
        min_start = int(min_start_period.get_start().timestamp() * 1000)

        self._min_start_index = index
