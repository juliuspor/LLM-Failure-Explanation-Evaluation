What happened (where it fails)
- The IndexError is raised inside substring_java -> SLICE_BLOCK line 32:
  if start < 0 or end < 0 or start > end or start > len(s) or end > len(s):
      raise IndexError(f"String index out of range: {end}")
- That means substring_java was called with end > len(text).

Why it happened (root cause, exact variables & lines)
- You try to keep upper within the string earlier:
    if upper == -1 or upper > len(text):
        upper = len(text)
- But immediately after you do:
    if upper < lower:
        upper = lower
- If lower is greater than len(text), this second assignment sets upper = lower which can be > len(text). That produces a call like substring_java(text, 0, upper) where upper > len(text), which triggers the IndexError at SLICE_BLOCK line 32 (because end > len(s)).

Concrete reproduction path:
- Pass lower > len(text) (for example text length = 10, lower = 12, upper = -1).
- First branch sets upper = len(text) (10).
- Second branch sees upper < lower (10 < 12) and sets upper = 12.
- Later you call substring_java(text, 0, 12) -> slice code checks end > len(s) and raises IndexError: "String index out of range: 12".

Precise fixes (step-by-step)
Option A — clamp upper after all adjustments (recommended minimal change):
- After the block that may raise upper = lower, add:
    if upper > len(text):
        upper = len(text)
This ensures you never call substring_java with end > len(text).

Option B — clamp lower at start (alternate):
- At the top, ensure lower is within bounds, for example:
    if lower < 0:
        lower = 0
    if lower > len(text):
        lower = len(text)
This prevents lower from ever pushing upper past the string length.

Option C — return early if lower >= len(text) (behavioral choice):
- If you consider "starting past end" to mean "no abbreviation needed", add:
    if lower >= len(text):
        return text
or if you want empty behavior, return "" or StringUtils.EMPTY accordingly.

Where to change (exact locations)
- Change immediately after:
    if upper < lower:
        upper = lower
  Add the clamp:
    if upper > len(text):
        upper = len(text)
- Or place the lower-clamp near the top, just after the len(text)/None checks.

Which fix to pick
- If you want minimal behavior change and safety: apply Option A.
- If the intention is that lower should be interpreted within the string bounds rather than forcing upper up, use Option B.
- If defining semantics for lower past the end should return the original text or empty string, use Option C.

This is an off-by-bounds bug caused by the order of adjustments to upper and lower: upper is clamped, then overwritten with (possibly out-of-range) lower.
