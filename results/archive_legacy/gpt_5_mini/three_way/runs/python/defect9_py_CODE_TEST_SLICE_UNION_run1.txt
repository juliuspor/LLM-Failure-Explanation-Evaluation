What happened (root cause)
- The bug is in the "guess result size" loop at lines 81–85:
  > greater = len(replacement_list[i]) - len(search_list[i])

  That code calls len(...) on both replacement_list[i] and search_list[i] without checking for None. In your failing test you pass replacement_list = ["c", None] (or anywhere an element is None). When i indexes the None entry, len(None) raises TypeError: "'NoneType' object has no len()". So the function crashes before it ever reaches the later null-safety checks (which exist in the matching loops).

Where it occurs in the code
- The problematic lines are inside:
  for i in range(search_length):
      greater = len(replacement_list[i]) - len(search_list[i])
      if greater > 0:
          increase += 3 * greater  # assume 3 matches
  (lines shown in your slice at 81–85)

Why it happens despite later checks
- Later, when finding matches, you do check:
    if no_more_matches_for_repl_index[i]
       or search_list[i] is None
       or len(search_list[i]) == 0
       or replacement_list[i] is None: continue
  but the early "increase" loop runs before that, and has no such guards. So None values in the arrays are unprotected there.

How to reproduce quickly
- Call replace_each("aba", ["a","b"], ["c", None]) — the code will raise TypeError at the "increase" loop.

Concrete fixes (step-by-step)
1. In the "increase" loop skip null/empty entries using the same conditions you use later. Replace the body at lines 81–85 with:

   for i in range(search_length):
       if search_list[i] is None or len(search_list[i]) == 0 or replacement_list[i] is None:
           continue
       greater = len(replacement_list[i]) - len(search_list[i])
       if greater > 0:
           increase += 3 * greater

2. (Optional) Make the check identical to the later matching guard to avoid future divergence: include no_more_matches_for_repl_index[i] if you want, though at this point that array is all False so it's not required.

3. Keep the existing cap: increase = min(increase, len(text) // 5)

Why this fix works
- It avoids calling len(...) on None, so TypeError cannot occur. It also keeps the "size guess" behavior correct by only measuring lengths for replacements/searches that will actually be used.

Minimal patch (one-line replacement)
- Replace:
    greater = len(replacement_list[i]) - len(search_list[i])
  with:
    if search_list[i] is None or len(search_list[i]) == 0 or replacement_list[i] is None:
        continue
    greater = len(replacement_list[i]) - len(search_list[i])

That resolves the failing tests (test_null_safety_inside_arrays_*).
