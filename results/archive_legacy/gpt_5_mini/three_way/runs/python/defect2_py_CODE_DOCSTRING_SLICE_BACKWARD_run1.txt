What happened (root cause)
- At line 54 you clamp the input into the local variable v:
  - v = max(value, self._lower_bound)
  - v = min(v, self._upper_bound)
- But at line 71 you compute g from value, not from the clamped v:
  - g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- So when value is outside the bounds, the calculation still uses the original (out‑of‑range) value and can produce g < 0 or g > 255, which triggers the ValueError at line 74–75.

Where it occurs
- The mismatch occurs between the clamping lines (54) and the calculation of g (71). The code clamps into v but never uses v in the computation.

Why it fails in plain terms
- You intended to clamp the input before mapping it into 0–255, but because you kept using value instead of v the clamp is ignored. The check that follows then correctly detects the out‑of‑range result and raises the error.

2. The error message at lines 74–75 is not informative — it doesn’t show the offending g or the bounds.

  def get_paint(self, value: float) -> Tuple[int, int, int]:
      v = max(value, self._lower_bound)

      if self._upper_bound == self._lower_bound:
