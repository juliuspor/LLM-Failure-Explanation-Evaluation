What happened (root cause)
- At line 176 your code tests: if x == int(x) and not math.isinf(x) and not math.isnan(x):
  - For x == -0.0 that condition is True because -0.0 == 0 and int(-0.0) == 0.
- At line 177 you do value = int(x). That converts -0.0 to the integer 0 and therefore loses the sign bit.
- Later (line 192) you do self.add(str(value)), which adds "0" to the output.
- Result: the negative zero sign is dropped; the test expected "-0.0" but your code printed "0".

Why this happens (why equality/int conversion fail to preserve sign)
- IEEE-754 floats have a negative zero that compares equal to positive zero. Numeric equality (x == 0) and int(x) do not preserve the sign bit. So using x==int(x) and int(x) to decide formatting will erase the negative zero information.

Where to change it (precise steps)
1. Detect negative zero before you convert to int or take the integer-branch:
   - Add a check before line 176. For example:

           self.add("-0.0")
           return
   - That ensures -0.0 is printed as "-0.0" and avoids losing the sign via int(x).

How to implement is_negative_zero (if you don't already have it)
- A reliable implementation is:
  - @staticmethod
    def is_negative_zero(x):
        return x == 0.0 and math.copysign(1.0, x) < 0.0
- This uses math.copysign to inspect the sign bit without changing the value.
