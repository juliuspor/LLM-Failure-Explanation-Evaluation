Why it fails (root cause)
1. Wrong notion of "element type":
   - line 336 uses type(array). For a Python list, type(array) is list (the container type), not the element type. That makes inferred_type wrong whenever an array is passed.
   - Example: array = ["a","b"] -> type(array) == list, not str. That is already incorrect logic for element-type inference.

2. Incorrect fallback and check when both array and element are None:
   - For ArrayUtils.add(None, None, expected_type=str), the code goes:
     - array is None, element is None -> neither branch at 335 or 337 sets inferred_type based on input
     - code falls through to the fallback at line 340 setting inferred_type = object
     - at line 346 the code checks if inferred_type == object and expected_type != object and then raises TypeError (line 347).
   - This produces the TypeError: the function refuses to proceed even though the caller supplied expected_type=str and the correct behavior is to create an array of that expected type (in Python: return [None]) rather than raise.

3. The logical effect:
   - When both inputs are None but the caller expects a concrete element type, the code refuses to use expected_type and raises a TypeError (or later tries to create an object[] and cast to String[] causing ClassCastException in Java port). The test expects [None] instead.

Exactly where to change (lines and variables)
- Replace the logic at/around lines 335–347. The key variables are:
  - array (the input array)
  - element (the item to append)
  - inferred_type (what the code is trying to compute)
  - expected_type (the caller-provided desired element type)

What to change (concrete steps)
1. Infer element type from actual elements, not from the list object:
   - If array is not None and len(array) > 0:
       - set inferred_type = type(array[0]) (or inspect all elements to find a common type if you want stricter inference)
   - If array is not None and len(array) == 0:
       - do not use type(array); instead use expected_type if provided, or object as fallback.

2. Use expected_type as authoritative fallback:
   - If expected_type is provided (not object / not None), use it whenever you cannot determine a better element type from array or element.
   - Only fall back to object when neither array nor element provides a type and expected_type is unspecified.

3. Accept None element values:
   - element can be None: that should not prevent using expected_type or creating a one-element array of expected_type.
