What failed
- The test passes None inside replacement_list (e.g. ["a"], [None]). At runtime the code executes this line in replace_each:
  - Line 82: greater = len(replacement_list[i]) - len(search_list[i])
- Because replacement_list[i] (or search_list[i]) is None, calling len(None) raises:
  - TypeError: object of type 'NoneType' has no len()
- This is why test_null_safety_inside_arrays_single and test_null_safety_inside_arrays_mixed fail.

Where it happened
- Method: replace_each (you showed its header at line 28)
- Loop start: line 81: for i in range(search_length):
- Failing expression: line 82: greater = len(replacement_list[i]) - len(search_list[i])

Why it happened (root cause)
- The code assumes every element in search_list and replacement_list is a string and calls len(...) on them unguarded.
- The tests (mirroring the Java StringUtils behavior) pass None to indicate "no replacement". The Python code does not implement the Java semantics and does not handle None, so len(None) crashes.

How to fix it (concrete steps)
1. Decide how None should behave. The Java tests imply None means "no replacement" / treat as empty string for sizing purposes.
2. Normalize the lists before using len(...) or compute lengths with None handled explicitly. For example, inside the loop replace lines 81–83 with guarded code:

   - rep = replacement_list[i] if replacement_list[i] is not None else ""
   - s = search_list[i] if search_list[i] is not None else ""
   - greater = len(rep) - len(s)
   - if greater > 0:
         increase += 3 * greater  # keep original heuristic

3. Alternatively, preprocess both lists once at function start:
   - search_list = [x if x is not None else "" for x in search_list]
   - replacement_list = [x if x is not None else "" for x in replacement_list]

4. Also ensure any later code that uses those elements uses the normalized values (not the original possibly-None items).

Why that fix works
- Replacing None with "" makes len(...) safe and matches the intended "no replacement" semantic so the tests expecting "aba" or "cbc" will pass.
- The change is minimal and local to the part that currently assumes non-None strings.
