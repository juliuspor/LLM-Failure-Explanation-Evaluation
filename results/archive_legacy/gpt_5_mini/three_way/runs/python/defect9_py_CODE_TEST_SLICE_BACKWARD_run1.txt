What happened (root cause)
- The bug is in the "size-guess" loop just after the first match search. The code at lines shown by SLICE_BACKWARD:
  - Line ~81-82:
    greater = len(replacement_list[i]) - len(search_list[i])
- That loop assumes each entry in search_list and replacement_list is a string and calls len(...) on them.
- But earlier code explicitly allows (and skips) None or empty entries when searching for matches:
  - The match-finding loops check:
    if no_more_matches_for_repl_index[i] or search_list[i] is None or len(search_list[i]) == 0 or replacement_list[i] is None:
        continue
- The size-guess loop does not repeat these null/empty checks. If replacement_list[i] or search_list[i] is None, len(None) raises a TypeError and the function crashes.
- This is exactly what your tests exercise:
  - test_null_safety_inside_arrays_single calls replace_each("aba", ["a"], [None]) → replacement_list[0] is None → len(replacement_list[0]) raises.
  - test_null_safety_inside_arrays_mixed calls replace_each("aba", ["a","b"], ["c", None]) → replacement_list[1] is None → same failure.

Where it occurs (specific names/lines)
- The failing variables: replacement_list[i] and/or search_list[i].
- The failing operation: len(replacement_list[i]) and/or len(search_list[i]) in the size-guess loop (the one preceded by the comment "# BUG: this loop is not null-safe for elements inside the arrays").
- This happens before the main replacement loop, so the function never reaches the logic that would otherwise skip None entries.

How to fix it (exact steps)
1. In the size-guess loop, skip entries that are None or empty, exactly the same way the match-finding loops do. Replace the loop body with a guard that continues when an element is invalid.
2. Concretely, change the loop to:

for i in range(search_length):
    if (
        no_more_matches_for_repl_index[i]
        or search_list[i] is None
        or len(search_list[i]) == 0
        or replacement_list[i] is None
    ):
        continue
    greater = len(replacement_list[i]) - len(search_list[i])
    if greater > 0:
        increase += 3 * greater

3. Optionally keep the final clamp:
    increase = min(increase, len(text) // 5)

Why this fixes it
- It prevents calling len(...) on None values, matching the null-safety used in the rest of the function. That lets the function return the original text or compute a safe buffer guess without raising TypeError when array elements are None.

Alternative minimal fix
- If you prefer a minimal guard, check before the len() call:
    if replacement_list[i] is None or search_list[i] is None or len(search_list[i]) == 0: 
        continue

Summary
- The failure is a TypeError from calling len() on None in the size-guess loop (lines ~81-82). Add the same null/empty checks used elsewhere before computing lengths to fix the issue.
