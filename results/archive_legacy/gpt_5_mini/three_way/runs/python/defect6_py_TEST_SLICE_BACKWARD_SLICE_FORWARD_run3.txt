  - 176: if x == int(x) and not math.isinf(x) and not math.isnan(x):
  - 177:     value = int(x)
- int(-0.0) yields 0, so the comparison x == int(x) is True for x == -0.0, and value becomes 0 (sign lost).
- Later the integer branch prints the integer form (line 192): self.add(str(value)) which produces "0".
- Because the branch converts -0.0 to an integer 0 before any negative-zero check, the negative zero sign is discarded and the output becomes "0" instead of the required "-0.0".
- In other words, the integer-special-case code (lines 176–194) runs for -0.0 and loses the sign; there is no earlier check to detect and preserve negative zero.

Why int(-0.0) causes the bug
- Converting to int removes the floating-point sign bit; int(-0.0) == 0 and has no negative sign to carry to str(value).
- Negative zero is a distinct floating-point sign bit, not preserved by integer casting; so the sign must be checked on the float before casting.

Why these fixes work
- They detect the float sign bit while x is still a float (before int(x) removes it), so you can produce the required "-0.0" text.

Extra notes / gotchas
- Don’t rely on comparisons like x == 0.0 alone; use math.copysign or an is_negative_zero helper to distinguish -0.0 from +0.0.
- The same approach preserves negative infinity or NaN behavior only if you check not math.isinf(x) and not math.isnan(x) as the code already does.
- Make sure to return (or otherwise skip the integer branch) after handling negative zero so no later code overwrites the output.
