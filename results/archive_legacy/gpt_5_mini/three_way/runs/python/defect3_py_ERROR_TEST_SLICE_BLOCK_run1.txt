What happened
- The test input "\U0001F630A" contains an emoji (code point U+1F630) that is encoded as a UTF-16 surrogate pair (two UTF-16 code units) followed by the ASCII 'A' (one UTF-16 code unit). So the UTF-16 code_units sequence length is 3.
- The code calls code_point_at(code_units, index) which checks index bounds at lines 35–36 and raises IndexError if index >= len(code_units).
- The IndexError message shows index = 2, which is equal to len(code_units) in the failing call path (or later an index > 2 was used). That means the code attempted to read a code unit at position 2 when it was already past the end of the available units for the operation it intended to perform.

Why it failed (root cause)
- The translator is treating surrogate pairs incorrectly when advancing the position (variable index) through the UTF-16 code_units:
  - When a surrogate pair is encountered, it consumes two UTF-16 code units (a high surrogate then a low surrogate).
  - The translate loop increments index by the wrong amount after handling a surrogate pair (or checks bounds against the wrong length), so a subsequent call to code_point_at is made with index equal to or beyond len(code_units).
- Concretely:
  - Starting index = 0: code_point_at reads the high surrogate and low surrogate -> returns one code point and should advance index by 2 (two code units).
  - If the code advances by 1 instead (or fails to check that 2 units were consumed), the next iteration calls code_point_at with index = 1 which points to the low surrogate (invalid start), or if it advanced by 3 or otherwise miscalculated, it calls with index = 3 (out of range). The bound check at lines 35–36 then raises IndexError: "String index out of range: 2".
- The visible symptom (IndexError at code_point_at bound check) is the direct result of incorrect index advancement/consumption logic in the translate loop, not of code_point_at itself.

Where in the code it occurs
- Bound check that raised the error: lines 35–36 in code_point_at:
  35 |     if index < 0 or index >= len(code_units):
  36 |         raise IndexError(f"String index out of range: {index}")
- The mistake is in the translate loop (likely CsvEscaper.translate_string) that calls to_utf16_code_units, then iterates using index and calls code_point_at. That loop advances index incorrectly after encountering surrogate pairs, causing code_point_at to be called with an out-of-range index.

How to fix it (concrete steps)
1. Make sure to iterate over UTF-16 code units correctly:
   - Compute code_units = to_utf16_code_units(s) once.
   - Use a while loop: while index < len(code_units):
       - Use code_point = code_point_at(code_units, index)
       - Determine units_consumed: if code_point > 0xFFFF then units_consumed = 2 else units_consumed = 1
       - Advance index by units_consumed
   - Always check bounds before attempting to read a low surrogate: if a high surrogate is found at index and index + 1 >= len(code_units), handle it as malformed or use a fallback (don’t call code_point_at beyond end).

2. Alternatively, avoid manual UTF-16 unit handling in Python:
   - Python strings are sequences of Unicode code points. Iterate directly over characters in the Python string (for ch in s:) which will treat surrogate pairs as single characters on narrow builds, and always avoid index arithmetic errors. This is simplest and least error-prone.

3. Specific code patch idea:
   - After code_point_at returns a code point, advance index like:
       if code_point > 0xFFFF:
           index += 2
       else:
           index += 1
   - Ensure code_point_at itself checks for a missing low surrogate and does not read code_units[index+1] without verifying index+1 < len(code_units).

Why this will fix the test
- For "\U0001F630A", the emoji consumes two code units so index becomes 2 after it’s processed. The next call should read the 'A' at index 2, which is in-range. With correct consumption logic (advance by 2 for the surrogate), code_point_at will be called only with valid indices and no IndexError will be raised.

Summary
- Root cause: index advancement/consumption logic is wrong when handling UTF-16 surrogate pairs, so code_point_at gets called with an out-of-range index (triggering the check at lines 35–36).
- Fix: ensure the loop advances index by the number of UTF-16 code units actually consumed (2 for a surrogate pair, 1 otherwise), and guard any read of index+1 with a bounds check. Or iterate over Python code points directly to avoid UTF-16 manual handling.
