What fails
- The tests call GrayPaintScale.get_paint(-0.5) and get_paint(1.5) expecting the method to clamp out-of-range inputs to the nearest bound (black or white). Instead the implementation raises a ValueError.
- Where it happens: in get_paint, at the calculation on line 71 and the subsequent check on lines 74–75:

  - Line 71:
    g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

  - Lines 74–75:
    if g < 0 or g > 255:
        raise ValueError(f"Color parameter outside of expected range: Red Green Blue")

Why it fails (root cause)
- For input values outside the bounds (e.g. -0.5 when lower_bound is 0.0), the computed g becomes negative. The code explicitly raises a ValueError when g is outside [0,255]. The unit tests expect those values to be clamped, not to raise exceptions. So the code violates the expected behavior by raising an exception at lines 74–75 instead of clamping.
- The failure therefore occurs in get_paint, because the method enforces an error on out-of-range results instead of adjusting them to valid color channel values.

Precise fix steps
1. Decide where to clamp:
   - Option A (preferred): Clamp the input value before mapping to gray. This keeps the mapping formula simple and avoids intermediate out-of-range g.
     Example change (replace lines 71–75):
       v = min(max(value, self._lower_bound), self._upper_bound)
       g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
   - Option B: Compute g and then clamp g to [0,255] instead of raising:
       g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
       g = max(0, min(255, g))

2. Remove the ValueError branch that raises for g out of range (lines 74–75). Replace it with one of the clamping implementations above.

3. (Optional) Handle the special case where upper_bound == lower_bound to avoid division by zero:
   - If self._upper_bound == self._lower_bound: set g = 0 (or 255) depending on desired semantics; most likely set g = 0.

Why these fixes work
- Clamping the input (or the computed g) ensures negative g and g > 255 are converted to valid channel values (0..255) instead of causing an exception, matching the test expectations for lower- and upper-bound inputs.
- Clamping before computing g avoids intermediate values that would trigger the ValueError.

Exact minimal code example to replace lines 71–75:
  # clamp input value to bounds
  v = min(max(value, self._lower_bound), self._upper_bound)
  # avoid division by zero if bounds are equal
  if self._upper_bound == self.__lower_bound:
      g = 0
  else:
      g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
  # ensure g is within [0,255]
  g = max(0, min(255, g))

This change will make tests for -0.5 and 1.5 produce (0,0,0) and (255,255,255) respectively instead of raising ValueError.
