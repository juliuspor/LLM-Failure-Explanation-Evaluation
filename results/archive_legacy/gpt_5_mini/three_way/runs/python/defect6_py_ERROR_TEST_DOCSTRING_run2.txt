What happened
- The failing assertion (Expected "-0.0", got "0") comes from TestCodeConsumerAddNumber.test_negative_zero, which calls CodeConsumer.add_number(-0.0). The code produced "0" instead of "-0.0", so the sign on negative zero was lost.

Why it happened (root cause)
- add_number contains a branch that treats values that are integral as integers and writes them with int formatting. A typical implementation looks like this:

  if number.is_integer():
      self.append(str(int(number)))
  else:
      self.append(str(number))

  For -0.0, number.is_integer() is True and int(-0.0) == 0, so the code appends "0". That conversion removes the negative sign, so -0.0 becomes "0".

- The code ignores the special case of negative zero before converting to int. The repository already has a utility is_negative_zero (tested by TestNegativeZeroUtility) but add_number does not use it; therefore the negative zero path isn't handled.

Where in the code this occurs
- Method: CodeConsumer.add_number
- Variables: number (the float passed in), self._output (or whatever append writes to)
- Specific failing test: TestCodeConsumerAddNumber.test_negative_zero in the test file you supplied

How to fix it (concrete steps)
1. Detect negative zero before the integer-branch. Use the existing utility CodeConsumer.is_negative_zero(number) (or implement equivalent math.copysign/struct.signbit check).
2. If number is negative zero, append the string "-0.0" and return without doing integer formatting.
3. Otherwise keep the current logic (integer formatting for true integers, normal float formatting otherwise).

A minimal patch example (illustrative; adapt to your exact method/line numbers):

- In CodeConsumer.add_number(self, number): find the start of the method and insert the negative-zero check near the top:

    # special-case negative zero to preserve sign
    if self.is_negative_zero(number):
        self.append("-0.0")
        return

- Keep the existing integer branch after that:

    if number.is_integer():
        self.append(str(int(number)))
    else:
        self.append(str(number))

Why this fixes it
- By checking is_negative_zero first, you avoid converting -0.0 to int( -0.0 ) == 0 and losing the sign. The check preserves the distinct representation required by the test and by JavaScript-like output needs ("-0.0").

Optional: alternative detection (if is_negative_zero is not implemented)
- Use math.copysign or struct to detect sign bit. For example:

    import math
    if number == 0.0 and math.copysign(1.0, number) < 0:
        self.append("-0.0")
        return

But since your test suite already contains CodeConsumer.is_negative_zero and tests it, prefer calling that utility.

What to verify after the change
- Run TestCodeConsumerAddNumber.test_negative_zero and the other number tests to ensure:
  - negative zero becomes "-0.0"
  - positive zero becomes "0"
  - integer behavior for other integers remains the same

Summary
- Root cause: integer-branch converts -0.0 to int 0 before checking sign, removing the negative sign.
- Fix: check is_negative_zero(number) first in add_number and append "-0.0" in that case.
