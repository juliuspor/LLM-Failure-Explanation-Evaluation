Why this is wrong (root cause)
- The method intends to build a single total offset in minutes from hours_offset and minutes_offset. For negative hour offsets the minutes part is supposed to be subtracted (e.g. -1 hour and 15 minutes -> -75 minutes), so it is legitimate for the minutes parameter to be thought of as positive 15 but combined into a negative total when hours_offset < 0.
- But the code validates minutes_offset for being in [0, 59] before it applies the sign of hours_offset (the sign is applied later at lines where minutes_offset is reassigned):
  ```
  hours_in_minutes = hours_offset * 60
  if hours_in_minutes < 0:
      minutes_offset = hours_in_minutes - minutes_offset
  else:
      minutes_offset = hours_in_minutes + minutes_offset
  ```
  Because validation happens earlier, any input with a negative minutes_offset (for example tests that pass -15 to indicate negative 15 minutes together with a negative hour) is rejected immediately. In short: the sign normalization is done after validation, so valid combinations are being rejected.
