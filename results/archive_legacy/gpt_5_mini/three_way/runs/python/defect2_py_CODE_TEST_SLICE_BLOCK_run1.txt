What happened (root cause)
- In get_paint you correctly clamp the input into the local variable v:
  - v = max(value, self._lower_bound)
  - v = min(v, self._upper_bound)
- But when you compute g you use the original parameter value instead of the clamped v:
  - g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- So when value is outside [self._lower_bound, self._upper_bound] the computed g can be < 0 or > 255 and triggers the ValueError at lines 74–75:
  - if g < 0 or g > 255: raise ValueError(...)
- That is why tests such as get_paint(-0.5) or get_paint(1.5) fail: the input was supposed to be clamped but the calculation ignored the clamp and produced an out-of-range channel value that raises the exception.

Where it occurs (line references)
- Clamping happens just above the calculation (v = ...).
- Wrong variable used at the g calculation line (the line shown in your snippet).
- The exception is raised at lines 74–75 (the SLICE_BLOCK you provided).

How to fix it (concrete steps)
1. Use the clamped value v when calculating g:
   - g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
2. Defend against division-by-zero if lower and upper bounds could be equal:
   - if self._upper_bound == self._lower_bound:
       raise ValueError("upper_bound must be greater than lower_bound")
   - Put this check before the computation.
3. (Optional) Improve the error message so it reports the actual invalid value if you keep the range check.
4. (Optional) If you prefer rounding to nearest integer instead of truncation, use round(...) before casting to int.
