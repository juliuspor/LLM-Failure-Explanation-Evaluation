What happened (what failed)
- At line 335 the code sets inferred_type = type(array). That takes the Python type of the list object itself (i.e. <class 'list'>), not the component/element type you want.
- On the failing test path (test_add_null_to_null_with_expected_type) both array and element are None, so execution falls through to line 340 where inferred_type = object.
- Then at lines 346–351 the code checks if inferred_type == object and expected_type != object and raises a TypeError. Since expected_type was passed as str, the check raises:
  "Cannot cast object list to str list (ClassCastException: ...)"
- In short: for the case array is None and element is None but an expected_type was supplied, the function incorrectly treats the inferred type as object and immediately raises instead of using the expected_type. That is why the test fails.

Why this is the root cause (clear, specific)
- type(array) at line 336 is wrong for determining the element type. It returns the list class, not the type of list elements. If you intended to infer element type from an existing non-empty list you need to inspect the elements, not the list object.
- The code never uses expected_type to set inferred_type when both array and element are None. The single-path fallback to object makes the later guard (lines 346–351) always fail when expected_type is supplied.
- The offending lines are 335–346 (inference) and 346–351 (the guard that raises). Together they cause the function to raise when it should return [None] with the expected_type honoured.

Where to change (exact places)
- Replace inferred_type = type(array) at line 336 with logic that finds the element/component type from the contents of array (if available), or uses expected_type if array is empty/None and element is None.
- Change the fallback at line 340 so that if expected_type is provided and both array and element are None, inferred_type = expected_type instead of object.
- Rework the guard at lines 346–351 so it only raises when there is a real incompatibility between inferred_type (from existing elements or element) and expected_type, not simply because you fell back to object.

For the specific test you should not raise when both array and element were None but expected_type was provided — instead use expected_type to construct the list.

Summary
- Root cause: wrong inference at line 336 (type(array) used instead of element type) and falling back to object at line 340 without honoring expected_type, combined with the strict guard at lines 346–351 that raises for the case expected_type=str and both array and element None.
