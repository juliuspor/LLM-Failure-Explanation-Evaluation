What happened (what failed)
- The test for negative zero expects add_number(-0.0) to emit "-0.0".
- The code at line 176 classifies -0.0 as an integer because the test x == int(x) is True for -0.0 (since -0.0 == 0).
- That sends execution into the integer branch (lines 176–192). At line 177 you do value = int(x), which produces 0 and loses the negative sign.
- Later you call self.add(str(value)) (line 192) and emit "0" (no sign), so the test comparing "-0.0" to "0" fails.

Root cause (why it happened)
- The condition x == int(x) is a correct integer-detection for numeric values, but it does not distinguish negative zero from integer zero. In IEEE floats -0.0 == 0.0, so that equality hides the negative sign.
- The code converts the float to an int (value = int(x)) before checking or preserving the negative-zero property, so the sign is discarded early.

Where it occurs
- The misclassification happens at line 176 (if x == int(x) and not math.isinf(x) and not math.isnan(x):).
- Losing the sign happens at line 177 (value = int(x)) and the emission occurs at line 192 (self.add(str(value))).

Concrete fixes (exact steps you can apply)
1. Detect negative zero before the integer branch and handle it explicitly. If CodeConsumer already has the is_negative_zero utility (the tests call CodeConsumer.is_negative_zero), use it first:
   - Add this check immediately before line 176:
     if self.is_negative_zero(x):
         self.add("-0.0")
         return
   This ensures -0.0 never reaches the integer-conversion branch.

2. Alternatively, change the integer branch condition so negative zero is excluded:
   - Replace the condition on line 176 with:

Either change will prevent loss of the negative sign.

Why these fixes work
- They detect the special IEEE negative-zero case before int(x) is used, preserving sign by emitting "-0.0" (or allowing the float-handling branch to use str(x), which yields "-0.0" in Python).
- They are precise (only affect negative zero), so other behavior (integer formatting, scientific notation logic starting at line 180, etc.) remains unchanged.

Extra note
