What happened (root cause)
- After clamping the input into v:
      v = max(value, self._lower_bound)
      v = min(v, self._upper_bound)
  the code still computes g using the original value:
      g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
  Because it used value instead of the clamped v, out-of-range inputs (for example value = -0.5) produced a negative g and triggered the ValueError check. This is why test_lower_bound_clamping fails: the method is supposed to clamp first but then ignores v and continues with the un-clamped value.

Where it occurs
- In get_paint, immediately after the clamping lines, on the line that sets g (the line beginning with "g = int((value - ...").

Why that causes the test failure
- For value = -0.5 and lower_bound = 0.0, the expression (value - lower_bound) / (upper_bound - lower_bound) becomes negative, so g < 0 and the following check raises ValueError("Color parameter outside of expected range: Red Green Blue"). The test expects (0,0,0) instead.

Also note a possible secondary issue
- If self._upper_bound == self._lower_bound you will get a division by zero. Even once you fix the main bug, you should handle that case or assert bounds are different at construction.

Concrete fix (step-by-step)
1. Use the clamped variable v when computing g:
      g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
2. Optionally guard against division by zero (bounds equal) — either raise a clear exception in the constructor or handle it in get_paint:
      span = self._upper_bound - self._lower_bound
      if span == 0.0:
          raise ValueError("upper_bound and lower_bound must differ")
      frac = (v - self._lower_bound) / span
      g = int(frac * 255.0)
3. Consider using round() instead of int() if you want symmetric rounding (int() truncates toward zero).

Minimal replacement code
    v = max(value, self._lower_bound)
    v = min(v, self._upper_bound)
    span = self._upper_bound - self._lower_bound
    if span == 0.0:
        raise ValueError("upper_bound and lower_bound must not be equal")
    g = int((v - self._lower_bound) / span * 255.0)
    if g < 0 or g > 255:
        raise ValueError(...)
    return (g, g, g)

That change uses the clamped v and prevents the ValueError in your failing tests.
