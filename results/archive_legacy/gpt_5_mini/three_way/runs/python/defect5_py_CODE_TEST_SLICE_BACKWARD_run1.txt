What happens (root cause)
- At line 335 you set inferred_type = type(array). For a Python list this yields <class 'list'> — not the component/element type you intend. So inferred_type becomes "list" instead of the element type (e.g. str or int). That is the wrong vocabulary for later decisions.
- At lines 337–340, when both array and element are None you set inferred_type = object. In your test test_add_null_to_null_with_expected_type you call ArrayUtils.add(None, None, expected_type=str). Because array is None and element is None the code sets inferred_type = object (line 340) instead of using the provided expected_type.
- Then at line 346 you check if inferred_type == object and expected_type != object and raise a TypeError. That triggers in the test and causes the failure: you raise TypeError when the caller supplied an expected_type, but you never used it to choose the inferred_type.

Where it occurs (specific lines / variables)
- Line 335: inferred_type = type(array) is wrong because type(array) == list, not the element class.
- Line 340: inferred_type = object when both array and element are None; ignores expected_type.
- Line 346: the conditional inferred_type == object and expected_type != object raises TypeError; this is the observed failure for add(None, None, expected_type=str).

Why this produces the observed failure
- The test expects ArrayUtils.add(None, None, expected_type=str) to return [None] without error. But because inferred_type becomes object, the code treats that as an incompatible situation and raises TypeError instead of producing a list whose element-type is considered str. The code never uses expected_type to set the inferred_type, so the presence of expected_type is ignored until used only to throw an error.

Concrete fixes (explicit steps and exact places to change)
1. Infer element type correctly from array:
   - Replace inferred_type = type(array) (line 335) with logic that inspects array contents:
     - If array is not None and len(array) > 0: use the first non-None element's type (e.g. next((type(x) for x in array if x is not None), None)).
     - If array is empty or all elements are None, do not set inferred_type to list; consider expected_type next.
2. Respect expected_type when array and element are both None:
   - If array is None and element is None and expected_type is not None, set inferred_type = expected_type (instead of object).
3. Change the raising logic at line 346:
   - Do not raise simply because inferred_type == object and expected_type != object. Instead, if expected_type is provided you should use expected_type to construct the new list (or at least allow it). Only raise if you truly cannot create a list of expected_type.
4. Example corrected selection logic (pseudo-code to replace lines 335–340 and the check at 346):

    if array is not None:
        # try to infer from existing elements, not from list type
        inferred_type = next((type(x) for x in array if x is not None), None)
    elif element is not None:

    elif expected_type is not None:
        inferred_type = expected_type
    else:

    if expected_type is not None:
        # prefer expected_type if inferred is None/object
        if inferred_type is None or inferred_type is object:
            inferred_type = expected_type

Notes
- The intent is to pick a component (element) type, not the container type.
- Use the caller's expected_type whenever you can't reliably infer a more specific element type from array or element.
