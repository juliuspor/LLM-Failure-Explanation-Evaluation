What happened
- The code fails with TypeError: object of type 'NoneType' has no len() because it calls len(...) on a None element inside the loop that estimates buffer growth.
- The offending code is the loop immediately after the comment "# BUG: this loop is not null-safe for elements inside the arrays":  
  for i in range(search_length):
      greater = len(replacement_list[i]) - len(search_list[i])
      if greater > 0:
          increase += 3 * greater

Why it fails (root cause)
- The function allows search_list and replacement_list to contain None values and intends to ignore such pairs elsewhere (see the two later loops that check "search_list[i] is None" and "replacement_list[i] is None").
- But in the "increase" estimation loop there is no such check. If either replacement_list[i] or search_list[i] is None, calling len(replacement_list[i]) or len(search_list[i]) raises TypeError because None has no length.
- In short: the code tries to compute lengths for items that may be None; the other loops guard against None, but this one does not.

Where it occurs
- In replace_each(), in the loop that computes "increase" (the buffer-size guess) — the line with "greater = len(replacement_list[i]) - len(search_list[i])".

How to fix it (concrete steps)
1. Skip None or empty search entries in that loop exactly the same way the matching loops do. For example replace the problematic loop with:

    for i in range(search_length):
        if search_list[i] is None or len(search_list[i]) == 0 or replacement_list[i] is None:
            continue
        greater = len(replacement_list[i]) - len(search_list[i])
        if greater > 0:
            increase += 3 * greater

2. Alternatively, prefilter the input arrays into parallel lists that contain only valid (non-None, non-empty-search, non-None-replacement) pairs, and then use those filtered lists everywhere (including estimating increase).

3. Less preferred: treat None as length 0 by using safe_len helpers, but that hides the intent; it's clearer to skip invalid pairs.

Which fix I recommend
- Apply step 1: add the same guards used elsewhere in the function to the "increase" loop. That is simple, consistent with the rest of the function, and avoids calling len(...) on None.

Result
- After this change, len(...) will only be called on actual strings, so the TypeError goes away and None entries will be ignored as originally intended.
