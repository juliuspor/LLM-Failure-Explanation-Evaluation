What happened (root cause)
- The loop that computes increase (lines 83–85) calls len(replacement_list[i]) and len(search_list[i]) without first checking whether those elements are None.
- If either replacement_list[i] or search_list[i] is None, calling len(...) raises a TypeError and the function crashes before it ever reaches the later logic that correctly ignores None entries (the later loops do check for None at lines 102–105).
- So even though the code intends to skip None search/replacement pairs, it crashes earlier in that size-estimate loop.

Where it occurs (exact lines/variables)
- The problematic loop is the one starting just before line 83:
  for i in range(search_length):
      greater = len(replacement_list[i]) - len(search_list[i])
      if greater > 0:
          increase += 3 * greater  # assume 3 matches
  (lines 83–85 in the slice)
- The crash happens on len(replacement_list[i]) or len(search_list[i]) when replacement_list[i] or search_list[i] is None.

Why it happens (step-by-step)
1. Code validates top-level lists (not None) and equal length, but allows None entries inside those lists.
2. The size-estimate loop assumes each element is a str and calls len(...) on each element.
3. If an element is None, len(None) raises TypeError (object of type 'NoneType' has no len()), causing the function to fail.
4. Later loops that actually perform replacements do check for None (lines 102–105), so if the size-estimate loop were skipped or made null-safe, the function would behave as intended.

How to fix it (concrete steps)
Option A — Minimal fix: skip None/empty elements in the size-estimate loop using the same conditions as the later loops:

Replace the size-estimate loop with:

for i in range(search_length):
    if (
        no_more_matches_for_repl_index[i]
        or search_list[i] is None
        or len(search_list[i]) == 0
        or replacement_list[i] is None
    ):
        continue
    greater = len(replacement_list[i]) - len(search_list[i])
    if greater > 0:
        increase += 3 * greater  # assume 3 matches

Option B — Pre-filter lists: build filtered lists of only valid (search, replacement) pairs up-front, then operate only on those lists (removes the need to repeatedly check in loops).

Option C — Defensive: wrap len() calls in a helper that treats None as length 0, but that is less clear than skipping invalid pairs.

What to test after the fix
- Provide search_list/replacement_list that include None entries: function should no longer crash and should ignore those pairs.
- Provide empty strings in search_list (should be ignored).
- Validate normal behavior with only valid strings remains unchanged.

Summary
The crash is caused by calling len(...) on None in the size-estimate loop (lines 83–85). Make that loop null-safe by skipping entries where search_list[i] is None or empty or replacement_list[i] is None (the same checks used later), or filter out invalid pairs beforehand.
