What happened (symptom)
- The function replace_each crashes with a TypeError when either search_list or replacement_list contains None. The crash happens even though later code explicitly skips None entries — so callers expect None entries to be ignored, not to crash.

Where it happens (exact location)
- The crash is in the loop that computes increase (around the second for i in range(search_length) — shown in your slice as lines 81–82).
- Specifically this line causes the error:
  greater = len(replacement_list[i]) - len(search_list[i])
  If replacement_list[i] or search_list[i] is None, len(...) raises TypeError: object of type 'NoneType' has no len().

Why it happens (root cause)
- Earlier in the function you correctly guarded each element when searching (you skip index i when search_list[i] is None, empty, or replacement_list[i] is None). But the "increase" loop does not apply those same guards and unconditionally calls len(...) on both arrays' elements. If any element is None the len() call raises.
- In short: the code assumes every array element is a non-None string when computing increase, but that assumption is false. The failure is a direct result of calling len(None).

How to fix it (concrete steps)
1. Apply the same null/empty checks used elsewhere before using len(...) in the increase loop. Concretely, skip an index if:
   - search_list[i] is None
   - len(search_list[i]) == 0
   - replacement_list[i] is None
2. Then only compute greater when both strings are present.

Patch example (replace the buggy loop):
- Replace this block:
    for i in range(search_length):
        greater = len(replacement_list[i]) - len(search_list[i])
        if greater > 0:
            increase += 3 * greater  # assume 3 matches
    increase = min(increase, len(text) // 5)
- With this guarded version:
    for i in range(search_length):
        if (
            search_list[i] is None
            or len(search_list[i]) == 0
            or replacement_list[i] is None
        ):
            continue
        greater = len(replacement_list[i]) - len(search_list[i])
        if greater > 0:
            increase += 3 * greater  # assume 3 matches
    increase = min(increase, len(text) // 5)

Why this fix is correct
- It prevents len(None) calls by skipping None entries exactly like the subsequent matching loops do, preserving the intended behavior of ignoring None/empty search or None replacement values and avoiding the crash.

Alternative (if you prefer defensive style)
- You could use explicit type checks (isinstance(..., str)) instead of len(...) checks, if you want to ensure only real strings are treated. But the guarded checks above match the existing function logic and are the simplest drop-in fix.
