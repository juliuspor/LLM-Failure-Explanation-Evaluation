What happened
- The failure occurs inside replace_each in the loop that computes increase (the "guess on the size of the result buffer").
- That loop reads len(replacement_list[i]) and len(search_list[i]) without checking whether those array elements are None.
- If either replacement_list[i] or search_list[i] is None (as in the tests), calling len(...) raises TypeError: object of type 'NoneType' has no len(), and the function crashes before the rest of the null-safety checks are used.

Where it occurs (exact location in your code)
- In replace_each, the offending loop is this one:
  for i in range(search_length):
      greater = len(replacement_list[i]) - len(search_list[i])
      if greater > 0:
          increase += 3 * greater
- The loop happens after you validated overall list lengths but before you skip null/empty individual entries. You already have correct null/empty checks later (the matching loops), but you forgot to make this size-estimate loop null-safe.

Why this leads to the failing tests
- test_null_safety_inside_arrays_single calls replace_each("aba", ["a"], [None]). When the increase loop tries len(replacement_list[0]) it raises TypeError. The test expects the code to ignore None replacements and return the original string; instead the function crashes early.

Concrete fixes (step-by-step)
1. Make the increase loop skip entries that are None or have empty search strings — use the same guard conditions used later:
   - if search_list[i] is None or len(search_list[i]) == 0 or replacement_list[i] is None: continue
2. Keep the rest of the function unchanged. With that guard the increase computation will not call len(None).

Minimal code patch (replace the buggy loop with this)
    # BUGFIX: skip null/empty entries like the matching loops do
    for i in range(search_length):
        if search_list[i] is None or len(search_list[i]) == 0 or replacement_list[i] is None:
            continue
        greater = len(replacement_list[i]) - len(search_list[i])
        if greater > 0:
            increase += 3 * greater

Why this fix is correct
- It uses the same conditions already used in the matching loops, so behavior is consistent: None or empty search strings and None replacements are ignored.
- It prevents calling len(...) on None and preserves the intended optimization for estimating buffer size.

Optional alternative
- If you prefer, you can compute lengths with safe defaults (e.g., use 0 if value is None) instead of continuing. But that would be semantically different; skipping entries matches the rest of the function and the expected behavior in the tests.
