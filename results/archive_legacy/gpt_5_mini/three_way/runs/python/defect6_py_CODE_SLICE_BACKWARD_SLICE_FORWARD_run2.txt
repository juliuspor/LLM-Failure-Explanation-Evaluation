What goes wrong
- The bug is inside the loop that strips trailing zeros (lines 180–183 in your slice). You try to test whether removing another trailing zero and increasing the exponent would still reconstruct the original integer, using:
  check_val = (mantissa // 10) * (10 ** (exp + 1))
  if check_val == value:

- That expression is wrong. After you have reduced mantissa (mantissa //= 10) and increased exp once, you should be reconstructing the original value as mantissa * (10 ** exp). Instead your check computes (mantissa // 10) * (10 ** (exp + 1)), i.e. it divides mantissa again before multiplying. This compares the wrong quantity to value and makes the loop logic incorrect.

- Practically this causes the loop to either stop too early or to compute the wrong mantissa/exp pair. For some inputs it will produce an incorrect string (wrong scientific representation), and for other inputs it may behave unpredictably.

Where it happens (specific lines/variables)
- Line 181: while mantissa % 10 == 0:
- Line 182: check_val = (mantissa // 10) * (10 ** (exp + 1))
- Line 183: if check_val == value:
  You compare check_val to value (the original integer). The mistake is using mantissa // 10 and (exp + 1) in the same expression instead of using the current mantissa and current exp to reconstruct the original.

Why that is wrong (concrete example)
- Suppose value = 1000, mantissa starts at 1000, exp = 0.
  - First iteration: check_val = (1000 // 10) * 10**1 = 100 * 10 = 1000 → equal → you do mantissa //= 10 → mantissa = 100, exp = 1 (okay so far).
  - Second iteration: check_val = (100 // 10) * 10**2 = 10 * 100 = 1000 → equal → mantissa = 10, exp = 2.
  - Third iteration: check_val = (10 // 10) * 10**3 = 1 * 1000 = 1000 → equal → mantissa = 1, exp = 3.
  That sequence happens to work here, but the formula is fragile and logically inverted; for different sequences (or if the order of operations or timing changes) it will break. The intended invariant is: mantissa * (10**exp) == value. Your check should verify that after reducing mantissa by a factor of 10 and increasing exp by 1, the new mantissa*10**new_exp still equals value — not divide mantissa again inside the same check.

How to fix (clear step-by-step)
Option A — simpler, correct loop:
1. Start with value = int(x).
2. Let mantissa = value, exp = 0.
3. While mantissa % 10 == 0:
     - mantissa //= 10
     - exp += 1
   (No check_val needed; dividing by 10 while divisible by 10 is sufficient.)
4. Emit mantissa and exp if exp > threshold.

Option B — keep the “check and then change” style:
1. Compute check_val = mantissa * (10 ** exp).
2. If you want to test one more step without modifying mantissa yet:
     - candidate = (mantissa // 10) * (10 ** (exp + 1))  — but this is awkward.
   Better is to check whether mantissa % 10 == 0 and then do:
     - new_mantissa = mantissa // 10
     - new_exp = exp + 1
     - if new_mantissa * (10 ** new_exp) == value:
         mantissa = new_mantissa
         exp = new_exp
     - else: break

Concrete replacement (short):
Replace the whole inner block with:

    value = int(x)
    mantissa = value
    exp = 0
    while mantissa % 10 == 0:
        mantissa //= 10
        exp += 1

This preserves the invariant mantissa * 10**exp == value and avoids the incorrect check_val logic.

Why this fix is safe
- It directly removes factors of 10 from the integer until no more exist; it does not rely on a fragile reconstructed equality and it matches the intent of printing mantissa + "E" + exp.
