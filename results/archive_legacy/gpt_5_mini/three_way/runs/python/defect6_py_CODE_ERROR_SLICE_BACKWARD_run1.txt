What happened (root cause)
- At line 176 your code tests if x is an exact integer: `if x == int(x) and ...`.
- For x = -0.0, `int(x)` is 0 and `x == int(x)` is True, so you go into the integer branch.
- `value = int(x)` sets `value` to 0 (positive zero) and you eventually do `self.add(str(value))`, which produces "0".
- The sign of -0.0 is lost because converting to `int` and then to `str` removes the negative sign bit.
- Also, earlier you try to detect negative inputs with `if x < 0 and prev == '-':` — that condition is False for -0.0 because `-0.0 < 0` is False. So you never handled negative zero there either.

Where it occurs (specific variables/lines)
- The problematic behavior originates at the integer check starting at line 176 (`if x == int(x) ...`).
- `value = int(x)` and `self.add(str(value))` are the exact operations that lose the negative zero sign.
- The earlier sign-check `if x < 0 and prev == '-':` (a few lines above) also fails to detect -0.0.

Why this is happening (simple explanation)
- Floating-point -0.0 compares equal to +0.0 and converting to int produces +0. Python comparisons and int conversion do not preserve the sign bit of negative zero, so you end up formatting a positive "0".

How to fix it (concrete steps)
1. Detect negative zero explicitly using math.copysign. Example check:
   - is_neg_zero = (x == 0.0 and math.copysign(1.0, x) < 0)
2. Use that check where you need to treat negative values differently:
   - Replace `if x < 0 and prev == '-':` with `if math.copysign(1.0, x) < 0 and prev == '-':` so -0.0 is handled there too.
3. In the integer branch, before converting to int, preserve sign for negative zero:
   - If is_neg_zero is True, output "-0" (or "-0.0" if you prefer the decimal form) instead of converting to int and losing the sign.
4. Optionally, compute `value = int(x)` only when not neg-zero.
