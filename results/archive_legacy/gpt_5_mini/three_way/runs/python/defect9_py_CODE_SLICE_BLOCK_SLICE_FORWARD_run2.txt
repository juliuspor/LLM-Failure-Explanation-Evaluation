What happened (what error, where it occurred)
- The function raises a TypeError at runtime on this line:
  - line 82: greater = len(replacement_list[i]) - len(search_list[i])
- The error is: "object of type 'NoneType' has no len()" when either replacement_list[i] or search_list[i] is None.

Why it happens (root cause)
- Earlier in the function you explicitly skip None/empty search strings and None replacements when looking for matches (see the if-block at lines ~34–45 and repeated at ~100–106). Those checks prevent calling .find or computing lengths there.
- The loop at lines 81–84 (the “get a good guess on the size of the result buffer” loop) does not perform the same None/empty checks before calling len(...) on both arrays. If either array contains a None element, len(None) raises the TypeError and the function fails.
- In short: you call len(...) on list elements that may be None without guarding them.

Exactly where to change (concrete fixes)
1. Add the same guards used elsewhere before computing lengths. For example, replace the loop at lines 81–84 with:
   for i in range(search_length):
       if (
           no_more_matches_for_repl_index[i]
           or search_list[i] is None
           or len(search_list[i]) == 0
           or replacement_list[i] is None
       ):
           continue
       greater = len(replacement_list[i]) - len(search_list[i])
       if greater > 0:
           increase += 3 * greater

2. Alternatively, explicitly check for None/empty before calling len:
   for i in range(search_length):
       if replacement_list[i] is None or search_list[i] is None or len(search_list[i]) == 0:
           continue
       ...

3. If you want defensive safety, you can wrap len calls:
       s = search_list[i]
       r = replacement_list[i]
       if not s or r is None:
           continue
       greater = len(r) - len(s)
   (Note: "not s" treats empty string as false and will skip it, which matches current behavior.)

Tests to add
- A unit test where search_list contains None (e.g. search_list = ["a", None], replacement_list = ["x", "y"]) — should not raise.
- A unit test where replacement_list contains None — should not raise.
- Tests for empty search strings in search_list — behavior should remain unchanged.

Why this fix is safe
- It makes the sizing loop consistent with the match-search loops that already guard against None/empty entries.
- It prevents len(None) from being called while preserving the intended heuristic to estimate increase only for valid pairs.

Summary
- Root cause: unguarded len(...) calls on potentially None list elements at lines 81–84.
- Fix: add the same None/empty checks used elsewhere (or equivalent checks) before computing lengths.
