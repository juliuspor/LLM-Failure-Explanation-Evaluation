What happens
- At line 22 (the for loop: for pt in range(consumed):) the code calls code_point_at(code_units, pos) each iteration.
- code_point_at checks the index and raises IndexError if index >= len(code_units) (see lines 35–36 in the code slice).
- During the loop pos is incremented inside the loop by char_count(...). After one iteration pos can become equal to length; the loop still runs because range(consumed) may have more iterations, so the next iteration calls code_point_at(code_units, pos) with pos == length and code_point_at raises IndexError ("String index out of range").

Why that is wrong (root cause)
- The loop assumes it is safe to call code_point_at(code_units, pos) consumed times without checking bounds.
- But pos is advanced in the body, so later iterations can pass an out-of-range index to code_point_at.
- The code_point_at implementation enforces bounds (lines 35–36), so the out-of-range call causes the crash.

Exactly where it fails
- The failure occurs inside the for pt in range(consumed) loop (the lines after the comment "Contract with translators ...") when code_point_at(code_units, pos) is called with pos >= len(code_units).
- The IndexError comes from code_point_at at the bounds check on line 35–36.
