What happened (root cause)
- The failure occurs in ArrayUtils.add when both the input array and the new element are None, but the caller passed expected_type=str. The code decides the array component type incorrectly and then raises a TypeError (mimicking a Java ClassCastException).
- Specifically:
  - At line 336 the code uses type(array) when array is not None. type(array) returns the list class (e.g. <class 'list'>), not the element type. That is wrong for inferring element/component type.
  - At lines 337–340 the code tries fallbacks: if element is not None it uses type(element), else it sets inferred_type = object (line 340) when both are None.
  - At lines 346–351 there is a guard: if inferred_type == object and expected_type != object: raise TypeError(...). Because both array and element are None, inferred_type == object, expected_type is str, so the guard fires and raises the error. That is exactly the behavior your test demonstrates (it expects [None], but the method raises).

- Decision that sets inferred_type to object happens at line 340.
- The exception is thrown at lines 346–351.
- The incorrect use of type(array) (line 336) means even when array is non-empty you are not getting the element type (you get the list class), so type inference logic is flawed more generally.

Why that logic is wrong
- type(array) gives the container type (list), not the type of elements in the container. To infer an element/component type you must inspect elements (e.g., array[0] or all elements), or accept the explicit expected_type provided by the caller.
- When both array and element are None, you have no information to infer element type; the correct behavior when expected_type is provided is to use expected_type, not to treat lack of information as an error. Raising an exception there prevents valid use cases such as creating a new list of None values typed as strings.
