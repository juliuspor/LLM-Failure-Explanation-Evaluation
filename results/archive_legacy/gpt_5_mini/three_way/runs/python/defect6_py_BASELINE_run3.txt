What happened (root cause)
- At line 176 you treat any float equal to its int conversion as an integer: if x == int(x) ...
- For x == -0.0 that comparison is True (because -0.0 == 0).
- At line 177 you set value = int(x) which becomes 0 (sign lost).
- Later, at line 192 you call self.add(str(value)) so you output "0".
- The code never hits the else branch at line 194 where you would have used str(x) (which would preserve the "-0.0" text), and you never check negative-zero explicitly.
- Net result: the sign-bit on -0.0 is discarded by int(x) and you output "0" instead of the required "-0.0".

Where it occurs (exact lines)
- Problem check: line 176: if x == int(x) and not math.isinf(x) and not math.isnan(x):
- Loss of sign: line 177: value = int(x)
- Wrong output: line 192: self.add(str(value))
- Alternative branch that would preserve sign: line 194: self.add(str(x)) — but it never runs for -0.0 because -0.0 == int(-0.0).

Why this matters
- IEEE-754 negative zero exists and equals positive zero numerically, but the sign bit can be important for source-to-source printing. The integer conversion int(x) removes that sign information.

Concrete fixes (step-by-step)
Option A — special-case negative zero first (recommended)
1. At the beginning of add_number, check CodeConsumer.is_negative_zero(x) (you already have that utility in tests).
2. If True, call self.add("-0.0") and return immediately before doing the integer check.

Example (exact spot to add):
- Insert just before line 176:
    if self.is_negative_zero(x):
        self.add("-0.0")
        return

Option B — modify the integer branch guard
1. Keep the integer formatting branch but refuse to treat negative zero as an integer:
- Change line 176 to:
    if not self.is_negative_zero(x) and x == int(x) and not math.isinf(x) and not math.isnan(x):

This will let -0.0 fall through to the else branch and be printed with str(x) (which yields "-0.0").

Option C — use math.copysign if you prefer no helper
1. Detect negative zero with math.copysign(1.0, x) < 0 and handle as in Option A:
    if x == 0.0 and math.copysign(1.0, x) < 0:
        self.add("-0.0")
        return

Which to choose
- Option A is simplest and most explicit: add the negative-zero special-case immediately and return. It uses the existing is_negative_zero utility and makes the intention clear.
- Option B is also acceptable if you prefer changing a single condition instead of adding an early return.

No other parts of the number-handling logic need to change for this bug; the fix is just to detect and preserve negative zero before calling int(x) or choosing the integer-formatting path.
