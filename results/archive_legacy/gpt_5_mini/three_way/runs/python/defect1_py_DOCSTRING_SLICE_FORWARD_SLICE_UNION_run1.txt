What went wrong
- Root cause: the code computes the total offset minutes incorrectly when both hours_offset and minutes_offset are negative.
- Where it happens: line 402 in your slice. That line does
  minutes_offset = hours_in_minutes - minutes_offset
  but at that point minutes_offset is negative (e.g. -15) for the (-2, -15) case.
- Why that fails: subtracting a negative minute value makes the total less negative instead of more negative, so the resulting offset is wrong.

Concrete example (shows the bug)
- Input: hours_offset = -2, minutes_offset = -15
- hours_in_minutes = hours_offset * 60 = -120
- Code at line 402 executes: minutes_offset = hours_in_minutes - minutes_offset
  => minutes_offset = -120 - (-15) = -105
- -105 minutes = -01:45, but the expected offset for (-2, -15) is -02:15 (-135 minutes).
- So the code computes -01:45 instead of -02:15.

Why the current expression is wrong
- The code assumes minutes_offset at that point should be added to hours_in_minutes, but it uses subtraction unconditionally.
- When minutes_offset is negative, subtracting it increases the total (moves toward zero) rather than decreasing it further into the negative, which is the intended effect for negative minutes with a negative hour.

Where to change it and how to fix it (concrete steps)
1. Replace the problematic line (402) with addition, so the total minutes are computed as:
   minutes_total = hours_in_minutes + minutes_offset
   (use a new variable name minutes_total or overwrite minutes_offset if you prefer)
   This works for all sign combinations because minutes_offset already carries its sign.
2. Keep the existing range checks (line 396) that validate minutes_offset is in -59..+59 and that disallow negative minutes when hours are positive — those checks are fine.
3. Use minutes_total for the subsequent multiplication:
   offset = FieldUtils.safe_multiply(minutes_total, DateTimeConstants.MILLIS_PER_MINUTE)
4. Add unit tests that cover:
   - Positive hour + positive minute (2,15) => +02:15
   - Positive hour + negative minute should still raise (2,-15) => error
   - Zero hour + negative minute (0,-15) => -00:15
   - Negative hour + positive minute (-2,15) => -02:15
   - Negative hour + negative minute (-2,-15) => -02:15 (the failing case)

Why this fix is safe
- If minutes_offset is positive, addition and the original subtraction-then-branching logic that produced addition anyway will match expected behavior.
- If minutes_offset is negative, addition correctly moves the total further negative.
- Using hours_in_minutes + minutes_offset handles all allowed sign combinations without conditionally negating or subtracting the minute value.

In short: line 402 subtracts an already-negative minutes_offset, which produces the wrong sign/duration. Replace that subtraction with addition (hours_in_minutes + minutes_offset) and use that total when converting to milliseconds.
