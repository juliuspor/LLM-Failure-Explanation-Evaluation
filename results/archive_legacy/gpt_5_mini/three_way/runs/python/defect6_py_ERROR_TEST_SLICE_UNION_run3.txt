What happened (root cause)
- At line 176 the code tests if x is an integer: if x == int(x) ...
- For x == -0.0 that comparison is True because -0.0 == 0.0, and int(-0.0) yields 0.
- The code then sets value = int(x) (value == 0) and later reaches line 192 which does self.add(str(value)), so it outputs "0".
- That loses the negative sign of negative zero. The test expects "-0.0", so the assertion fails.

Where it occurs (exact places)
- The problematic logic is in the integer-handling branch shown in the slice: lines 176–194.
  - Line 176: if x == int(x) and not math.isinf(x) and not math.isnan(x):
  - Line 177: value = int(x)
  - ... (logic to decide scientific notation)
  - Line 192: self.add(str(value))  <-- prints "0" for -0.0
- There is also a fallback at line 194: self.add(str(x)), but that is not reached for -0.0 because the integer branch already handled it.

Why this loses the sign
- In Python -0.0 compares equal to 0.0, and int(-0.0) is 0. Converting that integer back to string produces "0".
- The code never checks whether x was negative zero before converting to int, so it cannot preserve the sign.

How to fix it (clear, actionable steps)
1. Detect negative zero before converting to int. Use the existing utility CodeConsumer.is_negative_zero(x) or math.copysign to detect the sign of zero. Example condition:
   - if CodeConsumer.is_negative_zero(x):
         self.add("-0.0")
         return
2. Place that check at the top of the integer branch (immediately before line 176) or even earlier in add_number so negative zero is handled first, e.g.:

   if CodeConsumer.is_negative_zero(x):
       # preserve negative zero sign explicitly
       self.add("-0.0")
       return

3. If you prefer to keep a single code path, you can also special-case when value == 0 after value = int(x):
   - After line 177:
         if value == 0 and math.copysign(1.0, x) < 0:
             self.add("-0.0")
             return

4. Add tests (already present) to ensure:
   - CodeConsumer.is_negative_zero(-0.0) returns True
   - add_number(-0.0) yields "-0.0"

Why these fixes work
- They detect that x was negative zero before the integer conversion that destroys the sign, and explicitly output "-0.0". That preserves both the numeric formatting the code wants for integer-like numbers and the semantic sign information for negative zero.

Notes and caveats
- Do not rely on str(x) at the end of the function for negative zero; str(-0.0) yields "-0.0" but that code path is not executed for integer-like values because of the integer branch.
- Keep this special-case small and early so it doesn't interfere with other formatting decisions (scientific notation logic, etc.).
