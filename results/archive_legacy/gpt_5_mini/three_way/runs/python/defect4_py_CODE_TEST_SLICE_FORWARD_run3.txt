What happened
- The code that updates self._max_middle_index reads the wrong existing index when it fetches the period to compare against:
  - In the "max middle" block you call:
    max_middle_period = self.get_data_item(self._min_middle_index).get_period()
  - That uses self._min_middle_index instead of self._max_middle_index. Because of this, max_middle is computed from the period at the min-middle index, not the current max-middle index.

Why it fails (root cause)
- The comparison that decides whether to set self._max_middle_index compares the new middle to the wrong reference value (min_middle). So if min_middle and max_middle are different, the decision can be wrong and self._max_middle_index can be set incorrectly (or left unchanged incorrectly).
- This is exactly why the test shows expected get_max_middle_index() == 1 but the code returns 3: the code compared the new period's middle to the period at index self._min_middle_index, producing the wrong outcome.

Where it occurs
- In this snippet (the incorrect block):
  if self._max_middle_index >= 0:
      max_middle_period = self.get_data_item(self._min_middle_index).get_period()
      ...
      if middle > max_middle:
          self._max_middle_index = index

- The bug is the use of self._min_middle_index on the first line above; it should be self._max_middle_index.

Exact fix (minimal, concrete change)
Replace that one line:
  max_middle_period = self.get_data_item(self._min_middle_index).get_period()
with:
  max_middle_period = self.get_data_item(self._max_middle_index).get_period()

Full corrected block for clarity:
  if self._max_middle_index >= 0:
      max_middle_period = self.get_data_item(self._max_middle_index).get_period()
      if isinstance(max_middle_period, SimpleTimePeriod):
          s = max_middle_period.get_start_millis()
          e = max_middle_period.get_end_millis()
      else:
          s = int(max_middle_period.get_start().timestamp() * 1000)
          e = int(max_middle_period.get_end().timestamp() * 1000)
      max_middle = s + (e - s) // 2
      if middle > max_middle:
          self._max_middle_index = index
  else:
      self._max_middle_index = index

Suggested small defensive improvement (optional)
- Add an assertion or logging that the index you fetch is >= 0, to help catch index-mixups in future:
  assert self._max_middle_index >= 0
  before using it (or check and raise a clear error).

Why this fix solves the test failures
- Using self._max_middle_index ensures you compare the new period's middle to the current maximum-middle period, so the comparison and potential update are based on the intended reference and will produce the expected index values in your tests.
