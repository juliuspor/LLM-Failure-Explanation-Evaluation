- That happens because minutes_offset sometimes ends up outside the valid range 0–59 before that check.
- The nearby code that produces minutes_offset is at lines 402 and 404:
  - 402: minutes_offset = hours_in_minutes - minutes_offset
  - 404: minutes_offset = hours_in_minutes + minutes_offset
- Because those lines mix hours_in_minutes (an hours*60 value) and minutes_offset (already in minutes), minutes_offset becomes much larger than 59 (or negative), triggering the ValueError at line 397 when the code validates minutes.

Why it happens (root cause)
- The code tries to combine hour and minute parts into a single minute offset, but it subtracts or adds hours_in_minutes directly to minutes_offset instead of combining normalized minute parts.
- hours_in_minutes holds something like abs(hours) * 60. minutes_offset holds the minutes component (0..59 expected).
- Doing hours_in_minutes + minutes_offset yields a value like (abs(hours)*60 + minutes), e.g. 2 hours + 30 minutes → 150. That value is >59, so the subsequent validation thinks the minute component is out of range and raises the exception.
- In short: the code is treating a combined total-minute quantity as if it were the minute-of-hour component, then validating it as if it should be in 0–59.

What exactly to change (concrete fixes with steps and lines)
1. Keep minutes_offset strictly as the minute-of-hour value (0..59) before validating it.
2. Compute the total offset (in minutes) separately from the minute-of-hour component, then convert to milliseconds.
