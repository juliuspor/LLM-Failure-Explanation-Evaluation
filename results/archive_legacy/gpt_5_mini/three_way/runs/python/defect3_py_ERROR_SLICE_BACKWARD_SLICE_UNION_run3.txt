What happened (where it failed)
- The IndexError was raised at lines 35–36:
  - The code checks: if index < 0 or index >= len(code_units): raise IndexError(f"String index out of range: {index}")
  - That check fired because index was 2 and len(code_units) was <= 2, so index >= len(code_units) was true.

Why it happened (root cause)
- The code is working with UTF‑16 code units in the list code_units, but index was computed or supplied in a different coordinate system (most likely “Python string character” / Unicode code point indexes).
- UTF‑16 represents some Unicode characters (outside the Basic Multilingual Plane) as two 16‑bit code units (a surrogate pair). That means len(code_units) can be different from len(s) and from other index meanings.
- In short: index is too large for the code_units list because the caller computed index in the wrong unit (code points or original string indices) instead of in UTF‑16 code units expected here.

Where in the code this mismatch matters
- to_utf16_code_units builds code_units (line 12).
- The guard at lines 35–36 compares index against len(code_units) and raises.
- Later code (lines 38–41) assumes index points to a code unit and may access code_units[index+1] to check for a surrogate pair.

Concrete steps to fix it (clear actionable changes)
1. Decide what index means in your API:
   - If index should be a UTF‑16 code unit index, the caller must supply that. Document it explicitly.
   - If index should be a Python string index (code point index), convert it to a UTF‑16 code unit index before using code_units.

2. If you want the function to accept a Python string index (recommended for Python callers), add conversion from code point index to code unit index before the range check. Example algorithm:
   - Walk the Python string s up to the given code point index, counting how many UTF‑16 code units each character produces (1 for BMP code points, 2 for code points >= 0x10000).
   - Use that count as the code_unit_index and compare against len(code_units).

3. Alternatively, if index is provided externally and should never exceed len(code_units)-1, add validation where index is computed (caller side) so it uses the same unit system as code_units.

4. Add a clear error message and documentation: state whether index is a UTF‑16 code unit index or a code point index. That prevents future confusion.

Example (brief sketch of conversion for step 2)
- To convert a Python string code point index i to a UTF‑16 code unit index:
  - code_unit_index = 0
  - for each character ch in s[:i]:
      if ord(ch) >= 0x10000:
          code_unit_index += 2
      else:
          code_unit_index += 1

- Then use code_unit_index in the existing guard (lines 35–36) and accesses (lines 38–41).

Summary
- The IndexError is produced because index (value 2) does not match the length of code_units: you mixed up code point/string indexes with UTF‑16 code unit indexes. Fix by choosing one index convention and converting indices to the code_units coordinate system before checking or accessing code_units. Document the choice.
