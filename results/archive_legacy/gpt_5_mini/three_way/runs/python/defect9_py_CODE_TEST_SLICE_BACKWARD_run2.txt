What happened (what fails)
- The function crashes with a TypeError when any element of search_list or replacement_list is None.
- The crash happens in the "estimate result size" loop (around lines shown in the slice, roughly these lines in replace_each):

      for i in range(search_length):
          greater = len(replacement_list[i]) - len(search_list[i])
          if greater > 0:
              increase += 3 * greater

- If replacement_list[i] or search_list[i] is None, calling len(None) raises TypeError: object of type 'NoneType' has no len().

Why it happens (root cause)
- The code later (in the search loops) explicitly skips entries when search_list[i] is None, search_list[i] is empty, or replacement_list[i] is None:
  
      if no_more_matches_for_repl_index[i] or search_list[i] is None or len(search_list[i]) == 0 or replacement_list[i] is None:
          continue

  That shows the function is intended to be null-safe for elements inside the arrays.
- But the earlier "increase" loop does not perform the same null checks before calling len(), so it assumes every element is a non-null string. That assumption is false for the failing tests where replacement_list contains None; len(None) triggers the TypeError before the function can skip the null entries.

Where it occurs (specific variables / lines)
- In replace_each, the failing variables are replacement_list[i] and search_list[i] used inside the increase loop.
- Exact failing expression: greater = len(replacement_list[i]) - len(search_list[i])
- This is executed before the later null-safe loops, so the function blows up on inputs like replacement_list = [None].

1. Guard the increase calculation with the same null/empty checks already used later. For example:

      for i in range(search_length):

              continue
          greater = len(replacement_list[i]) - len(search_list[i])
          if greater > 0:
              increase += 3 * greater

2. Alternatively, check both values are strings before using len():
