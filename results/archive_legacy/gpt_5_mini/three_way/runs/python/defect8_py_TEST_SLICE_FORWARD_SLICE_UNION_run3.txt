What happened (where it fails)
- The failing input is "fr__POSIX".
- The code enters to_locale with locale_str = "fr__POSIX" and length = 9.
- In the branch that handles longer strings the code reads:
  - ch3 = locale_str[3]
  - ch4 = locale_str[4]
  - then it checks whether ch3 and ch4 are uppercase letters (lines 237–240).
- For "fr__POSIX" the characters are:
  - locale_str[0] = 'f', [1] = 'r', [2] = '_', [3] = '_', [4] = 'P', ...
- The code expects a country code at positions 3–4 (uppercase letters) and so ch3 ('_') fails the test ch3 >= 'A' and ch3 <= 'Z' and the code raises ValueError("Invalid locale format: fr__POSIX") at line 239–240.

Root cause (concise)
- The function assumes that if the input is longer than a language-only form it must contain a 2-letter country code at indices 3–4 and validates those chars immediately.
- It does not account for the valid "language__variant" form where the country is empty and two consecutive underscores appear (positions 2 and 3 are '_'). In that case position 3 is '_' (not an uppercase letter), so the uppercase check incorrectly fails and raises the ValueError.

Exactly where to change (what to change, step-by-step)
1. Before accessing and validating ch3 and ch4 (before lines 237–240), check for the empty-country pattern: locale_str[2] == '_' and locale_str[3] == '_'.
2. If that pattern matches, return Locale(locale_str[0:2], "", locale_str[4:]) (variant begins at index 4).
3. Only after ruling out the double-underscore case should you validate that positions 3 and 4 are uppercase letters and proceed to construct country/variant locales.

Concrete patch idea (pseudo-diff showing place & logic)
- Find the branch that executes for length >= 5 (the code that sets ch3/ch4 and checks them).
- Insert before ch3 = locale_str[3]:
    if locale_str[2] == '_' and locale_str[3] == '_':
        return Locale(locale_str[0:2], "", locale_str[4:])
- Then continue with the existing ch3/ch4 checks for the normal language_country(...) and language_country_variant(...) cases.

Why this fix works
- It explicitly handles the valid "language__variant" format so the code does not try to validate '_' as if it were a country-letter.
- It preserves existing validation for the normal language_country and language_country_variant forms.

One-line alternative (if you prefer minimal change)

        raise ValueError(...)
  but the explicit double-underscore check above is clearer and safer.
