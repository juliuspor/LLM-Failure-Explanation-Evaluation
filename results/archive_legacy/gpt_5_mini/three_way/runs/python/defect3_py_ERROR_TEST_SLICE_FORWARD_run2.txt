What happened (what failed)
- The test input "\U0001F630A" ends up causing code_point_at (the function shown in the slice) to be called with index equal to len(code_units). That makes the first access on line 38:
  high = code_units[index]
  raise IndexError: "String index out of range: 2" (index 2 in a length-2 sequence).

Why it happened (root cause)
- The function assumes the caller never passes an out-of-range index, so it reads code_units[index] before checking bounds.
- The bounds check on line 39 only protects the read of the low surrogate:
  if 0xD800 <= high <= 0xDBFF and index + 1 < len(code_units):
    low = code_units[index + 1]
  But that does not prevent the initial access high = code_units[index] when index == len(code_units).
- In this test the translate loop (caller) advanced the index incorrectly when handling surrogate pairs. That loop let index reach len(code_units) and still called code_point_at. So the immediate cause is the out-of-range call; the underlying cause is the caller advancing index past the end (off-by-one when consuming surrogate pairs).
- Concretely: for input consisting of a surrogate pair plus another character, the translate-loop consumed the pair, advanced the index by 2, and then the loop logic advanced it again (or otherwise called code_point_at with index == len). That produced the attempt to read code_units[2] when valid indices are 0 and 1.

- The crash occurs at line 38 in the shown slice (high = code_units[index]) inside code_point_at (or whatever function this slice is from).
- The unsafe check is on line 39: the check for index+1 < len(code_units) happens too late (after the access).
