What happened (where)
- The crash happens in the "guess size" loop at lines 81–84:
  > greater = len(replacement_list[i]) - len(search_list[i])

- If either replacement_list[i] or search_list[i] is None, calling len(...) raises TypeError: "object of type 'NoneType' has no len()".

Why it happens (root cause)
- Earlier, the code correctly guards against None/empty entries when scanning for matches (the for-loop starting at line 59 and the inner loop later around line 102). Those loops skip entries when search_list[i] is None or empty or replacement_list[i] is None.
- The "guess size" loop (lines 81–84) does not perform the same null-safety checks before calling len(...) on the array elements. So when a replacement is None (or a search element is None), len(...) is invoked on None and Python raises TypeError.
- In short: the code assumes all array elements are non-None for that size-estimation loop, but the API allows None entries which should be ignored.

Exactly where to change (concrete fixes)
1. Add the same guard used elsewhere before using len(...) in the increase loop. Replace the loop at lines 81–84 with something like:

    for i in range(search_length):
        if (
            no_more_matches_for_repl_index[i]
            or search_list[i] is None
            or len(search_list[i]) == 0
            or replacement_list[i] is None
        ):
            continue
        greater = len(replacement_list[i]) - len(search_list[i])
        if greater > 0:
            increase += 3 * greater  # assume 3 matches

This ensures you never call len(...) on None.

2. Alternatively, compute lengths defensively:

    s = search_list[i]
    r = replacement_list[i]
    if not s or r is None:
        continue
    greater = len(r) - len(s)
    ...

Either approach matches the existing null/empty checks already used elsewhere in the function and prevents the TypeError.

Why this fix is correct
- It keeps the size-guessing optimization but only considers valid (non-None, non-empty) search/replacement pairs, matching the function’s intended behavior (ignore None replacements) and avoiding len(None).
