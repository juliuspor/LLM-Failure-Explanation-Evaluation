What happened (where, and what went wrong)
- The bug is in the accumulation of hours and minutes into total minutes at lines 402–404 of the shown slice:
  - hours_in_minutes = hours_offset * 60
  - if hours_in_minutes < 0:
        minutes_offset = hours_in_minutes - minutes_offset
    else:
        minutes_offset = hours_in_minutes + minutes_offset
- The problem is the subtraction on the first branch (hours_in_minutes - minutes_offset). That produces the wrong result when both hours_offset and minutes_offset are negative.

Concrete example that shows the wrong arithmetic
- Input: hours_offset = -2, minutes_offset = -15
- hours_in_minutes = -2 * 60 = -120
- Code does minutes_offset = hours_in_minutes - minutes_offset = -120 - (-15) = -105
  - This yields total minutes = -105 -> total millis = -105 * 60000 which corresponds to -01:45, not the expected -02:15.
- The correct total minutes for (-2, -15) is -120 + (-15) = -135 -> -02:15.

Why the code produced the wrong sign
- The code treats minutes_offset as a value to subtract when hours_in_minutes is negative. Subtracting a negative minutes_offset flips its sign (double negation), reducing the magnitude instead of increasing it.
- The arithmetic should combine hours_in_minutes and minutes_offset by addition (hours_in_minutes + minutes_offset) to produce the true total minutes regardless of sign.

Where to fix it (exact lines and what to change)
- Replace the conditional logic at lines 402–404 with a single, correct combination:
  - Compute total_minutes = hours_in_minutes + minutes_offset
  - Then use total_minutes to compute offset:
        offset = FieldUtils.safe_multiply(total_minutes, DateTimeConstants.MILLIS_PER_MINUTE)
- That fixes the incorrect subtraction that produced -105 instead of -135 for (-2, -15).

Additional note about input validation (tests expect this)
- The tests expect certain sign handling and also expect a ValueError for mixed-sign inputs like (2, -15). The current code does not explicitly reject mixed-sign pairs. After fixing the arithmetic, also add explicit validation before combining:
  - If hours_offset > 0 and minutes_offset < 0: raise ValueError("Mixed sign not allowed")
  - If hours_offset < 0 and minutes_offset > 0: raise ValueError("Mixed sign not allowed")
  - (Keep allowing (0, ±minutes) and (±hours, ±minutes) according to your intended policy.)
- Put this validation before computing hours_in_minutes so callers get a clear ValueError for invalid combos.
