What fails
- Where: get_paint (lines shown); specifically the calculation at line 71 and the range check at line 74–75:
  - line 71: g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
  - lines 74–75: if g < 0 or g > 255: raise ValueError(...)
- What happens: g can be less than 0 or greater than 255 and trigger the ValueError on line 75. In addition, if self._upper_bound == self._lower_bound the code will raise a ZeroDivisionError at line 71 before you reach the check.

Why it happens (root causes)
1. value outside the bounds:
   - If value < self._lower_bound then (value - lower_bound) is negative → the fraction is negative → g < 0.
   - If value > self._upper_bound then the fraction > 1 → g > 255.
   The code only checks g after computing it; it does not validate value against lower/upper first.

2. invalid bounds (upper == lower):
   - If self._upper_bound == self._lower_bound the division at line 71 divides by zero, causing a ZeroDivisionError (this happens before the g range check).

3. unhelpful error message:
   - The raised ValueError text is "Color parameter outside of expected range: Red Green Blue" and does not include the actual offending values (g, value, lower_bound, upper_bound), so it is not informative.

How to fix (explicit steps)
1. Validate bounds in __init__:
   - In __init__, reject or fix equal/invalid bounds. Example:
     - if upper_bound <= lower_bound: raise ValueError(f"upper_bound must be > lower_bound; got lower={lower_bound}, upper={upper_bound}")
   This prevents the ZeroDivisionError.

2. Validate or clamp value before scaling:
   - Option A (reject out-of-range input): at the start of get_paint, check
     - if value < self._lower_bound or value > self._upper_bound: raise ValueError(f"value {value} outside [{self._lower_bound}, {self._upper_bound}]")
   - Option B (clamp into range and continue): clamp value to the interval before computing g:
     - value = min(max(value, self._lower_bound), self._upper_bound)

3. Compute and ensure g in [0,255]:
   - After computing the scaled float, convert and clamp to integer:
     - ratio = (value - self._lower_bound) / (self._upper_bound - self._lower_bound)
     - g = int(round(ratio * 255))
     - g = max(0, min(255, g))

4. Improve the error message:
   - If you still raise ValueError, include the actual values: raise ValueError(f"Computed color component out of range: g={g}, value={value}, lower={self._lower_bound}, upper={self._upper_bound}")

Summary
- The immediate failure comes from computing g from a value that lies outside the configured [lower_bound, upper_bound] (producing g < 0 or > 255), or from dividing by zero when upper_bound == lower_bound. Fixes: validate bounds in __init__, validate or clamp value in get_paint before scaling, and clamp g + improve error messages so the failure is explicit and actionable.
