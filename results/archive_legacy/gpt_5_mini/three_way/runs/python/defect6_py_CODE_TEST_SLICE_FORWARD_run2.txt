What happened (where in the code)
- Root cause: at line 177 you do value = int(x). That converts -0.0 to 0 and thus loses the negative zero sign.
- Because of that, the branch at line 176–193 that treats integer-valued floats runs for -0.0 (x == int(x) is True for -0.0). The code then ends up calling self.add(str(value)) at line 192 and writes "0" instead of "-0.0".
- The negative-zero case never reaches the fallback else that would call self.add(str(x)) (line 194), which would have preserved "-0.0".

Why this matters for the test
- Test test_negative_zero expects "-0.0".
- Converting to int too early (line 177) strips the sign bit of negative zero and you produce "0", so the assertion fails.

Exact variables / lines referenced
- x: the incoming float (can be -0.0)
- line 176: branch condition that allows integer-valued floats to be treated specially
- line 177: value = int(x) — this is where -0.0 becomes 0
- line 192: self.add(str(value)) — prints "0"
- line 194: self.add(str(x)) — would have printed "-0.0" but is not reached for -0.0

What to change (clear steps)
1. Detect negative zero before you convert x to int. Add an early check like:
   - if CodeConsumer.is_negative_zero(x): self.add("-0.0"); return
   or
   - if math.copysign(1.0, x) < 0 and x == 0.0: self.add("-0.0"); return
2. Alternatively, change the integer branch condition so it excludes negative zero:
   - replace if x == int(x) ... with if x == int(x) and not self.is_negative_zero(x):
     then negative zero will fall through to the float printing path and keep its sign.
3. Ensure your is_negative_zero utility inspects the sign bit (e.g. math.copysign or struct.unpack) and is used before int(x) conversion.

Minimal example fix (conceptual)
- At start of add_number:
    if self.is_negative_zero(x):
        self.add("-0.0")
        return

Why these fixes work
- They avoid calling int(x) on -0.0, so the sign bit is preserved and "-0.0" is emitted as expected by the tests.

Other notes
- The rest of your integer-handling logic (mantissa/exp) is fine once negative zero is excluded; the problem is purely the premature int() conversion.
