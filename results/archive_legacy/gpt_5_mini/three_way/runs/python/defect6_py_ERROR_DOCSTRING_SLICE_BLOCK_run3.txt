What happened (where and why)
- The code enters the integer branch at line 176: if x == int(x) and not math.isinf(x) and not math.isnan(x):. For x = -0.0 that condition is true because -0.0 == 0.
- On line 177 you do value = int(x). int(-0.0) produces 0 and loses the negative sign bit. That makes value == 0 and mantissa == 0 (lines 178–179).
- Later, because exp <= 2, you call self.add(str(value)) on line 192. str(0) is "0", so the sign is lost and the output is "0" instead of "-0.0".

Root cause summary
- Converting -0.0 to int at line 177 discards the sign of negative zero; the code never checks the float sign before converting, so negative zero is treated as plain zero.

Exact place to fix
- Add a sign-preservation check before int(x) (i.e., before line 177). Use math.copysign or an equivalent to detect negative zero.

Concrete fixes (copy/paste-ready)
Option A — preserve "-0.0" immediately and return:
- Insert just after line 176:
    if x == 0.0 and math.copysign(1.0, x) < 0:
        self.add("-0.0")
        return
  This prevents int(x) from being called and preserves the negative-zero representation.

Option B — preserve sign while continuing integer formatting:
- Insert just after line 176:
    neg_zero = (x == 0.0 and math.copysign(1.0, x) < 0)
  Then after computing value (line 177) and before self.add(...):
    if neg_zero:
        self.add("-0.0")
        return
  Or, if you want to include sign into value string generation, build sign = "-" if neg_zero else "" and use self.add(sign + str(value)) on line 192 (and also handle the mantissa branch on line 190).

Which to use
- Use Option A if you want a simple explicit representation "-0.0".
- Use Option B if you prefer to keep the rest of the integer-formatting logic but still preserve a negative zero output.

Why these fixes work
- math.copysign(1.0, x) returns -1.0 for negative-zero floats, so checking x == 0.0 and math.copysign(1.0, x) < 0 detects negative zero before int() erases its sign bit.

No other parts of the code need to change for this bug — the loss of sign happens at the int(x) call on line 177, and handling the sign before that call fixes the failing test.
