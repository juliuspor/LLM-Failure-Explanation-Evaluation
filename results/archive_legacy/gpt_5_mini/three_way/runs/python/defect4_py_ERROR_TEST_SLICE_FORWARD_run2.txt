What happened
- The failing assertion (Expected max_middle_index=1, got 3) is caused by _update_bounds() assigning the wrong index when it updates the "max middle" bound.
- In other words, the code compares the computed middle value of the new time period to the stored max middle value correctly, but when it decides "this one is larger", it writes the wrong index variable into self._max_middle_index. That wrong index ends up being 3 for your test input instead of 1.

Why it happened (root cause)
- Inside TimePeriodValues._update_bounds(), the code computes a middle value (call it mid) and compares it to the current maximum middle. That comparison is correct.
- When the code updates the bound it should do:
  self._max_middle_value = mid
  self._max_middle_index = i
  where i is the index of the newly added element.
- The buggy code instead assigns some other index variable (for example: using a different index variable name from another comparison, or accidentally using self._max_start_index, or using an index variable from the start/end comparison). Because of that, the stored index points at a different element (index 3) even though the highest middle value is at index 1.

Why your test exposes it
- Your test inserts times with middle points: index 0 -> 50, index 1 -> 150 (true max), index 2 -> 25, index 3 -> 75.
- The comparison logic sees the 150 value and should set max_middle_index to 1. But because the wrong variable is written, it ends up storing 3 instead. The failing assertion reveals exactly that.
