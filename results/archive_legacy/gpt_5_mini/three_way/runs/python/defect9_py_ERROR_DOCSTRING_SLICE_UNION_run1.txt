What happened (root cause)
- The code assumes every entry in search_list and replacement_list is a real string in several places before it ever checks for None. That causes a TypeError when the code calls len(...) on a None value.
- Concretely:
  - At lines 81–85 (the "increase" estimation loop) the code does:
    greater = len(replacement_list[i]) - len(search_list[i])
    If either replacement_list[i] or search_list[i] is None, len(...) raises TypeError.
  - Later, at line 94 the code sets
    start = text_index + len(search_list[replace_index])
    This again calls len(...) on search_list[replace_index] without ensuring that entry is not None.
- The code only checks for None at lines 102–105:
    ... or search_list[i] is None ... or replacement_list[i] is None
  but that check runs later when scanning for matches. By that time the earlier len(...) calls have already crashed.

Where it occurs (specific lines)
- Crash occurs in the pre-scan/estimation loop at lines 81–85 (len(replacement_list[i]) and len(search_list[i])) or later at line 94 (len(search_list[replace_index])).
- The None checks exist at lines 102–105 but are too late to prevent the earlier len(...) calls.

Why this is a bug (behavior vs intention)
- Docstring says replacements that are None should be ignored. The code tries to ignore them later, but because it uses len(...) on entries before ignoring them, the function still crashes on None entries instead of skipping them.
