What went wrong
- In the _update_bounds method, inside the "max middle" block you read the wrong index when looking up the period to compare against.
- The faulty line is:
  max_middle_period = self.get_data_item(self._min_middle_index).get_period()
  — it uses self._min_middle_index instead of self._max_middle_index.

Why that breaks things (concrete example from your test)
- After adding the first two periods, _max_middle_index is 1 (mid = 150).
- After adding the third period (index 2, mid = 25) the _min_middle_index becomes 2.
- When the fourth period (index 3, mid = 75) is added, the code should compare 75 to the current max middle (150) — but because of the bug it compares 75 to the period at _min_middle_index (25).
- Since 75 > 25, the code sets _max_middle_index = 3 (wrong). That explains your test failure (expected 1, got 3).

Where it occurs in the code
- In the block starting with:
    if self._max_middle_index >= 0:
  the call to get_data_item(...) uses self._min_middle_index instead of self._max_middle_index.

Minimal fix
- Replace the offending line with:
    max_middle_period = self.get_data_item(self._max_middle_index).get_period()

Steps to apply and verify
1. Edit _update_bounds and change that single line as shown above.
2. Run your unit tests (the suite in your TEST block). The test_max_middle_index_simple and the generic tests should now pass.
3. Optionally add an explicit unit test that exercises updating max_middle_index after a min_middle_index was just set (but your existing test already reproduces the issue).

No other parts of _update_bounds need changing for this bug — the rest of the comparisons use the correct index variables.
