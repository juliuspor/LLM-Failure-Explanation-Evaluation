What happened (where)
- The test failure happens in the number formatting branch shown in the slice (lines 176–194).
- The code first checks if x == int(x) (line 176) and then converts the value with value = int(x) (line 177).
- For -0.0 in Python, x == int(x) is True and int(-0.0) is 0. That makes value and mantissa become 0 and the code eventually calls self.add(str(value)) (line 192).
- str(0) is "0", so the negative sign of -0.0 is dropped and "-0.0" becomes "0". That is why test_negative_zero fails.

Why it happens (root cause)
- Converting the float to an int (line 177) removes the sign information for zero. There is no check for the special case "negative zero" before this conversion.
- The code assumes integer-ness implies safe integer formatting, but that assumption ignores the special IEEE-754 negative zero. In short: the sign bit on -0.0 is lost when you do int(x) or str(int(x)), so the code prints "0" instead of "-0.0".

Exactly where to fix it
- Insert a negative-zero check before the integer handling at or before line 176.
- Use a reliable negative-zero test such as math.copysign(1.0, x) < 0 and x == 0.0, or the existing utility CodeConsumer.is_negative_zero(x) if present.

Concrete fix (what to change, step-by-step)
1. Before the integer branch (before line 176), add:

    if CodeConsumer.is_negative_zero(x):
        # preserve sign for negative zero
        self.add("-0.0")
        return

   Or, if you don't want to call the utility:

    if x == 0.0 and math.copysign(1.0, x) < 0:
        self.add("-0.0")
        return

2. Leave the rest of the integer formatting code unchanged; it will only be reached for ordinary integers and zeros that are positive.

Why this fix works
- The check preserves the negative sign for -0.0 before any conversion to int strips the sign.
- It only affects the special case of negative zero and does not change formatting behavior for positive zero, integers, or other floats.

Additional note
- The current integer path (lines 176–192) already turns positive 0.0 into "0", which is correct. The only missing piece was detecting negative zero before int(x) is computed.
