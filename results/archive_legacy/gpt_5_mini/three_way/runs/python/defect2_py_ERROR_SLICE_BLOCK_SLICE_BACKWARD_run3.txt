What happened (where it failed)
- The exception is raised at get_paint, line 74–75:
  - Line 71 computes g from value: 
    g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
  - Line 74 checks g and raises the ValueError if g < 0 or g > 255. The message is: "Color parameter outside of expected range: Red Green Blue".
- So the code failed because g ended up less than 0 or greater than 255.

Why it happened (root causes)
1. value is outside the configured bounds
   - If value < self._lower_bound then (value - self._lower_bound) is negative → g < 0.
   - If value > self._upper_bound then the fraction > 1 → g > 255.
   - There is no check that the input value lies inside [self._lower_bound, self._upper_bound] before mapping it to 0..255.

2. upper_bound equals lower_bound (division by zero or invalid fraction)
   - If self._upper_bound == self._lower_bound the denominator is zero. That either raises ZeroDivisionError or produces an invalid float result that can make g outside 0..255; the code does not prevent or handle that.

Exactly where to look and what to do (concrete fixes)
- Add input validation in __init__ and in get_paint:

  1) In __init__ (lines around 30–44)
     - Validate bounds so upper_bound > lower_bound. Example:
       if upper_bound <= lower_bound:
           raise ValueError("upper_bound must be greater than lower_bound")
     - This prevents the division-by-zero path.

  2) In get_paint (lines 54–75)
     - Validate the incoming value and clamp or reject it before computing g. Two options:
       - Reject out-of-range values with a clear error:
         if value < self._lower_bound or value > self._upper_bound:
             raise ValueError(f"value {value} outside bounds [{self._lower_bound}, {self._upper_bound}]")
       - Or clamp into range so g always stays within 0..255:
         value_clamped = max(self._lower_bound, min(value, self._upper_bound))
         fraction = (value_clamped - self._lower_bound) / (self._upper_bound - self._lower_bound)
         g = int(fraction * 255.0)

  3) Keep the explicit g check as a sanity assertion if you want:
     if g < 0 or g > 255:
         raise AssertionError(f"Computed g out of expected range: {g}")

- Which fix to choose depends on intended behavior:
  - If callers must only pass values inside bounds, raise a ValueError in get_paint (clear message referencing value, lower_bound, upper_bound).
  - If callers may pass out-of-range values and you want graceful handling, clamp the value before mapping.

Summary
- The failure is because g is computed from a value that is outside [self._lower_bound, self._upper_bound] or because upper_bound == lower_bound, so g falls outside 0..255 and triggers the ValueError at lines 74–75.
- Fix by validating bounds in __init__ and either rejecting or clamping input values in get_paint.
