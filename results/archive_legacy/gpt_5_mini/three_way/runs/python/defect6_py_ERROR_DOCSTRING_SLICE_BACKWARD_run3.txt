What happened (where: line 176)
- On line 176 you test: if x == int(x) and not math.isinf(x) and not math.isnan(x):
- That condition is true for x == -0.0 because int(-0.0) == 0 and (-0.0) == 0 is True.
- Later (immediately after that line) the code converts x to an integer or otherwise normalizes it to a plain zero value (for example by returning int(x) or float(int(x))).
- Converting -0.0 to an int and back loses the IEEE-754 negative-zero sign bit. int(-0.0) → 0 and float(0) → 0.0 (positive zero). The test therefore received 0 instead of the expected -0.0.

Why the root cause is the comparison and conversion
- The equality check x == int(x) does not distinguish -0.0 from +0.0; it only checks numeric value.
- The conversion int(x) / float(int(x)) is what discards the negative-zero sign. That is the exact operation that produced 0 instead of -0.0.

How to fix it (concrete steps you can apply)
1. Preserve sign when x is exactly zero
   - Before converting, detect the zero sign and return a zero with the same sign:

           if x == 0.0:
               return math.copysign(0.0, x)   # keeps -0.0 when x is -0.0
           return int(x)
   - This avoids turning -0.0 into 0.0.

2. Alternatively, avoid converting to int when you need to keep float sign semantics
   - If you need a float return value, don’t use int(x) for zero cases. Use:

3. If you really want an integer for nonzero exact-integer floats but preserve -0.0 as a float

Why these fixes work
- math.copysign(0.0, x) constructs a floating zero with the same sign as x, so -0.0 remains -0.0.
- They avoid the int/float roundtrip that drops the negative-zero sign.
