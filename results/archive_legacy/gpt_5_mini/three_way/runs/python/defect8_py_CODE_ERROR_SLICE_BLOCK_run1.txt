What happened (where it failed)
- Input: "fr__POSIX".
- Failure occurs at the check on lines 239–240 (the ch3/ch4 uppercase-check):
  ch3 = locale_str[3]
  ch4 = locale_str[4]
  if ch3 < 'A' or ch3 > 'Z' or ch4 < 'A' or ch4 > 'Z':
      raise ValueError(f"Invalid locale format: {locale_str}")
- For "fr__POSIX":
  - locale_str[2] == '_' (OK)
  - locale_str[3] == '_' (because the country is empty)
  - locale_str[4] == 'P'
- The code expects positions 3 and 4 to always be uppercase letters (a two-letter country code). ch3 is '_' which fails the comparison ch3 < 'A', so the function raises ValueError.

Why it happened (root cause)
- The implementation assumes that when length != 2 the string will always contain a two-letter country code at indices 3 and 4 and a separating underscore at index 5. It does not handle the valid locale form where the country is omitted but a variant exists (language + two underscores + variant, e.g. "fr__POSIX").
- In "fr__POSIX" the country is empty and the format uses two underscores in a row. The code treats the first underscore as the separator and then insists the next two chars are letters, which is false when the next char is the second underscore.

How to fix it (concrete steps)
1. Detect the empty-country case (double underscore at positions 2 and 3) before enforcing the two-letter country checks.
2. If you see locale_str[2] == '_' and locale_str[3] == '_', set country = "" and set variant = locale_str[4:] (variant may be empty or non-empty). Return a Locale(language, "", variant).
3. Otherwise, keep the existing country-two-letters and variant-after-position-5 logic.
