What happened (root cause)
- In the "max middle" block you are comparing the new period's middle to the wrong stored period. The code fetches the period at self._min_middle_index instead of self._max_middle_index:
  - Current line: max_middle_period = self.get_data_item(self._min_middle_index).get_period()
  - It should use self._max_middle_index.
- Because you compared against the min_middle period instead of the current max_middle period, the comparison logic produced an incorrect winner and updated self._max_middle_index wrongly (so your test expected 1 but got 3).

Where it occurs in the code
- Inside the block that begins with:
  if self._max_middle_index >= 0:
- The erroneous line is the one that starts with max_middle_period = self.get_data_item(...).get_period() — it uses _min_middle_index.

Why that causes the failing assertion
- You intended to keep track of the item with the largest middle. Instead, you looked up the minimum-middle item as the baseline for deciding whether the current item is larger. That yields wrong comparisons when the min and max middle are different items, and can cause self._max_middle_index to be set to the wrong index (3 in your failing test).

Exact fix
- Replace the single incorrect lookup with the max index:
  - Change:
    max_middle_period = self.get_data_item(self._min_middle_index).get_period()
  - To:
    max_middle_period = self.get_data_item(self._max_middle_index).get_period()

Recommended follow-ups
1. Make the one-line change above and run tests (this should fix the AssertionError reported).
2. Add a unit test that inserts multiple periods with differing starts/ends and verifies all six index-tracking members (_min_start_index, _max_start_index, _min_middle_index, _max_middle_index, _min_end_index, _max_end_index) — this will catch regressions from similar copy/paste errors.
3. Optionally add a small internal assertion right after reading max_middle_period:
   assert self._max_middle_index >= 0
   (or check that the fetched period is not None) to make such mistakes fail fast.

No other blocks have an obvious symmetric typo, but doing step (2) will ensure there are no other copy/paste index errors.
