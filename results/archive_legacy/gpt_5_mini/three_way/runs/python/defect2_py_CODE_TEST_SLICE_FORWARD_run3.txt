What happened (root cause)
- In get_paint you correctly clamp the input into v:
  - v = max(value, self._lower_bound)
  - v = min(v, self._upper_bound)
- But when you compute g you accidentally use the original value variable instead of the clamped v:
  - g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- That means the clamping has no effect on the calculation. If value is outside the bounds (e.g. -0.5 or 1.5) g can be < 0 or > 255 and the code then raises the ValueError on the next lines. That is why tests that pass -0.5 or 1.5 fail.

Where it occurs (exact lines)
- v is set on the two clamp lines.
- The bug is at the line computing g — replace value with v there.
- The exception is raised immediately after when you check g < 0 or g > 255.

Why the tests fail
- test_lower_bound_clamping uses value -0.5. Because g is computed from the unclamped value, g becomes negative, triggering the ValueError instead of returning (0, 0, 0).
- test_upper_bound_clamping uses 1.5. Similarly g > 255 triggers the exception instead of returning (255, 255, 255).

How to fix it (concrete steps)
1. Change the g calculation to use the clamped v:
   - g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
2. Add a defensive check to avoid division by zero if upper and lower bounds are equal:
   - denom = (self._upper_bound - self._lower_bound)
   - if denom == 0.0: return (0, 0, 0) or (255,255,255) depending on desired behavior—or raise a clear error.
3. (Optional) If you prefer rounding rather than truncation for midpoints, use round(...) before int().

Minimal corrected get_paint
- Replace the g calculation line with:
  - denom = self._upper_bound - self._lower_bound
  - if denom == 0.0: g = 0  # or handle as desired
  - else: g = int((v - self._lower_bound) / denom * 255.0)

That single change (using v instead of value) fixes the test failures caused by the misplaced variable.
