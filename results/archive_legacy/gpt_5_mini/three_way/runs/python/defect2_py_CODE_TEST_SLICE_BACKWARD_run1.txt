What happened (root cause, where it occurred)
- In get_paint you clamp the input into v on lines 54–57:
  - v = max(value, self._lower_bound)
  - v = min(v, self._upper_bound)
- But on line 71 you compute g using the original value variable instead of the clamped v:
  - g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- If value is outside the bounds (e.g. -0.5 or 1.5 in the tests), the calculation uses that out‑of‑range value and yields g < 0 or g > 255, which triggers the ValueError on line 74.
- So the clamp is performed but ignored in the interpolation step because you used the wrong variable (value instead of v).

Why it fails (simple cause)
- Wrong variable used for the interpolation. The clamped value (v) is never used in the formula, so out‑of‑range inputs are not actually clamped before producing g.

Exact place to change
- Replace the interpolation line (line 71) with one that uses v:
  - g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

Also consider an additional necessary check
- If self._upper_bound == self._lower_bound you would divide by zero. Add a guard to avoid ZeroDivisionError (return black or white, or raise a clear error).

Minimal concrete fix (patch)
- Change line 71 to:
  g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- And optionally add before that:
  if self._upper_bound == self._lower_bound:
      raise ValueError("upper_bound must be greater than lower_bound")

Why this fixes the tests
- Using v ensures values below lower_bound are treated as lower_bound (0 → black) and values above upper_bound are treated as upper_bound (1 → white), so g will be in [0,255] and the ValueError will not be raised for clamped inputs.
