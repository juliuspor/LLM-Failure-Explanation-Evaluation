What happened (where and why)
- In get_paint you correctly clamp the incoming value into v:
  - v = max(value, self._lower_bound)
  - v = min(v, self._upper_bound)
  (these are at the start of get_paint, lines shown above).
- But when you compute the gray channel you use the original parameter value, not the clamped v:
  - g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
  (this is line 71 in your slice).
- If value is outside the bounds (e.g. -0.5 or 1.5), the expression uses that out-of-range value and produces g < 0 or g > 255, which triggers the ValueError at line 74–75. That is why tests that pass values below lower_bound or above upper_bound fail.

Root cause, plain and specific
- You intended to use the clamped variable v in the interpolation, but you accidentally left value in the formula. So clamping does nothing for the computed g.

Exact fix (steps)
1. Replace value with v in the interpolation:
   - g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
2. Also guard against zero range (upper == lower) to avoid division by zero. Example:
   - if self._upper_bound == self._lower_bound:
         g = 0  # or 255, or raise; choose behavior consistent with your API
   - or raise a ValueError in __init__ if bounds are equal/invalid.
3. (Optional) Improve the error message to show the actual g or parameter name if you keep the range check.

Minimal corrected method
- Change the single line shown above to use v instead of value, and add a zero-range check if needed.

Why this will make tests pass
- Using v ensures values outside the bounds are clamped before mapping, so -0.5 becomes lower_bound -> g = 0 and 1.5 becomes upper_bound -> g = 255, matching the unit tests.
