What happened (where it fails)
- The failure occurs inside create_number() in the branch that handles trailing "l" or "L" qualifiers. That branch starts at the line:
  if last_char in ("l", "L"):
- The code computes numeric = val[:-1] and then calls char_at(numeric, 0) without ensuring numeric is non-empty.
- If val is "l" or "L", numeric is the empty string "". char_at(numeric, 0) hits the bounds check in char_at (lines 30–31) and raises IndexError("String index out of range: 0"). The char_at return on line 32 is never reached because the index was invalid.

Why it happens (root cause)
- The code assumes numeric has at least one character when it calls char_at(numeric, 0). It does not guard the case len(val) == 1 (so numeric == ""), so an IndexError is thrown from char_at. The failing call is exactly the expression char_at(numeric, 0) in the if condition:
    if (char_at(numeric, 0) == "-" and numeric[1:].isdigit()) or numeric.isdigit():
- The IndexError originates in char_at's bounds check:
  30 |     if index < 0 or index >= len(s):
  31 |         raise IndexError(f"String index out of range: {index}")
  32 |     return s[index]

Concrete fixes (exact steps you can apply)
1) Short, explicit guard before calling char_at:
   Replace the start of the "l"/"L" block with:
     if last_char in ("l", "L"):
         if len(numeric) == 0:
             raise ValueError(f"{val} is not a valid number.")
         if (char_at(numeric, 0) == "-" and numeric[1:].isdigit()) or numeric.isdigit():
             return int(numeric)
         raise ValueError(f"{val} is not a valid number.")
   - This prevents char_at from being called with an empty string.

2) Better: avoid char_at entirely and use str methods (cleaner and safer):
   if last_char in ("l", "L"):
       if numeric.isdigit() or (numeric.startswith("-") and numeric[1:].isdigit()):
           return int(numeric)
       raise ValueError(f"{val} is not a valid number.")
   - This both fixes the empty-string case and makes the intent clearer.

Tests to add
- Ensure these inputs behave predictably:
  - "l" and "L" → should raise ValueError (not IndexError)
  - "-l" and "-L" → should raise ValueError (or be handled according to intended semantics)
  - "123l" and "123L" → return int(123)
  - "-123l" and "-123L" → return int(-123)

Summary
- Root cause: char_at(numeric, 0) is called when numeric can be empty, causing an IndexError in char_at (lines 30–31).
- Fix by checking len(numeric) > 0 first or by replacing the char_at logic with numeric.startswith("-") and numeric[1:].isdigit() as shown above.
