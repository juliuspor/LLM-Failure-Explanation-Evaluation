What happens
- At line 396 the code rejects any negative minutes:
  if minutes_offset < 0 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")

- The docstring (and intended behavior) allow minutes to be negative (range -59..+59) for some combinations (for example (0, -15) and (-2, -15) are allowed). Because the code only accepts 0..59, callers that pass a valid negative minutes value are incorrectly rejected at that check.

Why that causes the failure (root cause)
- The minutes range check is too strict: it enforces minutes_offset >= 0 rather than allowing -59 <= minutes_offset <= 59.
- Later code (lines 402–405) expects minutes_offset to possibly be negative and combines it with hours_in_minutes:
    hours_in_minutes = hours_offset * 60
    if hours_in_minutes < 0:
        minutes_offset = hours_in_minutes - minutes_offset
    else:
        minutes_offset = hours_in_minutes + minutes_offset
  Because callers with negative minutes never get past the earlier check, the code never reaches that combination logic for those valid inputs.

Where it occurs
- The incorrect check is at line 396 (the minutes range validation).
- The combination logic that assumes negative minutes could be present is at lines 402–405.

How to fix (concrete steps)
1. Allow minutes to be in -59..+59. Change the range check to:
   if minutes_offset < -59 or minutes_offset > 59:
       raise ValueError(f"Minutes out of range: {minutes_offset}")

2. Enforce the sign rule described in the docstring: minutes must not be negative when hours are positive. Add a check after the range check:
   if hours_offset > 0 and minutes_offset < 0:
       raise ValueError("Minutes must be positive when hours is positive")

3. (Optional) Keep the special-case for hours_offset == 0 and minutes_offset == 0 that returns UTC (line 394). Your combination logic at lines 402–405 can then safely assume minutes_offset is within -59..59 and that illegal sign combos have been rejected.

Minimal corrected snippet:
    if minutes_offset < -59 or minutes_offset > 59:
        raise ValueError(f"Minutes out of range: {minutes_offset}")
    if hours_offset > 0 and minutes_offset < 0:

Summary
- The code rejects valid negative minute values because the minutes validation only permits 0..59 (line 396). Allowing -59..59 and adding the specific sign-combination check fixes the failure and matches the docstring.
