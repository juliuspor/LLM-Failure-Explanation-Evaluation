What happened (where and what failed)
- The ValueError is raised at lines 396–397:
  - Line 396: if minutes_offset < 0 or minutes_offset > 59:
  - Line 397:     raise ValueError(f"Minutes out of range: {minutes_offset}")
- The exception message shows minutes_offset is -15. That means code arrived at the minutes-range check with a negative minutes_offset (-15), so the check correctly failed and raised the exception.

Why this happened (root cause)
- minutes_offset is being used for two different things:
  1. the parsed minute component (0..59), and
  2. the signed minute total that will be combined with hours to make a full offset.
- The validation (lines 396–397) runs after minutes_offset has already had sign applied (or otherwise been made negative) instead of validating the raw minute component first.
- As a result, when you pass a negative time zone like "-01:15" the minutes component winds up as -15 before the range check, and the check treats that as invalid and raises ValueError.

Exactly where to look in the code
- The validation happens at lines 396–397.
- The code that modifies minutes_offset appears at lines 402 and 404:
  - 402: minutes_offset = hours_in_minutes - minutes_offset
  - 404: minutes_offset = hours_in_minutes + minutes_offset
  Those lines show minutes_offset being combined with hours_in_minutes after sign-handling, which indicates minutes_offset has been turned negative or otherwise adjusted before the validation.

How to fix it (concrete steps)
1. Separate parsing/validation of the minute component from sign application:
   - Parse minutes into a local variable parsed_minutes (always non-negative, 0..59).
   - Validate parsed_minutes with the existing range check.
   - After validation, apply the sign and combine with hours to produce minutes_offset.

2. Example change (showing exact intent relative to your lines):
   - Before (problematic):
     396: if minutes_offset < 0 or minutes_offset > 59:
     397:     raise ValueError(f"Minutes out of range: {minutes_offset}")
     ...
     402: minutes_offset = hours_in_minutes - minutes_offset   # sign may already be applied
     404: minutes_offset = hours_in_minutes + minutes_offset
   - After (fixed):
     396: parsed_minutes = minutes_offset  # rename so we keep the raw parsed value
     397: if parsed_minutes < 0 or parsed_minutes > 59:
     398:     raise ValueError(f"Minutes out of range: {parsed_minutes}")
     ...
     402: # now apply sign using parsed_minutes
     403: if negative_sign:
     404:     minutes_offset = hours_in_minutes - parsed_minutes
     405: else:
     406:     minutes_offset = hours_in_minutes + parsed_minutes

3. Alternatively, validate abs(minutes_offset) if you prefer minimal edits:
   - Replace the check with:
     if abs(minutes_offset) > 59:
         raise ValueError(f"Minutes out of range: {minutes_offset}")
   - But the explicit parsed_minutes approach is clearer and avoids misinterpreting intermediate values.

Why these fixes work
- They ensure validation is performed on the raw minute component (0..59) before any sign or combination logic changes the value to negative or larger totals. That prevents valid inputs like "-01:15" from failing the range check.

Summary
- Root cause: minutes component is made negative before the minutes-range validation. The range check runs on that signed value and throws because -15 < 0.
- Fix: validate the unsigned parsed minute value (0..59) first, then apply sign and combine with hours.
