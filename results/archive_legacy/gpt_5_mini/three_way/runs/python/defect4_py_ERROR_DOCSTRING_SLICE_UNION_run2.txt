What happened (root cause)
- At lines 340–349 you try to update self._min_start_index only when the new period's start is earlier than the stored minimum, but line 349 then assigns self._min_start_index = index unconditionally and overwrites the conditional update. In other words, the code that is supposed to run only when the index has not been set or when a new minimum is found is always executed.
- Specifically:
  - Line 340 checks whether self._min_start_index >= 0.
  - Lines 341–347 compare the current period start to the stored minimum and set self._min_start_index = index only when start < min_start (line 347).
  - But line 349 immediately does self._min_start_index = index regardless of the comparison, wiping out the correct minimum index.

Why that makes your test fail
- Because the unconditional assignment at line 349 updates the minimum-start index to the current index even when the current period is not earlier, the stored min (and any related "max_middle_index" logic elsewhere) ends up being the last-seen index instead of the actual earliest index. The test expected max_middle_index = 1 but got 3 because the index was overwritten when it should not have been.

Where it occurs
- The incorrect behavior is in the block shown around lines 340–349 (the _min_start_index update). The same pattern likely exists for the corresponding maximum-index update code elsewhere; check for any unconditional assignments that follow conditional comparisons.

1. Replace the unconditional assignment at line 349 with an else branch so you only initialize the stored index when it has not been set:

         ...  # compute min_start, compare

             self._min_start_index = index
         self._min_start_index = index   # <-- remove this
