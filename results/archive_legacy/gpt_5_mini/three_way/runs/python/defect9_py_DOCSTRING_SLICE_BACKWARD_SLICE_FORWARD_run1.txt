What happened (what crashed, where)
- The code crashes with TypeError: "object of type 'NoneType' has no len()" at line 82:
  > greater = len(replacement_list[i]) - len(search_list[i])
- That happens when either replacement_list[i] or search_list[i] is None.
- The crash occurs inside the length-estimate loop that runs before the main replace loop (the loop that starts around line 59 and includes lines 81–85).

Why it failed (root cause)
- The function intends to ignore None entries (see the later check at lines 102–105 that tests "search_list[i] is None" and "replacement_list[i] is None").
- But the earlier sizing loop (lines 81–85) calls len(replacement_list[i]) and len(search_list[i]) without first checking for None.
- Because that guard is only present later, a None in either list causes len(None) to run and raise the TypeError before the code ever reaches the part that would ignore the None entry.

Exactly where to change it (what to edit)
- In the sizing loop (the for i in range(search_length) loop that contains lines 81–85), add the same None / empty-string checks used later (lines 102–105) before calling len().
- Concretely, at the start of that loop add:
  if search_list[i] is None or replacement_list[i] is None or len(search_list[i]) == 0:
      continue
  Then you can safely do:
      greater = len(replacement_list[i]) - len(search_list[i])
      if greater > 0:
          increase += 3 * greater

Why that fix works
- It prevents calling len() on None values, matching the function's intended behavior of ignoring None entries.
- It keeps the sizing estimate consistent with the later logic that skips those replacements.

Alternative (if you prefer)
- Instead of skipping, you could coerce None to '' earlier, but that changes semantics. The clear, minimal fix is to add the guard before any len() calls in the sizing loop.
