What failed
- The crash happens at this line (inside replace_each):  
  greater = len(replacement_list[i]) - len(search_list[i])  (around lines 81–84 in your slice).
- If any element replacement_list[i] or search_list[i] is None, calling len(None) raises a TypeError: "object of type 'NoneType' has no len()".
- The code earlier checks for None/empty values when searching matches (those checks are at lines 59–76 and again inside the loop at lines 102–106), but the "increase" loop (lines 81–85) runs before those checks and assumes every array element is a non-None string. The BUG comment already points to this being "not null-safe", but that is exactly the root cause.

Where it happens
- In replace_each(), after computing search_length, the code computes an estimated increase with:
    for i in range(search_length):
        greater = len(replacement_list[i]) - len(search_list[i])
        ...
  This runs even if search_list[i] or replacement_list[i] is None (or if search_list[i] == ""), causing the TypeError on len(None).

Why it happens (concise)
- The function accepts Optional[List[Optional[str]]], so elements may be None.
- The increase loop calls len(...) on list elements without guarding for None or empty strings.
- Later logic properly skips None/empty entries, but it's too late: the function already blew up.

Precise fixes (pick one)

1) Skip None/empty entries in the increase loop (recommended minimal change)
Replace the loop with a guarded version:

    for i in range(search_length):
        if search_list[i] is None or len(search_list[i]) == 0 or replacement_list[i] is None:
            continue
        greater = len(replacement_list[i]) - len(search_list[i])
        if greater > 0:
            increase += 3 * greater

This mirrors the checks you already perform later and prevents len(None).

2) Filter the lists up front and compute using filtered lists
Create new lists that only contain valid (non-None, non-empty search, non-None replacement) pairs, and use those for the rest of the function. That avoids repeating checks and simplifies later loops.

    pairs = [
        (s, r)
        for s, r in zip(search_list, replacement_list)
        if s is not None and len(s) > 0 and r is not None
    ]
    if not pairs:
        return text
    # then use pairs instead of indexing into search_list/replacement_list

3) Defensive try/except (not preferred)
Wrap len calls in try/except to catch TypeError — but this hides problems and is less clear than explicit checks.

What to change (exact lines)
- Change the loop beginning around line 81 (the "BUG" comment area) to include the guard shown in fix (1), or replace it by the filtering in fix (2). That addresses the TypeError at the line computing greater.

Why these fixes work
- They prevent calling len(...) on None by skipping any pair that is already excluded by the later matching logic. After the change, the "increase" estimate only inspects valid string pairs and cannot raise TypeError.
