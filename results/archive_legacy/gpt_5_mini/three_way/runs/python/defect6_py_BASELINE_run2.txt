What happened (root cause)
- At line 176 the code tests if x == int(x). For -0.0 this is True because in Python -0.0 == 0.
- Then at line 177 value = int(x) produces value == 0 and mantissa = 0.
- The integer branch (lines 189–192) finishes by calling self.add(str(value)) (line 192), which turns 0 into the string "0".
- The sign of the original float (negative zero) is lost before any sign-preserving check runs, so add_number(-0.0) prints "0" instead of "-0.0".

Why exactly that loses the sign
- Converting -0.0 to the integer 0 removes the negative sign because ints have no negative zero concept.
- The code never checks for negative zero before the int conversion; therefore the negative-zero special case is never observed and the negative sign is not emitted.

Where this occurs (lines and variables)
- Decision: line 176 if x == int(x) and not math.isinf(x) and not math.isnan(x)
- Conversion: line 177 value = int(x)
- Output: line 192 self.add(str(value)) — this is where the negative zero becomes "0"
- Variables involved: x (float input), value (int produced by int(x)), mantissa, exp.

How to fix it (concrete steps you can apply)
1. Detect negative zero before doing int(x). Add a test at the top of add_number, right after you have x and before line 176:
   - If the codebase already has CodeConsumer.is_negative_zero, call it. Example (pseudocode):
     if self.is_negative_zero(x):
         self.add("-0.0")
         return
   - If you don't have that utility, implement the detection with math.copysign:
     def is_negative_zero(v):
         return v == 0.0 and math.copysign(1.0, v) < 0.0
     then use the same early-return.

2. Alternative minimal change: if you prefer not to early-return, handle the special case just before writing the integer value:
   - Replace the block that writes str(value) (line 192) with:
     if value == 0 and math.copysign(1.0, x) < 0.0:
         self.add("-0.0")
     else:
         self.add(str(value))

Why that fix works
- Checking math.copysign(1.0, x) reveals the sign bit for zero-valued floats; for -0.0 it returns -1.0. That lets you distinguish -0.0 from +0.0 even though both compare equal to 0 and int(x) == 0.
- Performing this check before or instead of converting-and-stringifying prevents the integer conversion from discarding the negative sign.

One-line summary
The code loses the negative-zero sign because it converts -0.0 to int( -0.0 ) == 0 and then calls str(0); detect negative zero (e.g. via is_negative_zero or math.copysign) before converting to int and output "-0.0" explicitly.
