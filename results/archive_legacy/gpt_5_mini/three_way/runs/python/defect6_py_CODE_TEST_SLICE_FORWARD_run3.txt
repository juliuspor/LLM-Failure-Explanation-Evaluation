What happened (root cause)
- At line 177 you do value = int(x). For x == -0.0 this converts the float -0.0 to the integer 0 and loses the sign bit.
- Because of that the code takes the integer branch (the guard at line 176: if x == int(x) and not math.isinf(x) and not math.isnan(x):) and eventually executes self.add(str(value)) at line 192, which produces "0" not "-0.0".
- The test expects negative zero to preserve its sign and be printed as "-0.0", so the output becomes "0" and the test fails.

Where this occurs in the code
- Entry to the problematic branch: the check at the start of the integer branch (line ~176).
- Loss of sign: value = int(x) at line 177.
- Final emission of the wrong text: self.add(str(value)) at line 192.
- Also relevant: the earlier check x < 0 (used at top to insert a space when following a '-') will treat -0.0 as not negative in Python, so any logic that relies on x < 0 to detect sign will also miss -0.0.

Why int(x) is the culprit
- int(-0.0) is 0; converting to an integer discards the float sign bit. The equality x == int(x) is true for -0.0, so you enter the integer branch and never see that x was a negative zero.

Concrete fixes (step-by-step)
1) Detect negative zero before converting to int or before using x < 0:
   - Add a check at the top of add_number:
     if self.is_negative_zero(x):
         self.add("-0.0")
         return
   - This ensures -0.0 is handled and emitted correctly before any integer conversion or formatting.

2) Make the sign test robust (optional but recommended):
   - Replace comparisons like x < 0 when you want to know whether the float has a negative sign with a sign-bit test. For example:
       is_neg = math.copysign(1.0, x) < 0.0
     Use is_neg in places where you need to know whether the original float was negative, including -0.0.

3) If you cannot add is_negative_zero, implement it like:
   @staticmethod
   def is_negative_zero(x):
       return x == 0.0 and math.copysign(1.0, x) < 0.0

4) If you need to preserve "-0.0" formatting in the integer branch as well (safer):
   - Before value = int(x) check:
       if x == 0.0 and math.copysign(1.0, x) < 0.0:
           self.add("-0.0")
           return
   - Then continue with existing integer handling only for non-negative-zero values.

Summary
- The failure is caused by converting -0.0 to int (line 177), which loses the sign and causes the code path that emits "0" (line 192). Fix by detecting negative zero (using math.copysign or an is_negative_zero helper) before any int conversion and emitting "-0.0" immediately. This also fixes other logic that relies on x < 0 which misses -0.0.
