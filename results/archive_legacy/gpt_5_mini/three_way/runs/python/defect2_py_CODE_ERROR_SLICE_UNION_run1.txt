What happened (root cause)
- You clamp the input into variable v: 
  - "v = max(value, self._lower_bound)" and "v = min(v, self._upper_bound)" (the clamping lines).
- But when you compute g you use the original value, not the clamped v:
  - Line 71: g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- If value is outside the bounds, that formula produces a number < 0 or > 255, so the check at line 74 raises the ValueError you saw.

Where it occurs
- The mismatch is between the clamping variables and the variable used in the calculation: the clamping result is stored in v, but the calculation uses value (line 71), so the clamp is effectively ignored.

How to fix it (concrete steps)
1. Use the clamped v in the formula instead of value.
2. Handle the degenerate case when upper_bound == lower_bound to avoid division by zero.
3. After converting to int, ensure g is clamped to [0,255] or raise a clearer error only if that is desired.

Corrected code example:
    def get_paint(self, value: float) -> Tuple[int, int, int]:
        # Clamp value to bounds
        v = max(value, self._lower_bound)
        v = min(v, self._upper_bound)

        # Handle zero range to avoid division by zero
        if self._upper_bound == self._lower_bound:
            g = 0  # or 255, depending on desired behavior
        else:
            g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

        # Clamp final integer to valid RGB range
        g = max(0, min(255, g))

        return (g, g, g)

That change ensures the clamped value is used (fixing the ValueError) and avoids division-by-zero and out-of-range integers.
