What happened (root cause)
- The TypeError comes from line 82: len(replacement_list[i]) - len(search_list[i]). That raises "object of type 'NoneType' has no len()" when either replacement_list[i] or search_list[i] is None.
- The code only checks for None values later (lines 102–105), but the problematic len() calls happen earlier in the "increase" calculation (lines 81–85). So if any entry in search_list or replacement_list is None, the code will crash at line 82 before it ever reaches the later safety checks.

Where exactly it fails
- In replace_each (definition starts at line 28), you compute search_length = len(search_list) (line 52) and then run the loop that calculates increase (lines 81–85). Inside that loop you call len(replacement_list[i]) and len(search_list[i]) (line 82) without checking whether those elements are None.
- The code that would have excluded None entries runs later (lines 102–105), but it’s too late: the error already happened.

Why that causes the TypeError
- Python’s len(None) is invalid, so when replacement_list[i] or search_list[i] is None the interpreter raises TypeError at line 82.

Concrete fixes (explicit steps)
Pick one of these, depending on desired behavior:

1) Skip None entries when computing increase
- Change the increase loop (lines 81–85) to skip indices where either element is None:
  - Before calling len(), check:
    if replacement_list[i] is None or search_list[i] is None:
        continue
  - Then compute greater = len(replacement_list[i]) - len(search_list[i]) as before.

2) Normalize inputs at function start
- Immediately after search_length = len(search_list) (line 52), validate/normalize lists:
  - If you want None to act like empty string:
    for i in range(search_length):
        if search_list[i] is None: search_list[i] = ""
        if replacement_list[i] is None: replacement_list[i] = ""
  - Or raise a clear ValueError:
    for i, (s, r) in enumerate(zip(search_list, replacement_list)):
        if s is None or r is None:
            raise ValueError(f"search_list[{i}] or replacement_list[{i}] is None")

3) Move the existing safety checks earlier
- Move the block that sets no_more_matches_for_repl_index and checks for None (currently around lines 102–105) to before any len(...) calls so that None entries are handled before you compute increase.

Why these fixes work
- They prevent calling len() on None by either converting None to a string (so len() is valid), skipping the entries so len() is never called, or failing fast with a clear error that points to the offending index.

Which to choose
- If None is unexpected and indicates a caller bug: validate early and raise ValueError with the index (option 2 raising).
- If None should be treated as empty string: normalize to "" at the start (option 2 normalizing).
- If you simply want to avoid accounting for None entries in the size/increase estimate: skip them in the increase loop (option 1).

Small example patch (skip None entries in the increase loop)
Replace the loop at lines 81–85 with:
for i in range(search_length):
    if search_list[i] is None or replacement_list[i] is None:
        continue
    greater = len(replacement_list[i]) - len(search_list[i])
    if greater > 0:
        increase += 3 * greater

This change prevents the TypeError and keeps the rest of the function’s logic intact.
