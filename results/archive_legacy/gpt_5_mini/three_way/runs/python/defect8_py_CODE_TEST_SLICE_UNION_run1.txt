What happened (where it fails)
- The code fails on inputs like "fr__POSIX" (language + empty country + variant).
- Failure happens when the code reads ch3 = locale_str[3] and ch4 = locale_str[4] and then checks they are uppercase letters (lines shown in the slice: ch3/ch4 checks at lines 237–240). For "fr__POSIX" ch3 is '_' (the second underscore), so the check ch3 < 'A' or ch3 > 'Z' is true and a ValueError is raised with "Invalid locale format: fr__POSIX". In short: the code assumes positions 3 and 4 always contain an uppercase country code, but they can be underscores when the country is empty.

Why it happens (root cause)
- The method assumes one of only two forms after the language: either a 2-letter country (positions 3–4 are letters) or a country+variant with another underscore at position 5. It never handles the valid third form where the country is empty and the string is language + "_" + "_" + variant (language__variant).
- Concretely:
  - The code checks locale_str[2] == '_' (ok).
  - Immediately it unconditionally treats locale_str[3:5] as the country and validates them as uppercase letters (lines 237–240). That validation fails when locale_str[3] == '_' (empty country), because the code doesn't have a branch to detect an empty country and parse the variant starting at locale_str[4:].

How to fix it (precise steps)
1. After confirming locale_str[2] == '_', check whether locale_str[3] is '_' (empty country case) before validating characters 3 and 4:
   - If locale_str[3] == '_':
     - Treat country = "".
     - The variant (must be non-empty) begins at locale_str[4:], so return Locale(locale_str[0:2], "", locale_str[4:]) after validating variant length > 0 if you want.
2. Otherwise (locale_str[3] != '_'), proceed with the existing flow: validate locale_str[3] and locale_str[4] are uppercase letters, then either return Locale(lang, country) when length==5 or, if there's a variant, require locale_str[5]=='_' and return Locale(lang, country, locale_str[6:]).

Minimal example patch (conceptual)
Replace the block where ch3/ch4 are read and validated with:

    # after checking locale_str[2] == '_'
    if locale_str[3] == '_':
        # empty country, variant starts at index 4
        if length <= 4:
            raise ValueError(f"Invalid locale format: {locale_str}")  # no variant present
        return Locale(locale_str[0:2], "", locale_str[4:])
    else:
        ch3 = locale_str[3]
        ch4 = locale_str[4]
        if ch3 < 'A' or ch3 > 'Z' or ch4 < 'A' or ch4 > 'Z':
            raise ValueError(f"Invalid locale format: {locale_str}")
        if length == 5:
            return Locale(locale_str[0:2], locale_str[3:5])
        if locale_str[5] != '_':
            raise ValueError(f"Invalid locale format: {locale_str}")
        return Locale(locale_str[0:2], locale_str[3:5], locale_str[6:])

Why this fix works
- It explicitly recognizes the language__variant pattern (empty country) and parses the variant starting at index 4, avoiding the wrong uppercase-letter check on an underscore.
- All existing cases (language only, language_country, language_country_variant) remain covered by the existing logic.
