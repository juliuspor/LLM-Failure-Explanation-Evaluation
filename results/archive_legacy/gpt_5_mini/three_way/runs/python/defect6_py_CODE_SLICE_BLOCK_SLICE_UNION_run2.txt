What happened
- At runtime the code crashes because the variable exp is used before it’s been assigned.
- The problematic use occurs at line 182: check_val = (mantissa // 10) * (10 ** (exp + 1)) — this references exp even though, in the failing version of the source, exp was not initialized.
- You can see the symptom in the SLICE_UNION view: mantissa is set at line 178, the code enters the loop at 181, but there is no exp = 0 line before the first use at 182. That causes a NameError / UnboundLocalError.

Why that happened
- In the working (full) code exp is initialized at line 179: exp = 0. In the broken (merged/edited) code that initialization was removed or accidentally moved, so exp does not exist when the while loop runs.
- The loop and the check_val calculation assume exp exists even before the abs(x) >= 100 guard finishes changing it. If exp is missing, the interpreter raises an error at the first evaluation of check_val.

Exactly where it occurs
- Failure point: line 182 in the shown slice: check_val = (mantissa // 10) * (10 ** (exp + 1))
- Missing initialization should have been at line 179: exp = 0

Concrete fixes (choose one)
1) Restore the initialization (minimal fix)
   - Add exp = 0 before the if abs(x) >= 100: guard. For example:
     line 179: exp = 0
   - This guarantees exp exists when line 182 runs.

2) Move/init exp immediately before it’s needed (clearer scope)
   - Put exp = 0 directly before the while loop (right before line 181) so it’s obviously available to the loop.

3) Avoid relying on exp in check_val by computing check_val differently
   - Replace the check that uses exp with a comparison using value and mantissa directly, e.g.:
     if (mantissa // 10) * (10 ** (exp + 1)) == value
   - But this still needs exp. Better alternative: compute trailing-zero count with arithmetic:
     temp = value
     exp = 0
     while temp % 10 == 0:
         temp //= 10
         exp += 1
     mantissa = temp
   - This is simpler and avoids the extra check_val computation.

- add_number(self, 99.0) should not enter the while loop and should output "99"
- add_number(self, -4.0) and cases with negative prev char to confirm the space-handling logic still works
- Run unit tests or a small script that exercises both branches (abs(x) >= 100 and < 100) and NaN/inf branches

Summary
The root cause is a missing initialization of exp (line 179). Restore exp = 0 before it’s referenced in the while loop (or refactor the loop to compute trailing zeros differently). This will stop the NameError/UnboundLocalError at line 182.
