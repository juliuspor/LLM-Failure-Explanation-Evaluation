What happened
- The failure happens inside the while loop after a translator handled a surrogate pair. The code that advances pos is wrong and causes code_point_at to be called with pos == len(code_units), which raises IndexError("String index out of range: 2").

Where it occurs (lines from your snippet)
- After consumed = self.translate_codepoint(code_units, pos, out) (the translator returned consumed > 0), this block runs:

    for pt in range(consumed):
        pos += char_count(code_point_at(code_units, pos))

- The IndexError is raised inside code_point_at when it checks index >= len(code_units) (your slice shows the check at lines 35–36).

Why it fails (root cause)
- Suppose the translator consumed a surrogate pair at current pos (consumed == 1 or 2 depending on your contract; in your comment it indicates the translator handled a surrogate pair). The loop runs range(consumed) times. On the first iteration you call code_point_at(code_units, pos) and get a code point whose char_count is 2, so you add 2 to pos and pos becomes len(code_units) (for example pos goes from 0 to 2).
- The for-loop then goes into the next iteration and tries code_point_at(code_units, pos). But pos == len(code_units) is out of bounds for code_point_at's index check, so code_point_at raises IndexError("String index out of range: 2").
- In short: you advance pos by the code point's char count inside a loop that repeats consumed times, calling code_point_at again after pos already reached the end. The code therefore calls code_point_at with an invalid index.

What to change (exact steps)
- Do not call code_point_at(pos) repeatedly while pos may have been advanced past the end. Instead advance pos by the number of code units consumed directly.
- Replace the erroneous loop with one of these correct options (choose the one that matches your consumed semantics):

  Option A — If consumed is the number of UTF-16 code units consumed (1 for BMP char, 2 for surrogate pair):
    Replace the for-loop with:
      pos += consumed

  Option B — If consumed counts code points but you must move by code units (rare):
    Compute the code point once, get its char_count once, and add it once. Example:
      cp = code_point_at(code_units, pos)
      pos += char_count(cp)

- Do NOT call code_point_at(code_units, pos) again after pos has been incremented to the end without checking bounds.

Why this fix works
- Both suggested fixes ensure you only compute the code point at the current valid pos once and then advance pos by the correct number of UTF-16 code units. That prevents the second call to code_point_at with pos == len(code_units) and eliminates the IndexError.

Summary in one line
- The loop advances pos by the code-point size repeatedly and ends up calling code_point_at with pos == len(code_units); fix by advancing pos by consumed (or compute code point once and add its char_count once).
