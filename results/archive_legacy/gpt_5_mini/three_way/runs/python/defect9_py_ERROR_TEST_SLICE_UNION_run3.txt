What happened (root cause)
- The TypeError "object of type 'NoneType' has no len()" is raised because the code calls len(...) on a None value.
- In hit09_stringutils.py, method replace_each, the offending call happens in the loop that computes "increase" (see SLICE_UNION lines 81–85). Specifically line 82 does:
  greater = len(replacement_list[i]) - len(search_list[i])
- In the failing test replacement_list contains None (replacement_list = [None]). So replacement_list[i] is None and len(None) raises TypeError before any null checks run.

Where it occurs (exact variables / lines)
- Line 52 sets search_length = len(search_list) — this is fine because search_list itself is a list.
- The error occurs in the for loop starting at line 81:
  - Line 82: uses len(replacement_list[i]) and len(search_list[i]) without checking for None.
- Later in the function there is a guard (lines 102–107) that checks:
  or search_list[i] is None
  or len(search_list[i]) == 0
  or replacement_list[i] is None
  ...
  but that check comes after the earlier len(...) calls. So the code tries to measure lengths before it ever filters out None entries.

Why this causes failure for your tests
- Your tests intentionally include None inside the replacement list (to assert null-safety). The code attempts to compute an expected increase in output buffer size by calling len(replacement_list[i]) before it ever sees that replacement_list[i] is None, so it crashes on the first such None.

How to fix it (concrete steps)
1. Do not call len(...) on entries until you have checked that the entry is not None. Change the "increase" loop to skip pairs where either search_list[i] or replacement_list[i] is None. For example:
   for i in range(search_length):
       if search_list[i] is None or replacement_list[i] is None:
           continue
       greater = len(replacement_list[i]) - len(search_list[i])
       if greater > 0:
           increase += 3 * greater

2. Also audit every other place where the code calls len(search_list[i]) or len(replacement_list[i]) (for example line 94: start = text_index + len(search_list[replace_index]) and any text.find(...) calls) and ensure the code never calls len() or .find() on a None entry. Add the same guards there or ensure you never select an index whose search_list[index] is None.

3. Optionally simplify and centralize validation:
   - At the start of replace_each, validate inputs and build a filtered pair-list that excludes any pair where search_list[i] is None or replacement_list[i] is None (or where search_list[i] is empty), and then operate only on that filtered list. This prevents having to pepper guards everywhere.

Why these steps fix it
- They ensure the code never calls len() on None, so the TypeError cannot occur.
- They preserve the intended "null-safety inside arrays" behavior: when replacement_list[i] is None the function should treat it as "no replacement" (as the tests expect), rather than crash.

Summary
- Root cause: len(replacement_list[i]) is invoked before checking whether replacement_list[i] is None (line 82).
- Fix: check for None (and empty search strings) before any len(...) or .find(...) calls — either skip those entries in the early "increase" loop or build a filtered list of valid search/replacement pairs and use that everywhere.
