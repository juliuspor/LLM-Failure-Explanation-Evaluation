What happened (what failed)
- The function raises IndexError: "String index out of range: X" inside substring_java at line 34 (return s[start:end]). That means substring_java was called with end > len(text).
- substring_java is called in this function with end equal to either upper or index (calls at the branches that return substring_java(text, 0, upper) or substring_java(text, 0, index)).

Why it happened (root cause)
- You never validate lower against the length of text. That allows the variable lower to be greater than len(text).
- When that happens, this line runs:
    if upper < lower:
        upper = lower
  (so upper becomes a value > len(text))
- Later you call:
    result = substring_java(text, 0, upper)
  or
    return substring_java(text, 0, upper) + ...
  with upper > len(text). substring_java checks end > len(s) and raises IndexError (line 32–34 in the slice code), producing the failure.

Where in the code it occurs (specific lines)
- The problematic assignment is the if upper < lower: upper = lower check (the code that can make upper > len(text)).
- The exception is raised inside substring_java when returning s[start:end] (line 34 of the slice code) after substring_java(text, 0, upper) is called.

How to fix it (concrete steps)
You must ensure lower (and therefore upper after the adjustment) cannot exceed len(text) before calling substring_java. Two safe options:

1) Clamp lower to the string length
- After handling the upper == -1 / upper > len(text) case, add:
    if lower > len(text):
        lower = len(text)
- Then your existing if upper < lower: upper = lower will not produce upper > len(text).

2) Or clamp upper when you adjust it
- Replace:
    if upper < lower:
        upper = lower
  with:
    if upper < lower:
        upper = min(lower, len(text))

3) Or return early when lower is out of range (if that makes sense for your API):
    if lower > len(text):
        return text  # or substring_java(text, 0, len(text))

Which to choose depends on intended behavior:
- If you want "start at or after lower" semantics where an out-of-range lower should be treated as the end of the string, use option 1 or 3.
- If you want to preserve lower but never request an end > len(text), use option 2.
