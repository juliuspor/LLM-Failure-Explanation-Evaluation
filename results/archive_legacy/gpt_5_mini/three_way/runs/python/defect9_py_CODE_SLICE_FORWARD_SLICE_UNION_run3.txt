What happens
- At lines 81–85 (the "guess result buffer size" loop) the code does:
  greater = len(replacement_list[i]) - len(search_list[i])
  If either replacement_list[i] or search_list[i] is None this raises a TypeError ("object of type 'NoneType' has no len()") and the function crashes.

Why it fails (root cause)
- Earlier loops (lines ~59–76 and ~102–114) explicitly skip entries when search_list[i] is None, search_list[i] is empty, or replacement_list[i] is None. But the sizing loop at lines 81–85 does not perform the same null/empty checks. So when any element of search_list or replacement_list is None (or search_list[i] is ""), the call to len(...) blows up. The comment in the code even notes this: "BUG: this loop is not null-safe for elements inside the arrays."

Where it fails (exact location)
- The failure occurs in the loop starting at line 81:
  81 |         for i in range(search_length):
  82 |             greater = len(replacement_list[i]) - len(search_list[i])
  83 |             if greater > 0:
  84 |                 increase += 3 * greater  # assume 3 matches
  85 |         increase = min(increase, len(text) // 5)
- Example repro call:
  replace_each("abc", ["a", None], ["x", "y"])
  This will raise TypeError at line 82 when i == 1.

How to fix it (concrete steps)
1. Apply the same null/empty checks that are used elsewhere before using len(...). Replace the loop at lines 81–85 with:
   for i in range(search_length):
       if (
           no_more_matches_for_repl_index[i]
           or search_list[i] is None
           or len(search_list[i]) == 0
           or replacement_list[i] is None
       ):
           continue
       greater = len(replacement_list[i]) - len(search_list[i])
       if greater > 0:
           increase += 3 * greater
   increase = min(increase, len(text) // 5)

2. (Optional) If you prefer compactness, you can factor the condition into a helper function is_valid_pair(i) and reuse it in all three places so future edits stay consistent.

Why this fix works
- It prevents calling len(...) on None and matches the guard conditions used in the match-finding loops, so the sizing calculation only uses valid (non-null, non-empty) pairs.

No other changes are required to explain the crash.
