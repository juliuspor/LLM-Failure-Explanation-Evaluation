What failed
- The function DateTimeZone.for_offset_hours_minutes is rejecting valid negative minute values because of the check on minutes_offset at lines 396–397:
  - Lines shown:
    396 |         if minutes_offset < 0 or minutes_offset > 59:
    397 |             raise ValueError(f"Minutes out of range: {minutes_offset}")
- That condition treats any negative minute as out of range. Your tests call for_offset_hours_minutes with negative minutes (for example (0, -15), (0, -15) and (-2, -15)), so those calls raise ValueError immediately. That is why tests expecting negative-minute offsets fail.

Where it happens in the code
- Method: DateTimeZone.for_offset_hours_minutes
- Variable: minutes_offset
- Exact lines: 396–397 (the minutes range check)
- Downstream logic (lines 402 and 404) appears to handle sign adjustments (minutes_offset = hours_in_minutes - minutes_offset or hours_in_minutes + minutes_offset), so the implementation intends to accept negative minutes, but the early check prevents that code from ever running for negative minute values.

Why this is the root cause
- The code both intends to handle negative minutes and also forbids them. The contradiction is caused by the wrong bounds check:
  - It enforces 0..59, but the intended valid range is -59..+59 (per the rest of the logic and the tests).
- Because the negative-minute inputs are rejected before the sign-handling logic executes, tests that expect negative-minute offsets either fail or raise ValueError incorrectly.
