What happened (where and what failed)
- The bug is in the size-guess loop at lines 81–84 (the slice you gave). That loop calls len(replacement_list[i]) and len(search_list[i]) without checking whether either element is None.
- When one of the arrays contains None (for example replacement_list[0] is None in your tests), len(None) raises TypeError. That is exactly what breaks the tests that pass None inside the arrays.

Why it happens (root cause)
- Earlier in the function you carefully skip null/empty search strings and null replacements when searching for matches (the same checks appear in both the initial find loop and the loop inside the while).
- The "increase" estimation loop does not repeat those null/empty checks. It assumes every element is a str and calls len(...) directly.
- So for inputs like search_list = ["a"] and replacement_list = [None] the code reaches:
    greater = len(replacement_list[i]) - len(search_list[i])
  which becomes len(None) and raises TypeError.

Exact location
- Function: replace_each(...)
- Faulty lines: the "BUG" loop that starts at line 81 in your snippet:
    for i in range(search_length):
        greater = len(replacement_list[i]) - len(search_list[i])
        if greater > 0:
            increase += 3 * greater

How to fix (specific steps)
1. Skip any indices that are already marked no_more_matches_for_repl_index[i], or where search_list[i] is None or empty, or where replacement_list[i] is None — the same conditions used elsewhere in the function.
2. Replace the loop with the guarded version. For example:
    for i in range(search_length):
        if (
            no_more_matches_for_repl_index[i]
            or search_list[i] is None
            or len(search_list[i]) == 0
            or replacement_list[i] is None
        ):
            continue
        greater = len(replacement_list[i]) - len(search_list[i])
        if greater > 0:
            increase += 3 * greater

3. Keep the following line unchanged:
    increase = min(increase, len(text) // 5)

Why that fix works
- It prevents calling len(...) on None, matching the null-safety logic used elsewhere in the function. The size estimate will skip entries that should be ignored and will not raise TypeError.

Optional: small note on behavior
- The estimation loop is only an optimization; skipping problematic entries is safe because it only affects preallocation sizing, not the correctness of replacements.
