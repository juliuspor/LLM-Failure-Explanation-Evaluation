What happened
- The test fails with a TypeError: "object of type 'NoneType' has no len()" coming from line 82. That is because the code calls len(replacement_list[i]) and len(search_list[i]) without checking whether those entries are None.
- In the failing test test_null_safety_inside_arrays_single you pass replacement_list = [None]. At runtime the loop at lines 81–84 executes and tries to evaluate len(replacement_list[0]) (line 82), which raises the exception before the later null-safety checks run.

Where it occurs (exact lines)
- The immediate failure is at:
  - line 81: for i in range(search_length):
  - line 82: greater = len(replacement_list[i]) - len(search_list[i])
  - raised TypeError because replacement_list[i] (or search_list[i]) is None.
- Note there are null-safety checks later at lines 102–105:
  - those checks test "search_list[i] is None" or "replacement_list[i] is None" etc., but they execute after the length calculation above. So the code detects None too late.

Why this is the root cause
- The algorithm tries to estimate buffer growth (variable increase) by subtracting lengths of search and replacement entries. That estimate logically needs the lengths; but it does not guard against None entries before calling len(). The later logic expects None to mean "skip this replacement", but the preflight estimate step doesn't respect that and crashes.

How to fix (concrete steps)
1. In the loop that computes increase (lines 81–84), skip any index where either search_list[i] or replacement_list[i] is None or where search_list[i] is empty. Example replacement for that block:
   - if search_list[i] is None or replacement_list[i] is None or len(search_list[i]) == 0:
         continue
     greater = len(replacement_list[i]) - len(search_list[i])
     if greater > 0:
         increase += 3 * greater
2. This mirrors the later checks (lines 102–105) and prevents calling len(None).
3. Optionally add a unit test that exercises mixed None and non-None entries (your existing tests already do this) and confirm the estimate step no longer throws.

What you will observe after the fix
- The length-estimate loop will skip None entries and not call len(None). The null-safety checks later will still skip those replacements, so tests like test_null_safety_inside_arrays_single and test_null_safety_inside_arrays_mixed will pass.
