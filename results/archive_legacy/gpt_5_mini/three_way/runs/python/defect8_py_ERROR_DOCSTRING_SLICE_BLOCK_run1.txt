What happened (where it failed)
- Method: LocaleUtils.to_locale
- Input: locale_str = "fr__POSIX"
- Failure: ValueError("Invalid locale format: fr__POSIX")
- Location: the check at lines 239–240:
  ```
  if ch3 < 'A' or ch3 > 'Z' or ch4 < 'A' or ch4 > 'Z':
      raise ValueError(f"Invalid locale format: {locale_str}")
  ```

Why it failed (root cause)
- The code expects the pattern language[_COUNTRY][_variant], where:
  - language must be two lowercase letters,
  - COUNTRY (if present) must be two uppercase letters,
  - fields are separated by underscores.
- Your input "fr__POSIX" contains two consecutive underscores, meaning there is no COUNTRY part (empty string) and a variant "POSIX".
- In the implementation, after detecting the first underscore the code unconditionally reads ch3 and ch4 (the two characters it expects to be the COUNTRY letters).
  - For "fr__POSIX", ch3 is the second underscore '_' and ch4 is 'P'.
  - The check ch3 < 'A' or ch3 > 'Z' fails because '_' is not between 'A' and 'Z', so the code raises the ValueError.
- In short: the code assumes that if there is an underscore after the language then a two-letter COUNTRY must follow immediately. It does not allow the valid form language__variant where the country is empty.

Exactly what to change (clear, step-by-step fixes)
Option A — minimal fix (handle empty country before validating ch3/ch4)
1. Before validating ch3 and ch4, test whether ch3 == '_'. If so, treat the country as the empty string and parse the remainder as the variant.
2. Only perform the ch3/ch4 uppercase-letter check when ch3 is not '_'.
Concretely (pseudo-change around lines 235–240):
  - if ch3 == '_':
      - country = ""
      - variant = remainder after this underscore
  - else:
      - validate ch3 and ch4 are 'A'..'Z' (existing check)
      - set country = ch3+ch4
      - variant = remainder after ch4 (if any)

Option B — cleaner robust parsing (recommended)
1. Split the input into at most three parts using underscore: parts = locale_str.split('_', 2)
2. Validate parts:
   - parts[0] must be length 2 and all lowercase
   - if len(parts) >= 2 and parts[1] != "": parts[1] must be length 2 and all uppercase (otherwise country = "")
   - if len(parts) == 3: variant = parts[2] (can be any non-empty string)
3. This handles "fr", "fr_FR", "fr__POSIX", and "fr_FR_POSIX" correctly.

Option C — regex validation
Use a regex that allows an empty country when a variant exists:
  - r'^[a-z]{2}(?:_(?:[A-Z]{2})?(?:_.+)?)?$'
Then extract groups for country and variant accordingly.

Why these fixes work
- They stop assuming a country is always present when the first underscore exists.
- They explicitly permit the empty country case (double underscore) and correctly treat the following text as the variant.
- They keep the existing strict checks for language lowercase and country uppercase when country is present.

Which variables and lines to edit
- Edit LocaleUtils.to_locale, the code around the check at lines 239–240 where ch3 and ch4 are read and validated.
- Add the ch3 == '_' branch (or replace the manual index checks with split/regex parsing as in Option B/C).

This will prevent the unexpected ValueError for "fr__POSIX" and correctly produce language="fr", country="", variant="POSIX".
