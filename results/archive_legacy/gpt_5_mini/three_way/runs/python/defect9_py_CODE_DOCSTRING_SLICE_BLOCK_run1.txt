What happened (what fails)
- The code crashes with a TypeError at lines 81–84:
  > TypeError: object of type 'NoneType' has no len()
- That happens when either search_list[i] or replacement_list[i] is None because the code calls len(...) on them without checking for None.

Where it happens (precise location)
- In the first sizing loop:
  ```
  81 |         for i in range(search_length):
  82 |             greater = len(replacement_list[i]) - len(search_list[i])
  83 |             if greater > 0:
  84 |                 increase += 3 * greater  # assume 3 matches
  ```
  This loop runs before the later loops that properly skip None/empty entries. So if search_list[i] or replacement_list[i] is None, calling len(...) crashes the function.

Why it happens (root cause)
- The function intends to ignore entries where the search string is None/empty or the replacement is None (see the checks in the "Find the first match" loop and the "Find the next earliest match" loop).
- But the pre-sizing loop assumes every element is a string and unconditionally uses len(...) on both lists. That assumption is false when lists contain None, so the len(...) call raises a TypeError and the function crashes, even though later logic would have skipped those entries.

How to fix it (concrete changes)
Option A — Minimal fix: skip None/empty elements in the sizing loop. Replace the loop with:
```
for i in range(search_length):
    if search_list[i] is None or len(search_list[i]) == 0 or replacement_list[i] is None:
        continue
    greater = len(replacement_list[i]) - len(search_list[i])
    if greater > 0:
        increase += 3 * greater
```
This matches the same acceptance criteria used elsewhere in the function.

Option B — Slightly safer: factor the "is usable" test into a small helper and use it in all places (reduces duplication):

def usable(i):
    return (
        search_list[i] is not None
        and len(search_list[i]) > 0

for i in range(search_length):

Why these fixes solve it
- They avoid calling len(...) on None by ensuring both search_list[i] and replacement_list[i] are non-None (and search is non-empty) before using len(...).
- They maintain the intended logic of ignoring None/empty entries while letting the sizing loop estimate buffer growth only for entries that will actually be applied.

Summary
- Root cause: calling len(...) on None in the pre-sizing loop (lines 81–84).
- Fix: add the same None/empty checks used elsewhere before computing greater, or centralize that test and reuse it.
