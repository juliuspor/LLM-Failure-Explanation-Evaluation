What happened (where)
- The failure happens in the part of LocaleUtils.to_locale that parses 5+ character locale strings — specifically the code at/around lines 239–247 shown in your slice.
- At line 239 the code reads characters it expects to be the two-letter country code into variables ch3 and ch4 and then checks:
  if ch3 < 'A' or ch3 > 'Z' or ch4 < 'A' or ch4 > 'Z':
      raise ValueError(f"Invalid locale format: {locale_str}")
  (lines 239–240)
- For the input "fr__POSIX" the string characters are:
  index: 0='f', 1='r', 2='_', 3='_', 4='P', ...
  so ch3 = locale_str[3] = '_' and ch4 = locale_str[4] = 'P'.
- The check expects both ch3 and ch4 to be uppercase letters 'A'..'Z'. Because ch3 is '_' this condition fails and a ValueError is raised (your test error: "Invalid locale format because position 3 is '_' not uppercase letter").

Why it happened (root cause)
- The parser assumes that any 5+ character input that reaches this branch must contain a two-letter country code at indexes 3 and 4. It does not account for the valid Java-style form language__variant (language, empty country, variant) where there are two underscores in a row and the country is intentionally empty.
- Concretely: the code treats the second underscore of the "__" as a letter of the country and therefore rejects the input when that character is '_'.

Where to change it (concrete steps)
1. Detect the empty-country pattern before enforcing the uppercase-country check.
   - Add a branch before the uppercase-country validation (before lines 239–240) that checks for double underscore at positions 2 and 3:
     if locale_str[2] == '_' and locale_str[3] == '_':
         return Locale(locale_str[0:2], "", locale_str[4:])
   - For "fr__POSIX" this returns Locale("fr", "", "POSIX") because variant starts at index 4.
2. If you prefer to keep a single place of validation, change the validation logic so it allows ch3 == '_' (empty country) and then handles the variant accordingly. But the explicit double-underscore check is simpler and clearer.

Exact variables and lines to touch
- Lines to modify: just before the block at line 239 that reads ch3/ch4 and raises ValueError.
- Variables: locale_str, ch3, ch4
- Add the double-underscore check using locale_str[2] and locale_str[3], and return Locale(locale_str[0:2], "", locale_str[4:]) when true.

Example minimal fix (pseudo-patch)
- Insert before the existing validation:
  if locale_str[2] == '_' and locale_str[3] == '_':
      return Locale(locale_str[0:2], "", locale_str[4:])
- Keep the current uppercase-country validation for the normal language_country and language_country_variant cases.

Why this fix works
- It explicitly recognizes the language__variant form and returns the correct Locale with an empty country.
- It prevents the code from treating '_' as a country-letter and avoids the false "Invalid locale format" error for inputs like "fr__POSIX".
