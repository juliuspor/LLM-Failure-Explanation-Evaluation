What fails
- The code raises IndexError in code_point_at(...) (the check at slice line 35) while executing the loop in translate(...).
- This happens when the code calls code_point_at(code_units, pos) with pos == len(code_units). The code_point_at function checks "if index < 0 or index >= len(code_units)" and raises IndexError if index >= len(code_units).

Why it happens (root cause), with the exact variables/lines
1. You convert the input to UTF-16 code units and set:
   - pos = 0 (translate method)
   - length = len(code_units) (translate method)
2. You call consumed = self.translate_codepoint(code_units, pos, out).
   - The contract (comment) says translate_codepoint works in codepoints and "just took care of a surrogate pair".
   - It returns a positive consumed value when it handled one or more codepoints.
3. When consumed > 0 you run:
   for pt in range(consumed):
       pos += char_count(code_point_at(code_units, pos))
   (translate method)
4. The bug: code_point_at(code_units, pos) is called without guarding that pos is still < length. During the loop pos is advanced based on char_count(...). After some increments pos can become equal to length; on the next iteration code_point_at is called with index == len(code_units), and code_point_at raises IndexError (slice lines 35–36). In short: you call code_point_at with an out-of-range index because you increment pos in the loop and then call code_point_at again.

Why the code is wrong conceptually
- translate_codepoint returns the number of codepoints it consumed, but the loop advances pos by the number of UTF-16 code units per codepoint by repeatedly calling code_point_at at the current pos. That requires pos remain valid at each iteration. Because pos is changed inside the loop and there is no bounds check before each code_point_at call, the loop may try to read past the end of code_units.
- The code assumes there is always another code unit available when calling code_point_at; that assumption fails at the end of input.

Summary, succinctly
