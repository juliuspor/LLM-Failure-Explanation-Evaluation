What happened (where, exactly)
- The TypeError comes from this loop, right after the comment "# BUG: this loop is not null-safe for elements inside the arrays":

      for i in range(search_length):
          greater = len(replacement_list[i]) - len(search_list[i])
          if greater > 0:
              increase += 3 * greater  # assume 3 matches

- If either replacement_list[i] or search_list[i] is None, calling len(...) raises the observed error: "object of type 'NoneType' has no len()".
- The failing test test_null_safety_inside_arrays_single passes replacement_list = [None]. That makes replacement_list[0] None, so len(replacement_list[0]) triggers the TypeError.

Why it happened (root cause)
- Earlier in the function the code correctly treats None or empty search strings and None replacements as "skip this pair" when searching for matches (the two loops that call text.find(...) check search_list[i] is None, len(...) == 0, or replacement_list[i] is None).
- But this pre-sizing loop (the one that computes increase) does not perform the same null/empty checks. It blindly computes lengths for every pair and therefore chokes when an element is None.
- In short: inconsistent null-safety. The search/replacement loops check for None; the size-estimate loop does not.

How to fix it (concrete steps and exact code)
Option A — skip null/empty items (preferred, matches other logic):

Replace the buggy loop with:

    for i in range(search_length):
        if search_list[i] is None or len(search_list[i]) == 0 or replacement_list[i] is None:
            continue
        greater = len(replacement_list[i]) - len(search_list[i])
        if greater > 0:
            increase += 3 * greater  # assume 3 matches

Option B — treat None as zero-length explicitly (less consistent with other checks):

    for i in range(search_length):
        rep_len = len(replacement_list[i]) if replacement_list[i] is not None else 0
        search_len = len(search_list[i]) if search_list[i] is not None else 0
        greater = rep_len - search_len
        if greater > 0:
            increase += 3 * greater

Why Option A is better here
- The rest of the function already skips pairs where search is None/empty or replacement is None; Option A keeps that behaviour for the sizing pass so you won't over-count or reserve space for entries that are intentionally ignored.
