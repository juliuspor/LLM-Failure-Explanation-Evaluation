What happened (root cause)
- The code assumes replace_index always points to an entry whose search string is a str. At line 94 you do:
  start = text_index + len(search_list[replace_index])
  If search_list[replace_index] is None, len(None) raises TypeError: 'NoneType' has no len().
- Why replace_index can be an index whose search_list entry is None:
  - The loop that chooses replace_index runs at lines 102–113. That loop is intended to skip candidates whose search_list[i] is None or whose replacement_list[i] is None, but the code later still uses replace_index without re-checking that search_list[replace_index] is not None. In other words, selection and use are not guarded consistently: replace_index can end up referencing a None search_list and then len(...) is called on it at line 94.
  - The crash therefore happens when a None appears in search_list or replacement_list and the selection logic lets replace_index refer to that index, so len(search_list[replace_index]) is evaluated on None.

Exactly where it fails
- The exception is raised at line 94: start = text_index + len(search_list[replace_index]) because search_list[replace_index] is None.
- The selection logic that sets replace_index is in lines 102–113; those lines are responsible for allowing a None index through (or not protecting the later len call).

What to change (concrete fixes, step-by-step)
Pick one of these clear fixes:

1) Guard before using len(...) (minimal, safe)
- Immediately before line 94, add an explicit check:
  if replace_index is None or search_list[replace_index] is None:
      # nothing to replace; append rest of text and break/continue appropriately
      parts.append(text[start:])
      break
  Then compute start using len(search_list[replace_index]).
- This prevents calling len(None).

2) Skip None entries when choosing replace_index (prevents ever selecting them)
- In the candidate-selection loop (lines 102–113), ensure indices with None are skipped. For example, add at the top of that loop:
  if search_list[i] is None or replacement_list[i] is None:
      no_more_matches_for_repl_index[i] = True
      continue
- That guarantees replace_index will never be set to an index whose search_list is None.

3) Reorder or combine checks so len() is never evaluated on a None
- If you prefer a one-line guard where len() is used, replace len(search_list[replace_index]) with:
  s = search_list[replace_index]
  if not s:  # covers None and empty string
      parts.append(text[start:])
      break
  start = text_index + len(s)

Which fix to choose
- If you want minimal change, use (1).
- If you want the cleanest logic (never consider None entries at all), use (2).

Why these fixes solve the exact problem
- They ensure len() is only ever called on actual strings, so the TypeError is impossible.
- They make the selection and consumption of replacements consistent: indices marked as None are either skipped up-front or explicitly handled at use-time.
