What happened (root cause)
- The failure is in TimePeriodValues._update_bounds when it updates the "max middle" tracking.
- Inside _update_bounds the code computes the current period's middle time into a variable named middle (or mid). When it compares that value to the saved maximum it uses the wrong variable for the comparison — it compares middle to the index variable instead of to the saved numeric maximum.
- Concretely, instead of doing
    if middle > max_middle:
it does something like
    if middle > max_middle_index:
or
    if middle > index
which compares a timestamp (milliseconds) to an index (an integer position). That makes the condition true for the wrong rows and so self._max_middle_index gets set to the wrong index (for your test it ends up 3 instead of 1).

Where it occurs
- The bug is in TimePeriodValues._update_bounds. The failing behavior you see in test_max_middle_index_simple (expected get_max_middle_index() == 1 but got 3) is a direct symptom.
- Look for the block that:
  - computes middle (midpoint of period start and end),
  - fetches the current saved max middle (e.g. max_middle or max_middle_value) using self._max_middle_index,
  - then compares the newly computed middle with that saved max,
  - and assigns self._max_middle_index = index when a new max is found.
- One of those comparison lines uses the index variable (self._max_middle_index or index) instead of the saved numeric value (max_middle). That wrong variable name causes the faulty behavior.

Why that causes the wrong index
- Example: suppose existing max_middle is 150 and its index is 1. The code incorrectly compares middle (e.g. 75) to max_middle_index (1). 75 > 1 is true, so it treats the 75 value as a new maximum and sets _max_middle_index to that record's index (e.g. 3). That exactly matches the failing output you saw (returned index 3 instead of 1).
