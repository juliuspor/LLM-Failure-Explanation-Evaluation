What happened (where it failed)
- At line 176 your code runs: if x == int(x) and not math.isinf(x) and not math.isnan(x):
- For x == -0.0 this condition is true because -0.0 == 0.0 and int(-0.0) == 0.
- Downstream you convert or return int(x) (or otherwise produce an integer 0), so the code returns 0 (a positive zero integer) instead of preserving negative zero.
- The test expected -0.0, but got 0, so the assertion failed.

Why this happens (root cause)
- IEEE floating-point has a negative zero bit-pattern distinct from positive zero, but comparisons treat -0.0 == 0.0 as True.
- int(-0.0) produces 0 and loses the sign bit entirely. Your condition uses x == int(x) and then collapses the value to an integer or to a non-signed-zero float, so the negative-zero information is lost before returning.

Exactly where to look
- Line 176: the equality check x == int(x) is the trigger — it accepts -0.0 as an “integer-valued” float.
- Immediately after that branch the code converts/returns int(x) or a value derived from int(x), which removes the negative-zero sign.

Concrete fixes (step-by-step)
1. Decide whether you must preserve negative zero. If yes, do not turn -0.0 into an int or otherwise erase its sign.
2. Detect negative zero explicitly before you convert:
   - Use math.copysign or a sign test: math.copysign(1.0, x) < 0 or (x == 0.0 and math.copysign(1.0, x) < 0). Example:
     - if x == int(x) and not math.isinf(x) and not math.isnan(x):
         if x == 0.0 and math.copysign(1.0, x) < 0:
             return -0.0   # preserve negative zero
         return int(x)
3. Or always return a float instead of an int when x is zero:
     - if x == int(x) and not math.isinf(x) and not math.isnan(x):
         return float(int(x)) if x != 0.0 else x
   That keeps -0.0 when x is -0.0 because you return x unchanged for zero.
4. If you need a reliable bit-level test, use struct.unpack to check the sign bit, but math.copysign is simpler and clear.

Summary
- The bug is that the equality check treats -0.0 as an integer-valued float and your code converts it to 0, losing the negative-zero sign. Detect negative zero explicitly (using math.copysign or returning x unchanged for zero) before converting to int, and the test will pass.
