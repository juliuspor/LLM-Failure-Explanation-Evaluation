What failed
- In the "max middle" section you fetch the wrong item. This line:
  max_middle_period = self.get_data_item(self._min_middle_index).get_period()
  should use _max_middle_index, not _min_middle_index.

Where it occurs (approx.)
- Inside _update_bounds(), in the block that begins with:
    if self._max_middle_index >= 0:
  the code uses self._min_middle_index when calling get_data_item(...).get_period().

Why it causes the failure
- If _max_middle_index >= 0 but _min_middle_index is still -1, get_data_item(self._min_middle_index) will get an invalid index and typically raise an IndexError (or return the wrong item if negative indexing is allowed).
- Even if both indices are valid, using the min index for the max comparison compares the new item's middle against the wrong existing middle value. That prevents correct updates to _max_middle_index and corrupts the min/max tracking logic.

Exact spot to change
- Replace:
    max_middle_period = self.get_data_item(self._min_middle_index).get_period()
  with:
    max_middle_period = self.get_data_item(self._max_middle_index).get_period()

Suggested fixes (concrete steps)
1. Make the single-line fix above.
2. Run unit tests (or add tests) that cover:
   - adding the first period (ensures indices initialize from -1)
   - adding periods whose middle should update min and max separately
   - the case where _min_middle_index < 0 while _max_middle_index >= 0 (to expose the current failure)
3. (Optional, recommended) Reduce duplicate code by adding a small helper that returns start/middle/end in milliseconds for a period. That lowers the chance of similar copy-paste typos.
