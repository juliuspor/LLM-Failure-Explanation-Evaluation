What happened (root cause)
- You correctly clamp the input into variable v:
  v = max(value, self._lower_bound)
  v = min(v, self._upper_bound)
- But when you compute the gray channel you still use the original value variable instead of the clamped v:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- If value is outside the bounds (e.g. -0.5 or 1.5), the formula computes a g < 0 or > 255 and the subsequent check raises ValueError. That is why tests that expect clamping fail.

Where it occurs
- The bug is in the line that computes g: it should use the clamped v, not value. The clamping lines above are effectively ignored because you never use v afterward.

Why that produces the failure
- Tests call get_paint(-0.5). v becomes self._lower_bound (0.0), but g is calculated from -0.5, producing a negative g -> ValueError. The code raises instead of returning black (0,0,0).

Exact fixes (step-by-step)
1. Use the clamped variable v when calculating g:
   Replace
     g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
   with
     g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

2. Protect against zero division if lower and upper bounds are equal:
   if self._upper_bound == self._lower_bound:
       g = 0  # or raise an explicit ValueError explaining equal bounds are invalid

3. Ensure g is final-clamped to [0,255] (defensive coding):
   g = max(0, min(255, g))

4. (Optional) Use rounding if you want nearest integer instead of truncation:
   g = int(round(...))

Minimal corrected get_paint body
- Clamp to v (as you already do).
- Compute g using v.
- Handle equal bounds and final clamp.

That will stop the ValueError for out-of-range inputs and make the method return expected colors for the tests.
