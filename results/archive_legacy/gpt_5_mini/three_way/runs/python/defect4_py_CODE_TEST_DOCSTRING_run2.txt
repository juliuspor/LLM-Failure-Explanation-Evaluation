What happened (root cause)
- In the "max middle" block you fetch the stored period using the wrong index variable:
  max_middle_period = self.get_data_item(self._min_middle_index).get_period()
  — that line uses self._min_middle_index instead of self._max_middle_index.
- Because you compare the incoming period's middle against the min_middle period (not the max_middle), the comparison middle > max_middle is performed against the wrong baseline. That explains the test failure where the final max middle index became 3 instead of the expected 1.

- In the block that updates self._max_middle_index (the fourth if-block in _update_bounds). The incorrect call is:
  max_middle_period = self.get_data_item(self._min_middle_index).get_period()
  This should reference self._max_middle_index.

Why this causes the observed behavior (step-by-step for your failing test)
- You add periods with mids: index 0 -> 50, index 1 -> 150, index 2 -> 25, index 3 -> 75.
- When processing later entries, the code that compares to update _max_middle_index is using the period at _min_middle_index (the smallest middle), not the current best max.
- Since _min_middle_index points to a very small middle (e.g. 25 at index 2), most later middles are greater than that, so the condition middle > max_middle (where max_middle is actually the min middle) will be true often and the code will repeatedly set _max_middle_index to the most recently seen index. That explains why you end up with index 3 instead of 1.
