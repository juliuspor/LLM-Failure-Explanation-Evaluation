What happened
- The code in LocaleUtils.to_locale rejects strings like "fr__POSIX" because it expects a two-letter country code at positions 3–4 and validates those characters as uppercase letters.
- For "fr__POSIX" position 3 contains '_' (the second underscore), so the test at lines 237–239:
  ch3 = locale_str[3]
  ch4 = locale_str[4]
  if ch3 < 'A' or ch3 > 'Z' or ch4 < 'A' or ch4 > 'Z':
      raise ValueError(f"Invalid locale format: {locale_str}")
  raises ValueError. The message you saw ("Invalid locale format because position 3 is '_' not uppercase letter") is exactly this check failing.

Why it fails (root cause)
- The implementation assumes the format is either:
  - "ll" (language only), or
  - "ll_CC" (language + country), or
  - "ll_CC_variant" (language + country + variant)
- It has no branch to handle the "ll__variant" case (empty country, double underscore). In that case the country fields (ch3/ch4) are underscores, not uppercase letters, so the uppercase-letter validation fails.
- The relevant variables and lines:
  - length is computed at line 222.
  - the problematic indexing and validation are at lines 237–239 (ch3 = locale_str[3], ch4 = locale_str[4], uppercase check).
  - the code raises ValueError at line 239 when ch3 is '_' (not between 'A' and 'Z').

       return Locale(locale_str[0:2], "", locale_str[4:])
   This detects "ll__variant" (language + empty country + variant) and returns Locale(language, "", variant).
2. Keep existing checks for the normal "ll_CC" and "ll_CC_variant" cases afterwards.
3. Ensure you do length/bounds checks before indexing ch3/ch4 (you already compute length at line 222; use that to avoid IndexError).

Why this fix works

- It preserves the strict validation for normal country codes and variants.
