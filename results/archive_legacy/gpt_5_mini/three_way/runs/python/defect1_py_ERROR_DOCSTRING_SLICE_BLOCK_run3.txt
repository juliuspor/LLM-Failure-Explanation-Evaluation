What happened (what, where)
- The factory method that "Gets a time zone instance for the specified offset" raised ValueError("Minutes out of range: -15") at lines 396–397.
- The offending check is these lines:
  ```
  396 |         if minutes_offset < 0 or minutes_offset > 59:
  397 |             raise ValueError(f"Minutes out of range: {minutes_offset}")
  ```
- The code rejected minutes_offset = -15 even though the docstring allows negative minutes in some cases (for example (0, -15) → -00:15 and (-2, -15) → -02:15).

Why it failed (root cause)
- The condition minutes_offset < 0 unconditionally forbids any negative minutes. That contradicts the documented behavior which allows negative minutes when hours_offset is zero or negative.
- The check only enforces the range 0..59. The documented valid range for minutes is actually -59..+59 (with additional sign rules relative to hours), not 0..59.
- Because of that unconditional negative check, a valid call such as hours_offset = -2, minutes_offset = -15 (or hours_offset = 0, minutes_offset = -15) is wrongly rejected at line 396.

What should be changed (concrete fixes, with steps and exact code)
1. Enforce allowed absolute range for minutes: minutes_offset must be between -59 and +59.
2. Reject the combination where hours_offset > 0 and minutes_offset < 0 (docstring example: (2, -15) is invalid).
3. Continue to check the overall offset does not exceed +/- 23:59:59.000 if your code has that later.

Suggested replacement (exact change to lines 396–397)
Replace:
```
if minutes_offset < 0 or minutes_offset > 59:
    raise ValueError(f"Minutes out of range: {minutes_offset}")
```
With:
```
# minutes must be in -59..+59
if minutes_offset < -59 or minutes_offset > 59:
    raise ValueError(f"Minutes out of range: {minutes_offset}")

# a positive hour with negative minutes is not allowed (e.g. (2, -15) invalid)
if hours_offset > 0 and minutes_offset < 0:
    raise ValueError(f"Positive hour with negative minutes: hours={hours_offset}, minutes={minutes_offset}")
```

Examples (to confirm behavior)
- Allowed: (2, 15) -> +02:15
- Allowed: (2, 0) -> +02:00
- Disallowed: (2, -15) -> raises (explicit check above)
- Allowed: (0, -15) -> -00:15
- Allowed: (-2, -15) -> -02:15

Additional note
- After this fix, ensure there is an overall bounds check for the total offset (hours*3600 + minutes*60) so the final offset does not exceed +/- 23:59:59 if that constraint is required (your docstring mentions it). If such a check exists elsewhere, it will now receive correct minute values.

This addresses the immediate root cause (unconditional rejection of negative minutes at lines 396–397) and gives the exact replacement logic to match the documented behavior.
