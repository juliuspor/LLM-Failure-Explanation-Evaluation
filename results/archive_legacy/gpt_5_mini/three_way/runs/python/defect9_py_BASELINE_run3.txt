What happened (what failed)
- The TypeError "object of type 'NoneType' has no len()" is raised in the loop at lines 81–84:
  ```
  for i in range(search_length):
      greater = len(replacement_list[i]) - len(search_list[i])
      if greater > 0:
          increase += 3 * greater
  ```
- That code calls len(replacement_list[i]) and len(search_list[i]) with no null checks. If either replacement_list[i] or search_list[i] is None, len(None) raises the TypeError seen in the test.

Why it happens (root cause)
- Earlier in the function (lines ~59–75 and again later in the matching loop) you correctly skip entries where search_list[i] is None, search_list[i] is empty, or replacement_list[i] is None:
  ```
  if (
      no_more_matches_for_repl_index[i]
      or search_list[i] is None
      or len(search_list[i]) == 0
      or replacement_list[i] is None
  ):
      continue
  ```
- But the "guess result buffer size" loop that computes increase (lines 81–84) runs over all indices without the same null/empty checks. When a test provides replacement_list entries that are None (e.g. ["c", None] in test_null_safety_inside_arrays_mixed), that loop will try len(None) and crash. The bug is that this loop is not null-safe for elements inside the arrays (the comment even points this out).

Where it occurs (exact variables & lines)
- Faulty code: lines 81–84:
  - Variables involved: replacement_list[i], search_list[i]
  - Operation causing error: len(replacement_list[i]) when replacement_list[i] is None
- Triggering test: TestStringUtilsReplaceEach.test_null_safety_inside_arrays_mixed calls
  replace_each("aba", ["a", "b"], ["c", None])

How to fix it (concrete steps)
1. Skip indices where search_list[i] is None or empty or replacement_list[i] is None — same guards used in the match-finding loops.
2. Replace the loop with a guarded version. For example:
   ```
   for i in range(search_length):
       if search_list[i] is None or len(search_list[i]) == 0 or replacement_list[i] is None:
           continue
       greater = len(replacement_list[i]) - len(search_list[i])
       if greater > 0:
           increase += 3 * greater
   ```
3. (Optional) Keep the no_more_matches_for_repl_index check if you also want to exclude indices already known to have no matches:
   ```
   if no_more_matches_for_repl_index[i] or ...
   ```
4. Run the test suite to confirm the TypeError is gone and that null replacements are still ignored.

Why this fix is correct
- It applies the same null/empty checks used elsewhere in the function, so the increase estimate only uses valid string entries and will not attempt len(None). This restores the intended "null-safe" behavior described in the docstring and the tests.
