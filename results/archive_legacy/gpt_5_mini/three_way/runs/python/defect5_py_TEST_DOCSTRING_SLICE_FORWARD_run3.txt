What happens
- The failing case is test_add_null_to_null_with_expected_type (near the top of the test file). It calls:
  ArrayUtils.add(None, None, expected_type=str)
  The test expects a one-element list [None].
- The implementation of ArrayUtils.add is doing a strict type check that treats None like any other value. Concretely, it checks something like:
    if expected_type is not None and not isinstance(element, expected_type):
        raise TypeError("element type mismatch")
  Because element is None, isinstance(None, str) is False, so the method raises TypeError instead of returning [None].

Why that is wrong
- The docstring says: "If the input list is None, a new one element list is returned whose component type is the same as the element, unless the element itself is None, in which case the return type is List[object]." That implies None should be acceptable for any expected_type and the method should return [None] rather than raising.
- In other words, None is a valid element value and must not be rejected just because it is not an instance of the expected element type.

Where it occurs (what to change)
- In ArrayUtils.add, locate the type-check block that looks at expected_type and element/type compatibility. It will be right before the code that constructs the result list.
- Change the type check so that None is exempt. Concretely, replace:

    if expected_type is not None and not isinstance(element, expected_type):
        raise TypeError(...)

  with:

    # Allow None values regardless of expected_type
    if expected_type is not None and element is not None and not isinstance(element, expected_type):
        raise TypeError(...)

- Do the same change (the same reasoning) in ArrayUtils.add_at_index if it has an equivalent check for expected_type/element.

Why this fix works
- With the change, when element is None the isinstance check is skipped and the method can create and return [None] (or append None to a copy of array). This matches the docstring and the test's expectation.
- It also preserves correct runtime type checks for non-None elements: if you call ArrayUtils.add(arr, "x", expected_type=int) the check still raises TypeError.

Example patch (conceptual)
- Before:

    def add(array, element, expected_type=None):
        # ...
        if expected_type is not None and not isinstance(element, expected_type):
            raise TypeError("element type mismatch")
        # create and return new list

- After:

    def add(array, element, expected_type=None):
        # ...
        # Allow None to be stored regardless of expected_type
        if expected_type is not None and element is not None and not isinstance(element, expected_type):
            raise TypeError("element type mismatch")
        # create and return new list

Other notes
- This is a logic bug (overly-strict type check), not an import/path or test problem.
- If you want to be extra explicit, you can also document the behavior in the function docstring: "None is allowed as an element even when expected_type is supplied."
