- You intended to clamp value into [self._lower_bound, self._upper_bound] using v (lines 66–69), but you never use v in the computation of g. That lets out-of-range input produce g < 0 or g > 255 and triggers the ValueError.

Why it happens (root cause)
- Root cause: a simple variable mixup. You compute v = max(value, self._lower_bound); v = min(v, self._upper_bound) but then ignore v and keep using value in the formula for g. If value is less than lower_bound or greater than upper_bound, the numerator (value - lower_bound) will be outside the [0, upper-lower] range, so the fraction can be < 0 or > 1 and g becomes outside 0..255.
- Secondary edge-case: if self._upper_bound == self._lower_bound, the denominator is zero and (value - self._lower_bound) / (...) will raise ZeroDivisionError or produce Infinity — you should guard against that.

    v = max(value, self._lower_bound)
    v = min(v, self._upper_bound)

    # Prevent division by zero
    if self._upper_bound == self._lower_bound:
