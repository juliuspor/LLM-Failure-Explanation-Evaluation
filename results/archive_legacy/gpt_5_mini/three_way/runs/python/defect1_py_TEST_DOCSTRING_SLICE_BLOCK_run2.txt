What failed
- The test suite exercises negative minute offsets (e.g. DateTimeZone.for_offset_hours_minutes(0, -15), (-2, -15), (-2, 15) etc.).
- The implementation rejects any negative minutes unconditionally at lines 396–397:

  if minutes_offset < 0 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")

- That check only permits minutes in [0, 59], but the docstring and tests expect minutes to be allowed in [-59, +59]. As a result, calls that should be valid (minutes negative when hours are zero or negative) raise ValueError instead of returning the expected zone.

Where it occurs (exact names/lines)
- Method: DateTimeZone.for_offset_hours_minutes
- Variables: minutes_offset (checked), hours_offset (used elsewhere in method)
- Faulty lines: 396–397 (the minute-range check)

Why it happens (root cause)
- The minutes range validation is wrong: it treats negative minutes as out-of-range even though the docstring says minutes must be in -59..+59. The code uses minutes_offset < 0 rather than minutes_offset < -59.
- Because of that, tests that pass negative minute values fail immediately at that validation step.

What to change (concrete steps)
1. Allow minutes in the full range -59..+59:
   Replace the current check with:
   if minutes_offset < -59 or minutes_offset > 59:
       raise ValueError(f"Minutes out of range: {minutes_offset}")

2. Enforce the sign-combination rules described by the docstring:
   - If hours_offset > 0 and minutes_offset < 0: raise ValueError (positive hours with negative minutes is invalid).
   - If hours_offset < 0 and minutes_offset > 0: accept it (treat total offset as negative, i.e. -hours - minutes), as docstring shows (-2, 15) -> -02:15.
   - Zero hours may combine with positive or negative minutes per docstring.

   Example code sketch to add after the range check:
   if hours_offset > 0 and minutes_offset < 0:
       raise ValueError("Positive hours with negative minutes is not allowed")
   # minutes with negative hours or zero hours are allowed per docstring

3. Keep existing bounds check for hours_offset (ensure -23..+23) and final total-offset limit check (<= +/- 23:59:59.000) if present.

Suggested replacement fragment (lines ~396 onward)
