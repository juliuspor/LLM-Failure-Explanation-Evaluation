What happens
- At lines 335–338 you try to infer the element type with:
  - if array is not None: inferred_type = type(array)
  - elif element is not None: inferred_type = type(element)
- That is wrong: type(array) returns the container class (e.g. list), not the element type. So when you pass a non-empty list you get inferred_type = list, not e.g. int or str.
- When both array and element are None you fall through to the branch that sets inferred_type = object (line ~340).
- Immediately after (lines 346–347) the code does:
    if inferred_type == object and expected_type != object:
        raise TypeError(...)
  So if the caller provided expected_type=str but both array and element were None, the function raises a TypeError instead of returning [None].

Where this shows up in the tests
- test_add_null_to_null_with_expected_type calls ArrayUtils.add(None, None, expected_type=str). According to the test comment it should return [None], but the code sets inferred_type = object and then raises at line 346–347 because expected_type is str. That is the direct cause of the failure.

Why that is wrong (concise)
- type(array) is the wrong thing to inspect for element type.
- Raising when both array and element are None ignores the caller’s clear intent when they provided expected_type — the function should use expected_type to construct the result instead of throwing.

How to fix (specific steps and code)
1. Infer element type from the array contents, not from the array object:
   - If array is not None and non-empty, find a first non-None element and use type(that_element).
   - If array is empty ([]) or contains only None, don't set inferred_type = object automatically — fall back to expected_type if provided, else object.
2. If expected_type is supplied, prefer it when you can’t infer a more specific element type (do not raise simply because inferred_type == object).
3. Do not raise TypeError merely because inferred_type == object and expected_type != object; instead use expected_type to construct/return the array.

Minimal concrete replacement (pseudo-Python to insert around lines 335–347):

    # old:
    # if array is not None:
    #     inferred_type = type(array)
    # elif element is not None:
    #     inferred_type = type(element)
    # ...
    # inferred_type = object
    # if inferred_type == object and expected_type != object:
    #     raise TypeError(...)

    # new:
    if array is not None:
        # Try to infer element type from first non-None element
        inferred_type = None
        for e in array:
            if e is not None:
                inferred_type = type(e)
                break
        if inferred_type is None:
            # array exists but is empty or all-None
            inferred_type = expected_type if expected_type is not None else object
    elif element is not None:
        inferred_type = type(element)
    else:
        # both array and element are None -> use expected_type if provided
        inferred_type = expected_type if expected_type is not None else object

    # Do not raise simply because inferred_type == object and expected_type != object.
    # Instead, prefer expected_type when provided:
    if expected_type is not None and inferred_type != expected_type:
        # optionally validate compatibility, else set element type to expected_type
        inferred_type = expected_type

Notes and checks
- Use expected_type if the caller passed one (the test passed expected_type=str).
- If you need runtime validation (to mirror Java type safety) you could check that non-None elements are instances of expected_type and raise only when there's an actual mismatch (not just because inferred_type is object).
- Ensure you treat an actual array (list) differently from the element type everywhere else in the code; replace uses of type(array) with the inferred element type logic above.

Summary
- Root cause: type(array) used instead of element type and an unconditional raise when inference yields object while expected_type is provided.
- Fix: infer element type from array contents, fall back to expected_type when inference fails, and do not raise in the None/None/expected_type case. This will make ArrayUtils.add(None, None, expected_type=str) return [None] as the test expects.
