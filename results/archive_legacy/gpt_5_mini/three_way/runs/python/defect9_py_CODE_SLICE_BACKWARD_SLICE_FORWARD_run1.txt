What happened (root cause)
- The failure occurs in the loop that computes the estimated output-size increase (around lines 81–84).
- That loop does:
    greater = len(replacement_list[i]) - len(search_list[i])
  If either replacement_list[i] or search_list[i] is None, calling len(None) raises a TypeError:
    TypeError: object of type 'NoneType' has no len()
- The rest of the function already skips None/empty values when scanning for matches (see the checks at the first matching loop around lines 59–70 and again in the while-loop at lines 102–105). But this precompute loop did not perform the same null/empty checks, so it can crash when any element is None.

Where it occurs
- Function: replace_each (top of snippet).
- Exact location: the "BUG" comment and the loop immediately below it (lines 81–84 in your slice):
    for i in range(search_length):
        greater = len(replacement_list[i]) - len(search_list[i])
        if greater > 0:
            increase += 3 * greater

Why it happened
- The code assumes every element in search_list and replacement_list is a str. That assumption is false: the function accepts Optional[List[Optional[str]]] and later code explicitly handles None values. The precompute loop forgot to apply the same guards, so a None element causes len(None) to raise.

How to fix it (concrete steps)
1. Apply the same null/empty guards in that loop before computing lengths. For example, replace the buggy loop with:

    for i in range(search_length):
        if search_list[i] is None or replacement_list[i] is None or len(search_list[i]) == 0:
            continue
        greater = len(replacement_list[i]) - len(search_list[i])
        if greater > 0:
            increase += 3 * greater

This mirrors the checks at lines 59–70 and 102–105 so you won't call len(None).

2. (Optional) If you prefer a defensive style, you can compute safe lengths without repeated checks:

    for i in range(search_length):
        s = search_list[i]
        r = replacement_list[i]
        if not s or r is None:    # s falsy catches None and empty string
            continue
        greater = len(r) - len(s)
        if greater > 0:
            increase += 3 * greater

3. Add a unit test that includes None in either list to ensure the function no longer raises:
    - text = "aabbcc"
    - search_list = ["aa", None, "cc"]
    - replacement_list = ["x", "y", None]
  Call replace_each and assert it doesn't raise (and that behavior matches your intended semantics — currently None replacements are skipped).

Why this fix is correct
- It prevents calling len() on None by skipping entries that other parts of the function already skip.
- It keeps the estimation logic intact while making it consistent with the rest of replace_each.

Notes
- If you intend None entries to be treated differently (for example, as empty string or to cause errors earlier), adjust the guard or input validation accordingly. The minimal fix above preserves the current function's apparent intent of ignoring None/empty search entries and None replacements.
