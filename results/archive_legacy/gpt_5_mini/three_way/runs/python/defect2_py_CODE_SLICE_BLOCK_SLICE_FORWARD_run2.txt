What happened (where it failed)
- At line that computes g you used the original parameter value instead of the clamped local v:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
  (this is the line that leads to the ValueError at lines 74–75).
- Because you clamp into v but then compute g from value, g can be outside 0–255 when value is outside the bounds. That triggers the ValueError at lines 74–75.

Why it happened (root cause)
- You intended to map the clamped value into the 0–255 range, but you accidentally used the un-clamped variable value in the formula. If value < self._lower_bound or value > self._upper_bound, the numerator (value - self._lower_bound) produces results that map outside [0,255], so g ends up <0 or >255 and the check raises.
- There is a second potential issue: if self._upper_bound == self._lower_bound the denominator becomes zero, which would raise a ZeroDivisionError. The code currently does not handle that case.

How to fix it (clear step-by-step changes)
1. Use the clamped variable v in the mapping formula (replace value with v).
2. Protect against zero-length range (upper == lower) by returning a defined color or raising a clear exception.
3. Optionally clamp g to [0,255] after converting to int to be defensive.

Minimal corrected code (apply these replacements)
    def get_paint(self, value: float) -> Tuple[int, int, int]:
        # Clamp value to bounds
        v = max(value, self._lower_bound)
        v = min(v, self._upper_bound)
        
        # Guard against division by zero
        if self._upper_bound == self._lower_bound:
            # choose behavior: return middle gray, or raise a clear error
            return (128, 128, 128)
        
        # Use clamped v here (was using value)
        g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
        
        # Defensive clamp to 0..255
        if g < 0:
            g = 0
        elif g > 255:
            g = 255
        
        return (g, g, g)

What to expect after the fix
- Values outside bounds will be mapped correctly to 0 or 255 instead of causing ValueError.
- You will no longer get ZeroDivisionError when upper==lower (because it's handled explicitly).
- The ValueError at lines 74–75 will stop occurring for out-of-range g once you use v and/or clamp defensively.
