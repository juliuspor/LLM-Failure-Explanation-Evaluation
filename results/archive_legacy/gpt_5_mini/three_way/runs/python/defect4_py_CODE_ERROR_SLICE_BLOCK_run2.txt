What happened (root cause)
- In the "max middle" section you fetch the wrong existing period. Instead of using self._max_middle_index you use self._min_middle_index:
  max_middle_period = self.get_data_item(self._min_middle_index).get_period()
- Because you compare the new middle against the period at min_middle_index instead of the one at max_middle_index, the comparison can be wrong and a later item will be chosen as the max middle. That explains the failed assertion: the code compared against the wrong baseline and set self._max_middle_index to 3 when it should have stayed 1.

Where it occurs (exact line/variables)
- The error is in the "if self._max_middle_index >= 0:" block:
  - Wrong variable used: self._min_middle_index (fetch)
  - Intended variable: self._max_middle_index
- Variables involved: max_middle_period, self._min_middle_index, self._max_middle_index, middle, max_middle

Why that causes the observed result
- The algorithm should compare the new middle value to the current maximum middle value (from index self._max_middle_index). By using self._min_middle_index you compare against some other period (the minimum-middle period), which typically has a smaller middle value — so many new entries look larger and incorrectly replace the max index. That yields an incorrect self._max_middle_index (3 instead of 1).

How to fix it (concrete steps)
1. Replace the wrong fetch line inside the max-middle block:
   - Change
     max_middle_period = self.get_data_item(self._min_middle_index).get_period()
     to
     max_middle_period = self.get_data_item(self._max_middle_index).get_period()
2. Run the unit tests. The failing test expecting max_middle_index=1 should now pass.
3. Optional hardening:
   - Add a small unit test specifically exercising _min_middle_index/_max_middle_index updates.
   - Consider small code review checks or a grep for other places where _min_... vs _max_... mix-ups could exist.

Minimal patch (concept)
- Locate the max-middle block and change min_middle_index -> max_middle_index as shown above.

That single-line mistake is the direct cause; fixing it will stop the wrong period being used for comparison and will restore the correct max_middle_index behavior.
