What happened (where it failed)
- Failing input: "fr__POSIX"
- Failure occurs after the code confirms locale_str[2] == '_' and then reads ch3 = locale_str[3] and ch4 = locale_str[4].
- At lines 239–240 the code does this check:
  if ch3 < 'A' or ch3 > 'Z' or ch4 < 'A' or ch4 > 'Z':
      raise ValueError(f"Invalid locale format: {locale_str}")

- For "fr__POSIX", locale_str[3] is '_' (the second underscore). That is not between 'A' and 'Z', so the check raises ValueError. In short: the code expects a two-letter uppercase country at positions 3–4, but the input is the language__variant form (empty country) where position 3 is '_' and the variant begins later.

Why it happened (root cause)
- The implementation assumes that after the language and the first '_' there must always be two uppercase letters for a country (positions 3 and 4). It does not handle the valid Java-style locale form language__variant where the country is empty and there are two underscores in a row.
- The failing check is exactly at lines 239–240 (the ch3/ch4 A-Z validation). It treats '_' as an invalid character rather than as the marker for an empty country.

Consequence visible in tests
- Test test_locale_with_empty_country_variant and the other language__variant tests expect Locale("fr", "", "POSIX") etc., but the function raises ValueError because ch3 == '_' fails the A–Z check.
