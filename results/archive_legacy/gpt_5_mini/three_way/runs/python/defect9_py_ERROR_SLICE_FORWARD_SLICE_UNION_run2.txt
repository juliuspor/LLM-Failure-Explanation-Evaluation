What happened (root cause)
- The code calls len(...) on a value that can be None. That raises TypeError: "object of type 'NoneType' has no len()".
- The offending len calls appear at these lines:
  - line 52: search_length = len(search_list)
  - line 82: greater = len(replacement_list[i]) - len(search_list[i])
  - line 85: increase = min(increase, len(text) // 5)
  - line 104: or len(search_list[i]) == 0
- The most likely immediate trigger is line 82 or line 85, because both use len() on values that can be None before the code checks for None at line 104.

Why it fails where it does
- The code checks for None values for search_list[i] and replacement_list[i] only later (line 102–105). But earlier (line 82) it already computes len(search_list[i]) and len(replacement_list[i]). If either search_list[i] or replacement_list[i] is None, len(...) at line 82 raises the TypeError before the later guard runs.
- Also, if the function caller passed text = None, the len(text) at line 85 will raise the same error. Either case produces the same message, but the ordering shows the check for None in the inner loop comes too late.

Where it occurred (concrete variables & lines)
- If search_list contains None at index i:
  - line 82: greater = len(replacement_list[i]) - len(search_list[i]) => TypeError because search_list[i] is None.
- If replacement_list contains None at index i:
  - line 82: len(replacement_list[i]) raises TypeError.
- If text is None:
  - line 85: len(text) // 5 raises TypeError.
- The code later attempts to skip None entries at line 102–105, but that skip happens after the dangerous len(...) call at line 82.

Steps to fix (explicit, actionable)
1. Validate inputs at function start:
   - If text is None, raise a clear error:
     - e.g. if text is None: raise ValueError("text must be a string, not None")
   - If search_list or replacement_list is None (the whole list), raise ValueError.
2. Avoid calling len(...) on individual list items before checking them. Move the per-item None/emptiness checks before any len() use. Concretely:
   - Replace the loop that computes greater at line 81–85 with something like:
     for i in range(search_length):
         if search_list[i] is None or replacement_list[i] is None:
             continue  # or raise, depending on intended behavior
         if len(search_list[i]) == 0:
             continue
         greater = len(replacement_list[i]) - len(search_list[i])
         ...
3. If the intended behavior is to treat None as an empty string, convert them explicitly instead of calling len():
   - e.g. s = '' if search_list[i] is None else search_list[i]
         r = '' if replacement_list[i] is None else replacement_list[i]
         greater = len(r) - len(s)
4. Add unit tests that pass None for text, or insert None into search_list/replacement_list to verify the function fails with a clear message or handles it as intended.

Recommended quick change (minimal patch)
- At top of replace_each:
    if text is None:
        raise ValueError("replace_each: text must be a string, not None")
    if search_list is None or replacement_list is None:
        raise ValueError("replace_each: search_list and replacement_list must not be None")
- In the loop before line 82, add explicit guards:
    if search_list[i] is None or replacement_list[i] is None:
        continue
    if len(search_list[i]) == 0:
        continue

This ensures len(...) is only called on actual strings and prevents the NoneType TypeError.
