What happened (root cause)
- The code tries to infer the element/component type using these lines:
  - if array is not None:
        inferred_type = type(array)
  - elif element is not None:
        inferred_type = type(element)
  - else:
        inferred_type = object

  That is wrong: type(array) returns the Python list class (list), not the type of the elements inside the list. So when array is non-None you should be using the element type (e.g. str, int), not list. This mistaken inference breaks downstream logic that expects a component/element type.

- A concrete test failure occurs in test_add_null_to_null_with_expected_type:
  - array is None and element is None, so inferred_type becomes object (the else branch).
  - The code then calls ArrayUtils._copy_list_grow1(array, inferred_type) and after that checks:
        if expected_type is not None:
            if inferred_type == object and expected_type != object:
                raise TypeError(...)
  - Because inferred_type == object and expected_type == str, the code raises a TypeError. But the test expects ArrayUtils.add(None, None, expected_type=str) to succeed and return [None]. The code raises when it should instead accept expected_type as the element type (or at least not treat expected_type as an incompatible cast).

Where exactly in the code the bug is:
- The incorrect use of type(array) (first branch) — this is the wrong value for inferred_type.
- The raising TypeError block that treats inferred_type == object and expected_type != object as an unrecoverable cast error. In the situation array is None and element is None, expected_type should be used instead of rejecting.

Why that causes the observed failure:
- For the test case array=None, element=None, expected_type=str the code sets inferred_type = object and then raises TypeError because it treats object -> str as an invalid cast, so the call fails with an exception instead of returning [None].
- For other cases, using type(array) (a list class) instead of element-type can cause _copy_list_grow1 and any type-based checks to behave incorrectly (the code will think the component type is 'list' rather than str/int/NoneType), causing incorrect list creation or mismatches.

How to fix it (concrete steps and exact places to change)
1) Fix inference when array is provided (replace the type(array) line)
   Replace:
       if array is not None:
           inferred_type = type(array)
   With logic that uses the element/component type:
       if array is not None:
           # if array has at least one non-None element, use its type
           for elem in array:
               if elem is not None:
                   inferred_type = type(elem)
                   break
           else:
               # array is empty or all elements are None: fall back to expected_type or object
               inferred_type = expected_type if expected_type is not None else object

   Rationale: the component type should be the type of elements in the list, not the list itself. If array is empty or contains only None, we cannot infer an element type from it — use expected_type if caller provided it, otherwise object.

2) Handle the case array is None and element is None with expected_type provided
   Replace the TypeError block:
       if expected_type is not None:
           if inferred_type == object and expected_type != object:
               raise TypeError(...)
   With either:
       if expected_type is not None and inferred_type == object:
           # trust the caller's expected_type as the component type (no TypeError)
           inferred_type = expected_type

   Or, if you want stricter checking, only raise when there is a real incompatible existing component type (not when both array and element are None). But for this code and the tests, accepting expected_type when inference produced object is the correct behavior.

3) Keep assignment to the new list element as before:
       new_list[len(new_list) - 1] = element
   (No change needed here — the main issue is the inferred_type passed to _copy_list_grow1 and the unnecessary TypeError.)

Summary patch (conceptual)
- Replace use of type(array) with element-type inference from array contents (first non-None element), falling back to expected_type or object.
- Replace the TypeError-raising branch with assigning inferred_type = expected_type when inferred_type == object and expected_type is provided.

Why this addresses all requirements in the docstring
- If array is None and element is None but expected_type is provided, we should return a one-element list whose component type is expected_type. The suggested change sets inferred_type = expected_type in that situation so _copy_list_grow1 will create the right list, and no TypeError is thrown.
- If array is non-empty, the code now correctly uses an element's type as the component type.
- If array is empty or all-None, we now use expected_type when available, otherwise object — matching the docstring behavior.
