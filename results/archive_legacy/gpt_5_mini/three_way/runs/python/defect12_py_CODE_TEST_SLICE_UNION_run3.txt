What happened (root cause)
- In abbreviate(), when lower is greater than the length of text, the code sets upper = len(text) and then immediately does:
  if upper < lower:
      upper = lower
  That makes upper larger than len(text) (upper becomes lower). Later substring_java(text, 0, upper) is called with end > len(text). substring_java validates indices and raises IndexError ("String index out of range") at the check on line 32 in SLICE_UNION because end > len(s).

Where it occurs (exact lines / variables)
- The problematic logic is these lines in abbreviate():
    if upper == -1 or upper > len(text):
        upper = len(text)
    if upper < lower:
        upper = lower
  - For the failing test: text length = 10, lower = 15, upper = 20.
    - upper is set to 10 by the first if.
    - second if sees 10 < 15 and sets upper = 15.
    - substring_java(text, 0, upper) is then called with end = 15 > len(text) and substring_java raises at SLICE_UNION line 32:
      if start < 0 or end < 0 or start > end or start > len(s) or end > len(s):
          raise IndexError(...)

Why the test fails (concrete example)
- Test calls WordUtils.abbreviate("0123456789", 15, 20, None).
- After the current normalization code, upper becomes 15 while text length is 10.
- substring_java(text, 0, 15) triggers the slice-range check and raises IndexError.

How to fix it (concrete steps / suggested code changes)
Option A — Return the original text when lower is beyond length (matches the test expectation):
- After setting upper to len(text), add a check:
    if lower >= len(text):
        return text
  This makes the method return "0123456789" for the failing test.

Option B — Clamp lower to upper (less desirable for this API, but safe):
- Change the second normalization line to update lower instead of upper:
    if upper < lower:
        lower = upper
  Then substring_java(text, 0, upper) will have end <= len(text).
  But option A more directly matches the test comment ("Original bug case (Lang-45)") where lower beyond length should return the whole string.

Exact place to change
- Modify the block:
    if upper == -1 or upper > len(text):
        upper = len(text)
    if upper < lower:
        upper = lower
  to either
  (Option A)
    if upper == -1 or upper > len(text):
        upper = len(text)
    if lower >= len(text):
        return text
  or
  (Option B)
    if upper == -1 or upper > len(text):
        upper = len(text)
    if upper < lower:
        lower = upper

Why these fixes work
- They prevent calling substring_java(...) with end > len(text), avoiding the IndexError raised by the slice-range check (SLICE_UNION lines 32–34).
- Option A also matches the intended behavior in your unit test: if the lower bound is beyond the string length, return the original string.

Summary
- Bug: upper gets set to a value greater than len(text) (via upper = lower) and substring_java is called with end > len(text).
- Fix: avoid increasing upper beyond len(text) — either return text when lower >= len(text) or clamp lower to upper — so substring_java is never called with an out-of-range end index.
