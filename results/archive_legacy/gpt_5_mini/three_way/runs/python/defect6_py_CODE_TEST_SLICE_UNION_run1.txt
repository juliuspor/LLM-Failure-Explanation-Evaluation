What happened (root cause)
- The code treats any float that compares equal to its int form as an integer and prints int(x) (lines 176–178 and 190–192).
- In Python -0.0 == 0 is True, so add_number(-0.0) enters that integer branch. value = int(x) becomes 0 and mantissa = 0, and then the code does self.add(str(value)) (line 192), which writes "0".
- That loses the negative sign bit of -0.0, so the test expecting "-0.0" fails. The loss occurs at the integer branch decision and output (around lines 176–192).

Where it occurs (specific lines)
- Decision to treat x as integer: line 176: if x == int(x) and not math.isinf(x) and not math.isnan(x):
- Conversion to integer: line 177: value = int(x)
- Final integer output that drops sign: line 192: self.add(str(value))

Why that logic is wrong for negative zero
- Equality x == int(x) is true for -0.0, so the code doesn't distinguish between +0.0 and -0.0.
- Converting to int(x) yields 0 (no sign), and printing str(value) loses the negative-zero sign bit; there is no string "-0" or "-0.0" produced.

Concrete fixes (step-by-step)
1. Detect negative zero before the integer branch and handle it explicitly. For example, add at or before line 176:
   - if self.is_negative_zero(x):
       self.add("-0.0")
       return
   (The unit tests include/is expecting an is_negative_zero utility; use it.)

2. Alternatively, exclude negative zero from the integer branch condition:
   - Change line 176 to:
       if x == int(x) and not math.isinf(x) and not math.isnan(x) and not self.is_negative_zero(x):

3. If you don't have is_negative_zero yet, implement it as:
   - @staticmethod
     def is_negative_zero(v: float) -> bool:
         return v == 0.0 and math.copysign(1.0, v) < 0.0
   This checks the sign bit without losing precision.

4. Make sure to return (or otherwise avoid falling through) after handling negative zero so the later integer branch does not run.

Notes about other code parts
- The earlier spacing logic uses x < 0 to decide whether to add a space after a preceding '-', but in Python x < 0 is False for -0.0, so that code will not treat -0.0 as negative for spacing. That is separate from the negative-zero printing bug and the tests only require the sign be preserved in output.

Summary (what to change)
- Add an explicit negative-zero check (using is_negative_zero) before converting x to int or printing integer form. This prevents treating -0.0 as a plain integer 0 and preserves the sign by printing "-0.0".
