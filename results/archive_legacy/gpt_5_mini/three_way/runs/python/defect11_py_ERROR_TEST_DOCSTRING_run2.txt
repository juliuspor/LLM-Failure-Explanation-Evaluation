Why this occurs (root cause)
- Inside RandomStringUtils.random you compute a range length or otherwise call into the Python random functions (random.randrange / random.randint) using a value derived from end - start.
- When start == end (or start > end), the computed bound passed to random.randrange / randint is zero or negative.
- Python's random functions immediately raise a ValueError (from the random module) with the generic message "bound must be positive", before any code in RandomStringUtils can raise a clearer ValueError that mentions start and end.
- In other words, there is no early validation of the start/end arguments; the bad input triggers an internal random module error rather than your intended, descriptive error.
