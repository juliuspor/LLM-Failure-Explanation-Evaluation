What happened (where and why)
- In GrayPaintScale.get_paint at lines 71–75 you convert the input value to a gray channel g:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)  (line 71)
- Immediately after, the code checks:
  if g < 0 or g > 255:
      raise ValueError(...)  (lines 74–75)

- For values outside the configured range this produces g < 0 or g > 255 and raises a ValueError instead of clamping to black/white. Concretely:
  - For default bounds (lower=0.0, upper=1.0) and value = -0.5:
    g = int((-0.5 - 0.0) / (1.0 - 0.0) * 255.0) = int(-127.5) → -127 → triggers raise.
  - For value = 1.5: g = int(1.5 * 255) = 382 → triggers raise.
  - For custom bounds lower=10.0 upper=20.0 and value = 5.0:
    g = int((5-10)/10 * 255) = int(-127.5) → -127 → triggers raise.
- The tests expect clamping (negative values -> (0,0,0); >upper -> (255,255,255)), but the code raises an exception instead. So the root cause is the ValueError check (lines 74–75) instead of clamping behavior.

What to change (clear, specific steps)
1. Clamp the input value to the [self._lower_bound, self._upper_bound] range before converting, or clamp g to [0,255] after computing it.
2. Remove the raise ValueError for out-of-range g.

Minimal, safe fix examples (pick one)

A) Clamp the input value first (recommended):
- Replace lines 71–75 with:
  value = max(self._lower_bound, min(value, self._upper_bound))
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
  g = max(0, min(255, g))

B) Clamp g after compute:
- Replace lines 71–75 with:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
  if g < 0:
      g = 0
  elif g > 255:
      g = 255

Either change prevents the ValueError and produces the expected (0,0,0) and (255,255,255) for inputs outside bounds. If you choose (A) you also avoid issues when upper==lower (you should still guard that case separately to avoid division by zero).
