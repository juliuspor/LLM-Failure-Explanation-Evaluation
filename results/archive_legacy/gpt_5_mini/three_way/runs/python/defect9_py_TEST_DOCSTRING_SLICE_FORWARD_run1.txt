What happened
- The method pre-computes an "increase" amount (to estimate how much bigger the result will be) by calling len(...) on entries of replacement_list.
- If replacement_list contains None, len(None) raises TypeError (NoneType has no len()), so the code crashes before it gets to later checks that are intended to ignore None replacements.

Where it occurs (lines from your slice)
- The crash happens during the "increase" calculation around lines 83–85:
  - line 84: increase += 3 * greater  (where greater was computed using len(replacement_list[i]))
- Later in the method there is a guard that correctly ignores None values:
  - lines 102–105 check replacement_list[i] is None (and similar checks) before searching/choosing that replacement.
- But that guard comes after the earlier increase calculation, so the code already tried to call len(replacement_list[i]) when replacement_list[i] was None.

Why that causes the tests to fail
- Your tests pass None in replacement_list (e.g. ["c", None]). The increase-phase still calls len(None) and throws TypeError, so the function never reaches the logic that would ignore None replacements and continue normally. That matches the reported bug (Lang-39): a None in the replacement list should be treated as "ignore this replacement", not crash.

How to fix it (concrete steps)
1. In the loop that computes increase (the code around lines 77–86), skip entries where replacement_list[i] is None (and also where search_list[i] is None or empty), the same way you do later.
   - Option A (skip): add a condition so you do not call len(...) when replacement_list[i] is None.
   - Option B (tolerant length): compute replacement_len = 0 if replacement_list[i] is None else len(replacement_list[i]); use replacement_len in place of len(replacement_list[i]).
2. Keep the later guard (lines 102–105) as-is — it already prevents matching/choosing None entries during the main loop.
3. Run the provided tests to confirm the crash is gone.

Minimal patch example (conceptual)
- Before:
    greater = len(replacement_list[i]) - len(search_list[i])
- After:
    if replacement_list[i] is None or search_list[i] is None or len(search_list[i]) == 0:
        continue
    greater = len(replacement_list[i]) - len(search_list[i])

This ensures len(...) is never called on None and matches the later logic that ignores None entries.

Summary
The root cause is a len(...) call on replacement_list[i] before the code checks for replacement_list[i] being None. Move or duplicate the None-check into the increase-calculation loop (or treat None as length 0) so the function no longer crashes when replacements contain None.
