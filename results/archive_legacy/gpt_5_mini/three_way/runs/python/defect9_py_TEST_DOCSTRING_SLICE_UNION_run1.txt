What goes wrong
- Root cause: the code calls len(...) on items from search_list and replacement_list before it ever checks whether those items are None.
- Where it happens: in the first sizing loop at line 82:
  - Line 81/82: greater = len(replacement_list[i]) - len(search_list[i])
  - If either replacement_list[i] or search_list[i] is None, len(None) raises TypeError (or in Python: "object of type 'NoneType' has no len()").
- Why that is inconsistent: the code later performs None checks when deciding matches (lines 102–105), e.g. it treats None as "ignore this replacement":
  - Lines 102–105 test: no_more_matches_for_repl_index[i] or search_list[i] is None or len(search_list[i]) == 0 or replacement_list[i] is None
  - Those checks assume None entries are allowed and should be skipped, but they come after the earlier length computation that already crashed.

Concrete example from tests
- In test_null_safety_inside_arrays_single the call is:
  result = StringUtils.replace_each("aba", ["a"], [None])
- At line 82 the code tries len(replacement_list[0]) where replacement_list[0] is None → crash before any of the None-handling logic at lines 102–105 runs.

What to change (explicit steps)
1. Protect the initial sizing loop (lines 81–85) from None values:
   - Before calling len(...) skip indices where search_list[i] is None or replacement_list[i] is None.
   - Replace:
       greater = len(replacement_list[i]) - len(search_list[i])
     with something like:
       if search_list[i] is None or replacement_list[i] is None:
           continue
       greater = len(replacement_list[i]) - len(search_list[i])
2. Also ensure any other early code that calls len(...) or text.find(...) on search_list[i] or replacement_list[i] is guarded against None. For example, in the match-finding loop (line 108) you already check for search_list[i] is None at 102–105, but make sure those checks are evaluated before any use of search_list[i].
3. Optionally (safer): build a filtered list of valid (search, replacement) pairs at the start:
   - valid_pairs = [(s,r) for s,r in zip(search_list,replacement_list) if s is not None and r is not None and s != ""]
   - then operate only on valid_pairs everywhere (also update search_length), which centralizes the None/empty-string filtering.

Why this fixes it
- Skipping None entries before calling len(...) or text.find(...) prevents the TypeError and makes the code consistent: any None replacements are ignored both for sizing and for matching, matching the intended behavior described in the docstring and the Java tests (Lang-39).

Summary
- The failure is a simple ordering bug: code uses len(...) on possibly-None list entries at line 82 before the later None checks at lines 102–105. Fix by checking for None (or filtering pairs) before any use of len(...) or text.find(...).
