What happened
- The code raises ValueError("Minutes out of range: -15") inside DateTimeZone.for_offset_hours_minutes because of this check:
  if minutes_offset < 0 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")

- That line rejects any negative minutes. The test calls DateTimeZone.for_offset_hours_minutes(0, -15) expecting a valid offset of -00:15, so the function fails the test immediately.

Why it fails (root cause)
- The minutes validation is wrong. The minutes range should be -59..+59, not 0..59. By requiring minutes_offset >= 0, you incorrectly forbid legal negative minute offsets such as -15. This check is the immediate cause of the ValueError seen in the test.

Related logic notes you should be aware of
- The later code attempts to combine hours and minutes into total minutes using this logic:
    hours_in_minutes = hours_offset * 60
    if hours_in_minutes < 0:
        minutes_offset = hours_in_minutes - minutes_offset
    else:
        minutes_offset = hours_in_minutes + minutes_offset
  That logic expects minutes_offset to be able to be negative in some cases and is written to treat (-2, 15) as -2h - 15m (by subtracting minutes when hours are negative). So the initial validation must allow negative minutes to support that behavior.

What to change (concrete steps)
1. Replace the incorrect minutes validation:
   if minutes_offset < 0 or minutes_offset > 59:
       raise ValueError(f"Minutes out of range: {minutes_offset}")

   with a correct range check that allows negative minutes:
   if minutes_offset < -59 or minutes_offset > 59:
       raise ValueError(f"Minutes out of range: {minutes_offset}")

2. Add explicit rejection for the invalid mixed-sign case hours > 0 and minutes < 0 (the tests expect this to raise). Add this after checking hours and the new minutes range:
   if hours_offset > 0 and minutes_offset < 0:
       raise ValueError("Positive hours with negative minutes is not allowed")

3. Keep the existing combination logic that handles negative hours with positive minutes (so (-2, 15) becomes -2h -15m). With the corrected validation, the rest of the method will produce correct offsets.

Summary of changes (code fragment)
