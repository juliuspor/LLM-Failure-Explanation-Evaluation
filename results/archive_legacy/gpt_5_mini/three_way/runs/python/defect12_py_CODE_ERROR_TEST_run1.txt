What happened
- The code is a direct port of Java behavior but doesn't handle Python string indexing the same way Java does.
- In the failing test you call abbreviate("0123456789", 15, 20, None). Here len(text) == 10 and lower == 15.
- The code executes:
  - if upper == -1 or upper > len(text): upper = len(text)  -> upper becomes 10
  - if upper < lower: upper = lower  -> upper becomes 15 (now > len(text))
  - index = StringUtils.index_of(text, " ", lower)  -> calls index_of with start index 15
- In Java, String.indexOf(startIndex >= length) returns -1; in the Python string/index implementation used here, attempting to search or index with start >= len(text) triggers an IndexError: "String index out of range: 15". That's the exception you saw.

Why this is the root cause (clear link to code)
- The problematic lines are:
  - if upper < lower: upper = lower
  - index = StringUtils.index_of(text, " ", lower)
- By forcing upper = lower when lower > len(text), the code allows start indices larger than the string length to be passed into functions that don't tolerate them in Python.
- The immediate crash comes from calling index_of(text, lower) with lower == 15 while text length is 10.

How to fix it (concrete steps and exact changes)
Option A — simplest and matches the test expectation (return original when lower is beyond length):
1. After the empty-string check, add:
   if lower >= len(text):
       return text
This short-circuits the method for lower beyond the string and matches the expected result in the test.

Option B — clamp bounds and continue (if you prefer to preserve flow):
1. Replace or add before the index_of call:
   if lower > len(text):
       lower = len(text)
2. Ensure upper is also clamped to len(text) after you possibly set upper = lower:
   if upper > len(text):
       upper = len(text)
This prevents passing start indices > len(text) to index_of and avoids substrings with end > len(text).

Why these fixes work
- They prevent calling index_of or substring helper with start/end indices that exceed the Python string length, avoiding IndexError.
- Option A follows the original Java semantics expected by the test (if the starting position is beyond the string, return the original string).
- Option B preserves more of the original branching but ensures all index values are clamped to valid Python ranges.

Pick Option A if you want the simplest, test-passing fix; pick Option B if you want to preserve more of the original logic but must be careful to clamp both lower and upper before any string operations.
