What happened (root cause)
- In get_paint you correctly clamp the input into v:
  v = max(value, self._lower_bound)
  v = min(v, self._upper_bound)
- But directly afterward you compute g using the original value variable instead of the clamped v:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- So when value is outside the bounds (for example -0.5), the clamping has no effect on the numerator and g becomes negative. That negative g triggers your explicit range check and raises the ValueError shown in the test.

Where it occurs
- Inside GrayPaintScale.get_paint, the faulty line is the one that computes g (the line that uses value rather than v).

Why the test fails (concrete example)
- Default bounds appear to be 0.0..1.0. For value = -0.5:
  - v becomes 0.0 after clamping.
  - But g is computed from value: (−0.5 − 0.0)/(1.0 − 0.0)*255 = −127.5 -> int(...) -> −127
  - The subsequent if g < 0 or g > 255 raises ValueError.

Exact fixes (clear, actionable steps)
1. Use the clamped variable v when computing g:
   Replace
     g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
   with
     g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

2. Add a guard against division by zero if lower == upper:
   if self._upper_bound == self._lower_bound:
       g = 0  # or 255, or decide a sensible default; or raise a clearer error
   else:
       g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

3. (Optional) If you want exact behavior of the tests, ensure int() rounds down as expected (tests expect 0.5 -> 127). You can keep int() which truncates toward zero for positive numbers.

4. (Optional) Improve the error message to show g and the inputs so debugging is easier:
   raise ValueError(f"Color parameter outside of expected range: g={g}, lower={self._lower_bound}, upper={self._upper_bound}")

After applying step 1 (and step 2 if needed), the clamping will actually be used in the interpolation and the ValueError raised in the tests will no longer occur.
