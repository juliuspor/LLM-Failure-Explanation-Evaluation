What fails
- Running the tests raises a TypeError at the top of replace_each when the code does len(replacement_list[i]) while replacement_list[i] is None.
- That happens for the first test (replacement_list == [None]) and the second test (one replacement is None).

Where it fails (exact places)
- In the size-estimate loop starting at line 81:
  - Line 82: greater = len(replacement_list[i]) - len(search_list[i])
    - If replacement_list[i] is None this raises: TypeError: object of type 'NoneType' has no len().
- Later, at lines 102–105 the code already checks for None:
  - The check "or replacement_list[i] is None" appears there, but that check is after the earlier len(...) call, so it is too late.

Why it fails (root cause, plain)
- The function computes a quick "increase" estimate by calling len(replacement_list[i]) and len(search_list[i]) for every index i before it ever verifies whether either item is None or empty.
- When a replacement is None (which the tests treat as "do not replace this search string"), calling len(None) raises the TypeError. The later guard that skips None replacements cannot run because the exception has already occurred.

How to fix (concrete steps)
1. In the size-estimate loop (around line 81), skip indices where either search_list[i] or replacement_list[i] is None or where search_list[i] is empty before calling len(...). For example:
   - Replace the body at lines 81–85 with something like:
     if search_list[i] is None or replacement_list[i] is None or len(search_list[i]) == 0:
         continue
     greater = len(replacement_list[i]) - len(search_list[i])
     if greater > 0:
         increase += 3 * greater
2. Ensure the same defensive checks are present before any use of len(...) or text.find(...) so the code never calls len(None) or text.find(None).
3. Run the tests. The behaviour the unit tests expect is: a None replacement should be treated as "no replacement" (string unchanged where that search would be used).

Why this fix works
- It prevents len(None) from being called by skipping indices that are already known to be invalid for replacement (None or empty search). The later logic that finds actual matches already checks those conditions; putting the same checks before the estimate removes the exception while preserving the original skip semantics.
