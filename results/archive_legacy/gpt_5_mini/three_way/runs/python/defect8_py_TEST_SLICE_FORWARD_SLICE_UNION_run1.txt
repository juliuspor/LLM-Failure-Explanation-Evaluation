What happened (where it fails)
- The failing input "fr__POSIX" goes into LocaleUtils.to_locale with locale_str == "fr__POSIX".
- At line 237 the code sets ch3 = locale_str[3] and ch4 = locale_str[4].
  - For "fr__POSIX": locale_str[3] == '_' and locale_str[4] == 'P'.
- Lines 239–240 then check that ch3 and ch4 are uppercase A–Z and raise ValueError if not:
  - Since ch3 == '_' is not in 'A'..'Z', the code raises ValueError("Invalid locale format: fr__POSIX").
- That is exactly why the test expecting Locale("fr", "", "POSIX") fails: the method assumes the country is always two uppercase letters and rejects the double-underscore (empty country) case.

Why it happened (root cause)
- The code enforces a two-letter country at positions 3 and 4 before considering the possibility of an empty country (double underscore). It never checks for an underscore at position 3 to mean "empty country".
- Concretely:
  - The test input contains language + '_' + '_' + variant (language__variant).
  - The implementation unconditionally validates locale_str[3] and locale_str[4] as uppercase letters (lines 237–240) and only later deals with variants (lines 245–247). That ordering rejects '_' at position 3 as invalid instead of treating it as the empty country marker.

Exactly where to change it (step-by-step fix)
1. After confirming locale_str[2] == '_' (the separator after the language), check for the double-underscore case before validating country letters:
   - If locale_str[3] == '_', treat it as empty country and set variant = locale_str[4:].
   - Return Locale(locale_str[0:2], "", variant).
   - For "fr__POSIX", that yields Locale("fr", "", "POSIX") because variant is locale_str[4:].
2. Otherwise (locale_str[3] != '_'), continue validating locale_str[3] and locale_str[4] as uppercase A–Z and proceed with country or country+variant handling as before.
