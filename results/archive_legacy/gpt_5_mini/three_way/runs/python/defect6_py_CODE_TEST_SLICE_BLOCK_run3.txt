What happened (root cause)
- At line 176 you treat -0.0 as an integer-equivalent because x == int(x) is True for -0.0 (since -0.0 == 0).
- You then do value = int(x) at line 177. int(-0.0) produces 0 (a positive integer), so the sign information carried by the floating-point -0.0 is lost.
- Later (line 192) you call self.add(str(value)), which adds "0" to the output. That is why test_negative_zero fails: the code prints "0" instead of "-0.0".

Where it occurs
- The sign is lost between lines 176–177: the equality check and conversion to int throw away the negative zero sign.
- The code path that prints integers (lines 189–192) never sees a negative-zero special-case, so it prints the unsigned integer string.

Why int(x) and equality are the problem
- Python floats obey IEEE-754: -0.0 compares equal to 0.0, and int(-0.0) == 0. But IEEE negative zero is a distinct bit pattern that you must check for explicitly (it is not preserved by int()).
- The checks math.isinf(x) and math.isnan(x) do not help here because -0.0 is a finite, non-NaN value.

Concrete, actionable fixes (pick one)

1) Quick explicit negative-zero early-return (recommended)
- Add a check before the integer branch (before line 176). For example:
    if x == 0.0 and math.copysign(1.0, x) < 0:
        self.add("-0.0")
        return
- This uses math.copysign to detect the negative sign on zero and immediately prints "-0.0" so later integer conversion never runs.

2) Use the existing utility is_negative_zero (if implemented)
- If CodeConsumer.is_negative_zero(x) exists (your tests assume it does), use it:

        self.add("-0.0")
        return

3) Alternative detection using reciprocal
- Detect negative zero with 1.0/x == float('-inf'):
    if x == 0.0 and 1.0/x == float('-inf'):
        self.add("-0.0")
        return

Notes on placement and behavior
- Place the check immediately before the block starting at line 176 so you don't enter the int-conversion path.
- Ensure you return after adding "-0.0" so the function doesn't continue and overwrite or append further output.
- Do not rely on str(int(x)) or str(x) for negative zero — both will lose the negative sign if you first convert to int or if you let default float->str convert to "0.0" without preserving sign.

Example patch (contextual)
