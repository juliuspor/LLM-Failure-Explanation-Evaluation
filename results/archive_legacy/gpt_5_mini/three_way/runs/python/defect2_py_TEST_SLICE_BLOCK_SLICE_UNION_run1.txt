What happened (where and why)
- At line 71 you compute g:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- At lines 74–75 you immediately raise a ValueError when g is outside 0..255:
  if g < 0 or g > 255:
      raise ValueError(...)
- For inputs outside the configured bounds (e.g. value = -0.5 in test_lower_bound_clamping or 1.5 in test_upper_bound_clamping) the computed g becomes negative or >255, so the code raises instead of clamping. That is exactly why the tests fail: they expect clamping but the implementation throws an exception (lines 71 and 74–75 are the root cause).

Other potential edge case to handle
- If upper_bound == lower_bound the division (line 71) will cause a division-by-zero. Even if you fix the clamping logic, you should guard against that.

How to fix it (concrete steps and code)
1. Normalize value first and clamp it to [0.0, 1.0] before mapping to 0..255.
2. Compute g from the clamped normalized value.
3. Remove the exception for out-of-range g (you will never produce out-of-range g after clamping).
4. Guard the division against upper == lower.

Replacement (drop in instead of lines 71–75):

  # compute normalized position between lower and upper, guarding division-by-zero
  if self._upper_bound == self._lower_bound:
      normalized = 0.0  # or 1.0 or 0.5 depending on desired behavior when bounds equal
  else:
      normalized = (value - self._lower_bound) / (self._upper_bound - self._lower_bound)
  # clamp normalized to [0.0, 1.0]
  if normalized < 0.0:
      normalized = 0.0
  elif normalized > 1.0:
      normalized = 1.0
  # map to 0..255
  g = int(normalized * 255.0)

Then return (g, g, g) instead of raising an exception.

Why this fixes the tests
- For value = -0.5 normalized becomes negative but is clamped to 0.0, so g = 0 -> (0,0,0).
- For value = 1.5 normalized becomes >1.0 but is clamped to 1.0, so g = 255 -> (255,255,255).
- For value = 0.5 normalized = 0.5 -> g = int(127.5) = 127 -> (127,127,127).
- You also avoid division-by-zero when lower == upper by handling that case explicitly.

That directly addresses the failing tests and the exact lines responsible (71 and 74–75).
