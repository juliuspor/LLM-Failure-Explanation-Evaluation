What happened (root cause)
- At lines 176–177 the code tests if x == int(x) and then does value = int(x). For x == -0.0 this is True: int(-0.0) == 0 and -0.0 == 0.0, so the code enters the integer branch.
- That integer branch (lines 177–192) converts the float to the integer value 0 and eventually calls self.add(str(value)) (line 192). str(0) is "0", so the negative sign is lost.
- If the code had taken the else branch (line 193–194) it would call self.add(str(x)), and str(-0.0) yields "-0.0", preserving the sign. But because the integer branch runs first, negative zero is normalized to plain zero and the sign is thrown away.

Where it occurs
- The problematic code is in the block starting at line 176:
  - The check x == int(x) (line 176) is what causes -0.0 to be treated as integer zero.
  - The sign is lost at line 177 (value = int(x)) and finally at line 192 (self.add(str(value))).

Why the tests fail
- The test test_negative_zero expects "-0.0" but the implementation outputs "0" because negative zero was detected as an integer zero and printed via str(value). So the assertion fails.

Concrete fixes (step-by-step)
1. Detect negative zero before you coerce to int. Add an early check using a reliable negative-zero test:
   - Use math.copysign or a utility like CodeConsumer.is_negative_zero(x). A correct test is:
     - x == 0.0 and math.copysign(1.0, x) < 0.0
     - or, if you already have CodeConsumer.is_negative_zero, call that.
2. If x is negative zero, emit "-0.0" directly and return, so the integer-branch logic is skipped.
3. Otherwise continue the existing integer / non-integer handling.

Minimal patch example (insert before line 176):
- Pseudocode to add just above the if:
  if CodeConsumer.is_negative_zero(x):
      self.add("-0.0")
      return

Or, if you don't have that helper in scope:
  if x == 0.0 and math.copysign(1.0, x) < 0.0:
      self.add("-0.0")
      return

Why this fixes it
- It prevents -0.0 from being classified as an integer zero and losing its sign. The function will now output "-0.0" exactly as the test expects.

Notes
- Do not simply change the final else to always use str(x); that will break the desired integer formatting (e.g., printing "42" not "42.0") and the scientific-notation logic.
- The check must be done before value = int(x) (i.e., before line 177) so the sign is preserved.
