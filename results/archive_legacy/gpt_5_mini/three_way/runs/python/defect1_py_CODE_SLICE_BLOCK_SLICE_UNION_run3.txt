What happened (where it fails)
- The exception is raised at line 396:
  if minutes_offset < 0 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")
  That means a caller passed a negative minutes_offset (for example -30) and the method throws "Minutes out of range".

Why that is the root cause (why it fails)
- The method validates minutes_offset as only 0..59, but the rest of the method treats the sign of the overall offset using hours_offset and mutates minutes_offset to become the total offset in minutes:
  hours_in_minutes = hours_offset * 60
  if hours_in_minutes < 0:
      minutes_offset = hours_in_minutes - minutes_offset
  else:
      minutes_offset = hours_in_minutes + minutes_offset
- This shows the method expects minutes_offset to be provided as an unsigned component (0..59) and to apply the negativity based on hours_offset. But callers commonly pass a signed minutes_offset (e.g. hours_offset = -1, minutes_offset = -30) to mean "-1:30". Those calls get rejected at the minutes range check even though the later code would have produced the right total offset.

Why this is confusing and error-prone
- The same identifier minutes_offset is used both for the "minutes component" parameter and later for the computed total minutes. That makes the code easy to misuse and hard to reason about.
- The validation and composition logic disagree about whether minutes_offset carries sign information.

Concrete, actionable fixes (step-by-step)
1. Decide one canonical API contract:
   - Option A: minutes_offset is always non-negative (0..59) and the sign comes only from hours_offset.
   - Option B: minutes_offset may be negative (−59..59) and the sign of the total offset is determined by combining hours_offset and minutes_offset properly.
2. Implement the chosen contract and make the code explicit (do not reuse minutes_offset for different meanings). For example, using Option A (current intent appears to be this):

   - Keep the existing validation at line 396 (minutes_offset must be 0..59).
   - Replace the in-place mutation with a new variable total_minutes for clarity:
       hours_in_minutes = hours_offset * 60
       if hours_in_minutes < 0:
           total_minutes = hours_in_minutes - minutes_offset
       else:
           total_minutes = hours_in_minutes + minutes_offset
       offset = FieldUtils.safe_multiply(total_minutes, DateTimeConstants.MILLIS_PER_MINUTE)

   - Return cls.for_offset_millis(offset).

3. Or, if you want to accept signed minutes (Option B), change the validation and computation:
   - Validate minutes_offset is between -59 and 59 (inclusive).
   - Compute total_minutes = hours_in_minutes + minutes_offset (no extra subtraction), but you must ensure callers consistently pass minutes with correct sign; or normalize the sign when hours_offset is nonzero.
   - Example: if you want (-1, 30) and (-1, -30) both to be meaningful, define rules and normalize accordingly.

4. Add unit tests for edge cases:
   - (hours_offset, minutes_offset) = (0, 0) -> UTC
   - (1, 30) -> +1:30
   - (-1, 30) -> -1:30 (if Option A)
   - (-1, -30) -> either error or -1:30 depending on chosen contract
   - Bounds like (-23, 59), (23, 59) and invalids like (24,0), (0,60), (0,-1)

Summary
- The failure comes from a mismatch between the input validation (minutes_offset must be 0..59) and the way callers actually pass minute sign (some pass negative minutes). Fix by picking one clear contract, stop reusing minutes_offset for two meanings (use a separate total_minutes variable), and adjust validation and tests accordingly.
