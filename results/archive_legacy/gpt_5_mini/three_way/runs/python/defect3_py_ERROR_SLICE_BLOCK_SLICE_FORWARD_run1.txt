What happened (where it failed)
- The IndexError was raised at lines 35–36:
  - if index < 0 or index >= len(code_units):
  -     raise IndexError(f"String index out of range: {index}")
- The value of index was 2 and len(code_units) was 2, so index >= len(code_units) is true and the code raises "String index out of range: 2".

Why it happened (root cause)
- Valid indices into code_units when len(code_units) == 2 are 0 and 1. The caller passed index == 2 (one past the last element).
- That means the caller is using an off-by-one index (for example iterating to len(code_units) instead of len(code_units) - 1), or is otherwise computing an index value that can equal the length.
- The check at lines 39–41 (index + 1 < len(code_units)) protects the surrogate-pair read, but it does not prevent the initial read at line 38 (high = code_units[index]) — the guard at 35–36 already prevents that, and it is this guard that raises the IndexError because the caller supplied index == 2.

Exactly where to look in your code
- Find the caller of this function (the place that passes index). It is passing index == len(code_units).
- Look for a loop or range like for index in range(len(code_units)) where you then call this function using that index but you intended to call it only for existing code unit positions (0..len-1). A common mistake: using range(len(code_units) + 1) or otherwise computing index values up to len(code_units).

Why the surrogate logic at lines 39–41 is not the problem
- The surrogate check correctly ensures index+1 < len(code_units) before reading code_units[index + 1]. The actual exception came earlier when index itself was out of range; fixing the caller index will make the surrogate-path code operate as intended.
