What happened
- The code raises TypeError: object of type 'NoneType' has no len() because it calls len(...) on a None element inside replacement_list (or search_list).
- This occurs in the "increase" sizing loop:

      # BUG: this loop is not null-safe for elements inside the arrays
      for i in range(search_length):
          greater = len(replacement_list[i]) - len(search_list[i])
          if greater > 0:
              increase += 3 * greater  # assume 3 matches

- If replacement_list[i] (or search_list[i]) is None, len(...) raises the exception. The test that fails uses replacement_list = [None], so len(None) triggers the TypeError.

Why it happens (root cause)
- Earlier in the function the code already guards against None/empty search strings and None replacements when searching for matches (the for loops that set no_more_matches_for_repl_index and when finding next matches use checks like search_list[i] is None or len(search_list[i]) == 0 or replacement_list[i] is None). But the increase-sizing loop runs before those later loops and does not use the same null checks. The comment even notes the bug: "this loop is not null-safe for elements inside the arrays".
- Concretely: replacement_list[i] may be None, so len(replacement_list[i]) raises TypeError.

Where it occurs (variables/method names & location)
- Function: replace_each(...)
- Problematic loop: the one labeled with "BUG: this loop is not null-safe for elements inside the arrays", which computes variable greater by calling len(replacement_list[i]) and len(search_list[i]).
- Example failing test: test_null_safety_inside_arrays_single calls StringUtils.replace_each("aba", ["a"], [None]) — replacement_list[0] is None, so len(replacement_list[0]) raises.

How to fix it (concrete steps)
Option A — skip None/empty entries (recommended, matches later behavior)
1. Replace the increase-sizing loop with the same null/empty checks used elsewhere. Example replacement:

    increase = 0
    for i in range(search_length):
        if search_list[i] is None or len(search_list[i]) == 0 or replacement_list[i] is None:
            continue
        greater = len(replacement_list[i]) - len(search_list[i])
        if greater > 0:
            increase += 3 * greater

    increase = min(increase, len(text) // 5)

2. Run tests; the TypeError should be gone and tests for null-safety should pass.

Option B — treat None as length 0 (less consistent with the rest of the function)
- Use lengths like (len(replacement_list[i]) if replacement_list[i] is not None else 0) and the same for search_list. But this may be less correct semantically because entries with replacement None are intended to be skipped completely.

Why Option A is preferred
- The rest of the function already treats entries with search_list[i] is None, empty search strings, or replacement_list[i] is None as "skip" cases. Making the increase-sizing loop follow the same rule keeps behavior consistent and prevents trying to size based on entries that won't be used.

Summary
- The immediate root cause is the increase loop calling len() on list elements without null checks. Fix by adding the same null/empty checks used elsewhere in the function (skip entries where search_list[i] is None/empty or replacement_list[i] is None). After that change, the TypeError will be eliminated and the tests will pass.
