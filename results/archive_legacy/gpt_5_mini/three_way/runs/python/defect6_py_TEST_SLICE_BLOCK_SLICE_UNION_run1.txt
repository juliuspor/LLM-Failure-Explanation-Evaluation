What happened (root cause)
- At line 176 the code tests if x is an integer: if x == int(x) and not math.isinf(x) and not math.isnan(x).
- For x == -0.0 that comparison is true (because -0.0 == 0.0).
- At line 177 value = int(x) converts -0.0 to the integer 0 and discards the sign.
- Because the integer branch runs, the code eventually calls self.add(str(value)) at line 192, which produces "0" (no sign, no decimal).
- The branch that would have produced "-0.0" (the else at line 193/194 that did self.add(str(x))) is never reached for -0.0, so the negative sign is lost.

Where it occurs (specific lines and variables)
- Decision happens at line 176 using variable x.
- Sign is lost at line 177 (value = int(x)) and line 192 (self.add(str(value))).
- The code that would preserve the original string form (self.add(str(x))) is at line 194 but is in the else branch and not executed for -0.0.

Why this breaks the test
- The test expects "-0.0". Because -0.0 compares equal to 0 and is treated as an integer here, the code prints "0" instead. That is exactly the observed failure: expected "-0.0", got "0".

The code treats -0.0 as an ordinary integer because -0.0 == int(-0.0) is true, and int(-0.0) strips the sign. Add a negative-zero check (before the integer branch) or explicitly preserve sign when value == 0 to fix the failing test.
