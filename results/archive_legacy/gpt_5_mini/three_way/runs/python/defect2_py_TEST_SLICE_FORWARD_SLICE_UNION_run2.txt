What failed
- The test calls GrayPaintScale.get_paint with values outside [lower_bound, upper_bound] and expects the method to clamp to black/white.
- Instead, get_paint computes the gray channel g at line 71 and then raises a ValueError if g < 0 or g > 255 (lines 74–75). That causes the tests to fail because they expect clamping, not an exception.

Where it happens (exact lines)
- Constructor stores bounds: lines 30–44 set self._lower_bound and self._upper_bound.
- Gray value computed at line 71:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- Exception raised at lines 74–75 when g is out of [0,255]:
  if g < 0 or g > 255:
      raise ValueError(f"Color parameter outside of expected range: Red Green Blue")

Why it fails (root cause)
- The code treats out-of-range inputs by throwing an exception instead of clamping them to the allowed range. Tests call get_paint(-0.5) and get_paint(1.5); these produce g < 0 or g > 255 and therefore the ValueError is raised.
- The intention in tests (and typical behavior for a paint scale) is that values below lower_bound map to black (0,0,0) and values above upper_bound map to white (255,255,255). The current code enforces a hard error instead.

How to fix it (concrete steps and exact changes)
Option A — Clamp the computed channel (simple change)
- Replace the check-and-raise at lines 74–75 with clamping logic:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
  if g < 0:
      g = 0
  elif g > 255:
      g = 255
- Then return the color tuple (g, g, g).

Option B — Clamp the input value first (preferred, clearer intent)
- Clamp value to bounds before computing g:
  if value < self._lower_bound:
      value = self._lower_bound
  elif value > self._upper_bound:
      value = self._upper_bound
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
  return (g, g, g)

Option C — Use min/max in a single expression
  t = (value - self._lower_bound) / (self._upper_bound - self._lower_bound)
  t = max(0.0, min(1.0, t))
  g = int(t * 255.0)
  return (g, g, g)

Additional defensive change (important if someone sets equal bounds)
- If upper_bound == lower_bound the division will raise ZeroDivisionError. Add a guard:
  if self._upper_bound == self._lower_bound:
      g = 0  # or 255, or raise a clear ValueError about invalid bounds
  else:
      ...compute as above...

Summary
- The failure is caused by raising ValueError when g is out of [0,255] (lines 74–75) instead of clamping. Fix by clamping either the input value or the computed gray channel, and optionally guard against upper == lower to avoid division by zero.
