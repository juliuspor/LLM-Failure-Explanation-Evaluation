What happens
- If val ends with "l" or "L", the code runs the block starting at
  if last_char in ("l", "L"):
- It sets numeric = val[:-1]. For val = "L" or "l" this makes numeric == "".
- The code then calls char_at(numeric, 0) without checking numeric’s length.
- char_at checks bounds (lines shown in the slice block) and raises IndexError("String index out of range: 0") when index 0 is out of range:
  - see the slice block: if index < 0 or index >= len(s): raise IndexError(...)
  - that is exactly triggered when numeric == "".

So the root cause: an out-of-bounds access to numeric at char_at(numeric, 0) when numeric is empty (e.g. input "L" or "l"), causing an IndexError instead of the intended ValueError for an invalid number.

Where it occurs (precise names/places)
- Function: create_number
- Branch: the last_char in ("l", "L") branch
- Faulty call: char_at(numeric, 0)
- Failure raised by char_at implementation (slice block lines 30–32), which enforces bounds and raises IndexError.

Why it happens (concise)
- The code assumes numeric has at least one character before calling char_at(numeric, 0), but val[:-1] can be empty when val was a single-character qualifier like "L". No guard exists for len(numeric) == 0.

How to reproduce quickly
- Call create_number("L") or create_number("l") — you get IndexError("String index out of range: 0").

Exact fixes (concrete, minimal changes)
Option A — prefer using Python string methods (cleanest):
Replace the offending condition

    if (char_at(numeric, 0) == "-" and numeric[1:].isdigit()) or numeric.isdigit():

with

    if (numeric.startswith("-") and numeric[1:].isdigit()) or numeric.isdigit():

This avoids char_at entirely and safely handles empty numeric.

Option B — add explicit length guard before calling char_at:
Immediately before the char_at usage, add

    if len(numeric) == 0:
        raise ValueError(f"{val} is not a valid number.")

or change the if to

    if len(numeric) > 0 and ((numeric[0] == "-" and numeric[1:].isdigit()) or numeric.isdigit()):

Either option prevents the IndexError by ensuring char_at is never called on an empty string.

Why these fixes work
- Option A uses safe, built-in methods that return False for empty strings and never raise.
- Option B prevents an out-of-range access by checking len(numeric) first and produces the intended ValueError for invalid input.

Pick Option A for simplicity and readability.
