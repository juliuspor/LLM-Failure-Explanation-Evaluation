What happened (root cause)
- The error "object of type 'NoneType' has no len()" comes from this loop (lines 81–85):
  ```
  for i in range(search_length):
      greater = len(replacement_list[i]) - len(search_list[i])
      if greater > 0:
          increase += 3 * greater
  ```
- That loop calls len(replacement_list[i]) and len(search_list[i]) without checking whether either element is None.
- Earlier in the function you explicitly treat None elements as invalid (for example, the match-finding loops on lines ~59–78 and ~102–113 both skip items when search_list[i] is None or replacement_list[i] is None). But the "get a good guess" loop does not perform those same checks, so when either search_list[i] or replacement_list[i] is None you get len(None) and the TypeError.
- In short: the code assumes array elements are non-null in that one place but they can be None. That mismatch is exactly what produced the exception.

Where it occurs (precise references)
- Problem lines: 81–85 (the "increase" estimation loop).
- Variables involved: search_list[i], replacement_list[i], and the len() calls.
- Related logic that is correct elsewhere: the match-finding loops (lines ~59–78 and ~102–113) already guard against None by checking
  ```
  if no_more_matches_for_repl_index[i] or search_list[i] is None or len(search_list[i]) == 0 or replacement_list[i] is None:
      continue
  ```

How to fix it (clear steps)
1. Make the length-estimation loop null-safe by skipping entries where either element is None or the search string is empty. Example change:
   ```
   for i in range(search_length):
       if search_list[i] is None or replacement_list[i] is None or len(search_list[i]) == 0:
           continue
       greater = len(replacement_list[i]) - len(search_list[i])
       if greater > 0:
           increase += 3 * greater
   ```
2. Alternatively, compute lengths defensively:
   ```
   s_len = len(search_list[i]) if search_list[i] is not None else 0
   r_len = len(replacement_list[i]) if replacement_list[i] is not None else 0
   greater = r_len - s_len
   ```
   but you should also skip empty search strings (s_len == 0) because the rest of the code ignores them.
3. After making the loop null-safe, the TypeError will go away. Keep the same skipping rules you used later (None or empty search strings) to keep behavior consistent.

Why this fix is correct
- It matches the same guards you already use elsewhere, so you won't try to compute lengths for entries the algorithm intentionally ignores.
- It prevents len(None) from being called, which is the direct cause of the TypeError.

Optional: small note on "increase"
- The "increase" value is only a heuristic for pre-sizing an output buffer and is unrelated to correctness; skipping None/empty entries preserves the heuristic while avoiding the exception.
