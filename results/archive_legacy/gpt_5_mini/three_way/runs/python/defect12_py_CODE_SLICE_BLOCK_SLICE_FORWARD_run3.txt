What happened (where it fails)
- The IndexError is raised inside substring_java (the slice function). The guard on lines 32–33 throws because end > len(s).
- That call happens when the code reaches a branch that does substring_java(text, 0, upper). Because upper can end up greater than len(text), substring_java is called with end > len(text) and the slice function raises.

Why it happens (root cause, with variable names and line numbers)
1. You first clamp upper to text length on these lines:
   if upper == -1 or upper > len(text):
       upper = len(text)
   (lines just after the empty-text check)

2. Immediately after you do:
   if upper < lower:
       upper = lower

   If lower > len(text), the first step set upper = len(text) but the second step then sets upper = lower, so upper becomes larger than len(text).

3. Later you call substring_java(text, 0, upper) (three places: the index == -1 branch, the index > upper branch, and the final return). When upper > len(text), substring_java passes end > len(s) into the slice function. The slice function rejects that and raises IndexError on lines 32–33:
   if start < 0 or end < 0 or start > end or start > len(s) or end > len(s):
       raise IndexError(f"String index out of range: {end}")

So the bug is the order of clamping: you clamp upper to len(text) and then overwrite it with lower without re-checking that the new upper is still within the string bounds.

How to fix it (concrete steps)
Option A — clamp upper after ensuring it's at least lower:
- Replace the two-step logic with one that enforces both bounds in the right order. For example:
  upper = len(text) if upper == -1 else upper
  if upper < lower:
      upper = lower
  if upper > len(text):
      upper = len(text)

Option B — use min/max in one expression:
- Compute upper as:
  upper = len(text) if upper == -1 else max(lower, min(upper, len(text)))
  (or equivalently: upper = min(len(text), max(lower, upper)) and handle upper == -1 by mapping it to len(text) first)

Either fix guarantees upper <= len(text) before any substring_java(text, 0, upper) call.

Why these fixes work
- They prevent end being larger than len(text) when calling substring_java, so the slice guard (lines 32–33) will not be triggered and no IndexError will be raised.
