What happened (root cause)
- The code assumes that when a locale string has more than a 2-letter language it must have a 2-letter country at positions 3–4. It therefore reads locale_str[3] and locale_str[4] and immediately checks they are uppercase letters (lines 237–240).
- For an input like "fr__POSIX" the character at index 3 is '_' (the second underscore of the language__variant form), not an uppercase letter. The check at line 239 (if ch3 < 'A' or ch3 > 'Z' ...) therefore fails and a ValueError is raised with "Invalid locale format: fr__POSIX".
- In short: the code does not handle the language__variant format (empty country between two underscores). It treats the '_' at index 3 as an invalid country character instead of recognizing that the country is empty and the remainder is the variant.

Where it occurs (exact lines)
- length is computed at line 222.
- The problematic indexing and validation are at lines 237–240:

  - ch4 = locale_str[4]
  - if ch3 < 'A' or ch3 > 'Z' or ch4 < 'A' or ch4 > 'Z': raise ValueError(...)
- The code then extracts the country with locale_str[3:5] and later expects a '_' at position 5 for variant handling (lines 243–247). Those assumptions are invalid for "language__variant".

Why that condition is incorrect
- It assumes country is always present and always two uppercase letters. It never considers the possibility of an empty country (i.e., two consecutive underscores: "_" then "_") indicating language + empty country + variant.
- It also indexes positions 3 and 4 without first checking whether the string actually uses the double-underscore empty-country pattern, so it treats '_' as an invalid country character rather than a separator.
