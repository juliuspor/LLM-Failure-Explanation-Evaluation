What happened
- The failing test calls StringUtils.replace_each("aba", ["a"], [None]).
- The implementation of replace_each in hit09_stringutils.py assumes every element of search_list and replacement_list is a str.
- At some point the method calls len(...) on an element from one of those lists (for example len(search_list[i]) or len(replacement_list[i]) or len(search) after assigning search = search_list[i]).
- If that element is None the call becomes len(None) and Python raises TypeError: object of type 'NoneType' has no len() — exactly the error you saw.

Why it happens (root cause)
- The code does not check for None values inside the arrays before using them. The method treats None like a string and immediately uses string operations (len, indexing, slicing, substring matching).
- The original Java behavior (Lang-39) expects null replacement entries to be ignored, not treated as empty strings or crashed on. The Python code fails to implement that null-safety.

- In hit09_stringutils.py, inside the replace_each function/method. Look for the loop that iterates over search_list / replacement_list and lines that compute lengths or do substring matching, e.g.:
  - search = search_list[i]
  - replace = replacement_list[i]
  - start = len(search)         <-- this is the exact point that will fail if search is None
  or any direct call like len(replacement_list[i]).
  (If you open hit09_stringutils.py, grep for len(search) or len(search_list) inside replace_each to find the exact line.)

Concrete fix (step-by-step)
1. Guard against None before using len() or other string operations. Skip any pair where either search or replacement is None (to match the expected behavior of ignoring those replacements).
2. Also guard against empty search strings (""), since replacing with an empty search usually needs special handling to avoid infinite loops.
