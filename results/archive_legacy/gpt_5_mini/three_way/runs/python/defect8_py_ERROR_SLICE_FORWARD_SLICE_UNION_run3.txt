What happened (where it failed)
- The error was raised in to_locale (see slices around lines 231–247).
- The code computes ch3 = locale_str[3] and ch4 = locale_str[4] and then checks they are uppercase letters (lines 237–240).
- For the input "fr__POSIX", ch3 is '_' (the second underscore), not an uppercase letter, so the check fails and ValueError("Invalid locale format: fr__POSIX") is raised.

Why it failed (root cause)
- The parser only recognizes locales with either:
  - a 2-letter language only ("fr"), or
  - language + underscore + 2-letter country ("fr_FR"), or
  - language + underscore + country + underscore + variant ("fr_FR_POSIX").
- It does NOT handle the valid form where the country is omitted but the variant is present, which is written with two underscores between language and variant ("fr__POSIX").
- In "fr__POSIX" the second underscore occupies the position where the code expects the first letter of a 2-letter country (ch3), so the uppercase-letter check at lines 237–240 fails and the code raises the ValueError.

Exactly where to change (concrete steps)
1. In to_locale, before you read ch3/ch4 or validate them, add a special-case branch for the "language + empty country + variant" form (double underscore).
   - For "fr__POSIX":
     - language = locale_str[0:2]
     - variant = locale_str[4:] (variant starts after the two underscores)
2. Return a Locale with no country (use None or empty string per your Locale API) and the variant.
