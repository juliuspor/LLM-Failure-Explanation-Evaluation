What happened (root cause)
- At line 336 the code does: inferred_type = type(array).
  - type(array) returns the list class (e.g. <class 'list'>), not the element type inside the list.
  - Later code expects inferred_type to represent the element type, so all subsequent type checks are wrong.
- Consequence: when array is not None the routine will treat inferred_type as list, and the check at line 346
  if inferred_type == object and expected_type != object:
  can misfire or never behave as intended because inferred_type will rarely equal the element type or object.
- Additional case: if array is an empty list the code should inspect element or expected_type to infer an element type, but using type(array) gives no element information and loses the ability to validate types properly.
- Also, when both array and element are None the code sets inferred_type = object at line 340. That may be fine, but combined with the mistaken type(array) use it triggers that TypeError at line 346 in unexpected situations.
