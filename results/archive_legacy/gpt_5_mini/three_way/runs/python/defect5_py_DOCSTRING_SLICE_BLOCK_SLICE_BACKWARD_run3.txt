What happened (where): At line 346 your function raises a TypeError. That raise is triggered because inferred_type was set to object earlier (line ~340) and expected_type is not object. The code that decides inferred_type is wrong: at line 336 you do

    inferred_type = type(array)

and at line 337 you fall back to

    inferred_type = type(element)

and finally at line 340 you set

    inferred_type = object

Those assignments are the root cause: you are using type(array) (the list object’s type) instead of the element type, and you treat a lack of information by setting inferred_type = object and then immediately using that to decide whether to raise.

Why that causes the failure (why this logic is incorrect):
- type(array) returns the list’s class (e.g. <class 'list'>), not the class of elements contained in the list. That makes inferred_type meaningless for element-type compatibility checks.
- If array is None and element is None you set inferred_type = object (line ~340). Later you check

    if inferred_type == object and expected_type != object:
        raise TypeError(...)

So when both array and element are None, the function assumes “unknown element type” (object) and then raises because expected_type was provided. In that situation there is no element to check, so raising here is incorrect — you should accept the expected_type as the intended element type or delay the check until an element is added.
- Conversely, if array is not None but contains elements, type(array) does not tell you the element type; you then may perform the wrong compatibility checks or fail to detect incompatible element types.
