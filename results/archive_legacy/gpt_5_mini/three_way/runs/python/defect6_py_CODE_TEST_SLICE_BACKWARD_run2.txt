What happened (where it fails)
- At the start of add_number you do:
  prev = self.get_last_char()
  if x < 0 and prev == '-':
      self.add(" ")
  This check is supposed to detect a negative number after a minus and insert a space. For x == -0.0, the expression x < 0 is False (in IEEE floats -0.0 compares equal to +0.0 and is not less than 0). So the code treats -0.0 as non-negative here.

- Then at the branch starting on the line shown in the slice (line 176):
  if x == int(x) and not math.isinf(x) and not math.isnan(x):
      value = int(x)
      ...
      self.add(str(value))
  For x == -0.0 this condition is True (because -0.0 == 0 and int(-0.0) == 0). You convert the float to an integer (value = int(x) → 0) and then add str(value) → "0". The sign bit is lost entirely and you end up printing "0" instead of "-0.0".

Root cause (why it happened)
- The code uses numeric comparisons and int(x) conversion which erase the negative zero sign. In particular:
  - The check x < 0 fails to detect negative zero.
  - Converting -0.0 to int produces 0, losing the sign bit.
- Negative zero is distinct at the bit level (sign bit set) even though it compares equal to +0.0, so any logic that relies only on comparisons or int conversion will lose its sign.

Exactly where to look
- The get_last_char / space-insertion check (the first if using x < 0).
- The integer-path starting at the line you provided (line 176): "if x == int(x) ... value = int(x) ... self.add(str(value))".

Concrete fixes (step-by-step)
1. Add a robust negative-zero detector. Example implementation to put on the class (used by your tests):
   def is_negative_zero(x: float) -> bool:
       return x == 0.0 and math.copysign(1.0, x) < 0.0
   - This checks value is zero and the sign bit is negative using math.copysign. It detects -0.0 reliably.

2. Use that detector in the space-insertion check:
   prev = self.get_last_char()
   if (x < 0 or self.is_negative_zero(x)) and prev == '-':
       self.add(" ")
   - This ensures a negative zero after a minus gets the same spacing as any other negative.

3. Short-circuit the integer branch for negative zero so you preserve the sign and a decimal point:
   # before the "if x == int(x)" branch, add:
   if self.is_negative_zero(x):
       # preserve sign and float presentation from the original code expectations
       self.add("-0.0")
       return
   - This prevents int(x) conversion from wiping the sign and prints "-0.0" exactly as expected by the tests.

4. Optionally: ensure the integer branch excludes negative zero explicitly:
   if x == int(x) and not math.isinf(x) and not math.isnan(x) and not self.is_negative_zero(x):
       ...
   (but step 3 above is clearer and returns early)

Why these steps fix the failure
- They stop using comparisons and int conversion for the special case -0.0, so the sign bit is preserved.
- They make the same space-insertion logic apply to -0.0 as to other negatives.
