What happened (where it fails)

    if minutes_offset < 0 or minutes_offset > 59:

- That check rejects any negative minutes_offset (minutes < 0) unconditionally.
- According to the docstring and your tests, negative minutes are allowed in many cases (for example (0, -15) -> -00:15, (-2, -15) -> -02:15). Why it fails (root cause)
- The code treats the minutes range as 0..59 instead of -59..+59. The variable minutes_offset is being validated against an incorrect lower bound (0) rather than the correct lower bound (-59).
- This is a logic bug in the validation, not a problem with the tests. The docstring explicitly states "The minutes value must be in the range -59 to +59."

Where exactly (variables, method, lines)
- Method: DateTimeZone.for_offset_hours_minutes
- Variable: minutes_offset
- Lines: 396–397 (the two-line condition shown above) — this is where negative minutes are rejected.

2. Preserve the additional rule that positive hours cannot be combined with negative minutes (the docstring says +ve hour with -ve minute is ValueError). Implement explicit sign checking for invalid mixed signs:
   - If hours_offset > 0 and minutes_offset < 0: raise ValueError
   - (The docstring allows (-2, +15) and (-2, -15), so negative hours with positive minutes is OK.)
