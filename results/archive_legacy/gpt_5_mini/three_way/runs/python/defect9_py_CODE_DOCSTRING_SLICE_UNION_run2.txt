What happened (what fails)
- The code crashes with a TypeError when either search_list[i] or replacement_list[i] is None. The crash occurs in the loop that computes the initial buffer-size guess (lines 81–85 in your slice), because it calls len(...) on possibly-None elements:
  greater = len(replacement_list[i]) - len(search_list[i])
  If replacement_list[i] or search_list[i] is None, len(None) raises TypeError.

Why it fails (root cause)
- Earlier you deliberately skip None or empty entries when searching/replacing (see the guard used at lines 59–76 and again at lines 100–106). But the "increase" estimation loop (lines 81–85) does not use the same guard — it assumes every entry is a non-null string and so calls len(...) unconditionally. That mismatch is the root cause: a null-safe matching loop exists, but the size-estimate loop is not null-safe.

Where it happens (exact lines/variables)
- Lines 81–85:
    for i in range(search_length):
        greater = len(replacement_list[i]) - len(search_list[i])
        if greater > 0:
            increase += 3 * greater  # assume 3 matches
  Variables involved: search_length, search_list[i], replacement_list[i], increase.
- The later matching loops (lines ~59–76 and ~100–113) correctly check:
    if no_more_matches_for_repl_index[i] or search_list[i] is None or len(search_list[i]) == 0 or replacement_list[i] is None:
        continue
  The estimation loop must use the same checks.

How to fix it (explicit steps and example patch)
1. Use the same guard used elsewhere before calling len(...) on the pair:
   - Skip the element if search_list[i] is None, len(search_list[i]) == 0, or replacement_list[i] is None.
2. Then compute the length difference only for the safe elements.

Minimal replacement for lines 81–85:
    for i in range(search_length):
        # Skip null/empty search strings or null replacements (same guard as later loops)
        if search_list[i] is None or len(search_list[i]) == 0 or replacement_list[i] is None:
            continue
        greater = len(replacement_list[i]) - len(search_list[i])
        if greater > 0:
            increase += 3 * greater  # assume 3 matches

Alternative safe approach:
- Treat None as length 0 instead of skipping, e.g.:
    s_len = len(search_list[i]) if search_list[i] is not None else 0
    r_len = len(replacement_list[i]) if replacement_list[i] is not None else 0
    greater = r_len - s_len
  But that changes the behavior — prefer the first fix to match the intended "ignore None entries" semantics.

Why this fix is correct
- It makes the buffer-size estimation loop consistent with the matching/replacement loops: it won't call len(None), and it will ignore the same entries that are ignored later. That removes the TypeError while preserving intended behavior.

Summary
- Crash location: the increase estimation loop (lines 81–85) calls len() on possibly None list elements.
- Fix: add the same null/empty checks used elsewhere before calling len(...), or otherwise handle None safely.
