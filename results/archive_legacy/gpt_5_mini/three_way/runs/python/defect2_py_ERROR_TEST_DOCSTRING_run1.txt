What happened
- The test calls GrayPaintScale.get_paint with values outside the scale (e.g. -0.5 and 1.5).
- The implementation of get_paint computes RGB by scaling the raw value (or by scaling before clamping) and then passes those numbers to the code that validates/constructs the color.
- Because the value was not clamped first, the computed red/green/blue fell outside 0–255 and the color constructor/validator raised ValueError: "Color parameter outside of expected range: Red Green Blue".

Why it fails (root cause)
- Variable names in your method: value (the input), lower / upper (scale bounds), and the computed gray/color components (e.g. gray, r, g, b).
- The code scales value to an RGB component without clamping value to the [lower, upper] range first (or it clamps after scaling). For values < lower or > upper the scaled component becomes < 0 or > 255.
- The color creation/validation code detects that r/g/b are out of the valid 0–255 range and raises the ValueError. The test expects get_paint to clamp out-of-range inputs to black/white instead of propagating a bad component to the color constructor.

Where it occurs
- Inside GrayPaintScale.get_paint, at the lines that:
  1) compute the scaled gray value from value, and then
  2) construct/return the RGB tuple (or a Color object).
- The exception message you saw ("Color parameter outside of expected range: Red Green Blue") is generated by the color-construction/validation code when r/g/b are invalid.

Exact reason in terms of variables/operations
- If you compute gray_component = int((value - lower) / (upper - lower) * 255.0) while value is -0.5 (and lower is 0.0), gray_component becomes negative (e.g. -127), and passing (-127, -127, -127) triggers the ValueError.
- Similarly for 1.5 you get a value > 255 (e.g. 382), also invalid.

Concrete fixes (step-by-step)
1. Clamp the input value to the bounds before scaling:
   - if value < self.lower: value = self.lower
   - if value > self.upper: value = self.upper
2. Compute the normalized fraction safely:
   - if self.upper == self.lower: define normalized = 0.0 (or 1.0) to avoid division by zero
   - else normalized = (value - self.lower) / (self.upper - self.lower)
3. Scale and clamp the component into integer 0..255:
   - gray = int(normalized * 255.0)
   - gray = max(0, min(255, gray))
4. Return the RGB tuple: return (gray, gray, gray)

Why this fixes the tests
- Clamping value before scaling ensures any input below lower becomes exactly lower (black) and any above upper becomes upper (white), so computed r/g/b will always be in 0..255 and no ValueError will be raised.
