What happened (where it fails)
- In get_paint, you clamp the input into v on lines 2–4:
    v = max(value, self._lower_bound)
    v = min(v, self._upper_bound)
  but on line 6 you compute g using value instead of the clamped v:
    g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- Because you used value (the original, unclamped input) the computed g can be < 0 or > 255 when value is outside the bounds, so the subsequent range check raises ValueError (the error you saw).

Why that causes the exact error
- The clamp assignments set v into the allowed range, but they have no effect because you never use v in the calculation. The check is validating g derived from the original value, therefore clamping had no effect and the ValueError is triggered.

Other potential problem to watch for
- If self._upper_bound == self._lower_bound, the denominator (self._upper_bound - self._lower_bound) will be zero and the division will raise ZeroDivisionError. That can occur separate from the current ValueError.

How to fix it (concrete steps)
1. Use the clamped variable v in the computation instead of value:
    g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
2. Prevent division-by-zero by handling equal bounds before dividing. For example:
    if self._upper_bound == self._lower_bound:
        g = 0  # or 255, or some sensible default
    else:
        g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
3. Improve the diagnostic message so it shows the computed g and the inputs when raising an error, to aid debugging:
    raise ValueError(f"Color parameter out of range: g={g} from value={value}, bounds=({self._lower_bound},{self._upper_bound})")

Minimal corrected example
    def get_paint(self, value: float) -> Tuple[int, int, int]:
        v = max(value, self._lower_bound)
        v = min(v, self._upper_bound)

        if self._upper_bound == self._lower_bound:
            g = 0  # choose appropriate default
        else:
            g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

        if g < 0 or g > 255:
            raise ValueError(f"Color parameter out of range: g={g} from value={value}, bounds=({self._lower_bound},{self._upper_bound})")

        return (g, g, g)

This fixes the immediate bug (uses the clamped v) and guards against division-by-zero.
