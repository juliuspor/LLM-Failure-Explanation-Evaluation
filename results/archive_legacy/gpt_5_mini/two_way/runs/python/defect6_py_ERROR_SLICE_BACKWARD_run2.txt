What happened (root cause, where)
- At line 176 you check if x == int(x) and then later convert x to an int. For x == -0.0 that equality is true because IEEE-754 treats -0.0 == 0.0. So the code enters the branch and converts -0.0 into the integer 0, losing the negative-zero sign.
- The failing assertion "Expected -0.0, got 0" shows exactly that: the code returned an int 0 instead of preserving the float -0.0. The problematic check is on line 176: if x == int(x) and not math.isinf(x) and not math.isnan(x):

Why this happens (simple, precise)
- In Python floats follow IEEE-754: -0.0 compares equal to 0.0. The expression x == int(x) therefore returns True for x == -0.0.
- Converting -0.0 to int yields 0 (an int has no signed zero), so the negative sign is lost and you return 0 instead of preserving -0.0.

1. Detect signed zero before converting. Use math.copysign to see whether x is a negative zero:

2. Change the condition so you do not treat negative zero as an integer-like value. For example, replace line 176 with:

3. That keeps -0.0 out of the int-conversion branch and preserves the float -0.0 return path.
