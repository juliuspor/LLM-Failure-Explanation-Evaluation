What happens
- The bug lives in this sequence:
  1) consumed = self.translate_codepoint(code_units, pos, out)  
  2) if consumed != 0: for pt in range(consumed): pos += char_count(code_point_at(code_units, pos))

- The code assumes translate_codepoint returns a count of codepoints (Unicode scalar values). It then advances pos one codepoint at a time by calling code_point_at(...) and adding char_count(...) for each codepoint.

- In your actual implementation translate_codepoint returns the number of UTF-16 code units (chars) consumed, not the number of codepoints. For any supplementary character encoded as a surrogate pair (two UTF-16 code units), translate_codepoint will return 2. The caller then runs the for-loop twice and on each iteration calls code_point_at(...) which interprets the surrogate pair as a single codepoint the first iteration (char_count -> 2) and then as the second half of the pair (or the next codepoint) the next iteration (char_count -> 1). The net effect is that pos is advanced too far (2 + 1 = 3) or otherwise incorrectly, which leads to skipped characters, out-of-range reads, or an infinite loop depending on input.

Why it fails (root cause)
- Mismatch of units: translate_codepoint's return value and the caller's interpretation of that value use different units (UTF-16 code units vs. Unicode codepoints). The code that advances pos multiplies the advancement for surrogate pairs because it treats a "2" returned for a surrogate pair as "two codepoints" and adds char_count twice. In short: you are double-counting surrogate pairs.

Exactly where (names from code)
- Problem call: consumed = self.translate_codepoint(code_units, pos, out)
- Problem loop: for pt in range(consumed): pos += char_count(code_point_at(code_units, pos))

How to fix (concrete steps)
Pick one of these consistent options and implement it:

Option A — make translate_codepoint return number of codepoints
1. Change translate_codepoint so it returns how many Unicode codepoints it consumed (1 for a BMP char, 1 for a surrogate pair, 0 for “I didn’t consume anything”, etc.).
2. Keep the existing loop:
   for pt in range(consumed):
       pos += char_count(code_point_at(code_units, pos))
   This will now advance pos by the correct number of UTF-16 code units (char_count will be 2 for a surrogate pair the single time it’s counted).

Option B — keep translate_codepoint returning UTF-16 code units and advance pos directly
1. Treat consumed as the number of UTF-16 code units consumed. Remove the per-codepoint loop and just do:
   pos += consumed
2. Do not call code_point_at or char_count in that branch.

Option C — return units and make the loop use units instead of codepoints
1. If you must detect codepoints while advancing, compute units consumed from the codepoints explicitly by reading code_point_at repeatedly but ensure you call it consumed times in terms of codepoints (not units). This is more complex and usually unnecessary.

Which to choose
- Prefer Option B if translate_codepoint already returns UTF-16 code units (simpler and less error-prone).
- Prefer Option A if the higher-level API contract is supposed to be “translators work in codepoints”.

Minimal concrete change (if translate_codepoint returns units): replace the for-loop with
    pos += consumed

If translate_codepoint should return codepoints, change translate_codepoint implementations to count codepoints (1 per BMP or surrogate pair) and leave the loop as-is.

Proof this fixes it
- With the unit-consistent change you will no longer double-count surrogate pairs. pos will advance exactly the number of UTF-16 code units that were actually consumed, avoiding skipped chars, out-of-range reads, and infinite loops.
