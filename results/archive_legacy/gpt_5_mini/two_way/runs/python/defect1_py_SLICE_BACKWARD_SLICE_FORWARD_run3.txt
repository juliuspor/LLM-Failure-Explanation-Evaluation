What happened (where it fails)
- At line 396 the code checks minutes_offset and raises ValueError if minutes_offset < 0 or minutes_offset > 59.
- Earlier in the same routine the code applies the sign to the parsed minute value. That means minutes_offset can already be negative when it reaches line 396.
- Because the sign was applied to the minute component before the range check, a valid offset like "-05" (5 minutes with a negative sign) produces minutes_offset = -5 and triggers the "Minutes out of range" ValueError even though the minute digits themselves are valid.

Why it fails (root cause)
- The variable minutes_offset is used for two different meanings without ordering the operations correctly:
  1. Initially it is the parsed minute component that must be in 0..59.
  2. Later the code reuses the same variable to hold the total minutes offset (hours_in_minutes ± minutes_offset) at lines 402 and 404.
- The code applies the sign to the minute component before validating it (so the validation is checking a signed value instead of the raw minute digits). This incorrect ordering makes valid inputs with a negative sign fail the range check.

Exactly where to change it (concrete steps)
1. Keep the raw minute component separate from any sign application. For example:
   - Use parsed_minutes (or minutes_unsigned) to hold the numeric minutes parsed from the string.
   - Validate parsed_minutes with the current check: if parsed_minutes < 0 or parsed_minutes > 59: raise ValueError...
2. Only after validation apply the sign or combine with hours:
   - If sign is negative: total_minutes = - (hours_in_minutes + parsed_minutes)
   - If sign is positive: total_minutes = hours_in_minutes + parsed_minutes
   - Assign minutes_offset (or a new variable e.g. total_minutes) as the combined value and continue to lines 402/404.
3. Avoid reusing minutes_offset for both the minute component and the total-minute value; use distinct names to prevent the same error reoccurring.

Minimal code sketch (illustrates the ordering change)
- Parse hours and minutes into hours_in_minutes and parsed_minutes.
- Validate parsed_minutes:
    if parsed_minutes < 0 or parsed_minutes > 59:

- Compute total minutes with sign:
    if sign == '-':

    else:

- Use total_minutes where the code currently uses minutes_offset for combination and multiplication.

That change fixes the direct root cause: the validation must run on the raw minute digits (unsigned) before any sign or aggregation is applied.
