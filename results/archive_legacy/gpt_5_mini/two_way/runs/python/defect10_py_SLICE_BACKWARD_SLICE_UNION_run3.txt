What happened
- At lines 30–31 the code does this check:
  if index < 0 or index >= len(s):
      raise IndexError(f"String index out of range: {index}")
- That check runs in both the [SLICE_BACKWARD] and [SLICE_UNION] code paths. Because the check rejects any negative index, any call that passes a negative index (for example index == -1) hits that raise and the function fails with IndexError.

Why it fails (root cause)
- The variable s is the string being indexed and index is the requested position.
- The code treats negative indexes as out of range (line 30), but Python commonly allows negative indexes (e.g. s[-1] means the last character).
- In the backward-slicing path ([SLICE_BACKWARD]) you must accept and normalize negative indexes, but the unconditional "index < 0" branch prevents that and raises the exception before any normalization or conversion happens.
- In short: the bounds check is wrong for negative-index semantics and is executed before converting negative indexes into a valid positive index.

Where it occurs
- Lines 30–31 in both [SLICE_BACKWARD] and [SLICE_UNION]. The raise on line 31 is the immediate symptom.

How to fix it (concrete steps)
1. Decide whether you want to support negative indexes (typical for Python string indexing). If yes, apply step 2. If you want to disallow negatives intentionally, document that and keep the current check.
2. Normalize negative indexes to a non-negative index before the bounds check. For example:
   - If index < 0: index += len(s)
   - Then check bounds with: if index < 0 or index >= len(s): raise IndexError(...)
   - Return s[index] afterwards.
3. Make the change in the [SLICE_BACKWARD] block (and confirm [SLICE_UNION] behavior is correct). Concretely, change the code around lines 30–31 to:
   if index < 0:
       index += len(s)
   if index < 0 or index >= len(s):
       raise IndexError(f"String index out of range: {index}")
   return s[index]
4. Add tests:
