What went wrong (root cause)
- At line 178 you set mantissa = value. If value is 0, then mantissa % 10 == 0 is true and mantissa //= 10 at line 184 leaves mantissa at 0 again. The loop at line 181 never changes the condition and exp keeps increasing forever. In short: the while loop can become an infinite loop when mantissa becomes 0.

Where it happens
- The infinite loop occurs in the block lines 180–187, specifically the while at line 181 and the division at line 184. For x == 0 (or any integer that becomes 0 after repeated //= 10), check_val computed at line 182 is always equal to value (both 0), so the loop never breaks at line 186.

Why the check_val logic fails
- check_val = (mantissa // 10) * (10 ** (exp + 1)) intends to test whether removing one factor-of-10 from mantissa and restoring scale yields the original value. That reasoning is OK for nonzero mantissa, but once mantissa becomes 0 the test becomes tautologically true (0 == value when value is 0) and the algorithm cannot detect that it should stop.

How to reproduce quickly
- Call this branch with x == 0.0 (x == int(x) true, not inf/nan) — program hangs / never returns.

Concrete fixes (clear steps)
1. Prevent entering the infinite loop when mantissa is 0:
   - Replace the loop condition with a guard that requires mantissa to be nonzero:
     while mantissa != 0 and mantissa % 10 == 0:
   This stops the loop when mantissa has become 0.

2. Alternatively, keep a copy of the original value and test against it differently:
   - Use a loop that tests whether (value % (10 ** (exp + 1))) == 0 instead of manipulating mantissa down to 0. Example:
     while value % (10 ** (exp + 1)) == 0:
         mantissa = value // (10 ** (exp + 1))
         exp += 1
   This avoids dividing mantissa to 0 and directly checks trailing zeros in the original integer.

3. (Optional) Add a safety bound:
   - If you expect exp to never exceed some size, add a max_exp break to avoid runaway loops in unexpected cases.
