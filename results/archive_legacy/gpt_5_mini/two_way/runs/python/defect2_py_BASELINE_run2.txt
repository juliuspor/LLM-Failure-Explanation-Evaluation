What happened (root cause)
- In get_paint you attempt to clamp the input value to the valid range into local variable v:
  - lines ~56–58: 
    v = max(value, self._lower_bound)
    v = min(v, self._upper_bound)
- But when you compute the gray channel g you use the original value instead of the clamped v:
  - line 71:
    g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- Because of that, an out‑of‑range input like value = -0.5 produces a negative g (or >255 for values > upper bound) and triggers the range check at line 74:
  if g < 0 or g > 255: raise ValueError(...)
- The failing test (test_lower_bound_clamping) passes -0.5 expecting it to be clamped to lower_bound (0.0) and return (0,0,0), but your code calculates g from -0.5 and raises the ValueError instead.

Where it occurs (specific lines)
- Clamping is done at lines ~56–58 (v = max(...), v = min(...))
- The incorrect computation is on line 71 (uses value)
- The exception is raised on line 74 (range check)

How to fix (concrete steps)
1. Use the clamped variable v when computing g. Replace line 71 with:
   g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
2. Add a check to avoid division by zero if upper_bound == lower_bound:
   if self._upper_bound == self._lower_bound:
       g = 0  # or 255 or whatever your intended behavior is for degenerate range
   else:
       compute g as above
3. (Optional) Ensure final g is clamped to 0..255 (safer than relying on the earlier check):
   g = max(0, min(255, g))

Why this will fix the tests
- Using v guarantees out‑of‑range inputs are mapped to the bounds before conversion, so -0.5 -> v=0.0 -> g=0 and 1.5 -> v=1.0 -> g=255, satisfying the test expectations.

Summary
- Bug: computation uses value instead of clamped v (line 71).
- Effect: out-of-range inputs produce invalid RGB and raise ValueError at line 74.
- Fix: compute g from v and add a zero‑division guard; optionally clamp g to 0..255.
