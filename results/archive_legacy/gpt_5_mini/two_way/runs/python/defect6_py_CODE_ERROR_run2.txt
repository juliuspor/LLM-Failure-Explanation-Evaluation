What happened (where it fails)
- The test passed in x = -0.0.
- Execution goes into the integer branch: the code checks if x == int(x) (True for -0.0) and then does value = int(x).
- int(-0.0) produces 0 (line: value = int(x)), so the sign is lost.
- Later the code does self.add(str(value)) (line near the end of the integer branch) and writes "0".
- The earlier check if x < 0 and prev == '-' does not help because in Python -0.0 < 0 is False, so that block never runs.

Root cause
- Converting -0.0 to an int collapses the negative-zero sign; equality x == int(x) also treats -0.0 as 0. Those two facts together make the function treat -0.0 as plain 0 and emit "0" instead of "-0.0".

Exactly where to look
- The sign is lost at value = int(x).
- The string that gets emitted is built by self.add(str(value)) in the integer branch.
- The conditional that might avoid the problem (if x < 0 and prev == '-') does not trigger for -0.0 because x < 0 is False.

Concrete fixes (pick one)

1) Explicit negative-zero check (recommended, minimal change)
Insert this before the integer branch:
    if x == 0.0 and math.copysign(1.0, x) < 0:
        self.add("-0.0")
        return
This uses math.copysign to detect a negative zero and emits "-0.0" directly.

2) Preserve sign when formatting integer-valued floats
After computing value = int(x), when you would emit str(value), special-case zero using copysign:
    if value == 0 and math.copysign(1.0, x) < 0:
        self.add("-0")
    else:
        self.add(str(value))
Or if you want "-0.0" specifically, emit "-0.0" instead of "-0".

3) Avoid int() conversion for formatting
Format x while preserving its sign with a float-to-string formatter rather than int conversion, for example:
    s = ("{:.0f}".format(x))
This will produce "-0" for -0.0; if you need "-0.0" specifically, format accordingly:
    if x == 0.0 and math.copysign(1.0, x) < 0:
        s = "-0.0"
    else:
        s = format(x, ".0f")  # or whatever integer formatting you want
    self.add(s)

Which to choose
- If the test expects "-0.0", use option 1 (explicit negative-zero check and emit "-0.0"), or option 2 modified to emit "-0.0".
- If you only need to preserve a "negative" zero sign in integer output, option 2 is fine.

Summary
The code treats -0.0 as 0 because int(-0.0) == 0 and comparisons collapse the sign; detect negative zero with math.copysign and handle it explicitly before converting to int (or preserve sign when producing the final string).
