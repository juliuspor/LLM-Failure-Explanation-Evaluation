What happened (root cause)
- The failure happens inside the block that handles trailing long-qualifiers:
  if last_char in ("l", "L"):
      if (char_at(numeric, 0) == "-" and numeric[1:].isdigit()) or numeric.isdigit():
          return int(numeric)
- For input "l" or "L" you get:
  - val = "l"
  - last_char = "l" (not a digit) → numeric = val[:-1] → numeric == "" (empty string)
  - Then the code calls char_at(numeric, 0). That function checks index bounds (lines 30–31) and raises IndexError("String index out of range: 0") because numeric has length 0.
- The test expects a ValueError for "l"/"L" (bad input). Instead the code raises IndexError due to indexing an empty string. The IndexError originates at the char_at(numeric, 0) call inside the last_char in ("l","L") branch.

Where it occurs in the snippet

        # BUG: missing guard for len(val) == 1 and non-digit input.

Why the code is wrong (clear reason)
- The code assumes numeric (val without the qualifier) has at least one character and indexes numeric[0] (via char_at) and numeric[1:] without checking length.
- For single-character inputs that are only the qualifier (e.g., "l", "L"), numeric is empty and indexing numeric[0] is invalid. The code should treat these as invalid numbers and raise ValueError, not propagate IndexError.

Minimal corrected code fragment
    if last_char in ("l", "L"):
        if len(numeric) > 0 and ((numeric[0] == "-" and numeric[1:].isdigit()) or numeric.isdigit()):
            return int(numeric)

Why this fixes it
- The len(numeric) > 0 check prevents any attempt to read numeric[0] or numeric[1:] when numeric is empty, so the function will raise the expected ValueError for inputs like "l" and "L" instead of IndexError.

Summary
- Root cause: indexing an empty string (char_at(numeric, 0)) when val is just the qualifier ("l"/"L").
- Fix: guard against empty numeric before indexing, or explicitly check len(val) == 1 and raise ValueError.
