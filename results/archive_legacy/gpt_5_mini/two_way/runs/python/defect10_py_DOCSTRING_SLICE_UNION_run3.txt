What happened (root cause)
- The parser removes a trailing qualifier character ('l' or 'L') but does not check whether anything remains in the string afterwards. For the input "l" (or "L") the code does this:
  1. s is "l".
  2. The qualifier-stripping code trims the final character and sets s = "".
  3. Later code attempts to access s[0] to examine the first character. That access goes through the helper shown in [SLICE_UNION].
  4. Because index 0 is out of range for the empty string, slice_union raises IndexError at lines 30–32:
     - 30: if index < 0 or index >= len(s):
     - 31:     raise IndexError(f"String index out of range: {index}")
     - 32: return s[index]
- The caller code expects a ValueError for invalid numeric literals, but instead gets IndexError from slice_union. In short: stripping the qualifier can produce an empty s, and the code never validates s before calling s[0].

Where it occurred (specific names/lines)
- Variable: s (the string being parsed) becomes empty after qualifier stripping.
- Faulty access happens when the code reads s[0] (through slice_union). The explicit IndexError is raised by slice_union lines 30–32.
- The qualifier removal code path (the place that sets s = s[:-1] or similar) is the step that makes s empty for input "l" / "L".

Why this is wrong
- An empty string is not a valid numeric literal; the parser should treat "l" as an invalid input and raise ValueError (per the intended behavior). Instead it allows an out-of-range index, producing IndexError which leaks an implementation detail and is the wrong exception type.

Concrete fixes (step-by-step)
1. After you strip the qualifier, immediately check whether s is empty:
   - if not s:
       raise ValueError(f"Invalid numeric literal: '{original_input}'")
   This prevents any later s[0] access on an empty string.
2. Alternatively (or in addition), before every direct indexed access like s[0], check the length:
   - if len(s) == 0:
       raise ValueError("Invalid numeric literal")
3. If you want to preserve the original input in error messages, keep original_input in a local variable before modifying s and use it in the ValueError.
4. Add a unit test for the exact cases "l" and "L" asserting that ValueError is raised. Also add tests for other single-character qualifiers and for valid multi-character numeric literals with qualifiers.

Why these fixes are sufficient
- They ensure invalid inputs that become empty after qualifier stripping are rejected with ValueError instead of causing an IndexError inside slice_union. That fixes Lang-44 and gives the caller the correct, well-defined error.
