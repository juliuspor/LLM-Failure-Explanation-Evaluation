What happened (root cause)
- At line 349 the code unconditionally sets self._min_start_index = index, which overwrites whatever the comparison at lines 346–347 just did. That means the "keep the index of the minimum start" logic never works: after the method runs self._min_start_index will always be set to the current index regardless of whether start < min_start.
- Concretely:
  - Lines 340–347 try to compare the new period's start (start) with the stored minimum start (min_start) and only update self._min_start_index when the new start is earlier.
  - Then line 349 executes and assigns self._min_start_index = index no matter what, undoing the conditional update.
- Because of that stray unconditional assignment at line 349, the minimum index tracking fails — the stored index will always be the most recently processed index.

Where it occurs
- The bug is in the block beginning at line 340 (if self._min_start_index >= 0) and the unconditional assignment at line 349. These lines are responsible for updating self._min_start_index.

Why it fails (step-by-step)
1. If self._min_start_index >= 0, the code loads the existing min period and computes min_start (lines 341–346).
2. It checks if start < min_start and updates self._min_start_index = index when the new start is earlier (lines 346–347).
3. Immediately after, line 349 executes and sets self._min_start_index = index regardless of the comparison result, so the conditional update is meaningless.
4. The stored minimum index therefore becomes the last index processed, not the index with the earliest start.

Precise fix (recommended code change)
Replace the unconditional assignment with an else branch so you only set the stored index when there is no prior min. For example (showing relevant lines):

 340 |     if self._min_start_index >= 0:
 341 |         min_start_period = self.get_data_item(self._min_start_index).get_period()
 342 |         if isinstance(min_start_period, SimpleTimePeriod):
 343 |             min_start = min_start_period.get_start_millis()
 345 |             min_start = int(min_start_period.get_start().timestamp() * 1000)
 346 |         if start < min_start:
 347 |             self._min_start_index = index
 349 |     else:
 350 |         # no previous min, set this index as the minimum
 351 |         self._min_start_index = index

Additional checks to add
- Ensure min_start is always defined before the comparison (line 346). The current code has two ways to set min_start (get_start_millis() vs timestamp), so verify both branches set min_start consistently.
- Add unit tests that:
  - Insert a first period -> expect self._min_start_index == its index.
  - Insert a later period earlier in time -> expect index updated to the earlier one.
  - Insert a later period later in time -> expect index unchanged.

Why this fix solves it
- Using an else ensures you only set self._min_start_index to the current index when there is no prior minimum. When a prior min exists, the conditional update (start < min_start) will control whether the stored index changes. This removes the overwriting caused by the unconditional assignment at line 349.
