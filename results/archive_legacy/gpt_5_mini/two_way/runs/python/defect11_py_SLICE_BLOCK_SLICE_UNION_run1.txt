What happened (where it failed)
- The exception is raised on line 52: `if gap <= 0: raise ValueError("bound must be positive")`.
- `gap` is computed on line 49 as `gap = end - start`.
- If the caller supplies `start == end` or `end < start`, then `gap` is 0 or negative and the code raises.

Why it failed (root cause)
- The code treats `end` as an inclusive bound but computes `gap` as if `end` were exclusive. With inclusive bounds you must allow `start == end` (one legal value). Using `end - start` gives 0 for a single allowed character, which the code interprets as an error.
- Example that triggers it: calling `random(count, start=65, end=65, letters=True, numbers=False)` should be valid for the single character code 65, but `gap = 65 - 65 = 0` and line 52 raises.

Where to change it (exact steps)
1. Change the gap calculation to include the end bound:
   - Replace line 49 `gap = end - start` with `gap = end - start + 1`.
2. Keep the guard but it will now correctly only reject `end < start`:
   - Line 52 can remain `if gap <= 0: raise ValueError("bound must be positive")`.
3. No other changes are required if you continue to use `_RANDOM.randrange(gap)` on line 56; it will now produce a value in `[0, gap-1]` and `start + ...` will cover `start..end` inclusive.

Alternative (if you intended end to be exclusive)
- If `end` was intended as exclusive, document that and change the check to `if gap < 1: ...` (or keep `gap <= 0` and ensure callers pass an exclusive upper bound).
- But given the current behavior (code sets `end = 127` and `start = 0` in lines 46–47 expecting inclusive range of printable ASCII), making `gap = end - start + 1` is the correct fix.

Quick reproduction
- Call `random(1, 65, 65, True, False)` and you will hit the ValueError with the current code. After the change in step 1 the same call will return a character for code point 65.
