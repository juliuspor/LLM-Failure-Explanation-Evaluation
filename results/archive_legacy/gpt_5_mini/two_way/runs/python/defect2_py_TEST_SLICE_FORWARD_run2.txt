What fails
- The tests fail because GrayPaintScale.get_paint does not clamp the input value into the lower..upper range before computing the grayscale channel. As a result, values < lower produce negative channel numbers and values > upper produce numbers > 255, which later cause errors (or wrong colors) when the code constructs the RGB tuple / Color object.

What happens (step‑by‑step)
1. The test calls gps.get_paint(-0.5) (default GrayPaintScale bounds: lower = 0.0, upper = 1.0).
2. Inside get_paint the code computes the normalized fraction like:
   fraction = (value - self.lower) / (self.upper - self.lower)
   and then computes a channel value:
   channel = int(fraction * 255)
3. For value = -0.5 this gives fraction = -0.5 and channel = int(-127.5) -> -127.
4. The code then tries to use that channel value as an RGB component (or to build a Color/PIL object). Because RGB channels must be in 0..255, you either get wrong output (negative tuple) or an exception later when a library validates channel ranges (this is the failure the tests surface).
5. The same logic breaks values > upper (e.g. 1.5 -> fraction 1.5 -> channel 382 -> >255), so tests expecting clamping to black/white fail as well.

Where to change (exact method and variable names)
- Change GrayPaintScale.get_paint. Before computing fraction or before turning it into an int channel, clamp the input value into [self.lower, self.upper]. Concretely, inside get_paint replace the raw mapping code with a clamped mapping.

Suggested fix (explicit steps)
1. Open hit02_grayscale.py and find the get_paint method.
2. Locate the lines that compute the fraction and channel (the lines that reference value, self.lower, self.upper and compute a 0..255 channel).
3. Replace them with code that clamps value, then computes the channel from the clamped fraction.

Minimal correct mapping (pseudocode you can paste into get_paint)
- clamp value:
  v = max(self.lower, min(self.upper, value))
- compute fraction safely:
  fraction = (v - self.lower) / (self.upper - self.lower)
- compute channel and ensure it is an int in 0..255:
  channel = int(fraction * 255)
  channel = max(0, min(255, channel))
- return an RGB tuple:
  return (channel, channel, channel)

Why this fixes it
- Clamping v to [lower, upper] guarantees fraction is in [0.0, 1.0], so fraction*255 is in [0.0, 255.0]. After int() and a final clamp you always produce a valid 0..255 channel that matches the tests’ expectations: values below lower become 0 (black), values above upper become 255 (white), and middle values interpolate.

Notes
- If your code uses round() instead of int(), be aware that round(127.5) may produce 128; the tests expect int(127.5) -> 127. If you want the tests to pass exactly as written, use int(truncated) semantics or adjust the test.
- Also make sure self.upper != self.lower (avoid division by zero). If same, decide on behavior (always return black or white, or raise a clear exception).
