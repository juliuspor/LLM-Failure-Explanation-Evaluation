What failed
- The code at line 82 calls len(replacement_list[i]) without first ensuring replacement_list has an element at index i and that the element is a string (not None). That can raise either:
  - IndexError if replacement_list is shorter than search_list, or
  - TypeError if replacement_list[i] is None (len(None) is invalid).

Where it happens
- In replace_each:
  - search_length is set at line 52: search_length = len(search_list)
  - The loop at line 81: for i in range(search_length):
  - The problematic call is at line 82: greater = len(replacement_list[i]) - len(search_list[i])
  - No checks that replacement_list has length >= search_length or that replacement_list[i] is a valid string happen before this loop (the later checks around lines 102–108 that test for None or zero-length search strings run after this estimation).

Why that causes failure
- The code assumes replacement_list is a same-length list of strings. If replacement_list is shorter than search_list, attempting replacement_list[i] for a missing index causes an IndexError at line 82. If replacement_list[i] exists but is None, len(replacement_list[i]) raises TypeError at line 82. Because the size/validity checks for search_list/replacement_list entries occur later (lines 102–105), this estimation runs on unvalidated data and can crash before those checks run.

Concrete steps to fix (apply one or more)
1. Validate lengths up front (preferred)
   - After line 52 add a check that replacement_list has the same length as search_list:
     - if len(replacement_list) != search_length: raise ValueError("replacement_list must be same length as search_list")
   - This makes the assumption explicit and fails early with a clear message.

2. Or guard access in the loop
   - Replace the loop at 81–84 with iteration over indices that exist in both lists:
     - for i in range(min(search_length, len(replacement_list))):
         greater = len(replacement_list[i]) - len(search_list[i])
     - This prevents IndexError, but you must still decide what to do for unmatched indices (skip, error, or treat as empty replacement).

3. Or check for None/type before len()
   - Inside the loop, do:
     - if i >= len(replacement_list) or replacement_list[i] is None:
         continue  # or treat as "" or raise, depending on intended behavior
     - greater = len(replacement_list[i]) - len(search_list[i])
   - This avoids TypeError and IndexError.

4. Move the length-estimate after the per-item validation
   - The code’s later logic (lines 102–108) already checks for None or empty search entries. Move the "increase" calculation to after those checks, or reuse the validated flags (no_more_matches_for_repl_index / validated lists) so len(...) is only called on known-good strings.

Which to choose
- If the function requires one-to-one lists, use step 1 (raise ValueError). That makes the contract explicit and prevents subtle bugs.
- If differing lengths are allowed, pick step 2 or 3 and document the intended behavior for missing or None replacements.

Summary
The failure occurs because line 82 assumes replacement_list[i] exists and is a string, but search_length is derived only from search_list (line 52) and validation of replacement_list entries happens later (lines 102–108). Fix by validating replacement_list length/contents before calling len(replacement_list[i]) or by guarding that access.
