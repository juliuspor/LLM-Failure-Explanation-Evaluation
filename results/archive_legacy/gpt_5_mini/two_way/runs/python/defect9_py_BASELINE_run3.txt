What happened (root cause)
- The crash occurs at the loop starting on line 81 (shown in the slice block). On line 82 the code does:
  greater = len(replacement_list[i]) - len(search_list[i])
- If replacement_list[i] is None (or search_list[i] is None), calling len(...) raises TypeError: object of type 'NoneType' has no len().
- The code already has null-safety checks later when scanning for matches (lines 102–105), but this "increase" loop does not check for None or empty entries. The inline comment even says "BUG: this loop is not null-safe for elements inside the arrays" — that is exactly the problem.

How the failing test triggers it
- The test calls replace_each("aba", ["a"], [None]) or replace_each("aba", ["a","b"], ["c", None]).
- For those inputs replacement_list contains None, so the first time the loop at line 81 runs it attempts len(replacement_list[i]) and crashes.

Where to change the code (concrete steps)
1. In the loop at lines 81–84, skip entries that are invalid (None or empty search strings or None replacement), the same way the later match-finding loops do.
2. Only compute lengths when both search_list[i] and replacement_list[i] are non-None and search_list[i] is not "".

Minimal corrected version of that loop:

    for i in range(search_length):
        if (
            search_list[i] is None
            or len(search_list[i]) == 0
            or replacement_list[i] is None
        ):
            continue
        greater = len(replacement_list[i]) - len(search_list[i])
        if greater > 0:
            increase += 3 * greater  # assume 3 matches

Why this fixes it
- It prevents calling len() on None. It makes the "increase" estimation loop consistent with the null checks used later in the function, so inputs with None entries are ignored rather than causing a TypeError.

Notes
- No other logic needs changing; the rest of the function already guards against None/empty entries when finding matches.
