What happened
- The test fails because replace_each tried to use a None value as if it were a string.
- In the replace_each implementation (inside StringUtils.replace_each) the code loops through the parallel arrays search_list and replacement_list and performs string operations such as text.replace(search, replacement) (or concatenation, find, etc.) without first handling the case replacement == None.
- Python's str methods require the replacement argument to be a str. Calling text.replace(search, None) raises a TypeError: replacement must be str, not None. That is the immediate cause of the crash when a replacement_list element is None.

Where it occurs (variables and location)
- Method: StringUtils.replace_each
- Variables involved: text (the input string), search_list, replacement_list, and the loop index i or the loop variables search and replacement.
- Problem spot: inside the loop where the code calls a string method using replacement (for example text.replace(search_list[i], replacement_list[i]) or otherwise concatenates replacement_list[i] to text) without guarding replacement_list[i] against None.

Why it is a bug (root cause)
- The test expects None in replacement_list to be treated as “ignore this replacement” (same as the Java StringUtils behavior being tested). Instead, the implementation treats None as if it were a string and passes it to a str operation, which fails in Python.
- The code is missing an explicit check for replacement is None (and possibly for search is None) before performing the replacement.

Concrete steps to fix (explicit, actionable)
1. In StringUtils.replace_each, locate the loop that iterates through the parallel arrays (something like: for i in range(len(search_list)) or for search, replacement in zip(search_list, replacement_list)): that is where the call to a str method happens.
2. Add an explicit guard that skips that iteration when the replacement is None (and also consider skipping when the search term is None). Example logic:
   - If search is None: continue
   - If replacement is None: continue
   This implements “ignore None entries” as the tests expect.
3. Replace the problematic line(s) where you call the string operation with guarded code. For example, change code like this:
   - old: text = text.replace(search, replacement)
   - new:
     if search is None or replacement is None:
         continue
     text = text.replace(search, replacement)
4. If the implementation uses indices, do the same check using replacement_list[i] and search_list[i].
5. Run the unit tests to confirm:
   - test_null_safety_inside_arrays_single should pass (None replacement ignored).
   - test_null_safety_inside_arrays_mixed should pass (one replacement applied, the None ignored).
   - test_replace_each_basic should still pass.

Notes and edge cases
- Ensure you check both search and replacement entries; a None search should also be ignored (passing None as the search argument to str.replace would also raise an error).
- Keep the length handling consistent (e.g., if search_list and replacement_list lengths differ your code should either handle that case or raise a clear error).
- If your implementation uses a more complex algorithm (multiple passes to avoid overlapping replacements), apply the same None-guard at the points where the replacement string is actually used.

With those checks in place, calling replace_each("aba", ["a","b"], ["c", None]) will: apply the "a"->"c" replacement, skip the "b"->None replacement, and produce the expected "cbc".
