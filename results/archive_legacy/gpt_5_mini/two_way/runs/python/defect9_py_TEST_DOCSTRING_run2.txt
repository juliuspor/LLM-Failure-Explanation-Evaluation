What happened
- The failing tests call StringUtils.replace_each(text, search_list, replacement_list) with replacement_list containing None (Python value None).
- Inside your replace_each implementation the code assumes every replacement is a string and uses string operations on it (for example concatenation, .replace, len(), or ''.join). When it reaches a None it raises a TypeError (e.g. "can only join str (not NoneType)") or AttributeError (if it tries repl.replace(...)). That is why the test crashes instead of ignoring that replacement.

Where it occurs
- The problem is inside the StringUtils.replace_each function (the code that iterates over search_list / replacement_list and applies replacements). The variables involved are:
  - search_list (the list of substrings to find),
  - replacement_list (the list that can contain None),
  - search or s (the current search string),
  - repl (the current replacement value),
  - text or result (the string being transformed),
  - loop index i.
- The crash point is the line where the code uses repl as a string (for example joining replacement_list, calling repl.replace(...), doing result = result.replace(search, repl), or accessing len(repl)). Any such use will fail when repl is None.

Why it is a bug (root cause)
- The tests (and the original Java bug Lang-39) expect that a None/null in replacement_list means "do not replace this search string" (or "ignore this pair"), not "treat None as a string".
- The implementation does not check for None and therefore does not follow the intended contract: it should skip a pair where the replacement is None.

How to fix it (concrete steps)
1. Before doing any string operations on a replacement, check whether it is None and skip that pair.
2. Ensure the same for search entries: skip pairs where the search entry is None or an empty string.
3. If you build helper structures (like a combined regex, a map of search->replacement, or call ''.join(replacement_list)), make those constructions only from non-None, non-empty entries.

Minimal concrete patch (replace the core loop)
- If your code currently does something like:
    for i in range(len(search_list)):
        search = search_list[i]
        repl = replacement_list[i]
        result = result.replace(search, repl)
  change it to:
    for i in range(len(search_list)):
        search = search_list[i]
        repl = replacement_list[i]
        # skip if search or replacement is None (or if search is empty)
        if search is None or repl is None:
            continue
        if search == "":
            continue
        result = result.replace(search, repl)

- If you build a combined string from replacement_list (e.g. ''.join(replacement_list)) instead do:
    safe_replacements = [r for r in replacement_list if r is not None]
    # use safe_replacements for any join or processing that assumes strings

Edge cases to handle
- search_list and replacement_list lengths: keep current behavior (usually both lists must be same length). If you change to skipping entries, do not silently re-pair remaining items — keep pairing by index and skip pairs where either element is None.
- If search is the empty string (""), decide behavior (most implementations skip empty search values because replacing empty string is ambiguous). The test suite expects safe behavior (do not crash).

Example change summary
- Add checks in StringUtils.replace_each:
    if search_list is None or replacement_list is None:
        return text
    for i, search in enumerate(search_list):
        repl = replacement_list[i]
        if search is None or repl is None:
            continue
        # apply replacement

This ensures the function ignores None replacement entries (matching the tests / Lang-39) and prevents attempts to call string methods on None.
