What happened (summary)
- The code raises IndexError at the check in code_point_at (lines 35–36) because the translator advances the index incorrectly when it encounters a UTF‑16 surrogate pair. That produces an index value that is outside the bounds of the UTF‑16 code_units array passed to code_point_at.

Why it happens (root cause, plain)
- The translator converts the input string into UTF‑16 code units (to_utf16_code_units) and then walks those code units using a variable named index.
- When it detects a supplementary code point (a character encoded as a surrogate pair in UTF‑16), the translator advances index by 2 to skip the two code units that make the pair.
- But the loop surrounding that logic also advances the iteration (or otherwise advances index again), so index ends up incremented one time too many. On the next call to code_point_at(code_units, index) the index equals or exceeds len(code_units), so the check at lines 35–36 raises IndexError: "String index out of range: {index}".
- In short: index is advanced twice for surrogate pairs (once for the surrogate handling and once by the loop), so the code asks for a code unit past the end of the code_units list.

Where it happens (specific references)
- Failing check: code_point_at, lines 35–36:

      raise IndexError(f"String index out of range: {index}")
- Offending variables and methods: index (the loop index), code_units (from to_utf16_code_units), and the call site in the translator (CsvEscaper.translate_string / CharSequenceTranslator.translate) where the code advances index after handling a surrogate pair.
