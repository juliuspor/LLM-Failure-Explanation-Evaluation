What happens
- Root cause: when lower is larger than len(text), the code raises an IndexError inside substring_java because upper gets set to that too-large lower, so substring_java(...) is called with end > len(s). The slice bounds check in SLICE_UNION (lines 32–34) detects end > len(s) and raises "String index out of range: {end}".

Where it occurs (exact places)
1. At the start you clamp upper to the text length:
   if upper == -1 or upper > len(text):
       upper = len(text)
   (this runs at the beginning of the function)
2. Immediately after you do:
   if upper < lower:
       upper = lower
   If lower > len(text), this assignment makes upper > len(text).
3. Later you call substring_java(text, 0, upper) (several places). substring_java performs a bounds check (SLICE_UNION lines 32–34) and raises because end (= upper) > len(text).

Concrete example that triggers the failure
- Call abbreviate("abc", lower=5, upper=3, append_to_end="..."):
  - Step 1: upper is 3, not -1 and not > len(text) (=3), so upper stays 3.
  - Step 2: upper < lower (3 < 5) so upper = lower → upper becomes 5.
  - Step 3: substring_java(text, 0, upper) is called with end = 5, but len(text) = 3 → IndexError from SLICE_UNION.

Why this is the bug (simple explanation)
- You never validate or clamp lower against the text length before you use it to set upper. That lets upper become larger than len(text) and later leads substring_java to be called with an invalid end index.

How to fix it (clear steps)
Pick one of these fixes and apply it near the start of abbreviate (before you set upper <-> lower):

Option A — clamp lower to valid range
- After checking text is not None/empty, add:
  if lower < 0:
      lower = 0
  if lower > len(text):
      lower = len(text)
- This prevents upper from being set to a value > len(text).

Option B — ensure upper is clamped after you adjust it to lower
- Keep your existing logic but after the line if upper < lower: upper = lower add:
  if upper > len(text):
      upper = len(text)
- This guarantees any assignment from lower won't leave upper > len(text).

Option C — validate inputs and fail early
- At the top, validate lower and upper and raise a clear ValueError when lower is out of [0, len(text)] or upper < -1, so the caller knows they passed bad bounds.

Minimal code change I recommend (Option A, exact placement)
- Right after the empty-text check, add:

    if lower < 0:
        lower = 0
    if lower > len(text):
        lower = len(text)

This preserves the intended behavior (you still try to find a space at or after lower) and guarantees upper will never be set larger than len(text), so substring_java(...) won't be called with end > len(s).

Why this fix is safe
- It prevents upper from being raised above len(text) by an oversized lower and resolves the IndexError at SLICE_UNION lines 32–34 without changing other logic flows.
