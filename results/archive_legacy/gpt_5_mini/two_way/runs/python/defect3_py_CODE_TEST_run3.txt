What happened (root cause)
- The code advances the input index incorrectly after a translator handles a codepoint. The incorrect lines are:

      for pt in range(consumed):
          pos += char_count(code_point_at(code_units, pos))

  This is in the while loop after

      consumed = self.translate_codepoint(code_units, pos, out)

- Problem: translate_codepoint has already consumed N UTF-16 code units (a surrogate pair is two code units). The loop then attempts to advance pos by repeatedly calling code_point_at(code_units, pos) and adding char_count(...) — but that re-reads the same positions and interprets them as new codepoints while incrementing pos inside the loop. That leads to two concrete failures:
  1. Wrong advancement: you end up adding the UTF-16 code unit length of successive codepoints, which double-counts or miscounts the units consumed. In the surrogate-pair case you can advance too far or too little.
  2. IndexError: code_point_at(code_units, pos) can read a code unit beyond the end of code_units (or attempt to read a second half of a surrogate when none exists) because pos is advanced inside the loop while the loop still expects to call code_point_at on a valid starting position. That is exactly what the failing test demonstrates: when translating a surrogate pair followed by another char, the index arithmetic causes an out-of-bounds read and IndexError.

Why this is wrong (concrete example)
- Suppose code_units contains a surrogate pair at indices 0 and 1 representing one codepoint, and an ASCII char at index 2.
- translate_codepoint(code_units, 0, out) handles the surrogate pair and returns consumed = 1 (or 2 — the comment is ambiguous), meaning "I handled one codepoint (which is a surrogate pair)". The code then enters the for loop range(consumed).
- On the first iteration it calls code_point_at(code_units, pos) where pos is still 0, calculates char_count(...) (2), and sets pos += 2 -> pos = 2.
- If consumed > 1 the loop will run again and call code_point_at(code_units, pos) with pos == 2. If pos == length-1 and code_point_at expects a following surrogate or tries to read ahead, it can read beyond the array and raise IndexError. Even if not out-of-bounds, this logic is redundant and fragile — you should not re-parse codepoints already handled by translate_codepoint.

Where it occurs (specific variables/lines)
- The problematic variable is pos and the problematic method calls are:
  - translate_codepoint(code_units, pos, out) which returns consumed
  - the loop body: char_count(code_point_at(code_units, pos))
- The incorrect update is pos += char_count(...), done inside a loop for pt in range(consumed)

How to fix (concrete steps)
1. Treat consumed as the number of UTF-16 code units (or number of codepoints) that the translator consumed. Do not re-parse the same positions to decide how far to advance.
2. Replace the whole for-loop with a single deterministic advance. Two safe options, depending on what consumed represents:
   - If translate_codepoint returns the number of UTF-16 code units consumed, advance by that many units:
         pos += consumed
   - If translate_codepoint returns the number of codepoints consumed (1 for one codepoint, even if it was a surrogate pair), then convert that to code units once by using char_count on the codepoint at the original pos, for example:
         first_cp = code_point_at(code_units, pos)
         pos += char_count(first_cp) * consumed
     (But usually the translator should document which it returns; prefer the first option.)
3. Remove the loop that repeatedly calls code_point_at during advancement. That loop is the source of double-counting and out-of-bounds reads.

Suggested minimal change (direct replacement)
- Replace:

      for pt in range(consumed):
          pos += char_count(code_point_at(code_units, pos))

  with (most likely correct):

      pos += consumed

Tests to run after fix
- All provided unit tests, especially:
  - test_translate_surrogate_pair_with_suffix (failing case)
  - test_translate_single_surrogate_pair
  - test_translate_surrogate_at_end
- Also fuzz mixed-surrogate/non-surrogate boundary cases.

Summary
- The bug is an incorrect index-advance after a translator consumes a codepoint. The loop that calls code_point_at while incrementing pos re-parses and miscounts code units and can read out-of-bounds. Fix by advancing pos deterministically (pos += consumed) instead of re-reading codepoints.
