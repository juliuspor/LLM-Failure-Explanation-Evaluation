What happened (root cause)
- On line 338 you set inferred_type = type(array). That returns the Python type of the list object itself (e.g. builtins.list), not the element/component type inside the list.
- ArrayUtils._copy_list_grow1 (called on line 342) expects an element/component type (so it can make a new list/array of that component type). Because you passed the list class instead of the element class, ArrayUtils receives the wrong type and creates an incorrectly typed/structured result. That leads to the later logic failing and the TypeError check on line 346 being meaningless for the actual problem.
- This also breaks the empty-list case: if array is an empty list you have no element to query, but you still set inferred_type = type(array) (list) instead of falling back to object or element type. The expected_type comparison on line 346 (inferred_type == object) will be false, so the code will not raise the intended error even though the copy/resize step was given a wrong component type.

- Line 338: inferred_type = type(array) — wrong value (list class instead of element type).
- Line 342: new_list = ArrayUtils._copy_list_grow1(array, inferred_type) — receives wrong type and produces a malformed result.
- Line 346: if inferred_type == object and expected_type != object: — this check is ineffective because inferred_type will be list, not an element type or object.

Why that causes a crash/incorrect behaviour
- ArrayUtils._copy_list_grow1 acts on the component type you supply. Passing the list class causes it to create an array/list with the wrong component metadata or allocation, which later causes either an immediate failure when assigning element into new_list[-1] (line 343) or an incorrect runtime type that makes the cast-check logic meaningless.
- The code intended to detect when only an object-typed array exists and an explicit expected_type prevents casting; but because inferred_type is wrong, that guard never triggers correctly.
