What happened
- At line 396 the code checks:
  if minutes_offset < 0 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")
- The code raised ValueError because minutes_offset was -15. So execution stopped at line 396 and never reached the later adjustments at lines 402–405 that handle negative minute offsets.

Why that is the root cause
- The later code (lines 402–405) contains logic that intends to handle negative minute offsets (it has two branches that change minutes_offset by adding or subtracting it from hours_in_minutes). That means negative minute values are valid for the overall algorithm, but the early check at line 396 forbids any negative value unconditionally. In short: the validation is too strict and happens before the code that would accept and normalize negative minutes.

Where exactly it fails
- Failure occurs at line 396 with minutes_offset == -15. Because of the condition there, you get ValueError("Minutes out of range: -15") before reaching the normalization logic at lines 402–405.

How to fix it (concrete steps)
1. Decide the valid range you want for minutes_offset:
   - If negative minute offsets should be allowed but only down to -59, then check the absolute value.
   - If negative minutes are never allowed, remove/adjust the later normalization code instead; but given lines 402–405 exist, the intended behavior is almost certainly to allow negatives.

2. Replace the check at line 396 with one that allows negative minutes within -59..59. For example change line 396 to:
   if abs(minutes_offset) > 59:
       raise ValueError(f"Minutes out of range: {minutes_offset}")

   This accepts -15 and rejects values like -60 or 60.

3. Keep the normalization logic at lines 402–405. Verify those branches handle negative values correctly (e.g. the branch that adds minutes_offset to hours_in_minutes should be used for negative minute offsets).

4. Add unit tests that exercise:
   - minutes_offset = -15 (should not raise)
   - minutes_offset = 15 (should not raise)
   - minutes_offset = -60 and 60 (should raise)

Why this fix works
- It removes the premature rejection of valid negative minute offsets while still enforcing that the absolute minute quantity is less than 60. The normalization code that follows can then run and produce the intended offset milliseconds (via FieldUtils.safe_multiply and cls.for_offset_millis).

Optional note
- If you prefer an explicit check instead of abs(): change the condition to:
   if minutes_offset < -59 or minutes_offset > 59:
       raise ValueError(...)
Either form is correct; the key is to allow negative minute values in the intended range.
