The bug occurs because translate increments pos incorrectly when translators consume codepoints. The existing loop after consumed!=0 advances pos by char_count(code_point_at(code_units, pos)) for each pt in range(consumed) but calling code_point_at may raise IndexError if pos is at end (e.g., translator consumed last codepoint and loop still tries to read). Need to advance pos by code units consumed directly: for consumed codepoints, determine number of UTF-16 code units per codepoint and advance accordingly without calling code_point_at at out-of-range positions. Simpler: for pt in range(consumed): if pos < length: advance by char_count(code_point_at(code_units, pos)); else break. But calling code_point_at can still raise if pos==length; so check pos < length before calling. Implement safer logic: for _ in range(consumed): if pos >= length: break; cp = code_point_at(code_units, pos); pos += char_count(cp). Also ensure translators returning consumed value reflect codepoints consumed. That fixes IndexError.
