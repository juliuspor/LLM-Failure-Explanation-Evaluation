What happened (where it failed)
- The code enters the integer branch at line 176 because for x = -0.0, the expression x == int(x) is True (int(-0.0) == 0).
- At line 177 you do value = int(x). That converts -0.0 to the integer 0 and loses the negative sign.
- Later at line 192 you call self.add(str(value)), which produces "0". The test expected "-0.0" so the assertion fails with "Expected -0.0, got 0".

Why it happens (root cause)
- Floating negative zero is a distinct value from positive zero but compares equal to 0.0 and to the integer 0. Converting it with int(x) discards the sign bit.
- The code treats any float equal to an integer as an integer and formats using value, so negative-zero floats get normalized to plain "0" instead of preserving "-0.0".

Exact locations to change
- Line 176: the branch condition allows -0.0 through.  
- Line 177: value = int(x) is where the sign is lost.  
- Line 192: self.add(str(value)) is what emits "0" instead of "-0.0".

Concrete fixes (step-by-step)
Option A — preserve negative zero by early return
1. Add a check right after entering the integer branch (immediately after line 176):
   - if x == 0.0 and math.copysign(1.0, x) < 0:
       self.add(str(x))
       return
2. This keeps "-0.0" formatted by the existing floating formatting (str(x)) and skips the integer conversion.

Option B — preserve sign when value == 0
1. After value = int(x) (line 177), before emitting, check:
   - if value == 0 and math.copysign(1.0, x) < 0:
       self.add(str(x))
       return
2. This only special-cases the zero result and uses the original float string to preserve "-0.0".

Option C — detect negative zero via sign bit
1. If you prefer a lower-level check, use struct.unpack('!Q', struct.pack('!d', x))[0] & (1<<63) to test the sign bit. But math.copysign(1.0, x) < 0 is simpler and portable.

Recommended quick patch (minimal change)
Insert after line 176:
- if x == 0.0 and math.copysign(1.0, x) < 0:
    self.add(str(x))
    return

How to test
- Add a unit test: x = -0.0 -> expect "-0.0"
- Also test x = 0.0 -> expect "0" (unchanged)
- Test other integer-like floats (e.g., 42.0, -42.0) to ensure behavior is unchanged.

That is the direct cause and concrete fixes; the simplest safe fix is to special-case signed zero before you convert to int so the negative sign is preserved.
