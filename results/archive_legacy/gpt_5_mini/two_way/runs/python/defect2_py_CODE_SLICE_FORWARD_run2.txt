What happened
- You clamp the input into variable v:
  - v = max(value, self._lower_bound)
  - v = min(v, self._upper_bound)
- But when you compute g you use value instead of v:
  - g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

Because you used value, the clamping is ignored. If value is outside the bounds the computed g can be < 0 or > 255, which triggers the ValueError check a few lines later.

Where it occurs
- The bug is on the line that assigns g (the one using (value - self._lower_bound) ...). The two prior lines that assign v perform the clamp but v is never used.

Why it fails (concrete example)
- Suppose self._lower_bound = 0.0, self._upper_bound = 1.0 and value = 2.0.
  - v becomes 1.0 (clamped)
  - g is computed from value = 2.0, so g = int((2.0 - 0.0) / 1.0 * 255) = 510
  - g > 255, so ValueError is raised.

Exact fixes (replace these lines)
1) Use the clamped v when computing g:
   - Replace
     g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
     with
     g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

2) Also guard against a zero-width range to avoid division by zero:
   - If self._upper_bound == self._lower_bound, decide on behavior (e.g., return 0 or 255 or midpoint). Example:
     if self._upper_bound == self._lower_bound:
         g = 0  # or 127, or raise a clearer error
     else:
         g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

3) Optionally ensure g is clamped to [0,255] after conversion:
   - g = max(0, min(255, g))

Summary of minimal change
- Change the g computation to use v instead of value and add a zero-division guard. This fixes the mismatch between the intended clamp and the value used for scaling.
