What happened
- The test passes None inside the replacements array (e.g. ["a"], [None]).
- Inside StringUtils.replace_each the code calls len(...) on replacement values (something like len(repl) or len(replacements[i])) without checking whether the replacement is None first.
- Calling len(None) raises TypeError: object of type 'NoneType' has no len(), which is exactly the test failure you're seeing.

Why it fails (root cause)
- The code assumes every element of the replacements list is a string and uses its length (for index math or building the result) before verifying it is not None.
- The tests expect a None replacement to be treated as a no-op (Java tests in the suite use null to mean "leave the original text alone" in these cases). The current implementation does not guard against None and immediately calls len(replacement), causing the TypeError.

Where to look in your code
- Method: StringUtils.replace_each
- Variables to inspect: the replacements parameter (often named replacements or replace_list) and any local variable named repl, replacement, or repl_len.
- The failing call is the one that does len(repl) (or len(replacements[i])) without a prior None check.

How to fix it (concrete steps)
1. Find the loop that iterates through the pairs (search_list and replacements) and that computes the replacement length (or otherwise uses len on the replacement).
2. Add a None check before calling len() or using the replacement string. Example logic to implement:
   - If replacements[i] is None, treat it as a "no-op" and skip replacing that search token.
   - Otherwise use the replacement string as before.
3. Example patch (pseudo-Python showing the exact pattern to change):

   Before (problematic):
   for i, search in enumerate(search_list):
       repl = replacements[i]
       repl_len = len(repl)        # <-- TypeError when repl is None
       # ... use repl_len ...

   After (fixed):

What the tests expect

Summary
