What happened (what failed)
- The code throws IndexError from slice_backward complaining "String index out of range: end" when calling substring_java. That means substring_java was called with end > len(text).
- The call that causes it is substring_java(text, 0, upper) (or less commonly substring_java(text, 0, index)). Those calls pass an end index that can exceed the string length.

Why it happened (root cause)
- You clamp upper to len(text) early:
  if upper == -1 or upper > len(text):
      upper = len(text)
- But later you do:
  if upper < lower:
      upper = lower
  This can increase upper above len(text) if lower > len(text). The code never checks or clamps lower against len(text).
- So when lower > len(text) (a valid input possibility), upper is set to that larger lower value, and you then call substring_java(text, 0, upper) with upper > len(text) -> IndexError thrown by the substring function because end is out of range.

Where it occurred
- The problematic flow:
  1. upper initially <= len(text) so not clamped to len(text).
  2. lower > len(text).
  3. upper < lower true → upper = lower (now > len(text)).
  4. substring_java(text, 0, upper) is executed.
- The exception originates in slice_backward (the substring implementation) at its start/end validation and is raised there (lines shown in your slice_backward snippet).

How to fix it (concrete steps)
Option A — clamp lower to the string length (simple and safe)
- Before using lower, add: lower = max(0, min(lower, len(text)))
- Keep the rest of your logic. This prevents lower from exceeding len(text) and therefore prevents upper from being set > len(text).

Option B — clamp upper again after the upper<lower assignment (minimal change)
- After you set upper = lower, immediately clamp:
    if upper > len(text):
        upper = len(text)
- Also ensure lower is non-negative if that’s a requirement:
    if lower < 0:
        lower = 0

Option C — validate inputs and raise a clear error
- If passing a lower beyond the string should be considered a caller error, check:
    if lower < 0 or lower > len(text):
        raise ValueError(f"lower must be between 0 and {len(text)}: {lower}")
  This fails fast with a clear message rather than causing an IndexError inside substring_java.

Which I recommend
- If you want robust behavior that truncates rather than errors, use Option A (clamp lower).
- If out-of-range lower should be invalid, use Option C and raise a clear ValueError.
Either way, the key fix is to ensure lower and upper are both clamped to the valid 0..len(text) range before calling substring_java(text, 0, end).
