What happened
- On these lines you clamp the input into v:
  - v = max(value, self._lower_bound)
  - v = min(v, self._upper_bound)
- But on the next line you compute g using the original value, not the clamped v:
  - g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

Because you used value instead of v, a value outside the bounds will produce g < 0 or g > 255, which triggers the ValueError you raise a few lines later. Concretely, passing value < self._lower_bound or value > self._upper_bound causes the calculation to be outside the expected 0–255 range even though you intended to clamp it.

Two additional issues to watch for
1. Zero division: if self._upper_bound == self._lower_bound, the denominator (self._upper_bound - self._lower_bound) is zero and the code raises ZeroDivisionError.
2. The error message in the raise line is unhelpful:
   - raise ValueError(f"Color parameter outside of expected range: Red Green Blue")
   It’s a static string and doesn’t include the offending value or g.

How to fix it (step‑by‑step)
1. Use the clamped value v when computing g:
   - g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
2. Guard against zero span before dividing:
   - denom = self._upper_bound - self._lower_bound
   - if denom == 0: raise ValueError("upper_bound and lower_bound must differ")
3. Improve the ValueError message to include the computed g and the input value for easier debugging.

Corrected code example
    def get_paint(self, value: float) -> Tuple[int, int, int]:
        # Clamp value to bounds
        v = max(value, self._lower_bound)
        v = min(v, self._upper_bound)
        
        denom = self._upper_bound - self._lower_bound
        if denom == 0:
            raise ValueError("upper_bound and lower_bound must not be equal")
        
        # Use the clamped v (not the original value)
        g = int((v - self._lower_bound) / denom * 255.0)
        
        if g < 0 or g > 255:
            raise ValueError(f"Computed color component out of range: g={g}, value={value}")
        
        return (g, g, g)

Summary
- Root cause: g is computed from value instead of the clamped v (line with g = int(...)).
- Secondary failure: possible division by zero when bounds are equal.
- Fix: compute g from v and add a denom check; improve the error message.
