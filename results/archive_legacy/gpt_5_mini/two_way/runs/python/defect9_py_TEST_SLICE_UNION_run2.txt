What failed
- The tests pass replacement_list entries that are None (e.g. ["c", None] or [None]).
- In replace_each, the code assumes every replacement_list[i] is a string and calls len(replacement_list[i]) without checking for None.
- That causes a TypeError: object of type 'NoneType' has no len().

Where it happens (exact location)
- The failing code is in the loop starting at line 81:
  - Line 82: greater = len(replacement_list[i]) - len(search_list[i])
  - If replacement_list[i] is None, len(replacement_list[i]) raises the error.
- Later code (lines 102–105) does check for replacement_list[i] is None before searching:
  - That check is too late — the code already crashed at line 82.

Why this happens (root cause)
- The function tries to estimate how much the output string may grow by precomputing an "increase" (lines 81–85).
- That estimation loop unconditionally indexes and calls len() on replacement_list entries.
- The intended null-safety checks happen only later when scanning the text (lines 102–105), so None values are allowed by tests but the early estimation step can't handle them.

How to fix (concrete steps)
1. Skip None (and empty) replacements when computing the preallocation/estimate:
   - Change the loop at lines 81–85 to check for None and empty strings before calling len().
   - Example replacement for that loop:
     for i in range(search_length):
         if search_list[i] is None or len(search_list[i]) == 0:
             continue
         if replacement_list[i] is None:
             # treat None as no replacement -> no growth
             continue
         greater = len(replacement_list[i]) - len(search_list[i])
         if greater > 0:
             increase += 3 * greater
2. Alternatively, treat None as "" for sizing:
   - Use replacement_len = len(replacement_list[i]) if replacement_list[i] is not None else 0
   - Then compute greater = replacement_len - len(search_list[i])
3. Ensure any other place that assumes replacement_list[i] is a string also checks for None before calling string methods.

Why this fix works
- It prevents calling len() on None by short-circuiting when replacement_list[i] is None.
- It makes the estimation step consistent with later guards (lines 102–105) that already avoid processing None replacements.

Summary
- Root cause: len(replacement_list[i]) is called without guarding against replacement_list[i] being None (lines 81–85).
- Fix: add a check for replacement_list[i] is None (or treat None as empty) before calling len(), so the early size-estimate loop matches the later null-safety checks.
