What happened (root cause)
- In the "max middle" block you fetch the period for the wrong index:
  - Code uses self.get_data_item(self._min_middle_index).get_period() inside the if self._max_middle_index >= 0: branch.
  - That is a clear typo: the branch intends to examine the currently recorded max-middle item, but it actually reads the min-middle item.

Why that causes a failure (where and why)
- Location: inside the block beginning at
  if self._max_middle_index >= 0:
- Variable names involved: self._max_middle_index (checked), self._min_middle_index (used).
- Consequences:
  - If _min_middle_index != _max_middle_index you will compare the new period's middle against the wrong existing middle value, so _max_middle_index may never be updated when it should be, and the maintained extremes become incorrect.
  - If _min_middle_index is -1 while _max_middle_index >= 0, you call get_data_item(-1).get_period(). Depending on get_data_item, that may return the last element (unexpected) or raise an IndexError — causing a crash. That explains crashes or incorrect behavior.

Exact fix (step-by-step)
1. Replace the wrong index reference with the correct one: use self._max_middle_index when fetching the existing max-middle period.
2. Add a unit test that inserts at least two periods so both min and max middle branches are exercised and verify indices are updated correctly.
3. (Optional hardening) Validate that _*_index values are within bounds before calling get_data_item, or ensure get_data_item raises a clear error for invalid indices.

Corrected code snippet (only the changed lines)
Replace this block:
    if self._max_middle_index >= 0:
        max_middle_period = self.get_data_item(self._min_middle_index).get_period()
        ...
with:
    if self._max_middle_index >= 0:
        max_middle_period = self.get_data_item(self._max_middle_index).get_period()
        ...

Why this fix is sufficient
- It uses the value you actually checked (_max_middle_index) to retrieve the current "max middle" period, so comparisons are done against the right item and you avoid the out-of-bounds/wrong-item behavior described above.

Suggested tests to catch regressions
