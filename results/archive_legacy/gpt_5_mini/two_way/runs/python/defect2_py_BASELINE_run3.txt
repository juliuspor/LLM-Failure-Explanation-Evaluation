What happened (root cause)
- In get_paint you clamp the input into v on lines 55–57:
  - v = max(value, self._lower_bound)
  - v = min(v, self._upper_bound)
- But on line 71 you compute g using the original value, not the clamped v:
  - g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- So when value is outside the bounds (e.g. -0.5 or 1.5), the un-clamped expression yields a g < 0 or g > 255 and triggers the ValueError at line 74:
  - if g < 0 or g > 255: raise ValueError(...)

Why it failed for your tests
- test_lower_bound_clamping passes value -0.5; although v becomes 0.0 after clamping, g is computed from value -0.5 -> negative g -> exception.
- test_upper_bound_clamping passes 1.5; value 1.5 produces g > 255 -> exception.
- The code intended to clamp but never used the clamped variable v when computing g.

Where to change (concrete fixes)
1) Use the clamped v when computing g (recommended):
   - Replace line 71 with:
     g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

2) Also guard against zero range (lower == upper) to avoid division by zero. For example, before computing g:
   if self._upper_bound == self._lower_bound:
       g = 0  # or 255, or a defined behavior; choose sensible default
   else:
       g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

3) Optionally clamp the computed g to [0,255] instead of raising:
   g = max(0, min(255, g))

Minimal patch (apply in get_paint):
- Replace the computation and validation with:
    if self._upper_bound == self._lower_bound:
        g = 0
    else:
        g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
    g = max(0, min(255, g))

Summary
- The bug is using value instead of the clamped v when computing g (line 71), causing out-of-range g and the ValueError raised at line 74. Fix by computing g from v and handle the equal-bounds case.
