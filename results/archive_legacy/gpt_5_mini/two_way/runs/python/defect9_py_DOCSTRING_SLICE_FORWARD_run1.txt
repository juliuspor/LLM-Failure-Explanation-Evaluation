What happened (root cause)
- At lines 102–105 the code checks whether a candidate replacement should be ignored:
  no_more_matches_for_repl_index[i]
  or search_list[i] is None
  or len(search_list[i]) == 0
  or replacement_list[i] is None
- But that check does not actually skip the candidate because there is no continue/else branch after it. The next statement (line 108) always runs:
  temp_index = text.find(search_list[i], start)
- When search_list[i] is None (or otherwise invalid), text.find(search_list[i], start) raises TypeError. That is exactly why the function crashes when either list contains None even though you intended to ignore those entries.

Where it crashes
- The immediate crash happens at line 108 (text.find(search_list[i], start)) when search_list[i] is None.
- A secondary crash can happen later at line 94/line 115 when code uses len(search_list[replace_index]) or otherwise assumes replace_index points to a valid string; those assumptions also rely on the guard actually skipping None entries.

Why the existing guard fails
- The condition at 102–105 correctly identifies entries to ignore, but it does nothing because the result of the condition isn’t acted on (no continue/else). In short: the guard is present but not applied.

How to fix (concrete steps)
1. Make the guard skip the candidate by adding a continue:
   if (no_more_matches_for_repl_index[i]
       or search_list[i] is None
       or len(search_list[i]) == 0
       or replacement_list[i] is None):
       continue
   temp_index = text.find(search_list[i], start)
   This ensures find() is never called with search_list[i] == None.

2. Add defensive checks before you use replace_index later:
   - Before parts.append(replacement_list[replace_index]) (line 92), ensure replacement_list[replace_index] is not None.
   - Before start = text_index + len(search_list[replace_index]) (line 94), ensure search_list[replace_index] is a string (not None) and has length.

3. (Optional) Prefer clearer structure: compute a boolean skip = (no_more_matches... or search_list[i] is None ...), if skip: continue; else do find. That avoids evaluating len() on None because search_list[i] is checked first.

Why this fix is safe
- The fix restores the intended behavior: entries with search_list or replacement_list equal to None are ignored, so find() and len() are only called on real strings. It also prevents replace_index from being set to a None entry.

Summary
The code detects entries that should be ignored but never actually skips them. Because of that, text.find(...) is called with None (line 108) and len(None) may be called later (line 94), causing the crash. Add a continue (or otherwise branch the logic) immediately after the condition at lines 102–105 and add defensive checks before using replace_index to fix the bug.
