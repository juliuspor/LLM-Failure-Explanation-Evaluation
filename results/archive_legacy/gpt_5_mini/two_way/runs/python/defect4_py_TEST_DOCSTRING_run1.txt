What happened (short): _update_bounds updates the stored index for the maximum middle value using the wrong variable. When the code finds a new maximum "middle" it sets max_middle_index from a loop/local index (or some other variable) instead of using the function argument index. That causes the wrong index to be remembered (e.g. returning 3 instead of 1 in your test).

Where it occurs: inside TimePeriodValues._update_bounds (the method under test). The relevant variables are:
- period — the TimePeriod passed into _update_bounds
- index — the index argument that identifies the period being added
- middle (or mid) — the computed middle time of the period
- max_middle (or maxMiddle) — the currently-known maximum middle value
- max_middle_index (or maxMiddleIndex) — the stored index corresponding to max_middle

Why it fails (root cause):
- _update_bounds computes the middle for the new period and compares it to max_middle.
- When the new middle is larger, the code correctly updates max_middle, but assigns max_middle_index from the wrong variable — typically a loop variable (often named i) or some other local like lastIndex — instead of using the index argument.
- As a result, max_middle_index does not point to the newly-added period but to whatever that wrong variable held (for example the current size-1, a different loop position, or a stale index). That explains test_max_middle_index_simple: the 150 middle at index=1 is recognized as the maximum middle value, but max_middle_index was set to 3 (the wrong source variable) and so get_max_middle_index() returns 3 instead of 1.

Concrete example of the incorrect pattern (likely present in your file):
- Incorrect (pseudo-code):

      self.max_middle = middle
      self.max_middle_index = i     # WRONG: i is not the 'index' argument

Correct change (what to change, exact assignment):

      self.max_middle_index = index  # use the function argument 'index'
