What happened
- The code crashes when either search_list or replacement_list contains None because this loop tries to call len(...) on None:
  for i in range(search_length):
      greater = len(replacement_list[i]) - len(search_list[i])
- len(None) raises TypeError: "object of type 'NoneType' has no len()", so the function fails before it reaches the main replacement loop.

Where it occurs
- The failing code is the "increase" calculation block immediately after the initial match-finding and the comment "# BUG: this loop is not null-safe for elements inside the arrays".
- Specifically the offending expression is len(replacement_list[i]) - len(search_list[i]) (the first line inside that for loop).

Why it happens (root cause)
- Earlier in the function you correctly skip None/empty entries when searching for matches (you check search_list[i] is None, len(search_list[i]) == 0, and replacement_list[i] is None).
- But the "increase" loop does not apply the same checks. It assumes every search_list[i] and replacement_list[i] is a non-None string and directly calls len(...) on them.
- If either element is None (or one side is None and the other a string), len(None) is invalid, causing the crash. This is why the function crashes even though those replacements were intended to be ignored.

How to fix it (clear, actionable steps)
1. Use the same skip conditions you already use elsewhere before calling len():
   - If no_more_matches_for_repl_index[i] is True, or search_list[i] is None, or len(search_list[i]) == 0, or replacement_list[i] is None, then skip this i.
2. Replace the "increase" loop with a null-safe version. Example patch:

    increase = 0
    for i in range(search_length):
        if (
            no_more_matches_for_repl_index[i]
            or search_list[i] is None
            or len(search_list[i]) == 0
            or replacement_list[i] is None
        ):
            continue
        greater = len(replacement_list[i]) - len(search_list[i])
        if greater > 0:
            increase += 3 * greater

3. Alternatively, filter the input arrays up-front into pairs you actually plan to use (remove pairs where search is None/empty or replacement is None). Then use the filtered lists everywhere; this reduces duplicated checks and makes the code safer and clearer.

Why these fixes work
- They prevent calling len(...) on None by ensuring only valid string pairs reach the length calculation.
- They make the "increase" calculation consistent with the match-finding logic, avoiding the TypeError and preserving the intended behavior of ignoring None entries.
