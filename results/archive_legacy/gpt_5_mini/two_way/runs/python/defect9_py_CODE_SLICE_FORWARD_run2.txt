What happened (root cause)
- The loop that computes "increase" (lines 83–85) calls len(replacement_list[i]) and len(search_list[i]) without checking for None. If either array element is None, len(None) raises TypeError: "'NoneType' object has no len()".
- That means the function can fail before it ever does any replacements, when you pass search_list or replacement_list that contain None entries.

Where it occurs
- Exact failing expression: greater = len(replacement_list[i]) - len(search_list[i]) (line 83 in your slice).
- The earlier match-finding loop (lines ~30–50) correctly guards against None and empty strings, but the "increase" loop does not. So the None check is missing only in this size-estimate loop.

What the exception looks like
- Typical stack/exception you would see: TypeError: object of type 'NoneType' has no len(), raised at line 83 (the len(...) call).

Why this is the problem
- The function allows None inside the lists (it skips them later when searching). But this particular loop assumes every element is a str and calls len() unconditionally. That assumption is false, so the code crashes.

How to fix it (concrete steps)
1. In the "increase" loop, skip entries where either search_list[i] or replacement_list[i] is None or where search_list[i] is empty. Example replacement for lines 79–85:

    for i in range(search_length):
        s = search_list[i]
        r = replacement_list[i]
        if s is None or r is None or len(s) == 0:
            continue
        greater = len(r) - len(s)
        if greater > 0:
            increase += 3 * greater  # assume 3 matches

2. Alternatively, if you prefer treating None as empty strings, use:

    s_len = len(search_list[i]) if search_list[i] is not None else 0
    r_len = len(replacement_list[i]) if replacement_list[i] is not None else 0
    greater = r_len - s_len

but skipping None is closer to how the rest of the function behaves.

3. After making that change, remove the comment "BUG: this loop is not null-safe..." (or update it to reflect the fix).

Why that fix is sufficient
- It mirrors the guards used in the later matching loops, preventing len(None) calls.
- The rest of the function already skips None/empty entries when searching and when appending replacements, so this makes the size estimate code consistent and avoids the TypeError.

Other notes
- The later uses of replacement_list[replace_index] and search_list[replace_index] are safe because replace_index is selected only from indices that passed the None/empty checks in the matching loop.
- If you rely on the "increase" heuristic, skipping None entries preserves the intended behavior.
