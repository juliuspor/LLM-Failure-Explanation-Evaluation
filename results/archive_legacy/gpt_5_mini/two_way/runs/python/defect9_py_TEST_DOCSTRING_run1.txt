What happened
- The tests call StringUtils.replace_each(text, search_list, replacement_list) with some entries in replacement_list equal to None (Python None corresponds to Java null).
- The implementation of StringUtils.replace_each does not check for None before using the replacement value. It treats each replacement entry as if it were a string and uses it in string operations (concatenation, replace/find/index, or similar).
- When the code reaches a None replacement it tries to use it as a string and Python raises an error (for example TypeError: can’t concatenate str and NoneType or AttributeError: 'NoneType' object has no attribute 'find'). That is why the tests fail instead of returning the original or partially replaced string.

Where it occurs (variables and method)
- Inside StringUtils.replace_each, the code loops over the provided lists (the arguments named search_list and replacement_list).
- During that loop the implementation takes replacement = replacement_list[i] (and likely search = search_list[i]) and then immediately uses replacement in string operations without checking if replacement is None.
- The failing tests are:
  - test_null_safety_inside_arrays_single: replacement_list = [None] — expected to ignore the replacement and return "aba", but the code errors when using replacement.
  - test_null_safety_inside_arrays_mixed: replacement_list = ["c", None] — expected to apply "a"→"c" and ignore "b"→None, but code errors when it reaches the None entry.

Why this is the root cause
- The intended behavior (from the docstring and the Java original) is: if either the search entry or the replacement entry is null/None, that pair should be ignored.
- The implementation violates that contract because it never skips pairs with None; it directly passes the replacement into string operations, causing an exception. That is the direct root cause.

Concrete steps to fix it
1. In StringUtils.replace_each, before using search_list[i] or replacement_list[i], explicitly check for None and skip that pair:
   - if search_list[i] is None: continue
   - if replacement_list[i] is None: continue
2. Also handle empty search strings if the original implementation needs to ignore them:
   - if search_list[i] == "": continue
3. Make sure any code that builds a new string never tries to concatenate or call string methods on a None value.
4. Add unit tests (the three you already have) to verify:
   - single None in replacements is ignored (returns original string),
   - mixed None entries are ignored for those pairs only,
   - normal replacements still work.

Minimal example of the change (pseudo-Python within the function loop)
- for i in range(len(search_list)):
    s = search_list[i]
    r = replacement_list[i]
    if s is None or r is None:
        continue
    # proceed using s and r as strings

Why this will fix the failure
- Skipping any pair where either the search or replacement is None prevents treating None as a string and thus prevents the TypeError/AttributeError. That matches the documented/expected behavior and will make your tests pass.
