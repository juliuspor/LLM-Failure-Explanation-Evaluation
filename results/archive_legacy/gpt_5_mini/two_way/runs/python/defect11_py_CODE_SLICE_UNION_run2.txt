What goes wrong
- Root cause: when callers pass an end value that is less than or equal to start, gap = end - start becomes <= 0 and _RANDOM.randrange(gap) is called with a non‑positive bound. That triggers the ValueError("bound must be positive") at the check on line 52 and/or inside randrange when reached.
- Where it happens: gap is computed on line 49, then the code checks gap <= 0 and raises ValueError("bound must be positive") on line 52. The actual randrange call that would fail is on line 56: _RANDOM.randrange(gap).
- Why it fails for callers: the function never validates the relationship between the start and end parameters in a descriptive way. A caller who passes start >= end gets an exception with the unhelpful message "bound must be positive" instead of a clear explanation that end must be greater than start.

Concrete steps to reproduce
- Call random(5, start=100, end=50, letters=False, numbers=False). gap = -50 -> raises ValueError("bound must be positive") at line 52 (or at randrange on line 56 if the explicit check were removed).

Precise fixes (apply any one)
1) Validate and give a clear message
   Replace the current check at line 52 with:
     if end <= start:
         raise ValueError(f"end ({end}) must be greater than start ({start})")
   This detects the condition earlier and tells the caller exactly what is wrong.

2) Alternatively, accept swapped inputs (if you want to be lenient)
   Before computing gap, normalize:
     if end < start:
         start, end = end, start
   Then proceed. (Decide whether this matches your API expectations.)

3) Also consider additional validation (optional but recommended)
   - Ensure start and end are in valid Unicode codepoint range if you intend to call chr(start + ...):
       if not (0 <= start <= 0x10FFFF) or not (0 <= end <= 0x10FFFF):
           raise ValueError("start and end must be valid Unicode code points")
   - Make the error message explicit rather than reusing Random#nextInt wording.

Why this advice fixes the problem
- The explicit check prevents calling randrange() with a non-positive bound and gives a clear, actionable message that end must be greater than start (or swaps values if you choose normalization). This removes the confusing "bound must be positive" symptom and points the developer to the real input mistake.
