What happened
- The tests pass a None inside the replacement arrays (e.g. ["a"], [None] and ["a","b"], ["c", None]).
- Inside StringUtils.replace_each your code calls len(...) on elements of the search/replacement arrays without checking whether those elements are None.
- When the loop reaches a None element and executes len(search) (or len(replacement)), Python raises TypeError: object of type 'NoneType' has no len().

Where it occurs
- In StringUtils.replace_each, in the loop over the search/replacement lists. The error comes from a line like:
  len(search)  # or len(replacement)
  where search (or replacement) is None for the test cases.

Why that causes the failing tests
- The Java tests you’re porting expect null/None in the replacement array to be treated as "no operation" (skip that pair), not to crash. Because you call len(...) without a None check, the function raises an exception instead of returning the unchanged string.

How to fix it (concrete steps)
1. Locate the loop in replace_each that iterates the search and replacement arrays. It will look something like:
   for i in range(len(search_list)):
       search = search_list[i]
       repl = replacement_list[i]
       if len(search) == 0: ...
       # or other code that immediately uses len(search) or len(repl)

2. Add explicit None checks before any len(...) or other operations that assume a string. If either the search or the replacement is None, skip that pair (do not attempt len()):
   for i in range(len(search_list)):
       search = search_list[i]
       repl = replacement_list[i]
       if search is None or repl is None:
           continue
       if len(search) == 0:
           continue
       # existing replacement logic

   This implements the expected "null-safety": a None replacement acts as a no-op.

3. If your logic treats an empty search string specially, keep that handling (the code above preserves that).

Minimal code patch example
- Replace the part that reads/uses search and repl with:

   for i in range(len(search_list)):
       search = search_list[i]
       repl = replacement_list[i]
       # Skip any pair containing None (null-safe behavior)
       if search is None or repl is None:
           continue
       # Now it's safe to call len(search) or use search/repl as strings
       if len(search) == 0:
           continue
       # (rest of replacement logic)

Why this exact fix
- It prevents calling len(None) which causes the TypeError.
- It matches the Java behavior in the tests: a null replacement should not change the input string (i.e. be treated as a skipped mapping).
- It’s minimal and keeps existing behavior for non-None elements.
