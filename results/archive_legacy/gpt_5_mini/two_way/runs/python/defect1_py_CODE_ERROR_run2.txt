What happened (where it failed)
- The method raises ValueError at the minutes validation:
  if minutes_offset < 0 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset})
  The error message in the test shows "Minutes out of range: -15", so minutes_offset was -15 when that check ran. That is the immediate failure.

Why it happens (root cause)
- The method currently requires minutes_offset to be in the range 0..59. That rejects negative minute values, but the API (and the rest of the method) expect callers to be able to pass a negative minutes value when requesting a negative offset (for example hours_offset = -1, minutes_offset = -15 to mean -1 hour 15 minutes). In short: the validation happens too strictly/too early and does not allow negative minute input.
- After that check the code tries to combine hours and minutes into a single minutes count with:
    hours_in_minutes = hours_offset * 60
    if hours_in_minutes < 0:
        minutes_offset = hours_in_minutes - minutes_offset
    else:
        minutes_offset = hours_in_minutes + minutes_offset
  That code assumes minutes_offset can be negative for negative hours (it subtracts minutes_offset for negative hours). But because of the earlier check, negative minutes are rejected before this combination happens.

Where in the code to look
- The failing check is the third if in your snippet:
    if minutes_offset < 0 or minutes_offset > 59:
        raise ValueError(f"Minutes out of range: {minutes_offset}")
  That check is executed before you combine hours and minutes into total minutes.

1. Allow minutes_offset to be negative (range -59..59) so callers can pass -15.
2. Keep the combination logic the same.

Change the validation to:

        raise ValueError(f"Minutes out of range: {minutes_offset}")

Option B — require minutes be non-negative and adjust validation/combination order:
1. Keep minutes_offset required to be 0..59 for input.
2. If you want to allow callers to pass a negative minutes to mean negative fractional hour, document that API or normalize earlier.
3. Alternatively move the validation so you validate the absolute minutes after combining hours and minutes into total minutes.
