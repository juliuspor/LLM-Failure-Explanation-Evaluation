- inferred_type becomes object in the common case where the code cannot determine a more specific element type. That happens, for example, when:
  - array is None and element is None, or
  - array has mixed types, or
  - the code intentionally uses object as a fallback inference.
- The check at line 346 does not consider whether the caller actually provided an element or whether the caller provided expected_type to assert a type. It simply assumes that an object-typed list must never be converted to a more specific list and throws immediately.
- In the documented behavior you want: "If the input list is None, a new one element list is returned whose component type is the same as the element, unless the element itself is None, in which case the return type is List[object]."

- The error message also suggests a Java-style ClassCastException and "[Ljava.lang.Object;" which indicates the check was translated from Java semantics, but in this implementation it prevents valid cases where expected_type should be used or element is None.

- The failure occurs at the check on inferred_type and expected_type shown at lines 346–351. The variables involved are:
