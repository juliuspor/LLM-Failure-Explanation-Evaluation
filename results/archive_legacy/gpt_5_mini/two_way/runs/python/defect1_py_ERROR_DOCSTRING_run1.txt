What happened
- You called the offset factory with hours_offset = 2 and minutes_offset = -15 (or otherwise passed a negative minutes value when the hour argument is positive).
- The factory method performs input validation and rejects the (+hours, -minutes) combination. That triggers the ValueError with the message "Minutes out of range: -15".

Why it failed (root cause)
- The docstring documents the rule: when hours is positive, minutes must be zero or positive. A pair like (2, -15) is explicitly disallowed.
- The code that throws the exception is the method that constructs the zone from hours and minutes (the factory; in Joda-Time this is DateTimeZone.forOffsetHoursMinutes(int hours, int minutes)). During parameter checks it tests:
  - that minutes_offset is between -59 and +59, and
  - that minutes_offset is not negative when hours_offset > 0.
  Passing minutes_offset = -15 fails that second test, so the method raises ValueError("Minutes out of range: -15").

Where it occurred
- In the offset factory method (the method documented by the docstring). The exception originates from the initial argument-validation block inside that method before any offset arithmetic is attempted.

How to fix it (concrete steps)
1. If you intended a positive 2 hour 15 minute offset, pass minutes_offset = 15:
   - Example: DateTimeZone.forOffsetHoursMinutes(2, 15)
2. If you intended a negative 2 hour 15 minute offset, make hours negative and minutes positive:
   - Example: DateTimeZone.forOffsetHoursMinutes(-2, 15)  // yields -02:15
3. If you intended a 2 hour minus-15 minutes effect (nonstandard), convert to an equivalent single signed offset in minutes and pass appropriately:
   - Compute totalMinutes = hours_offset * 60 + minutes_offset, then derive signed hours/minutes so minutes has the same sign as hours. For (2, -15):
     - totalMinutes = 2*60 + (-15) = 105 minutes = +1 hour 45 minutes
     - Call DateTimeZone.forOffsetHoursMinutes(1, 45)
4. Alternatively, if your API accepts a single-minute offset, use that form (if available) to avoid sign-rule issues.

Why not just pass (2, -15)
- The API explicitly forbids mixing a positive hours value with a negative minutes value; it's a validation rule, not a bug in the exception.
