What happened
- Given val = "l" (or "L"), the code enters the branch for qualifiers:
  - last_char = val[-1] -> 'l'
  - numeric = val[:-1] -> "" (empty string)
- Inside the L-qualifier branch the code evaluates:
  (char_at(numeric, 0) == "-" and numeric[1:].isdigit()) or numeric.isdigit()
  The expression calls char_at(numeric, 0) on an empty string.
- Calling char_at (or indexing numeric[0]) on an empty string raises IndexError ("String index out of range: 0") instead of raising the intended ValueError.

Why it fails (root cause)
- The code assumes numeric has at least one character before it checks numeric[0] / numeric[1:].
- There is no guard for the case len(val) == 1 (so numeric == "").
- Evaluation of char_at(numeric, 0) happens before numeric.isdigit() in the combined boolean expression, so the out-of-range access occurs first and crashes the function.

Where it occurs in the snippet
- In the block:
    if last_char in ("l", "L"):
        # BUG: missing guard for len(val) == 1 and non-digit input.
        if (char_at(numeric, 0) == "-" and numeric[1:].isdigit()) or numeric.isdigit():
            return int(numeric)
        raise ValueError(f"{val} is not a valid number.")
  The IndexError is raised by char_at(numeric, 0) when numeric == "" (i.e. val length is 1).

Concrete fixes (step-by-step)
Option A — explicit empty check (clear, minimal change)
1. Add a guard before accessing numeric[0] or numeric[1:]:
    if last_char in ("l", "L"):
        if len(numeric) == 0:
            raise ValueError(f"{val} is not a valid number.")
        if (char_at(numeric, 0) == "-" and numeric[1:].isdigit()) or numeric.isdigit():
            return int(numeric)
        raise ValueError(f"{val} is not a valid number.")
2. This prevents calling char_at on an empty string and yields the intended ValueError for input "l" / "L".

Option B — avoid indexing by using safe string methods (preferred)
1. Replace the inner condition with methods that are safe on empty strings:
    if last_char in ("l", "L"):
        if numeric.isdigit() or (numeric.startswith("-") and numeric[1:].isdigit()):
            return int(numeric)
        raise ValueError(f"{val} is not a valid number.")
2. numeric.startswith("-") returns False for empty numeric, so numeric[1:] is only evaluated when numeric actually starts with "-" and therefore has at least one character; this avoids IndexError and is clearer.

Testing to add
- Unit tests that cover:
  - val = "l" and val = "L" (should raise ValueError)
  - val = "5l", val = "-5l" (should return int(5) and int(-5))
  - val = "" (already raises ValueError)
  - val = None (returns None)
- Add an assertion or test that the function never raises IndexError for any one-character non-digit input.

Summary
The crash is caused by indexing numeric[0] (char_at(numeric, 0)) when numeric is empty (val length 1). Add a len(numeric) check before indexing, or rewrite the condition using numeric.isdigit() and numeric.startswith("-") to avoid direct indexing. Either change fixes the IndexError and makes the function raise the intended ValueError for inputs like "l" or "L".
