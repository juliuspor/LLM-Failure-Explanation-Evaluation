What happened (what failed)
- The test for negative zero expected the string "-0.0" but the code emitted "0". The root cause is that the code treats -0.0 as an integer 0 and prints value (which is "0"), which loses the negative sign.

Where it happens (exact lines and variables)
- At line 177 the code does: value = int(x)
- It then uses that integer to decide how to print. Because int(-0.0) -> 0 and -0.0 == 0 is True, the code takes the integer branch and eventually calls self.add(str(value)) (line 192). That produces "0" and so the sign from x is lost.
- If the code had reached the final fallback self.add(str(x)) (line 194) it would have preserved the sign because str(-0.0) is "-0.0", but it does not reach that because the integer branch ran earlier.

Why that causes the failing test
- Equality x == value treats -0.0 as equal to  integer 0. Using value (the int) to print therefore discards the sign information that is present in the original float x. The specific check/order of branches causes the code to prefer printing str(value) instead of str(x) for values that compare equal, which is fine for ordinary zeros but incorrect for negative zero.

Concrete fixes (step-by-step, specific edits)
1. Detect negative zero before taking the integer/shortcut branch. Use the existing utility or math.copysign:
   - After computing value = int(x) (line 177) or before using value for printing, insert:
     if CodeConsumer.is_negative_zero(x):
         self.add(str(x))
         return
   This ensures -0.0 goes to the float-format path and prints "-0.0".

2. Alternatively, when printing the integer branch, special-case x == 0.0:
   - Replace the place that calls self.add(str(value)) (line 192) with:
     if value == 0 and math.copysign(1.0, x) < 0:
         self.add(str(x))
     else:
         self.add(str(value))
   This preserves sign for negative zero while keeping the integer optimization for ordinary zeros.

3. If you prefer fewer checks, move the is_negative_zero check to the top of the method (before any integer conversion/formatting), and handle it explicitly:
   if CodeConsumer.is_negative_zero(x):
       self.add("-0.0")
       return
   Then normal logic can proceed unchanged.

Why these fixes work
- They avoid converting -0.0 into an un-signed integer and ensure the printed representation comes from the original float x (or is explicitly "-0.0"), thereby preserving the sign bit that equality/intification loses.

Summary
- The code converts x to int (value) and uses str(value) when x == value. For x == -0.0 this yields "0", dropping the sign. Fix by checking is_negative_zero(x) (or math.copysign) before emitting str(value), and print str(x) (or "-0.0") in that special case. This directly addresses lines 177 and 192 where the sign was lost.
