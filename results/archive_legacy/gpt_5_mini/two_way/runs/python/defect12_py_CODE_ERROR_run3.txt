What happened (root cause)
- The code sets upper = lower when upper < lower, but that assignment happens before you guarantee upper ≤ len(text).
- If lower is greater than the text length, upper becomes a value larger than len(text). Later you call substring_java(text, 0, upper) and pass an end index that is past the string length.
- substring_java (like Java's substring) throws IndexError / "String index out of range" when the end index > text.length(), which is exactly the error you saw: "String index out of range: 15".

Where it occurs in your code
- The problematic sequence is:
  1. if upper < lower: upper = lower
  2. ... later ... result = substring_java(text, 0, upper)   (also used in two other places)
- If lower > len(text) (for example lower == 15 while text.length() == 12), upper becomes 15 and the subsequent substring_java(text, 0, upper) call triggers the IndexError.

Why the current checks are wrong
- You check "if upper == -1 or upper > len(text): upper = len(text)" first, then you check "if upper < lower: upper = lower". That ordering allows upper to be set back to a value > len(text). You need to ensure the final upper value is capped to the string length.

Concrete fixes (choose one)
1) After adjusting for lower, re-cap upper against the text length (small change):
   - Move or add a cap after the lower check:
     if upper < lower:
         upper = lower
     if upper == -1 or upper > len(text):
         upper = len(text)

2) Compute upper in a single expression (safer and clearer):
   - Normalize upper to a valid range before any substring calls:
     if upper == -1:
         upper = len(text)
     upper = max(lower, upper)
     upper = min(upper, len(text))

3) Validate inputs early and return/raise if lower is out of range:
   - If lower > len(text) you could decide to return the whole string or None; for example:
     if lower > len(text):
         return substring_java(text, 0, len(text)) + StringUtils.default_string(append_to_end)

Minimum patch (example):
Replace these two lines:
    if upper == -1 or upper > len(text):
        upper = len(text)
    if upper < lower:
        upper = lower
With:
    if upper == -1:
        upper = len(text)
    if upper < lower:
        upper = lower
    if upper > len(text):
        upper = len(text)

Why this fixes it
- The final value of upper is guaranteed to be between lower and len(text), so you will never pass an end index larger than the string length into substring_java.

Summary
- Root cause: ordering of bounds checks lets upper end up > len(text).
- Fix: ensure upper is re-capped against len(text) after you set upper = lower (or normalize in a single step). This stops substring_java from receiving an out-of-range end index.
