What happened (where and what failed)
- In get_paint you clamp the input into v on lines 3–5:
  - v = max(value, self._lower_bound)
  - v = min(v, self._upper_bound)
- But on the next line you compute g from value, not from v:
  - g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- Because you used value instead of the clamped v, g can end up outside 0–255 even though you intended to clamp first. That causes the check on lines 74–75 to raise the ValueError:
  - if g < 0 or g > 255: raise ValueError(...)

Why this is the root cause
- You explicitly create a clamped variable v but never use it. The math still uses the original value, so any out-of-range input will produce an out-of-range g and trigger the exception.
- There is a second related issue: if self._upper_bound == self._lower_bound, the denominator (self._upper_bound - self._lower_bound) is zero and the division will raise ZeroDivisionError. That can happen even if you used v.

Exact fixes (step-by-step)
1. Use the clamped variable v when computing g:
   - change the g line to use v instead of value.
2. Protect against zero division:
   - if self._upper_bound == self._lower_bound, decide on a defined result (e.g., return (0,0,0) or (255,255,255) or 128 gray) and return early.
3. Optionally ensure numeric rounding and final clamp of g to [0,255] before returning.

Suggested corrected implementation
- Replace your function body with something like:

    def get_paint(self, value: float) -> Tuple[int, int, int]:
        # Clamp value to bounds
        v = max(value, self._lower_bound)
        v = min(v, self._upper_bound)

        # Guard against zero range
        span = self._upper_bound - self._lower_bound
        if span == 0.0:
            # choose a sensible default; here mid gray
            return (128, 128, 128)

        # Use the clamped v, not the original value
        g = int(round((v - self._lower_bound) / span * 255.0))

        # Final clamp to 0..255 just in case
        g = max(0, min(255, g))

        return (g, g, g)

Why this resolves the failure
- Using v guarantees the mapped value is based on the clamped input, so g stays in the expected range.
- The zero-range check prevents division by zero.
- The final clamp prevents any tiny floating-point rounding from producing g outside 0–255.

References to your code lines
- Clamping occurs at lines where v is assigned (your lines 3–5).
- The buggy computation is at the line computing g (your line after clamping).
- The exception raised is at lines 74–75 in your slice block.
