What happened (root cause)
- At line 92 the code unconditionally does parts.append(replacement_list[replace_index]). If replacement_list[replace_index] is None this later causes a crash when the code tries to build/join the output (''.join(parts) fails on a None), because join requires strings.
- The intent was to ignore None entries in search_list/replacement_list, but the code that marks those entries as “no more matches / ignored” is inside the match-finding loop (lines 102–105). That check runs after a replacement index has already been chosen and used at line 92.
- In short: the code chooses a replace_index, immediately uses replacement_list[replace_index] (line 92), and only afterward marks entries with replacement_list[i] is None as ignorable (lines 102–105). The order is wrong — the None-check happens too late.

Where it happens (specific lines / variables)

Why that causes a crash
- If replacement_list[replace_index] is None, parts contains a None. Later when code attempts ''.join(parts) (or otherwise expects strings) Python raises TypeError. The None was supposed to be ignored, but it wasn’t because the code checked for None only in the next phase of the loop instead of before using the replacement.

Concrete fixes (exact steps you can apply)
1. Prevent using a None replacement at the point of use (before line 92). Replace line 92 with a guarded append:
   - if replacement_list[replace_index] is None:
       parts.append('')   # or skip appending entirely
     else:
       parts.append(replacement_list[replace_index])
   This ensures no None gets into parts.

3. Also add a safety check that replace_index is valid before using it:
   - if replace_index is None or replace_index < 0: break or continue the outer loop
   This guards against accidental -1/None values being used.

Which of the two fixes I recommend
- Best: combine (2) and (1). Mark unusable slots up-front (so they are never considered in the find loop), and keep the guarded append as a defensive programming measure. That both fixes the bug and prevents regressions if future changes reorder logic again.
