What happened (where it fails)
- The failure occurs at get_paint, line 71:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- If self._upper_bound == self._lower_bound, that expression tries to divide by zero and raises a ZeroDivisionError at that line.

Why it fails (root cause)
- The code never guarantees the denominator (self._upper_bound - self._lower_bound) is nonzero.
- That can happen if the caller constructs the object with equal bounds or if bounds were set incorrectly later. The division by zero is the direct cause.
- A secondary problem: when the denominator is nonzero but bounds are reversed (upper < lower) or value falls outside the bounds, g can be negative or >255, and the code then raises a ValueError (line 75) with an unhelpful message "Color parameter outside of expected range: Red Green Blue" that does not show which parameter/value failed.

How to fix it (concrete steps)
1. Validate bounds in __init__ (lines 30–44):
   - Add a check after assigning the fields to ensure upper > lower (or at least upper != lower), e.g.
     if upper_bound == lower_bound:
         raise ValueError(f"upper_bound ({upper_bound}) must not equal lower_bound ({lower_bound})")
   - Prefer upper_bound > lower_bound if that is required for your mapping.

2. Defend in get_paint (line 71) in case bounds get changed after init:
   - Compute denom = self._upper_bound - self._lower_bound and explicitly check it before dividing:

     ratio = (value - self._lower_bound) / denom

   - This makes the failure explicit and points to the offending bounds.

3. Make the error message at line 75 useful:
