What happened (where)
- The failure happens inside add_number, at the check on the first line of the integer branch:
  if x == int(x) and not math.isinf(x) and not math.isnan(x):
- For x == -0.0 this condition is True because in Python -0.0 == 0.0, so the code treats -0.0 as an integer value.
- The next line value = int(x) produces value == 0 (losing the negative sign). Later the path ends up calling self.add(str(value)) which writes "0".
- The test expects "-0.0", so the sign bit of negative zero was lost by converting to int and then to string.

Why it happens (root cause)
- Comparing x == int(x) collapses -0.0 to 0.0, so the sign is indistinguishable at that point.
- Converting to int (value = int(x)) discards the sign bit; int(-0.0) == 0.
- There is no subsequent check for negative zero, so negative-zero input is printed as "0" instead of "-0.0".

Exactly where to change (concrete fix steps)
1. Detect negative zero before the integer branch and handle it specially. For example, add at the top of add_number, after prev = self.get_last_char():
   if CodeConsumer.is_negative_zero(x):
       # preserve sign; use "-0.0" (test expects this)
       self.add("-0.0")
       return
2. Alternatively, if you prefer inline detection without a helper:
   if x == 0.0 and math.copysign(1.0, x) < 0:
       self.add("-0.0")
       return
3. Ensure this check runs before any int(x) conversion or equality that would collapse sign.

Exactly what to change in your code (drop-in patch)
- Insert the negative-zero check immediately after prev = self.get_last_char(), e.g.:

Why that fix works
- math.copysign(1.0, x) < 0 detects the sign bit even when the value compares equal to +0.0, so negative zero is recognized before int(x) removes the sign.
- Printing "-0.0" directly preserves the negative zero in output as the test expects.

Other notes and pitfalls
- Do not rely on x == int(x) to detect integer-ness when you must preserve the sign of zero.
- Avoid using int(x) before checking for negative zero.
- Using CodeConsumer.is_negative_zero(x) (if available) is a clear semantic solution; the call should happen before integer-conversion logic.
