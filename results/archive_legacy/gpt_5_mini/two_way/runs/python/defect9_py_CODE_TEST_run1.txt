What happened (what fails)
- The code raises a TypeError when either search_list[i] or replacement_list[i] is None because the loop that computes increase calls len(...) on those items without checking for None.
- The failing loop starts at the comment "# BUG: this loop is not null-safe for elements inside the arrays" and iterates:
  for i in range(search_length):
      greater = len(replacement_list[i]) - len(search_list[i])
      ...
  Calling len(None) raises: TypeError: object of type 'NoneType' has no len().

Where it occurs (exact place & variables)
- Function: replace_each
- Location: the "increase" estimation loop immediately after the first match search (the loop annotated with the BUG comment).
- Variables involved: replacement_list[i], search_list[i], and search_length.
- Line causing the exception: the expression len(replacement_list[i]) or len(search_list[i]) inside that loop.

Why it happens (root cause)
- Earlier in the function you correctly treat None/empty search strings and None replacements as "skip" when looking for matches:
    if search_list[i] is None or len(search_list[i]) == 0 or replacement_list[i] is None:
        continue
  That ensures the main matching loops don't break on None.
- But the separate "increase" loop does not perform the same null/empty checks and unconditionally computes len(replacement_list[i]) and len(search_list[i]). For tests that include None in the arrays (for example replacement_list contains None), len(None) is invalid and throws a TypeError.
- The tests that reproduce this are:
  - test_null_safety_inside_arrays_single: replacement_list = [None] — fails when increase loop runs.
  - test_null_safety_inside_arrays_mixed: replacement_list contains None — also fails.

How to fix it (concrete steps)
1. Make the "increase" loop null/empty-safe by skipping elements where search_list[i] is None, search_list[i] == "", or replacement_list[i] is None — the same conditions used elsewhere in the function.
2. Replace the loop with one of these concrete alternatives:

Option A — explicitly check before using len():
    for i in range(search_length):
        if (
            search_list[i] is None
            or len(search_list[i]) == 0
            or replacement_list[i] is None
        ):
            continue
        greater = len(replacement_list[i]) - len(search_list[i])
        if greater > 0:
            increase += 3 * greater

Option B — use safe lengths (shorter):
    for i in range(search_length):
        s = search_list[i]
        r = replacement_list[i]
        if not s or r is None:  # beware that not s treats "" as True and None as True
            continue
        greater = len(r) - len(s)
        if greater > 0:
            increase += 3 * greater

3. Keep the rest of the function unchanged. With this fix, tests using None inside the arrays will no longer hit len(None) and the method will behave as intended.

Why this fix is correct
- It preserves the original intention: skip null/empty searches and null replacements for both the match-finding and the buffer-size estimation phases.
- It prevents the TypeError at the line computing len(...) by ensuring we only call len on real strings.

Minimal patch to apply
- Replace the three-line body of the "increase" loop with the Option A code snippet above (i.e., add the same null/empty checks used elsewhere before computing lengths).
