What failed
- The function to_utf16_code_units (starts at line 10) raises IndexError at lines 35–36:

- That check fired because the value in index is not measured in the same units as len(code_units). In other words, index was computed relative to something else (the input string or Unicode code points) while code_units is a list of UTF‑16 code units. Because some Unicode code points become two UTF‑16 code units (surrogate pairs), index can be greater than or equal to len(code_units) even though it looks valid for the original string.

- Variable names and lines:

  - index is later tested against len(code_units) at lines 35–36.
- The function computes or uses index in a way that assumes one list element per character of s (one index per code point), but code_units can have more elements than characters because some characters produce two code units.
- That mismatch (index measured in code points vs. code_units measured in UTF‑16 units) makes the boundary check invalid. When index refers to a code point position that maps to a later position in code_units, index >= len(code_units) becomes true and you get this IndexError.

Where it occurred
- Inside to_utf16_code_units, at the boundary check on line 35. The check compares index against len(code_units) but index was derived earlier in the function using a different unit (string characters/code points), so the comparison is inconsistent.
