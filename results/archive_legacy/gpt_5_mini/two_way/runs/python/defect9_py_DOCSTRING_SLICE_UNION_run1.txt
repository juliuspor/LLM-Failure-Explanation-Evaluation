What happened (root cause)
- At lines 82–84 the code calculates:
  greater = len(replacement_list[i]) - len(search_list[i])
  This runs for every i in range(search_length) (loop starting at line 81).
- If either replacement_list[i] or search_list[i] is None, calling len(None) raises TypeError and the function crashes before it gets to any later logic that would ignore None entries.
- The later code that tries to ignore None entries (the condition around lines 102–105 which checks search_list[i] is None or replacement_list[i] is None) is reached only after the earlier length calculation. In short: the code checks for None too late — after it already used len() on the possibly-None values.

Where it fails (specific lines)
- Line 82–84: len(replacement_list[i]) and len(search_list[i]) are used unguarded. This is the immediate crash site.
- The loop that does this starts at line 81 and runs before the main replacement loop (so crashes before replacements).
- The later check that would have skipped None entries is around lines 102–105, but it never gets a chance to run because of the earlier crash.

Why the later check doesn't protect you
- The condition at lines 102–105 is intended to skip a pattern if search_list[i] is None or replacement_list[i] is None (and other cases). But that check happens after the code has already tried to compute lengths for all entries at lines 82–84. Therefore the check cannot prevent the earlier TypeError.

Summary
