What happened (what failed)
- The code turns lower=15 and upper=20 into upper=15 (because of this line: if upper < lower: upper = lower).
- Later the function calls substring_java(text, 0, upper) with upper == 15, but len(text) == 10.
- substring_java raises IndexError because the end index (15) is greater than the string length (10). The slice-check code shows the exception (String index out of range: 15).

Why it happened (root cause)
- The function enforces upper >= lower unconditionally (upper = lower) even when lower is beyond the input length.
- There is no early check for lower > len(text). For inputs where lower > len(text), the logic should not try to take a substring up to lower (which is past the end) — it should simply return the original text.
- Concretely:
  - After normalizing upper with: if upper == -1 or upper > len(text): upper = len(text)  (so upper becomes 10),
  - then the next line if upper < lower: upper = lower sets upper = 15,
  - then substring_java(text, 0, upper) runs with end=15 -> IndexError.

Where it occurs (lines / expressions)
- The problematic sequence is:
  - if upper == -1 or upper > len(text): upper = len(text)
  - if upper < lower: upper = lower
  - result = substring_java(text, 0, upper)   <-- raises IndexError when lower > len(text)
- The error message shows end index 15, matching lower=15 being used as the end.

How to fix it (specific steps / patch)
Option A — simplest, return original text if lower is beyond the string:
1. After checking for empty text, add a guard:
   if lower > len(text):
       return text
This prevents any attempt to slice past the end.

Option B — preserve the existing intent of using upper but avoid making upper > len(text):
1. Change the adjustment so upper is never set to lower if lower is > len(text). For example:
   if upper == -1 or upper > len(text):
       upper = len(text)
   if lower > len(text):
       return text
   if upper < lower:
       upper = lower

Either option will satisfy the test case (WordUtils.abbreviate("0123456789", 15, 20, None) should return "0123456789").

Why these fixes are correct
- They stop the function from asking substring_java for an end index greater than len(text).
- They match the documented/expected behavior (original bug Lang-45): when lower is beyond the string length, return the full string (do not try to abbreviate).
