What happened (root cause)
- At line 176 the code does:
  if x == int(x) and not math.isinf(x) and not math.isnan(x):
- For x == -0.0 that condition is True because int(-0.0) == 0 and -0.0 == 0.0 holds.
- The branch for "integer-valued" numbers then formats x as an integer (losing the negative sign on zero) and ends up appending "0" (or "0" with no fractional part) instead of preserving the negative zero sign.
- So negative zero is being treated as an integer-valued number and its sign is lost before any special negative-zero handling is applied. The failing test (test_negative_zero) expects "-0.0", but the code emitted "0" (or "0" without sign), hence the assertion failure.

Where it occurs
- The problematic conditional is at the reported slice line:
    176 |         if x == int(x) and not math.isinf(x) and not math.isnan(x):
- This is inside CodeConsumer.add_number (the method under test). The variable x is the float being printed.

Why it happens (how sign is lost)
- Python's numeric equality and int conversion do not preserve negative zero: int(-0.0) -> 0 and -0.0 == 0.0 is True.
- The integer branch assumes sign/formatting suitable for whole numbers and does not specially detect negative zero first, so the negative sign is dropped.

Concrete fixes (clear steps)
1. Special-case negative zero before the integer-valued branch. Use the existing utility or a reliable detection:
   - CodeConsumer.is_negative_zero(x) is provided in the tests; use that.
   - Or use math.copysign(1.0, x) < 0, or struct.unpack to test sign bit.

2. Insert the check before line 176. If x is negative zero, append "-0.0" (or whatever canonical representation you want to preserve) and return.

Minimal code change example (pseudocode showing exact placement)
- Replace around line 175–177 with:

    # Special-case negative zero so its sign is preserved
    if CodeConsumer.is_negative_zero(x):
        self.append("-0.0")
        return

    if x == int(x) and not math.isinf(x) and not math.isnan(x):
        ...

Notes about alternatives
- You could also format zero values while preserving sign using math.copysign or by checking `math.copysign(1.0, x) < 0`.
- Do not rely on the integer branch to preserve sign for -0.0 — special-casing is the simplest, safest approach.

Summary
- The test fails because the integer-valued branch (line 176) runs for -0.0 and loses the negative sign. Fix by detecting negative zero before that branch (use CodeConsumer.is_negative_zero or math.copysign) and append "-0.0" explicitly.
