What happened (where it fails)
- The failure happens in the surrogate-pair handling path of code_point_at (slice lines 38–44). That function computes and returns the Unicode code point:
  - when it finds a high surrogate followed by a low surrogate it returns 0x10000 + ((high - 0xD800) << 10) + (low - 0xDC00)  (line 43),
  - otherwise it returns high (line 44).
- The caller (the translator loop) does not treat the returned value as a code point; it treats it as the number of UTF-16 code units to advance the index by. In other words, the code is doing index += cp (or otherwise using cp as a count) instead of index += 1 (or 2 when the code point is a surrogate pair).

Why that causes an IndexError
- For a surrogate pair (e.g. the emoji \U0001F630), code_point_at returns a large integer representing the code point (~0x1F630 = 128,560).
- The caller uses that returned integer as the amount to advance the index. So index gets increased by ~128,560 instead of by 2, jumping well past the length of the code_units array. The next access into code_units raises IndexError.
- The failing test "translate_surrogate_pair_with_suffix" (\U0001F630A) exercises exactly this: code_point_at at index 0 returns the large code point, the caller advances the index incorrectly, and the translator crashes when it tries to read out-of-range code_units entries.

Root cause (concise)
- Mismatch of contract: code_point_at returns a Unicode code point integer, but the caller expects (or treats the return value as) a unit count. The code that advances index should use 1 for BMP characters and 2 for surrogate pairs; it must not add the numeric code point value to index.

Concrete fixes (step-by-step)
Option A — change the caller (recommended, minimal):
1. Find the translator loop where you call cp = code_point_at(code_units, index).
2. After computing cp, compute the number of UTF-16 code units consumed:
   - consumed = 2 if cp > 0xFFFF else 1
3. Advance the index with index += consumed (not index += cp).
4. Keep writing/escaping cp as the code point value.

Option B — change code_point_at to return both values:
1. Modify code_point_at to return a tuple (code_point, consumed_units). For example:
   - return (0x10000 + ..., 2) when you detect a valid surrogate pair
   - return (high, 1) otherwise
2. Update the caller to unpack cp, consumed = code_point_at(...); then use index += consumed.

Option C — if the existing intent was that code_point_at should return the number of units consumed (less likely), rewrite it so it returns 1 or 2 and stop returning the code point. This is only correct if the caller really only needs the consumed count.

What to check after fixing
- Ensure the translator loop uses the cp value as a code point (for escaping/output) and uses consumed (1 or 2) for index advancement.
- Re-run the provided unit tests: test_translate_surrogate_pair_with_suffix, test_translate_single_surrogate_pair, and others.

References to variables/lines
- code_point_at uses code_units and index (slice lines 38–44). It returns the computed code point for surrogate pairs (line 43) and high for non-pairs (line 44).
- The bug appears when index is advanced using that returned value (the code point) instead of using 1 or 2 UTF-16 code units.

This explains why you get an IndexError: the code advances the index by the numeric code point value instead of the number of UTF-16 code units consumed. Fix by advancing index by 1 or 2 (or return the consumed-unit count from code_point_at).
