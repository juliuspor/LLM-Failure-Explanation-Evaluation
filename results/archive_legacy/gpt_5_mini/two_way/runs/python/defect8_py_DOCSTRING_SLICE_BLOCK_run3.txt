What happened (where it fails)
- The failure occurs in to_locale(), at the check on lines 239–240:
  if ch3 < 'A' or ch3 > 'Z' or ch4 < 'A' or ch4 > 'Z':
    raise ValueError(f"Invalid locale format: {locale_str}")
- The exception you see is a TypeError like:
  TypeError: '<' not supported between instances of 'int' and 'str'
- That means ch3 and/or ch4 are integers while the code is comparing them to string characters ('A' and 'Z').

Why it happened (root cause)
- Earlier in the function ch3 and ch4 were set using ord(...), e.g.:
    ch3 = ord(locale_str[3])
    ch4 = ord(locale_str[4])
  (this is the only way they would be ints at this point)
- The comparison on lines 239–240 mixes types: comparing an int (ch3) to a str ('A') is invalid in Python and raises TypeError.
- The check intends to test whether those characters are between 'A' and 'Z', but because ord() was used, the operands are different types so the comparison cannot run.

Exactly what to change (concrete steps)
Option A — keep characters (recommended)
1. Stop using ord() when extracting ch3 and ch4. Make them single-character strings:
     ch3 = locale_str[3]
     ch4 = locale_str[4]
2. Keep the existing check, or make it clearer:
     if ch3 < 'A' or ch3 > 'Z' or ch4 < 'A' or ch4 > 'Z':
         raise ValueError(...)
   or equivalently:
     if not ('A' <= ch3 <= 'Z' and 'A' <= ch4 <= 'Z'):
         raise ValueError(...)

Option B — use integer comparisons (if you want ord values)
1. Keep ord() but compare to ord('A') / ord('Z'):
     if ch3 < ord('A') or ch3 > ord('Z') or ch4 < ord('A') or ch4 > ord('Z'):
         raise ValueError(...)

Extra note
- Also ensure you only index locale_str[3] and [4] after confirming the string is long enough; otherwise you'll get IndexError before these checks. Validate length first.

Summary
- Root cause: mixing int (from ord) with str in comparisons at lines 239–240 causes a TypeError.
- Fix by making ch3/ch4 the same type as the bounds (either both strings or both ints) and validating indices before indexing.
