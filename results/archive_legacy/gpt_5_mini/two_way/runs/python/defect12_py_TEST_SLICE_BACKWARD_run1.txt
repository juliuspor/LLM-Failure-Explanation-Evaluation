What happened
- The tests call WordUtils.abbreviate with upper = -1 (meaning "no limit").
- That value ends up passed into slice_backward as the variable end.
- In slice_backward, lines 32–33 contain this check:

  if start < 0 or end < 0 or start > end or start > len(s) or end > len(s):
      raise IndexError(f"String index out of range: {end}")

- Because end == -1, the condition end < 0 is true and slice_backward raises IndexError (message "String index out of range: -1").
- Result: tests that expect "-1 means no upper limit" fail with an IndexError.

Why this is the root cause
- The code treats any negative end index as an invalid index. But the rest of the code (and the test suite) uses -1 as a sentinel meaning "no upper limit." The check does not respect that sentinel and thus rejects the valid input for the intended API.

Where to change
- File: hit12_wordutils.py
- Function: slice_backward
- Problem lines: 32–33 (the if that rejects negative end values)

Concrete fixes (pick one)
1) Allow -1 as a valid sentinel in the check:
   Replace the failing condition with something like:
   if start < 0 or (end < 0 and end != -1) or start > end and end != -1 or start > len(s) or (end > len(s) and end != -1):
       raise IndexError(f"String index out of range: {end}")

   (Adjust parentheses to match the exact intended semantics.)

2) Normalize the sentinel before calling slice_backward:
   - If end == -1, set end = len(s) (or otherwise interpret -1 to mean "no limit") before the index validation and slicing.

3) Special-case end == -1 at the very top of slice_backward:
   if end == -1:
       end = len(s)

   then keep the rest of the existing validation (but still allow start <= end).

Which to choose
- The simplest and safest change: at the start of slice_backward, convert the sentinel into the effective index:
  if end == -1:
      end = len(s)
  Then the existing checks work unchanged.

Example concrete patch (minimal):
  def slice_backward(s, start, end):
      if end == -1:
          end = len(s)
      if start < 0 or end < 0 or start > end or start > len(s) or end > len(s):
          raise IndexError(f"String index out of range: {end}")
      ...

After applying one of these fixes, passing upper = -1 will not trigger the IndexError and the tests (including test_abbreviate_no_limit) should pass.
