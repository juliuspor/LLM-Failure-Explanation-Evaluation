What happens (where)
- At line 38 you load high = code_units[index].
- Lines 39–41 detect a high surrogate (0xD800..0xDBFF) and look at the next code unit (low = code_units[index + 1]).
- If the next unit is a low surrogate (0xDC00..0xDFFF) you combine them and return the full code point at line 43.
- If the next unit is not a low surrogate (or there is no next unit) the function falls through to line 44 and returns high — i.e. the raw high-surrogate value (a number between 0xD800 and 0xDBFF).

Why that fails (root cause)
- A high surrogate value by itself is not a valid Unicode scalar. Returning the raw high-surrogate integer leaks an invalid code point into the rest of the pipeline.
- Downstream code that expects valid Unicode scalars (or that will encode/serialize the value) can fail, raise exceptions, or produce invalid output when given a lone high surrogate.
- The problem occurs specifically when a high surrogate is followed by a non-low surrogate (or when the high surrogate is at the end of the list). The code never treats that situation as an error or a replacement (line 39–44), it merely returns high unchanged.
