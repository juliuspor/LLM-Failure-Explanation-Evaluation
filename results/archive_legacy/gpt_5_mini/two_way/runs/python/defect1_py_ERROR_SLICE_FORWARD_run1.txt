- After that multiplication the code either validated minutes or passed minutes_offset onward to cls.for_offset_millis, but minutes_offset already contains an invalid negative minute value (-15), hence the ValueError.

Why it failed (root cause)
- The code reuses the variable minutes_offset for two different purposes: initially it holds the parsed minute component (0..59), then it is reused to hold the total offset in minutes (hours*60 ± minutes). That reuse combined with incorrect sign handling produces a negative minute component (−15).
- Specifically, when computing a negative offset the code does:
  minutes_offset = hours_in_minutes - minutes_offset
  which results in hours*60 - minutes (for example 1*60 - 15 = 45), and then in another branch:
  minutes_offset = hours_in_minutes + minutes_offset
  which mixes already-changed values and can leave minutes_offset with the wrong sign and magnitude.
- The correct arithmetic for a negative offset should produce minutes_offset = -(hours_in_minutes + parsed_minutes) or apply the sign to the combined total. Instead, the code performs subtraction/addition in a way that leaves the minute part negative (e.g. −15), and that triggers the "Minutes out of range" check.
