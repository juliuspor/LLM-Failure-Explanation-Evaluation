What happened (where it fails)
- At line 335 you check array, at 337 you check element, and at 340 you set inferred_type = object.
- Because line 340 is not inside an else branch, inferred_type = object runs unconditionally every time execution reaches that point — it overwrites whatever you set at lines 336 or 338.
- Later, at line 346 you do:
  if inferred_type == object and expected_type != object:
      raise TypeError(
  That condition becomes true whenever expected_type != object, because inferred_type was just forced to object at line 340. That causes a TypeError to be raised even when array or element provided a concrete type.

Why this is the root cause (simple)
- The code’s intent is clearly to set inferred_type from array, or from element, or fall back to object only when both are absent. But because of incorrect indentation / a missing else, the fallback assignment is executed every time instead of only when both are absent. That makes the later type check always fail in the common case.
