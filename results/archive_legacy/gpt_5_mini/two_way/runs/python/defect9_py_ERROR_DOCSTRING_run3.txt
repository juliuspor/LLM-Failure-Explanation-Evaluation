What failed
- The TypeError "object of type 'NoneType' has no len()" happens because the code calls len(...) on an element taken from search_list or replacement_list that is actually None.
- The crash occurs inside the function that performs the replacements (I'll call it replace_all below), not when measuring the outer lists. The code assumes every element in search_list and replacement_list is a string and calls len(search) or search.startswith(...) etc. on each element without first checking if that element is None.

Why it happens (root cause)
- Variables involved: search_list (list of search strings), replacement_list (list of replacement strings), and the loop variables search and replacement you get when iterating those lists.
- Example problematic pattern in replace_all:
  - for search, replacement in zip(search_list, replacement_list):
      if len(search) == 0: ...
      # or: if search.startswith(...): ...
  - If search is None, the call len(search) (or search.startswith) raises TypeError because NoneType has no length and no string methods.
- The docstring says None entries should be ignored, but the code never checks for None before treating elements as strings. So the "ignore None" requirement is not implemented, and encountering None causes the crash.

Where it occurs in the code
- Inside replace_all (the function that iterates zip(search_list, replacement_list)).
- Immediately before any call that assumes search (or replacement) is a string: len(search), search == "", search in target, search.startswith(...), replacement is used in string concatenation, etc.
- Any of these calls on a None entry triggers the TypeError.

Concrete, step-by-step fixes you can apply
1. Decide the intended semantics for None in either list:
   - If a pair (search, replacement) should be ignored when search is None OR replacement is None, skip that pair.
   - If only search==None should be ignored but replacement==None should be treated as the empty string (or vice versa), implement that rule explicitly.

2. Filter or guard before using len() / string methods. Two safe approaches:

   A. Filter pairs up-front (recommended, keeps loop body simple)
   - Replace the iteration with:
       pairs = [(s, r) for s, r in zip(search_list, replacement_list) if s is not None and r is not None]
       for search, replacement in pairs:
           # now safe to call len(search), search in subject, etc.
   - This ensures you never call string methods on None.

   B. Guard inside the loop
   - At the top of the loop do:
       for search, replacement in zip(search_list, replacement_list):
           if search is None or replacement is None:
               continue   # ignore this replacement pair
           # now safe

3. If you do any length checks using len() on entries, make sure you call len() only on strings:
   - Replace len(search) with either:

   - Or after filtering/guarding, len(search) is fine.

4. Add unit tests that exercise the failing case:
   - Call replace_all(target, ["a", None, "b"], ["x", "y", None]) and assert the function does not raise and returns the expected string given your chosen semantics (e.g., pairs containing None ignored).

Example minimal patch (pseudocode)
- Current (buggy) loop:
    for search, replacement in zip(search_list, replacement_list):
        if len(search) == 0:
            continue
        result = result.replace(search, replacement)

- Replace with:

            continue
        if len(search) == 0:
            continue
        result = result.replace(search, replacement)

Why this fixes it
- The TypeError is thrown because len() was called on None. By skipping pairs where search or replacement is None before calling len() or other string methods, you stop calling those methods on None and the function no longer crashes when the lists contain None.
