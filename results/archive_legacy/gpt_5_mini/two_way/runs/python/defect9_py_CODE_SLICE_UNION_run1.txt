What happens
- Root cause: the "increase" sizing loop (lines 81–85) calls len(replacement_list[i]) and len(search_list[i]) without checking whether either element is None. If any element in search_list or replacement_list is None you get a TypeError: "'NoneType' object has no len()".
- Where it occurs: the for loop starting at line 81:
  greater = len(replacement_list[i]) - len(search_list[i])
  This runs before the code that skips None/empty items in the matching loops (the checks at lines 28–49 and 102–113), so it hits None elements and fails.

Why it fails (in simple terms)
- The function signature allows list elements to be Optional[str]. Later code assumes some elements may be None and explicitly guards against them when searching (lines 102–105). But the "increase" loop forgets those guards and blindly calls len(...) on each element. len(None) raises an exception, so the function crashes before doing any replacements.

Exact lines to fix
- Faulty lines: 81–85 (the loop computing increase).
- Related lines that are safe: the matching loops check for None at 28–49 and 102–113. The append at line 92 and the len(...) at line 94 are safe because replace_index was chosen from only guarded candidates.

Concrete fixes (step-by-step)
1. Add the same null/empty checks inside the "increase" loop before calling len():
   - skip if search_list[i] is None
   - skip if replacement_list[i] is None
   - skip if len(search_list[i]) == 0
2. Then compute greater only for valid items.

Minimal replacement code for lines 81–85 (apply in your function):
for i in range(search_length):
    if search_list[i] is None or len(search_list[i]) == 0 or replacement_list[i] is None:
        continue
    greater = len(replacement_list[i]) - len(search_list[i])
    if greater > 0:
        increase += 3 * greater

- Or filter out None entries before computing search_length and all loops (requires more refactor).

Why this fix is correct
- It mirrors the existing checks used when searching for matches, so "increase" only accounts for elements that will actually be used. It prevents len(None) calls and keeps the rest of the algorithm unchanged.
