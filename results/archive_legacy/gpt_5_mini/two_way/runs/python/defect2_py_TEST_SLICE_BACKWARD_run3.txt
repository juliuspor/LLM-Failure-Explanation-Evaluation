What happened (root cause)
- At line 71 you compute g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0).
- At lines 74–75 the code raises ValueError if g < 0 or g > 255:

  71 | g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
  74 | if g < 0 or g > 255:
  75 |     raise ValueError(f"Color parameter outside of expected range: Red Green Blue")

- The test suite expects values outside the configured bounds to be clamped to the nearest end (below lower -> black (0,0,0); above upper -> white (255,255,255)). Instead, the implementation throws an exception when the computed g falls outside 0–255. That is why tests calling get_paint(-0.5) or get_paint(1.5) fail — the code refuses to map out-of-range inputs and raises ValueError.

Where it happened
- In get_paint at the calculation and the subsequent check around lines 71 and 74–75 (see snippet above). The check triggers for negative g (value < lower_bound) and for g > 255 (value > upper_bound).

Why it happened (mechanics)
- The method computes the grayscale channel directly from the raw value without first clamping the input to the valid range. For example, value = -0.5 gives a negative numerator, producing a negative g; the code treats that as an error and raises ValueError instead of returning black.
- The intention in the tests (and typical behavior for paint scales) is clamping, not erroring.
- (Side note) The error message f-string on line 75 is static text and does not show the offending numbers; that makes debugging less helpful.

Precise fixes (what to change and where)
1. Clamp the input value to the configured bounds before computing g (preferred):
   - Replace the code around line 71 with:
     - clamp_value = min(max(value, self._lower_bound), self._upper_bound)
     - g = int((clamp_value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
   - Remove the ValueError branch that raises for g outside 0–255 (because clamping guarantees g is within range).

2. Alternatively, clamp the computed g to [0, 255] after computing it:
   - g = int(…)
   - g = max(0, min(255, g))
   - Return (g, g, g)

3. Improve the error handling/message:
   - If you still want to raise for invalid bounds (upper == lower or other configuration errors), raise only when upper == lower (division by zero). Use a clear message including the bound values:
     - if self._upper_bound == self._lower_bound:
         raise ValueError(f"upper_bound ({self._upper_bound}) must differ from lower_bound ({self._lower_bound})")

Exactly what to change (minimal patch)

Why this satisfies the tests
- Values below lower_bound become lower_bound → maps to 0.
- Values above upper_bound become upper_bound → maps to 255.
- Values in between map linearly and produce the expected 127 for 0.5 with default bounds.
