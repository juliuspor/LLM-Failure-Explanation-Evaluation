The bug is that after a translator consumes codepoints, the loop increments pos by calling code_point_at without ensuring pos is still a valid index; this can call code_point_at with pos == len(code_units) and raise IndexError. Fix: keep initial validations, and when consumed>0, increment pos in a loop but check pos < length before calling code_point_at, breaking if we've reached the end. This prevents out-of-range access while preserving behavior.
