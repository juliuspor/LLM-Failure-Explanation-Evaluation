- What it does: when deciding the array element type it sets inferred_type = type(array). That returns the Python type of the container (e.g. list), not the element type expected inside the list.
- Why that fails: in your tests you call ArrayUtils.add(None, None, expected_type=str). For this call:
  - array is None, element is None so the code falls into the "both None" branch and sets inferred_type = object (line 340).
  - expected_type is str, so the check at line 346 (inferred_type == object and expected_type != object) is true.
  - The method then raises TypeError (line 347) — but your tests expect it to return ["None"] or a list of the expected_type. So instead of returning a one-element list using expected_type, the code treats the lack of concrete type info as an error.
- Additional bug: when array is not None the code uses type(array) (line 336). For a Python list this gives <class 'list'>, not the element type (e.g. <class 'str'>). That will break any logic that assumes inferred_type represents the contained element type.

- Lines 337–340 and 346–347: the handling of element==None and array==None sets inferred_type = object then rejects expected_type if it is more specific.
