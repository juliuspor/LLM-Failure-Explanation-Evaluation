Exact fix (steps)
- Replace the raise with clamping logic:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
  # clamp to valid byte range
  if g < 0:
      g = 0
  elif g > 255:
      g = 255
  return (g, g, g)

- Add a check for zero range to produce a clear error or handle it:
  if self._upper_bound == self._lower_bound:
      raise ValueError("upper_bound must not equal lower_bound")

Why this fixes it
- The tests expect negative inputs to map to black (0,0,0) and inputs > upper_bound to map to white (255,255,255). Clamping g to [0,255] implements that behavior.
- The zero-range guard prevents a division-by-zero and gives a clear, specific failure if the bounds are misused.

Optional small improvements
- Compute the normalized value with float and clamp that before converting to int:
  norm = (value - self._lower_bound) / (self._upper_bound - self._lower_bound)
  norm = max(0.0, min(1.0, norm))
  g = int(norm * 255.0)

This keeps the semantics clearer and avoids negative/overflow intermediate values.
