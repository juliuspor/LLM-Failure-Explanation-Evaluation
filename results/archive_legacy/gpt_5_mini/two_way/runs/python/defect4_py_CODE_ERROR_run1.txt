What happened (root cause)
- In the "max middle" block you fetch the period using the wrong stored index.
- Specifically, this line (near the start of the max-middle block):

      max_middle_period = self.get_data_item(self._min_middle_index).get_period()

  should use self._max_middle_index, not self._min_middle_index. Because of that you compare the new item's middle against the period at _min_middle_index instead of against the current max candidate. That incorrect comparison can fail to update _max_middle_index correctly and produces the wrong result (your test expected max_middle_index=1 but the code set it to 3).

Why that causes the observed failure
- The code computes middle = start + ((end - start) // 2).
- When checking for a new maximum middle you compare middle > max_middle.
- But max_middle is computed from the period at _min_middle_index (wrong item). If that wrong item's middle is larger or smaller in the wrong way, the conditional will evaluate incorrectly and either skip updating or update incorrectly, producing the wrong index (3 in your test).

Exactly where to change it
- Replace the incorrect get_data_item call in the max-middle block:

    max_middle_period = self.get_data_item(self._min_middle_index).get_period()

  with:

    max_middle_period = self.get_data_item(self._max_middle_index).get_period()

Suggested fix (concrete patch)
- Edit the max-middle block to read:

    if self._max_middle_index >= 0:
        max_middle_period = self.get_data_item(self._max_middle_index).get_period()
        if isinstance(max_middle_period, SimpleTimePeriod):
            s = max_middle_period.get_start_millis()
            e = max_middle_period.get_end_millis()
        else:
            s = int(max_middle_period.get_start().timestamp() * 1000)
            e = int(max_middle_period.get_end().timestamp() * 1000)
        max_middle = s + (e - s) // 2
        if middle > max_middle:
            self._max_middle_index = index
    else:
        self._max_middle_index = index
