What happened (where it failed)
- The failure happens inside get_paint at line 71:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- If self._upper_bound == self._lower_bound, the expression divides by zero and Python raises a ZeroDivisionError before execution ever reaches the if at line 74.
- If the denominator is nonzero but value is outside the [lower_bound, upper_bound] interval, g can be < 0 or > 255 and the method deliberately raises ValueError at line 74.

Why this happens (root cause)
- Root cause 1 (crash): No check that self._upper_bound != self._lower_bound before using the denominator, so division by zero can occur at line 71.
- Root cause 2 (runtime error raised by code): The method expects value to lie inside the bounds; when it does not, the computed g falls outside 0–255 and triggers the ValueError at line 74. That is intentional but the error message is unhelpful (see below).

Exact places to change and what to do (step-by-step fixes)
1) Prevent division by zero at construction
   - In __init__ (lines shown around 30 and 43–44), validate the bounds:
     - require self._upper_bound > self._lower_bound
     - if not, raise ValueError("upper_bound must be greater than lower_bound")
   - This guarantees the denominator in get_paint (line 71) is never zero.

2) Make get_paint handle out-of-range value explicitly and give clear message
   Option A — reject out-of-range values:
   - At the start of get_paint (before line 71) check:
       if value < self._lower_bound or value > self._upper_bound:
           raise ValueError(f"value {value} outside bounds [{self._lower_bound}, {self._upper_bound}]")
   - This prevents computing g for invalid inputs and gives a precise message.

   Option B — clamp value into the bounds (if you prefer robustness instead of error):
   - Replace the computation with a clamped value:
       clamped = max(self._lower_bound, min(value, self._upper_bound))
       g = int((clamped - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
   - This avoids ValueError at line 74 because g will be in 0–255.
