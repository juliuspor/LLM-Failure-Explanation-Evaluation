What happened (root cause)
- The code assumes both search_list[i] and replacement_list[i] are valid strings before it ever checks them. That causes a crash when one of them is None.
- The first place this happens is in the "increase" computation at lines 81–85:
  - Line 82: greater = len(replacement_list[i]) - len(search_list[i])
  - If either replacement_list[i] or search_list[i] is None, len(...) raises a TypeError.
- A second, related crash source is at line 108:
  - Line 108: temp_index = text.find(search_list[i], start)
  - If search_list[i] is None, text.find(None, start) raises a TypeError before the later guard at lines 102–105 would skip that index.

Why the later checks don't help
- Lines 102–105 contain the correct guard conditions (no_more_matches_for_repl_index[i] or search_list[i] is None or len(search_list[i]) == 0 or replacement_list[i] is None). But those checks run later than the operations that already used len(...) and text.find(...). So the code checks for None too late.

Where it occurs (line references)
- Calculation that crashes first: lines 81–85 (len(...) used without guards).
- Search/find that crashes later: line 108 (text.find called on search_list[i] before checking for None).
- The guard that was intended to prevent this is at lines 102–105, but it is reached after the problematic calls.

Concrete, step-by-step fixes to apply
1. Skip None/empty entries before using them when computing "increase" (lines 81–85).
   - Replace the body of that loop with an early continue if either element is None or the search string is empty:
     - if search_list[i] is None or replacement_list[i] is None or len(search_list[i]) == 0: continue
     - then compute greater = len(replacement_list[i]) - len(search_list[i])...
2. Before calling text.find(search_list[i], start) at line 108, ensure search_list[i] is not None and not empty and replacement_list[i] is not None. For example:
   - if (no_more_matches_for_repl_index[i] or search_list[i] is None or len(search_list[i]) == 0 or replacement_list[i] is None): continue
   - else temp_index = text.find(search_list[i], start)
3. Optionally, make the loops robust to mismatched list lengths:
   - Set search_length = min(len(search_list), len(replacement_list)) to avoid indexing beyond the shorter list.
4. Ensure you never append a None replacement into parts (line 92). The existing guard at 102–105 suggests replacements with None should be ignored; ensure that check is in effect before appending at line 92.

Why these fixes work
- They put the guard (search_list[i] is None or replacement_list[i] is None or len(search_list[i]) == 0) before any use of len(...) or text.find(...).
- That prevents TypeError from calling len(None) or text.find(None, ...), and avoids appending None replacements.

Summary
- Crash cause: len(...) and text.find(...) are called on None because the None checks occur later (lines 81–85 and 108 happen before 102–105).
- Fix: move/duplicate the None and empty-string checks so they run before any len(...) or text.find(...) usage, and optionally clamp search_length to the shorter of the two lists.
