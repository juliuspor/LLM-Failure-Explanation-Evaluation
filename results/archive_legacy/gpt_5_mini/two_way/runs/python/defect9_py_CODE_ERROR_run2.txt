What went wrong
- The TypeError arises in the "get a good guess on the size of the result buffer" loop:
  for i in range(search_length):
      greater = len(replacement_list[i]) - len(search_list[i])
  This loop calls len(...) on both replacement_list[i] and search_list[i].
- Because the function accepts Optional[List[Optional[str]]], individual entries may be None. If either search_list[i] or replacement_list[i] is None, len(None) raises TypeError: object of type 'NoneType' has no len().
- The code later correctly checks for None/empty entries when searching for matches, but this estimation loop does not — that is the root cause and the exact location of the failure (the "increase" loop, just before start = 0 / parts: List[str] = []).

How to fix it (concrete steps)
1. Skip entries that are None (or empty search strings) when computing increase:
   for i in range(search_length):
       if search_list[i] is None or replacement_list[i] is None or len(search_list[i]) == 0:
           continue
       greater = len(replacement_list[i]) - len(search_list[i])
       if greater > 0:
           increase += 3 * greater

2. Alternatively, use zip and a comprehension to only consider valid pairs:
   for s, r in zip(search_list, replacement_list):
       if s and r:
           greater = len(r) - len(s)
           if greater > 0:
               increase += 3 * greater

3. (Optional) Keep the existing clamp:
   increase = min(increase, len(text) // 5)

Why these fixes work
- They prevent calling len() on None by explicitly skipping None entries (and skip empty search strings consistent with the rest of the function). That removes the TypeError and makes the buffer-size estimate null-safe.

Where to change (exact spot)
- Modify the loop that begins with:
    # BUG: this loop is not null-safe for elements inside the arrays
    for i in range(search_length):
      ...
  Replace its body with one of the safe versions above.

That fixes the TypeError and keeps the later logic (which already handles None/empty entries) consistent.
