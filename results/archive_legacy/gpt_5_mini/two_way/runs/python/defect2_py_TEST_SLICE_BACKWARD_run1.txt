What fails
- Where: get_paint (starts at line ~54); the specific check is at lines 74–75.
- What: instead of clamping out-of-range inputs to black (0,0,0) or white (255,255,255), the method raises a ValueError when the computed gray channel g is < 0 or > 255:
  - line 71 computes g: g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
  - lines 74–75 raise ValueError("Color parameter outside of expected range: Red Green Blue") if g < 0 or g > 255
- Why that breaks the tests: the tests call get_paint(-0.5) and get_paint(1.5) expecting clamping to black and white respectively. Because the code raises an exception for g outside [0,255], the tests fail instead of receiving (0,0,0) or (255,255,255).

Root cause (clear, specific)
- The method enforces a hard error for g out of [0,255] (lines 74–75) instead of clamping values. The variable names involved are value (input), self._lower_bound, self._upper_bound (used to normalize), and g (the computed 0..255 channel).
- The error is raised after computing g; it never clamps either the normalized fraction or the final channel values.

Concrete fixes (specific steps and minimal code to apply)
1. Option A — clamp the input value to bounds before converting:
   - Replace the normalization step with:
     - v = min(max(value, self._lower_bound), self._upper_bound)
     - compute g using v instead of value:
       g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
   - Remove the raise at lines 74–75.
2. Option B — clamp the computed channel g after calculating it:
   - Keep existing compute line (71), then do:
     if g < 0:
         g = 0
     elif g > 255:
         g = 255
   - Remove the raise at lines 74–75.
3. Also improve the error message (if you keep any error for other cases) to include the actual value and bounds, but for these tests you should not raise for out-of-range paint values.

Minimal replacement example (apply in get_paint)
- Replace the compute-and-check block with:
  v = min(max(value, self._lower_bound), self._upper_bound)
  g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
  # ensure numeric safety
  g = max(0, min(255, g))
  return (g, g, g)

Notes
- This also avoids potential division-by-zero if upper == lower; you may want to handle that case explicitly (raise or return black/white) depending on intended behavior.
- The current exception text ("Color parameter outside of expected range: Red Green Blue") is ambiguous; if you keep exceptions for other reasons, include the offending value and bounds.
