- The input string length is 10 (indices 0..9). The method then tries to access the string at index 15 (or otherwise uses 15 as a direct index into the string).
- Accessing input[15] (or any index >= len(input)) raises IndexError: "String index out of range: 15".

Where it occurs
- Inside WordUtils.abbreviate, at the line that uses the lower parameter as a direct string index or attempts a substring that requires input[lower] to exist. For example, code like:
  - value = input[lower]  OR
  - prefix = input[:lower+1]  (if lower is used as an inclusive index)
  will fail when lower >= len(input). The exception message shows the offending index (15), which matches the test's lower argument.

Why it happened (root cause)
- The method assumes the lower index falls within the input string and does not handle the case lower >= len(input). It is missing the guard that if the requested lower bound is beyond the string length, the function should simply return the original string (or otherwise handle it safely).
- The code also appears not to clamp upper to the string length or to treat negative/`-1` upper as "no limit" before using upper-related indexing.

Concrete steps to fix
1. Check for null/None input first and return None.
2. If len(input) == 0 return "" immediately.
3. Normalize/clamp indices before indexing:
   - If lower < 0, set lower = 0.
   - If upper == -1, treat it as "no limit" (set upper = len(input)).
   - If upper > len(input), set upper = len(input).
4. If lower >= len(input) then return the original input (no abbreviation needed).
5. Use safe slicing (input[a:b]) instead of single-index accesses wherever possible; slicing never raises IndexError for out-of-range indices.
