What happened (root cause, plain):

- In TimePeriodValues._update_bounds the code that updates the stored index for a min/max bound sometimes assigns the new index unconditionally, so the conditional comparison is ignored and the bound index is always set to the most recently-visited index.
- Concretely, in the slice you provided the min-start update does this:
  - It computes start for the new period.
  - It loads min_start_period = self.get_data_item(self._min_start_index) and computes min_start.
  - It compares if start < min_start, and inside that if-block sets self._min_start_index = index (line ~346–347).
  - Then, immediately afterward (line ~349) there is another assignment self._min_start_index = index that is executed unconditionally — this overwrites the earlier conditional update (or sets the min index even when the condition is false).
- Because of that unconditional assignment the code ends up returning the last index seen as the min_start_index (and the same pattern is present for other bounds, e.g. the max_middle_index case). That explains your failing test: the true max-middle was at index 1 but the code ended up returning index 3 (the later item) because of the overwrite.
