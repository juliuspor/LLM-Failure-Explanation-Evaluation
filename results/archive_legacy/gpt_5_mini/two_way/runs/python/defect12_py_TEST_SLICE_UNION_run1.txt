What happened
- The test test_abbreviate_lower_beyond_length calls WordUtils.abbreviate("0123456789", 15, 20, None). That eventually calls slice_union with start = 15 and end = 20 while s has length 10.
- slice_union's guard at line 32 is too strict:

  Line 32:
  if start < 0 or end < 0 or start > end or start > len(s) or end > len(s):
      raise IndexError(f"String index out of range: {end}")

- Because start (15) and end (20) are greater than len(s) (10), the guard raises IndexError (line 33) and the test fails.

Why it failed (root cause)
- slice_union assumes start and end must always be inside the string bounds and raises an exception when they are not. But WordUtils.abbreviate expects out-of-range slice indices to be handled gracefully (for example by clamping to the string length or returning the original string), not by raising.
- Concretely: start > len(s) and end > len(s) trigger the explicit "start > len(s) or end > len(s)" check at line 32, which causes the IndexError at line 33. That is the exact place the failure occurs.

How to fix it (concrete steps)
Option A — Clamp indices before slicing (recommended)
1. Remove the "or start > len(s) or end > len(s)" part from the guard at line 32.
2. Add clamping right before the return, for example:
   start = max(0, min(start, len(s)))
   end   = max(0, min(end, len(s)))
3. Then return s[start:end].

Result: start and end outside [0, len(s)] become len(s) (or 0 if negative), avoiding IndexError and matching WordUtils.abbreviate expectations.

Option B — Accept out-of-range starts/ends as valid by changing the check
1. Change the guard to only reject negative indices or start > end:
   if start < 0 or end < 0 or start > end:
       raise IndexError(...)
2. After that, clamp start/end as in step 2 above before slicing.

What to change in exact lines
- Current (lines 32–34):
  if start < 0 or end < 0 or start > end or start > len(s) or end > len(s):
      raise IndexError(f"String index out of range: {end}")
  return s[start:end]
- Suggested replacement:
  if start < 0 or end < 0 or start > end:
      raise IndexError(f"String index out of range: {end}")
  start = min(max(start, 0), len(s))
  end   = min(max(end,   0), len(s))
  return s[start:end]

Why this fix is correct
- It prevents IndexError for cases where abbreviate is asked to slice beyond the string length (the failing test case).
- It preserves a check for clearly invalid inputs (negative indices or start > end).
- It makes slice_union behave in the way WordUtils.abbreviate expects: out-of-range indices are handled by clamping instead of throwing exceptions.
