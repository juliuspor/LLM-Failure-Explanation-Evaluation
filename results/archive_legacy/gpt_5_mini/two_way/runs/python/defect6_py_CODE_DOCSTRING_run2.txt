What happens
- The failure occurs at the if line:
  if x == int(x) and not math.isinf(x) and not math.isnan(x):
- When x is NaN (float('nan')) or +/-Inf (float('inf')), int(x) is evaluated and raises an exception (ValueError for NaN, OverflowError for Inf) before Python ever evaluates math.isinf(x) or math.isnan(x). That is why the code "fails" (raises) for those inputs.

Why it fails (root cause)
- The operands of the and expression are evaluated left-to-right, but Python does not reorder them to avoid exceptions; the leftmost expression x == int(x) requires calling int(x) immediately. The intent was to skip int(x) when x is NaN/Inf, but the current order does the int() conversion first, so you try to convert NaN/Inf to int and get an exception.

Where exactly
- The problematic expression is on the if line that tests "x == int(x) and not math.isinf(x) and not math.isnan(x)". The int(x) call is what throws (for NaN/Inf) and happens before the math checks.

How to reproduce quickly
- Call add_number(float('nan')) or add_number(float('inf')). You will get:
  - ValueError: cannot convert float NaN to integer (for NaN), or
  - OverflowError: cannot convert float infinity to integer (for Inf)

Precise fixes (pick one)
1) Check finiteness first (recommended)
   Replace the if with:
     if not math.isinf(x) and not math.isnan(x) and x == int(x):
   or better:
     if math.isfinite(x) and x == int(x):
   Reason: math.isfinite(x) is true only for non-Inf, non-NaN floats, so int(x) is only called when safe.

2) Avoid int(x) until after safety check
   You can write:
     if not math.isinf(x) and not math.isnan(x):
         if x == int(x):
             value = int(x)
             ...
   This makes the intent explicit and prevents int() from being called on NaN/Inf.

3) Use math.isfinite and eliminate redundant checks:
     if math.isfinite(x) and x == int(x):
         ...

Notes
- Do not try to rely on short-circuiting by reordering the same expression with int(x) first; int(x) must not be called on NaN/Inf at all.
- After applying one of the fixes above, the rest of the function will run as intended for integer-valued finite floats.
