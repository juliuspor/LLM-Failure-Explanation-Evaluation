What happened (root cause)
- In the "max middle" block you fetch the wrong data item. On this line you call:
  self.get_data_item(self._min_middle_index).get_period()
  but you meant to use self._max_middle_index. That is the single root cause.

Where it occurs (exact location)
- Inside the if block starting with:
  if self._max_middle_index >= 0:
- Specifically the line:
  max_middle_period = self.get_data_item(self._min_middle_index).get_period()
  (the next lines compute s/e and max_middle from max_middle_period).

Why that breaks things
- You compare the new middle to the middle of the period at _min_middle_index instead of at _max_middle_index, so _max_middle_index will be updated using the wrong reference value. That produces incorrect max-middle tracking (logical bug).
- If _min_middle_index is -1 or otherwise invalid while _max_middle_index >= 0, calling get_data_item(self._min_middle_index) can raise an IndexError (or produce an invalid period), so this bug can also produce runtime errors in some states.

What you should change (concrete steps)
1. Replace the wrong index with the correct one:
   Change
     max_middle_period = self.get_data_item(self._min_middle_index).get_period()
   to
     max_middle_period = self.get_data_item(self._max_middle_index).get_period()

2. Add a unit test that exercises updating _max_middle_index (insert periods with increasing/decreasing middles) to catch regression.

3. Optional hardening: validate indices before use or centralize the period-to-millis/middle calculation to avoid similar copy-paste mistakes. For example, add an assertion or guard:
   assert 0 <= self._max_middle_index < len(self._data_items)

Corrected snippet (only the changed block)
- Replace the entire "max middle" block with:
  if self._max_middle_index >= 0:
      max_middle_period = self.get_data_item(self._max_middle_index).get_period()
      if isinstance(max_middle_period, SimpleTimePeriod):
          s = max_middle_period.get_start_millis()
          e = max_middle_period.get_end_millis()
      else:
          s = int(max_middle_period.get_start().timestamp() * 1000)
          e = int(max_middle_period.get_end().timestamp() * 1000)
      max_middle = s + (e - s) // 2
      if middle > max_middle:

      self._max_middle_index = index

Why this fix solves it
- Using self._max_middle_index reads the correct existing item to compare against. That prevents incorrect updates and avoids the potential IndexError caused by accidentally using _min_middle_index.

That single change should fix both the logic error and the potential runtime failure.
