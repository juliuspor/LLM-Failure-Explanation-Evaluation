What happened (what failed)
- The code raised IndexError at line 36:
  raise IndexError(f"String index out of range: {index}")
  with index == 2.
- That check compares index against len(code_units) (line 35: if index < 0 or index >= len(code_units):).
- The caller passed index==2 thinking it was an index into the original Python string s (a character index).
- But code_units is a list of UTF‑16 code units created in to_utf16_code_units(s). For strings containing characters that become surrogate pairs in UTF‑16, len(code_units) can be smaller or larger than len(s). In this run len(code_units) was 2, so index 2 was out of range for code_units even though index 2 might be a valid character index in s.

Why it happened (root cause)
- The code mixes two different index spaces:
  - index is a character index into the Python string s (each Python str element is a Unicode code point).
  - code_units is a list of 16‑bit UTF‑16 code units (some code points become two code units).
- The boundary check at line 35 uses len(code_units) to validate index, but the index refers to characters in s, not positions in code_units. That mismatch caused a false out‑of‑range error.

- to_utf16_code_units(s: str) builds code_units (line ~10–line ~34).
- At line 35 the code does: if index < 0 or index >= len(code_units):
