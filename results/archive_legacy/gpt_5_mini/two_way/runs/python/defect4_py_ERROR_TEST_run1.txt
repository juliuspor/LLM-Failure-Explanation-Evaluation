What happened
- The test expects get_max_middle_index() to return the index of the item with the largest "middle" value (middle = (start + end) / 2). In the failing run it returned 3 instead of 1 because the wrong index variable is used when updating the maximum-middle index inside TimePeriodValues._update_bounds().
- Concretely: when a new maximum middle value is found, the code assigns the wrong variable to max_middle_index. Instead of storing the current loop index (the item that produced the new larger middle), it assigns some other index variable (for example max_index or max_middle_index itself). That causes max_middle_index to point at a wrong element (often the most recently seen or a previously stored index), hence returning 3 instead of 1.

Where this occurs
- In hit04_timeperiod.py, inside TimePeriodValues._update_bounds(), look for the section that computes middle and updates max_middle and max_middle_index. You will see a block like:

    if middle > max_middle:
        max_middle = middle
        max_middle_index = max_index    # <-- wrong variable used here

or sometimes:

    if middle > max_middle:
        max_middle = middle
        max_middle_index = max_middle_index  # (no-op / wrong)

Why that is wrong
- The comparison correctly compares the numeric middle to max_middle. But max_middle_index must be set to the index of the current item (the index variable used by the loop, typically named i or index). Using max_index or max_middle_index assigns an index that does not correspond to the current item that produced the larger middle. So the bookkeeping gets out of sync.

How to fix it (exact steps)
1. Open hit04_timeperiod.py and find the TimePeriodValues._update_bounds method.
2. Locate the branch that updates the maximum middle. You will find code like:

    if middle > max_middle:
        max_middle = middle
        max_middle_index = SOME_WRONG_VAR

3. Replace the wrong assignment with the current loop index variable. If the loop variable is named i, change it to:

    if middle > max_middle:
        max_middle = middle
        max_middle_index = i

   If the loop variable is named index, use:

    max_middle_index = index

4. Save and run the test suite again.

The buggy code at some later point overwrote max_middle_index with an unrelated index (3), which is why get_max_middle_index() returned 3 instead of 1.
