What happened (where)
- The ValueError is raised at line 246: the code checks if locale_str[5] != '_' and then raises ValueError(f"Invalid locale format: {locale_str}").
- The failing input is "fr__POSIX". That string reached this check and triggered the exception.

Why it happens (root cause)
- The parsing code assumes the locale string always has a non-empty country code at positions 3–4 (it slices locale_str[3:5]) and that, if there is a variant, the variant separator '_' will be at index 5.
- For "fr__POSIX" there are two underscores in a row. Indexes of "fr__POSIX" are:
  - 0='f', 1='r', 2='_', 3='_', 4='P', 5='O', ...
- Because the country part is empty, the characters are shifted: the code reads locale_str[3:5] as "_P" (not a valid country) and then checks locale_str[5] expecting '_' but finds 'O'. That makes the condition at line 245 true and causes the ValueError at line 246.
- In short: the code only supports formats like "xx_YY" or "xx_YY_variant". It does not handle the valid-but-different form with an empty country and a variant separated by a double underscore ("language__variant"), so the index-based parsing is misaligned and rejects "fr__POSIX".

How to fix it (concrete steps)
1. Stop relying on fixed index positions. Parse on underscores so empty fields are preserved:
   - Use parts = locale_str.split('_', 2) to split into at most three parts: [language], [country], [variant].
   - This keeps an empty country as an empty string (parts[1] == '') and places the variant in parts[2].
2. Validate the parts explicitly:
   - Ensure parts[0] (language) is present and length==2.
   - If parts has length >= 2, accept parts[1] even if empty (if your Locale type allows no country).
   - If parts has length == 3, treat parts[2] as variant (may be required to be non-empty).
3. Replace the index-based branches at lines 243–247 with this split-based logic and construct Locale(language=parts[0], country=parts[1] or None, variant=parts[2] if present).
