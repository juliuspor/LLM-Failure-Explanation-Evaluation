What happened (where it failed)
- The failing check is in DateTimeZone.for_offset_hours_minutes, lines 396-397:
  if minutes_offset < 0 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")
- That condition rejects any negative minutes (minutes_offset < 0).
- Several tests pass negative minute values (for example: (0, -15), (-2, -15), and the mixed-sign tests). Those tests expect negative minutes to be allowed in the range -59..+59 or to be handled according to the hour sign rules. Because the code rejects negative minutes unconditionally, tests like test_zero_hours_negative_minutes, test_negative_hours_negative_minutes, and test_mixed_signs_invalid fail.

Why it failed (root cause)
- The code enforces minutes to be in [0, 59] only, but the specification/tests expect minutes to be allowed in [-59, +59] (and to be combined with the hour sign in a specific way).
- The variable involved is minutes_offset. The validation at lines 396-397 is the wrong range check: it should allow negative minutes down to -59.

Exactly which tests break
- test_zero_hours_negative_minutes: calls DateTimeZone.for_offset_hours_minutes(0, -15) — currently raises ValueError at the minutes check.
- test_negative_hours_negative_minutes: (-2, -15) — same rejection.
- test_mixed_signs_invalid: the part expecting (-2, 15) to work may pass or fail depending on later logic, but the part expecting (2, -15) to raise relies on different validation; the immediate problem is that negative minutes are blocked even where they should be permitted.

   - Replace the current check (lines 396-397) with:
     if minutes_offset < -59 or minutes_offset > 59:
         raise ValueError(f"Minutes out of range: {minutes_offset}")
2. Enforce/implement sign rules consistent with the docstring/tests:
   - If hours > 0 and minutes < 0: raise ValueError (mixed signs where hour positive and minute negative is invalid).
