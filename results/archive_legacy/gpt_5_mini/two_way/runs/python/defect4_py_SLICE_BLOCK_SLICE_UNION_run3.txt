What happens
- At line 253 you initialize self._min_start_index = -1.
- In the code that processes each time period (lines 331–349) you intend to set self._min_start_index to the index of the period with the smallest start time.
- You correctly compare start to the current min when self._min_start_index >= 0 (lines 340–347).
- But immediately after that comparison you unconditionally execute self._min_start_index = index (line 349). That overwrites whatever you just computed and sets _min_start_index to the current index every time.

Why it fails (root cause)
- The unconditional assignment at line 349 means _min_start_index always ends up as the last processed index, not the index of the minimum start time. The comparison code (lines 340–347) never has a lasting effect because it is always clobbered right afterward.
- In the first iteration, when _min_start_index is -1, you also never set the min inside the intended branch because the intended assignment for the "first item" case is the one at line 349 — but it is misplaced and executed for every iteration.

Exactly where to change it
- Replace the unconditional assignment at line 349 with an else branch so the current index is only set when there is no previous minimum (i.e., when _min_start_index < 0).
Concretely, change this fragment:

 340 |         if self._min_start_index >= 0:
 341 |             min_start_period = self.get_data_item(self._min_start_index).get_period()
 342 |             if isinstance(min_start_period, SimpleTimePeriod):
 343 |                 min_start = min_start_period.get_start_millis()
 345 |                 min_start = int(min_start_period.get_start().timestamp() * 1000)
 346 |             if start < min_start:
 347 |                 self._min_start_index = index
 349 |             self._min_start_index = index

to:

 340 |         if self._min_start_index >= 0:

 342 |             if isinstance(min_start_period, SimpleTimePeriod):

Why this fixes it
- Now, on the first processed period (_min_start_index == -1), the else branch sets the initial minimum to the current index.
- On subsequent periods, you only update _min_start_index when start < min_start, so it will correctly track the index with the minimum start time.

No other changes are required, assuming get_data_item(...) and the start extraction logic are correct.
