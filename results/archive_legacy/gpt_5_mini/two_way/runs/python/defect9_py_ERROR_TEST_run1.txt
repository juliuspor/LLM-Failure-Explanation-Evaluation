What happened (root cause)
- The TypeError comes from calling len(...) on a None value inside your replace_each implementation.
- The failing tests pass a replacements list that contains None entries (e.g. replace_list = ["c", None]). Your code assumes every entry in replace_list is a string and immediately does something like len(repl) (or uses repl in a string operation) without checking for None.
- When the code reaches the replacement for which repl is None, Python raises TypeError: object of type 'NoneType' has no len().

Where it occurs
- In hit09_stringutils.py, inside the replace_each function. Look for the loop that iterates search_list / replace_list and the line that fetches the replacement:
  repl = replace_list[i]
  ... then a call like len(repl) or using repl in slicing/concatenation.
- That len(repl) (or any use that expects a string) is the exact place the exception originates.

Why that causes the tests to fail
- The tests expect null-safety inside the arrays: a None replacement should be treated as “no replacement” (the Java Apache Commons StringUtils behavior). Instead, your code tries to measure or manipulate the None and crashes.

How to fix it (concrete steps)
1. Open hit09_stringutils.py and find replace_each(text, search_list, replace_list).
2. In the loop where you do:
     repl = replace_list[i]
   add a guard before any call that assumes repl is a string:
     if repl is None:
         # skip this replacement (no-op), or treat as "leave text unchanged for this search"
         continue
   or explicitly set repl to a safe value if your algorithm expects a string (but note tests expect a no-op, not deletion):
     if repl is None:
         repl = None  # and then handle by skipping usage below
3. Make sure you do not call len(repl) or perform string operations on repl unless repl is not None.
4. Also ensure you handle None inside search_list entries similarly if you allow them (the tests only show None in replace_list).
5. Run the tests.

Minimal patch example (conceptual — adjust to your function structure)
- If your loop looks like this:

Why this matches the expected behavior
- Apache Commons StringUtils.replaceEach treats a null replacement as "do not replace". Skipping the replacement when repl is None reproduces the expected results:
  - replace_each("aba", ["a"], [None]) -> "aba"
  - replace_each("aba", ["a","b"], ["c", None]) -> "cbc"
