What happened (where it fails)
- In abbreviate() you adjust upper with these lines (around the top of the method):
  - if upper == -1 or upper > len(text): upper = len(text)
  - if upper < lower: upper = lower
- When called with text length 10, lower=15, upper=20 (the failing test):
  1. The first if sets upper = len(text) -> upper becomes 10.
  2. The second if sees upper < lower (10 < 15) and sets upper = lower -> upper becomes 15.
- That leaves upper = 15 while text length is 10. Later the code calls substring_java(text, 0, upper). Because upper > len(text), substring_java is asked to slice past the end of the string and raises an error (or otherwise returns an incorrect result). The code never returns early for the case "lower is beyond text length", so we end up with an invalid upper.

Why it fails (root cause)
- The line "if upper < lower: upper = lower" blindly sets upper to lower even when lower > len(text). That allows upper to end up greater than len(text). The method then tries to use upper as a slice index into text (substring_java(text, 0, upper)), which is invalid. The logic intended to enforce upper >= lower inadvertently violates upper <= len(text).

Exact variables & locations to note
- Variables: lower, upper, text
- Problem lines:
  - "if upper == -1 or upper > len(text): upper = len(text)"
  - "if upper < lower: upper = lower"
  - use site: "result = substring_java(text, 0, upper)" (and similar later calls that use upper)

How to fix it (concrete steps)
Option A — return early when lower is beyond the text (closest to Java behavior):
- Add this check after handling None/empty and after normalizing upper to len(text):
  - if lower >= len(text):
      return text
  This matches the intended behavior: if the lower bound is beyond the string, return the whole string unchanged.

Option B — clamp upper correctly (alternate):
- Instead of setting upper = lower unconditionally, compute upper = max(lower, min(upper, len(text))). Concretely:
  - if upper == -1: upper = len(text)
  - else: upper = min(upper, len(text))
  - if lower >= len(text): return text
  - if upper < lower: upper = lower  # now safe because lower < len(text)
  This ensures upper never exceeds len(text) and returns early when lower is out of range.

Suggested minimal patch (Option A, exact replacement):
- After
    if upper == -1 or upper > len(text):
        upper = len(text)
  insert:
    if lower >= len(text):
        return text

Why this fix works
- It prevents substring_java from being invoked with an upper index greater than the string length. It also matches the expected behavior for the test case: WordUtils.abbreviate("0123456789", 15, 20, None) should return the original string.

Summary
- Root cause: "if upper < lower: upper = lower" can push upper past len(text) when lower > len(text), causing invalid substring indices.
- Fix: either return text when lower >= len(text) or clamp upper properly before any substring calls.
