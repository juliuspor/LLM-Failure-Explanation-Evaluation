What happened (where it fails)
- At line 396 the code rejects any negative minutes unconditionally:
  if minutes_offset < 0 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")
  This throws for inputs that the docstring says are allowed, e.g. hours_offset = -2, minutes_offset = -15 (line 396).

- The sign arithmetic that computes a total minute offset is wrong at lines 402 and 404:
  402: minutes_offset = hours_in_minutes - minutes_offset
  404: minutes_offset = hours_in_minutes + minutes_offset
  Those branches invert the sign in the wrong case. For example, for hours_offset = -2 and minutes_offset = 15:
    hours_in_minutes = -120
    correct total minutes = -120 + 15 = -105  (result -02:15)
    but the code on line 402 does -120 - 15 = -135 (wrong)
  So the code produces the wrong offset sign/value for negative hours with positive minutes.

Why it fails (root cause)
- The validation at line 396 is too strict: it forbids any negative minutes even though the docstring allows negative minutes when hours are zero or negative.
- The sign handling (lines 402/404) is implemented incorrectly: the subtraction is used where an addition is required. In fact the correct combined minutes is simply hours_in_minutes + minutes_offset in the allowed cases; the existing code sometimes subtracts and thus flips the sign.

Exactly where (lines and variables)
- Line 396: rejects minutes_offset < 0 unconditionally — prevents valid inputs (minutes_offset) when hours_offset <= 0.
- Line 402: minutes_offset = hours_in_minutes - minutes_offset — incorrect sign for negative hours + positive minutes.
- Line 404: minutes_offset = hours_in_minutes + minutes_offset — used in the other branch but the branching logic makes 402/404 apply incorrectly.
- Line 405: offset = FieldUtils.safe_multiply(minutes_offset, DateTimeConstants.MILLIS_PER_MINUTE) — uses the incorrectly computed minutes_offset to produce a wrong millisecond offset (or an exception if earlier checks reject).
