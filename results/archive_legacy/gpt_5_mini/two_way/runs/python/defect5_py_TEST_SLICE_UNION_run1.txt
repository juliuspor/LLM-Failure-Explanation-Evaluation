Why it fails (root cause)
1. At line 336 the code does:
   inferred_type = type(array)
   But when array is a Python list, type(array) is the list class (e.g. <class 'list'>) — not the element type. So the code is using the container's type instead of the element type.
2. When both array and element are None (your test case), the code falls through and sets:

   (line 340).
3. Immediately after (line 346) the code does:

       raise TypeError(...)
   Since inferred_type == object and you passed expected_type=str, that condition is true and a TypeError is raised.
4. The net result: when both array and element are None the function treats the element type as “unknown/object” and then refuses to proceed because expected_type is non-object — even though the caller explicitly asked for expected_type=str and would reasonably expect [None] to be returned.

Why the logic is wrong (explicit)
- The code conflates the array object type with the element type (type(array) vs type(array element)).
- The code treats 'unknown' (object) as an error when expected_type is provided, instead of using expected_type to decide the resulting array element type.
- The offending checks are at the exact lines you showed: inferred_type calculation at 336 and 340, and the raising decision at 346–347.
