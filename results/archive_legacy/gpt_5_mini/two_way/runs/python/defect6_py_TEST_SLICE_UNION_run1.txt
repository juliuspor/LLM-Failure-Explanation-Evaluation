What happened (where, exactly)
- The failure occurs in the numeric formatting branch starting at line 176:
  - line 176: if x == int(x) and not math.isinf(x) and not math.isnan(x):
  - line 177: value = int(x)
  - later (line 192) self.add(str(value)) is used for whole numbers.
- For x == -0.0 Python treats -0.0 == 0.0 as True. That makes the code enter the integer branch.
- At line 177 value = int(x) converts -0.0 to the integer 0, which loses the negative sign.
- As a result the code emits "0" (line 192) instead of "-0.0". This is why the test expecting "-0.0" fails.

Why it happens (root cause)
- The test expects the sign of negative zero to be preserved as "-0.0".
- The check x == int(x) collapses -0.0 to 0.0 because IEEE negative zero compares equal to positive zero in Python.
- Converting to int removes the sign information; nothing later in this branch checks for negative zero, so the sign is lost permanently.

Precise place to change
- Insert a negative-zero check before the integer branch at or before line 176. Use an explicit negative-zero test (do not rely on x == int(x)).
- The repository already has a utility CodeConsumer.is_negative_zero(x) (your tests exercise it). Use that function or an equivalent test (e.g. math.copysign(1.0, x) < 0 and x == 0.0).

Concrete fixes (step-by-step)
1. At the top of the number-formatting method, before line 176, add:
   - if CodeConsumer.is_negative_zero(x):
       self.add("-0.0")
       return
   This guarantees -0.0 is emitted correctly and avoids entering the integer branch.

2. Alternatively, inside the integer branch (after line 177), handle the special zero case:
   - if value == 0 and math.copysign(1.0, x) < 0:
       self.add("-0.0")
       return
   This keeps the existing integer-branch logic but preserves negative-zero sign.

3. Do not rely on str(x) to fix this later in the method because the integer branch already emitted str(value) and returned; you must intercept before that emission.

Why these fixes work
- CodeConsumer.is_negative_zero or math.copysign detects the sign bit independently of numeric equality, so you can detect negative zero even though -0.0 == 0.0.
- Emitting "-0.0" before converting to int preserves the sign in the output exactly as the test expects.

This is explicit, clear, and prevents the lost-sign conversion at line 177.
