What happened (where it fails)
- At lines 396–397 the code unconditionally rejects negative minute offsets:
  if minutes_offset < 0 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")
- That check causes the function to fail whenever minutes_offset is negative, even in cases the docstring explicitly allows (for example (0, -15) or (-2, -15)).

Why it fails (root cause)
- The validation treats the minutes range as 0..59 only. The docstring allows minutes in -59..+59, and also allows negative minutes in some sign combinations (specifically when hours is zero or negative). The code enforces the wrong sign rule: it forbids negative minutes regardless of hours_offset.
- In short: minutes are being validated without considering hours_offset, so valid inputs per the doc are rejected.

How to fix it (concrete steps and exact conditions)
1. Allow minutes in the full allowed magnitude range:
   - Replace the current range check with one that accepts -59..+59:
     if minutes_offset < -59 or minutes_offset > 59:
         raise ValueError(f"Minutes out of range: {minutes_offset}")
2. Enforce the sign combination rule described in the docstring:
   - Add a check that forbids a positive hour with negative minutes:
     if hours_offset > 0 and minutes_offset < 0:
         raise ValueError(f"Positive hours with negative minutes: {hours_offset}, {minutes_offset}")
   - (No extra check is needed for negative hours with negative minutes because the doc allows both (-2, 15) and (-2, -15) and the offset computation should normalize sign accordingly.)
3. Keep any existing checks that validate hours range and total absolute offset limit (e.g., +/- 23:59:59.000), unchanged or implement them after the above checks.

Concrete replacement example (pseudo-code around lines 396–397):
  # allow minutes in -59..+59
  if minutes_offset < -59 or minutes_offset > 59:

- The first change accepts negative minute values that are within the documented magnitude limit.
- The second change preserves the documented prohibition of a positive hour with a negative minute (e.g., (2, -15) should still raise ValueError).
- Together they match the docstring combinations and stop rejecting valid inputs.
