What happened (what fails)
- The function replace_each (line 28) crashes with a TypeError when either search_list[i] or replacement_list[i] is None.
- The crash happens at the second loop where the code computes a length difference:
  - Line 81–82: greater = len(replacement_list[i]) - len(search_list[i])
- If replacement_list[i] or search_list[i] is None, len(...) raises TypeError: object of type 'NoneType' has no len(), so the function crashes even though those pairs were supposed to be ignored.

Why it happens (root cause)
- Earlier code sets search_length = len(search_list) (line 52) and iterates i over range(search_length) (line 59 and again line 81) using the original lists and index i.
- The code assumes every entry search_list[i] and replacement_list[i] is a string when it later calls len(...) on them.
- There is no guard in the second loop (lines 81–82) to skip entries that are None. Even if some earlier logic intended to ignore None entries, that intent is lost because the code still indexes the original lists and unconditionally calls len(...) on the elements.
- In short: the code indexes pairs by position and later performs operations that require non-None values, but it never checks for None before using len().

- Line 52: search_length = len(search_list) — establishes the loop range from the original list length.
- Line 59: for i in range(search_length): — first loop (likely does some setup).
- Line 81–82: for i in range(search_length): ... greater = len(replacement_list[i]) - len(search_list[i]) — here the code calls len() on elements that may be None, causing the crash.
- Variables involved: search_list, replacement_list, i, greater.

How to reproduce quickly
- Call replace_each with search_list = ["a", None] and replacement_list = ["b", "c"] (or any None in either list). The code will reach the length computation on the None entry and raise TypeError.

Precise fixes (step-by-step, unambiguous)
Choose one of these concrete fixes and apply it in replace_each:

- Before computing lengths at lines 81–82, add an explicit guard:
  - if search_list[i] is None or replacement_list[i] is None:
      continue
- This keeps using the index-based approach but prevents calling len() on None.

Which to pick
- If the intention is to ignore pairs that contain None: use option (1) or (2). Option (2) is cleaner and removes multiple index checks.
- If the intended behavior is to treat None as empty replacement/search: use option (3).

Final note
- The bug is not a mysterious runtime failure — it is a simple missing None check before calling len() on search_list[i] or replacement_list[i]. Fix by guarding or pre-filtering the lists as shown.
