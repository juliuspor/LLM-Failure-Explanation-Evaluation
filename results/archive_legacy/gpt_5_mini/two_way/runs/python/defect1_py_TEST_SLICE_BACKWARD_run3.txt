What failed
- The tests call DateTimeZone.for_offset_hours_minutes with negative minutes (e.g. (0, -15), (-2, -15)) and expect the method to accept them (and treat sign correctly).
- The implementation currently rejects negative minute values unconditionally at lines 396–397:

  396 |         if minutes_offset < 0 or minutes_offset > 59:
  397 |             raise ValueError(f"Minutes out of range: {minutes_offset}")

  That condition raises ValueError whenever minutes_offset is negative, so calls like for_offset_hours_minutes(0, -15) or for_offset_hours_minutes(-2, -15) fail there. The failure occurs inside DateTimeZone.for_offset_hours_minutes when it validates the minutes parameter (variable minutes_offset), not in the test code.

Why it fails (root cause)
- The code treats minutes_offset as valid only in [0, 59]. The tests (and the documented behavior you expect) allow minutes in the range -59..+59 and require special handling of sign combinations (e.g. negative hours + positive minutes should be treated as a negative total offset).
- So the direct cause is the wrong validation range (minutes_offset < 0) in that if statement. Because minutes_offset is rejected before any sign-handling logic runs, several tests fail.
