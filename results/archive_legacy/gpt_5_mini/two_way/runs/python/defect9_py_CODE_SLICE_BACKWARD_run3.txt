What happened (where it fails)
- The failure occurs in the loop that estimates "increase" starting at the slice lines 81–82:
  for i in range(search_length):
      greater = len(replacement_list[i]) - len(search_list[i])
- If any element of search_list or replacement_list is None, calling len(...) on that element raises:
  TypeError: object of type 'NoneType' has no len()
- Earlier in the function you explicitly guard against None/empty items when locating matches (the for-loops that call text.find), but that same null-safety is missing in the "increase" loop. So the function can proceed past the initial checks (they only check the lists themselves and lengths, not their elements) and then crash here when it tries to take len() of a None element.

Why it happens (root cause)
- The root cause is that the code assumes every element in search_list and replacement_list is a non-None string when computing lengths for the buffer size guess. It does not perform the same null/empty checks used elsewhere, so a None element causes len(None) and a TypeError.

Exactly what to change (step-by-step fix)
1. At the loop that computes increase (lines ~81–82), skip elements that the rest of the function already treats as ignored:
   - skip if search_list[i] is None
   - skip if len(search_list[i]) == 0
   - skip if replacement_list[i] is None

2. Replace the loop with this safe version:

    for i in range(search_length):
        if (
            search_list[i] is None
            or len(search_list[i]) == 0
            or replacement_list[i] is None
        ):
            continue
        greater = len(replacement_list[i]) - len(search_list[i])
        if greater > 0:
            increase += 3 * greater  # assume 3 matches

3. (Optional) You can also use a helper predicate or reuse the same condition used later in the function to avoid duplication.

Why this fix works
- The fix mirrors the null/empty checks used elsewhere in the function, preventing len() from being called on None. That removes the TypeError and preserves the intended heuristic for pre-sizing the result buffer only for valid replacements.

Notes
- This is a correctness bug, not an algorithmic one; no other lines need to change to avoid this specific crash.
