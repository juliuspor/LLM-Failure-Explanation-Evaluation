What happened (where & why)

- At line 52 the code sets search_length = len(search_list). That determines how many indices the later loops will iterate.
- At lines 81–85 the code computes an "increase" estimate:

  > greater = len(replacement_list[i]) - len(search_list[i])   (line 82–83)

  This line tries to call len(...) on replacement_list[i] and search_list[i] unconditionally.
- If either search_list[i] or replacement_list[i] is None, calling len(None) raises a TypeError and the function crashes.

So the root cause is: the code iterates up to search_length and dereferences search_list[i] and replacement_list[i] with len(...) before it ever checks for None. The None checks exist later (lines 102–105), but they happen after those len(...) calls. In short: the code checks for None too late and performs len(...) on None at lines 82–83.

Other related places that assume non-None before checking
- The loop at lines 108–112 calls text.find(search_list[i], start). That is safe only if search_list[i] is not None; the code does guard against None at lines 102–105, but that guard must run before any access like text.find or len. The immediate crash reported is from the len(...) at 82–83.

Concrete, step-by-step fixes

1) Do not assume search_list and replacement_list contain no None entries. Skip those entries when you iterate:
   - Make search_length reflect the usable pairs, or loop differently. Example options:
     - Set search_length = min(len(search_list), len(replacement_list)) (avoid IndexError if lists differ).
     - In every loop that accesses list[i], check first:
       if search_list[i] is None or replacement_list[i] is None: continue

2) Fix the increase loop (lines 81–85) so it never calls len(...) on None. Change it to:

   for i in range(search_length):
       if search_list[i] is None or replacement_list[i] is None:
           continue
       greater = len(replacement_list[i]) - len(search_list[i])

3) Ensure the code paths that call text.find(...) (lines 108–112) only do so when search_list[i] is not None and not empty. If you already set no_more_matches_for_repl_index[i] for None entries earlier, the find call is safe; otherwise add an explicit guard before calling find.

4) (Optional robustness) Prefer iterating over pairs instead of indexing:

Why these fixes work

- They prevent len(None) and text.find(None, ...) by skipping None entries before any access.
- They keep the same intended behavior ("replacements should be ignored when either side is None") by continuing without attempting replacement work for those indices.
- Using min(len(...)) or zip avoids indexing past the shorter list.
