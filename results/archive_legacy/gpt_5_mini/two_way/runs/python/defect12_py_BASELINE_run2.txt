What happened
- The IndexError is raised when substring_java(text, 0, upper) is called with upper > len(text). The slice function checks end > len(s) and raises IndexError (see slice_block lines 32–34).
- In the failing test you call WordUtils.abbreviate("0123456789", 15, 20, None): text length = 10, lower = 15, upper = 20.

Where it goes wrong (line numbers refer to your function):
1. At the start you do:
   if upper == -1 or upper > len(text):
       upper = len(text)
   With the test values this sets upper = 10.
2. Immediately after you do:
   if upper < lower:
       upper = lower
   Now upper becomes 15 (lower), which is greater than len(text).
3. Later you call substring_java(text, 0, upper). Because upper is 15 > len(text), substring_java (the slice implementation) raises IndexError ("String index out of range: 15").

Why this is the root cause
- The code first clamps upper down to len(text), then inexplicably reassigns upper to lower when upper < lower, undoing the clamp and creating an out-of-range end index.
- The original Java behavior (and the test expectation) is to return the original text when lower > len(text). Your code never handles lower > len(text) early, so it ends up passing an invalid end index to substring_java.

Concrete fixes (pick one)

A — Early-return when lower beyond text length (recommended, matches Java/Lang-45):
Add this check near the top, before manipulating upper:
    if lower > len(text):
        return text
This prevents any out-of-range slice and produces the expected "0123456789" for the test.

B — Reorder/clamp correctly:
Ensure upper is finally clamped to len(text) before calling substring_java. For example:
    if upper == -1 or upper > len(text):
        upper = len(text)
    if upper < lower:
        # Instead of setting upper = lower, keep upper at len(text) or otherwise handle
        upper = len(text)
But this changes semantics — the clearer correct behavior is A.

Exact minimal patch (apply near top, before the upper/lower logic):
    if lower > len(text):
        return text

This fixes the IndexError and matches the expected behavior in your test suite.
