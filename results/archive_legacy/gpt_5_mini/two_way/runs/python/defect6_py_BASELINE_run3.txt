What happened (root cause)
- At line 176 you test if x == int(x). For x == -0.0 that expression is True because int(-0.0) == 0 and Python treats -0.0 == 0.0 as equal.
- Then at line 177 you do value = int(x) which converts -0.0 to the integer 0 and loses the sign.
- Later (lines 191–192) you call self.add(str(value)) which produces "0". So the negative sign of -0.0 is thrown away before you ever output it, producing "0" instead of "-0.0".

Where it occurs
- The wrong behavior originates at the integer-handling branch starting at line 176. The negative-zero case must be detected before converting to int(x) (line 177) or before taking the integer branch.

Why this matters
- IEEE floats have a distinct negative zero bit pattern even though -0.0 == 0.0 compares True. The intent in tests is to preserve that sign by printing "-0.0". Your integer-special-case pathway collapses -0.0 to 0.

Concrete fixes (clear steps and exact code to change)
Option A — detect negative zero first (recommended):
1. Add a negative-zero check before the integer branch. For example, just above line 176 insert:

    if self.is_negative_zero(x):
        self.add("-0.0")
        return

2. Ensure is_negative_zero is implemented, e.g.:

    @staticmethod
    def is_negative_zero(x: float) -> bool:
        return x == 0.0 and math.copysign(1.0, x) < 0.0

This uses math.copysign to detect the sign bit without relying on comparisons that collapse sign.

Option B — exclude negative zero from the integer branch:
Change the condition at line 176 from

    if x == int(x) and not math.isinf(x) and not math.isnan(x):

to

    if x == int(x) and not math.isinf(x) and not math.isnan(x) and not self.is_negative_zero(x):

and implement is_negative_zero as above. If negative zero is excluded from that branch, control will fall through to the final else where you do self.add(str(x)), which prints "-0.0" as desired.

Why these fixes work
- They detect the distinct negative-zero value before you convert to int and lose the sign bit. math.copysign inspects the sign without changing the numeric equality behavior.
