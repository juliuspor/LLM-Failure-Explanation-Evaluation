What happened (what failed)
- The code raises IndexError from code_point_at(...). The error comes from the check at lines 35–36:
  - 35 | if index < 0 or index >= len(code_units):
  - 36 |     raise IndexError(f"String index out of range: {index)")
- The call that triggers that check is inside the loop at the end of translate:
  for pt in range(consumed):
      pos += char_count(code_point_at(code_units, pos))

Why it failed (root cause)
- translate_codepoint returned a positive integer named consumed. By contract this number counts codepoints (not UTF-16 code units / chars).
- The loop runs consumed times and calls code_point_at(code_units, pos) on every iteration.
- But pos is advanced inside the loop by char_count(...). After some iterations pos can become equal to length (past the last valid index). On the next iteration the code still calls code_point_at(code_units, pos) and that call fails the index check (pos >= len(code_units)) and raises IndexError.
- In short: the code calls code_point_at with pos == len(code_units) (out of bounds) because it advances pos inside the loop and still tries to read another codepoint for the next iteration.

Where it happened (exact variables / lines)
- Variable: pos.
- Function: code_point_at(code_units, pos).
- The failing location: the for pt in range(consumed) loop at the end of translate (the three lines shown); the IndexError originates from the bounds check at lines 35–36 in code_point_at.

How to fix it (concrete steps)
Pick one of these precise fixes depending on intended semantics:

Option A — stop calling code_point_at when pos is out of bounds
- Replace the for loop with a loop that checks pos before calling code_point_at:
  for pt in range(consumed):
      if pos >= length:
          break
      pos += char_count(code_point_at(code_units, pos))
- This avoids the out-of-range call and keeps pos <= length.

Option B — advance pos safely using the consumed value (preferred if consumed counts codepoints)
- If consumed is the number of codepoints consumed starting at pos, compute advancement without calling code_point_at for an out-of-range pos:
  for pt in range(consumed):
      # get the codepoint only if there's data left
      if pos >= length:
          break
      pos += char_count(code_point_at(code_units, pos))

Option C — change the contract so translate_codepoint returns number of code units consumed
- If translate_codepoint returned number of UTF-16 code units consumed (1 or 2), you can simply do:
  pos += consumed
  (remove the per-codepoint loop entirely)
- Use this if translators are already handling surrogate pairs and should report units consumed, not codepoints.

Minimal patch I recommend (Option A) — change the loop to:

      if pos >= length:
          break
      pos += char_count(code_point_at(code_units, pos))

This explicitly prevents calling code_point_at with pos == length and fixes the IndexError while preserving the existing contract that consumed measures codepoints.
