What happened (where and what failed)
- The error occurs inside Paint.get_paint at the division on line 71:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- If self._upper_bound equals self._lower_bound, the denominator (self._upper_bound - self._lower_bound) is zero. That causes a ZeroDivisionError at line 71 before the code ever reaches the range check on lines 74–75.
- Because the division never completes, you never get to the ValueError branch; the program crashes with ZeroDivisionError at get_paint line 71.

Why it happened (root cause)
- You allow creating or using a Paint where lower_bound == upper_bound.
- The code assumes a non-zero range when computing the normalized position (value − lower_bound) / (upper_bound − lower_bound). When the range is zero, the expression is undefined => runtime exception.

How to reproduce reliably
- Create a Paint with equal bounds (for example via __init__(lower_bound=0.5, upper_bound=0.5)) or call get_paint when both bounds are equal:
  p = Paint(0.5, 0.5)
  p.get_paint(0.5)
- That call crashes with ZeroDivisionError at the line shown.

Concrete fixes (step-by-step changes you can apply)
Option A — Validate in __init__ and refuse invalid ranges
1. In __init__ (lines ~30–44) add a check:
   if upper_bound == lower_bound:
       raise ValueError("upper_bound must not equal lower_bound")
2. This prevents ever creating a Paint with zero-length range, so line 71 will not divide by zero.

Option B — Handle the zero-range case in get_paint
1. At the start of get_paint (before line 71) add:
   if self._upper_bound == self._lower_bound:
       # decide expected behavior: return 0 or 255 or a middle value
       return (0, 0, 0)  # or (255,255,255) or (128,128,128) depending on spec
2. This avoids the division and makes behavior explicit when bounds are equal.

Option C — Clamp/guard the denominator and final g value
1. Compute safe_range = self._upper_bound - self._lower_bound
2. If safe_range == 0, handle as in Option B; otherwise compute ratio.
3. After computing g, clamp it into [0,255] before the check:
   g = max(0, min(255, int(...)))
4. Remove or keep the ValueError depending on whether you prefer clamping or strict errors.

Which to choose
- If equal bounds represent invalid input: use Option A (fail early in __init__).
- If equal bounds can be valid and you need defined output: use Option B (explicit return).
- If you want to be permissive and avoid exceptions for small floating errors: clamp as in Option C.
