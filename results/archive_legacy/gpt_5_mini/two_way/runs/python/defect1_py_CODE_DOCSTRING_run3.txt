What happens
- The input validation at lines
  if minutes_offset < 0 or minutes_offset > 59:
      raise ValueError(f"Minutes out of range: {minutes_offset}")
  rejects any negative minutes. That contradicts the docstring which allows negative minutes when hours_offset is zero or negative (for example (0, -15) and (-2, -15) must be allowed). As a result calls like DateTimeZone.for_offset_hours_minutes(-2, -15) or (0, -15) fail immediately with "Minutes out of range".
- The combination logic that builds the total minutes is also wrong. These lines:
  hours_in_minutes = hours_offset * 60
  if hours_in_minutes < 0:
      minutes_offset = hours_in_minutes - minutes_offset
  else:
      minutes_offset = hours_in_minutes + minutes_offset
intend to combine hours and minutes but produce the wrong sign when minutes_offset is negative. Example:
  hours_offset = -2, minutes_offset = -15
  hours_in_minutes = -120
  minutes_offset becomes -120 - (-15) = -105 (i.e. -1:45) — but the correct total is -135 minutes (-2:15). The code subtracts the negative minutes instead of subtracting the absolute minutes.

Where this occurs
- Validation failure occurs at the minutes validation if-block (the lines shown above).
- Wrong arithmetic occurs in the try block where hours_in_minutes is combined with minutes_offset (the if hours_in_minutes < 0 branch).

Why this causes the observed failures
- Negative minutes are forbidden by validation so allowed inputs per the docstring are rejected.
- Even if you removed the strict negative check, the combination logic produces incorrect offsets for negative hours/minutes pairs, yielding the wrong total minutes (and therefore wrong milliseconds / wrong time zone).

How to fix it (specific steps and example code)
1. Change the minutes validation so it allows negative minutes but still enforces magnitude <= 59:
   if minutes_offset < -59 or minutes_offset > 59:
       raise ValueError(f"Minutes out of range: {minutes_offset}")
2. Enforce the rule that positive hours may not have negative minutes (per docstring). Add:
   if hours_offset > 0 and minutes_offset < 0:
       raise ValueError("Positive hours cannot have negative minutes")
3. Compute total minutes with explicit sign handling (do not reuse minutes_offset for the combined total). For example:
   hours_in_minutes = hours_offset * 60
   if hours_offset < 0:
       total_minutes = hours_in_minutes - abs(minutes_offset)
   elif hours_offset > 0:
       total_minutes = hours_in_minutes + minutes_offset  # minutes_offset guaranteed >= 0 here
   else:  # hours_offset == 0
       total_minutes = minutes_offset
4. Multiply total_minutes by MILLIS_PER_MINUTE and proceed:
   offset = FieldUtils.safe_multiply(total_minutes, DateTimeConstants.MILLIS_PER_MINUTE)

Concrete replacement snippet (lines to use in place of the validation + combination code)
    if minutes_offset < -59 or minutes_offset > 59:
        raise ValueError(f"Minutes out of range: {minutes_offset}")
    if hours_offset > 0 and minutes_offset < 0:
        raise ValueError("Positive hours cannot have negative minutes")
    hours_in_minutes = hours_offset * 60
    if hours_offset < 0:
        total_minutes = hours_in_minutes - abs(minutes_offset)
    elif hours_offset > 0:
        total_minutes = hours_in_minutes + minutes_offset
    else:
        total_minutes = minutes_offset
    offset = FieldUtils.safe_multiply(total_minutes, DateTimeConstants.MILLIS_PER_MINUTE)

This preserves the documented behavior:
- (2, 15) => +135 minutes
- (2, -15) => ValueError (positive hours with negative minutes)
- (0, -15) => -15 minutes
- (-2, -15) => -135 minutes

Summary
- Root causes: (1) minutes validation wrongly rejects negative minutes, and (2) the formula that combines hours and minutes has the wrong sign handling for negative hours/minutes.
- Fix by adjusting the validation range, enforcing the "positive hours cannot have negative minutes" rule, and computing total_minutes with explicit sign-aware logic.
