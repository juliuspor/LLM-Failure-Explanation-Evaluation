What happened (where it failed)
- The exception occurs at these lines (slice block 81–83):
  > for i in range(search_length):  
  > &nbsp;&nbsp;&nbsp;&nbsp;greater = len(replacement_list[i]) - len(search_list[i])

- The error TypeError: object of type 'NoneType' has no len() means either replacement_list[i] or search_list[i] is None, and len(None) is invalid.

Why it happens (root cause)
- Earlier in the function you intentionally treat None or empty search strings and None replacements as “ignore this pair” — see the guard used when finding matches (lines ~59–69 and again ~102–105):
  if no_more_matches_for_repl_index[i] or search_list[i] is None or len(search_list[i]) == 0 or replacement_list[i] is None: continue

- But the size-guess loop at lines 81–84 lacks that same null-safety check. So when replacement_list[i] or search_list[i] is None the code still reaches len(replacement_list[i]) or len(search_list[i]) and raises TypeError.

Which tests trigger it
- test_null_safety_inside_arrays_single: replacement_list = [None] causes len(replacement_list[0]) to fail.
- test_null_safety_inside_arrays_mixed: replacement_list[1] is None; the loop still tries len(replacement_list[1]) and fails.

How to fix it (concrete steps)
1. Skip any pair where search_list[i] is None or empty, or replacement_list[i] is None — same condition used elsewhere in the function.
2. Only compute lengths when both search_list[i] and replacement_list[i] are non-None (and search_list[i] not empty).

Exact change (drop-in replacement for the buggy loop)
Replace:
  for i in range(search_length):
      greater = len(replacement_list[i]) - len(search_list[i])
      if greater > 0:
          increase += 3 * greater  # assume 3 matches

With:
  for i in range(search_length):
      if search_list[i] is None or len(search_list[i]) == 0 or replacement_list[i] is None:
          continue
      greater = len(replacement_list[i]) - len(search_list[i])
      if greater > 0:
          increase += 3 * greater  # assume 3 matches

Why this fix is correct
- It matches the null-safety logic used later in the function (so behavior is consistent).
- It prevents calling len(...) on None and keeps the size-guess heuristic working only for valid pairs.

Optional note
- The comment in the code already notes this loop is "not null-safe for elements inside the arrays" — the fix implements that intended safety.
