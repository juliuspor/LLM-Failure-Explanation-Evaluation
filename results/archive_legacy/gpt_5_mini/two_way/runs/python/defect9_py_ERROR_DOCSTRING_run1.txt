What happened (root cause)
- The method replaceEach(text, search_list, replacement_list) crashes because, somewhere in the code, len(...) is called on an item that can be None. The exception TypeError: object of type 'NoneType' has no len() tells us exactly that: a None value reached a len(...) call.
- The None values come from search_list or replacement_list. The docstring says None replacements should be ignored, but the code still assumes every element of search_list and replacement_list is a string and directly calls len() on them.
- Concretely: in replaceEach you iterate through the parallel arrays search_list and replacement_list and at some point do something like len(search_list[i]) or len(replacement_list[i]). If replacement_list[i] (or search_list[i]) is None, Python raises TypeError.

Where this occurs (what to look for in your code)
- Method name: replaceEach (the routine that performs the replacements).
- Variables: search_list, replacement_list, text, i (the loop index).
- Exact kind of lines to inspect:
  - Any line that calls len(search_list[i]) or len(replacement_list[i]).
  - Any conditional that assumes a truthy length, e.g. if len(replacement_list[i]) == 0 or if len(search_list[i]) > 0.
  - Any place that slices or indexes the element, e.g. replacement_list[i][0:…], which also assumes a non-None string.
- Typical failing spot: before you decide whether to skip or perform a replacement you test length (or call .startswith, .find, slicing) on the element without first checking for None.

Why the code violates the intended behavior
- The docstring says "those replacements should be ignored." That requires the implementation to explicitly detect None elements and skip them. Instead the code treats None like a string and calls len() (or other string operations) on it, causing the crash.

Steps to reproduce quickly
- Call replaceEach("abc", ["a", None], ["x", "y"]) (or similar) — the function will raise TypeError at the first attempt to take len(None) or do string ops on None.
