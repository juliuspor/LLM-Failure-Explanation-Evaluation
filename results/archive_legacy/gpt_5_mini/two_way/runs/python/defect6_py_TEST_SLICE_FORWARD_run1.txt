What happened (what fails)
- The code prints plain "0" instead of "-0.0" for an input of -0.0. The negative sign is lost and the test test_negative_zero fails.

Where it happens (exact lines)
- The sign is lost at line 177 when the code does value = int(x). Converting the float -0.0 to int produces 0 (no sign). Later the code chooses a representation based on value/mantissa and eventually calls self.add(str(value)) (line 192) or self.add(str(x)) (line 194), so by the time a string is produced the information that x was negative zero is gone.

Why it happens (root cause)
- int(-0.0) == 0 — converting the float to an integer destroys the negative-zero sign bit. The code never checks for negative zero before converting or formatting, so it treats -0.0 the same as +0.0 when deciding which representation to emit. The utility to detect negative zero (CodeConsumer.is_negative_zero) exists, but it is not used here prior to the int cast/formatting, so the special case is missed.

Concrete steps to fix (explicit, actionable)
1. Detect negative zero before any int conversion or formatting:
   - At the start of add_number (before line 177), add a check like:
     if CodeConsumer.is_negative_zero(x):
         self.add("-0.0")
         return
   - This preserves the sign and the exact expected string "-0.0".
   - Use the existing is_negative_zero utility so the detection is robust.

2. Alternatively, if you prefer handling inline without the utility:
   - Use math.copysign or struct to detect sign:
     if x == 0.0 and math.copysign(1.0, x) < 0:
         self.add("-0.0")
         return

3. Ensure the negative-zero check occurs before value = int(x) (line 177) and before any other transformation that discards the sign.

Notes and rationale
- The key is to check for negative zero before any operation that normalizes the value (int cast, integer mantissa extraction, stringification of the integer 0). Once you call int(x) or otherwise compute a value that equals 0, the distinction between +0.0 and -0.0 is irrecoverably lost using those variables.
- The proposed fix is minimal and localized: one conditional early in add_number. It uses the existing is_negative_zero helper or math.copysign, so no invasive rewrites of the rest of the formatting logic are required.
