What happened
- In LocaleUtils.to_locale the code validates the country letters using variables ch3 and ch4 at lines 239–240:
  - Line 239 reads: if ch3 < 'A' or ch3 > 'Z' or ch4 < 'A' or ch4 > 'Z':
  - Line 240 raises ValueError(...) when that check fails.
- For the input "fr__POSIX" the characters at those positions are:
  - locale_str[2] (ch3) == '_' (an underscore)
  - locale_str[3] (ch4) == '_' (an underscore)
- The validation expects uppercase letters there, so ch3 ('_') < 'A' triggers the condition and a ValueError is raised. That is why the test for the language__variant format (empty country, double underscore) fails.

Why it fails (root cause)
- The method assumes there will always be two uppercase country letters when parsing a 7+ character locale string, and validates ch3/ch4 unconditionally.
- It does not handle the valid Java-style form language__variant (e.g. "fr__POSIX") where the country is intentionally empty and represented by two underscores. Because ch3 is '_' instead of an uppercase letter, the rigid check treats that as invalid and throws an error.

Where to change it (exact place)
- Modify the logic around lines 239–240 inside LocaleUtils.to_locale to detect and accept the double-underscore empty-country case before enforcing the uppercase-country-letter check.

Concrete fixes (step-by-step, specific)
Option A — Minimal fix around lines 239–240:
1. Before the uppercase-letter check, check for double underscore:
   - if locale_str[2] == '_' and locale_str[3] == '_':
       - treat country = "" and variant = locale_str[4:]
       - skip the ch3/ch4 uppercase check
2. Otherwise, perform the existing uppercase-letter validation on ch3/ch4 as now.

Pseudo-change (showing the conditional to add exactly around those lines):
- Replace the unconditional check at lines 239–240 with:
  if locale_str[2] == '_' and locale_str[3] == '_':
      # empty country; parse variant from index 4
      country = ""
      variant = locale_str[4:]
  else:
      if ch3 < 'A' or ch3 > 'Z' or ch4 < 'A' or ch4 > 'Z':
          raise ValueError(f"Invalid locale format: {locale_str}")
      # existing code to set country = ch3+ch4 and possibly variant follows

Option B — Cleaner parsing (recommended long-term):
1. Split the input by '_' into parts: parts = locale_str.split('_', 2)
   - This yields up to three components: language, country (maybe empty), variant (maybe absent).
2. Validate language (lowercase, length 2).
3. If parts length >= 2, country = parts[1] (may be empty string — accept that).
   - If non-empty, validate uppercase and length 2.
4. If parts length == 3, variant = parts[2].
5. This approach naturally supports language__variant without special-case checks.

Why these fixes work
- Both approaches explicitly accept an empty country when the input uses a double underscore. The current failure happens because the code treats '_' as invalid uppercase letters; the fixes either bypass that check in the empty-country case (Option A) or avoid positional character assumptions by splitting (Option B).

Summary
- Root cause: unconditional uppercase-letter check on ch3/ch4 treats '_' as invalid (lines 239–240).
- Fix: detect double underscore and accept an empty country, or parse by splitting on underscores and validate components.
