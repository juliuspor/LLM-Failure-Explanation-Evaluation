What happened (root cause)
- At the top you clamp upper to the string length:
  if upper == -1 or upper > len(text):
      upper = len(text)
- Immediately after you can raise upper again:
  if upper < lower:
      upper = lower

If lower is greater than len(text) (e.g. caller passed a too-large lower), that second line sets upper = lower, so upper can become > len(text) even though you just clamped it. Later you call substring_java(text, 0, upper). That call passes an end index larger than the string length, which hits the check in SLICE_BLOCK (lines 32–33) and raises IndexError: "String index out of range: {end}".

Where it fails
- The failure originates at the substring_java(...) call(s) that use upper as the end index (the branches that call substring_java(text, 0, upper)). The SLICE_BLOCK shows the thrown IndexError is because end > len(s).
- The assignment that lets that happen is the if upper < lower: upper = lower line (it can raise upper above len(text) if lower > len(text)).

How to fix (concrete steps)
1. Ensure upper is never set above len(text) after you adjust it. Either:
   - Move the lower-check before the clamp:
     - First: if upper < lower: upper = lower
     - Then: if upper == -1 or upper > len(text): upper = len(text)
   OR
   - Re-clamp after the lower adjustment:
     - Keep current order, but add after the lower adjustment: if upper > len(text): upper = len(text)
   OR
   - Use a single normalized expression: upper = min(len(text), max(0 if you want, lower), upper) depending on allowed values.

2. Also consider protecting lower itself: if lower > len(text) you probably want to treat that as lower = len(text) (or return the whole string/empty) — be explicit about desired behavior and normalize lower accordingly.

Minimal concrete change example (add one line):
    if upper < lower:
        upper = lower
    if upper > len(text):
        upper = len(text)

This guarantees substring_java(text, 0, upper) will never be called with end > len(text).
