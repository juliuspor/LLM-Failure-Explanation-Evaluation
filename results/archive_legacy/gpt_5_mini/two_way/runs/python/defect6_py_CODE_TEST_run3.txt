What happened (root cause)
- In add_number, the code first tests whether x is an integer with:
  if x == int(x) and not math.isinf(x) and not math.isnan(x):
- For x == -0.0 that condition is True because -0.0 == 0 (and int(-0.0) == 0).
- The code then sets value = int(x) → value == 0 and later does self.add(str(value)) → "0".
- That converts the negative zero into the plain string "0", so the sign is lost and the test expecting "-0.0" fails.

Where it occurs (exact spot)
- In your shown add_number implementation, the problematic behavior happens in the integer-handling branch starting at the line:
    if x == int(x) and not math.isinf(x) and not math.isnan(x):
  because the negative-zero detection happens too late (after x was already reduced to int(x) == 0).

Why this is wrong
- Negative zero is a distinct IEEE floating-point value that must be preserved in output when the sign matters. Converting to int(x) before checking sign removes the sign bit, so you can no longer detect negative zero.

Precise fixes (step-by-step)
1. Detect negative zero before you do the integer check. Use an explicit check for negative zero and emit a string that preserves the sign, then return. For example, insert right after you compute prev (and before the integer branch):
    # preserve negative zero sign
    if x == 0.0 and math.copysign(1.0, x) < 0:
        self.add("-0.0")
        return

   Or, if you have the utility CodeConsumer.is_negative_zero(x), do:
    if CodeConsumer.is_negative_zero(x):
        self.add("-0.0")
        return

2. Keep the rest of the code unchanged; it will now only run for non-negative-zero values and will not collapse -0.0 into 0.

Why this fix works
- The check x == 0.0 and math.copysign(1.0, x) < 0 detects negative zero reliably before any integer conversion wipes out the sign. Emitting "-0.0" there preserves the sign exactly as the tests expect.

Note on alternatives
- You could also change the logic to avoid calling int(x) when x is zero, or prefer formatting paths that preserve sign for floats (for example formatting floats directly with repr/format), but the minimal, explicit, and reliable fix is to detect negative zero first and return.

Concluding one-line patch
- Insert, immediately after prev = self.get_last_char():
    if x == 0.0 and math.copysign(1.0, x) < 0:
        self.add("-0.0")
        return

This prevents the integer branch from converting -0.0 into 0 and fixes the failing test.
