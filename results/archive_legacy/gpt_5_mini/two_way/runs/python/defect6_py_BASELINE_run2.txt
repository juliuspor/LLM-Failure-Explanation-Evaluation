What happened (where it fails)
- The failure occurs in the integer-handling branch starting at line 176:
  if x == int(x) and not math.isinf(x) and not math.isnan(x):
- For x = -0.0, the comparison x == int(x) is True because -0.0 == 0.0, and int(-0.0) returns 0. That sets value = 0 (line 177).
- Later the code does self.add(str(value)) (line 192), which writes "0" — the sign is lost.
- The code never preserves the negative-zero sign because it converts the float to an int and then to a string. The code treats -0.0 as the integer 0 and so prints "0" instead of "-0.0".

Why this happens (root cause)
- IEEE-754 negative zero (float value -0.0) compares equal to positive zero (0.0). int(-0.0) → 0 loses the sign bit. The branch at line 176 assumes exact-integer floats can be rendered as integers, but that assumption is wrong for -0.0 because printing an integer "0" drops the negative sign. The class has an is_negative_zero utility (tests call CodeConsumer.is_negative_zero), but the integer branch does not use it.
