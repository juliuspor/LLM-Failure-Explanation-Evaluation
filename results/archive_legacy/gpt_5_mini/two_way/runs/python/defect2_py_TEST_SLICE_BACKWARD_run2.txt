What happened (where it fails)
- Location: GrayPaintScale.get_paint, at the code that computes the gray channel and then checks it (see the slice).
- Exact lines:
  - Line 71: g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
  - Lines 74–75:
    if g < 0 or g > 255:
        raise ValueError(f"Color parameter outside of expected range: Red Green Blue")
- Why it fails for your tests:
  - Your tests call get_paint(-0.5) and get_paint(1.5) with the default bounds 0.0..1.0.
  - For value = -0.5, the expression on line 71 yields g = int((-0.5 - 0.0)/(1.0 - 0.0)*255.0) → int(-127.5) → -127.
  - Because g < 0, the code hits the if on line 74 and raises ValueError instead of returning a clamped black color.
  - Similarly, value = 1.5 produces g > 255 and triggers the same raise.
- Root cause: the implementation expects inputs to always be inside the configured bounds and raises an exception when the computed channel is outside 0..255, rather than clamping out-of-range inputs to the bounds as the tests (and the original Java behavior) expect.

Steps to fix (concrete, line-level)
Option A — clamp the input value before computing g (recommended)
- Replace line 71 with:
    value_clamped = min(max(value, self._lower_bound), self._upper_bound)
    g = int((value_clamped - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- Remove lines 74–75 (the ValueError raise). After computing g, ensure returned color uses g.

Option B — compute g then clamp the channel (alternate)
- Keep line 71, but replace the check/raise with:
    g = max(0, min(255, g))
- Then return (g, g, g).

Notes and one extra check
- If self._upper_bound == self._lower_bound you will get division by zero on line 71; you should also handle that case (e.g., treat any value as the lower bound or raise a clear error). Example safe guard:
    if self._upper_bound == self._lower_bound:
        g = 0
    else:
        ...compute g...
- With the recommended change (Option A), get_paint(-0.5) -> value_clamped = 0.0 -> g = 0 -> (0,0,0); get_paint(1.5) -> clamped to 1.0 -> g = 255 -> (255,255,255); and mid values behave as expected.

This directly fixes the ValueError you observed by clamping inputs (or g) instead of raising, matching the test expectations.
