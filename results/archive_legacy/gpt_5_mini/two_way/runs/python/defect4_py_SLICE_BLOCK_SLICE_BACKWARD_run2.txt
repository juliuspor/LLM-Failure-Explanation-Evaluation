What happens
- In __init__ you set self._min_start_index = -1 (line 253).
- Later, when you process a new time period you compute its start (lines ~331–336) and try to update the stored minimum start index (lines 340–349).
- The intended behavior is:
  - If _min_start_index < 0 (no minimum yet), set _min_start_index to the current index.
  - Else compare the current period start to the stored minimum start and update _min_start_index only if the current start is smaller.

Why it fails (root cause)
- At line 349 there is an unconditional assignment:
  self._min_start_index = index
  This sits outside the conditional blocks and therefore always runs. As a result, every time you process a period you overwrite the previous minimum index with the current index, regardless of the comparison at line 346 (start < min_start).
- Consequence: _min_start_index never reliably tracks the true minimum-start period; it always ends up pointing to the last processed index.
- Secondary risk (closely related): the conditional logic that computes min_start only runs when self._min_start_index >= 0 (line 340). If you intended to set _min_start_index for the first item, that must be done in an else branch or when _min_start_index < 0 — but the unconditional assignment at line 349 hides that intent and breaks correctness.

- The bug: the unconditional assignment at line 349 overwrites whatever the conditional did.
