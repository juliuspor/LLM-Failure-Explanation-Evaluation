What happened (where and why)
- At line 54 you compute a clamped variable v:
  v = max(value, self._lower_bound)
  v = min(v, self._upper_bound)
  So v is the clamped value you intend to use.
- At line 71 you compute g using the original value variable instead of v:
  g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
  Because value wasn't clamped here, passing a value < lower_bound or > upper_bound produces g outside 0–255 and triggers the ValueError at line 74.
- In short: you clamped into v but never used v in the grayscale calculation. The unused clamped variable v is the root cause.

Concrete fix (exact change)
- Replace value with v in the computation of g, and also guard against zero range:
  denom = (self._upper_bound - self._lower_bound)
  if denom == 0.0:
      g = 0
  else:
      g = int((v - self._lower_bound) / denom * 255.0)

Why this fixes it
- Using v ensures out-of-range inputs are first clamped to bounds (black/white), so g will always be between 0 and 255.
- Checking denom prevents division by zero if lower_bound == upper_bound.

Optional small note about rounding
- int() truncates toward zero; your tests expect 127 for 127.5, so int() is fine here. If you prefer nearest rounding, use round() before int().

Summary of steps to apply
1. Change the g computation to use the clamped v (see code above).
2. Add a check for zero denominator (upper_bound == lower_bound).
3. Re-run tests — the ValueError should disappear and clamping tests will pass.
