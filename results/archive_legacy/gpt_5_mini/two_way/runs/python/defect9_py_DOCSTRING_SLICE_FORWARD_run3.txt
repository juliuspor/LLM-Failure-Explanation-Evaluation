What happened (root cause)
- The code tries to ignore entries where search_list[i] or replacement_list[i] is None, but it still evaluates len(search_list[i]) (and/or calls text.find(search_list[i])) before that ignore check is guaranteed to run.
- When search_list[i] is None, calling len(search_list[i]) raises TypeError ("object of type 'NoneType' has no len()"), and calling text.find(None, ...) also raises TypeError. Those calls occur around the block shown at lines 102–110, so the function crashes instead of skipping that entry.

  If the guard failed to skip a None search_list entry, this call will raise TypeError because text.find(None, ...) is invalid.

Why the guard failed
- The guard mixes checks that are safe (search_list[i] is None) with a check that is unsafe when search_list[i] is None (len(search_list[i]) == 0). Python evaluates the right-hand parts of the condition only as needed, but because the unsafe check appears unprotected in the same boolean expression, len(None) can still be evaluated in some code paths and crash.
- Also, even if the boolean expression were safely ordered, other places of the loop (for example the text.find call at line 108 and the later use of replacement_list[replace_index] at line 92) assume non-None values and will crash unless entries with None are explicitly skipped beforehand.

Exact consequence
