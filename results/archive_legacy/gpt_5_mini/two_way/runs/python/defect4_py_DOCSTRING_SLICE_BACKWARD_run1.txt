What happened (root cause)
- The code always sets self._min_start_index to the current index on every call because of an unconditional assignment at line 349.
- Specifically:
  - Lines 340–347 do the intended check: when self._min_start_index >= 0, compute min_start from the stored data item and, if start < min_start, set self._min_start_index = index (line 347).
  - But immediately after that conditional block there is another assignment at line 349: self._min_start_index = index. That unconditional assignment overwrites any earlier decision and makes _min_start_index always equal to the most recent index.
- Consequence: the “minimum start” tracking never works — the index is replaced every time instead of being updated only when a genuinely smaller start is found (and also the first-time initialization is not handled correctly if the code relied on the conditional branch).

Where it occurred (exact lines and variables)
- Initialization: self._min_start_index = -1 (line 253).
- Decision & compare: if self._min_start_index >= 0: ... if start < min_start: self._min_start_index = index (lines 340–347).
- Problematic override: self._min_start_index = index (line 349) — this is the line that causes the wrong behavior.
- Accessor: def get_min_start_index(...) returns self._min_start_index (lines 437–439).

Why this caused the failure
- Because the unconditional assignment at line 349 always executes, the earlier conditional check and conditional assignment (line 347) are useless. The code cannot preserve the previously recorded minimum; it always uses the current index.

How to fix it (concrete steps)
1. Remove the unconditional assignment at line 349.
2. Ensure the first-time case (when _min_start_index == -1) sets _min_start_index to index. Concretely:
   - If self._min_start_index < 0: set self._min_start_index = index (initialize).
   - Else (self._min_start_index >= 0): compute min_start and only set self._min_start_index = index when start < min_start.
3. Make sure min_start is always defined before it is compared (the code already computes it inside the >= 0 branch).

Suggested replacement pseudocode (apply to the method body around lines 340–349):

  if self._min_start_index < 0:
      # first item -> initialize
      self._min_start_index = index
  else:
      min_start_period = self.get_data_item(self._min_start_index).get_period()
      if isinstance(min_start_period, SimpleTimePeriod):
          min_start = min_start_period.get_start_millis()
      else:
          min_start = int(min_start_period.get_start().timestamp() * 1000)
      if start < min_start:
          self._min_start_index = index

Notes
- Do not keep the unconditional assignment at line 349; that is the single line whose removal (and replacing with the explicit first-time branch above) fixes the logic.
- Verify that start and min_start use the correct timestamp path for both SimpleTimePeriod and other period types (the code shows two ways to get milliseconds; keep whichever is correct for your types).
