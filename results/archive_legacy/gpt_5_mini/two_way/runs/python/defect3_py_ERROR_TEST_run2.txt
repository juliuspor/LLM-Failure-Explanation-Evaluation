What happened (symptom)
- The test string is "\U0001F630A". In UTF-16 this is three 16-bit code units: [high-surrogate, low-surrogate, 'A'].
- The translator iterates over UTF‑16 code units and tries to combine a high + low surrogate into one code point.
- The code advances its index incorrectly when it sees a surrogate pair, then later tries to read input_chars[index+1] where index+1 is past the end of the array. That triggers IndexError: String index out of range: 2.

Where it occurs (methods / variables)
- In hit03_translator.py: methods translate_string (CsvEscaper), CharSequenceTranslator.translate (or whatever internal routine walks the UTF‑16 units), and the helper to_utf16_code_units / code_point_at are involved.
- The relevant local variables are the index variable (often named index or i), the current code unit (ch or cur), and the next code unit (next_ch).
- The crash happens when the code executes something like next_ch = input_chars[index + 1] without first ensuring index + 1 < len(input_chars).

Why it happens (root cause)
- The code treats a high surrogate specially but does not consistently advance the loop index by two when it consumes a surrogate pair. Common buggy patterns:
  1. Detect high surrogate at position i, compute code point from unit i and i+1, but only do i += 1 (or rely on the outer loop doing +1), leaving the next iteration pointing at the low surrogate or the trailing ASCII char.
  2. Later, that iteration again tries to access index+1 assuming a pair exists and reads past the end.
- Or: the code checks input_chars[index+1] to see if a low surrogate is present without checking that index+1 < len(input_chars). If the index happens to be the last position, this unconditional access raises IndexError.
- In the failing test the sequence of index values is:
  - 0 -> sees high surrogate, mishandled advancement
  - after incorrect advancement the code ends up calling input_chars[ index + 1 ] where index + 1 == 3 (out-of-range) or calling it when index == 2 and index+1 == 3 -> IndexError with message referencing index 2 (the code attempted to read the next pair check from the last code unit).

How to reproduce the exact failure
- Input: "\U0001F630A" (high surrogate, low surrogate, 'A')
- If translate logic tries to inspect index+1 without guarding bounds, it throws IndexError on the final check.

Concrete fixes (step-by-step)
1. Always check bounds before reading index+1:
   - Before doing next_ch = units[index + 1], do:
     if index + 1 >= len(units): handle as non-pair (treat high surrogate as standalone or append replacement) and advance index by 1.
2. When you successfully consume a surrogate pair, advance the index by 2, not 1:
   - If you detect a valid high surrogate followed by a valid low surrogate:
     - code_point = combine(units[index], units[index+1])
     - append output for code_point
     - index += 2
   - Else (no valid low surrogate) treat the high surrogate as a single code unit and index += 1.
3. Prefer a single helper that both returns the code point and the number of UTF-16 units consumed, e.g.:
   - def code_point_at(units, index):
       if index + 1 < len(units) and is_high(units[index]) and is_low(units[index+1]):
           return combined_code_point, 2
       else:
           return units[index], 1
   - Then loop using the returned consumed length: index += consumed.
4. Alternatively, operate on Python code points (the native str) rather than manually on UTF-16 code units so you don't need surrogate handling. For example, iterate the string directly (for ch in s:) and use ord(ch) — Python uses code points, so surrogate pairs are already combined (unless you explicitly converted to UTF-16 units).
