What happened
- The test that failed called DateTimeZone.for_offset_hours_minutes(0, -15). The implementation raised ValueError("Minutes out of range: -15").
- That message shows the code is rejecting -15 as an illegal minutes value even though -15 is a valid minute according to the tests and the docstring (minutes must be in -59..+59).

Root cause (where and why)
- In hit01_timezone.py, inside DateTimeZone.for_offset_hours_minutes, the minutes validation only accepts the range 0..59 instead of -59..59. The code that raises ValueError("Minutes out of range: %d" % minutes) is therefore triggered when minutes is negative (here minutes == -15).
- Because the implementation treats minutes as unsigned, any negative minutes (including valid ones like -15) fail the range check. That is why the test for (0, -15) fails.
