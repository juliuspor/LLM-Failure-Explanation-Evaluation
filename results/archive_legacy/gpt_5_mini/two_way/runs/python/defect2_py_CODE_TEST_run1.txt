What happened (root cause)
- The code clamps the input into v:
    v = max(value, self._lower_bound)
    v = min(v, self._upper_bound)
  but then computes g from the original value instead of the clamped v:
    g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- So when value is outside the bounds (e.g. -0.5), v becomes the clamped bound (0.0), but g is computed from value (-0.5). That yields a value outside 0..255, so the code raises the ValueError in the sanity check. This is exactly why test_lower_bound_clamping fails.
- Additionally: if self._upper_bound == self._lower_bound you get a division-by-zero in that same g computation. That edge case is not handled.

Where it occurs (specific variables/lines)
- The bug is in the line that computes g (uses value instead of v). Variables involved: value (uncamped), v (clamped). The error is triggered by the subsequent range check on g which raises ValueError for out-of-range g.

What to change (concrete steps)
1. Use the clamped variable v when computing g.
2. Guard against division by zero when upper == lower.
3. (Optional) Improve the ValueError message to include the actual parameter and value.

Minimal corrected code (replace the g computation and following guard):

    # after clamping to v
    # handle degenerate range
    range_width = self._upper_bound - self._lower_bound
    if range_width == 0.0:
        # if bounds are identical, return black or white (choose behavior you want)
        g = 0  # or 255, or 127 — pick the appropriate policy for equal bounds
    else:
        g = int((v - self._lower_bound) / range_width * 255.0)

    if g < 0 or g > 255:
        raise ValueError(f"Color parameter outside of expected range: {g}")

    return (g, g, g)

Why this fixes it
- Using v ensures values outside the bounds are mapped as intended (e.g. -0.5 -> clamped 0.0 -> g = 0).
- The range_width == 0.0 check avoids a division-by-zero if bounds are equal.
- Using the actual g in the error message makes debugging clearer.
