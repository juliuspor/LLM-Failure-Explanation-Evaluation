What happened (where and why)
- Location: GrayPaintScale.get_paint, lines:
  - v = max(value, self._lower_bound)
  - v = min(v, self._upper_bound)
  - g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
  - if g < 0 or g > 255: raise ValueError(...)

- Root cause: you clamp the input into v, but then compute g using the original value variable (value) instead of using the clamped variable v. For out-of-range inputs (e.g. value = -0.5 or value = 1.5), the calculation uses the unclamped value and produces g outside 0..255, so the code raises ValueError even though you intended to clamp first.

- Concrete symptom in tests:
  - test_lower_bound_clamping: value = -0.5 -> v becomes 0.0, but g uses value = -0.5, yields negative g and raises ValueError instead of returning (0,0,0).
  - test_upper_bound_clamping: value = 1.5 -> v becomes 1.0, but g uses 1.5, yields g > 255 and raises ValueError instead of returning (255,255,255).

Other issues to note (secondary, but important)
- The error message in the ValueError is unhelpful: f"Color parameter outside of expected range: Red Green Blue" — it prints a literal string and not the offending values. It should include actual numbers (e.g. g or the computed RGB).
- If self._upper_bound == self._lower_bound, you will get a division-by-zero at the g calculation. That should be guarded.

How to fix it (step-by-step)
1. Use the clamped value v in the gray calculation (replace value with v).
2. Improve the ValueError message to include the offending channel/value.
3. Optionally guard against zero range (upper == lower) to avoid division by zero.
4. (Optional) If you want rounding toward nearest instead of truncation, use round(...) before int conversion.

Minimal corrected code (replace the body of get_paint with):

    def get_paint(self, value: float) -> Tuple[int, int, int]:
        # Clamp value to bounds
        v = max(value, self._lower_bound)
        v = min(v, self._upper_bound)

        # Guard against zero range
        span = self._upper_bound - self._lower_bound
        if span == 0.0:
            g = 0 if v <= self._lower_bound else 255
        else:
            # Use clamped value v here (previous bug used 'value' instead of 'v')
            g = int((v - self._lower_bound) / span * 255.0)

        # Validate the range and give a useful message
        if g < 0 or g > 255:
            raise ValueError(f"Color parameter outside of expected range: g={g}")

        return (g, g, g)

Why this fix works
- By computing g from v (the clamped value) you ensure out-of-range inputs are first clamped and then mapped into 0..255 so the ValueError is not triggered incorrectly.
- The span check prevents division by zero if upper and lower bounds are equal.
- The improved error message shows the actual offending value for easier debugging.
