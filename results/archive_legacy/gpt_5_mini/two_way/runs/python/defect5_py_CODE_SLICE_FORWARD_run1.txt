What goes wrong (root cause)
- On this line you set inferred_type = type(array) when array is not None:
  - That returns the Python type of the list object itself (e.g. <class 'list'>), not the element type contained by the list.
- _copy_list_grow1 is apparently expecting an element type (the type to use for the new list’s storage), not the list’s type. So when you pass inferred_type = <class 'list'> instead of <class 'int'> (or whatever the element type is), the copy/grow operation creates a list with the wrong element-type assumption.
- The wrong type then propagates to the later checks and the ClassCast-like error you raise. The code fails because you used the wrong value for inferred_type — you used the type of the container (array) instead of the type of the contained elements.

Where it happens (exact lines)
- Line that chooses the inferred type (shown in your snippet):
  if array is not None:
      inferred_type = type(array)
- Later, that bad inferred_type is passed to:
  new_list = ArrayUtils._copy_list_grow1(array, inferred_type)
- And the incorrect inferred_type causes the subsequent assignment/new-list creation to fail or the cast check to raise.

Why that causes the error (simple sequence)
1. array is a list, so type(array) is list, not element type.
2. _copy_list_grow1 receives inferred_type == list and builds a new list under the wrong assumption.
3. You then assign element into the newly created slot and either a runtime type mismatch occurs in the underlying copy logic or your final check misdetects an object/element-type mismatch and raises the TypeError.

Concrete fixes (step-by-step)
1. Use the element type when array has elements, not the list type. For example:
   - If array is non-empty: inferred_type = type(array[0])
   - If array is empty and expected_type is provided: inferred_type = expected_type
   - Else if element is not None: inferred_type = type(element)
   - Else fallback to object
2. Defend against an empty array (len(array) == 0): you cannot infer the element type from the array, so prefer expected_type (if provided) or fall back to type(element) or object.
3. Update the check that raises TypeError to use the corrected inferred_type logic.

Minimal corrected code (replace only the inferred_type block)
    if array is not None and len(array) > 0:
        inferred_type = type(array[0])
    elif element is not None:
        inferred_type = type(element)
    elif expected_type is not None:
        inferred_type = expected_type
    else:
        inferred_type = object

Why this fix works
- When array has elements, type(array[0]) gives the element type, which is what _copy_list_grow1 expects.
- When array is empty, you fall back to expected_type or element, avoiding the earlier mistake of using list as the element type.
- This prevents creating a new list under the wrong type assumption and prevents the later ClassCast-like error.

Optional improvements
- If element types can be mixed, consider using a common base type or just using object to avoid type errors.
- Add unit tests: e.g., add([1, 2], 3) and add([], 3, expected_type=int) to confirm behavior.
