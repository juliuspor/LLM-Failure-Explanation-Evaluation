What happened (where it fails)
- The failure happens in translate(), inside the while loop after a translator returns a positive consumed value:
  - Line: consumed = self.translate_codepoint(code_units, pos, out)
  - Then this block runs (the loop at the end of the snippet):

    for pt in range(consumed):
        pos += char_count(code_point_at(code_units, pos))

- code_point_at() raises an IndexError when called with index == len(code_units) (see to_utf16_code_units slice: lines 35–36). That IndexError is the crash you see.

Why it fails (root cause)
- There is a mismatch in what consumed counts:
  - translate_codepoint(...) returns consumed, but the code treats consumed as "number of code points" and advances pos by repeatedly calling code_point_at(code_units, pos) and adding char_count(...) for each.
  - If consumed already counts code units (or if the translator already advanced through code units), calling code_point_at(code_units, pos) repeatedly may be invoked when pos == length (past the end), causing IndexError.
- More simply: the code calls code_point_at(code_units, pos) at a position that may be equal to length (i.e., past the last code unit) because pos is not advanced correctly before the call. That out-of-range call throws the IndexError at the check in code_point_at (lines 35–36).

Exactly where the mismatch appears
- translate(): line that calls translate_codepoint(...) and the for-loop that advances pos based on char_count(code_point_at(...)).
- to_utf16_code_units / code_point_at(): lines 35–44 show code_point_at performs an index range check and will raise IndexError if called with index >= len(code_units).

Concrete ways to reproduce the bug
- If translate_codepoint(...) returns a positive consumed value indicating it already consumed one or more code units but did not advance pos itself, the loop tries to advance pos by asking for the code point at pos. If pos is at the last code unit or at length, code_point_at(...) is called with an out-of-range index and fails.

How to fix it (specific steps)
1. Decide the unit of consumed and make it consistent:
   - Option A (simpler): If translate_codepoint returns the number of UTF-16 code units consumed, then replace the for-loop with a direct addition:
     - pos += consumed
     - Remove the for pt in range(consumed) loop entirely.
   - Option B: If translate_codepoint returns the number of code points consumed (not code units), keep the loop but ensure you read the code point safely:
     - Before calling code_point_at(code_units, pos) check pos < length. If pos >= length, break or raise a clear error.
     - Or compute each codepoint with bounds-checking: if pos >= length: raise ValueError("translate_codepoint signaled consumption beyond available input") so the error is explicit.

2. Make the contract explicit in code and comments:
   - Document whether translate_codepoint(code_units, pos, out) returns count in code units or code points.
   - If it returns code points, change its name or docstring to say so. If it returns code units, make the caller add consumed to pos directly.

3. Add a defensive check before calling code_point_at:
   - if pos >= length: raise IndexError(f"pos {pos} out of range (length {length})") — but better is to fix the unit mismatch as in step 1.

4. Add unit tests that cover:
   - Translators that consume surrogate pairs (2 code units).
   - Translators that consume single code units.
   - Cases where translate_codepoint returns consumed larger than remaining input (should fail with a clear error).

Example minimal change (if consumed is code units):
Replace these lines:

    for pt in range(consumed):
        pos += char_count(code_point_at(code_units, pos))

With:

    pos += consumed

Why this fix works
- If consumed counts UTF-16 code units, adding consumed directly advances pos the correct number of code units without calling code_point_at beyond the end. This removes the out-of-range call and resolves the IndexError.

Summary
- The crash is caused by calling code_point_at(code_units, pos) when pos may be at or past the end of code_units. The underlying reason is an inconsistent interpretation of what translate_codepoint(...) returns (code units vs. code points). Fix by making the return unit explicit and advancing pos in the same unit (most likely: pos += consumed).
