What happened (where it fails)
- The failure happens in random(...) at line 49–52:
  - Line 49: gap = end - start
  - Line 52: if gap <= 0: raise ValueError("bound must be positive")
- If a caller passes start == end (a valid single-value range) then gap == 0 and the code raises ValueError("bound must be positive"). Example: random(1, 65, 65, False, False) will raise even though generating a single character at code point 65 should be valid.

Why it happens (root cause)
- This is an off-by-one / wrong-condition error. gap = end - start computes the number of steps between start and end, exclusive of the endpoint. For an inclusive range you must count the endpoint too. With start == end the inclusive size is 1, but gap == 0 triggers the "must be positive" check.
- The check at line 52 therefore rejects valid ranges where end == start. It also doesn't clearly express the intended invariant (is end inclusive or exclusive?), which caused the incorrect comparison.

Precise fixes (edit these lines exactly)
Option A — treat end as inclusive (recommended):
- Change line 49 to:
  gap = end - start + 1
- Keep line 52 as-is:
  if gap <= 0: raise ValueError("bound must be positive")
This makes gap == 1 when start == end (valid), and raises only when end < start.

Option B — keep arithmetic but change the check to express the invariant directly:
- Keep line 49:
  gap = end - start
- Change line 52 to:
  if end < start: raise ValueError("end must be >= start")
This is clearer: it raises only when the caller supplied an invalid ordering.

Which to choose
- If random(...) is intended to treat end as inclusive (typical for character/code-point ranges), use Option A.
- If end was intended to be exclusive, the error message should be updated to explain that and you should keep current arithmetic but change the message to something like "end must be greater than start" — but based on the current behavior the code appears to have intended an inclusive end, so Option A is the correct fix.

- With Option A: start=65, end=65 -> gap = 1 -> allowed
- With current code: start=65, end=65 -> gap = 0 -> raises ValueError
