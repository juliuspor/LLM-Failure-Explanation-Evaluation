What happened (where it fails)
- Failure occurs in GrayPaintScale.get_paint at these lines:

  - Lines 74–75:

- When you call get_paint(-0.5) or get_paint(1.5) the computed g is outside the 0–255 range, so the code raises ValueError instead of clamping to black or white. The test expects values below lower_bound to be clamped to (0,0,0) and values above upper_bound to be clamped to (255,255,255).

Why it happened (root cause)
- The algorithm computes g directly from the raw input value without first clamping the input (or clamping g afterwards).
- The subsequent check raises an exception when g is out of range, rather than clamping to the nearest valid color.
- Additional edge case to watch: if upper_bound == lower_bound the division on line 71 will raise ZeroDivisionError; that isn’t handled either.

- Before computing g (line 71) clamp the input value to the allowed range:
  - value = max(self._lower_bound, min(self._upper_bound, value))
- Or after computing g, clamp g to 0..255 instead of raising an exception:
  - g = max(0, min(255, g))
- Also handle the case upper_bound == lower_bound to avoid division by zero (either raise a clear ValueError or define a deterministic behavior).
- Finally, the raised ValueError text is unhelpful; if you still raise for an error case use a clear message naming the offending variable and line (e.g., "upper_bound must be > lower_bound").

Concrete minimal fix (example)

  1) Clamp the input:
     value = max(self._lower_bound, min(self._upper_bound, value))
