What happened (where it fails)
- In get_paint (def at line 54) you clamp the input into v:
  - v = max(value, self._lower_bound)
  - v = min(v, self._upper_bound)
  These are the two lines right after the "Clamp value to bounds" comment.
- But when you compute g you use the original variable value instead of the clamped v:
  - g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)  (line shown at 71)
- If value is outside the bounds that you intended to clamp, that calculation can produce a number < 0 or > 255, which triggers the ValueError at the check on line 74:
  - if g < 0 or g > 255: raise ValueError(...)
- There is also a second failure mode: if self._upper_bound == self._lower_bound the expression divides by zero and will raise a ZeroDivisionError before you even get to the range check.

Why it fails (root cause)
- Root cause 1: you clamped into v but then ignored it and continued to use the original value variable for the conversion (so clamping had no effect).
- Root cause 2: you didn't guard against lower_bound == upper_bound, so you can divide by zero.

1. Use the clamped value v when computing g. Replace the calculation on line 71 with:
   - g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
2. Guard against equal bounds (avoid division by zero). Add a check before computing g, for example:

3. Optionally clamp g to [0,255] before the ValueError check to avoid floating rounding producing 256:
