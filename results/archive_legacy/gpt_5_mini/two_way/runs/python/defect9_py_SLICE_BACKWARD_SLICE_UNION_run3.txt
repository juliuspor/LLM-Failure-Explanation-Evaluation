What went wrong (root cause)
- The method replace_each sets search_length = len(search_list) (line 52) and then assumes replacement_list has at least that many elements. At line 82 the code does:
  greater = len(replacement_list[i]) - len(search_list[i])
  This will raise an IndexError if replacement_list has fewer elements than search_list, and will raise a TypeError if replacement_list[i] is None. Those checks for None/empty (lines 102–107) occur later in the code, after this early loop, so they do not prevent the failure.

Where it happens (exact lines)
- The failure occurs during the first loop that computes "increase" (around lines 81–85). Specifically:
  - search_length is computed at line 52.
  - The loop that uses replacement_list[i] runs at line 81–85.
  - The problematic access is at line 82: replacement_list[i].
- A related assumption later also can fail: at lines 92–94 the code does parts.append(replacement_list[replace_index]) and start = text_index + len(search_list[replace_index]) — these assume replace_index was set to a valid index (not -1) and that both lists contain a usable entry at that index. If there are no matches, replace_index may remain unset/invalid and that will also cause errors.

Why that causes the crash (plain language)
- The code decides how much extra space to reserve by looping for i in range(search_length) and indexing replacement_list[i] before the code has validated that replacement_list is the same length as search_list or that replacement_list[i] is non-None. If replacement_list is shorter, Python raises IndexError as soon as you try replacement_list[i]; if replacement_list[i] is None, calling len(...) raises TypeError. The later checks that try to guard empty or None entries come too late — after the code already accessed those entries.

Concrete fixes (step-by-step)
1. Make the length relationship explicit and check it at the start of replace_each (preferred):
   - If you expect replacement_list to match search_list 1:1, add:
     if len(replacement_list) != len(search_list):
         raise ValueError("replacement_list must have the same length as search_list")
   - This makes the contract clear and fails early with a helpful message.

2. Or, iterate only over the overlapping indices if mismatched lengths are allowed:
   - Replace search_length = len(search_list) with:
       search_length = min(len(search_list), len(replacement_list))
     This prevents IndexError and limits processing to indices that exist in both lists.

3. Always validate items before calling len(...) or using them:
   - In the increase calculation loop (lines 81–85), check:
       if search_list[i] is None or len(search_list[i]) == 0 or replacement_list[i] is None:
           continue
       greater = len(replacement_list[i]) - len(search_list[i])
   - That prevents TypeError/ValueError when items are None/empty.

4. Ensure replace_index is valid before using it:
   - Before parts.append(replacement_list[replace_index]) and before computing start using search_list[replace_index], check that replace_index is not -1 (or isn’t the sentinel you use when no match is found), and that the corresponding list items are valid.

Which fix to choose
- If the API expects the two lists to be paired 1:1, use fix (1). It gives a clear error message and avoids hidden bugs.
- If mismatched lengths are legitimately allowed, use (2) and (3) to make the behavior robust.

Summary
- Root cause: the code indexes replacement_list[i] (line 82) for i in range(len(search_list)) before confirming replacement_list is at least that long or that entries are non-None. The safety checks are performed later, so the code can raise IndexError or TypeError at the increase calculation step. Fix by validating lengths and entries up front or iterating only the overlapping indices and checking for None before calling len(...).
