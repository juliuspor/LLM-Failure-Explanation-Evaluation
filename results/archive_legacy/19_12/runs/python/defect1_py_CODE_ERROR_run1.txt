The error occurs due to a problem in the `DateTimeZone.for_offset_hours_minutes` method. This method is supposed to create a time zone offset using hours and minutes. However, it currently raises an error if you try to pass a negative value for minutes, regardless of whether the hours value is positive or negative. This behavior is incorrect for negative `hours_offset` values where allowing negative `minutes_offset` is valid.

### What Happened
The error message `ValueError: Minutes out of range: -15` indicates that the application attempted to construct a time zone with an hours offset of, for example, `-2` and a minutes offset of `-15`. The code erroneously flags this as invalid.

### Why It Happened
In `for_offset_hours_minutes`, there's a check that raises a `ValueError` if `minutes_offset < 0`. This is too stringent because it prevents negative minutes that should be legal within the context of a negative hours offset (e.g., creating `-02:15` as an offset).

### Where It Occurred
The error is raised in this section of the `for_offset_hours_minutes` method:

```python
if minutes_offset < 0 or minutes_offset > 59:
    raise ValueError(f"Minutes out of range: {minutes_offset}")
```

### Fix
To correct this, modify the condition to allow negative minutes only when the hours are also negative:

1. Identify this block in the `for_offset_hours_minutes` method.
2. Change it to:
   ```python
   if (hours_offset < 0 and minutes_offset < -59) or (hours_offset >= 0 and minutes_offset < 0) or minutes_offset > 59:
       raise ValueError(f"Minutes out of range: {minutes_offset}")
   ```

This revised check ensures that minutes are restricted appropriately based on the sign of the hours, allowing truly valid combinations like `-02:15`.
