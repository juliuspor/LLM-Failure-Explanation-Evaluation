What happened (root cause)
- At line 176 the code tests if x == int(x). For x == -0.0 that comparison is True because Python considers -0.0 == 0.0.
- Immediately at line 177 value = int(x) turns -0.0 into the integer 0, losing the sign bit.
- Later (line 192) self.add(str(value)) writes "0". The code never reaches the else branch (line 194) that would have printed a float string like "-0.0".
- In short: the integer-branch treats negative zero as the integer 0 and discards the negative sign, so "-0.0" becomes "0".

Where it occurs
- The mis-handling happens in the integer branch beginning at line 176 and concretely when value = int(x) (line 177) and when str(value) is added (line 192).

How to fix (concrete steps)
1. Detect negative zero before converting to int or entering the integer branch.
2. If x is negative zero, emit "-0.0" and return immediately so the rest of the integer logic doesn't run.

Exact code changes you can make
- Add a check at the top of add_number, before line 176. For example, using math.copysign:

    if x == 0.0 and math.copysign(1.0, x) < 0.0:
        self.add("-0.0")
        return

- Or factor it into a utility method is_negative_zero and use it:

    if self.is_negative_zero(x):
        self.add("-0.0")
        return

- If you already have CodeConsumer.is_negative_zero, call that:

    if CodeConsumer.is_negative_zero(x):
        self.add("-0.0")
        return

Why this fixes it
- The check runs before value = int(x), so the sign is examined while x is still a float and the negative sign on the zero can be detected by math.copysign (or other signbit methods). That prevents the integer branch from converting -0.0 to 0 and losing the sign.

Notes
- Do not try to detect negative zero by x < 0 (it is False for -0.0).
- Keep the early return so the existing logic for integers, scientific notation, infinities and NaN stays unchanged.
