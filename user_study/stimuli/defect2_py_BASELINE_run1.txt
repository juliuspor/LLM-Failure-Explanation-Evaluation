What happened (root cause)
- In get_paint you correctly clamp the input into the local variable v:
  - v = max(value, self._lower_bound)
  - v = min(v, self._upper_bound)
  But when you compute the grayscale component you accidentally use the original value variable instead of the clamped v:
  - g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
- Because of that mistake, out-of-range inputs (for example value = -0.5 in test_lower_bound_clamping) produce a g outside 0..255 and trigger the ValueError on line 74.
  - Example: with lower_bound = 0.0 and value = -0.5, v is clamped to 0.0 but g is computed from -0.5 -> negative g -> exception.

Where it occurs
- get_paint, line that computes g (slice shows this as line 71), and the subsequent range check (line 74) raise the exception.
- The clamping lines that set v appear earlier (slice around lines 54–59), but their result is never used for g.

What to change (concrete steps)
1. Use the clamped variable v when computing g instead of value:
   - Replace
     g = int((value - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
     with
     g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)
2. (Optional but recommended) Guard against zero denominator if upper_bound == lower_bound:
   - If self._upper_bound == self._lower_bound, return either 0 or 255 (or raise a clearer error). For example:
     if self._upper_bound == self._lower_bound:
         return (0, 0, 0)  # or raise ValueError("upper_bound must be greater than lower_bound")
3. After making (1), tests like value = -0.5 and 1.5 will be clamped correctly and g will always be in 0..255 (assuming bounds are valid), so the ValueError on line 74 will no longer be raised.

Minimal corrected code (the single-line fix)
- Change g computation to:
  g = int((v - self._lower_bound) / (self._upper_bound - self._lower_bound) * 255.0)

That fixes the failing tests by using the clamped value for interpolation.
