The code fails because the loop that advances pos after a translator consumes code points advances by UTF-16 code unit counts computed at the current (changing) pos, which can move past the end when consuming a surrogate pair at the end followed by another character. Specifically: after consumed = self.translate_codepoint(...), the for-loop does: for pt in range(consumed): pos += char_count(code_point_at(code_units, pos)). That calls code_point_at(code_units, pos) while pos is being incremented inside the loop. When a surrogate pair is at the current pos and consumed >= 1, the first iteration reads the pair (ok) and increments pos by 2; on the next iteration the code calls code_point_at with pos equal to the original pos+2, which may be equal to len(code_units) (off the end) when the pair was at the end of the string. code_point_at checks index < len(code_units) and raises IndexError when index==len(code_units). In short: the loop should inspect the codepoint/char_count at the original start position(s) rather than updating pos before the next lookup, or better iterate using a separate temp index or add the total number of code units consumed in one step. The bug occurs around the loop beginning at line where "for pt in range(consumed): pos += char_count(code_point_at(code_units, pos))"; code_point_at raises IndexError: String index out of range: 2 when pos equals length (2) after advancing past a surrogate pair. Fix: compute char_count(code_point_at(...)) for each needed step using a temp index (e.g., temp = pos; for pt in range(consumed): cp = code_point_at(code_units, temp); temp += char_count(cp); then set pos = temp), or sum char_counts without calling code_point_at past the end. Also validate bounds before calling code_point_at.
